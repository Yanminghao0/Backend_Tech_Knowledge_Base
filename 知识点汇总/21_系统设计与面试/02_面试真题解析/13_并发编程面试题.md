## 1. 锁机制

**Q: synchronized vs ReentrantLock？**

```
        synchronized                    ReentrantLock
关键字     JVM层面(JVM负责加锁释放)          API层面(手动lock/unlock，必须finally释放)
公平性     非公平                            可选公平/非公平
可中断     不可中断                           可中断(lockInterruptibly)
超时获取   不支持                             支持tryLock(timeout)
条件变量   一个(wait/notify)                 多个Condition
锁绑定     对象/类                            Lock对象
性能       JDK6+优化后接近                    略优

选择: 简单场景用synchronized，需要高级功能(公平/中断/超时/多条件)用ReentrantLock
```

**Q: synchronized的锁升级过程？**

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁 (不可逆)

偏向锁: 第一个线程访问，Mark Word记录线程ID，后续无CAS
  (JDK 15+默认禁用JEP 374，后续版本逐步移除)
轻量级锁: 多线程交替(无竞争)，CAS自旋
重量级锁: 真正竞争，操作系统互斥量，线程阻塞唤醒

Mark Word状态(64位):
  偏向锁:   [线程ID(54)][epoch(2)][age(4)][biased(1)][lock(01)]
  轻量级锁: [指向栈中锁记录指针(62)][lock(00)]
  重量级锁: [指向ObjectMonitor指针(62)][lock(10)]
  GC标记:   [空][lock(11)]
```

**Q: CAS原理？有什么问题？**

```
CAS(V, expected, new): 如果V==expected，则V=new，否则重试
底层: lock cmpxchg指令(原子性)

问题:
  1. ABA问题: A→B→A，CAS认为没变 → 解决: AtomicStampedReference(版本号)
  2. 自旋开销: 长时间CAS浪费CPU → 解决: 限制自旋次数
  3. 只保证一个变量原子 → 解决: AtomicReference包装对象
```

**Q: AQS原理？**

```
AbstractQueuedSynchronizer — JUC锁和同步器的基础框架

核心:
  state(volatile int): 同步状态(0=空闲, >0=持有)
  CLH队列: 双向链表，等待获取锁的线程封装为Node
  exclusiveOwnerThread: 持有锁的线程

ReentrantLock实现:
  非公平锁(默认): tryAcquire先CAS抢，失败再入队
  公平锁: tryAcquire先检查队列是否有前驱，有则排队

Semaphore: state=许可数，acquire减1，release加1
CountDownLatch: state=计数，countDown减1，await等待归零
```

## 2. volatile与JMM

**Q: volatile原理？**

```
两个语义:
  1. 可见性: 写操作立即刷新到主内存，读操作从主内存读
     底层: x86用lock前缀指令(锁缓存行+MESI协议)
  2. 有序性: 禁止指令重排序(内存屏障)
     写前插入StoreStore屏障，写后插入StoreLoad屏障
     读前插入LoadLoad屏障，读后插入LoadStore屏障

不保证原子性: i++是读-改-写三步，volatile不保证
  → 用AtomicInteger或synchronized
```

**Q: JMM(Java内存模型)的happens-before？**

```
8条规则:
  1. 程序顺序规则: 同线程中，前面的操作happens-before后面的
  2. 监视器锁规则: unlock happens-before 后续lock
  3. volatile规则: 写 happens-before 后续读
  4. 线程启动规则: start() happens-before 线程内所有操作
  5. 线程终止规则: 线程内操作 happens-before terminate()
  6. 线程中断规则: interrupt() happens-before 被中断线程检测到
  7. 对象终结规则: 初始化 happens-before finalize()
  8. 传递性: A→B, B→C, 则 A→C
```

**Q: ThreadLocal原理？内存泄漏？**

```
每个Thread有ThreadLocalMap
ThreadLocalMap的Entry: WeakReference<ThreadLocal> key + Object value

内存泄漏原因:
  ThreadLocal被GC回收(key=null)，但value还在(强引用)
  线程池中线程长期存活 → value一直不回收

解决: 用完务必remove()
```

## 3. 线程池

**Q: 线程池7个核心参数？**

```
1. corePoolSize:    核心线程数(不会被回收，除非allowCoreThreadTimeOut)
2. maximumPoolSize: 最大线程数
3. keepAliveTime:   非核心线程空闲存活时间
4. unit:            时间单位
5. workQueue:       任务队列(ArrayBlockingQueue/LinkedBlockingQueue/SynchronousQueue)
6. threadFactory:   线程工厂(命名/优先级/守护线程)
7. handler:         拒绝策略
```

**Q: 线程池execute流程？**

```
1. 线程数 < corePoolSize → 新建核心线程执行
2. 线程数 >= corePoolSize → 任务入队列
3. 队列满 + 线程数 < maxPoolSize → 新建非核心线程
4. 队列满 + 线程数 >= maxPoolSize → 拒绝策略
```

**Q: 4种拒绝策略？**

```
AbortPolicy(默认): 抛RejectedExecutionException
CallerRunsPolicy: 调用者线程执行(降级保护)
DiscardPolicy: 静默丢弃
DiscardOldestPolicy: 丢弃队列最老任务，重新提交
```

**Q: 线程池大小怎么设置？**

```
CPU密集型: N+1 (N=CPU核数，+1防偶发页面缺失等)
IO密集型: 2N 或 N*(1+等待时间/计算时间)
混合型: 拆分为CPU密集和IO密集两个线程池

获取CPU核数: Runtime.getRuntime().availableProcessors()

阿里规范: 不允许用Executors.newFixedThreadPool(队列无界OOM风险)
  → 必须用new ThreadPoolExecutor(...)手动创建
```

## 4. 并发容器

**Q: ConcurrentHashMap JDK 7 vs JDK 8？**

```
JDK 7: Segment[] + HashEntry[]，分段锁(16段)，并发度=段数
JDK 8: Node[] + 链表/红黑树，CAS + synchronized锁桶头节点
  - 并发度更高(每个桶独立锁)
  - 链表>8转红黑树
  - 查询O(logN) vs O(N)
  - 不需要ReentrantLock，用synchronized(轻量)
```

**Q: CopyOnWriteArrayList原理？**

```
写时复制:
  1. 加ReentrantLock
  2. 复制原数组→新数组
  3. 新数组添加元素
  4. 替换旧数组引用

读无锁(读旧数组快照)
适用: 读多写极少(监听器列表/配置缓存)
缺点: 写开销大(全量复制) + 最终一致性(读到旧数据)
```

## 5. 并发工具

**Q: CountDownLatch vs CyclicBarrier vs Semaphore？**

```
CountDownLatch: 一次性，N个线程都完成后再继续
  场景: 主线程等待N个子任务完成
  countDown()减1，await()等归零

CyclicBarrier: 可重用，N个线程互相等待到齐后一起继续
  场景: 多线程计算后汇总
  await()等待，到N后全部释放，可重置

Semaphore: 许可证，控制并发数
  场景: 限流(数据库连接池/接口限流)
  acquire()获取许可，release()释放
```

**Q: CompletableFuture常用API？**

```java
// 异步执行
CompletableFuture.supplyAsync(() -> queryUser(id), executor);

// 链式转换
future.thenApply(user -> user.getName())
      .thenAccept(System.out::println);

// 组合
CompletableFuture<String> f1 = CompletableFuture.supplyAsync(...);
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(...);
f1.thenCombine(f2, (r1, r2) -> r1 + r2);

// 等待全部完成
CompletableFuture.allOf(f1, f2, f3).join();

// 任一完成
CompletableFuture.anyOf(f1, f2, f3).join();

// 异常处理
future.exceptionally(ex -> "fallback")
      .whenComplete((result, ex) -> log(...));
```

## 6. 实战场景题

**Q: 三个线程交替打印ABC打印10次？**

```java
// 方案1: synchronized + wait/notify
private int state = 0; // 0=A, 1=B, 2=C

synchronized void print(String name, int targetState, int nextState) {
    for (int i = 0; i < 10; i++) {
        while (state != targetState) {
            wait();
        }
        System.out.print(name);
        state = nextState;
        notifyAll();
    }
}
// Thread-A: print("A", 0, 1)
// Thread-B: print("B", 1, 2)
// Thread-C: print("C", 2, 0)
```

```java
// 方案2: Lock + Condition(更灵活)
ReentrantLock lock = new ReentrantLock();
Condition condA = lock.newCondition();
Condition condB = lock.newCondition();
Condition condC = lock.newCondition();
// 唤醒精确的下一个线程，不用notifyAll
```

```java
// 方案3: LockSupport.park/unpark(最简单)
// Thread-A打印后unpark(B), B打印后unpark(C), C打印后unpark(A)
```

**Q: 生产者消费者(BlockingQueue实现)？**

```java
BlockingQueue<String> queue = new ArrayBlockingQueue<>(10);

// 生产者
new Thread(() -> {
    while (true) {
        queue.put(produce()); // 满了自动阻塞
    }
}).start();

// 消费者
new Thread(() -> {
    while (true) {
        String data = queue.take(); // 空了自动阻塞
        consume(data);
    }
}).start();
```

**Q: 如何控制方法并发数为N？**

```java
Semaphore semaphore = new Semaphore(5); // 最多5个并发

public void limitedMethod() {
    try {
        semaphore.acquire();
        doBusiness();
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        semaphore.release();
    }
}
```

## 7. 面试题速查

**Q1: synchronized vs Lock？**
```
JVM vs API / 非公平 vs 可选 / 不可中断 vs 可中断 / 1个条件 vs 多个Condition
```

**Q2: volatile作用？**
```
可见性(刷主内存) + 有序性(内存屏障禁重排)，不保证原子性

```

## 8. 补充面试题

**Q: synchronized底层实现(monitor)？**

```
字节码层面:
  monitorenter: 获取monitor所有权(计数器+1)
  monitorexit:  释放monitor(计数器-1, 为0时释放)

ObjectMonitor(C++):
  _owner:    持有锁的线程
  _count:    重入计数
  _WaitSet:  wait状态的线程集合
  _EntryList: 阻塞等待锁的线程集合

加锁流程:
  1. 线程进入_EntryList
  2. 获取monitor → _owner=当前线程, _count=1
  3. 重入 → _count++
  4. wait() → 进入_WaitSet, 释放monitor
  5. notify() → 从_WaitSet移到_EntryList
  6. 释放 → _count--, _owner=null
```

**Q: ReentrantReadWriteLock读写锁原理？**

```
AQS的state(int, 32位)拆分:
  高16位: 读锁(共享)获取次数
  低16位: 写锁(排他)重入次数

读锁(共享): 多个线程可同时持有
  → 高16位+1, ThreadLocal记录每个线程的重入次数

写锁(排他): 独占
  → 低16位+1

规则:
  - 有读锁时不能加写锁(避免写饥饿)
  - 有写锁时不能加读锁
  - 写锁可降级为读锁(先获取写锁→获取读锁→释放写锁)
  - 读锁不能升级为写锁(死锁风险)

场景: 缓存更新(读多写少)
  读: readLock.lock() → 读缓存 → unlock()
  写: writeLock.lock() → 写缓存 → unlock()
```

**Q: StampedLock(乐观读)？**

```
JDK 8引入, 比ReadWriteLock更高性能

三种模式:
  1. 写锁: writeLock() → 排他
  2. 悲观读锁: readLock() → 共享
  3. 乐观读: tryOptimisticRead() → 无锁! 返回stamp

乐观读流程:
  long stamp = lock.tryOptimisticRead(); // 乐观读(无锁)
  // ... 读取数据 ...
  if (!lock.validate(stamp)) {           // 检查读取期间是否有写
      stamp = lock.readLock();           // 升级为悲观读
      try { /* 重新读取 */ } finally { lock.unlockRead(stamp); }
  }

优势: 读多写少场景性能远超ReadWriteLock(乐观读不加锁)
缺点: 不可重入, 容易死锁(不在finally释放)
```

**Q: 锁消除和锁粗化？**

```
锁消除(JIT优化):
  JIT逃逸分析发现锁对象不会被其他线程访问 → 删除synchronized
  例: StringBuffer.append()是synchronized的
      但局部变量sb不会逃逸 → JIT消除锁

锁粗化(JIT优化):
  循环内每次synchronized → 合并为循环外一次synchronized
  for (int i = 0; i < 1000; i++) {
      synchronized(lock) { list.add(i); }
  }
  → JIT粗化为:
  synchronized(lock) { for(...) { list.add(i); } }
```

**Q: 为什么ConcurrentHashMap不允许null？**

```
原因: 解决歧义问题
  map.get(key)返回null → 可能是value=null, 也可能是key不存在
  在并发环境下无法用containsKey验证(两次操作之间可能被修改)

HashMap允许null: 单线程环境下可以用containsKey验证
ConcurrentHashMap禁止null: 多线程下containsKey验证不可靠

 concurrentHashMap.put(key, null); // NPE
```

**Q: BlockingQueue的7个实现类？**

```
1. ArrayBlockingQueue:    有界数组+FIFO, 单锁(put/take共用ReentrantLock)
2. LinkedBlockingQueue:   有界链表+FIFO, 双锁(put/take各自锁, 高并发更好)
3. SynchronousQueue:      容量0, put必须等take(直接传递, 线程池cachedPool用)
4. DelayQueue:            延迟队列, 元素到期才能take(定时任务用)
5. PriorityBlockingQueue: 无界+优先级(Comparator)
6. LinkedTransferQueue:   transfer()直接传递给消费者(无消费者时阻塞)
7. LinkedBlockingDeque:   双端队列(工作窃取用)

线程池中使用:
  newFixedThreadPool → LinkedBlockingQueue(无界! OOM风险)
  newCachedThreadPool → SynchronousQueue
  newScheduledThreadPool → DelayQueue
```

**Q: 线程的生命周期状态？**

```
NEW:           new Thread()创建但未start
RUNNABLE:      start()后, 可运行(可能在等CPU时间片)
BLOCKED:       等待synchronized锁
WAITING:       wait()/join()/LockSupport.park() (无超时)
TIMED_WAITING: sleep(t)/wait(t)/join(t)/parkNanos(t)
TERMINATED:    线程执行完毕

注意:
  - JDK线程状态没有"运行中"(RUNNING), RUNNABLE包含就绪和运行
  - BLOCKED只针对synchronized, ReentrantLock的等待是WAITING
  - 调用wait()会释放锁, 调用sleep()不会释放锁
```

**Q: 为什么不建议用Executors创建线程池？**

```
阿里规约禁止原因:
  newFixedThreadPool: LinkedBlockingQueue无界(Integer.MAX_VALUE)
    → 任务堆积导致OOM
  newCachedThreadPool: maximumPoolSize=Integer.MAX_VALUE
    → 线程无限创建导致OOM
  newSingleThreadExecutor: 同FixedThreadPool, 无界队列OOM

正确做法:
  new ThreadPoolExecutor(
      corePoolSize, maxPoolSize, keepAliveTime, unit,
      new ArrayBlockingQueue<>(1000),  // 有界队列
      new ThreadFactoryBuilder().setNameFormat("biz-%d").build(),
      new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
  );
```
