## 1. 线上问题排查

**Q: 线上CPU突然飙高，怎么排查？**

```
1. top -c → 找到CPU最高的Java进程PID
2. top -Hp <PID> → 找到CPU最高的线程TID
3. printf "%x\n" <TID> → 转十六进制(如0x1a3b)
4. jstack <PID> | grep -A 30 "0x1a3b" → 看线程在执行什么代码
5. 或用arthas: thread -n 3 → 直接看CPU最高的3个线程

常见原因:
  - 死循环/while(true)没有sleep
  - 频繁Full GC(对象创建太快/内存泄漏)
  - 正则回溯灾难
  - 序列化/反序列化大对象
```

**Q: 线上接口突然变慢，怎么排查？**

```
排查链路(由外到内):
  1. 网络: ping/mtr排除网络延迟
  2. 网关: Gateway日志看耗时(路由/限流/转发)
  3. 应用: APM(SkyWalking)看耗时分布(哪个方法慢)
  4. 数据库: 慢查询日志 + EXPLAIN(是否走索引/全表扫描)
  5. 缓存: Redis慢日志/缓存击穿(大量请求穿透到DB)
  6. 外部调用: HTTP/RPC超时(第三方接口变慢)
  7. JVM: jstat -gcutil看GC频率(STW停顿)
  8. 线程池: 是否打满(请求排队等待)
  9. 锁竞争: jstack看BLOCKED线程

arthas trace命令逐层看耗时:
  trace com.example.OrderService createOrder '#cost > 100'
```

**Q: 线上OOM怎么排查？**

```
1. 确认OOM类型:
   - Java heap space: 堆内存不足
   - Metaspace: 类元数据溢出(动态生成类太多)
   - Direct buffer memory: 直接内存不足(NIO)
   - GC overhead limit exceeded: GC花太多时间但回收很少

2. 导出堆转储:
   jmap -dump:format=b,file=heap.hprof <PID>
   或JVM参数 -XX:+HeapDumpOnOutOfMemoryError(提前配置)

3. MAT分析:
   - Dominator Tree → 找占用最大的对象
   - Leak Suspects → 自动分析泄漏点
   - 查看GC Root引用链 → 定位代码

常见泄漏: ThreadLocal未remove/静态集合增长/资源未关闭
```

**Q: 数据库连接池满怎么排查？**

```
1. 看连接池监控(HikariCP): active/idle/queued
2. jstack看线程是否大量WAITING(等待连接)
3. 排查慢SQL: SHOW PROCESSLIST / 慢查询日志
4. 排查事务未关闭: @Transactional方法耗时过长(长事务持有连接)
5. 排查连接泄漏: getConnection后未close(try-with-resources)
6. 临时措施: 增大maximumPoolSize(治标)/杀掉慢查询(治标)
```

## 2. 性能调优场景

**Q: 一个接口响应时间从50ms涨到500ms，你会怎么优化？**

```
分层排查+优化:
  数据库层:
    - EXPLAIN看执行计划(是否走索引/扫描行数)
    - 索引优化(联合索引/覆盖索引/避免回表)
    - 大表分页: 深度分页用游标(id > last_id)

  应用层:
    - 串行→并行(CompletableFuture)
    - 热点数据缓存(Redis)
    - 批量操作替代循环单条查询
    - 异步化(非核心链路走MQ)

  JVM层:
    - GC调优(减少Full GC)
    - 对象池化(减少GC压力)

  架构层:
    - 读写分离
    - 分库分表
    - CDN静态资源
```

**Q: 如何设计一个接口限流方案？**

```
单机: Guava RateLimiter(令牌桶) / Semaphore(并发数)
分布式: Redis + Lua脚本(原子性) / Sentinel

方案对比:
  计数器: 简单但有临界问题(窗口边界突发)
  滑动窗口: 解决临界问题(Sentinel默认)
  漏桶: 匀速流出(消息队列削峰)
  令牌桶: 允许突发(RateLimiter)

实战推荐: Sentinel(QPS限流/并发线程数限流/热点参数限流)
```

## 3. 编码场景题

**Q: 两个线程交替打印1-100？**

```java
// 方案: synchronized + wait/notify
private int num = 1;
private boolean flag = true; // true=线程A, false=线程B

synchronized void printA() {
    while (num <= 100) {
        while (!flag) { wait(); }
        if (num > 100) break;
        System.out.println("A: " + num++);
        flag = false;
        notifyAll();
    }
}

synchronized void printB() {
    while (num <= 100) {
        while (flag) { wait(); }
        if (num > 100) break;
        System.out.println("B: " + num++);
        flag = true;
        notifyAll();
    }
}
```

**Q: 写一个简单的延迟任务(不用MQ)？**

```java
// 方案1: DelayQueue(JUC内置)
```

## 4. 项目介绍(STAR法则)

**Q: 请介绍一下你的项目？**

```
STAR法则:
  Situation(背景): 业务场景+技术栈+团队规模
  Task(任务): 你负责什么+面临什么问题
  Action(行动): 你做了什么(技术方案+关键决策)
  Result(结果): 量化成果(性能提升x%/成本降低y%)

模板:
  "我们的项目是XX系统，基于Spring Cloud微服务架构，
  日均PV XX万。我负责XX模块，主要解决XX问题。
  我通过XX技术方案(具体说了什么技术选型、为什么选)，
  最终QPS从XX提升到XX，响应时间从XX降到XX。"

关键点:
  - 突出你的贡献(不是团队做了什么，而是你做了什么)
  - 量化结果(数字最有说服力)
  - 技术深度(选型理由、权衡、踩过的坑)
  - 30秒能说完(面试官会追问细节)
```

**Q: 项目中遇到最大的技术挑战是什么？**

```
回答框架:
  1. 描述问题: 具体场景+影响面
  2. 排查过程: 怎么定位的(体现方法论)
  3. 解决方案: 临时方案→根治方案
  4. 预防措施: 监控/告警/代码规范

反面教材(避免):
  - "没有遇到什么大问题" (说明没有深度参与)
  - "百度搜了一下改了配置就好了" (没有技术深度)
  - 只说现象不说方案 (没有解决问题能力)
```

## 8. 补充面试题

**Q: 线上频繁Full GC怎么排查？**

```
1. jstat -gcutil <pid> 1000 → 看各区域使用率和GC频率
  O区使用率持续>90% → 老年代空间不足
  M区使用率>95% → Metaspace不足

2. jmap -histo:live <pid> | head -20 → 看占用最大的对象类型

3. 排查方向:
  a. 内存泄漏: jmap -dump → MAT分析 → 找GC Root引用链
  b. 大对象: 大List/大数组直接进老年代
  c. 代码问题: static集合无限增长/ThreadLocal未remove/缓存无淘汰
  d. 参数问题: 堆太小/Survivor太小导致过早晋升
  e. System.gc(): 代码或框架显式调用(加-XX:+DisableExplicitGC)

4. 临时方案: 重启大法(治标不治本)
5. 根治: 修复代码/调整JVM参数/增加监控告警
```

**Q: Redis缓存雪崩/击穿/穿透的区别和解决方案？**

```
缓存穿透: 查询不存在的key, 缓存和DB都没有
  → 布隆过滤器(拦截不存在的key)
  → 缓存null值(短TTL)

缓存击穿: 热点key过期, 大量请求穿透到DB
  → 互斥锁(只让一个线程查DB, 其他等待)
  → 热点key永不过期(异步刷新)

缓存雪崩: 大量key同时过期, DB压力暴增
  → TTL加随机值(避免同时过期)
  → 多级缓存(本地缓存+Redis)
  → 熔断降级(Hystrix/Sentinel保护DB)
  → Redis集群高可用(防止Redis宕机导致全量雪崩)
```

**Q: 如何设计一个接口的幂等性？**

```
场景: 用户重复点击/网络重试/MQ重复消费

方案1: 唯一索引(数据库)
  INSERT时冲突忽略 → 防止重复创建

方案2: Token机制
  1. 前端先获取token
  2. 提交时带上token
  3. 服务端Redis SETNX(token) → 第一次成功, 重复返回

方案3: 状态机
  订单: 待支付→已支付(重复支付不变状态)

方案4: 乐观锁
  UPDATE order SET status=1, version=version+1
  WHERE id=? AND version=?

方案5: 分布式锁
  Redis锁(lockKey=业务ID) → 同一请求串行化
```

**Q: 两个线程交替打印1-100(扩展)？**

```java
// 方案: ReentrantLock + 2个Condition
ReentrantLock lock = new ReentrantLock();
Condition oddCond = lock.newCondition();   // 奇数线程
Condition evenCond = lock.newCondition();  // 偶数线程
int count = 1;

// 奇数线程
lock.lock();
try {
    while (count <= 100) {
        if (count % 2 == 0) oddCond.await(); // 偶数则等待
        System.out.println(Thread.currentThread().getName() + ": " + count++);
        evenCond.signal(); // 唤醒偶数线程
    }
} finally { lock.unlock(); }

// TransferQueue方案(最简洁)
TransferQueue<Integer> queue = new LinkedTransferQueue<>();
// 线程A: 取偶数打印, 放奇数
// 线程B: 取奇数打印, 放偶数
```
