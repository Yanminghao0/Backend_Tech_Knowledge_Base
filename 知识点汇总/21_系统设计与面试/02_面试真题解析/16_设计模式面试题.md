## 1. 设计模式六大原则

**Q: 设计模式的六大原则？**

```
1. 单一职责(SRP): 一个类只做一件事(Service/DAO分离)
2. 开闭原则(OCP): 对扩展开放，对修改关闭(策略模式+工厂)
3. 里氏替换(LSP): 子类能替换父类且不破坏行为(子类不削弱父类功能)
4. 依赖倒置(DIP): 依赖抽象不依赖具体(面向接口编程)
5. 接口隔离(ISP): 接口精细化，不强迫实现不需要的方法
6. 迪米特法则(LoD): 最少知道，只和直接朋友通信(降低耦合)
```

## 2. 创建型模式

**Q: 工厂方法 vs 抽象工厂？**

```
工厂方法: 一个工厂创建一种产品
  abstract class PaymentFactory { abstract Payment create(); }
  class AlipayFactory { Payment create() { return new Alipay(); } }

抽象工厂: 一个工厂创建一族产品
  abstract class UIFactory {
      abstract Button createButton();
      abstract Input createInput();
  }
  class WindowsUIFactory { ... }  // Windows风格按钮+输入框
  class MacUIFactory { ... }       // Mac风格按钮+输入框

区别: 工厂方法=单一产品, 抽象工厂=产品族(多维度)
```

**Q: 建造者模式？什么场景用？**

```
适用: 多参数构造(尤其有可选参数)
// 传统: new User("name", null, null, 25, null, "addr")  // 可读性差
// Builder:
User user = User.builder()
    .name("张三")
    .age(25)
    .email("zhangsan@test.com")
    .build();

Lombok @Builder注解自动生成
MyBatis SqlSession的Configuration.Builder
```

## 3. 结构型模式

**Q: 代理模式: 静态代理 vs JDK动态代理 vs CGLIB？**

| 维度 | 静态代理 | JDK动态代理 | CGLIB |
|------|---------|------------|-------|
| 原理 | 手写代理类 | Proxy+InvocationHandler | 字节码生成子类 |
| 要求 | 接口+实现 | 必须有接口 | 不需要接口(final类不行) |
| 性能 | - | 较快 | 更快(方法内联) |
| Spring | - | AOP默认(有接口时) | AOP备选(无接口时) |

```
JDK动态代理核心:
  Proxy.newProxyInstance(loader, interfaces, InvocationHandler)
  → 生成$Proxy0类实现接口，方法调用转发给InvocationHandler.invoke()

CGLIB核心:
  Enhancer.create(父类, MethodInterceptor)
  → 生成子类，方法调用转发给interceptor.intercept()
```

**Q: 装饰器 vs 代理 区别？**

```
装饰器: 增强功能，关注点在于"添加行为"
  BufferedReader(Reader) → 给Reader加缓冲能力
  用户知道用的是装饰器

代理: 控制访问，关注点在于"拦截"
  Spring AOP代理 → 在方法前后加事务/日志
  用户不知道用的是代理(透明的)

代码结构相似，意图不同:
  装饰器: 用户主动创建(wrapper)
  代理: 框架/容器创建(对用户透明)
```

## 4. 行为型模式

**Q: 模板方法模式？Spring中哪里用？**

```
定义骨架，子类实现细节:
  abstract class AbstractTemplate {
      void execute() {        // final防止覆写
          step1();
          step2();
          hook();             // 钩子方法(可选实现)
      }
      abstract void step1();
      abstract void step2();
      void hook() {}          // 默认空实现
  }

Spring中的应用:
  JdbcTemplate: execute(模板) → 各回调(ConnectionCallback/RowMapper)
  RestTemplate, RedisTemplate: 同模式
  AbstractApplicationContext.refresh(): 定义了Spring启动流程骨架
```

**Q: 责任链模式？**

```
请求沿着链传递，每个节点决定处理或传递给下一个

Spring中的应用:
  FilterChain: Servlet Filter链(doFilter递归)
  InterceptorChain: Spring MVC HandlerInterceptor链
  Spring Security: FilterChainProxy

代码:
  abstract class Handler {
      Handler next;
      void handle(Request req) {
          if (canHandle(req)) doHandle(req);
          else if (next != null) next.handle(req);
      }
      abstract boolean canHandle(Request req);
      abstract void doHandle(Request req);
  }
```

## 5. Spring中的设计模式

**Q: Spring框架用到了哪些设计模式？**

```
工厂模式:     BeanFactory/ApplicationContext创建Bean
单例模式:     Bean默认scope=singleton
代理模式:     AOP(JDK动态代理/CGLIB)
模板方法:     JdbcTemplate/RestTemplate/RedisTemplate
观察者模式:   ApplicationEvent + ApplicationListener
适配器模式:   HandlerAdapter适配不同Controller
装饰器模式:   BeanWrapper/各种Wrapper
责任链模式:   FilterChain/InterceptorChain
策略模式:     Resource接口(ClassPathResource/FileSystemResource)
适配器模式:  AdvisorAdapter适配不同Advice类型

面试答法: 挑3-5个你最熟悉的展开说，结合源码或实际使用
```

**Q: 实际项目中你用了哪些设计模式？(高频行为题)**

```
回答模板: 场景 → 问题 → 选用模式 → 效果

示例1(策略+工厂):
  "支付系统多种支付方式(支付宝/微信/银行)，原来用if-else判断，
  后续每次新增支付方式都要改核心代码。我用策略模式定义PayStrategy
  接口，每种支付一个实现类，工厂模式根据type获取策略。新增支付方式
  只需加实现类+注册，符合开闭原则。"

示例2(模板方法):
  "对接多个第三方数据源(银行/运营商/征信)，流程都是: 鉴权→请求→
  解析→存储。我用模板方法定义AbstractDataProvider骨架，各数据源
  只需实现鉴权和解析细节，复用70%代码。"

示例3(观察者):
  "订单创建后需要触发: 发邮件/加积分/推消息/更新统计。原来在一个
  方法里串行调用，耦合且慢。我用Spring事件机制(ApplicationEvent)，
  订单创建后发布事件，各模块监听处理，解耦且可异步。"
```

## 7. 补充面试题

**Q: 适配器模式？Spring中哪里用？**

```
适配器: 把一个接口转换成另一个接口(让不兼容的接口协同工作)

类适配器(继承):  Adapter extends Adaptee implements Target
对象适配器(组合): Adapter implements Target { Adaptee adaptee; }

Spring中的应用:
  HandlerAdapter: 适配不同类型的Controller
    SimpleControllerHandlerAdapter  → Controller接口
    RequestMappingHandlerAdapter    → @RequestMapping注解
    HttpRequestHandlerAdapter       → HttpRequestHandler接口
  → DispatcherServlet通过HandlerAdapter统一调用不同Controller

实际场景:
  老系统接口不兼容新系统 → 写适配器转换
  第三方SDK接口不统一 → 适配器统一封装
```

**Q: 外观模式(Facade)？**

```
为复杂子系统提供统一入口(简化调用)

// 复杂子系统
class CPU { void freeze() {} void execute() {} }
class Memory { void load() {} }
class Disk { void read() {} }

// 外观
class Computer {
    void start() {
        cpu.freeze();
        memory.load();
        disk.read();
        cpu.execute();
    }
}
// 客户端只需 new Computer().start()

Spring中的应用:
  JdbcOperations(JdbcTemplate) → 封装Connection/Statement/ResultSet
  SLF4J → 封装不同日志框架的统一接口
```

**Q: 组合模式(Composite)？**

```
树形结构, 统一处理叶子和组合节点

场景: 文件系统(文件+文件夹)/组织架构(员工+部门)/菜单树

// 组件接口
interface Component {
    void operation();
    void add(Component c);
    void remove(Component c);
}
// 叶子(文件)
class File implements Component { ... }
// 组合(文件夹)
class Directory implements Component {
    List<Component> children = new ArrayList<>();
    void operation() { children.forEach(Component::operation); }
}
```

**Q: 状态模式(State)？**

```
对象行为随状态变化(用类代替if-else状态判断)

场景: 订单状态(待支付→已支付→已发货→已完成/已取消)

interface OrderState {
    void handle(OrderContext ctx);
}
class PendingPayState implements OrderState {
    void handle(OrderContext ctx) { ctx.setState(new PaidState()); }
}
class PaidState implements OrderState {
    void handle(OrderContext ctx) { ctx.setState(new ShippedState()); }
}

vs 策略模式:
  策略: 客户端选择策略(外部控制)
  状态: 状态自己决定下一个状态(内部自动流转)
```

**Q: 迭代器模式(Iterator)？**

```
提供统一方式遍历集合, 不暴露内部结构

JDK中的应用:
  Iterator<E> { hasNext(); next(); remove(); }
  Iterable<E> { Iterator<E> iterator(); } → 支持for-each

实现:
  class MyList<T> implements Iterable<T> {
      @Override
      public Iterator<T> iterator() {
          return new Iterator<T>() {
              int cursor = 0;
              public boolean hasNext() { return cursor < size; }
              public T next() { return data[cursor++]; }
          };
      }
  }
```

**Q: 开闭原则在实际项目中怎么体现？**

```
对扩展开放(新增功能不改老代码), 对修改关闭(不修改已有代码)

反面(违反OCP):
  if (type.equals("alipay")) { ... }
  else if (type.equals("wechat")) { ... }
  // 新增支付方式需要修改这段代码

正面(符合OCP):
  interface PayStrategy { void pay(); }
  // 新增支付: 只加实现类, 不改老代码
  // 配合工厂模式自动注册

关键: 面向接口编程 + 多态 + 开闭原则三者配合
```

**Q: 实际项目中你用过哪些设计模式？(扩展版)**

```
回答框架: 场景→问题→选模式→效果

4. 适配器模式:
   "对接3个第三方支付SDK，接口各不相同。我定义统一PayApi接口，
   每个SDK写一个适配器实现，上层调用统一接口不感知差异。"

5. 责任链模式:
   "风控系统需要多重校验: 黑名单→频率→金额→设备指纹。
   我用责任链模式把每个校验封装为Handler，按顺序执行，
   任意一环不通过则中断。新增校验只需加Handler。"

6. 装饰器模式:
   "日志系统需要支持多种输出(文件/DB/Kafka)和多种格式(JSON/TEXT)。
   我用装饰器模式，基础Logger+FileDecorator+JsonDecorator自由组合。"

7. 模板方法:
   "对接多家银行接口，流程都是: 签名→请求→验签→解析。
   用AbstractBankClient定义骨架，各银行只实现差异部分。"

8. 状态模式:
   "订单状态机(待支付→已支付→已发货→已完成/已取消)，
   原来用switch-case 200多行，重构为状态模式，每个状态一个类，
   新增状态只需加类，代码清晰可维护。"
```

## 8. 更多面试题

**Q: 单例模式的反射攻击和序列化破坏？**

```
反射攻击:
  Constructor<EnumSingleton> c = EnumSingleton.class.getDeclaredConstructor();
  c.setAccessible(true);
  c.newInstance(); // 抛异常: 枚举不允许反射创建

  → 枚举单例天然防反射(JVM层面禁止)
  → DCL需在构造器中检查if(instance!=null) throw

序列化破坏:
  对象序列化→反序列化时, JVM会创建新对象(绕过构造器)
  → DCL需实现readResolve()返回单例实例
  → 枚举天然防序列化(JVM保证枚举的readResolve返回同一实例)

  private Object readResolve() {
      return INSTANCE; // 反序列化时返回单例
  }
```

**Q: Spring AOP用了哪些设计模式？**

```
1. 代理模式: JDK动态代理 / CGLIB → 核心就是代理
2. 策略模式: ProxyCreator选择代理策略(有接口JDK, 无接口CGLIB)
3. 责任链模式: Interceptor链(AspectJ Around/Before/After)
4. 模板方法: AbstractAutoProxyCreator定义创建代理的骨架
5. 适配器模式: AdvisorAdapter适配不同Advice类型
   (MethodBeforeAdviceAdapter/AfterReturningAdviceAdapter)
6. 装饰器模式: AdvisedSupport包装目标对象+切面配置

面试答法:
  "Spring AOP核心是代理模式, 有接口用JDK动态代理,
  没接口用CGLIB。切面执行用责任链模式, 各Interceptor
  按顺序执行。代理策略选择用策略模式。"
```

**Q: 实际项目中设计模式不能过度使用，怎么看？**

```
反模式(过度设计):
  - 简单CRUD强行套设计模式 → 增加复杂度
  - 只有一个实现就搞接口+工厂 → YAGNI原则
  - 需求未稳定就大量抽象 → 后期难以修改

原则:
  1. KISS(Keep It Simple): 简单优先, 需要时再引入模式
  2. YAGNI(You Ain't Gonna Need It): 不预设未来需求
  3. 三次法则: 同样代码出现3次才考虑抽象
  4. 优先组合而非继承
  5. 模式是手段不是目的 → 解决问题才是

面试回答:
  "设计模式是前人总结的最佳实践, 但不能为模式而模式。
  我的原则是: 遇到实际痛点(重复代码/if-else膨胀/扩展困难)
  时才引入模式, 且引入后要团队共识, 避免增加理解成本。"
```
