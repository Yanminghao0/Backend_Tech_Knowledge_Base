## 1. 微服务基础

**Q: 微服务 vs 单体架构？**

```
单体: 所有功能在一个WAR/JAR中，共享数据库
  优点: 简单、部署方便、调试容易
  缺点: 代码膨胀、技术栈单一、扩展只能整体扩展、一个Bug全挂

微服务: 按业务拆分独立服务，独立部署/数据库/技术栈
  优点: 独立部署、技术栈灵活、按需扩展、故障隔离
  缺点: 复杂度↑(分布式事务/服务治理/链路追踪)、运维成本↑
```

**Q: 微服务怎么拆分？**

```
原则: 单一职责、高内聚低耦合、DDD领域驱动
  - 按业务领域拆(用户/订单/商品/支付)
  - 按变更频率拆(高频变更独立出来)
  - 数据库独占(避免共享DB)

粒度: 一个微服务2-3人维护，一个服务一个明确的业务领域
拆太细: 分布式事务/网络开销/运维灾难
拆太粗: 失去微服务意义
```

## 2. 服务注册与发现

**Q: Nacos vs Eureka vs Zookeeper？**

| 维度 | Nacos | Eureka | Zookeeper |
|------|-------|--------|-----------|
| CAP | AP(默认)/CP(可切换) | AP | CP |
| 一致性 | 最终一致+Raft(CP模式) | 最终一致 | ZAB强一致 |
| 健康检查 | TCP/HTTP/自定义 | 心跳 | Session/临时节点 |
| 配置中心 | ✅ 内置 | ❌ | ❌ |
| 大规模 | 支持十万级 | 支持万级 | 不适合服务发现 |

**Q: CAP理论？为什么注册中心选AP？**

```
C(一致性) A(可用性) P(分区容错) — 三选二，网络分区P必选

注册中心选AP原因:
  - 服务发现要求高可用(注册中心挂了所有服务通信中断)
  - 短暂的数据不一致可接受(某节点还有旧服务列表，最多调用失败重试)
  - 选CP(ZK): 节点故障时leader选举期间不可用 → 服务发现中断
```

## 3. 服务熔断与限流

**Q: 熔断器三种状态？**

```
CLOSED(关闭): 正常请求，统计失败率
  失败率>阈值 → 熔断 → OPEN

OPEN(打开): 直接拒绝/走降级逻辑
  等待休眠时间 → HALF_OPEN

HALF_OPEN(半开): 放行少量请求探测
  成功 → CLOSED(恢复)
  失败 → OPEN(继续熔断)
```

**Q: Sentinel vs Hystrix？**

| 维度 | Sentinel | Hystrix |
|------|----------|---------|
| 限流 | ✅ QPS/线程数/关联/链路 | ❌ 仅熔断 |
| 熔断 | 慢调用比例/异常比例/异常数 | 异常比例 |
| 降级 | ✅ | ✅ |
| 控制台 | 实时监控+动态规则 | Dashboard(功能弱) |
| 规则配置 | 代码/控制台/Nacos动态推送 | 代码/配置 |
| 维护状态 | 活跃(阿里维护) | 停更(Netflix) |

**Q: Sentinel限流算法？**

```
1. 滑动窗口: 固定时间窗口滑动统计QPS
2. 漏桶: 请求匀速流出，超出部分排队/丢弃(削峰填谷)
3. 令牌桶: 固定速率发令牌，有令牌才能通过(允许突发)
4. 排队等待: 匀速排队，超出最大等待时间拒绝

QPS限流: 超过阈值直接拒绝(RateLimiter)
并发线程数限流: 超过阈值排队等待
```

## 4. API网关

**Q: Spring Cloud Gateway核心概念？**

```
Route(路由): ID + 目标URI + Predicate + Filter
Predicate(断言): 匹配请求(Path/Header/Method/Query/Host/Cookie/After/Before)
Filter(过滤器): 修改请求/响应(全局/路由级别)

执行流程:
  请求 → Predicate匹配 → Filter链(前置) → 转发 → Filter链(后置) → 响应

底层: Netty + Reactor(非阻塞异步)，性能是Zuul 1.x的1.6倍
```

**Q: 网关如何做灰度发布？**

```yaml
# 方案: 通过Header/Weight路由
spring:
  cloud:
    gateway:
      routes:
        - id: order-v1
          uri: lb://order-service-v1
          predicates:
            - Path=/api/order/**
            - Header=X-Version, v1        # 指定版本
          # 或用Weight
          # - Weight=group1, 90           # 90%流量到v1
        - id: order-v2
          uri: lb://order-service-v2
          predicates:
            - Path=/api/order/**
            - Header=X-Version, v2
          # - Weight=group1, 10           # 10%流量到v2
```

## 5. 服务间通信

**Q: Feign原理？如何集成Sentinel？**

```
Feign本质: JDK动态代理 + HTTP客户端(默认URLConnection)
  1. @FeignClient注解接口 → 扫描注册BeanDefinition
  2. 调用方法 → 代理对象拦截 → 解析注解构造HTTP请求
  3. 发送请求 → 解码响应

集成Sentinel:
  feign.sentinel.enabled=true
  @FeignClient(name="order", fallback=OrderFallback.class)
  → 调用失败自动走fallback类

优化: 连接池(Apache HttpClient/OkHttp替换默认URLConnection)
     超时: connectTimeout/readTimeout
     日志: FULL级别记录请求响应(生产用BASIC)
```

**Q: 同步调用 vs 异步消息？**

```
同步(Feign/RestTemplate):
  优点: 即时返回结果、编码简单
  缺点: 耦合、阻塞、级联失败(需要熔断降级)
  场景: 核心链路(下单→扣库存→创建订单)

异步(MQ):
  优点: 解耦、削峰、无阻塞
  缺点: 最终一致、复杂度↑
  场景: 非核心链路(下单后发邮件/加积分/推送通知)
```

## 7. 分布式链路追踪

**Q: TraceId如何跨服务传递？**

```
Sleuth自动注入HTTP Header:
  X-B3-TraceId: 64位hex(整条链路唯一)
  X-B3-SpanId:  64位hex(当前节点)
  X-B3-ParentSpanId: 父节点SpanId

传递链路:
  Gateway(生成TraceId) → Feign调用 → Header透传 → 下游Sleuth解析

MQ传递: 生产者在消息属性中注入TraceId，消费者解析
异步线程: Sleuth通过TaskDecorator自动传递

存储: Zipkin/Elasticsearch → 按TraceId查询整条链路
```

## 8. 面试题速查

**Q1: 微服务怎么拆？**
```
按业务领域(DDD)，单一职责，数据库独占，2-3人一个服务
```

**Q2: 注册中心AP vs CP？**
```
选AP(可用性优先)，短暂不一致可接受，CP在leader选举时不可用
```

**Q3: 熔断三状态？**
```
CLOSED(正常统计) → OPEN(熔断拒绝) → HALF_OPEN(探测恢复)
```

**Q4: Sentinel vs Hystrix？**
```
Sentinel: 限流+熔断+控制台，活跃维护
Hystrix: 仅熔断降级，已停更
```

**Q5: Gateway vs Zuul？**
```
Gateway: Netty异步非阻塞，性能1.6x Zuul1.x
Zuul1.x: 同步阻塞; Zuul2.x: 异步但Spring Cloud不集成
```

**Q6: Feign原理？**
```
JDK动态代理 → 解析注解构造HTTP请求 → 发送 → 解码响应
```

**Q7: 灰度发布？**
```
Header路由(X-Version) 或 Weight权重(90%v1+10%v2)
```

**Q8: 分布式事务方案？**
```
Seata AT(自动补偿) / TCC(手动补偿) / 消息事务(最终一致) / Saga(长事务)
```

**Q9: Nacos AP和CP怎么切换？**
```
AP(默认): 临时实例+心跳, Distro协议
CP: 永久实例+Raft协议, 配置spring.cloud.nacos.discovery.instance-type=persistent
```

**Q10: 微服务数据一致性怎么保证？**
```
核心链路: Seata AT/TCC强一致
非核心链路: 消息事务最终一致(本地消息表)
查询: 聚合查询用CQRS(读库冗余)
```

## 9. 补充面试题

**Q: 服务雪崩怎么防止？**

```
三级防护:
  1. 网关限流: Gateway + Redis限流(单机/分布式)
  2. 服务熔断: Sentinel熔断(慢调用比例/异常比例)
  3. 降级: fallback返回默认值/缓存数据

设计原则:
  - 快速失败: 不要等待超时(熔断器快速拒绝)
  - 舱壁隔离: 线程池/信号量隔离(一个服务挂不影响其他)
  - 限流: 控制入口流量(令牌桶/滑动窗口)
  - 降级: 有损服务(返回兜底数据)
```

**Q: 微服务如何做监控？**

```
三层监控:
  1. 基础设施: CPU/内存/磁盘/网络(Prometheus + node_exporter)
  2. 应用层: JVM/线程池/DB连接池(Prometheus + Micrometer)
  3. 业务层: QPS/响应时间/错误率/业务指标(自定义Metrics)

链路追踪: SkyWalking / Zipkin / Jaeger
日志聚合: ELK / Loki
告警: Alertmanager → 钉钉/飞书/企业微信
```

**Q: Nacos集群部署注意事项？**

```
1. 至少3节点(奇数, Raft选举)
2. 使用外置MySQL存储(不依赖内嵌Derby)
3. Nginx/LVS做负载均衡
4. 命名空间隔离: dev/test/prod
5. 数据持久化: 配置数据在MySQL, 服务发现数据在各节点本地(Distro协议)
```

**Q: Feign如何传递Token/Header？**

```java
// 方案1: @RequestHeader注解
@FeignClient(name = "order-service")
public interface OrderApi {
    @GetMapping("/orders/{id}")
    Order getOrder(@PathVariable Long id, @RequestHeader("Authorization") String token);
}

// 方案2: RequestInterceptor(推荐, 全局自动传递)
@Configuration
public class FeignConfig {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            HttpServletRequest req = ((ServletRequestAttributes)
                RequestContextHolder.getRequestAttributes()).getRequest();
            String token = req.getHeader("Authorization");
            if (token != null) {
                template.header("Authorization", token);
            }
        };
    }
}

// 方案3: Hystrix线程隔离下RequestContextHolder为null
// 解决: 自定义并发策略(HystrixConcurrencyStrategy)
```

## 10. 更多面试题

**Q: 微服务如何做服务降级？**

```
降级策略:
  1. 返回默认值: 用户服务挂了 → 返回"游客"用户信息
  2. 返回缓存: 商品服务挂了 → 返回Redis中的缓存数据
  3. 简化逻辑: 不调用推荐服务, 直接返回热门商品列表
  4. 异步化: 非核心操作写入MQ异步处理
  5. 限流降级: 超过阈值直接拒绝, 返回"系统繁忙"

Sentinel降级规则:
  - 慢调用比例: RT>500ms的比例>50% → 降级10s
  - 异常比例: 异常率>50% → 降级10s
  - 异常数: 1分钟内异常>100 → 降级10s

  @SentinelResource(value = "queryOrder",
      fallback = "queryOrderFallback") // 降级方法
```

**Q: 微服务版本兼容怎么处理？**

```
1. URI版本: /v1/users, /v2/users (简单直接)
2. Header版本: X-API-Version: 2
3. 兼容期: 新旧版本并存(至少3个月)
4. 废弃通知: Header加 Sunset: Sat, 31 Dec 2026 00:00:00 GMT
5. 监控: 统计旧版本调用量, 归零后下线

Spring Cloud Gateway路由:
  - Path=/v1/** → order-service-v1
  - Path=/v2/** → order-service-v2
```

