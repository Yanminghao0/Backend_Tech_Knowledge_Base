## 1. 通用问题

**Q: 为什么用消息队列？有什么缺点？**

```
用途(解耦/异步/削峰):
  解耦: 订单系统发消息，下游各系统自己消费，互不影响
  异步: 非核心链路异步化(发邮件/加积分)，响应时间从500ms→50ms
  削峰: 秒杀流量先入队列，下游按能力消费

缺点:
  系统复杂度↑: 引入MQ多一个组件，要考虑宕机/消息丢失/重复消费
  一致性: 异步处理可能失败，需要补偿机制
  运维成本: 集群部署/监控/容量规划
```

**Q: 如何保证消息不丢失？**

```
三个环节都可能丢:
  1. 生产端: 发送失败
     → 同步发送+重试(retries=3) + 开启确认(acks=all)
  2. MQ端: Broker宕机
     → Kafka: 多副本(min.insync.replicas=2)
     → RocketMQ: 同步刷盘+主从同步
     → RabbitMQ: 持久化(durable queue + persistent message)
  3. 消费端: 消费失败
     → 手动ACK(关闭自动确认)，处理成功再确认
```

**Q: 如何保证消息顺序性？**

```
全局有序: 一个Topic只一个Partition/Queue(牺牲并发)
分区有序: 同一key的消息发到同一Partition
  Kafka: 指定key → hash到同一Partition → 单线程消费
  RocketMQ: MessageQueueSelector选队列
  消费端: 单线程消费或按key分发到内存队列

注意: 消费失败重试会破坏顺序，需保证消费幂等
```

**Q: 如何处理消息重复消费(幂等性)？**

```
方案1: 数据库唯一键(INSERT冲突忽略)
方案2: Redis SETNX去重(messageId做key)
方案3: 状态机(订单: 待支付→已支付，重复消费不改变状态)
方案4: 全局唯一ID+去重表

推荐: 状态机(最优雅) + Redis去重(高性能)
```

**Q: 消息积压怎么处理？**

```
紧急处理:
  1. 扩容消费者数量(临时增加Consumer实例)
  2. 新建临时Topic+更多Partition，将积压消息转发过去
  3. 消费后直接落库/缓存，异步再处理(先消费后处理)

根治:
  1. 排查消费慢原因(慢SQL/外部调用超时)
  2. 优化消费逻辑(批量消费/异步处理)
  3. 增加监控告警(消费延迟>阈值报警)
```

## 2. Kafka核心

**Q: Kafka架构？**

```
Producer → Broker(Topic→Partition→Replica) → Consumer Group

核心概念:
  Topic: 逻辑分类
  Partition: 物理分片，有序(单分区内)，并行(多分区)
  Replica: 副本(Leader + Follower)，保证高可用
  Consumer Group: 同组内消费者各消费不同分区(负载均衡)
  Offset: 消费位置(存在__consumer_offsets Topic中)
  ISR: 与Leader同步的副本集合(in-sync replicas)
```

**Q: Kafka为什么快？**

```
1. 顺序写: 追加写磁盘，避免随机寻址(600MB/s vs 100KB/s)
2. 零拷贝: sendfile()系统调用，数据从页缓存直接到网卡
   传统: 磁盘→内核缓冲→用户缓冲→Socket缓冲→网卡(4次拷贝)
   零拷贝: 磁盘→内核缓冲→网卡(2次拷贝)
3. 页缓存: OS层面的缓存，利用系统空闲内存
4. 批量+压缩: 批量发送，减少网络IO
5. 分区并行: 多分区并发读写
```

**Q: Kafka的ack机制？**

```
acks=0:   发出去就完事，不等响应(可能丢)
acks=1:   Leader写入即返回(Leader宕机可能丢)
acks=all: 所有ISR副本写入才返回(最安全，最慢)

生产推荐: acks=all + min.insync.replicas=2
```

**Q: Kafka如何保证Exactly-Once？**

```
生产端: 幂等性(enable.idempotence=true) + 事务(transactional.id)
消费端: 事务型Consumer + 手动提交Offset
Kafka Connect: Exactly-Once语义支持

Kafka Streams: processing.guarantee=exactly_once_v2
底层: 两阶段提交(2PC) + 事务日志
```

## 3. RocketMQ核心

**Q: RocketMQ架构？**

```
Producer → NameServer → Broker(Master/Slave) → Consumer

NameServer: 轻量注册中心(无状态，可集群部署)
Broker: 消息存储(CommitLog + ConsumeQueue)
  CommitLog: 所有消息顺序写(6亿+消息单文件1GB)
  ConsumeQueue: 消息索引(按Topic+QueueId分)
Producer: 发送消息(同步/异步/单向)
Consumer: 拉取模式(Push本质是长轮询)

vs Kafka:
  Kafka依赖ZooKeeper/KRaft，RocketMQ用NameServer(更轻量)
  Kafka按Partition存文件，RocketMQ用统一CommitLog(顺序写更极致)
  RocketMQ支持延迟消息/事务消息/Kafka不支持
```

**Q: RocketMQ事务消息原理？**

```
1. Producer发半消息(Half Message) → Broker暂存(对Consumer不可见)
2. Broker返回确认 → Producer执行本地事务
3. 本地事务成功 → 发Commit → Broker投递消息
   本地事务失败 → 发Rollback → Broker删除半消息
4. 长时间未收到Commit/Rollback → 回查本地事务状态(最多15次)

适用场景: 分布式事务(订单+扣库存+加积分)
```

**Q: RocketMQ延迟消息？**

```
开源版: 18个延迟级别(1s/5s/10s/30s/1m/2m/.../2h)
  5.x+: 支持任意延迟时间
  原理: 延迟消息先存SCHEDULE_TOPIC_XXXX，定时线程到期后转到目标Topic

商业版: 任意延迟时间
```

## 4. RabbitMQ核心

**Q: RabbitMQ架构？**

```
Producer → Exchange → Queue → Consumer

Exchange类型:
  Direct: 精确匹配routing key
  Fanout: 广播到所有绑定队列
  Topic: 通配符匹配(*一个词，#零或多个词)
  Headers: 基于消息头匹配(少用)
```

**Q: RabbitMQ如何保证可靠性？**

```
1. 生产端: confirm模式(每条消息确认) + mandatory(不可路由时Return回调)
2. Broker: 持久化(durable=true + deliveryMode=2) + 镜像队列(HA)
3. 消费端: 手动ACK(autoAck=false) + 死信队列(DLX)
```

**Q: 死信队列(DLX)？**

```
消息进入死信队列的条件:
  1. 消息被拒绝(basic.reject/basic.nack)且requeue=false
  2. 消息TTL过期
  3. 队列达到最大长度

应用: 延迟消息(TTL+DLX组合实现)、异常消息处理
```

## 5. 选型对比

**Q: Kafka/RocketMQ/RabbitMQ怎么选？**

| 维度 | Kafka | RocketMQ | RabbitMQ |
|------|-------|----------|----------|
| 吞吐量 | 百万级/秒 | 十万级/秒 | 万级/秒 |
| 延迟 | ms级 | ms级 | us级 |
| 可靠性 | 副本+ISR | 同步刷盘+主从 | 持久化+镜像 |
| 顺序性 | 分区有序 | 队列有序 | 单队列有序 |
| 延迟消息 | ❌ | ✅(18级/任意) | ✅(TTL+DLX) |
| 事务消息 | ✅(2PC) | ✅(半消息) | ❌ |
| 消息回溯 | ✅(按Offset) | ✅(按时间) | ❌ |
| 协议 | 自定义 | 自定义 | AMQP |
| 适用 | 大数据/日志流 | 电商/金融 | 中小项目/ERP |

**选型建议:**
```
日志/大数据: Kafka(吞吐最高，生态最好)
电商/金融: RocketMQ(事务消息+延迟消息+高可靠)
中小项目: RabbitMQ(轻量、协议标准、管理界面好)
```

## 6. 面试题速查

**Q1: 消息不丢失三环节？**
```
生产端: 重试+确认(acks=all)
Broker: 多副本/持久化
消费端: 手动ACK
```

**Q2: 消息幂等？**
```
状态机 + 唯一键 + Redis去重
```

**Q3: 消息积压？**
```
扩容Consumer → 临时Topic转发 → 批量消费落库
```

**Q4: Kafka快的原因？**
```
顺序写 + 零拷贝(sendfile) + 页缓存 + 批量压缩 + 分区并行
```

**Q5: 事务消息原理？**
```
半消息→本地事务→Commit/Rollback→回查(15次)
```

**Q6: Kafka vs RocketMQ？**
```
Kafka: 吞吐高、大数据生态、无延迟消息
RocketMQ: 事务消息、延迟消息、NameServer轻量
```

## 7. 补充面试题

**Q: Kafka的rebalance机制？**

```
触发条件:
  1. 消费者加入/离开Consumer Group
  2. 订阅的Topic/Partition变化
  3. 消费者心跳超时(session.timeout.ms)

Rebalance流程:
  1. 选Group Coordinator(broker)
  2. Coordinator通知所有消费者"准备rebalance"
  3. 消费者发送JoinGroup请求
  4. 选出Consumer Leader
  5. Leader分配Partition策略(Range/RoundRobin/Sticky)
  6. Coordinator下发分配方案
  7. 消费者开始消费

问题: rebalance期间所有消费者停止消费(STW)
优化:
  Cooperative Rebalance(Kafka 2.4+): 增量rebalance, 只影响需要变更的分区的消费者
  Sticky Assignor: 尽量保持原分配, 减少变更
```

**Q: Kafka Partition分配策略？**

```
1. Range(默认): 按Topic, 每个消费者分配连续的Partition
   Topic-A: P0,P1 → Consumer1; P2,P3 → Consumer2
   Topic-B: P0,P1 → Consumer1; P2,P3 → Consumer2
   问题: 第一个消费者总是多分配(不公平)

2. RoundRobin: 所有Topic的Partition混合轮询
   P0→C1, P1→C2, P2→C1, P3→C2 (更公平)

3. Sticky: 尽量保持原有分配(减少rebalance时的分区迁移)
   rebalance时只把离线消费者的分区分配给其他消费者
   → 最大化利用本地缓存
```

**Q: Kafka高水位HW vs LEO？**

```
LEO(Log End Offset): 每个副本的日志末尾offset(下一条要写入的位置)
HW(High Watermark): 所有ISR副本中最小的LEO
  → 消费者只能消费HW之前的消息(保证已全部同步)

流程:
  1. Leader收到消息, LEO+1
  2. Follower拉取消息, LEO+1, 告知Leader自己的LEO
  3. Leader更新HW = min(所有ISR的LEO)
  4. Follower拉取时获取Leader的HW, 更新自己的HW

Leader故障时:
  新Leader的HW可能小于旧Leader → 未完全同步的消息丢失
  → min.insync.replicas + acks=all 保证不丢
```

**Q: RocketMQ存储模型？**

```
CommitLog: 所有消息统一存储(顺序写, 1GB/文件)
  → 不同于Kafka按Partition分文件, RocketMQ所有Topic共用一个CommitLog
  → 优点: 顺序写更极致(一个IO写所有Topic)
  → 缺点: 读取时需要索引定位

ConsumeQueue: 消息索引(按Topic+QueueId)
  每条: offset(8B) + size(4B) + tagCode(8B) = 20B
  → 消费者通过ConsumeQueue快速定位CommitLog中的消息

IndexFile: 按key/hash索引
  → 支持按messageId/keys快速查询

文件关系:
  Producer → CommitLog(写)
  CommitLog → 异步构建ConsumeQueue + IndexFile(读)
  Consumer → ConsumeQueue(查索引) → CommitLog(查消息)
```

**Q: RabbitMQ镜像队列 vs 仲裁队列？**

```
镜像队列(Mirrored Queue, 经典):
  原理: 主队列+镜像队列, 主负责读写, 同步到镜像
  问题: 主故障时可能丢失未同步消息, 网络分区时脑裂
  → RabbitMQ 4.0废弃

仲裁队列(Quorum Queue, 推荐):
  原理: 基于Raft协议, 多节点共识
  优点: 强一致(多数写入成功才确认), 无脑裂
  缺点: 性能略低于镜像队列
  → 生产推荐

创建:
  rabbitmqctl set_policy ha-all "^ha\." '{"queue-type":"quorum"}'
```

**Q: 消息队列如何实现分布式事务？**

```
方案1: 本地消息表(最常用)
  1. 业务操作+消息记录在同一个本地事务中
  2. 定时任务扫描消息表, 发送到MQ
  3. 消费者消费成功后回调确认
  4. 失败重试(幂等消费)
  → 最终一致, 简单可靠

方案2: RocketMQ事务消息
  1. 发送半消息(对消费者不可见)
  2. 执行本地事务
  3. Commit(投递) / Rollback(删除)
  4. 超时回查本地事务状态
  → 无需本地消息表, 但依赖MQ支持

方案3: Saga模式
  长流程事务, 每步有补偿操作
  → 适合跨多服务的长流程
```

**Q: 百万消息积压1小时怎么处理？**

```
紧急(5分钟内):
  1. 暂停上游生产(降级开关/限流)
  2. 扩容消费者(临时加机器/K8s扩副本)
  3. 消息直接落库不处理(先消费后处理)
     Consumer只做: INSERT INTO buffer_table
     异步线程从buffer_table消费处理

恢复:
  4. 修复消费慢的根因(慢SQL/外部依赖)
  5. 逐步消化积压(控制消费速度避免压垮下游)
  6. 消化完毕后恢复上游

预防:
  7. 消费延迟监控告警(>阈值报警)
  8. 容量规划(峰值3倍冗余)
  9. 压测验证消费能力
```
