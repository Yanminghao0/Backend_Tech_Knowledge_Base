## 1. HashMap底层原理

**Q: HashMap put方法的完整流程(源码级)？**

```java
// JDK 8 HashMap.putVal() 简化版
final V putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict) {
    Node<K,V>[] tab; int n, i;

    // 1. 数组未初始化 → 初始化(resize)
    if ((tab = table) == null || (n = tab.length) == 0)
        n = (tab = resize()).length;

    // 2. 桶为空 → 直接CAS放入
    if ((p = tab[i = (n - 1) & hash]) == null)
        tab[i] = newNode(hash, key, value, null);
    else {
        // 3. 桶非空
        Node<K,V> e; K k;
        // 3a. key相同(==或equals) → 直接覆盖
        if (p.hash == hash && ((k = p.key) == key || (key != null && key.equals(k))))
            e = p;
        // 3b. 红黑树 → 树插入
        else if (p instanceof TreeNode)
            e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);
        // 3c. 链表 → 遍历到尾部插入
        else {
            for (int binCount = 0; ; ++binCount) {
                if ((e = p.next) == null) {
                    p.next = newNode(hash, key, value, null);
                    // 链表长度>=8 → 树化
                    if (binCount >= TREEIFY_THRESHOLD - 1) // -1 for 1st
                        treeifyBin(tab, hash);
                    break;
                }
                if (e.hash == hash && ((k = e.key) == key || (key != null && key.equals(k))))
                    break;
                p = e;
            }
        }
        // 3d. 找到已有key → 更新value
        if (e != null) {
            V oldValue = e.value;
            if (!onlyIfAbsent || oldValue == null)
                e.value = value;
            return oldValue;
        }
    }
    // 4. size++ → 检查是否扩容
    if (++size > threshold)
        resize();
    return null;
}
```

**Q: HashMap的扰动函数为什么这样设计？**

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

原理: 高16位异或低16位
  - hashCode是32位int，但数组长度通常很小(16/32/64)
  - 计算桶位置: (n-1) & hash → 只用到低位bit
  - 如果hashCode高位变化大、低位变化小 → 大量冲突
  - 扰动函数让高位也参与运算 → 减少冲突

示例:
  hashCode = 0x12345678
  h >>> 16 = 0x00001234
  hash     = 0x12345678 ^ 0x00001234 = 0x1234444C
  → 高位信息"混入"了低位

为什么不用取模?
  hash % n 和 hash & (n-1) 结果相同(n是2的幂)
  但位运算比取模快5-10倍(CPU指令层面)
```

**Q: HashMap get方法流程？**

```
get(key):
  1. hash = hash(key) → 扰动
  2. index = (n-1) & hash → 定位桶
  3. 桶为空 → return null
  4. 桶头节点key匹配 → return value
  5. 是TreeNode → 红黑树查找O(logN)
  6. 是链表 → 遍历查找O(N)

为什么不加锁?
  HashMap是单线程设计，table和Node的value都是非volatile
  多线程下可能读到半初始化状态 → 所以需要ConcurrentHashMap
```

## 2. 红黑树原理

**Q: 红黑树的五大性质？**

```
1. 每个节点是红色或黑色
2. 根节点是黑色
3. null叶子是黑色(隐藏的NIL节点)
4. 红色节点的子节点必须是黑色(不能连续两个红)
5. 从任意节点到其叶子节点的所有路径包含相同数量的黑色节点(黑高相同)

推论:
  - 最短路径: 全黑 → 黑高h → 节点数 = h
  - 最长路径: 红黑交替 → 黑高h → 节点数 = 2h
  - 因此最长路径 ≤ 2 × 最短路径 → 近似平衡
  - 查找复杂度: O(logN)，最坏不超过2logN
```

**Q: 红黑树插入怎么自平衡？**

```
插入节点默认红色(不破坏黑高)

情况1: 父节点是黑色 → 直接插入，无需调整

情况2: 父节点是红色(违反性质4) → 需要调整
  2a. 叔叔是红色:
      父和叔变黑, 祖父变红
      把祖父当作新插入节点继续向上调整

  2b. 叔叔是黑色(或null):
      LL型(父在左, 新在左): 右旋祖父 + 父变黑祖父变红
      LR型(父在左, 新在右): 先左旋父 → 变成LL → 右旋祖父
      RR型(父在右, 新在右): 左旋祖父 + 父变黑祖父变红
      RL型(父在右, 新在左): 先右旋父 → 变成RR → 左旋祖父

左旋: 把右子提上来，自己变成左子
右旋: 把左子提上来，自己变成右子
```

**Q: 为什么HashMap选红黑树不选AVL树？**

```
AVL树: 严格平衡(左右子树高度差≤1) → 查找更快(严格O(logN))
红黑树: 近似平衡(最长≤2×最短) → 插入删除旋转更少

HashMap场景:
  - 哈希冲突时才用树(冲突率极低, 链表>8概率0.00000006)
  - 树化后插入/删除频率 > 查找频率(HashMap以put为主)
  - 红黑树插入最多2次旋转, 删除最多3次旋转
  - AVL树可能需要O(logN)次旋转(一直调整到根)
  → 红黑树的旋转次数更少, 综合性能更好

其他选择:
  - B+树(MySQL索引): 磁盘IO优化, 每节点存多个key → 减少IO次数
  - 跳表(ConcurrentSkipListMap): 无锁CAS友好, 不需要旋转
```

## 3. ConcurrentHashMap底层原理

**Q: ConcurrentHashMap put完整流程(源码级)？**

```java
// JDK 8 ConcurrentHashMap.putVal() 简化
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // 不允许null key/value(歧义问题)
    if (key == null || value == null) throw new NullPointerException();
    int hash = spread(key.hashCode()); // 扰动: (h ^ h>>>16) & HASH_BITS
    int binCount = 0;

    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;

        // 1. 数组未初始化 → initTable(CAS初始化)
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();

        // 2. 桶为空 → CAS插入(无锁!)
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            if (casTabAt(tab, i, null, new Node<>(hash, key, value, null)))
                break;
        }

        // 3. 桶是ForwardingNode(正在扩容) → 协助扩容
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);

        // 4. 桶非空 → synchronized锁桶头节点
        else {
            synchronized (f) {
                if (tabAt(tab, i) == f) { // double-check
                    // 4a. 链表 → 遍历插入/更新
                    if (fh >= 0) {
                        binCount = 1;
                        for (Node<K,V> e = f;; ++binCount) {
                            // key存在 → 更新
                            // key不存在 → 尾插
                        }
                    }
                    // 4b. 红黑树 → 树插入
                    else if (f instanceof TreeBin) {
                        // putTreeVal
                    }
                }
            }
            // 链表>8 → 树化
            if (binCount >= TREEIFY_THRESHOLD)
                treeifyBin(tab, i);
        }
    }
    // 5. size计数(baseCount + CounterCell)
    addCount(1L, binCount);
    return null;
}
```

**Q: ConcurrentHashMap get为什么无锁？**

```
get(key)全程不加锁:

1. tabAt(tab, i): volatile读
   底层: Unsafe.getObjectVolatile(tab, offset)
   → 保证看到最新值(不需要synchronized)

2. Node的val和next是volatile:
   volatile V val;
   volatile Node<K,V> next;
   → 读取时直接看到最新值

3. 扩容时的可见性:
   迁移完的桶放ForwardingNode(hash=MOVED)
   → get遇到MOVED → 转发到新表(nextTab)查找
   → 不阻塞读

为什么不需要volatile读tab数组本身?
   table字段是volatile:
   volatile Node<K,V>[] table;
   → table引用的变化(扩容时替换)对所有线程可见

  但table[i]不是volatile的 → 用tabAt(Unsafe volatile读)
```

**Q: ConcurrentHashMap size计数的原理(LongAdder思想)？**

```
JDK 8 ConcurrentHashMap.size:

// 核心字段
volatile long baseCount;        // 无竞争时直接CAS
volatile CounterCell[] counterCells; // 有竞争时分段计数

// @Contended 避免伪共享
static final class CounterCell {
    volatile long value;
}

addCount过程:
  1. 尝试CAS baseCount+1
  2. CAS失败(有竞争) → 随机选一个CounterCell → CAS cell.value+1
  3. cell CAS也失败 → 扩容counterCells(2倍) → 重试

size() = baseCount + sum(counterCells)
  → 遍历所有cell累加(非原子，所以size不是精确值)

为什么不用一个AtomicLong?
  高并发下所有线程CAS同一个值 → 大量自旋失败 → 性能下降
  分散到多个cell → 不同线程操作不同cell → 减少竞争
  → 这就是LongAdder的核心思想(空间换时间)

CounterCell用@Contended注解:
  填充缓存行(64字节) → 避免多个cell在同一缓存行(伪共享)
  → 一个cell修改不会导致其他cell的缓存行失效
```

## 4. volatile底层原理

**Q: volatile在x86上怎么实现可见性？**

```
volatile写: JVM生成lock前缀指令
  汇编: lock addl $0, 0(%rsp)  (或lock cmpxchg等)

lock前缀指令的作用:
  1. 锁缓存行(Lock Cache Line)
     → 当前CPU独占缓存行(其他CPU缓存行失效)
  2. 触发MESI协议
     → 其他CPU该缓存行从Modified→Invalid
     → 其他CPU读取时从主内存/当前CPU缓存重新加载
  3. 写屏障(Store Buffer刷入缓存)
     → 确保store buffer中的写操作刷入L1缓存 → 对其他CPU可见

如果数据跨缓存行(>64字节):
  → lock指令退化为总线锁(Bus Lock) → 性能很差
  → 所以@Contended填充到缓存行对齐很重要
```

**Q: MESI缓存一致性协议？**

```
CPU缓存行(L1/L2)的四种状态:
  M(Modified):  已修改，与内存不一致，只有本CPU有副本
  E(Exclusive): 独占，与内存一致，只有本CPU有副本
  S(Shared):    共享，与内存一致，多个CPU有副本
  I(Invalid):   无效，需要重新从内存加载

状态转换(volatile写场景):
  CPU-A写volatile变量(cache line状态 M→M):
    1. CPU-A发"Read Invalidate"消息 → 总线
    2. 其他CPU的该缓存行变为I(Invalid)
    3. CPU-A独占缓存行(M状态)
    4. CPU-B读该变量 → cache miss → 从CPU-A缓存拉取 → CPU-A变S

Store Buffer(写缓冲):
  CPU写数据时先写Store Buffer(不等总线确认)
  → 提高性能(不用等其他CPU确认)
  → 但引入"写重排序"问题 → 需要StoreLoad屏障

Invalidate Queue(失效队列):
  CPU收到Invalidate消息后立即回复ACK(不等实际失效)
  → 提高性能 → 引入"读重排序"问题 → 需要LoadLoad屏障
```

**Q: 四种内存屏障？**

```
1. LoadLoad屏障: Load1; LoadLoad; Load2
   → 确保Load1完成后再Load2(防止读重排序)

2. StoreStore屏障: Store1; StoreStore; Store2
   → 确保Store1刷入内存后再Store2(防止写重排序)

3. LoadStore屏障: Load1; LoadStore; Store2
   → 确保Load1完成后再Store2

4. StoreLoad屏障: Store1; StoreLoad; Load2
   → 确保Store1对其他CPU可见后再Load2
   → 最强屏障(开销最大, 等于lock指令)
   → 防止"写读重排序"(最常见的重排序)

volatile写前插入: StoreStore(确保前面写先完成)
volatile写后插入: StoreLoad(确保写对其他CPU可见)
volatile读后插入: LoadLoad + LoadStore(确保读后再后续操作)

JMM层面: volatile禁止指令重排序
硬件层面: 插入内存屏障指令
```

## 5. CAS底层原理

**Q: CAS在CPU层面怎么保证原子性？**

```
CAS = Compare And Swap
Java层面: Unsafe.compareAndSwapInt/Object
JVM层面: Atomic::cmpxchg (C++)
CPU层面: lock cmpxchg 指令(x86)

cmpxchg指令:
  比较eax寄存器与目标内存值
  相等 → 将新值写入内存, ZF标志位置1
  不等 → 将内存值加载到eax, ZF清0

lock前缀:
  单核: cmpxchg本身就是原子的
  多核: 需要lock前缀锁缓存行(或总线) → 保证cmpxchg的"比较+写入"原子执行

  lock cmpxchg [mem], new_value
  → 锁定缓存行 → 其他CPU不能修改该内存 → 原子完成比较+交换

总线锁 vs 缓存锁:
  缓存锁(默认): 锁定缓存行(MESI) → 性能好
  总线锁(回退): 数据跨缓存行或不支持缓存锁 → 锁总线 → 性能差
```

**Q: Unsafe类和CAS？**

```java
// Unsafe.compareAndSwapInt 的C++实现(JDK)
UNSAFE_ENTRY(jboolean, Unsafe_CompareAndSwapInt(JNIEnv *env, ...)) {
  // 最终调用:
  return Atomic::cmpxchg(exchange_value, dest, compare_value) == compare_value;
}

// Atomic::cmpxchg (x86)
inline jint Atomic::cmpxchg(jint exchange_value, volatile jint* dest, jint compare_value) {
  // 内联汇编
  __asm__ volatile (
    "lock cmpxchgl %1, (%3)"       // lock前缀 + cmpxchg指令
    : "=a" (exchange_value)         // eax寄存器
    : "r" (exchange_value), "a" (compare_value), "r" (dest)
    : "cc", "memory"
  );
}
```

## 6. Java对象头详解

**Q: 64位JVM对象头布局？**

```
普通对象头(64位 + 压缩指针 = 12字节):
  ┌──────────────────────────────────────────────────┐
  │                    Mark Word (64 bits)            │
  ├──────────────────────────────────────────────────┤
  │              Klass Pointer (32 bits, 压缩)        │
  └──────────────────────────────────────────────────┘

数组对象头多4字节(数组长度):
  ┌──────────────────────────────────────────────────┐
  │                    Mark Word (64 bits)            │
  ├──────────────────────────────────────────────────┤
  │              Klass Pointer (32 bits, 压缩)        │
  ├──────────────────────────────────────────────────┤
  │              Array Length (32 bits)               │
  └──────────────────────────────────────────────────┘

Mark Word各状态(64位):

无锁:    [unused:25][hash:31][unused:1][age:4][biased:0][lock:01]
偏向锁:  [threadId:54][epoch:2][unused:1][age:4][biased:1][lock:01]
轻量级锁:[ptr_to_lock_record:62][lock:00]
重量级锁:[ptr_to_heavyweight_monitor:62][lock:10]
GC标记:  [空][lock:11]

lock: 2位, 最后2bit
  01: 无锁/偏向锁(看biased bit区分)
  00: 轻量级锁
  10: 重量级锁
  11: GC标记

JOL验证对象头:
  <dependency>
    <groupId>org.openjdk.jol</groupId>
    <artifactId>jol-core</artifactId>
  </dependency>
  System.out.println(ClassLayout.parseInstance(obj).toPrintable());
```

**Q: 为什么Klass Pointer可以压缩？**

```
JVM堆地址通常 < 32GB
32位指针可寻址4GB, 但用对象对齐(8字节) → 32位 × 8 = 32GB
→ 堆 < 32GB时用压缩指针(4字节)代替8字节指针

-XX:+UseCompressedOops (默认开启, 堆<32GB)
-XX:ObjectAlignmentInBytes=8 (默认8字节对齐)

节省内存:
  不压缩: Object头 = 8(Mark) + 8(Klass) = 16字节
  压缩后: Object头 = 8(Mark) + 4(Klass) = 12字节 → 对齐到16
  Integer: 24字节(不压缩) → 16字节(压缩)
  大量对象场景节省20-30%内存
```

## 7. synchronized锁升级全流程

**Q: 偏向锁的获取和撤销流程？**

```
偏向锁获取:
  1. 检查Mark Word的biased bit=1, lock=01(可偏向)
  2. 检查threadId是否是自己:
     是 → 直接进入临界区(无需CAS)
     否 → CAS设置threadId为自己

偏向锁撤销(发生竞争时):
  1. 全局安全点(STW)
  2. 检查持有偏向锁的线程是否存活:
     不活跃 → 重置为无锁
     活跃 → 遍历栈帧, 找到锁记录
       如果还在持有 → 升级为轻量级锁
       如果已释放 → 重置为无锁(可重新偏向)
  3. 唤醒被阻塞在安全点的线程

批量重偏向:
  同一类的偏向锁撤销超过20次(阈值)
  → 该类所有对象的偏向锁重置为可重新偏向
  → 允许重新偏向到新线程

批量撤销:
  撤销超过40次
  → 该类禁用偏向锁(新对象直接从无锁开始)
```

**Q: 轻量级锁的加锁/解锁流程？**

```
加锁:
  1. 在当前栈帧创建Lock Record(锁记录)
  2. 拷贝对象Mark Word到Lock Record(Displaced Mark Word)
  3. CAS尝试将对象Mark Word替换为指向Lock Record的指针
     成功 → 锁标志位改为00(轻量级锁)
     失败 → 自旋重试(适应性自旋)
       自旋超过阈值 → 升级重量级锁

解锁:
  1. CAS将Displaced Mark Word替换回对象Mark Word
     成功 → 释放成功
     失败 → 说明有竞争(已升级为重量级锁)
       → 唤醒等待的线程

自旋:
  JDK 6: 固定次数自旋(-XX:PreBlockSpin=10)
  JDK 7+: 适应性自旋(根据上次成功率动态调整)
  JDK 11: 默认禁用自旋(-XX:+UseSpinning)
  → 因为CPU多核时代, 自旋浪费CPU不如直接park
```

**Q: 重量级锁的ObjectMonitor原理？**

```
C++ ObjectMonitor结构(简化):
class ObjectMonitor {
    void* volatile _owner;     // 持有锁的线程
    volatile int _count;       // 重入计数
    ObjectWaiter* _WaitSet;    // wait()的线程集合(双向链表)
    ObjectWaiter* _EntryList;  // 等待进入的线程集合(双向链表)
    volatile int _recursions;  // 重入次数
}

加锁流程(monitorenter):
  1. 线程进入_EntryList(如果_owner不为空)
  2. _owner为null → CAS设置_owner=当前线程
  3. 重入 → _recursions++
  4. 获取失败 → park(操作系统级阻塞, Linux用futex)
     → 线程状态 WAITING/BLOCKED

wait()流程:
  1. 释放锁(_owner=null, _recursions=0)
  2. 线程加入_WaitSet
  3. park(等待notify)

notify()流程:
  1. 从_WaitSet移出一个线程
  2. 加入_EntryList(重新竞争锁)

为什么重量级锁慢?
  1. 线程阻塞/唤醒需要OS内核态切换(user→kernel→user)
  2. 每次切换约1-10微秒
  3. 大量线程竞争时, 内核态切换成为瓶颈
  → 这就是synchronized优化的目标: 尽量不进入重量级锁
```

## 8. AQS源码原理

**Q: AQS的CLH队列变体？**

```
AQS队列结构:
  head(哨兵) → Node → Node → Node → tail
  每个Node: thread(线程) + waitStatus + prev + next

CLH原版: 自旋检查前驱节点状态
AQS变体: 不自旋, park阻塞, 前驱节点释放时unpark后继

Node.waitStatus:
  CANCELLED(1):   线程取消
  SIGNAL(-1):     后继需要被唤醒(当前节点释放时唤醒后继)
  CONDITION(-2):  在Condition队列中
  PROPAGATE(-3):  共享模式传播(后继可以传播唤醒)
  0:              初始状态

入队流程(acquire):
  1. 创建Node(prev=tail, thread=当前线程)
  2. CAS设置tail=newNode
  3. CAS设置前驱的next=newNode
  4. 检查前驱是否是head + tryAcquire成功 → 获取锁
  5. 否则: shouldParkAfterFailedAcquire → 设置前驱waitStatus=SIGNAL → park

出队流程(release):
  1. tryRelease成功
  2. head.waitStatus != 0 → unpark后继线程
  3. 后继线程被唤醒 → 重新tryAcquire → 成功 → 设置head=自己
```

**Q: AQS独占模式 vs 共享模式？**

```
独占模式(ReentrantLock):
  acquire(1):
    if (!tryAcquire(1) && 
        acquireQueued(addWaiter(EXCLUSIVE), 1))
      selfInterrupt();

  tryAcquire由子类实现:
    ReentrantLock.FairSync: 先检查队列有无人排队 → 排队
    ReentrantLock.NonFairSync: 直接CAS抢 → 抢不到再排队

共享模式(Semaphore/CountDownLatch):
  acquireShared(1):
    if (tryAcquireShared(1) < 0)  // <0表示获取失败
      doAcquireShared(1);

  doAcquireShared:
    入队 → park → 被唤醒 → tryAcquireShared
    如果成功(r >= 0) → 传播唤醒(setHeadAndPropagate)
    → 唤醒后继共享节点(级联唤醒)

  传播唤醒的意义:
    Semaphore: 一个许可释放 → 唤醒一个等待者
    CountDownLatch: count=0 → 唤醒所有等待者(级联传播)
```

**Q: ConditionObject原理？**

```
Condition是AQS的内部类, 维护独立的等待队列:

  同步队列(AQS的CLH队列): 等待获取锁的线程
  等待队列(Condition的队列): 调用await()的线程

await()流程:
  1. 创建Node(CONDITION模式) → 加入Condition等待队列
  2. 释放锁(fullyRelease: 释放所有重入)
  3. park(在Condition队列上等待)
  4. 被signal唤醒 → 从Condition队列移到AQS同步队列
  5. 重新竞争锁(acquireQueued)
  6. 从await()返回

signal()流程:
  1. 从Condition等待队列移出第一个Node
  2. 转移到AQS同步队列(enq: Node模式改为EXCLUSIVE)
  3. 设置前驱节点waitStatus=SIGNAL
  4. 如果前驱已取消或CAS失败 → unpark(立即唤醒)

应用: ReentrantLock.newCondition()
  生产者消费者: notFull.await()/notFull.signal()
  阻塞队列: put时notFull.await(), take后notFull.signal()
```

## 9. ReentrantLock源码

**Q: ReentrantLock公平锁tryAcquire源码？**

```java
// FairSync.tryAcquire
protected final boolean tryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        // 公平: 先检查队列有没有人排队
        if (!hasQueuedPredecessors() &&
            compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    }
    else if (current == getExclusiveOwnerThread()) {
        // 重入: state++
        int nextc = c + acquires;
        if (nextc < 0) throw new Error("Maximum lock count exceeded");
        setState(nextc);
        return true;
    }
    return false;
}
```

**Q: 非公平锁tryAcquire的区别？**

```java
// NonFairSync.tryAcquire
final boolean nonfairTryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        // 非公平: 直接CAS抢(不检查队列)
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    }
    // ... 重入逻辑相同
}
```

```
公平 vs 非公平区别只有一行:
  公平: !hasQueuedPredecessors() && CAS
  非公平: CAS (直接抢)

为什么默认非公平?
  公平锁每次都要检查队列 → 上下文切换多 → 吞吐量低
  非公平锁允许"插队" → 减少线程切换 → 吞吐量高约30%
  代价: 可能导致队列中的线程长时间拿不到锁(饥饿)
```

## 10. 线程池源码原理

**Q: 线程池ctl变量和状态流转？**

```java
// ThreadPoolExecutor用一个int的高3位存状态, 低29位存线程数
private final AtomicInteger ctl = new AtomicInteger(ctlOf(RUNNING, 0));

// 状态(高3位):
RUNNING    111: 接受新任务, 处理队列任务
SHUTDOWN   000: 不接受新任务, 处理队列任务 (shutdown())
STOP       001: 不接受新任务, 不处理队列, 中断进行中 (shutdownNow())
TIDYING    010: 所有任务终止, workerCount=0, 调terminated()
TERMINATED 011: terminated()执行完成

// 状态转换:
RUNNING → SHUTDOWN (shutdown())
RUNNING/SHUTDOWN → STOP (shutdownNow())
SHUTDOWN → TIDYING (队列+线程池空)
STOP → TIDYING (线程池空)
TIDYING → TERMINATED (terminated()完成)

// 为什么用一个int?
  → 线程数和工作状态需要原子性更新
  → 一个CAS操作同时更新两者(避免两个变量的CAS同步问题)
```

**Q: addWorker源码流程？**

```java
private boolean addWorker(Runnable firstTask, boolean core) {
    // 1. 外层循环: 检查状态和线程数
    retry:
    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c);
        // 状态检查: SHUTDOWN时不接受新任务(除非队列为空且firstTask==null)
        if (rs >= SHUTDOWN && !(rs == SHUTDOWN && firstTask == null && !workQueue.isEmpty()))
            return false;

        // 2. 内层循环: CAS增加线程数
        for (;;) {
            int wc = workerCountOf(c);
            if (wc >= CAPACITY || wc >= (core ? corePoolSize : maximumPoolSize))
                return false;
            if (compareAndIncrementWorkerCount(c))  // CAS +1
                break retry;
            // CAS失败 → 重读状态 → 重试
        }
    }

    // 3. 创建Worker并启动线程
    Worker w = new Worker(firstTask);
    Thread t = w.thread;
    workers.add(w);  // HashSet<Worker>
    t.start();        // 启动线程 → 执行runWorker

    return true;
}
```

**Q: runWorker源码(线程怎么持续获取任务)？**

```java
final void runWorker(Worker w) {
    Thread wt = Thread.currentThread();
    Runnable task = w.firstTask;
    w.firstTask = null;
    w.unlock(); // Worker继承AQS, 允许中断

    try {
        // 循环获取任务
        while (task != null || (task = getTask()) != null) {
            w.lock();  // 加锁(标识线程在工作)
            // 检查状态: 如果STOP → 中断
            if ((runStateAtLeast(ctl.get(), STOP) ||
                 (Thread.interrupted() && runStateAtLeast(ctl.get(), STOP))))
                wt.interrupt();

            try {
                beforeExecute(wt, task);  // 钩子方法
                try {
                    task.run();            // 执行任务!
                    afterExecute(task, null);  // 钩子方法
                } catch (Exception ex) {
                    afterExecute(task, ex);
                }
            } finally {
                task = null;
                w.completedTasks++;
                w.unlock();
            }
        }
    } finally {
        workerExited(w);  // 线程退出清理
    }
}

// getTask: 从队列获取任务(阻塞)
private Runnable getTask() {
    boolean timedOut = false;
    for (;;) {
        // 判断是否需要超时回收(非核心线程 or allowCoreThreadTimeOut)
        boolean timed = allowCoreThreadTimeOut || wc > corePoolSize;
        if (wc > maximumPoolSize || (timed && timedOut))
            // 超过max或超时 → 减少线程数 → return null → 线程退出
            return null;

        try {
            Runnable r = timed ?
                workQueue.poll(keepAliveTime, TimeUnit.NANOSECONDS) : // 超时获取
                workQueue.take();  // 阻塞获取
            if (r != null) return r;
            timedOut = true;
        } catch (InterruptedException retry) {
            timedOut = false;
        }
    }
}
```
