## 1. 测试覆盖率基础概念

### 1.1 什么是测试覆盖率

测试覆盖率（Test Coverage）是指测试代码执行时覆盖到的被测代码的比例。它回答的问题是："我的测试运行了多少代码？"

```
覆盖率 = (被测试执行到的代码 / 总代码) × 100%
```

> **重要区分：** 覆盖率 ≠ 测试质量。100% 覆盖率不代表所有场景都被测试到，只代表所有代码行都被执行过。

### 1.2 覆盖率的局限

| 覆盖率能告诉你 | 覆盖率不能告诉你 |
|--------------|----------------|
| 哪些代码没有被测试执行 | 测试是否正确验证了业务逻辑 |
| 哪些分支没有被覆盖 | 测试是否覆盖了边界条件 |
| 测试的广度 | 测试的深度和质量 |
| 代码的测试空白区域 | 代码是否真正正确 |

### 1.3 常见覆盖率工具对比

| 工具 | 语言 | 特点 | 集成方式 |
|------|------|------|---------|
| **JaCoCo** | Java | 最主流，支持行/分支/方法覆盖 | Maven/Gradle 插件 |
| Cobertura | Java | 老牌工具，已较少使用 | Maven 插件 |
| Istanbul/nyc | JavaScript/TypeScript | Node.js 生态主流 | npm 集成 |
| Coverage.py | Python | Python 生态标准 | pytest-cov |
| Go cover | Go | 内置工具 | go test -cover |

## 2. JaCoCo 配置详解

### 2.1 Maven 集成

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.jacoco</groupId>
            <artifactId>jacoco-maven-plugin</artifactId>
            <version>0.8.12</version>
            <executions>
                <!-- 1. 在测试运行前准备 JaCoCo Agent -->
                <execution>
                    <id>prepare-agent</id>
                    <goals>
                        <goal>prepare-agent</goal>
                    </goals>
                </execution>

                <!-- 2. 测试运行后生成报告 -->
                <execution>
                    <id>report</id>
                    <phase>test</phase>
                    <goals>
                        <goal>report</goal>
                    </goals>
                </execution>

                <!-- 3. 覆盖率检查（CI 门禁） -->
                <execution>
                    <id>check</id>
                    <goals>
                        <goal>check</goal>
                    </goals>
                    <configuration>
                        <rules>
                            <rule>
                                <element>BUNDLE</element>
                                <limits>
                                    <limit>
                                        <counter>LINE</counter>
                                        <value>COVEREDRATIO</value>
                                        <minimum>0.80</minimum>
                                    </limit>
                                    <limit>
                                        <counter>BRANCH</counter>
                                        <value>COVEREDRATIO</value>
                                        <minimum>0.75</minimum>
                                    </limit>
                                </limits>
                            </rule>
                        </rules>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

### 2.2 Gradle 集成

```groovy
plugins {
    id 'jacoco'
}

jacoco {
    toolVersion = '0.8.12'
}

test {
    finalizedBy jacocoTestReport
    jacoco {
        enabled = true
        destinationFile = file("$buildDir/jacoco/jacoco.exec")
    }
}

jacocoTestReport {
    dependsOn test
    reports {
        xml.required = true
        html.required = true
        csv.required = false
        html.outputLocation = layout.buildDirectory.dir('reports/jacoco/html')
        xml.outputLocation = layout.buildDirectory.file('reports/jacoco/jacoco.xml')
    }
}

jacocoTestCoverageVerification {
    violationRules {
        rule {
            limit {
                counter = 'LINE'
                value = 'COVEREDRATIO'
                minimum = 0.80
            }
            limit {
                counter = 'BRANCH'
                value = 'COVEREDRATIO'
                minimum = 0.75
            }
        }
    }
}

check.dependsOn jacocoTestCoverageVerification
```

### 2.3 排除不需要覆盖的类

```xml
<configuration>

        <!-- 排除配置类 -->
```

## 3. 覆盖率维度解析

### 3.1 行覆盖率（Line Coverage）

行覆盖率是最直观的指标，衡量有多少行代码被测试执行：

```java
public class Calculator {
    public int divide(int a, int b) {
        if (b == 0) {                    // 行1：if 条件
            throw new ArithmeticException("除数不能为零"); // 行2：异常分支
        }                                 // 行3
        return a / b;                     // 行4：正常返回
    }
}
```

如果测试只测了 `divide(10, 2)`，行覆盖率为 3/4 = 75%（行2未执行）。

### 3.2 分支覆盖率（Branch Coverage）

分支覆盖率衡量条件分支的覆盖情况：

```java
public String classify(int score) {
    if (score >= 90) {           // 分支1：true / false
        return "优秀";
    } else if (score >= 60) {    // 分支2：true / false
        return "及格";
    } else {                      // 分支3
        return "不及格";
    }
}
```

| 测试输入 | 覆盖的分支 | 分支覆盖率 |
|---------|-----------|-----------|
| score=95 | 分支1=true | 1/4 = 25% |
| score=95, score=70 | 分支1=false, 分支2=true | 2/4 = 50% |
| score=95, score=70, score=50 | 全部 | 4/4 = 100% |

### 3.3 方法覆盖率（Method Coverage）

```java
public class UserService {
    public User findById(Long id) { ... }     // 方法1
    public User create(String name) { ... }   // 方法2
    public void delete(Long id) { ... }        // 方法3
    public User update(User user) { ... }      // 方法4
}
```

如果测试只调用了 `findById` 和 `create`，方法覆盖率为 2/4 = 50%。

### 3.4 类覆盖率（Class Coverage）

类覆盖率衡量有多少类被至少一个测试调用过：

```java
// 如果 UserServiceTest 只测试了 UserService
// 但没有测试 EmailService，那么类覆盖率为 1/2 = 50%
```

### 3.5 指令覆盖率（Instruction Coverage）

JaCoCo 的最细粒度指标，统计 Java 字节码指令的覆盖情况：

```
指令覆盖率 = 被执行的字节码指令数 / 总字节码指令数
```

### 3.6 覆盖率维度对比

| 维度 | 粒度 | 直观性 | 价值 |
|------|------|--------|------|
| 行覆盖 | 代码行 | ⭐⭐⭐⭐⭐ | 最直观，常用 |
| 分支覆盖 | 条件分支 | ⭐⭐⭐⭐ | 发现遗漏的分支场景 |
| 方法覆盖 | 方法 | ⭐⭐⭐ | 宏观了解覆盖范围 |
| 类覆盖 | 类 | ⭐⭐ | 粗粒度概览 |
| 指令覆盖 | 字节码 | ⭐ | 最精确，但不直观 |

## 5. 覆盖率目标设定

### 5.1 合理的覆盖率目标

| 覆盖率范围 | 评价 | 适用场景 |
|-----------|------|---------|
| 90-100% | 优秀 | 核心业务逻辑、金融系统 |
| 80-90% | 良好 | 大多数业务系统 |
| 70-80% | 合格 | 边缘模块、工具类 |
| 60-70% | 需改进 | 遗留系统、过渡阶段 |
| <60% | 风险高 | 不应长期存在 |

### 5.2 按模块设定差异化目标

```xml
<rules>
    <!-- 核心业务模块：高要求 -->
    <rule>
        <element>BUNDLE</element>
        <includes>com.example.core.*</includes>
        <limits>
            <limit>
                <counter>LINE</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.90</minimum>
            </limit>
            <limit>
                <counter>BRANCH</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.85</minimum>
            </limit>
        </limits>
    </rule>

    <!-- 控制器层：中等要求 -->
    <rule>
        <element>BUNDLE</element>
        <includes>com.example.controller.*</includes>
        <limits>
            <limit>
                <counter>LINE</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.75</minimum>
            </limit>
        </limits>
    </rule>

    <!-- 全局底线 -->
    <rule>
        <element>BUNDLE</element>
        <limits>
            <limit>
                <counter>LINE</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.70</minimum>
            </limit>
        </limits>
    </rule>
</rules>
```

### 5.3 渐进式覆盖率提升

```xml
<!-- 使用 ratchet 策略：只允许覆盖率上升，不允许下降 -->
<configuration>
    <rules>
        <rule>
            <element>CLASS</element>
            <limits>
                <limit>
                    <counter>LINE</counter>
                    <value>COVEREDRATIO</value>
                    <!-- 当前基线，只增不减 -->
                    <minimum>0.75</minimum>
                </limit>
            </limits>
        </rule>
    </rules>
</configuration>
```

## 6. CI 门禁集成

### 6.1 Maven CI 门禁

```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.12</version>
    <executions>
        <execution>
            <id>prepare-agent</id>
            <goals><goal>prepare-agent</goal></goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>verify</phase>
            <goals><goal>report</goal></goals>
        </execution>
        <execution>
            <id>coverage-check</id>
            <phase>verify</phase>
            <goals><goal>check</goal></goals>
            <configuration>
                <haltOnFailure>true</haltOnFailure> <!-- 覆盖率不达标则构建失败 -->
                <rules>
                    <rule>
                        <element>BUNDLE</element>
                        <limits>
                            <limit>
                                <counter>LINE</counter>
                                <value>COVEREDRATIO</value>
                                <minimum>0.80</minimum>
                            </limit>
                            <limit>
                                <counter>BRANCH</counter>
                                <value>COVEREDRATIO</value>
                                <minimum>0.75</minimum>
                            </limit>
                            <limit>
                                <counter>METHOD</counter>
                                <value>COVEREDRATIO</value>
                                <minimum>0.80</minimum>
                            </limit>
                        </limits>
                    </rule>
                </rules>
            </configuration>
        </execution>
    </executions>
</plugin>
```

### 6.2 GitHub Actions 集成

```yaml
name: CI with Coverage

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'

      - name: Run tests with coverage
        run: mvn clean verify

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v4

          token: ${{ secrets.CODECOV_TOKEN }}

      - name: Upload JaCoCo HTML report
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: target/site/jacoco/
```

## 7. 多模块项目覆盖率聚合

### 7.1 问题：多模块覆盖率分散

在多模块 Maven 项目中，每个模块独立生成覆盖率报告，无法看到全局覆盖率。

### 7.2 使用 JaCoCo 聚合报告

```xml
<!-- 在聚合模块（通常是父 POM 或专门的 report 模块）中配置 -->
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.12</version>
    <executions>
        <execution>
            <id>report-aggregate</id>
            <phase>verify</phase>
            <goals>
                <goal>report-aggregate</goal>
            </goals>
            <configuration>
                <dataFileIncludes>
                    <dataFileInclude>**/jacoco.exec</dataFileInclude>
                </dataFileIncludes>
                <outputDirectory>${project.reporting.outputDirectory}/jacoco-aggregate</outputDirectory>
            </configuration>
        </execution>
    </executions>
    <dependencies>
        <!-- 依赖所有子模块 -->
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>core-module</artifactId>
            <version>${project.version}</version>
        </dependency>
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>web-module</artifactId>
            <version>${project.version}</version>
        </dependency>
    </dependencies>
</plugin>
```

### 7.3 项目结构示例

```
my-project/
├── pom.xml (父POM)
├── core-module/
│   ├── src/
│   └── pom.xml
├── web-module/
│   ├── src/
│   └── pom.xml
└── coverage-report/    ← 专门的覆盖率聚合模块
    └── pom.xml
```

```xml
<!-- coverage-report/pom.xml -->
<project>
    <artifactId>coverage-report</artifactId>
    <packaging>pom</packaging>

    <dependencies>
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>core-module</artifactId>
            <version>${project.version}</version>
        </dependency>
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>web-module</artifactId>
            <version>${project.version}</version>
        </dependency>
    </dependencies>

    <build>
        <plugins>
            <plugin>
                <groupId>org.jacoco</groupId>
                <artifactId>jacoco-maven-plugin</artifactId>
                <executions>
                    <execution>
                        <id>report-aggregate</id>
                        <phase>verify</phase>
                        <goals><goal>report-aggregate</goal></goals>
                    </execution>
                </executions>
            </plugin>
        </plugins>
    </build>
</project>
```

## 8. 常见误区与反模式

### 8.1 误区1：追求 100% 覆盖率

```java
// ❌ 为了覆盖率而写无意义的测试
@Test
void testGetter() {
    User user = new User(1L, "张三");
    assertEquals(1L, user.getId());     // 只是为了覆盖 getter
    assertEquals("张三", user.getName()); // 只是为了覆盖 getter
}

// ✅ 测试有意义的业务逻辑
@Test
void shouldRejectMinorUserRegistration() {
    User user = new User("张三", 16); // 未成年
    assertThrows(ValidationException.class, () -> registrationService.register(user));
}
```

### 8.2 误区2：高覆盖率 = 高质量

```java
// 覆盖率 100%，但测试毫无价值
@Test
void shouldReturnSomething() {
    UserService service = new UserService(repository);
    service.findById(1L); // 调用了但没断言
    // 覆盖率：100%
    // 价值：0
}
```

### 8.3 误区3：忽略分支覆盖

```java
public String getGrade(int score) {
    if (score >= 90) return "A";
    if (score >= 80) return "B";
    if (score >= 60) return "C";
    return "F";
}

// ❌ 只有一个测试，行覆盖率 100%，但分支覆盖率只有 50%
@Test
void testGetGrade() {
    assertEquals("A", service.getGrade(95));
}

// ✅ 覆盖所有分支
@Test
void testAllGrades() {
    assertEquals("A", service.getGrade(95));
    assertEquals("B", service.getGrade(85));
    assertEquals("C", service.getGrade(65));
    assertEquals("F", service.getGrade(30));
}
```

### 8.4 误区4：排除过多代码

```xml
<!-- ❌ 过度排除，导致覆盖率虚高 -->
<excludes>
    <exclude>**/service/**</exclude>    <!-- 排除了所有 Service -->
    <exclude>**/controller/**</exclude>  <!-- 排除了所有 Controller -->
    <exclude>**/repository/**</exclude>  <!-- 排除了所有 Repository -->
    <!-- 结果：只测了 DTO，覆盖率 95%，但毫无意义 -->
</excludes>

<!-- ✅ 只排除真正不需要测试的 -->
<excludes>
    <exclude>**/dto/**</exclude>
    <exclude>**/entity/**</exclude>
    <exclude>**/config/**</exclude>
    <exclude>**/Application.*</exclude>
</excludes>
```

### 8.5 误区5：一次性设定过高门槛

```
❌ 遗留系统覆盖率 30%，直接设门禁 80% → 所有人都无法通过 CI
✅ 渐进式提升：30% → 35% → 40% → ... → 80%
```

## 9. 质量指标体系

### 9.1 超越覆盖率的全面质量指标

| 指标 | 工具 | 说明 |
|------|------|------|
| 测试覆盖率 | JaCoCo | 行/分支/方法覆盖 |
| 代码复杂度 | SonarQube | 圈复杂度（Cyclomatic Complexity） |
| 代码重复 | SonarQube/PMD | 重复代码比例 |
| 代码规范 | Checkstyle/SpotBugs | 编码规范遵循 |
| 安全漏洞 | SpotBugs/OWASP | 静态安全分析 |
| 测试质量 | PiTest/Mutation Testing | 变异测试 |
| 技术债务 | SonarQube | 代码异味和债务 |

### 9.2 变异测试（Mutation Testing）

变异测试通过修改代码来检验测试的有效性：

```java
// 原始代码
public int add(int a, int b) {
    return a + b;
}

// 变异1：a + b → a - b
// 变异2：a + b → a * b
// 变异3：return a → return b
```

如果测试能检测出变异（测试失败），说明测试有效；如果测试仍然通过，说明测试不够。

### 9.3 SonarQube 质量门禁

```yaml
# SonarQube Quality Gate 示例
Quality Gate:
  - Coverage ≥ 80%
  - Branch Coverage ≥ 70%
  - Duplicated Lines ≤ 3%
  - Cyclomatic Complexity ≤ 15
  - Critical Issues = 0
  - Blocker Issues = 0
  - Technical Debt Ratio ≤ 5%
```

### 9.4 测试质量评估清单

```
□ 覆盖率：行覆盖 ≥ 80%，分支覆盖 ≥ 75%
□ 断言密度：每个测试方法至少1个有意义的断言
□ 测试命名：清晰表达测试意图（should_When_）
□ 测试隔离：测试之间无依赖
□ 测试速度：单元测试 < 1秒，集成测试 < 10秒
□ 测试稳定性：无 Flaky Test（不稳定测试）
□ 变异测试得分 ≥ 80%
```

## 10. 面试题速查

**Q1: 行覆盖率和分支覆盖率的区别？**
- 行覆盖率：被执行到的代码行比例
- 分支覆盖率：条件分支被覆盖的比例（if/else/switch 的每个分支）
- 分支覆盖率通常低于行覆盖率，更能反映测试完整性

**Q2: 100% 覆盖率意味着什么？**
- 只意味着所有代码行都被执行过
- 不代表所有边界条件都被测试
- 不代表测试断言充分

**Q3: JaCoCo 的 prepare-agent 和 report 的作用？**
- `prepare-agent`：在 JVM 启动时注入 JaCoCo Agent，收集执行数据
- `report`：将收集的数据生成 HTML/XML/CSV 报告

**Q4: 如何处理多模块项目的覆盖率聚合？**
- 使用 `report-aggregate` 目标
- 在专门的聚合模块中配置
- 依赖所有子模块

**Q5: 覆盖率门禁应该如何设置？**
- 全局底线：行覆盖 70-80%
- 核心模块：85-90%
- 渐进式提升，避免一次性设置过高门槛

**Q6: 什么是变异测试？它和覆盖率的关系？**
- 变异测试通过修改代码验证测试是否能检测到变更
- 覆盖率告诉你"代码被执行了"，变异测试告诉你"测试是否有效"

**Q7: 哪些类应该从覆盖率中排除？**
- DTO/Entity/Model 等纯数据类
- 配置类
- 自动生成的代码
- 主启动类

**Q8: 如何避免 Flaky Test 对覆盖率的影响？**
- 确保测试隔离性
- 使用 Testcontainers 替代 Mock 复杂依赖
- 在 CI 中监控测试稳定性
- 使用 `@RepeatedTest` 发现不稳定测试
