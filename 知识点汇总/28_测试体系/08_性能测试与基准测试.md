## 1. 性能测试概述

### 1.1 性能测试分类

```
┌──────────────────────────────────────────────────────────┐
│                    性能测试体系                            │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  1. 微基准测试 (Micro Benchmark)                          │
│     - 方法级别的性能测量                                   │
│     - 工具: JMH                                           │
│     - 关注: 单个方法的执行时间/吞吐量                       │
│     - 场景: 算法对比、数据结构选择、API优化                 │
│                                                          │
│  2. 集成性能测试                                          │
│     - 接口级别的性能测量                                   │
│     - 工具: JMeter, Gatling                               │
│     - 关注: API响应时间、吞吐量、错误率                     │
│     - 场景: 接口压测、并发测试                             │
│                                                          │
│  3. 系统性能测试 (Load Test)                              │
│     - 系统级别的负载测试                                   │
│     - 模拟真实用户负载                                     │
│     - 关注: 系统整体表现                                   │
│     - 场景: 峰值测试、容量规划                             │
│                                                          │
│  4. 压力测试 (Stress Test)                                │
│     - 超出系统承载能力的测试                                │
│     - 关注: 系统在极限条件下的行为                         │
│     - 场景: 找到系统瓶颈和崩溃点                           │
│                                                          │
│  5. 持久测试 (Soak/Endurance Test)                       │
│     - 长时间持续负载测试                                   │
│     - 关注: 内存泄漏、连接泄漏、性能衰减                    │
│     - 场景: 系统稳定性验证                                 │
│                                                          │
│  6. 峰值测试 (Spike Test)                                 │
│     - 瞬间高负载测试                                       │
│     - 关注: 系统对突发流量的处理能力                       │
│     - 场景: 秒杀、促销活动                                 │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 1.2 性能指标

```yaml
核心性能指标:

响应时间 (Response Time / Latency):
  - 平均响应时间
  - P50: 50%的请求响应时间低于此值
  - P90: 90%的请求响应时间低于此值
  - P95: 95%的请求响应时间低于此值
  - P99: 99%的请求响应时间低于此值
  - Max: 最大响应时间
  
  注意: 不要只看平均值，P99更能反映用户体验

吞吐量 (Throughput):
  - TPS (Transactions Per Second): 每秒事务数
  - QPS (Queries Per Second): 每秒查询数
  - RPS (Requests Per Second): 每秒请求数

资源利用率:
  - CPU使用率 (%)
  - 内存使用率 (%)
  - 磁盘I/O (IOPS, MB/s)
  - 网络I/O (Mbps)
  - GC频率和时间

错误率:
  - HTTP错误率 (4xx, 5xx)
  - 超时率
  - 业务错误率

并发指标:
  - 并发用户数
  - 同时连接数
  - 线程池活跃线程数
```

## 2. JMH微基准测试

### 2.1 为什么需要JMH

```java
// ❌ 自己写微基准测试的陷阱

public class NaiveBenchmark {
    public static void main(String[] args) {
        long start = System.currentTimeMillis();
        
        long sum = 0;
        for (int i = 0; i < 1000000; i++) {
            sum += calculate(i);
        }
        
        long end = System.currentTimeMillis();
        System.out.println("耗时: " + (end - start) + "ms");
    }
    
    private static int calculate(int x) {
        return x * x;
    }
}

// 陷阱1: JIT编译器优化
// JVM可能直接计算出结果（死代码消除），不实际执行循环
// sum变量没有被使用，JVM可能优化掉整个循环

// 陷阱2: 没有预热
// 第一次执行包含类加载、JIT编译等开销
// 需要先预热让JIT编译完成

// 陷阱3: 没有考虑GC
// GC暂停会干扰测量结果

// 陷阱4: System.currentTimeMillis()精度不够
// 应该使用System.nanoTime()

// JMH解决了以上所有问题
```

### 2.3 第一个JMH基准测试

```java
import org.openjdk.jmh.annotations.*;
import org.openjdk.jmh.results.format.ResultFormatType;
import org.openjdk.jmh.runner.Runner;
import org.openjdk.jmh.runner.RunnerException;
import org.openjdk.jmh.runner.options.Options;
import org.openjdk.jmh.runner.options.OptionsBuilder;

import java.util.concurrent.TimeUnit;

@BenchmarkMode(Mode.AverageTime)        // 测量模式: 平均时间
@OutputTimeUnit(TimeUnit.MILLISECONDS)   // 输出时间单位
@Warmup(iterations = 3, time = 1)       // 预热: 3轮，每轮1秒
@Measurement(iterations = 5, time = 1)  // 测量: 5轮，每轮1秒
@Fork(2)                                // Fork 2个进程
@State(Scope.Benchmark)                  // 状态共享范围
public class StringConcatBenchmark {
    
    private String prefix = "Hello";
    private String suffix = "World";
    private int iterations = 1000;
    
    // 字符串拼接基准测试
    @Benchmark
    public String concatWithPlus() {
        String result = "";
        for (int i = 0; i < iterations; i++) {
            result = result + prefix + suffix;
        }
        return result;
    }
    
    // StringBuilder基准测试
    @Benchmark
    public String concatWithStringBuilder() {
        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < iterations; i++) {
            sb.append(prefix).append(suffix);
        }
        return sb.toString();
    }
    
    // String.format基准测试
    @Benchmark
    public String concatWithFormat() {
        String result = "";
        for (int i = 0; i < iterations; i++) {
            result = String.format("%s%s", result, prefix + suffix);
        }
        return result;
    }
    
    public static void main(String[] args) throws RunnerException {
        Options opt = new OptionsBuilder()
            .include(StringConcatBenchmark.class.getSimpleName())
            .result("target/jmh-results.json")
            .resultFormat(ResultFormatType.JSON)
            .build();
        
        new Runner(opt).run();
    }
}

/* 输出示例:
Benchmark                              Mode  Cnt   Score   Error  Units
StringConcatBenchmark.concatWithPlus   avgt   10  45.234 ± 2.123  ms/op
StringConcatBenchmark.concatWithStringBuilder  avgt   10   0.123 ± 0.005  ms/op
StringConcatBenchmark.concatWithFormat avgt   10  85.678 ± 3.456  ms/op
*/
```

## 3. JMH注解详解

### 3.1 测量模式

```java
@BenchmarkMode({
    Mode.AverageTime,    // 平均执行时间
    Mode.Throughput,     // 吞吐量（操作/秒）
    Mode.SampleTime,     // 采样时间（收集延迟分布）
    Mode.SingleShotTime, // 单次执行时间（不预热）
    Mode.All             // 所有模式
})
public class BenchmarkModeDemo {
    
    @Benchmark
    @BenchmarkMode(Mode.Throughput)
    @OutputTimeUnit(TimeUnit.SECONDS)
    public void measureThroughput() throws InterruptedException {
        // 测量吞吐量: 每秒能执行多少次
        TimeUnit.MILLISECONDS.sleep(10);
    }
    
    @Benchmark
    @BenchmarkMode(Mode.AverageTime)
    @OutputTimeUnit(TimeUnit.MICROSECONDS)
    public void measureAverageTime() {
        // 测量平均执行时间
        Math.sqrt(Math.random());
    }
    
    @Benchmark
    @BenchmarkMode(Mode.SampleTime)
    @OutputTimeUnit(TimeUnit.MICROSECONDS)
    public void measureSampleTime() throws InterruptedException {
        // 采样时间: P50, P90, P99等分位数
        if (Math.random() > 0.99) {
            TimeUnit.MILLISECONDS.sleep(100);  // 模拟偶发慢请求
        } else {
            TimeUnit.MICROSECONDS.sleep(100);
        }
    }
}
```

### 3.2 状态管理

```java
public class StateDemo {
    
    // Benchmark: 所有线程共享同一个实例
    @State(Scope.Benchmark)
    public static class SharedState {
        private List<Integer> data;
        
        @Setup(Level.Trial)  // 每次Trial（整个基准测试）开始时执行
        public void setup() {
            data = new ArrayList<>();
            for (int i = 0; i < 10000; i++) {
                data.add(i);
            }
        }
        
        @TearDown(Level.Trial)  // 每次Trial结束时执行
        public void tearDown() {
            data.clear();
        }
    }
    
    // Thread: 每个线程有自己的实例
    @State(Scope.Thread)
    public static class ThreadState {
        private StringBuilder sb = new StringBuilder();
        
        @Setup(Level.Invocation)  // 每次调用前执行（谨慎使用！）
        public void reset() {
            sb.setLength(0);
        }
    }
    
    // Group: 同一组的线程共享
    @State(Scope.Group)
    public static class GroupState {
        private BlockingQueue<Integer> queue = new LinkedBlockingQueue<>();
    }
    
    @Benchmark
    public void useSharedState(SharedState state) {
        int sum = state.data.stream().mapToInt(Integer::intValue).sum();
    }
    
    @Benchmark
    public void useThreadState(ThreadState state) {
        for (int i = 0; i < 100; i++) {
            state.sb.append(i);
        }
    }
}
```

### 3.4 编译器控制

```java
public class CompilerControlDemo {
    
    @Benchmark
    @CompilerControl(CompilerControl.Mode.DONT_INLINE)  // 阻止方法内联
    public int dontInline() {
        return expensiveComputation();
    }
    
    @Benchmark
    @CompilerControl(CompilerControl.Mode.INLINE)  // 强制方法内联
    public int inline() {
        return expensiveComputation();
    }
    
    @Benchmark
    @CompilerControl(CompilerControl.Mode.EXCLUDE)  // 完全禁止JIT编译
    public int exclude() {
        return expensiveComputation();
    }
    
    // 防止死代码消除
    @Benchmark
    public int preventDCE(Blackhole bh) {
        int result = 0;
        for (int i = 0; i < 1000; i++) {
            result += i;
        }
        bh.consume(result);  // 消费结果，防止JVM优化掉
        return result;
    }
    
    private int expensiveComputation() {
        int x = 0;
        for (int i = 0; i < 100; i++) {
            x += i * i;
        }
        return x;
    }
}
```

## 5. Gatling脚本

### 5.1 Scala DSL脚本

```scala
// OrderSimulation.scala
import io.gatling.core.Predef._
import io.gatling.http.Predef._
import scala.concurrent.duration._

class OrderSimulation extends Simulation {
  
  // HTTP配置
  val httpProtocol = http
    .baseUrl("http://localhost:8080")
    .acceptHeader("application/json")
    .contentTypeHeader("application/json")
    .userAgentHeader("Gatling/Performance Test")
  
  // 场景数据
  val userIds = (1 to 1000).map(_.toString).toArray
  val feeder = Iterator.continually(
    Map(
      "userId" -> userIds(scala.util.Random.nextInt(userIds.length)),
      "quantity" -> (scala.util.Random.nextInt(10) + 1).toString
    )
  )
  
  // 场景1: 浏览商品 + 下单
  val orderScenario = scenario("Create Order")
    .feed(feeder)
    .exec(
      http("List Products")
        .get("/api/products")
        .check(status.is(200))
        .check(jsonPath("$[*].sku").findRandom.saveAs("productSku"))
    )
    .pause(1, 3)  // 模拟用户思考时间
    .exec(
      http("Create Order")
        .post("/api/orders")
        .body(StringBody("""{
          "userId": ${userId},
          "items": [
            {"sku": "${productSku}", "quantity": ${quantity}}
          ]
        }""")).asJson
        .check(status.is(201))
        .check(jsonPath("$.orderId").saveAs("orderId"))
    )
    .exec(
      http("Get Order")
        .get("/api/orders/${orderId}")
        .check(status.is(200))
    )
  
  // 场景2: 高频查询
  val browseScenario = scenario("Browse Products")
    .exec(
      http("List Products")
        .get("/api/products")
        .check(status.is(200))
    )
    .pause(500.millis, 2.seconds)
  
  // 负载模型
  setUp(
    orderScenario.inject(
      rampUsers(100).during(30.seconds),       // 30秒内启动100用户
      constantUsersPerSec(10).during(2.minutes), // 每秒10个用户持续2分钟
      rampUsersPerSec(10).to(50).during(1.minute) // 从10/s增加到50/s
    ),
    browseScenario.inject(
      rampUsers(500).during(60.seconds)
    )
  ).protocols(httpProtocol)
    .assertions(
      global.responseTime.max.lt(2000),         // 最大响应时间 < 2s
      global.responseTime.percentile3.lt(500),   // P95 < 500ms
      global.successfulRequests.percent.gt(95)   // 成功率 > 95%
    )
}
```

```java
public class BenchmarkPitfalls {
    
    // ❌ 陷阱1: 死代码消除
    @Benchmark
    public void deadCodeElimination() {
        int x = 0;
        for (int i = 0; i < 1000; i++) {
            x += i;  // x从未被使用，JVM可能优化掉整个循环
        }
    }
    
    // ✅ 使用Blackhole消费结果
    @Benchmark
    public void preventDCE(Blackhole bh) {
        int x = 0;
        for (int i = 0; i < 1000; i++) {
            x += i;
        }
        bh.consume(x);  // 防止优化
    }
    
    // ❌ 陷阱2: 常量折叠
    @Benchmark
    public int constantFolding() {
        int a = 1;
        int b = 2;
        return a + b;  // JVM可能直接返回3
    }
    
    // ✅ 使用@State提供非常量
    @State(Scope.Benchmark)
    public static class Values {
        public int a = 1;
        public int b = 2;
    }
    
    @Benchmark
    public int noFolding(Values v) {
        return v.a + v.b;  // 不能确定返回值
    }
    
    // ❌ 陷阱3: 循环内测量
    @Benchmark
    public void loopInBenchmark() {
        for (int i = 0; i < 10000; i++) {
            process(i);  // JMH会调用整个方法，循环可能被优化
        }
    }
    
    // ✅ JMH本身就是循环，不需要手动循环
    @Benchmark
    public void noManualLoop() {
        process(42);  // JMH会重复调用
    }
    
    // ❌ 陷阱4: @Setup(Level.Invocation)开销过大
    @State(Scope.Thread)
    public static class BadSetup {
        List<Integer> data;
        
        @Setup(Level.Invocation)  // 每次调用都执行！
        public void setup() {
            data = new ArrayList<>();
            for (int i = 0; i < 10000; i++) {
                data.add(i);
            }
        }
    }
    // @Setup(Level.Invocation)的开销可能比被测代码还大
    // 只有在预热后确保不影响测量时才使用
    
    // ❌ 陷阱5: 没有考虑GC
    // 如果基准测试产生大量对象，GC暂停会严重影响结果
    
    // ✅ 使用@State管理对象生命周期
    @State(Scope.Benchmark)
    public static class ReusableState {
        // 复用对象，减少GC压力
    }
```

## 8. 面试题速查

**Q1: 为什么不能直接用System.currentTimeMillis()做微基准测试？**
> 1）精度问题：currentTimeMillis()精度依赖OS，可能是毫秒级。2）没有预热：JIT编译需要时间，前几次执行包含编译开销。3）死代码消除：JVM可能优化掉未被使用的计算。4）GC干扰：没有隔离GC暂停的影响。5）没有统计：单次测量无法评估误差范围。JMH解决了所有这些问题。

**Q2: JMH的预热（Warmup）为什么重要？**
> JVM在运行时会进行JIT编译——将热点代码从字节码编译为机器码。JIT编译需要时间和CPU，如果没有预热，测量结果会包含编译开销。预热让JIT编译完成后才开始正式测量，确保测量的是稳定状态下的性能。JMH默认5轮预热，每轮1秒。

**Q3: Blackhole在JMH中的作用是什么？**
> Blackhole"消费"计算结果，防止JVM的死代码消除（DCE）优化。如果计算结果没有被使用（存储、返回或打印），JVM可能认为这段代码没有副作用而直接优化掉。Blackhole.consume()告诉JVM"这个值被使用了"，确保代码实际执行。此外，Blackhole还可以避免常量折叠——防止JVM在编译期直接计算出结果。

**Q4: JMeter和Gatling有什么区别？**
> JMeter基于GUI+XML配置，上手简单但脚本维护困难，资源消耗较大（每个虚拟用户一个线程）。Gatling基于代码（Scala/Java），脚本可版本控制，资源消耗小（基于Akka的Actor模型，单机可模拟数万用户）。Gatling的报告更现代，JMeter的插件生态更丰富。推荐新项目用Gatling，遗留项目或非技术团队用JMeter。

**Q5: 性能测试应该关注P99还是平均值？**
> 都要关注但侧重不同。平均值反映整体水平，但会被少量极端值拉偏。P99（99%的请求低于此值）反映尾部延迟，更能代表真实用户体验。一个平均50ms但P99=2000ms的系统，意味着1%的用户等2秒——在1000QPS下每秒10个用户受影响。SLA通常用P95/P99定义。

**Q6: 什么是性能基线？如何管理性能回归？**
> 性能基线是系统在特定版本和环境下的性能指标记录。管理回归的方法：1）建立基线——在稳定版本上测量并记录。2）设定阈值——允许的波动范围（如±20%）。3）CI集成——每次合并自动运行性能测试。4）对比分析——当前结果与基线对比，超出阈值则报警。5）定期更新基线——性能优化后更新基线。

**Q7: 什么是false sharing（伪共享）？如何避免？**
> 多核CPU中每个核心有自己的缓存行（通常64字节），当两个线程分别修改同一缓存行中的不同变量时，会导致缓存行频繁失效和重新加载，虽然变量不共享但缓存行共享。避免方法：使用@Contended注解填充缓存行（需要-XX:-RestrictContended），或手动添加padding字段。
