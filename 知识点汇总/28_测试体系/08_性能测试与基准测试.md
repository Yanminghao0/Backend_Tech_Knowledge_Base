## 1. 性能测试体系概览

### 1.1 性能测试分类

```
┌────────────────────────────────────────────────────┐
│                  性能测试体系                        │
├─────────────┬──────────────┬───────────────────────┤
│  微基准测试  │  负载测试     │   压力测试             │
│  (JMH)      │  (JMeter)    │   (Gatling)           │
├─────────────┼──────────────┼───────────────────────┤
│ 方法/代码段  │ API/服务级    │  高并发/极限场景       │
│ 纳秒~毫秒级  │ 秒~分钟级     │  分钟~小时级           │
│ 单线程为主   │ 多线程模拟    │  万级并发              │
└─────────────┴──────────────┴───────────────────────┘
```

### 1.2 性能指标

| 指标 | 缩写 | 说明 |
|------|------|------|
| 响应时间 | RT | 请求从发出到收到响应的时间 |
| 吞吐量 | TPS/QPS | 每秒处理的事务/查询数 |
| 并发数 | CC | 同时处理的请求数 |
| 错误率 | ER | 失败请求占总请求的比例 |
| 百分位延迟 | P90/P95/P99 | 90%/95%/99% 请求的响应时间 |
| 资源利用率 | CPU/Memory/IO | 系统资源使用情况 |

## 2. JMH 微基准测试基础

### 2.1 什么是 JMH

JMH（Java Microbenchmark Harness）是 OpenJDK 团队开发的微基准测试工具，专门用于测量 Java 代码片段的精确性能。它解决了 JVM 优化带来的测量偏差问题。

### 2.2 为什么需要 JMH

```java
// ❌ 朴素基准测试：结果不可靠
long start = System.currentTimeMillis();
for (int i = 0; i < 1000000; i++) {
    doSomething();
}
long end = System.currentTimeMillis();
System.out.println("耗时: " + (end - start) + "ms");
// 问题：JIT 编译、GC、死码消除等都会影响结果
```

JMH 解决的问题：
- **JIT 编译预热**：自动执行预热轮次
- **死码消除**：防止 JVM 优化掉被测代码
- **常量折叠**：避免编译期常量优化干扰
- **GC 干扰**：统计 GC 次数和时间
- **统计置信度**：提供误差范围和置信区间

### 2.3 Maven 依赖

```xml
<dependencies>
    <dependency>
        <groupId>org.openjdk.jmh</groupId>
        <artifactId>jmh-core</artifactId>
        <version>1.37</version>
    </dependency>
    <dependency>
        <groupId>org.openjdk.jmh</groupId>
        <artifactId>jmh-generator-annprocess</artifactId>

</dependencies>
```

### 2.4 第一个 JMH 基准测试

```java
import org.openjdk.jmh.annotations.*;
import org.openjdk.jmh.infra.Blackhole;
import java.util.concurrent.TimeUnit;

@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.NANOSECONDS)
@Warmup(iterations = 3, time = 1, timeUnit = TimeUnit.SECONDS)
@Measurement(iterations = 5, time = 1, timeUnit = TimeUnit.SECONDS)
@Fork(1)
@State(Scope.Thread)
public class StringConcatBenchmark {

    @Benchmark
    public void stringConcat(Blackhole bh) {
        String result = "Hello" + ", " + "World" + "!";
        bh.consume(result); // 防止死码消除
    }

    @Benchmark
    public void stringBuilder(Blackhole bh) {
        StringBuilder sb = new StringBuilder();
        sb.append("Hello").append(", ").append("World").append("!");
        bh.consume(sb.toString());
    }

    @Benchmark
    public void stringFormatted(Blackhole bh) {
        String result = String.format("%s, %s!", "Hello", "World");
        bh.consume(result);
    }
}
```

### 2.5 运行 JMH

```java
// 方式1：通过 main 方法运行
public class BenchmarkRunner {
    public static void main(String[] args) throws Exception {
        org.openjdk.jmh.Main.main(args);
    }
}

// 方式2：编程式运行
public class BenchmarkRunner {
    public static void main(String[] args) throws RunnerException {
        Options opt = new OptionsBuilder()
            .include(StringConcatBenchmark.class.getSimpleName())
            .forks(1)
            .build();
        new Runner(opt).run();
    }
}
```

## 3. JMH 注解详解

### 3.1 核心注解一览

| 注解 | 用途 | 示例 |
|------|------|------|
| `@Benchmark` | 标记基准测试方法 | `@Benchmark public void test()` |
| `@BenchmarkMode` | 测试模式 | `@BenchmarkMode(Mode.Throughput)` |
| `@OutputTimeUnit` | 输出时间单位 | `@OutputTimeUnit(TimeUnit.MILLISECONDS)` |
| `@Warmup` | 预热配置 | `@Warmup(iterations=3, time=1)` |
| `@Measurement` | 测量配置 | `@Measurement(iterations=5, time=1)` |
| `@Fork` | Fork 进程数 | `@Fork(2)` |
| `@State` | 状态对象作用域 | `@State(Scope.Thread)` |
| `@Setup` | 初始化方法 | `@Setup(Level.Trial)` |
| `@TearDown` | 清理方法 | `@TearDown(Level.Trial)` |
| `@Param` | 参数化测试 | `@Param({"1", "10", "100"})` |

### 3.2 @State 作用域

```java
// Thread：每个线程独立实例
@State(Scope.Thread)
public class ThreadState {
    private int counter = 0;
}

// Group：同一组线程共享
@State(Scope.Group)
public class GroupState {
    private Queue<Integer> queue = new ConcurrentLinkedQueue<>();
}

// Benchmark：所有线程共享同一实例
@State(Scope.Benchmark)
public class BenchmarkState {
    private List<String> data;
}

// 使用示例
@State(Scope.Thread)
public class MapBenchmark {

    private Map<String, Integer> map;

    @Setup(Level.Trial) // 每次试验前执行
    public void setUp() {
        map = new HashMap<>();
        for (int i = 0; i < 1000; i++) {
            map.put("key" + i, i);
        }
    }

    @Benchmark
    public void get(Blackhole bh) {
        bh.consume(map.get("key500"));
    }

    @TearDown(Level.Trial) // 每次试验后执行
    public void tearDown() {
        map.clear();
    }
}
```

### 3.3 @Param 参数化

```java
@State(Scope.Benchmark)
public class ListBenchmark {

    @Param({"10", "100", "1000", "10000"})
    private int size;

    private List<Integer> arrayList;
    private List<Integer> linkedList;

    @Setup
    public void setUp() {
        arrayList = new ArrayList<>();
        linkedList = new LinkedList<>();
        for (int i = 0; i < size; i++) {
            arrayList.add(i);
            linkedList.add(i);
        }
    }

    @Benchmark
    public int arrayListGet() {
        return arrayList.get(size / 2);
    }

    @Benchmark
    public int linkedListGet() {
        return linkedList.get(size / 2);
    }
}
```


## 4. JMH 测试模式与预热

### 4.1 BenchmarkMode 四种模式

```java
// 1. Throughput：吞吐量（ops/time，越大越好）
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.SECONDS)
public class ThroughputBenchmark {
    @Benchmark
    public void test() { }
}

// 2. AverageTime：平均时间（time/op，越小越好）
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.MICROSECONDS)
public class AverageTimeBenchmark {
    @Benchmark
    public void test() { }
}

// 3. SampleTime：采样时间（提供百分位统计）
@BenchmarkMode(Mode.SampleTime)
@OutputTimeUnit(TimeUnit.MICROSECONDS)
public class SampleTimeBenchmark {
    @Benchmark
    public void test() { }
}

// 4. SingleShotTime：单次执行时间（不预热，只跑一次）
@BenchmarkMode(Mode.SingleShotTime)
@OutputTimeUnit(TimeUnit.MILLISECONDS)
public class SingleShotBenchmark {
    @Benchmark
    public void test() { }
}

// 多模式组合
@BenchmarkMode({Mode.Throughput, Mode.AverageTime})
public class MultiModeBenchmark {
    @Benchmark
    public void test() { }
}
```

### 4.2 预热配置

```java
// 预热：让 JIT 编译器充分优化后再测量
@Warmup(
    iterations = 5,           // 预热轮数
    time = 2,                  // 每轮时间
    timeUnit = TimeUnit.SECONDS,
    batchSize = 1              // 每批次调用次数
)
@Measurement(
    iterations = 10,           // 测量轮数
    time = 2,
    timeUnit = TimeUnit.SECONDS
)
@Fork(
    value = 2,                 // fork 进程数（2个进程取平均，更可靠）
    jvmArgs = {"-Xms2G", "-Xmx2G"} // JVM 参数
)
public class WarmupBenchmark {
    @Benchmark
    public void test() { }
}
```

### 4.3 Fork 的意义

```
Fork 1（独立 JVM 进程）
  ├── Warmup（3轮 × 1秒）
  └── Measurement（5轮 × 1秒）

Fork 2（另一个独立 JVM 进程）
  ├── Warmup（3轮 × 1秒）
  └── Measurement（5轮 × 1秒）

最终结果 = Fork 1 和 Fork 2 的统计聚合
```

> **重要：** Fork 至少为 1。Fork 0 会在当前 JVM 中运行，结果受已有 JIT 编译和类加载状态影响，不推荐。

## 5. JMH 实战案例

### 5.1 字符串拼接性能对比

```java
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.NANOSECONDS)
@Warmup(iterations = 3, time = 1)
@Measurement(iterations = 5, time = 1)
@Fork(1)
@State(Scope.Benchmark)
public class StringConcatBenchmark {

    @Param({"10", "50", "100"})
    private int count;

    private String[] parts;

    @Setup
    public void setUp() {
        parts = new String[count];
        for (int i = 0; i < count; i++) {
            parts[i] = "part" + i;
        }
    }

    @Benchmark
    public String plusConcat() {
        String result = "";
        for (String part : parts) {
            result += part;
        }
        return result;
    }

    @Benchmark
    public String stringBuilderConcat() {
        StringBuilder sb = new StringBuilder();
        for (String part : parts) {
            sb.append(part);
        }
        return sb.toString();
    }

    @Benchmark
    public String stringJoin() {
        return String.join("", parts);
    }

    @Benchmark
    public String stringBufferConcat() {
        StringBuffer sb = new StringBuffer();
        for (String part : parts) {
            sb.append(part);
        }
        return sb.toString();
    }
}
```

### 5.2 JSON 序列化性能对比

```java
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.MILLISECONDS)
@Warmup(iterations = 3, time = 2)
@Measurement(iterations = 5, time = 2)
@Fork(1)
@State(Scope.Benchmark)
public class JsonSerializationBenchmark {

    private ObjectMapper objectMapper;
    private Gson gson;
    private User user;
    private String userJson;

    @Setup
    public void setUp() throws Exception {
        objectMapper = new ObjectMapper();
        gson = new Gson();

        user = new User(1L, "张三", "zhangsan@test.com", 25,
            List.of("reading", "coding"), Map.of("city", "Beijing"));
        userJson = objectMapper.writeValueAsString(user);
    }

    @Benchmark
    public String jacksonSerialize() throws Exception {
        return objectMapper.writeValueAsString(user);
    }

    @Benchmark
    public User jacksonDeserialize() throws Exception {
        return objectMapper.readValue(userJson, User.class);
    }

    @Benchmark
    public String gsonSerialize() {
        return gson.toJson(user);
    }

    @Benchmark
    public User gsonDeserialize() {
        return gson.fromJson(userJson, User.class);
    }
}
```

## 7. Gatling 脚本编写

### 7.1 Gatling 优势

| 特性 | JMeter | Gatling |

| 资源消耗 | 高 | 低 |
| 报告 | HTML | 丰富的 HTML |
| CI 集成 | 需要插件 | 原生支持 |
| 单机并发 | ~1000 | ~10000+ |

### 7.3 Gatling Java API 脚本

```java
import io.gatling.javaapi.core.*;
import io.gatling.javaapi.http.*;

import static io.gatling.javaapi.core.CoreDsl.*;
import static io.gatling.javaapi.http.HttpDsl.*;

public class ApiLoadTest extends Simulation {

    // HTTP 协议配置
    HttpProtocolBuilder httpProtocol = http
        .baseUrl("http://localhost:8080")
        .acceptHeader("application/json")
        .contentTypeHeader("application/json")
        .userAgentHeader("Gatling/Performance Test");

    // 场景1：用户查询
    ScenarioBuilder userQueryScenario = scenario("用户查询场景")
        .exec(
            http("获取用户列表")
                .get("/api/users")
                .check(status().is(200))
                .check(jsonPath("$[*].id").count().gt(0))
        )
        .pause(1, 3); // 1-3秒思考时间

    // 场景2：用户创建流程
    ScenarioBuilder userCreateScenario = scenario("用户创建场景")
        .feed(
            // 从 CSV 文件注入测试数据
            csv("users.csv").circular()
        )
        .exec(
            http("创建用户")
                .post("/api/users")
                .body(StringBody("""
                    {
                        "name": "#{name}",
                        "email": "#{email}",
                        "age": #{age}
                    }
                    """))
                .asJson()
                .check(status().is(201))
                .check(jsonPath("$.id").saveAs("userId"))
        )
        .exec(
            http("查询创建的用户")
                .get("/api/users/#{userId}")
                .check(status().is(200))
                .check(jsonPath("$.name").isEL("#{name}"))
        )
        .exec(
            http("删除用户")
                .delete("/api/users/#{userId}")
                .check(status().is(204))
        );

    // 负载注入策略
    {
        setUp(
            userQueryScenario.injectOpen(
                // 阶梯式加压：0 → 50 用户，每10秒加10
                rampUsersPerSec(1).to(50).during(60),
                // 保持50用户持续30秒
                constantUsersPerSec(50).during(30),
                // 降压：50 → 0 用户
                rampUsersPerSec(50).to(0).during(20)
            ),
            userCreateScenario.injectOpen(
                // 一次性注入20用户
                atOnceUsers(20),
                // 线性增加：10秒内达到100用户
                rampUsers(100).during(10)
            )
        ).protocols(httpProtocol)
         .assertions(
             global().responseTime().max().lt(500),    // 最大响应时间 < 500ms
             global().responseTime().percentile3().lt(200), // P95 < 200ms
             global().successfulRequests().percent().gt(99.0) // 成功率 > 99%
         );
    }
}
```

## 8. 性能基线管理

### 8.1 什么是性能基线

性能基线是系统在特定条件下的性能基准值，用于：
- 检测性能退化
- 验证性能优化效果
- 设定容量规划目标

### 8.2 建立性能基线

```java
// 基线配置文件：performance-baseline.yml
baselines:
```

## 9. 常见陷阱与最佳实践

### 9.1 JMH 常见陷阱

```java
// ❌ 陷阱1：死码消除 — JVM 优化掉了被测代码
@Benchmark
public void wrongBenchmark() {
    int x = 1 + 2; // JVM 会发现 x 没被使用，直接消除
}

// ✅ 使用 Blackhole 消费结果
@Benchmark
public void correctBenchmark(Blackhole bh) {
    int x = 1 + 2;
    bh.consume(x); // 防止优化
}

// ❌ 陷阱2：常量折叠 — 编译期计算
@Benchmark
public long wrongConstantFolding() {
    long x = 1000L * 60 * 60; // 编译器会直接算成 3600000L
    return x;
}

// ✅ 通过 @State 和 @Param 传入运行期值
@State(Scope.Thread)
public class CorrectBenchmark {
    @Param({"1000"})
    long base;

    @Benchmark
    public long correct(Blackhole bh) {
        long x = base * 60 * 60;
        bh.consume(x);
        return x;
    }
}

// ❌ 陷阱3：循环在 Benchmark 内部
@Benchmark
public void wrongLoop(Blackhole bh) {
    for (int i = 0; i < 1000; i++) { // JMH 会自动循环，你不需要手动循环
        bh.consume(i);
    }
}

// ✅ 让 JMH 控制循环
@Benchmark
public void correct(Blackhole bh) {
    bh.consume(doSomething());
}

// ❌ 陷阱4：共享状态干扰
@State(Scope.Benchmark) // 所有线程共享，可能引入竞争
private List<Integer> sharedList;

// ✅ 根据场景选择正确的作用域
@State(Scope.Thread) // 每个线程独立
private List<Integer> threadLocalList;
```

### 9.2 JMeter/Gatling 常见陷阱

```
❌ 陷阱1：测试环境与生产环境差异大
  - 测试环境1核2G，生产环境8核16G
  - 结果不可参考

✅ 尽量使用与生产相同配置的测试环境

❌ 陷阱2：忽略预热阶段
  - 直接开始测量，应用还没启动完成
  - JIT 还没充分编译

✅ 先运行预热负载，再开始正式测量

❌ 陷阱3：只看平均值
  - 平均 50ms，但 P99 是 2000ms
  - 少量慢请求被平均值掩盖

✅ 关注 P90/P95/P99 和最大值

❌ 陷阱4：忽视资源监控
  - 只看响应时间，不看 CPU/内存/IO
  - 无法定位瓶颈

✅ 同时监控服务器资源（CPU、内存、GC、IO）

❌ 陷阱5：并发用户数过高导致测试客户端瓶颈
  - JMeter 单机 5000 线程时本身成为瓶颈

✅ 使用分布式测试或改用 Gatling（事件驱动）
```

### 9.3 最佳实践清单

| 类别 | 实践 |
|------|------|
| JMH | Fork 至少1，预热充分，使用 Blackhole |
| JMH | 避免循环内嵌循环，让 JMH 控制迭代 |
| JMeter | 非GUI模式运行，关注P95/P99 |
| Gatling | 利用事件驱动模型，支持高并发 |
| 通用 | 测试环境接近生产环境 |
| 通用 | 先预热再测量 |
| 通用 | 监控服务器资源 |
| 通用 | 建立性能基线，持续检测退化 |
| 通用 | CI 中设置性能门禁 |

## 10. 面试题速查

**Q1: JMH 解决了什么问题？**
- 解决 JIT 编译预热、死码消除、常量折叠、GC 干扰等 JVM 优化导致的基准测试偏差

**Q2: JMH 的四种 BenchmarkMode 是什么？**
- Throughput：吞吐量（ops/time）
- AverageTime：平均时间（time/op）
- SampleTime：采样时间（提供百分位）
- SingleShotTime：单次执行时间

**Q3: JMH 中 @State 的三种作用域？**
- Scope.Thread：每个线程独立
- Scope.Group：同组线程共享
- Scope.Benchmark：所有线程共享

**Q4: 为什么 JMH 需要预热（Warmup）？**
- 让 JIT 编译器充分优化代码
- 让 GC 进入稳定状态
- 避免冷启动数据干扰测量结果

**Q5: JMeter 和 Gatling 的主要区别？**
- JMeter 基于线程模型，GUI 丰富，单机并发能力有限
- Gatling 基于事件驱动（Akka），代码即脚本，单机并发能力强

**Q6: 性能测试应该关注哪些指标？**
- 响应时间（平均、P90、P95、P99、最大值）
- 吞吐量（TPS/QPS）
- 错误率
- 资源利用率（CPU、内存、GC、IO）

**Q7: 什么是性能基线？为什么需要？**
- 性能基线是系统在特定条件下的性能基准
- 用于检测性能退化、验证优化效果、容量规划

**Q8: 什么是死码消除？JMH 如何防止？**
- JVM 会优化掉计算结果未被使用的代码
- JMH 使用 Blackhole 消费结果，防止 JVM 优化
