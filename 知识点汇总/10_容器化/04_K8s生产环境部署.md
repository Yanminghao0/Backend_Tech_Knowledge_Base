## 1. 生产环境集群规划

### 1.1 集群架构

```
┌─────────────────────────────────────────────────────────────┐
│                    K8s 生产集群架构                          │
│                                                             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                    │
│  │ Master-1│  │ Master-2│  │ Master-3│  ← 高可用控制面     │
│  │API+ETCD │  │API+ETCD │  │API+ETCD │    (负载均衡前置)    │
│  └─────────┘  └─────────┘  └─────────┘                    │
│       ↕            ↕            ↕                           │
│  ┌─────────────────────────────────────────┐               │
│  │          SLB / HAProxy (VIP)            │               │
│  └─────────────────────────────────────────┘               │
│       ↕            ↕            ↕                           │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐      │
│  │Worker-1 │  │Worker-2 │  │Worker-3 │  │Worker-4 │       │
│  │Ingress  │  │Ingress  │  │  Pod    │  │  Pod    │       │
│  │  Pods   │  │  Pods   │  │  Pods   │  │  Pods   │       │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘      │
│                                                             │
│  ┌─────────────────────────────────────────┐               │
│  │     存储: CSI (云盘/NFS/Ceph)            │               │
│  └─────────────────────────────────────────┘               │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 节点规划

| 节点角色 | 数量 | CPU | 内存 | 磁盘 | 说明 |
|---------|------|-----|------|------|------|
| Master | 3 | 8C | 16G | 100G SSD | API Server + etcd + Scheduler + Controller Manager |
| Worker | 5+ | 16C | 64G | 200G SSD | 运行业务Pod |
| Worker(Ingress) | 2 | 4C | 8G | 50G SSD | 专用Ingress节点 |
| Worker(监控) | 1 | 8C | 32G | 500G SSD | Prometheus + Grafana + ELK |

### 1.3 etcd备份策略

```bash
#!/bin/bash
# etcd 定期备份脚本
ETCD_ENDPOINTS="https://10.0.1.11:2379"
ETCD_CERT="/etc/etcd/pki/etcd.crt"
ETCD_KEY="/etc/etcd/pki/etcd.key"
ETCD_CACERT="/etc/etcd/pki/ca.crt"
BACKUP_DIR="/data/etcd-backup"
DATE=$(date +%Y%m%d_%H%M%S)

# 创建快照
ETCDCTL_API=3 etcdctl snapshot save ${BACKUP_DIR}/etcd-snapshot-${DATE}.db \
  --endpoints=${ETCD_ENDPOINTS} \
  --cert=${ETCD_CERT} \
  --key=${ETCD_KEY} \
  --cacert=${ETCD_CACERT}

# 验证快照
ETCDCTL_API=3 etcdctl snapshot status ${BACKUP_DIR}/etcd-snapshot-${DATE}.db

# 保留最近7天的备份
find ${BACKUP_DIR} -name "etcd-snapshot-*.db" -mtime +7 -delete

# Cron: 每6小时备份一次
# 0 */6 * * * /opt/scripts/etcd-backup.sh
```

## 2. 资源配额与QoS

### 2.1 资源配额

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: api-server
  labels:
    app: api-server
spec:
  containers:
    - name: api
      image: registry.example.com/api:v1.0.0
      resources:
        requests:          # 调度依据（软保证）
          cpu: "500m"      # 0.5个CPU核心
          memory: "512Mi"
        limits:            # 硬限制（超限会被限制/杀掉）
          cpu: "1000m"     # 1个CPU核心
          memory: "1Gi"
      # CPU超限：被节流（throttle），不会被杀
      # 内存超限：被OOM Kill（exit code 137）
```

### 2.2 QoS等级

Kubernetes根据requests和limits的配置，将Pod分为三个QoS等级：

```
┌─────────────────────────────────────────────────────────────┐
│                    QoS 等级划分                              │
│                                                             │
│  Guaranteed（最高优先级）                                    │
│    条件: requests == limits（CPU和内存都设置且相等）          │
│    行为: 最后被驱逐                                          │
│    适用: 核心服务（数据库、中间件）                          │
│                                                             │
│  Burstable（中等优先级）                                     │
│    条件: 不满足Guaranteed，但至少设置了一个request           │
│    行为: 中等优先级驱逐                                      │
│    适用: 普通业务服务                                        │
│                                                             │
│  BestEffort（最低优先级）                                    │
│    条件: 没有设置任何requests和limits                       │
│    行为: 最先被驱逐                                          │
│    适用: 临时任务、开发测试                                  │
└─────────────────────────────────────────────────────────────┘
```

```yaml
# Guaranteed — 生产核心服务
apiVersion: v1
kind: Pod
metadata:
  name: mysql-pod
spec:
  containers:
    - name: mysql
      image: mysql:8.0
      resources:
        requests:
          cpu: "2000m"
          memory: "4Gi"
        limits:
          cpu: "2000m"       # requests == limits
          memory: "4Gi"

---
# Burstable — 普通业务服务
apiVersion: v1
kind: Pod
metadata:
  name: api-pod
spec:
  containers:
    - name: api
      image: api:v1.0
      resources:
        requests:
          cpu: "500m"
          memory: "512Mi"
        limits:
          cpu: "1000m"       # limits > requests
          memory: "1Gi"

---
# BestEffort — 不推荐生产使用
apiVersion: v1
kind: Pod
metadata:
  name: batch-pod
spec:
  containers:
    - name: batch
      image: batch:v1.0
      # 没有设置resources
```

## 3. 健康检查

### 3.1 三种探针

```
Liveness Probe（存活探针）
  ├── 检查容器是否正常运行
  ├── 失败 → 重启容器（restartPolicy生效）
  └── 用途: 检测死锁、进程假死

Readiness Probe（就绪探针）
  ├── 检查容器是否准备好接收流量
  ├── 失败 → 从Service Endpoints移除（不接收新请求）
  └── 用途: 启动慢的容器、依赖检查

Startup Probe（启动探针）
  ├── 检查容器是否已成功启动
  ├── 启动期间禁用Liveness和Readiness
  ├── 失败 → 重启容器
  └── 用途: 启动慢的应用（如Java应用）
```

### 3.2 完整探针配置

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api-server
  template:
    metadata:
      labels:
        app: api-server
    spec:
      containers:
        - name: api
          image: registry.example.com/api:v1.0.0
          ports:
            - containerPort: 8080
          
          # 启动探针 — Java应用启动慢，给5分钟启动时间
          startupProbe:
            httpGet:
              path: /actuator/health
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 10
            failureThreshold: 30        # 10s × 30 = 300s = 5分钟
            timeoutSeconds: 5
          
          # 存活探针 — 检测死锁/假死
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            periodSeconds: 15
            failureThreshold: 3         # 连续3次失败重启
            timeoutSeconds: 5
          
          # 就绪探针 — 检查是否可以接收流量
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            periodSeconds: 10
            failureThreshold: 3         # 连续3次失败摘除流量
            timeoutSeconds: 3
            successThreshold: 1         # 1次成功恢复流量

---
# TCP探针示例 — 数据库类应用
apiVersion: v1
kind: Pod
metadata:
  name: redis-pod
spec:
  containers:
    - name: redis
      image: redis:7
      livenessProbe:
        tcpSocket:
          port: 6379
        periodSeconds: 10
        failureThreshold: 3

---
# 命令探针示例 — 自定义检查
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
    - name: app
      image: app:v1.0
      livenessProbe:
        exec:
          command:
            - /bin/sh
            - -c
            - "ps aux | grep myapp | grep -v grep"
        periodSeconds: 30
```

## 4. HPA水平自动扩缩容

### 4.1 基于CPU/内存的HPA

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 3                  # 最小副本数
  maxReplicas: 20                 # 最大副本数
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70  # CPU使用率目标70%
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80  # 内存使用率目标80%
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0   # 扩容立即执行
      policies:
        - type: Percent
          value: 100                   # 每次最多扩容100%
          periodSeconds: 60
        - type: Pods
          value: 4                     # 每次最多扩4个Pod
          periodSeconds: 60
      selectPolicy: Max
    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容冷却5分钟
      policies:
        - type: Percent
          value: 10                    # 每次最多缩容10%
          periodSeconds: 60
```

### 4.2 自定义指标HPA

```yaml
# 需要安装 Prometheus Adapter
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-custom-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 3
  maxReplicas: 30
  metrics:
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "1000"     # 每Pod 1000 QPS触发扩容
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

### 4.3 HPA扩缩容流程

```
                    HPA 控制循环（默认15秒）
                           │
                           ▼
               ┌───────────────────────┐
               │  获取当前指标值         │
               │  (CPU/Memory/Custom)  │
               └───────────┬───────────┘
                           │
                           ▼
               ┌───────────────────────┐
               │  计算期望副本数         │
               │  desired = current ×   │
               │  (actual / target)     │
               └───────────┬───────────┘
                           │
                    ┌──────┴──────┐
                    │             │
                 扩容            缩容
                    │             │
                    ▼             ▼
              立即执行      等待冷却期
              创建Pod       (5分钟)
```

## 5. PDB中断预算

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-pdb
  namespace: production
spec:
  minAvailable: 2              # 至少保持2个Pod可用
  # 或使用 maxUnavailable: 1
  selector:
    matchLabels:
      app: api-server

# PDB保证在自愿中断（如节点维护、滚动更新）时，
# 不会同时驱逐过多Pod，确保服务可用性
# 注意：PDB不影响非自愿中断（如节点故障）
```

## 6. ConfigMap与Secret

### 6.1 ConfigMap

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: production
data:
  application.yml: |
    server:
      port: 8080
    spring:
      datasource:
        url: jdbc:mysql://mysql-svc:3306/appdb
        driver-class-name: com.mysql.cj.jdbc.Driver
      redis:
        host: redis-svc
        port: 6379
    
    logging:
      level:
        com.example: DEBUG
    
    management:
      endpoints:
        web:
          exposure:
            include: health,info,metrics,prometheus
  LOG_LEVEL: "INFO"
  MAX_THREADS: "200"

---
# 以环境变量方式注入
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  template:
    spec:
      containers:
        - name: api
          image: api:v1.0
          env:
            - name: LOG_LEVEL
              valueFrom:
                configMapKeyRef:
                  name: app-config
                  key: LOG_LEVEL
            - name: MAX_THREADS
              valueFrom:
                configMapKeyRef:
                  name: app-config
                  key: MAX_THREADS
          envFrom:
            - configMapRef:
                name: app-config    # 注入所有key为环境变量

---
# 以Volume方式注入（推荐，支持热更新）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  template:
    spec:
      containers:
        - name: api
          image: api:v1.0
          volumeMounts:
            - name: config-volume
              mountPath: /app/config
              readOnly: true
      volumes:
        - name: config-volume
          configMap:
            name: app-config
            items:
              - key: application.yml
                path: application.yml
```

## 7. StatefulSet有状态应用

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
  namespace: production
spec:
  serviceName: mysql-headless       # 必须关联Headless Service
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      affinity:
        podAntiAffinity:            # 反亲和性，分散到不同节点
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchLabels:
                  app: mysql
              topologyKey: kubernetes.io/hostname
      containers:
        - name: mysql
          image: mysql:8.0
          ports:
            - containerPort: 3306
              name: mysql
          env:
            - name: MYSQL_ROOT_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: mysql-secret
                  key: root-password
          volumeMounts:
            - name: data
              mountPath: /var/lib/mysql
          resources:
            requests:
              cpu: "1000m"
              memory: "2Gi"
            limits:
              cpu: "2000m"
              memory: "4Gi"
  volumeClaimTemplates:             # 每个Pod自动创建独立PVC
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: fast-ssd
        resources:
          requests:
            storage: 100Gi

---
apiVersion: v1
kind: Service
metadata:
  name: mysql-headless
spec:
  clusterIP: None                   # Headless Service
  selector:
    app: mysql
  ports:
    - port: 3306
      targetPort: 3306

# StatefulSet Pod命名规则: mysql-0, mysql-1, mysql-2
# DNS解析: mysql-0.mysql-headless.production.svc.cluster.local
```

## 8. 持久化存储

### 8.1 PV/PVC/StorageClass

```yaml
# StorageClass — 动态供给
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/aws-ebs    # 或 csi.driver.example.com
parameters:
  type: gp3
  fsType: ext4
reclaimPolicy: Retain                 # Retain | Delete
allowVolumeExpansion: true
volumeBindingMode: WaitForFirstConsumer
mountOptions:
  - noatime
  - nobarrier

---
# PVC — 自动创建PV
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: app-data-pvc
  namespace: production
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: fast-ssd
  resources:
    requests:
      storage: 50Gi

---
# 在Pod中使用PVC
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
    - name: app
      image: app:v1.0
      volumeMounts:
        - name: data
          mountPath: /app/data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: app-data-pvc
```

### 8.2 AccessMode对比

| AccessMode | 缩写 | 说明 | 适用场景 |
|-----------|------|------|---------|
| ReadWriteOnce | RWO | 单节点读写 | 数据库、有状态应用 |
| ReadOnlyMany | ROX | 多节点只读 | 配置文件、静态资源 |
| ReadWriteMany | RWX | 多节点读写 | 共享存储(NFS/CephFS) |
| ReadWriteOncePod | RWOP | 单Pod读写 | K8s 1.22+，更严格隔离 |

## 9. 灰度发布

### 9.1 滚动更新

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1              # 滚动更新时最多多出1个Pod
      maxUnavailable: 0        # 滚动更新时不允许减少可用Pod
  selector:
    matchLabels:
      app: api-server
  template:
    metadata:
      labels:
        app: api-server
    spec:
      containers:
        - name: api
          image: registry.example.com/api:v2.0.0    # 更新镜像版本
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
```

  ports:

      targetPort: 8080

# 金丝雀发布流程:
# 1. 创建canary Deployment（1个副本）→ 10%流量
# 2. 观察监控指标（错误率、RT）
# 3. 逐步增加canary副本，减少stable副本

## 13. 生产排障指南

### 13.1 Pod状态排查

```bash
# Pod状态速查
kubectl get pods -n production

# 常见异常状态:
# Pending     → 调度失败（资源不足、亲和性不满足）
# ContainerCreating → 正在创建（拉镜像慢、挂载卷失败）
# CrashLoopBackOff → 容器反复崩溃重启
# ImagePullBackOff → 镜像拉取失败
# OOMKilled   → 内存不足被杀
# Evicted     → 被驱逐（节点资源不足）

# 查看Pod详情
kubectl describe pod <pod-name> -n production

# 查看Pod日志
kubectl logs <pod-name> -n production
kubectl logs <pod-name> -n production --previous    # 查看上次崩溃的日志
kubectl logs <pod-name> -n production -c <container> --tail=100

# 进入Pod排查
kubectl exec -it <pod-name> -n production -- /bin/sh
```

### 13.2 常见问题排查

```bash
# === Pod一直Pending ===
kubectl describe pod <pod-name> -n production
# 检查Events部分，常见原因:
#   - 0/5 nodes are available: insufficient cpu
#   - 0/5 nodes are available: node(s) had taints that the pod didn't tolerate
#   - persistentvolumeclaim "xxx" not found

# === Pod频繁重启(CrashLoopBackOff) ===
kubectl logs <pod-name> -n production --previous
# 检查:
#   - 应用启动日志是否有异常
#   - 是否OOM (kubectl describe pod, 查看Last State)
#   - 健康检查是否配置正确

# === 服务无法访问 ===
# 1. 检查Pod是否Running且Ready
kubectl get pods -n production -l app=api-server
# 2. 检查Service Endpoints
kubectl get endpoints api-service -n production
# 3. 检查Service配置
kubectl describe service api-service -n production
# 4. 从另一个Pod测试连通性
kubectl exec -it <debug-pod> -- curl http://api-service:80/actuator/health

# === 节点NotReady ===
kubectl describe node <node-name>
# 检查:
#   - kubelet是否正常运行
#   - 节点资源是否耗尽
#   - 网络是否正常
```
