## 1. ES集群架构概述

### 1.1 整体架构

Elasticsearch是一个基于Lucene的分布式搜索引擎，其核心设计理念是通过分片（Shard）和副本（Replica）实现数据的水平扩展和高可用。

```
┌─────────────────────────────────────────────────────────────┐
│                      ES Cluster                              │
│                                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                  │
│  │ Master-1 │  │ Master-2 │  │ Master-3 │  ← 主节点选举     │
│  └──────────┘  └──────────┘  └──────────┘                  │
│                                                              │
│  ┌──────────────────┐  ┌──────────────────┐                │
│  │  Data Node-1     │  │  Data Node-2     │                │
│  │ ┌─────┐ ┌─────┐ │  │ ┌─────┐ ┌─────┐ │                │
│  │ │P0   │ │R1   │ │  │ │P1   │ │R0   │ │  ← 数据分片     │
│  │ └─────┘ └─────┘ │  │ └─────┘ └─────┘ │                │
│  └──────────────────┘  └──────────────────┘                │
│                                                              │
│  ┌──────────────────┐  ┌──────────────────┐                │
│  │ Coord Node-1     │  │ Coord Node-2     │  ← 协调节点     │
│  └──────────────────┘  └──────────────────┘                │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心概念

**主分片（Primary Shard）**：

- 索引的数据被水平切分为多个分片，每个分片是一个独立的Lucene索引
- 主分片数量在索引创建时确定，后续不可修改（除非Reindex）
- 每个分片承载索引的一部分数据

**副本分片（Replica Shard）**：

- 主分片的拷贝，提供数据冗余和高可用
- 副本数量可以动态调整
- 副本分片不会被分配到与主分片相同的节点上
- 读请求可以由副本分片处理，提升查询吞吐

**发现机制（Discovery）**：

ES集群通过Zen Discovery模块实现节点发现和Master选举：

```
节点启动
   ↓
尝试连接 seed hosts（discovery.seed_hosts）
   ↓
加入集群 / 发起Master选举
   ↓
集群状态同步
```

```yaml
# elasticsearch.yml — 集群发现配置
cluster.name: production-es-cluster

# 种子节点列表（用于初始发现）
discovery.seed_hosts: ["10.0.1.11", "10.0.1.12", "10.0.1.13"]

# 具有Master资格的节点
cluster.initial_master_nodes: ["node-1", "node-2", "node-3"]

# 选举触发参数
discovery.zen.minimum_master_nodes: 2  # ES 7.x后自动管理
```

### 1.3 集群状态管理

ES集群状态（Cluster State）包含：

| 组件 | 说明 |
|------|------|
| 元数据（Metadata） | 索引定义、Mapping、Settings、别名 |
| 路由表（Routing Table） | 分片到节点的映射关系 |
| 分片分配（Allocation） | 分片的分配策略和决策 |
| 节点信息（Nodes） | 集群中所有节点的基本信息 |

Master节点负责维护集群状态并广播给所有节点，集群状态变更采用两阶段提交（2PC）保障一致性。

## 2. 节点角色与职责

### 2.1 节点角色详解

ES 7.x+通过 `node.roles` 配置节点角色：

```yaml
# elasticsearch.yml

# Master 节点 — 负责集群管理（索引创建删除、分片分配、Master选举）
node.roles: [master]

# Data 节点 — 存储数据，执行CRUD、搜索、聚合
node.roles: [data]

# Coordinating 节点 — 路由请求、聚合结果（不存储数据）
node.roles: []  # 空数组表示纯协调节点

# Ingest 节点 — 数据预处理管道（类似Logstash的filter）
node.roles: [ingest]

# Data Content 节点（ES 7.9+数据分层）
node.roles: [data_content]

# Data Hot 节点 — 热数据层
node.roles: [data_hot]

# Data Warm 节点 — 温数据层
node.roles: [data_warm]

# Data Cold 节点 — 冷数据层
node.roles: [data_cold]

# Data Frozen 节点 — 冻结数据层
node.roles: [data_frozen]
```

### 2.2 生产环境角色分离

生产环境强烈建议**角色分离**，避免资源争抢：

```yaml
# === Master 节点配置 ===
node.name: master-1
node.roles: [master]
# Master节点不需要大存储，但需要稳定
# 推荐配置：4C 8G SSD 50G

# === Data 节点配置 ===
node.name: data-1
node.roles: [data, data_hot]
# Data节点需要高配置CPU和大存储
# 推荐配置：16C 64G NVMe SSD 2T

# === Coordinating 节点配置 ===
node.name: coord-1
node.roles: []
# 协调节点需要高CPU和内存（聚合计算）
# 推荐配置：8C 16G
```

### 2.3 节点选型建议

| 节点类型 | CPU | 内存 | 磁盘 | 数量 | 职责 |
|---------|-----|------|------|------|------|
| Master | 4C | 8G | 50G SSD | 3（奇数） | 集群管理、元数据维护 |
| Data Hot | 16C | 64G | 2T NVMe | 按数据量 | 热数据存储与查询 |
| Data Warm | 8C | 32G | 4T HDD | 按数据量 | 温数据存储 |
| Coordinating | 8C | 16G | 50G SSD | 2+ | 请求路由、结果聚合 |
| Ingest | 4C | 8G | 50G | 1+ | 数据预处理 |

> ⚠️ **注意**：Master节点必须 ≥ 3个且为奇数，防止脑裂。ES 7.x+自动管理 `minimum_master_nodes`，但仍需确保奇数节点。

## 3. 分片策略与路由机制

### 3.1 分片数选择原则

分片数的选择直接影响集群的并发能力和资源利用：

```
分片数选择公式：
  分片数 = max(1, ⌈单分片大小目标 / 总数据量⌉)

经验值：
  - 单分片大小建议：10GB ~ 50GB
  - 分片数 = ⌈总数据量(GB) / 30⌉
  - 最大分片数 ≈ 数据节点数 × 20（每节点不超过20个分片）
```

**示例计算**：

```
场景：日志数据，每月 500GB，保留3个月 = 1.5TB
  分片数 = 1500 / 30 = 50 个分片
  副本数 = 1
  总分片 = 50 × 2 = 100 个

  如果有 5 个 Data 节点：
    每节点分片数 = 100 / 5 = 20 个（刚好满足）
```

### 3.2 路由规则

ES通过路由公式决定文档存储到哪个分片：

```java
// ES路由公式（源码级）
shard = hash(routing) % number_of_primary_shards

// 默认 routing = _id（文档ID）
// 可自定义 routing 值
```

**自定义路由示例**：

```json
// 使用自定义路由写入文档
PUT /orders/_doc/1?routing=user_1001
{
  "order_id": "1",
  "user_id": "user_1001",
  "amount": 99.00
}

// 使用相同路由查询，避免广播到所有分片
GET /orders/_search?routing=user_1001
{
  "query": {
    "term": { "user_id": "user_1001" }
  }
}
```

### 3.3 分片分配策略

```yaml
# 索引级别分片分配策略
PUT /my_index/_settings
{
  "index.routing.allocation.total_shards_per_node": 2,
  "index.routing.allocation.include._tier_preference": "data_hot"
}

# 集群级别分配策略
cluster:
  routing:
    allocation:
      node_concurrent_recoveries: 2      # 并发恢复数
      node_initial_primaries_recoveries: 4  # 主分片恢复并发数
      disk:
        watermark:
          low: 85%        # 磁盘使用率 > 85% 不再分配新分片
          high: 90%       # 磁盘使用率 > 90% 触发分片迁移
          flood_stage: 95%  # 磁盘使用率 > 95% 索引变为只读
```

### 3.4 分片分配过滤器

```json
// 将索引分配到指定节点
PUT /hot_index/_settings
{
  "index.routing.allocation.include._name": "data-1, data-2"
}

// 基于自定义属性分配
PUT /logs-2024.01/_settings
{
  "index.routing.allocation.include.box_type": "hot"
}
```

## 5. 写入性能优化

### 5.1 Bulk批量写入

```java
// Java API — Bulk批量写入
BulkRequest bulkRequest = new BulkRequest();

for (int i = 0; i < documents.size(); i++) {
    IndexRequest request = new IndexRequest("orders")
        .id(documents.get(i).getId())
        .source(JSON.toJSONString(documents.get(i)), XContentType.JSON);
    bulkRequest.add(request);

    // 每 5000 条提交一次
    if (i % 5000 == 0 && i > 0) {
        BulkResponse response = bulkRequest.execute().actionGet();
        if (response.hasFailures()) {
            log.error("Bulk写入失败: {}", response.buildFailureMessage());
        }
        bulkRequest = new BulkRequest();
    }
}
// 提交剩余数据
if (bulkRequest.numberOfActions() > 0) {
    bulkRequest.execute().actionGet();
}
```

**Bulk大小建议**：

```
单次Bulk大小：5MB ~ 15MB
单次Bulk文档数：1000 ~ 5000 条
并发Bulk线程数：2 ~ 4（根据CPU核心数调整）

注意：
  - Bulk不是越大越好，过大导致JVM GC压力
  - 多线程并发Bulk比单线程大Bulk更高效
  - 监控Bulk rejection（线程池满），适当降低并发
```

### 5.2 Refresh Interval优化

```json
// 默认 refresh_interval = 1s（每秒刷新一次，生成新Segment）
// 写入密集场景应增大或禁用
```

```json
// Translog耐久性配置
```

| 参数 | 默认值 | 调优建议 | 说明 |
|------|--------|---------|------|

## 6. 查询性能优化

### 6.1 Filter缓存

Filter上下文不参与评分，结果会被缓存，适合精确过滤：

```json
// ✅ 推荐：使用filter替代query做精确匹配
GET /products/_search
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "published" } },
        { "range": { "price": { "gte": 100, "lte": 500 } } },
        { "terms": { "category_id": [1, 2, 3] } }
      ],
      "must": [
        { "match": { "title": "手机" } }
      ]
    }
  }
}
```

```
Filter缓存原理：
  - 每个Segment维护一个Filter Cache
  - 命中缓存的Filter不会重复计算
  - 缓存大小由 indices.queries.cache.size 控制（默认10%）
  - 适合基数较低的字段（如status、type、category）
```

### 6.2 Doc Values

```json
// Doc Values默认开启，提供高效的排序和聚合
// 确保关键字段开启doc_values
PUT /products
{

    "properties": {
      "price": {
        "type": "scaled_float",
        "scaling_factor": 100,
        "doc_values": true    // 排序/聚合字段必须开启
      },
      "description": {
        "type": "text",
        "doc_values": false   // 纯文本搜索字段可关闭，节省空间
      }
    }
  }
}
```

### 6.3 路由查询


### 6.4 分页优化

```json

  "size": 20,

    { "created_at": "desc" },
    { "_id": "asc" }
  ],
  "query": { "match_all": {} }
}

// ✅ Scroll 适合大批量导出（不适合实时分页）

{
  "size": 1000,
  "query": { "match_all": {} }
}
```

## 7. JVM堆内存配置

### 7.1 堆内存大小

```bash
# jvm.options 配置文件
# ========================
# 堆内存不超过32GB（指针压缩阈值）
# 推荐堆内存 = 物理内存的50%

# 64G物理内存的Data节点
-Xms31g
-Xmx31g

# 16G物理内存的Data节点
-Xms8g
-Xmx8g

# 使用G1垃圾收集器（ES 7+默认）
-XX:+UseG1GC
-XX:MaxGCPauseMillis=50
-XX:G1ReservePercent=25
```

```
为什么不超过32G？

  JVM在堆内存 < 32GB时启用指针压缩（Compressed Oops）
  使用32位指针引用对象，而非64位指针
  节省约50%的内存引用开销
  一旦超过32GB，指针解压，内存效率反而下降

  实际建议：堆内存设置在31GB以下，留出安全余量
```

### 7.2 堆外内存

```
ES除了JVM堆内存外，还使用大量堆外内存：

  1. Lucene段文件缓存（通过mmap映射到堆外）
     - 由操作系统管理，不需要JVM堆内存
     - 建议预留50%物理内存给Lucene

  2. Netty网络传输缓冲区
     - 直接内存，不占用堆

  3. 聚合/排序临时数据
     - 大聚合操作会使用堆外内存

内存分配原则（64G物理内存）：
  ┌──────────────────────┐
  │  JVM Heap:  31G      │ ← Xms31g Xmx31g
  ├──────────────────────┤
  │  Lucene缓存: 31G     │ ← 操作系统mmap
  ├──────────────────────┤
  │  其他/系统: 2G       │ ← OS + 其他进程
  └──────────────────────┘
```
