## 日志体系概述

### 为什么需要日志？

**价值**：
- 🔍 问题排查：快速定位线上问题
- 📊 数据分析：业务数据分析
- 🚨 监控告警：异常日志告警
- 🔐 安全审计：操作记录追溯

### 日志收集架构

```
应用日志文件 → Filebeat → Kafka → Logstash → Elasticsearch → Kibana
                                                      ↓
                                                  告警系统
```

**架构选型**：

| 架构方案 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| Filebeat → ES → Kibana | 简单 | 无缓冲、处理能力弱 | 小规模(<10台) |
| Filebeat → Kafka → Logstash → ES | 高可用、可扩展 | 架构复杂 | 生产环境(推荐) |
| Filebeat → Logstash → ES | 中等复杂度 | Logstash单点 | 中等规模 |

## 日志规范

### 1. 日志级别

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| **ERROR** | 系统错误 | 数据库连接失败、外部接口调用失败 |
| **WARN** | 警告信息 | 缓存未命中、降级处理 |
| **INFO** | 关键操作 | 用户登录、订单创建、支付成功 |
| **DEBUG** | 调试信息 | 方法参数、中间结果 |

### 2. 日志格式（JSON规范）

**推荐JSON格式**：
```json
{
  "timestamp": "2025-01-15 14:30:00.123",
  "level": "INFO",
  "traceId": "abc123def456",
  "spanId": "span001",
  "service": "order-service",
  "instance": "10.0.1.10:8080",
  "class": "com.example.order.OrderController",
  "method": "createOrder",
  "message": "Order created successfully",
  "context": {
    "orderId": "ORD20250115001",
    "userId": "U10086",
    "amount": 299.99,
    "duration": 156
  },
  "thread": "http-nio-8080-exec-1",
  "env": "production"
}
```

**Logback JSON配置**：
```xml
<!-- logback-spring.xml -->
<configuration>
    <springProperty scope="context" name="APP_NAME" source="spring.application.name"/>

    <appender name="JSON_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>/var/log/app/${APP_NAME}.log</file>
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <customFields>{"service":"${APP_NAME}"}</customFields>
            <includeMdc>true</includeMdc>
            <includeContext>true</includeContext>
        </encoder>
        <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
            <fileNamePattern>/var/log/app/${APP_NAME}.%d{yyyy-MM-dd}.%i.log</fileNamePattern>
            <maxFileSize>200MB</maxFileSize>
            <maxHistory>30</maxHistory>
            <totalSizeCap>10GB</totalSizeCap>
        </rollingPolicy>
    </appender>

    <root level="INFO">
        <appender-ref ref="JSON_FILE"/>
    </root>
</configuration>
```

**Maven依赖**：
```xml
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

### 3. 日志内容规范

```java
// ✅ 推荐：结构化日志
log.info("Order created, orderId={}, userId={}, amount={}, duration={}ms",
    order.getId(), order.getUserId(), order.getAmount(), duration);

// ❌ 不推荐：字符串拼接
log.info("Order created: " + order.getId() + " for user " + order.getUserId());

// ✅ 推荐：MDC注入traceId
MDC.put("traceId", traceContext.getTraceId());
MDC.put("userId", currentUser.getId());
log.info("Processing payment");
MDC.clear();

// ❌ 不推荐：打印敏感信息
log.info("User login: password=123456");  // 绝对禁止
```

### 2. Logstash配置

```ruby
input {
  kafka {
    bootstrap_servers => "kafka:9092"
    topics => ["app-logs"]
    group_id => "logstash-consumer"
    consumer_threads => 4
    decorate_events => true
  }
}

filter {
  # JSON解析
  if [message] =~ /^\{/ {
    json {
      source => "message"
      target => "log"
    }
  }

  # Grok解析非结构化日志
  if [message] !~ /^\{/ {
    grok {
      match => {
        "message" => "%{TIMESTAMP_ISO8601:timestamp}\s+%{LOGLEVEL:level}\s+%{DATA:thread}\s+%{JAVACLASS:class}\s*-\s*%{GREEDYDATA:log_message}"
      }
    }
  }

  # 时间解析
  date {
    match => ["timestamp", "yyyy-MM-dd HH:mm:ss.SSS", "ISO8601"]
    target => "@timestamp"
    timezone => "Asia/Shanghai"
  }

  # 移除无用字段
  mutate {
    remove_field => ["@version", "prospector", "input", "log", "offset"]
  }

  # 错误日志标记
  if [level] == "ERROR" or [level] == "FATAL" {
    mutate {
      add_tag => ["error_log"]
    }
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "app-logs-%{+YYYY.MM.dd}"
    template_name => "app-logs"
    template_overwrite => true
  }
}
```

## 日志分析实战

### 1. 日志查询（Kibana KQL）

```
# 错误日志排查
level: "ERROR" and service: "order-service"

# 特定traceId全链路
traceId: "abc123def456"

# 慢请求排查
context.duration > 3000

# 异常堆栈搜索
message: *NullPointerException* and level: "ERROR"

# 时间范围 + 服务过滤
@timestamp >= "2025-01-15T10:00:00" and service: "payment-service"
```

### 2. 问题排查案例

**案例：订单创建超时排查**

步骤1：根据traceId查全链路
```
traceId: "trace-20250115-001"
```

步骤2：按服务分析耗时
```
service: "order-service" and method: "createOrder" and context.duration > 1000
```

步骤3：定位具体错误
```
traceId: "trace-20250115-001" and level: "ERROR"
→ 发现 inventory-service 返回超时
```

### 2. 日志告警策略

| 告警类型 | 条件 | 级别 | 示例 |
|---------|------|------|------|
| **错误率告警** | 5分钟ERROR>10条 | P1 | NullPointerException激增 |

## 最佳实践

### 1. 日志优化

- **异步日志**：使用AsyncAppender，避免日志IO阻塞业务线程
- **合理级别**：生产环境INFO，排查问题时动态调整为DEBUG
- **避免大日志**：单条日志不超过10KB，大对象只记录摘要

```xml
<!-- 异步日志配置 -->
<appender name="ASYNC" class="ch.qos.logback.classic.AsyncAppender">
    <queueSize>1024</queueSize>
    <discardingThreshold>0</discardingThreshold>
    <neverBlock>true</neverBlock>
    <appender-ref ref="JSON_FILE"/>
</appender>
```

### 2. 存储优化

- **ILM生命周期管理**：hot(7天) → warm(30天) → cold(90天) → delete
- **索引分片合理**：单分片不超过50GB，3主分片+1副本
- **冷热分离**：SSD存储热数据，HDD存储冷数据
- **日志压缩**：启用best_compression，节省40%存储空间

### 3. 查询优化

- **精确匹配用keyword**：level、service、traceId用term查询
- **避免通配符开头**：`*Exception` 性能差，改用 `Exception*`
- **限定时间范围**：始终指定@timestamp范围
- **使用索引模式**：`app-logs-2025.01.*` 比 `app-logs-*` 更高效
