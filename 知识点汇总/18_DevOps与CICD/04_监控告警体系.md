## 监控体系概述

### 监控层次

```
基础设施监控（CPU、内存、磁盘、网络）
    ↓
应用监控（QPS、RT、错误率）
    ↓
业务监控（订单量、GMV、转化率）
    ↓
用户体验监控（页面加载、交互延迟）
```

### 四个黄金指标

1. **延迟（Latency）**：请求响应时间
2. **流量（Traffic）**：系统吞吐量
3. **错误（Errors）**：错误率
4. **饱和度（Saturation）**：资源利用率

### 监控工具选型

| 工具 | 用途 | 优势 |
|------|------|------|
| **Prometheus** | 指标监控 | 时序数据库、强大的查询语言 |
| **Grafana** | 可视化 | 丰富的Dashboard模板 |
| **ELK** | 日志监控 | 日志收集、分析、可视化 |
| **Skywalking** | APM | 链路追踪、性能分析 |

## Prometheus+Grafana

### 1. Prometheus架构

```
应用 → Exporter → Prometheus → Grafana
                       ↓
                   Alertmanager
```

**Docker部署**：
```yaml
# docker-compose.yml
version: '3.8'
services:
  prometheus:
    image: prom/prometheus:v2.51.0
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
      - prometheus_data:/prometheus
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.retention.time=30d'

  grafana:
    image: grafana/grafana:10.4.0
    ports:
      - "3000:3000"
    volumes:
      - grafana_data:/var/lib/grafana
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=admin

volumes:
  prometheus_data:
  grafana_data:
```

### 2. Prometheus配置

**核心配置文件** `prometheus.yml`：
```yaml

  scrape_interval: 15s       # 采集间隔
```

**Spring Boot接入Prometheus**：

```yaml

```

### 3. PromQL查询语言

```promql
# HTTP请求QPS（每秒请求数）
rate(http_server_requests_seconds_count[5m])

# P99延迟
histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m]))

# 错误率
sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m]))
/
sum(rate(http_server_requests_seconds_count[5m]))

# JVM堆内存使用率
jvm_memory_used_bytes{area="heap"}
/
jvm_memory_max_bytes{area="heap"} * 100

# CPU使用率
100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# 内存使用率
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100

# 磁盘使用率
(1 - (node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"})) * 100
```

## ELK日志监控

### 1. ELK架构

```
应用日志 → Filebeat → Kafka → Logstash → Elasticsearch → Kibana
                                                      ↓
                                                  告警系统
```

### 2. Filebeat配置

```yaml
# filebeat.yml
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/app/*.log
    json.keys_under_root: true
    json.add_error_key: true
    fields:
      app: order-service
      env: production
    fields_under_root: true

output.kafka:
  hosts: ["kafka:9092"]
  topic: "app-logs"
  partition.round_robin:
    reachable_only: true
  required_acks: 1

processors:
  - add_host_metadata: ~
  - add_cloud_metadata: ~
```

### 3. Logstash配置

```ruby
# logstash.conf
input {
  kafka {
    bootstrap_servers => "kafka:9092"
    topics => ["app-logs"]
    group_id => "logstash"
    consumer_threads => 4
  }
}

filter {
  # 解析JSON日志
  json {
    source => "message"
    target => "log"
  }

  # 添加时间戳
  date {
    match => ["[log][timestamp]", "yyyy-MM-dd HH:mm:ss.SSS"]
    target => "@timestamp"
  }

  # 提取traceId用于链路追踪
  mutate {
    add_field => {
      "traceId" => "%{[log][traceId]}"
    }
  }

  # 错误日志标记
  if [log][level] == "ERROR" {
    mutate {
      add_tags => ["error"]
    }
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "app-logs-%{+YYYY.MM.dd}"
    template_name => "app-logs"
  }
}
```

### 4. Kibana查询

**常用KQL查询**：
```
# 查询错误日志
level: "ERROR" and message: *NullPointerException*

# 查询特定服务的慢请求
app: "order-service" and duration > 3000

# 查询特定traceId的全链路日志
traceId: "abc123def456"
```

## Skywalking链路追踪

### 1. Skywalking架构

```
应用(JavaAgent) → OAP Server → Storage(Elasticsearch/MySQL)
                      ↓
                   UI Dashboard
```

**Docker部署**：
```yaml
services:
  oap:
    image: apache/skywalking-oap-server:9.7.0
    ports:
      - "11800:11800"  # gRPC
      - "12800:12800"  # HTTP
    environment:
      SW_STORAGE: elasticsearch
      SW_STORAGE_ES_CLUSTER_NODES: elasticsearch:9200

  ui:
    image: apache/skywalking-ui:9.7.0
    ports:
      - "8080:8080"
    environment:
      SW_OAP_ADDRESS: http://oap:12800
```

### 2. Agent配置

**启动参数**：
```bash
java -javaagent:/path/to/skywalking-agent.jar \
     -Dskywalking.agent.service_name=order-service \
     -Dskywalking.collector.backend_grpc=oap:11800 \
     -Dskywalking.logging.level=INFO \
     -jar app.jar
```

**Spring Boot集成**：
```xml
<dependency>
    <groupId>org.apache.skywalking</groupId>
    <artifactId>apm-toolkit-trace</artifactId>
    <version>9.7.0</version>
</dependency>
<dependency>
    <groupId>org.apache.skywalking</groupId>
    <artifactId>apm-toolkit-logback-1.x</artifactId>
    <version>9.7.0</version>
</dependency>
```

**自定义Span**：
```java
import org.apache.skywalking.apm.toolkit.trace.TraceCrossThread;
import org.apache.skywalking.apm.toolkit.trace.ActiveSpan;

// 自定义标签
```

**Logback集成（日志中输出traceId）**：
```xml

<appender name="grpc-log" class="org.apache.skywalking.apm.toolkit.log.logback.v1.x.gRPCLogClientAppender">
    <grpcHost>oap</grpcHost>
    <grpcPort>11800</grpcPort>
</appender>
```

## 告警规则配置

### 1. 告警分级

| 级别 | 说明 | 响应时间 | 通知方式 |
|------|------|---------|---------|
| **P0** | 严重故障 | 立即 | 电话+短信+企微 |
| **P1** | 重要故障 | 15分钟 | 短信+企微 |
| **P2** | 一般告警 | 1小时 | 企微 |
| **P3** | 提示信息 | 24小时 | 邮件 |

### 2. Prometheus告警规则

```yaml
# alert_rules.yml
groups:
  - name: system_alerts
    rules:
      # CPU使用率告警
      - alert: HighCPUUsage
        expr: 100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
        for: 5m
        labels:
          severity: P1
        annotations:
          summary: "CPU使用率过高 {{ $value }}%"
          description: "实例 {{ $labels.instance }} CPU使用率持续5分钟超过80%"

      # 内存使用率告警
      - alert: HighMemoryUsage
        expr: (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 85
        for: 5m
        labels:
          severity: P1
        annotations:
          summary: "内存使用率过高 {{ $value }}%"

      # 磁盘空间不足
      - alert: DiskSpaceLow
        expr: (1 - (node_filesystem_avail_bytes / node_filesystem_size_bytes)) * 100 > 90
        for: 10m
        labels:
          severity: P2
        annotations:
          summary: "磁盘空间不足 {{ $value }}%"

  - name: app_alerts
    rules:
      # HTTP错误率告警
      - alert: HighErrorRate
        expr: sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m])) / sum(rate(http_server_requests_seconds_count[5m])) * 100 > 5
        for: 3m
        labels:
          severity: P0
        annotations:
          summary: "接口错误率过高 {{ $value }}%"

      # JVM堆内存告警
      - alert: JVMHeapHigh
        expr: jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} * 100 > 85
        for: 5m
        labels:
          severity: P1

      # 接口响应时间P99告警
      - alert: HighLatencyP99
        expr: histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m])) > 3
        for: 5m
        labels:
          severity: P2
        annotations:
          summary: "P99延迟过高 {{ $value }}s"
```

### 3. Alertmanager告警收敛

```yaml
# alertmanager.yml
global:
  resolve_timeout: 5m

route:
  group_by: ['alertname', 'cluster']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  receiver: 'wechat'
  routes:
    - match:
        severity: P0
      receiver: 'phone'
      repeat_interval: 5m
    - match:
        severity: P1
      receiver: 'wechat'

receivers:
  - name: 'wechat'
    webhook_configs:
      - url: 'http://webhook-adapter/wechat'
  - name: 'phone'
    webhook_configs:
      - url: 'http://webhook-adapter/phone'

# 告警抑制：P0触发时抑制P1
inhibit_rules:
  - source_match:
      severity: P0
    target_match:
      severity: P1
    equal: ['alertname', 'instance']
```

## 最佳实践

### 1. 监控指标设计

**RED方法（请求驱动服务）**：
- **Rate**：请求速率（QPS）
- **Errors**：错误率
- **Duration**：延迟分布（P50/P95/P99）

**USE方法（资源型系统）**：
- **Utilization**：使用率
- **Saturation**：饱和度
- **Errors**：错误数

### 2. Dashboard设计原则

- **分层展示**：全局概览 → 服务详情 → 实例级指标
- **关键指标突出**：使用大数字面板展示核心指标
- **阈值标色**：绿色正常、黄色警告、红色异常
- **时间范围**：默认最近1小时，支持自定义

### 3. 告警优化

- **避免告警风暴**：设置group_wait和group_interval，合并同类告警
- **设置合理for时间**：避免瞬时波动触发告警
- **分级通知**：P0电话、P1短信、P2企微、P3邮件
- **告警可操作性**：每个告警附带排查指南和处理步骤
- **定期review**：每月审查告警有效性，清理无效告警
