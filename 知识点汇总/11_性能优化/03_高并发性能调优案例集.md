## 1. 调优方法论

### 性能调优流程

```
1. 确定指标：RT < 200ms, QPS > 5000, 错误率 < 0.1%
2. 压测基线：用JMeter/Gatling建立性能基线
3. 监控定位：用APM(Skywalking/Prometheus)定位瓶颈
4. 分析根因：CPU/内存/IO/网络/数据库/代码
5. 优化实施：针对性优化
6. 验证效果：对比优化前后指标
7. 持续监控：防止回退
```

### 性能瓶颈定位

```
┌─────────────────────────────────────────┐
│              用户请求                     │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐    │
│  │ CDN │→│ Nginx│→│ 网关  │→│ 应用  │    │
│  └─────┘  └─────┘  └─────┘  └──┬──┘    │
│                                   │      │
│              ┌────────────────────┼──┐   │
│              │  ┌──────┐  ┌──────┐│  │   │
│              │  │ Redis│  │MySQL ││  │   │
│              │  └──────┘  └──────┘│  │   │
│              └────────────────────┘  │   │
└─────────────────────────────────────────┘

每一层都可能成为瓶颈：
  CDN/Nginx → 网络带宽、连接数
  网关 → 限流、路由性能
  应用 → JVM、代码逻辑、线程池
  Redis → 内存、网络、热点Key
  MySQL → 慢SQL、锁等待、连接数
```

## 2. 案例一：秒杀系统性能优化

### 场景

```
1000件商品，10万人同时抢购
原始方案：直接查库 → 下单 → 扣库存
问题：QPS 500，RT > 3s，大量超时
```

### 优化过程

```
第一层：缓存预热
  - 商品信息提前加载到Redis
  - 库存数量存Redis（SET stock:1001 1000）

第二层：异步化
  - 用户请求 → MQ → 异步处理
  - 快速返回"排队中" → 前端轮询结果

第三层：防超卖
  - Redis Lua脚本原子扣减库存
  - 库存≤0直接拒绝

第四层：限流
  - Sentinel限流：单用户1次/秒，全局10000 QPS
  - 超过限流 → 快速失败"活动太火爆"

第五层：前端优化
  - 按钮防抖（3秒内只能点一次）
  - 验证码拦截机器人
```

### 优化后代码

```java
@Service
public class SeckillService {
    
    @Autowired
    private RedisTemplate<String, String> redis;
    
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    // Lua脚本：原子扣减库存
    private static final String DEDUCT_SCRIPT = """
        local stock = redis.call('GET', KEYS[1])
        if not stock then return -1 end
        stock = tonumber(stock)
        if stock <= 0 then return 0 end
        redis.call('DECR', KEYS[1])
        return 1
        """;
    
    @SentinelResource(value = "seckill", blockHandler = "seckillBlocked")
    public SeckillResult seckill(Long userId, Long productId) {
        // 1. 限流检查（Sentinel处理）
        // 2. 用户防重
        String userKey = "seckill:user:" + userId + ":" + productId;
        Boolean isFirst = redis.opsForValue().setIfAbsent(userKey, "1", 60, TimeUnit.SECONDS);
        if (isFirst == null || !isFirst) {
            return SeckillResult.fail("您已参与过此活动");
        }
        
        // 3. 原子扣减库存
        Long result = redis.execute(
            new DefaultRedisScript<>(DEDUCT_SCRIPT, Long.class),
            List.of("seckill:stock:" + productId));
        
        if (result == null || result == 0) {
            return SeckillResult.fail("已售罄");
        }
        
        // 4. 发送MQ异步创建订单
        mqTemplate.asyncSend("seckill-order", 
            new SeckillMessage(userId, productId), 
            new SendCallback() {
                @Override
                public void onSuccess(SendResult r) {}
                @Override
                public void onException(Throwable e) {
                    // 库存回滚
                    redis.opsForValue().increment("seckill:stock:" + productId);
                }
            });
        
        return SeckillResult.success("抢购成功，订单创建中");
    }
}
```

### 优化效果

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| QPS | 500 | 50,000 | 100x |
| RT | 3s | 50ms | 60x |
| 超卖 | 有 | 无 | - |
| 错误率 | 30% | 0.01% | - |

## 3. 案例二：接口响应慢排查

### 场景

```
订单查询接口RT从100ms上升到2s
现象：偶发性变慢，非持续
```

### 排查过程

```
1. APM链路追踪（Skywalking）
   → 发现订单查询调用了用户服务3次（重复调用）

2. 代码分析
   → 循环中调用远程接口：
   for (Order order : orders) {
       User user = userService.getUser(order.getUserId());  // N+1问题
   }

3. 优化方案
   → 批量查询用户信息：
   Set<Long> userIds = orders.stream().map(Order::getUserId).collect(toSet());
   Map<Long, User> userMap = userService.batchGetUsers(userIds);
   for (Order order : orders) {
       User user = userMap.get(order.getUserId());
   }
```

### 其他常见慢接口原因

```
1. N+1查询 → 批量查询或JOIN
2. 循环调用RPC → 批量接口
3. 大对象序列化 → 精简返回字段
4. 数据库慢SQL → 加索引、优化SQL
5. 锁等待 → 减小锁粒度
6. GC停顿 → JVM调优
7. 网络抖动 → 重试 + 超时
```

## 4. 案例三：数据库CPU飙升

### 场景

```
MySQL CPU使用率从30%飙升至95%
现象：QPS没有明显增长，但CPU高
```

### 排查过程

```
1. 查看当前执行SQL
   SHOW PROCESSLIST;
   → 发现大量Waiting for lock metadata
   
2. 慢查询日志
   → 发现一条慢SQL：
   SELECT * FROM orders WHERE DATE(create_time) = '2026-07-05';
   → 函数操作导致索引失效，全表扫描

3. 优化SQL
   -- 改为范围查询
   SELECT * FROM orders 
   WHERE create_time >= '2026-07-05' 
     AND create_time < '2026-07-06';

4. 验证
   EXPLAIN → type从ALL变为range，rows从100万降到5000
```

### 其他CPU飙升原因

```
1. 全表扫描 → 加索引
2. 临时表排序 → Using filesort → 优化ORDER BY
3. 大量锁等待 → 优化事务（减小事务粒度）
4. 存储过程死循环 → 代码审查
5. 统计信息过期 → ANALYZE TABLE
```

## 5. 案例四：JVM Full GC频繁

### 场景

```
应用每隔2分钟Full GC一次，STW约3秒
现象：接口周期性超时
```

### 排查过程

```
1. 查看GC日志
   -Xms4g -Xmx4g -XX:+UseG1GC
   G1 Full GC间隔约2分钟，每次回收3GB
   
2. 分析Heap Dump
   jmap -dump:format=b,file=heap.hprof <pid>
   MAT分析 → 发现LocalCache占3.5GB
   
3. 根因
   Caffeine缓存未设置maximumSize，无限增长
   缓存了100万条商品数据，每条3KB = 3GB
   
4. 优化
   Caffeine.newBuilder()
       .maximumSize(10_000)      // 限制最大1万条
       .expireAfterWrite(30, TimeUnit.MINUTES)
       .build();
   
5. 效果
   Full GC间隔从2分钟 → 30分钟，STW从3s → 200ms
```

### JVM调优常用参数

```bash
# G1 GC推荐配置
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200        # 目标停顿时间
-XX:G1HeapRegionSize=16m        # Region大小
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发标记的堆占用率
-XX:MaxRAMPercentage=75.0       # 堆内存占容器内存比例

# GC日志
-Xlog:gc*:file=/var/log/gc.log:time,uptime:filecount=10,filesize=100m
```

## 6. 案例五：缓存命中率低

### 场景

```
Redis缓存命中率从95%降到60%
现象：数据库压力增大
```

### 排查过程

```
1. 分析缓存Key分布
   redis-cli --bigkeys → 发现大量小Key
   
2. 分析过期策略
   → 所有Key设置了相同TTL（30分钟）
   → 30分钟后大量Key同时过期 → 缓存雪崩
   
3. 优化
   // TTL加随机值
   int ttl = 1800 + random.nextInt(600);  // 30-40分钟
   redis.opsForValue().set(key, value, ttl, TimeUnit.SECONDS);
   
4. 进一步优化
   - 热点数据永不过期，后台异步刷新
   - 多级缓存：Caffeine(L1) + Redis(L2)
   - 布隆过滤器防穿透
```

### 缓存命中率公式

```
命中率 = 命中次数 / 总查询次数

低命中率原因：
  1. TTL过短 → 调整TTL
  2. Key设计不合理 → 优化Key结构
  3. 数据冷热不均 → 热点数据特殊处理
  4. 缓存穿透 → 布隆过滤器
  5. 缓存雪崩 → TTL加随机值
  6. 缓存击穿 → 互斥锁 + 提前刷新
```

## 7. 面试要点

### Q1: 如何排查接口慢的问题？

```
1. APM链路追踪定位慢在哪一步
2. 数据库：慢查询日志 + EXPLAIN
3. 应用：JVM GC日志 + 线程Dump
4. 网络：DNS + TCP RTT
5. 常见原因：N+1查询、慢SQL、GC停顿、锁等待、大对象序列化
```

### Q2: 如何设计一个高并发系统？

```
1. 缓存：多级缓存(Caffeine+Redis)，命中率>95%
2. 异步：MQ削峰填谷，非核心流程异步化
3. 限流：Sentinel保护系统，快速失败
4. 降级：非核心功能降级，保住核心链路
5. 拆分：读写分离 + 分库分表
6. 连接池：数据库/Redis/HTTP连接池调优
7. 无状态：应用无状态，水平扩展
```

### Q3: JVM频繁Full GC怎么排查？

```
1. 看GC日志：Full GC频率和回收量
2. jmap dump：MAT分析内存占用
3. 常见原因：
   - 内存泄漏（缓存无限增长）
   - 大对象直接进老年代
   - 老年代空间不足
   - Metaspace溢出
4. 解决：
   - 限制缓存大小
   - 调整新生代/老年代比例
   - 修复内存泄漏
```

### Q4: 如何防止缓存雪崩？

```
缓存雪崩：大量Key同时过期，请求全部打到DB

预防：
1. TTL加随机值（30分钟+0-10分钟随机）
2. 多级缓存（L1 Caffeine兜底）
3. 互斥锁（同一Key只允许一个请求查DB）
4. 提前刷新（TTL剩余20%时异步刷新）
5. 熔断降级（DB压力大时限流降级）
```

### Q5: 秒杀系统的核心设计？

```
1. 前端：按钮防抖 + 验证码 + 答题
2. 网关：限流（Sentinel） + 黑名单
3. 缓存：库存预存Redis + Lua原子扣减
4. 异步：MQ异步下单，快速响应用户
5. 防超卖：Redis Lua脚本 + 数据库乐观锁
6. 降级：库存售罄后直接拒绝，不查库
7. 监控：实时监控库存、订单量、RT
```
