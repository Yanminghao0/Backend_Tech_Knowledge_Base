## 慢SQL定位与分析

### 1.1 慢查询日志

慢查询日志是定位慢SQL的第一道防线，它记录所有执行时间超过阈值的SQL语句。

**开启与配置：**

```sql
-- 查看慢查询日志配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
SET GLOBAL long_query_time = 1;       -- 超过1秒的SQL记为慢查询
SET GLOBAL log_queries_not_using_indexes = 'ON';  -- 记录未使用索引的SQL
SET GLOBAL min_examined_row_limit = 100;  -- 扫描行数超过100才记录

-- 持久化配置（my.cnf）
-- [mysqld]
-- slow_query_log = 1
-- slow_query_log_file = /var/log/mysql/slow.log
-- long_query_time = 1
-- log_queries_not_using_indexes = 1
```

**分析慢查询日志：**

```bash
# 使用mysqldumpslow分析慢查询日志
# 按总耗时排序（找出最耗时的SQL）
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# 按次数排序（找出执行最频繁的SQL）
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# 按返回行数排序（找出扫描大量数据的SQL）
mysqldumpslow -s r -t 10 /var/log/mysql/slow.log

# 按锁时间排序
mysqldumpslow -s l -t 10 /var/log/mysql/slow.log

# 参数说明：
# -s: 排序方式 t=总时间 c=次数 r=返回行 l=锁时间
# -t: 返回前N条
# -g: 正则过滤
```

**使用pt-query-digest深度分析（推荐）：**

```bash
# 安装percona toolkit
yum install percona-toolkit

# 分析慢查询日志，生成详细报告
pt-query-digest /var/log/mysql/slow.log > slow_report.txt

# 只分析某个时间段的慢查询
pt-query-digest --since '2024-01-01 00:00:00' --until '2024-01-02 00:00:00' /var/log/mysql/slow.log

# 分析processlist
pt-query-digest --processlist h=localhost,u=root,p=password --interval 30 --run-time 300
```

pt-query-digest输出示例解读：

```

# ==== ============================ ============== ===== ======= ===== =========
#    1 0xABC123...                   120.5 45.2%    85  1.4176  0.12 SELECT orders
```

### 1.2 EXPLAIN执行计划分析

EXPLAIN是分析SQL执行计划的核心工具，能够揭示MySQL如何执行查询。

```sql
EXPLAIN SELECT o.order_id, o.user_id, u.user_name, o.total_amount
FROM orders o
JOIN users u ON o.user_id = u.user_id
WHERE o.status = 'PAID' AND o.create_time > '2024-01-01'
ORDER BY o.create_time DESC
LIMIT 20;
```

**EXPLAIN输出字段详解：**

```
+----+-------------+-------+------------+--------+-------------------+-------------------+---------+-------------------+------+----------+----------------------------------------------+
| id | select_type | table | partitions | type   | possible_keys     | key               | key_len | ref               | rows | filtered | Extra                                        |
+----+-------------+-------+------------+--------+-------------------+-------------------+---------+-------------------+------+----------+----------------------------------------------+
|  1 | SIMPLE      | o     | NULL       | ref    | idx_status,idx_time| idx_status_time   | 130     | const             | 1500 |    33.33 | Using where; Using index condition           |
|  1 | SIMPLE      | u     | NULL       | eq_ref | PRIMARY           | PRIMARY           | 8       | test.o.user_id    |    1 |   100.00 | NULL                                         |
+----+-------------+-------+------------+--------+-------------------+-------------------+---------+-------------------+------+----------+----------------------------------------------+
```

**关键字段解读：**

| 字段 | 说明 | 优化关注点 |
|------|------|------------|
| **type** | 访问类型 | system > const > eq_ref > ref > range > index > ALL。ALL为全表扫描，必须优化 |
| **key** | 实际使用的索引 | NULL表示未走索引 |
| **key_len** | 索引使用长度 | 判断联合索引用到了几个字段 |
| **rows** | 预估扫描行数 | 越小越好 |
| **filtered** | 过滤比例 | 百分比越小说明扫描了大量无用行 |
| **Extra** | 额外信息 | 出现Using filesort/Using temporary需要重点关注 |

**Extra字段重点关注项：**

```sql
-- Using filesort：无法利用索引排序，需要额外排序操作
EXPLAIN SELECT * FROM orders WHERE user_id = 100 ORDER BY create_time DESC;
-- 优化：创建 (user_id, create_time) 联合索引

-- Using temporary：使用临时表，常见于GROUP BY、DISTINCT
EXPLAIN SELECT status, COUNT(*) FROM orders GROUP BY status;
-- 优化：为GROUP BY字段创建索引

-- Using index condition：索引下推（ICP），5.6+优化
-- Using index：覆盖索引，不需要回表，性能最佳
-- Using join buffer：JOIN无索引，使用Block Nested-Loop，需要为JOIN字段加索引
```

### 1.3 实时SQL监控

```sql
-- 查看当前正在执行的SQL（information_schema）
SELECT 
    id,
    user,
    host,
    db,
    command,
    time AS duration_seconds,
    state,
    LEFT(info, 200) AS sql_text

    r.trx_id AS waiting_trx_id,
    r.trx_mysql_thread_id AS waiting_thread,
```

## 索引优化策略

### 2.1 索引类型与选择

```sql
-- 主键索引（聚簇索引）：InnoDB中数据按主键组织，建议使用自增整型
CREATE TABLE orders (
    order_id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    order_no VARCHAR(32) NOT NULL,
    user_id BIGINT NOT NULL,
    status VARCHAR(20) NOT NULL,
    total_amount DECIMAL(12,2) NOT NULL,
    create_time DATETIME NOT NULL,
    PRIMARY KEY (order_id),              -- 主键索引（聚簇索引）
    UNIQUE KEY uk_order_no (order_no),   -- 唯一索引
    KEY idx_user_id (user_id),           -- 普通索引
    KEY idx_status_time (status, create_time)  -- 联合索引
) ENGINE=InnoDB;

-- 索引选择原则：
-- 1. 区分度高的字段优先建索引（selectivity = DISTINCT值数 / 总行数）
SELECT 
    COUNT(DISTINCT status) / COUNT(*) AS status_selectivity,    -- 低区分度，不适合单独建索引
    COUNT(DISTINCT user_id) / COUNT(*) AS user_selectivity,     -- 高区分度，适合建索引
    COUNT(DISTINCT order_no) / COUNT(*) AS order_no_selectivity -- 接近1，适合建唯一索引
FROM orders;

-- 2. 频繁查询的WHERE、JOIN、ORDER BY、GROUP BY字段建索引
-- 3. 避免在低区分度字段单独建索引（如status只有3个值）
-- 4. 联合索引把区分度高的字段放前面
```

### 2.2 覆盖索引

覆盖索引指查询所需的所有字段都包含在索引中，无需回表查询，能大幅提升性能。

```sql
-- 假设有联合索引 idx_user_status (user_id, status)
-- 包含user_id, status，以及聚簇索引键order_id

-- ❌ 需要回表：查询了total_amount不在索引中
SELECT user_id, status, total_amount FROM orders WHERE user_id = 100;

-- ✅ 覆盖索引：查询字段都在索引中
SELECT user_id, status, order_id FROM orders WHERE user_id = 100;

-- ✅ 覆盖索引：利用索引的有序性避免filesort
SELECT user_id, status FROM orders WHERE user_id = 100 ORDER BY status;

-- 优化技巧：将高频查询字段加入联合索引实现覆盖索引
-- 原索引：idx_user_id (user_id)
-- 优化后：idx_user_status_amount (user_id, status, total_amount)
-- 这样以下查询可以走覆盖索引：
SELECT status, total_amount FROM orders WHERE user_id = 100;
```

**覆盖索引的Explain验证：**

```sql
-- 覆盖索引，Extra显示 Using index
EXPLAIN SELECT user_id, status, order_id FROM orders WHERE user_id = 100;
-- Extra: Using index

-- 非覆盖索引，Extra不显示 Using index
EXPLAIN SELECT user_id, status, total_amount FROM orders WHERE user_id = 100;
-- Extra: NULL（需要回表）
```

### 2.3 最左前缀原则

联合索引遵循最左前缀原则，查询必须从索引最左列开始才能使用索引。

```sql
-- 联合索引 idx_status_time (status, create_time)
-- 索引结构按 status 排序，相同status内按 create_time 排序

-- ✅ 使用索引：包含最左列status
SELECT * FROM orders WHERE status = 'PAID';
SELECT * FROM orders WHERE status = 'PAID' AND create_time > '2024-01-01';
SELECT * FROM orders WHERE status = 'PAID' AND create_time > '2024-01-01' ORDER BY create_time;

-- ❌ 无法使用索引：缺少最左列status
SELECT * FROM orders WHERE create_time > '2024-01-01';

-- ✅ 联合索引(status, create_time)均可使用：status等值+create_time范围，但total_amount不在索引中
SELECT * FROM orders WHERE status = 'PAID' AND create_time > '2024-01-01' AND total_amount > 100;
-- 只能使用 idx_status_time 的 status 部分

-- 范围查询会中断后续列的索引使用
-- ✅ status能走索引，create_time能走索引
SELECT * FROM orders WHERE status = 'PAID' AND create_time > '2024-01-01';
-- 但如果create_time是范围查询，其后面的列无法走索引
-- 联合索引 (a, b, c)：WHERE a=1 AND b>5 AND c=3 → 只有a和b能走索引
```

### 2.4 索引下推（Index Condition Pushdown, ICP）

索引下推是MySQL 5.6引入的优化，将WHERE条件的一部分下推到存储引擎层，在索引遍历过程中过滤，减少回表次数。

```sql
-- 假设有联合索引 idx_name_age (last_name, first_name, age)
-- 表结构：users(id, last_name, first_name, age, city, ...)

-- 无ICP（5.6之前）：
-- 1. 存储引擎根据 last_name 前缀从索引中找到所有匹配记录
-- 2. 回表取完整行数据
-- 3. MySQL Server层过滤 age > 18 的记录
-- 如果 last_name='Smith' 有10000条，但 age>18 只有100条，需要回表10000次

-- 有ICP（5.6+）：
-- 1. 存储引擎根据 last_name 前缀从索引中找到匹配记录
-- 2. 在索引层直接判断 age > 18 条件（age在索引中）
-- 3. 只有满足条件的记录才回表
-- 只需回表100次，大幅减少IO

-- 验证ICP是否生效
EXPLAIN SELECT * FROM users WHERE last_name = 'Smith' AND age > 18;
-- Extra: Using index condition  → ICP生效
-- Extra: Using where             → ICP未生效，条件在Server层过滤

-- 注意：ICP适用于：
-- 1. 联合索引的非最左前缀列上的范围条件
-- 2. LIKE 'prefix%' 前缀匹配
-- 3. 不适用于子查询、存储函数等
```

### 2.5 索引优化实战技巧

```sql
-- 1. 避免索引失效的常见场景

-- ❌ 函数操作导致索引失效
SELECT * FROM orders WHERE DATE(create_time) = '2024-01-01';
-- ✅ 改写为范围查询
SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2024-01-02';

-- ❌ 隐式类型转换导致索引失效
SELECT * FROM orders WHERE order_no = 12345;  -- order_no是VARCHAR
-- ✅ 使用正确类型
SELECT * FROM orders WHERE order_no = '12345';

-- ❌ 左模糊导致索引失效
SELECT * FROM users WHERE user_name LIKE '%张';
-- ✅ 使用右模糊（可以利用索引）
SELECT * FROM users WHERE user_name LIKE '张%';

-- ❌ OR连接条件，其中一个无索引，导致全部索引失效
SELECT * FROM orders WHERE user_id = 100 OR total_amount > 1000;
-- ✅ 使用UNION ALL改写
SELECT * FROM orders WHERE user_id = 100
UNION ALL
SELECT * FROM orders WHERE total_amount > 1000 AND user_id != 100;

-- ❌ != 或 <> 操作符通常不走索引
SELECT * FROM orders WHERE status != 'PAID';
-- ✅ 如果状态值有限，改写为IN
SELECT * FROM orders WHERE status IN ('PENDING', 'CANCELLED', 'REFUNDED');

-- 2. 前缀索引优化长字符串
-- 对于很长的字符串字段，使用前缀索引节省空间
ALTER TABLE articles ADD INDEX idx_content_prefix (content(20));
-- 选择合适前缀长度的方法：
SELECT 
    COUNT(DISTINCT LEFT(content, 10)) / COUNT(*) AS sel_10,
    COUNT(DISTINCT LEFT(content, 15)) / COUNT(*) AS sel_15,
    COUNT(DISTINCT LEFT(content, 20)) / COUNT(*) AS sel_20,
    COUNT(DISTINCT LEFT(content, 25)) / COUNT(*) AS sel_25
FROM articles;
-- 选择区分度接近完整列的前缀长度

-- 3. 查看索引使用情况，清理无用索引
SELECT 
    object_schema AS db,
    object_name AS table_name,
    index_name,
    count_read,
    count_fetch,
    count_insert,
    count_update,
    count_delete
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE object_schema = 'your_db'
ORDER BY count_read ASC;
-- count_read为0的索引可以考虑删除
```

## SQL改写技巧

### 3.1 JOIN vs 子查询

```sql
-- ❌ 子查询：MySQL对子查询优化较弱，可能产生临时表
SELECT * FROM orders 
WHERE user_id IN (SELECT user_id FROM users WHERE vip_level = 5);

-- ✅ JOIN改写：通常性能更好
SELECT o.* 
FROM orders o
INNER JOIN users u ON o.user_id = u.user_id
WHERE u.vip_level = 5;

-- ❌ 相关子查询：每行都执行子查询
SELECT order_id, 
       (SELECT user_name FROM users WHERE user_id = o.user_id) AS user_name,
       total_amount
FROM orders o
WHERE o.status = 'PAID';

-- ✅ JOIN改写
SELECT o.order_id, u.user_name, o.total_amount
FROM orders o
INNER JOIN users u ON o.user_id = u.user_id
WHERE o.status = 'PAID';

-- EXISTS vs IN：当外表小内表大时用IN，外表大内表小时用EXISTS
-- ✅ 外表大、子查询表小：用IN
SELECT * FROM large_table WHERE id IN (SELECT id FROM small_table WHERE flag = 1);

-- ✅ 外表小、子查询表大：用EXISTS
SELECT * FROM small_table s WHERE EXISTS (
    SELECT 1 FROM large_table l WHERE l.ref_id = s.id AND l.status = 'ACTIVE'
);
```

### 3.2 分页优化

```sql
-- ❌ 传统LIMIT分页：偏移量越大越慢
SELECT * FROM orders ORDER BY create_time DESC LIMIT 100000, 20;
-- MySQL需要扫描100020行，丢弃前100000行

-- ✅ 延迟关联（子查询+覆盖索引）
SELECT o.* FROM orders o
INNER JOIN (
    SELECT order_id FROM orders ORDER BY create_time DESC LIMIT 100000, 20
) t ON o.order_id = t.order_id;
-- 子查询利用覆盖索引快速定位order_id，再关联取完整数据

-- ✅ 游标分页（推荐）：记住上一页的最后一条记录
-- 第一页
SELECT * FROM orders WHERE status = 'PAID' ORDER BY create_time DESC, order_id DESC LIMIT 20;
-- 第二页（传上一页最后一条的create_time和order_id）
SELECT * FROM orders 
WHERE status = 'PAID' 
  AND (create_time < '2024-01-15 10:30:00' 
       OR (create_time = '2024-01-15 10:30:00' AND order_id < 10020))
ORDER BY create_time DESC, order_id DESC 
LIMIT 20;
-- 游标分页不受偏移量影响，性能稳定

-- ✅ BETWEEN AND 分页（已知页码范围）
SELECT * FROM orders WHERE order_id BETWEEN 100001 AND 100020 ORDER BY order_id;
```

### 3.3 批量操作优化

```java
// ==================== Java代码示例 ====================

// ❌ 逐条插入：每次都要建立网络往返和事务提交
for (Order order : orderList) {
    orderMapper.insert(order);  // 1000条 = 1000次DB交互
}

// ✅ 批量插入：一次网络往返
// MyBatis批量插入
@Insert({
    "<script>",
    "INSERT INTO orders (order_no, user_id, status, total_amount, create_time)",
    "VALUES ",
    "<foreach collection='list' item='item' separator=','>",
    "(#{item.orderNo}, #{item.userId}, #{item.status}, #{item.totalAmount}, #{item.createTime})",
    "</foreach>",
    "</script>"
})
int batchInsert(@Param("list") List<Order> orderList);

// 分批处理：避免单次SQL过大
public void batchInsertOrders(List<Order> allOrders) {
    int batchSize = 500;  // 每批500条
    for (int i = 0; i < allOrders.size(); i += batchSize) {
        int end = Math.min(i + batchSize, allOrders.size());
        List<Order> batch = allOrders.subList(i, end);
        orderMapper.batchInsert(batch);
    }
}

// ✅ 使用JDBC batch + rewriteBatchedStatements（最高性能）
public void jdbcBatchInsert(Connection conn, List<Order> orders) throws SQLException {
    String sql = "INSERT INTO orders (order_no, user_id, status, total_amount, create_time) VALUES (?, ?, ?, ?, ?)";
    try (PreparedStatement ps = conn.prepareStatement(sql)) {
        conn.setAutoCommit(false);
        for (Order order : orders) {
            ps.setString(1, order.getOrderNo());
            ps.setLong(2, order.getUserId());
            ps.setString(3, order.getStatus());
            ps.setBigDecimal(4, order.getTotalAmount());
            ps.setTimestamp(5, Timestamp.valueOf(order.getCreateTime()));
            ps.addBatch();
        }
        ps.executeBatch();
        conn.commit();
    }
}

// JDBC URL必须加上 rewriteBatchedStatements=true
// jdbc:mysql://localhost:3306/order_db?rewriteBatchedStatements=true&useServerPrepStmts=true
```

```sql
-- 批量更新：使用CASE WHEN
UPDATE orders SET status = CASE order_id
    WHEN 1001 THEN 'SHIPPED'
    WHEN 1002 THEN 'DELIVERED'
    WHEN 1003 THEN 'CANCELLED'
END
WHERE order_id IN (1001, 1002, 1003);

INSERT INTO orders (order_no, user_id, status, total_amount, create_time)
VALUES 
    ('NO001', 100, 'PAID', 99.00, NOW()),
    ('NO002', 101, 'PAID', 199.00, NOW()),
    ('NO003', 102, 'PAID', 299.00, NOW())
ON DUPLICATE KEY UPDATE 
    status = VALUES(status),
    total_amount = VALUES(total_amount);

-- 使用INSERT IGNORE避免唯一键冲突中断批量插入
INSERT IGNORE INTO user_tags (user_id, tag) VALUES (100, 'VIP'), (101, 'NEW'), (102, 'VIP');
```

### 3.4 其他SQL改写技巧

```sql
-- ❌ SELECT * 查询不需要的列
SELECT * FROM orders WHERE user_id = 100;
-- ✅ 只查需要的列
SELECT order_id, order_no, total_amount FROM orders WHERE user_id = 100;

-- ❌ COUNT(*) 在大表上慢
SELECT COUNT(*) FROM orders WHERE status = 'PAID';
-- ✅ 使用近似值或维护计数器
-- 方案1：使用估算值
SELECT table_rows FROM information_schema.tables WHERE table_name = 'orders';
-- 方案2：维护状态计数器表
SELECT paid_count FROM order_status_counter WHERE date = '2024-01-15';
-- 方案3：Redis缓存计数

-- ❌ DISTINCT 去重
SELECT DISTINCT user_id FROM orders WHERE status = 'PAID';
-- ✅ GROUP BY 去重（某些场景更高效）
SELECT user_id FROM orders WHERE status = 'PAID' GROUP BY user_id;

-- ❌ 多次查询拼装结果
-- 第一次查询
SELECT * FROM orders WHERE user_id = 100 AND status = 'PAID';
-- 第二次查询
SELECT * FROM orders WHERE user_id = 100 AND status = 'SHIPPED';
-- ✅ 合并为一次查询
SELECT * FROM orders WHERE user_id = 100 AND status IN ('PAID', 'SHIPPED');
```

## 连接池调优

### 4.1 HikariCP连接池配置

HikariCP是目前性能最高的JDBC连接池，Spring Boot 2.x默认使用。

```yaml
# Spring Boot application.yml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/order_db?useUnicode=true&characterEncoding=utf8&useSSL=false&serverTimezone=Asia/Shanghai&rewriteBatchedStatements=true
    username: root
```

### 4.2 连接池大小计算

```java
/**
 * 连接池大小计算公式（来自HikariCP官方文章）：
 * 
 * connections = ((core_count * 2) + effective_spindle_count)
 * 
 * 例如：4核CPU + 1块SSD = (4 * 2) + 1 = 9
 * 
 * 关键理解：
 * 1. 数据库操作大部分时间在等待磁盘IO，不在消耗CPU
 * 2. 连接数过多会导致上下文切换开销，反而降低性能
 * 3. 微服务多实例部署时，总连接数 = 实例数 * 单实例连接数
 *    需确保总连接数 < MySQL max_connections
 * 
 * 测试建议：
 * 1. 使用阶梯压测找到最优连接数
 * 2. 观察CPU使用率和QPS的关系
 * 3. 连接数翻倍不等于性能翻倍，边际递减明显
 */
```

## 实战案例分析

### 案例：订单列表接口慢查询优化

**问题现象：** 订单列表接口响应时间从200ms逐渐增长到3s，数据库CPU使用率达90%。

**排查过程：**

```sql
-- 1. 查看慢查询日志，发现以下SQL执行频繁且耗时
# Time: 2024-01-15T10:30:00
# Query_time: 3.254  Lock_time: 0.001  Rows_sent: 20  Rows_examined: 1850000
SELECT * FROM orders 
WHERE user_id = 10086 
  AND status IN ('PAID', 'SHIPPED') 
  AND create_time > '2023-06-01'
ORDER BY create_time DESC 
LIMIT 20;

-- 2. EXPLAIN分析
EXPLAIN SELECT * FROM orders 
WHERE user_id = 10086 
  AND status IN ('PAID', 'SHIPPED') 
  AND create_time > '2023-06-01'
ORDER BY create_time DESC 
LIMIT 20;

-- 结果：
-- type: ALL（全表扫描）
-- key: NULL（未使用索引）
-- rows: 1850000（扫描185万行）
-- Extra: Using where; Using filesort（使用了文件排序）
```

**优化方案：**

```sql
-- 优化1：创建联合索引
-- 分析查询条件：user_id（等值）+ status（范围）+ create_time（范围+排序）
-- 最优索引：user_id在前（等值查询），create_time在后（排序优化）
ALTER TABLE orders ADD INDEX idx_user_status_time (user_id, status, create_time);

-- 优化后EXPLAIN：
-- type: ref（索引查找）
-- key: idx_user_status_time
-- rows: 50（只扫描50行）
-- Extra: Using index condition（索引下推生效）

-- 优化2：覆盖索引避免回表
-- 将查询改为只取order_id，再关联查询
SELECT o.* FROM orders o
INNER JOIN (
    SELECT order_id FROM orders 
    WHERE user_id = 10086 
      AND status IN ('PAID', 'SHIPPED') 
      AND create_time > '2023-06-01'
    ORDER BY create_time DESC 
    LIMIT 20
) t ON o.order_id = t.order_id;

-- 优化3：进一步优化索引，将status放入覆盖索引
ALTER TABLE orders DROP INDEX idx_user_status_time;
ALTER TABLE orders ADD INDEX idx_user_status_time_covering (user_id, status, create_time, order_id, total_amount);
-- 这样主查询也能走覆盖索引

-- 优化后效果：
-- 响应时间：3250ms → 15ms（提升217倍）
-- 扫描行数：1,850,000 → 20
-- 数据库CPU使用率：90% → 15%
-- Using filesort → 消除
```

### 问题1：索引建了但没生效

```sql
-- 常见原因排查清单：

-- 1. 隐式类型转换
EXPLAIN SELECT * FROM orders WHERE order_no = 12345;  -- order_no是VARCHAR
-- 解决：确保参数类型与字段类型一致

-- 2. 函数包裹索引列
EXPLAIN SELECT * FROM orders WHERE YEAR(create_time) = 2024;
-- 解决：改写为范围查询
EXPLAIN SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';

-- 3. OR条件中有无索引列
EXPLAIN SELECT * FROM orders WHERE user_id = 100 OR remark LIKE '%紧急%';
-- 解决：确保OR两侧都有索引，或改用UNION

-- 4. 优化器认为全表扫描更快（小表或索引区分度低）
-- 查看优化器决策
EXPLAIN FORMAT=JSON SELECT * FROM small_table WHERE indexed_col = 1;
-- 如果表很小（< 几百行），优化器可能选择全表扫描，这是正确行为

-- 5. 使用FORCE INDEX强制走索引（谨慎使用）
SELECT * FROM orders FORCE INDEX(idx_user_id) WHERE user_id = 100;
```

### 问题2：索引过多导致写入变慢

```sql
-- 每个索引都会增加写入开销（INSERT/UPDATE/DELETE需要维护索引）
-- 评估索引数量是否合理
SELECT 
    table_schema AS db,
    table_name,
    COUNT(*) AS index_count,
    GROUP_CONCAT(index_name) AS index_names
FROM information_schema.statistics
WHERE table_schema = 'order_db'
GROUP BY table_schema, table_name
ORDER BY index_count DESC;
-- 建议：单表索引数不超过5~7个

-- 索引冗余检测
-- 如果存在 (a, b, c) 索引，则 (a) 和 (a, b) 是冗余的
SELECT 
    a.table_schema,
    a.table_name,
    a.index_name AS redundant_index,
    b.index_name AS covering_index,
    GROUP_CONCAT(a.column_name ORDER BY a.seq_in_index) AS redundant_cols
FROM information_schema.statistics a
JOIN information_schema.statistics b 
    ON a.table_schema = b.table_schema 
    AND a.table_name = b.table_name
    AND a.index_name != b.index_name
    AND a.seq_in_index <= b.seq_in_index
    AND a.column_name = b.column_name
WHERE a.table_schema = 'order_db'
GROUP BY a.table_schema, a.table_name, a.index_name, b.index_name
HAVING COUNT(*) = (
    SELECT COUNT(*) FROM information_schema.statistics c 
    WHERE c.table_schema = a.table_schema 
    AND c.table_name = a.table_name 
    AND c.index_name = a.index_name
);
```
