## 1. 慢SQL定位

### 1.1 开启慢查询日志

```sql
-- 查看慢查询日志状态
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
SET GLOBAL long_query_time = 1;  -- 超过1秒记录

-- 永久生效（my.cnf）
[mysqld]
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1
log_queries_not_using_indexes = 1  -- 记录未使用索引的查询
```

### 1.2 EXPLAIN执行计划

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 10086;
```

| 列名 | 说明 | 关注点 |
|------|------|--------|
| id | 查询序号 | 越大越先执行 |
| select_type | 查询类型 | SIMPLE最好 |
| table | 表名 | |
| type | 访问类型 | **重点：至少达到range** |
| possible_keys | 可能用到的索引 | |
| key | 实际使用的索引 | **NULL表示未用索引** |
| key_len | 索引使用长度 | 判断联合索引用了几个字段 |
| rows | 扫描行数 | **越小越好** |
| Extra | 额外信息 | **Using filesort/temporary需要优化** |

```
type访问类型（从好到差）：
system > const > eq_ref > ref > range > index > ALL
  │        │        │       │       │       │      │
  │        │        │       │       │       │      └─ 全表扫描（最差）
  │        │        │       │       │       └─ 全索引扫描
  │        │        │       │       └─ 范围扫描（BETWEEN/IN/>/<）
  │        │        │       └─ 非唯一索引等值查找
  │        │        └─ 唯一索引等值查找（JOIN）
  │        └─ 主键/唯一索引等值查找
  └─ 表只有一行

Extra关键信息：
Using index        — 覆盖索引（好）
Using where        — 使用WHERE过滤
Using temporary    — 使用临时表（差，需优化）
Using filesort     — 文件排序（差，需优化）
Using join buffer  — 使用连接缓冲区（差，需优化）
```

### 1.3 EXPLAIN实战分析

```sql
-- ❌ 差：全表扫描
EXPLAIN SELECT * FROM orders WHERE order_no = 'ORD20240101001';
-- type: ALL, key: NULL, rows: 1000000

-- 添加索引
ALTER TABLE orders ADD INDEX idx_order_no (order_no);

-- ✅ 好：使用索引
EXPLAIN SELECT * FROM orders WHERE order_no = 'ORD20240101001';
-- type: ref, key: idx_order_no, rows: 1

-- ❌ 差：索引失效（函数操作）
EXPLAIN SELECT * FROM orders WHERE DATE(create_time) = '2024-01-01';
-- type: ALL, key: NULL

-- ✅ 好：改写为范围查询
EXPLAIN SELECT * FROM orders 
WHERE create_time >= '2024-01-01 00:00:00' 
  AND create_time < '2024-01-02 00:00:00';
-- type: range, key: idx_create_time
```

## 2. 索引优化策略

### 2.1 索引类型选择

| 索引类型 | 适用场景 | 示例 |
|----------|----------|------|
| 主键索引 | 唯一标识 | id BIGINT PRIMARY KEY |
| 唯一索引 | 唯一约束 | order_no UNIQUE |
| 普通索引 | 等值/范围查询 | user_id |
| 联合索引 | 多条件查询 | (user_id, status, create_time) |
| 覆盖索引 | 避免回表 | 查询字段全在索引中 |
| 前缀索引 | 长字符串 | url(20) |

### 2.2 联合索引与最左前缀原则

```sql
-- 联合索引 (a, b, c)
ALTER TABLE t ADD INDEX idx_abc (a, b, c);

-- B+树索引结构：
-- a有序 → a相同时b有序 → a,b相同时c有序

-- ✅ 能用到索引：
WHERE a = 1                     -- 用到a
WHERE a = 1 AND b = 2           -- 用到a, b
WHERE a = 1 AND b = 2 AND c = 3 -- 用到a, b, c
WHERE a = 1 AND b > 2           -- 用到a, b（b范围扫描）
WHERE a = 1 ORDER BY b          -- 用到a，b用于排序

-- ❌ 不能用到索引（或部分失效）：
WHERE b = 2                     -- 跳过a，索引失效
WHERE b = 2 AND c = 3           -- 跳过a，索引失效
WHERE a = 1 AND c = 3           -- 只用到a，c无法使用
WHERE a > 1 AND b = 2           -- a范围查询，b无法使用索引排序
```

### 2.3 覆盖索引

```sql
-- ❌ 回表查询（需要两次IO）
SELECT * FROM orders WHERE user_id = 10086;
-- 1. 在idx_user_id中找到主键id
-- 2. 回表到聚簇索引找完整数据

-- ✅ 覆盖索引（一次IO，Extra显示Using index）
SELECT user_id, order_no, status FROM orders WHERE user_id = 10086;
-- 需要索引：idx_user_id_order_no_status (user_id, order_no, status)

-- Extra: Using index 表示覆盖索引
EXPLAIN SELECT user_id, order_no FROM orders WHERE user_id = 10086;
```

### 2.4 索引下推ICP（Index Condition Pushdown）

```sql
-- MySQL 5.6+特性
-- 联合索引 (name, age)
SELECT * FROM users WHERE name LIKE '张%' AND age > 20;

-- 无ICP（5.6之前）：
-- 1. 在索引中找到所有name LIKE '张%'的记录（可能1000条）
-- 2. 逐条回表查age判断 → 1000次回表

-- 有ICP（5.6之后）：
-- 1. 在索引中找到name LIKE '张%'的记录
-- 2. 在索引层直接过滤age > 20（可能只剩100条）
-- 3. 只对100条回表 → 100次回表
```

### 2.5 索引失效场景汇总

```sql
-- 1. 函数操作
WHERE YEAR(create_time) = 2024          -- ❌ 失效
WHERE DATE(create_time) = '2024-01-01'  -- ❌ 失效
-- ✅ 改写：WHERE create_time >= '2024-01-01' AND create_time < '2024-01-02'

-- 2. 隐式类型转换
WHERE order_no = 12345          -- ❌ 失效（order_no是varchar）
-- ✅ 改写：WHERE order_no = '12345'

-- 3. 前导模糊查询
WHERE name LIKE '%张三'         -- ❌ 失效
WHERE name LIKE '%张三%'        -- ❌ 失效
-- ✅ 改写：WHERE name LIKE '张三%'  -- 只有右模糊能用索引

-- 4. OR条件（一侧无索引则全失效）
WHERE user_id = 1 OR order_no = 'ORD001'  -- 两侧都有索引才走index_merge

-- 5. != / NOT IN / NOT EXISTS
WHERE status != 1               -- 通常全表扫描
-- ✅ 改写：WHERE status IN (0, 2, 3)

-- 6. IS NOT NULL（某些场景）
WHERE delete_time IS NOT NULL   -- 可能失效

-- 7. 计算
WHERE id + 1 = 100              -- ❌ 失效
-- ✅ 改写：WHERE id = 99

-- 8. 联合索引跳过左列
-- 索引(a, b, c)：WHERE b = 1  -- ❌ 失效
```

## 3. SQL改写技巧

### 3.1 JOIN vs 子查询

```sql
-- ❌ 子查询（通常慢，产生临时表）
SELECT * FROM orders 
WHERE user_id IN (SELECT id FROM users WHERE level > 5);

-- ✅ JOIN（通常快）
SELECT o.* FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE u.level > 5;
```

### 3.2 分页优化

```sql
-- ❌ 深度分页（LIMIT越大越慢）
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;
-- 扫描1000010行，丢弃前1000000行

-- ✅ 方案1：子查询优化（先查ID再JOIN）
SELECT * FROM orders o
INNER JOIN (SELECT id FROM orders ORDER BY id LIMIT 1000000, 10) t
ON o.id = t.id;

-- ✅ 方案2：游标分页（记住上一页最大ID）
SELECT * FROM orders WHERE id > 1000000 ORDER BY id LIMIT 10;
-- 直接定位，O(1)

-- ✅ 方案3：游标分页（带排序字段）
-- 假设按create_time DESC分页，上一页最后一条: id=12345, create_time='2024-01-01'
SELECT * FROM orders 
WHERE (create_time < '2024-01-01') OR (create_time = '2024-01-01' AND id < 12345)
ORDER BY create_time DESC, id DESC 
LIMIT 10;
```

### 3.3 批量操作

```sql
-- ❌ 循环单条插入（N次网络IO）
-- Java循环执行：INSERT INTO orders (...) VALUES (...);  ×1000次

-- ✅ 批量插入（1次网络IO）
INSERT INTO orders (order_no, user_id, amount) VALUES
('ORD001', 1001, 100.00),
('ORD002', 1002, 200.00),
...
('ORD1000', 1003, 300.00);

-- MyBatis批量插入
<insert id="batchInsert">
    INSERT INTO orders (order_no, user_id, amount)
    VALUES
    <foreach collection="list" item="item" separator=",">
        (#{item.orderNo}, #{item.userId}, #{item.amount})
    </foreach>
</insert>

-- 大批量分批（每批1000）
-- batch_size = 1000
-- 避免单条SQL过大、undo log过大
```

### 3.4 COUNT优化

```sql
-- ❌ COUNT(*) 全表扫描
SELECT COUNT(*) FROM orders WHERE status = 1;

-- ✅ 方案1：汇总表（实时维护）
SELECT count FROM order_stats WHERE status = 1;

-- ✅ 方案2：Redis计数器
-- 写入订单时 INCR order:status:1:count

-- ✅ 方案3：近似值（不需要精确）
SELECT COUNT(*) FROM orders WHERE status = 1; -- 用EXPLAIN估算
EXPLAIN SELECT * FROM orders WHERE status = 1; -- rows字段估算

-- ✅ 方案4：分时段统计
SELECT DATE(create_time) AS dt, COUNT(*) AS cnt 
FROM orders 
WHERE create_time >= '2024-01-01'
GROUP BY DATE(create_time);
```

## 4. 连接池调优

### 4.1 HikariCP配置

```yaml
spring:
  datasource:
    hikari:
      # 连接池大小（公式：poolSize = ((核心数 * 2) + 有效磁盘数)）
      # 或参考：connections = ((core_count * 2) + effective_spindle_count)
      maximum-pool-size: 20       # 最大连接数
      minimum-idle: 10             # 最小空闲连接（建议=maximum-pool-size）
      
      # 超时配置
      connection-timeout: 30000    # 获取连接超时30s
      idle-timeout: 600000         # 空闲连接超时10min
      max-lifetime: 1800000        # 连接最大存活30min
      validation-timeout: 5000     # 连接验证超时5s
      
      # 连接验证
      connection-test-query: SELECT 1  # 连接有效性检查SQL
      # HikariCP推荐使用JDBC4的isValid()方法，不需要test-query
      
      # 泄漏检测
      leak-detection-threshold: 60000  # 60s未归还则记录泄漏日志
```

### 4.2 连接池大小计算

```
公式（PostgreSQL Wiki）：
  optimal_pool_size = ( (核心数 × 2) + 有效磁盘数 )

示例：
  4核CPU + 1块SSD = (4 × 2) + 1 = 9个连接
  8核CPU + 1块SSD = (8 × 2) + 1 = 17个连接

注意事项：
1. 连接数不是越多越好
2. 连接过多反而增加数据库上下文切换开销
3. 公式适用于CPU密集型，IO密集型可适当增大
4. 微服务多实例时需考虑：DB连接上限 / 实例数 = 单实例pool_size

数据库端配置：
  SHOW VARIABLES LIKE 'max_connections';  -- MySQL默认151
  SET GLOBAL max_connections = 500;
```

## 5. MySQL参数调优

### 5.1 InnoDB核心参数

```ini
[mysqld]
# === 内存相关 ===
# InnoDB缓冲池大小（最关键参数，建议物理内存的50-70%）
innodb_buffer_pool_size = 4G

# 缓冲池实例数（每个实例至少1G，4G内存建议4个实例）
innodb_buffer_pool_instances = 4

# === 日志相关 ===
# Redo Log大小（影响crash恢复时间和写入性能）
innodb_log_file_size = 1G

# Redo Log缓冲区
innodb_log_buffer_size = 16M

# === 刷盘策略 ===
# 0=每秒刷（性能高，可能丢1秒数据）
# 1=每次提交刷（默认，最安全）
# 2=每次提交写OS buffer，每秒刷盘（折中）
innodb_flush_log_at_trx_commit = 1

# === IO相关 ===
# 异步IO（建议开启）
innodb_use_native_aio = 1

# 读IO线程数
innodb_read_io_threads = 8

# 写IO线程数  
innodb_write_io_threads = 8

# === 并发相关 ===
# 并发线程限制（0=不限制）
innodb_thread_concurrency = 0

# === 其他 ===
# 独立表空间（每个表独立.ibd文件）
innodb_file_per_table = 1

# 隔离级别（READ-COMMITTED减少锁竞争）
transaction-isolation = READ-COMMITTED
```

### 5.2 内存分配建议

```
服务器内存：16G
├── InnoDB Buffer Pool：8G（50%）
│   └── 缓存数据页+索引页（最关键）
├── 连接缓冲区：2G
│   └── 每个连接 sort_buffer + join_buffer + read_buffer
├── Redo Log Buffer：64M
├── 查询缓存：0（MySQL 8.0已移除，不建议使用）
├── 临时表：1G
└── OS + MySQL进程：5G
```

## 6. 读写分离与分库分表

### 6.1 读写分离

```
读写分离架构：
              ┌──────────┐
              │ Application│
              └─────┬─────┘
                    │
            ┌───────┴───────┐
            │  Proxy/ShardingSphere
            └───────┬───────┘
                    │
         ┌──────────┼──────────┐
         ▼          ▼          ▼
    ┌────────┐ ┌────────┐ ┌────────┐
    │ Master │ │ Slave1 │ │ Slave2 │
    │ (写)   │ │ (读)   │ │ (读)   │
    └────────┘ └────────┘ └────────┘
         │          ▲          ▲
         └──────────┴──────────┘
              主从复制
```

### 6.2 分库分表策略

```
分库分表时机：
- 单表数据量 > 1000万
- 单库数据量 > 100G
- 单表QPS > 5000
- 响应时间明显下降

分片策略：
┌──────────────────────────────────────────┐
│ 垂直分库：按业务拆分                         │
│   user_db    → 用户相关表                    │
│   order_db   → 订单相关表                    │
│   product_db → 商品相关表                    │
├──────────────────────────────────────────┤
│ 垂直分表：按字段拆分                         │
│   orders (热字段) → id, user_id, amount     │
│   orders_detail (冷字段) → id, description  │
├──────────────────────────────────────────┤
│ 水平分表：按行拆分                           │
│   orders_0 → user_id % 4 == 0             │
│   orders_1 → user_id % 4 == 1             │
│   orders_2 → user_id % 4 == 2             │
│   orders_3 → user_id % 4 == 3             │
├──────────────────────────────────────────┤
│ 分片键选择：                                │
│   取模：user_id % N（均匀，但扩容麻烦）       │
│   范围：0-1000万→表0, 1000万-2000万→表1     │
│   一致性Hash：扩容迁移最少                   │
└──────────────────────────────────────────┘
```

## 7. 实战案例分析

### 案例：订单查询从3秒优化到50ms

```sql
-- 原始SQL（3.2秒）
SELECT * FROM orders o
LEFT JOIN users u ON o.user_id = u.id
LEFT JOIN products p ON o.product_id = p.id
WHERE o.status = 1 
  AND o.create_time > '2024-01-01'
ORDER BY o.create_time DESC
LIMIT 20;

-- EXPLAIN分析：
-- 1. type=ALL（orders全表扫描）
-- 2. key=NULL（未用索引）
-- 3. Extra=Using filesort（文件排序）
-- 4. rows=5000000（扫描500万行）

-- 优化步骤1：加索引
ALTER TABLE orders ADD INDEX idx_status_create (status, create_time);
-- 复合索引，覆盖status过滤+create_time排序

-- 优化步骤2：避免SELECT *（覆盖索引）
SELECT o.id, o.order_no, o.amount, o.create_time,
       u.name AS user_name, p.name AS product_name
FROM orders o
LEFT JOIN users u ON o.user_id = u.id
LEFT JOIN products p ON o.product_id = p.id
WHERE o.status = 1 
  AND o.create_time > '2024-01-01'
ORDER BY o.create_time DESC
LIMIT 20;
-- 优化后：type=ref, key=idx_status_create, rows=200, Extra=Using index
-- 耗时：50ms
```

## 8. 面试题速查

**Q1: 如何定位慢SQL？**
```
1. 开启慢查询日志（slow_query_log）
2. 设置阈值（long_query_time=1s）
3. 使用EXPLAIN分析执行计划
4. 关注type、key、rows、Extra字段
5. 使用pt-query-digest分析慢日志Top SQL
```

**Q2: 索引失效的常见场景？**
```
1. 函数操作：WHERE YEAR(date) = 2024
2. 隐式类型转换：WHERE varchar_col = 123
3. 前导模糊：LIKE '%abc'
4. OR条件一侧无索引
5. 联合索引跳过左列
6. != / NOT IN
7. 计算：WHERE id + 1 = 100
```

**Q3: 联合索引最左前缀原则？**
```
索引(a, b, c)：
- a → 用到索引
- a, b → 用到a, b
- a, b, c → 全部用到
- b, c → 索引失效
- a, c → 只用到a
- a > 1, b → a范围查询后b失效
```

**Q4: 如何优化深度分页？**
```
方案1：子查询先取ID再JOIN
方案2：游标分页（WHERE id > last_id LIMIT 10）
方案3：搜索引擎（ES处理分页）
```

**Q5: HikariCP连接池大小如何设置？**
```
公式：pool_size = (CPU核心数 × 2) + 有效磁盘数
原则：不是越多越好，过多增加DB上下文切换
注意：微服务多实例时 DB总连接数 / 实例数 = 单实例pool_size
```
