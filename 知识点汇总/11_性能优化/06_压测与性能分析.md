## 1. 压测工具对比

| 工具 | 语言 | 特点 | 适用场景 |
|------|------|------|----------|
| JMeter | Java | GUI+CLI，功能全面，插件丰富 | HTTP/API/DB压测 |
| wrk | C | 轻量极速，Lua扩展 | 快速HTTP压测 |
| wrk2 | C | 支持精确延迟统计 | P99/P999精确测量 |
| Gatling | Scala | DSL脚本，报告精美 | CI/CD集成压测 |
| k6 | Go | JavaScript脚本，云原生 | 微服务/云原生压测 |
| Locust | Python | 代码即脚本，分布式 | 灵定场景定制 |
| hey | Go | 单文件，极简 | 快速验证 |

```
选型建议：
├── 快速验证 → hey / wrk（一行命令）
├── 正式压测 → JMeter（功能全）或 k6（脚本灵活）
├── CI/CD集成 → k6 / Gatling（脚本驱动）
├── 分布式压测 → JMeter分布式 / Locust
└── 精确延迟 → wrk2（修正了协调遗漏效应）
```

## 2. JMeter实战

### 2.1 核心概念

```
JMeter测试计划结构：
Test Plan（测试计划）
├── Thread Group（线程组）
│   ├── HTTP Request（取样器）
│   ├── HTTP Header Manager（配置元件）
│   ├── CSV Data Set Config（参数化）
│   ├── Timer（定时器）
│   ├── Assertion（断言）
│   └── Listener（监听器）
│       ├── Summary Report
│       ├── Aggregate Report
│       └── View Results Tree
└── setUp/tearDown Thread Group
```

### 2.2 CLI模式压测（推荐）

```bash
# GUI模式仅用于调试，正式压测用CLI模式
jmeter -n -t test_plan.jmx -l result.jtl -e -o ./report

# 参数说明：
# -n  非GUI模式
# -t  测试计划文件
# -l  结果文件
# -e  生成HTML报告
# -o  报告输出目录

# 示例
jmeter -n -t order_api.jmx -l result.jtl -e -o ./report -Jthreads=100 -Jrampup=10 -Jloop=100
```

### 2.3 动态参数化

```bash
# CSV数据文件
# users.csv:
# 1001,password1
# 1002,password2
# 1003,password3

# JMeter中配置CSV Data Set Config
# Filename: users.csv
# Variable Names: username,password
# Delimiter: ,
# Recycle on EOF: True
```

### 2.4 常用断言

```
Response Code断言：
  Pattern: 200
  → 响应码不是200则标记失败

JSON断言（JSON Path）：
  Assert JSON Path exists: $.code
  Additionally assert value: 200
  → 检查响应JSON中code字段是否为200

Response Time断言：
  Pattern: < 500
  → 响应时间超过500ms则失败
```

### 3.2 Lua脚本扩展


## 4. 性能指标解读

### 4.1 核心指标

```
┌─────────────────────────────────────────────┐
│  QPS = 并发数 / 平均RT                       │
│  并发数 = QPS × 平均RT                       │
│  RT = 1 / QPS × 并发数                       │
└─────────────────────────────────────────────┘

示例：
  100并发 × 50ms RT = 2000 QPS
  200并发 × 50ms RT = 4000 QPS
  100并发 × 200ms RT = 500 QPS
```

### 4.2 分位值分析

```
延迟分布：
  Avg（平均）：容易被长尾请求拉高，不推荐单独使用
  P50（中位数）：50%请求的响应时间
  P90：90%请求的响应时间
  P99：99%请求的响应时间 ← 核心指标
  P999：99.9%请求的响应时间

示例（10000次请求）：
  P50 = 20ms  → 5000次请求在20ms内完成
  P99 = 100ms → 9900次请求在100ms内完成
  P999 = 500ms → 9990次请求在500ms内完成
  Max = 2000ms → 最慢的一次花了2秒

分析：
  P50和P99差距大 → 有长尾请求，可能是GC或锁竞争
  P99和Max差距大 → 偶发极端延迟，可能是网络抖动
```

### 4.3 错误率

```
错误率 = 失败请求数 / 总请求数

错误类型：
  HTTP 4xx → 客户端错误（参数错误、权限不足）
  HTTP 5xx → 服务端错误（代码异常、资源不足）
  超时 → 请求超时（连接超时/读取超时）
  连接拒绝 → 服务不可用（线程池满、连接池满）

目标：错误率 < 0.1%
```

## 5. 压测策略

### 5.1 阶梯加压

```
逐步增加并发，找出系统拐点：

  并发数
  500 ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
                    ┌─── 拐点：RT开始上升
  400 ─ ─ ─ ─ ─ ─ ─┤ ─ ─ ─ ─ ─ ─
                    │
  300 ─ ─ ─ ─ ─ ─ ─┤ ─ ─ ─ ─ ─ ─
                    │
  200 ─ ─ ─ ─ ─ ─ ─┤ ─ ─ ─ ─ ─ ─
                    │
  100 ─ ─ ─ ─ ─ ─ ─┤ ─ ─ ─ ─ ─ ─
        ─────────────────────────→ 时间
        0s   30s   60s   90s  120s

目的：找到最大QPS和对应并发数
```

```bash
# JMeter阶梯加压插件：Stepping Thread Group
# 或使用命令行参数动态调整
# 0-30s: 100并发
# 30-60s: 200并发

# 90-120s: 400并发
```

### 5.2 恒压测试

```
固定并发数，持续运行较长时间，观察稳定性：

  并发数
  300 ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
       │                               │
       │                               │
       │                               │
  ─────┼───────────────────────────────┼──→ 时间
       0s                            30min

目的：检测内存泄漏、连接泄漏、GC退化
```

## 6. 性能瓶颈分析

### 6.1 瓶颈分析框架

```
压测结果异常 →
├── RT高（P99 > 目标值）
│   ├── CPU使用率高？→ 代码计算密集，profiler分析热点
│   ├── 内存/GC频繁？→ jstat分析GC，dump分析对象
│   ├── 磁盘IO高？→ iostat分析，可能是日志写盘
│   ├── 网络带宽满？→ iftop/nethogs分析
│   ├── DB慢查询？→ 慢查询日志 + EXPLAIN
│   ├── 连接池等待？→ 增大连接池或优化SQL
│   └── 线程阻塞？→ jstack分析线程状态
├── 错误率高
│   ├── 502/503 → 服务不可用（OOM/线程池满）
│   ├── 超时 → RT过高导致超时
│   ├── 连接拒绝 → 连接池满/backlog满
│   └── 429 → 限流触发
└── QPS上不去
    ├── CPU满 → 增加实例或优化代码
    ├── 线程数不够 → 增大线程池
    ├── DB连接池满 → 增大连接池
    └── 外部依赖瓶颈 → 排查下游服务
```

### 6.2 系统监控命令

```bash
# CPU
top -H -p <pid>          # 查看线程CPU使用
mpstat 1                  # CPU各核心使用率

# 内存
free -m                   # 内存使用概况
jmap -heap <pid>          # JVM堆内存

# 磁盘IO
iostat -x 1               # 磁盘IO详情
iotop                     # 哪个进程在读写磁盘

# 网络
iftop                     # 网络流量
netstat -an | grep TIME_WAIT | wc -l  # TIME_WAIT连接数
ss -s                     # socket统计

# 线程
jstack <pid> | grep -c "BLOCKED"  # 阻塞线程数
jstack <pid> > thread_dump.txt   # 线程转储

# GC
jstat -gcutil <pid> 1000  # 每秒GC统计
```

## 7. 分布式压测

### 7.1 JMeter分布式架构

```
              ┌──────────────┐
              │  Master节点   │ ← 控制机
              │  (JMeter GUI) │
              └──────┬───────┘
                     │
     ┌───────────────┼───────────────┐
     ▼               ▼               ▼
┌─────────┐    ┌─────────┐    ┌─────────┐
│ Slave1  │    │ Slave2  │    │ Slave3  │ ← 执行机
│ 1000并发 │    │ 1000并发 │    │ 1000并发 │
└─────────┘    └─────────┘    └─────────┘

总并发 = Slave数 × 单机并发数
3台Slave × 1000并发 = 3000总并发
```

```bash
# Slave节点启动
jmeter-server -Djava.rmi.server.hostname=192.168.1.101

# Master节点配置（jmeter.properties）
remote_hosts=192.168.1.101,192.168.1.102,192.168.1.103

# Master启动分布式压测
jmeter -n -t test.jmx -r -l result.jtl -e -o ./report
# -r  启动所有远程节点
```

## 8. 实战案例：秒杀系统压测

### 8.1 压测目标

```
场景：1000个商品，10万人参与秒杀
目标：
  QPS ≥ 10000
  P99 RT ≤ 200ms
  错误率 ≤ 0.1%
  超卖 = 0
```

### 8.2 压测脚本

```lua
-- seckill.lua
local url = "/seckill/" .. tonumber(ARGV[1]) .. "/execute"
local userId = math.random(1, 100000)
local productId = math.random(1, 1000)

wrk.method = "POST"
wrk.body = string.format('{"userId":%d,"productId":%d}', userId, productId)
wrk.headers["Content-Type"] = "application/json"
wrk.headers["Authorization"] = "Bearer " .. generateToken(userId)

request = function()
    return wrk.format("POST", url)
end
```

### 8.3 压测执行

```bash
# 第一轮：阶梯加压（找出拐点）
# 1000 → 2000 → 5000 → 8000 → 10000并发
wrk -t16 -c1000 -d60s -s seckill.lua http://seckill-api:8080
wrk -t16 -c2000 -d60s -s seckill.lua http://seckill-api:8080
wrk -t16 -c5000 -d60s -s seckill.lua http://seckill-api:8080
wrk -t16 -c8000 -d60s -s seckill.lua http://seckill-api:8080
wrk -t16 -c10000 -d60s -s seckill.lua http://seckill-api:8080

# 第二轮：恒压测试（稳定性验证）
wrk -t16 -c5000 -d30m -s seckill.lua http://seckill-api:8080

# 第三轮：脉冲测试（突发流量）
wrk -t16 -c20000 -d10s -s seckill.lua http://seckill-api:8080
```

### 8.4 压测结果分析

```
第一轮阶梯加压结果：
┌──────────┬────────┬──────────┬──────────┬────────┐
│ 并发数   │ QPS    │ P50 RT   │ P99 RT   │ 错误率 │
├──────────┼────────┼──────────┼──────────┼────────┤
│ 1,000    │ 8,200  │ 35ms     │ 85ms     │ 0.00%  │
│ 2,000    │ 12,500 │ 55ms     │ 120ms    │ 0.00%  │
│ 5,000    │ 15,800 │ 120ms    │ 280ms    │ 0.05%  │ ← 拐点
│ 8,000    │ 14,200 │ 250ms    │ 850ms    │ 1.20%  │ ← 退化
│ 10,000   │ 11,800 │ 450ms    │ 1500ms   │ 5.80%  │ ← 雪崩
└──────────┴────────┴──────────┴──────────┴────────┘

分析：
1. 拐点在5000并发，QPS=15800，P99=280ms
2. 超过5000后QPS反而下降，RT急剧上升
3. 错误率超过0.1%目标

优化方向：
1. Redis预减库存（减少DB压力）
2. 异步化下单（MQ削峰）
3. 增加限流（Sentinel，5000QPS阈值）
4. 水平扩容（4实例 → 8实例）
```

## 9. 面试题速查

**Q1: 压测的QPS和RT如何计算？**
```
Little's Law：并发数 = QPS × RT
  示例：100并发 × 50ms RT = 2000 QPS
  示例：500 QPS × 200ms RT = 100并发
```

**Q2: 为什么P99比平均值更重要？**
```
平均值会被大量快速请求拉低，掩盖长尾问题
P99反映99%用户的体验，是更严格的SLA指标
示例：10000次请求，平均20ms，但P99=500ms
  → 100个用户体验差（0.5秒）
```

**Q3: 如何找出系统的最大QPS？**
```
阶梯加压法：
1. 从小并发开始（如100）
2. 逐步增加并发（100→200→500→1000）
3. 记录每个级别的QPS和RT
4. QPS不再增长且RT开始急剧上升的点 = 最大QPS
```

**Q4: 压测时CPU 100%怎么排查？**
```
1. top -H -p <pid> 找出CPU高的线程
2. jstack <pid> | grep <thread_hex> 查看线程在做什么
3. 如果是GC线程 → jstat看GC频率，dump分析对象
4. 如果是业务线程 → profiler分析代码热点
5. 如果是序列化 → 检查大对象序列化/反序列化
```

**Q5: 分布式压测为什么要用多台机器？**
```
单机瓶颈：
1. 单机TCP端口数限制（65535 - 保留端口）
2. 单机CPU/网卡带宽限制
3. 单机JMeter自身JVM开销
多机分布式：3台Slave × 1000并发 = 3000并发，突破单机限制
```
