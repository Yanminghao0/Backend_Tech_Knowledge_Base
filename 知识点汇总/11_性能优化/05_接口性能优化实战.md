## 1. 性能指标

| 指标 | 全称 | 含义 | 目标 |
|------|------|------|------|
| RT | Response Time | 响应时间 | P99 < 200ms |
| QPS | Queries Per Second | 每秒请求数 | 根据业务定 |
| TPS | Transactions Per Second | 每秒事务数 | 根据业务定 |
| 并发数 | Concurrency | 同时处理请求数 | QPS × RT |
| 吞吐量 | Throughput | 单位时间处理数据量 | 越高越好 |
| 错误率 | Error Rate | 失败请求占比 | < 0.1% |

```
RT分位值：
  P50：50%的请求在此时长内完成（中位数）
  P90：90%的请求在此时长内完成
  P99：99%的请求在此时长内完成（重点关注）
  P999：99.9%的请求在此时长内完成

经验值：
  P99 < 100ms  — 极速（搜索、推荐）
  P99 < 200ms  — 优秀（核心交易）
  P99 < 500ms  — 可接受（普通查询）
  P99 < 1000ms — 需优化
  P99 > 3000ms — 严重问题
```

## 3. 串行转并行

### 3.1 CompletableFuture并行

```java
// ❌ 串行调用（总耗时 = A + B + C = 600ms）
public OrderDetail getOrderDetail(Long orderId) {
    Order order = orderService.getById(orderId);          // 200ms
    User user = userService.getById(order.getUserId());   // 200ms
    Product product = productService.getById(order.getProductId()); // 200ms
    return new OrderDetail(order, user, product);
}

// ✅ 并行调用（总耗时 = max(A, B, C) = 200ms）
public OrderDetail getOrderDetail(Long orderId) {
    Order order = orderService.getById(orderId);  // 必须先获取order

    CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(
            () -> userService.getById(order.getUserId()));
    CompletableFuture<Product> productFuture = CompletableFuture.supplyAsync(
            () -> productService.getById(order.getProductId()));

    // 等待全部完成
    CompletableFuture.allOf(userFuture, productFuture).join();

    return new OrderDetail(order, userFuture.join(), productFuture.join());
}

// ✅ 带超时的并行
public OrderDetail getOrderDetailWithTimeout(Long orderId) {
    Order order = orderService.getById(orderId);

    CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(
            () -> userService.getById(order.getUserId()))
            .orTimeout(500, TimeUnit.MILLISECONDS)          // 单个超时500ms
            .exceptionally(ex -> User.defaultUser());        // 降级

    CompletableFuture<Product> productFuture = CompletableFuture.supplyAsync(
            () -> productService.getById(order.getProductId()))
            .orTimeout(500, TimeUnit.MILLISECONDS)
            .exceptionally(ex -> Product.defaultProduct());

    CompletableFuture.allOf(userFuture, productFuture).join();
    return new OrderDetail(order, userFuture.join(), productFuture.join());
}
```

### 3.2 自定义线程池

```java
@Configuration
public class AsyncConfig {

    @Bean("orderThreadPool")
    public ThreadPoolExecutor orderThreadPool() {
        return new ThreadPoolExecutor(
                20,                     // corePoolSize
                50,                     // maxPoolSize
                60, TimeUnit.SECONDS,   // keepAlive
                new LinkedBlockingQueue<>(1000),  // 工作队列
                new ThreadFactoryBuilder()
                        .setNameFormat("order-pool-%d")
                        .build(),
                new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
        );
    }
}

// 使用自定义线程池
@Service
public class OrderService {

    @Autowired
    @Qualifier("orderThreadPool")
    private ThreadPoolExecutor threadPool;

    public OrderDetail getOrderDetail(Long orderId) {
        Order order = orderService.getById(orderId);

        CompletableFuture<User> userFuture = CompletableFuture
                .supplyAsync(() -> userService.getById(order.getUserId()), threadPool);
        CompletableFuture<Product> productFuture = CompletableFuture
                .supplyAsync(() -> productService.getById(order.getProductId()), threadPool);

        CompletableFuture.allOf(userFuture, productFuture).join();
        return new OrderDetail(order, userFuture.join(), productFuture.join());
    }
}
```

### 4.2 MQ异步解耦

```java
// 同步调用第三方接口（慢且不可靠）
public void createOrder(Order order) {
    orderMapper.insert(order);
    inventoryService.deduct(order.getProductId());    // 远程调用
    pointsService.addPoints(order.getUserId());        // 远程调用
    couponService.useCoupon(order.getCouponId());     // 远程调用
}
// 问题：任一远程调用失败导致整体失败

// MQ异步（可靠）
@Service
public class OrderService {

    @Autowired
    private RocketMQTemplate mqTemplate;

    public void createOrder(Order order) {
        orderMapper.insert(order);  // 只做核心操作

        // 发送MQ消息，异步处理
        mqTemplate.asyncSend("order-created", order, ...);
        // 总耗时：DB写入 + MQ发送
    }
}

// 消费者
@RocketMQMessageListener(topic = "order-created", consumerGroup = "order-consumer")
public class OrderEventListener implements RocketMQListener<Order> {
    @Override
    public void onMessage(Order order) {
        inventoryService.deduct(order.getProductId());
        pointsService.addPoints(order.getUserId());
        couponService.useCoupon(order.getCouponId());
    }
}
```

## 5. 缓存策略

### 5.1 多级缓存

```java
// 多级缓存架构
// L1: Caffeine本地缓存（纳秒级）
// L2: Redis分布式缓存（毫秒级）
// L3: 数据库

@Service
public class ProductService {

    @Autowired
    private CacheManager cacheManager;  // Caffeine
    @Autowired
    private RedisTemplate<String, Product> redisTemplate;
    @Autowired
    private ProductMapper productMapper;

    // Caffeine本地缓存（L1）
    @Cacheable(value = "product:local", key = "#id", unless = "#result == null")
    public Product getProduct(Long id) {
        // 先查Redis（L2）
        Product product = redisTemplate.opsForValue().get("product:" + id);
        if (product != null) {
            return product;
        }

        // 再查DB（L3）
        product = productMapper.selectById(id);
        if (product != null) {
            // 回填Redis
            redisTemplate.opsForValue().set("product:" + id, product, 30, TimeUnit.MINUTES);
        }
        return product;
    }
}

// Caffeine配置
@Configuration
public class CacheConfig {

    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager manager = new CaffeineCacheManager();
        manager.setCaffeine(Caffeine.newBuilder()
                .initialCapacity(100)               // 初始容量
                .maximumSize(1000)                  // 最大缓存数
                .expireAfterWrite(5, TimeUnit.MINUTES)  // 写后过期
                .recordStats());                     // 记录统计
        return manager;
    }
}
```

### 5.2 缓存穿透/击穿/雪崩防护

```java
// 缓存穿透：查询不存在的数据
// 解决：空值缓存 + 布隆过滤器
public Product getProduct(Long id) {
    String key = "product:" + id;
    
    // 1. 布隆过滤器快速判断
    if (!bloomFilter.mightContain(id)) {
        return null;  // 一定不存在
    }

    // 2. 查缓存
    Product product = redisTemplate.opsForValue().get(key);
    if (product != null) {
        return product;
    }

    // 3. 查DB
    product = productMapper.selectById(id);
    if (product != null) {
        redisTemplate.opsForValue().set(key, product, 30, TimeUnit.MINUTES);
    } else {
        // 空值缓存（短过期）
        redisTemplate.opsForValue().set(key, "NULL", 5, TimeUnit.MINUTES);
    }
    return product;
}

// 缓存击穿：热点Key过期
// 解决：互斥锁（单线程回源）
public Product getProductWithLock(Long id) {
    String key = "product:" + id;
    Product product = redisTemplate.opsForValue().get(key);
    if (product != null) {
        return product;
    }

    // 获取分布式锁
    String lockKey = "lock:product:" + id;
    try {
        boolean locked = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);
        if (locked) {
            // 双重检查
            product = redisTemplate.opsForValue().get(key);
            if (product != null) return product;

            // 查DB并回填
            product = productMapper.selectById(id);
            redisTemplate.opsForValue().set(key, product, 30, TimeUnit.MINUTES);
            return product;
        } else {
            // 等待重试
            Thread.sleep(50);
            return getProductWithLock(id);
        }
    } finally {
        redisTemplate.delete(lockKey);
    }
}
```

## 6. 数据库层优化

### 6.1 N+1查询问题

```java
// ❌ N+1查询（1次查列表 + N次查关联）
List<Order> orders = orderMapper.selectList(null);  // 1次
for (Order order : orders) {
    User user = userMapper.selectById(order.getUserId());  // N次
    order.setUser(user);
}
// 100条订单 = 101次SQL

// ✅ 批量查询
List<Order> orders = orderMapper.selectList(null);  // 1次
Set<Long> userIds = orders.stream().map(Order::getUserId).collect(Collectors.toSet());
List<User> users = userMapper.selectBatchIds(userIds);  // 1次
Map<Long, User> userMap = users.stream().collect(Collectors.toMap(User::getId, u -> u));
orders.forEach(o -> o.setUser(userMap.get(o.getUserId())));
// 100条订单 = 2次SQL
```

### 6.2 批量操作

```java
// ❌ 循环单条插入
for (Order order : orders) {
    orderMapper.insert(order);  // 1000次SQL
}

// ✅ 批量插入
orderMapper.batchInsert(orders);  // 1次SQL

// MyBatis批量插入
@Insert("<script>" +
    "INSERT INTO orders (order_no, user_id, amount) VALUES " +
    "<foreach collection='list' item='item' separator=','>" +
    "(#{item.orderNo}, #{item.userId}, #{item.amount})" +
    "</foreach>" +
    "</script>")
void batchInsert(@Param("list") List<Order> orders);

// 大批量分批（每批500-1000）
Lists.partition(orders, 500).forEach(orderMapper::batchInsert);
```

```java
        PoolingHttpClientConnectionManager connManager = 
                new PoolingHttpClientConnectionManager();
```

## 8. JVM层优化

### 8.1 GC停顿优化

```bash
# 查看GC情况
jstat -gcutil <pid> 1000 10

# 输出示例：
#  S0     S1     E      O      M     CCS    YGC   YGCT   FGC  FGCT   GCT
#  0.00  45.23  67.89  34.56  95.12  91.34   23   0.234    2  0.567  0.801

# 关注点：
# FGC（Full GC次数）应该接近0
# FGCT（Full GC时间）越短越好
# O（老年代使用率）不应持续增长
```

```bash
# G1 GC调优（JDK 9+默认）
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200          # 目标停顿时间200ms
-XX:G1HeapRegionSize=16m          # Region大小
-XX:InitiatingHeapOccupancyPercent=45  # 堆占用45%触发并发标记

# ZGC（JDK 15+，超低延迟）
-XX:+UseZGC
-XX:SoftMaxHeapSize=4g
# ZGC停顿时间 < 1ms（不管堆大小）
```

## 9. 实战案例：2000ms→50ms

### 9.1 优化前

```java
// 订单详情接口 — 2000ms
@GetMapping("/orders/{id}")
public OrderDetail getOrderDetail(@PathVariable Long id) {
    // 1. 查订单（200ms，无索引）
    Order order = orderMapper.selectById(id);

    // 2. 查用户（300ms，远程调用）
    User user = userService.getUser(order.getUserId());

    // 3. 查商品（300ms，远程调用）
    Product product = productService.getProduct(order.getProductId());

    // 4. 查评论（500ms，N+1查询）
    List<Comment> comments = commentMapper.selectByProductId(order.getProductId());
    for (Comment comment : comments) {
        User commentUser = userMapper.selectById(comment.getUserId());
        comment.setUser(commentUser);
    }

    // 5. 发埋点（200ms，同步HTTP）
    analyticsService.track("view_order", id);

    // 6. 序列化大对象（500ms）
    return buildOrderDetail(order, user, product, comments);
}
// 总耗时：200 + 300 + 300 + 500 + 200 + 500 = 2000ms
```

### 9.2 优化后

```java
// 优化后 — 50ms
@GetMapping("/orders/{id}")
public OrderDetail getOrderDetail(@PathVariable Long id) {
    // 1. 查订单（5ms，加索引）
    Order order = orderMapper.selectById(id);

    // 2+3. 并行查用户和商品（20ms，CompletableFuture）
    CompletableFuture<User> userFuture = CompletableFuture
            .supplyAsync(() -> userService.getUser(order.getUserId()), threadPool)
            .orTimeout(100, TimeUnit.MILLISECONDS)
            .exceptionally(ex -> User.defaultUser());

    CompletableFuture<Product> productFuture = CompletableFuture
            .supplyAsync(() -> productService.getProduct(order.getProductId()), threadPool)
            .orTimeout(100, TimeUnit.MILLISECONDS)
            .exceptionally(ex -> Product.defaultProduct());

    // 4. 查评论（10ms，批量查询+缓存）
    List<Comment> comments = getCommentsWithCache(order.getProductId());

    // 5. 发埋点（异步，0ms不阻塞）
    CompletableFuture.runAsync(() -> analyticsService.track("view_order", id), threadPool);

    // 等待并行结果
    CompletableFuture.allOf(userFuture, productFuture).join();

    // 6. 精简返回字段（15ms）
    return buildOrderDetail(order, userFuture.join(), productFuture.join(), comments);
}
// 总耗时：5 + 20 + 10 + 0 + 15 = 50ms

// 优化措施总结：
// 1. 加索引：200ms → 5ms
// 2. 串行转并行：300+300ms → 20ms
// 3. N+1改批量+缓存：500ms → 10ms
// 4. 异步埋点：200ms → 0ms
// 5. 精简字段：500ms → 15ms
```

## 10. 面试题速查

**Q1: 接口性能优化的常见手段？**
```
1. 串行转并行（CompletableFuture）
2. 异步化（@Async / MQ）
3. 加缓存（多级缓存、Redis）
4. 数据库优化（索引、批量、N+1修复）
5. 减少网络IO（连接池、Gzip）
6. JVM调优（GC停顿）
7. 精简数据（避免大对象序列化）
```

**Q2: 如何定位接口性能瓶颈？**
```
1. Arthas trace命令定位方法级耗时
2. 慢查询日志定位SQL
3. SkyWalking全链路追踪
4. jstat分析GC停顿
5. 按照瓶颈定位决策树逐层排查
```

**Q3: N+1查询问题如何解决？**
```
1. JOIN查询
2. 批量查询：先查列表，再批量查关联，代码中组装
3. MyBatis关联查询（resultMap + collection）
```

**Q4: 缓存穿透/击穿/雪崩的区别和解决方案？**
```
穿透：查询不存在的数据 → 布隆过滤器 + 空值缓存
击穿：热点Key过期 → 互斥锁 + 永不过期
雪崩：大量Key同时过期 → 过期时间加随机值 + 多级缓存
```
