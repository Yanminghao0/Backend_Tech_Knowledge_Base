## 1. Zookeeper整体架构

### 1.1 核心概念

```
Zookeeper集群：

┌──────────┐   ┌──────────┐   ┌──────────┐
│  ZK-1    │   │  ZK-2    │   │  ZK-3    │
│ (Leader) │←→│(Follower)│←→│(Follower)│
│          │   │          │   │          │
│ 处理读写   │   │ 处理读    │   │ 处理读    │

└──────────┘   └──────────┘   └──────────┘
      ↕              ↕              ↕
  ┌──────────────────────────────────────┐
  │         客户端 (Client)               │
  │  Dubbo / Kafka / HBase / 自研应用      │
  └──────────────────────────────────────┘
```

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| ZNode | 数据节点，类似文件系统的文件/目录 |
| zxid | 事务ID（64位：epoch(32) + counter(32)） |
| epoch | 纪元，Leader的任期号 |
| Session | 客户端会话，有心跳保活 |
| Watcher | 数据变更监听器 |
| Quorum | 法定人数，集群中过半节点数 |

## 2. ZAB协议

### 2.1 ZAB概述

```
ZAB（ZooKeeper Atomic Broadcast）协议是ZK的原子广播协议
保证所有副本以相同顺序应用相同事务

两种核心模式：
1. 崩溃恢复模式（Recovery）
   - Leader选举
   - 数据同步
   
2. 消息广播模式（Broadcast）
   - 类似两阶段提交（2PC）
   - Leader广播Proposal给所有Follower
   - 过半Follower ACK后Commit
```

### 2.2 消息广播流程

```
Leader收到写请求后的广播流程：

┌──────────┐                     ┌──────────┐  ┌──────────┐
│  Leader  │                     │Follower 1│  │Follower 2│
│          │   1. PROPOSAL       │          │  │          │
│  生成     │───────────────────→│          │  │          │
│  Proposal│──────────────────────────────────→│          │
│          │                     │          │  │          │
│          │   2. ACK            │          │  │          │
│          │←───────────────────│          │  │          │
│          │←──────────────────────────────────│          │
│          │                     │          │  │          │
│ 过半ACK   │   3. COMMIT        │          │  │          │
│          │───────────────────→│          │  │          │
│          │──────────────────────────────────→│          │
└──────────┘                     └──────────┘  └──────────┘

关键点：
  - 不需要所有Follower ACK，只需过半（Quorum）
  - Leader为每个Proposal分配递增的zxid
  - Follower按zxid顺序应用事务（保证顺序一致）
```

### 2.3 ZAB消息广播源码

```java
// Leader.java
public class Leader {

    private final List<LearnerHandler> learners; // 所有Follower连接

    /**
     * 广播Proposal
     */
    void propose(Request request) throws IOException {
        // 1. 生成Proposal（分配zxid）
        long zxid = zk.getZxid(); // 获取下一个zxid
        QuorumPacket proposal = new QuorumPacket(
            Leader.PROPOSAL, zxid, request.getSerializeData(), null);

        // 2. 写入Leader本地事务日志
        zk.getZKDatabase().getLog().append(proposal);

        // 3. 广播给所有Follower
        for (LearnerHandler fh : learners) {
            fh.queuePacket(proposal); // 异步发送
        }

        // 4. 等待过半ACK
        proposalTracker.addProposal(zxid, proposal);
    }
}

// LearnerHandler——Leader与每个Follower的通信处理器
public class LearnerHandler extends ZooKeeperThread {

    /**
     * 处理Follower的ACK
     */
    void processAck(long zxid, long followerSid) {
        // 1. 记录ACK
        proposalTracker.addAck(zxid, followerSid);

        // 2. 检查是否过半
        if (proposalTracker.hasQuorum(zxid)) {
            // 3. 过半——发送COMMIT
            QuorumPacket commit = new QuorumPacket(
                Leader.COMMIT, zxid, null, null);

            for (LearnerHandler fh : leader.getLearners()) {
                fh.queuePacket(commit);
            }

            // 4. Leader本地也Commit
            leader.zk.commit(zxid);
        }
    }
}
```

### 2.4 zxid结构

```
zxid是64位整数：epoch(高32位) + counter(低32位)

┌────────────────┬────────────────┐
│  epoch (32位)   │  counter (32位) │
└────────────────┴────────────────┘

epoch：Leader的纪元号，每次Leader选举后+1
counter：Leader内递增的事务序号

zxid的大小关系保证：
1. 同一epoch内，zxid越大事务越新
2. 不同epoch，epoch大的zxid更大（新Leader的事务更"新"）

zxid的作用：
1. 标识事务的全局顺序
2. Leader选举时比较zxid决定谁的数据最新
3. 数据同步时用zxid确定差异
```

## 3. Leader选举

### 3.1 选举触发条件

```
触发Leader选举的场景：
1. 集群启动（初始选举）
2. Leader崩溃
3. Follower与Leader失去联系（超时）
4. 手动触发

选举算法：FastLeaderElection（快速选举算法）
```

### 3.2 选举参数

```java
public class FastLeaderElection {

    // 选举中的投票信息
    static public class Notification {
        long leader;           // 被推荐为Leader的服务器ID
        long zxid;             // 被推荐者的最大zxid
        long electionEpoch;    // 选举轮次
        QuorumPeer.ServerState state; // 发送者的状态
        long sid;              // 发送者的服务器ID
    }

    // 投票
    static public class Vote {
        final long id;         // 被投票的服务器ID
        final long zxid;       // 被投票者的zxid
        final long electionEpoch; // 选举轮次
        final long peerEpoch;  // 被投票者的epoch
    }
}
```

### 3.3 选举规则

```
比较顺序（选出数据最新的节点作为Leader）：

1. 比较epoch（peerEpoch）
   → epoch大的优先（经历过更多Leader轮次，数据更完整）

2. 如果epoch相同，比较zxid
   → zxid大的优先（包含更多事务，数据更完整）

3. 如果zxid相同，比较serverId（myid）
   → serverId大的优先（配置中指定的ID）

总结：先比epoch → 再比zxid → 最后比myid
```

### 3.4 FastLeaderElection源码

```java
public class FastLeaderElection {

    volatile Vote currentVote;  // 当前投票

    /**
     * 开始选举
     */
    public Vote lookForLeader() throws InterruptedException {
        // 1. 初始化：投自己
        self.updateElectionEpoch();
        long electionEpoch = self.getElectionEpoch();
        currentVote = new Vote(self.getId(),
            self.getLastLoggedZxid(),  // 自己的zxid
            electionEpoch,
            self.getPeerEpoch());       // 自己的epoch

        // 2. 广播自己的投票
        sendNotifications();

        // 3. 循环处理收到的投票
        Map<Long, Vote> recvset = new HashMap<>(); // 收到的投票

        while (true) {
            Notification n = recvqueue.take(); // 阻塞等待

            switch (n.state) {
                case LOOKING:
                    // 对方也在选举中
                    if (n.electionEpoch > electionEpoch) {
                        // 对方的选举轮次更大——更新自己的轮次
                        electionEpoch = n.electionEpoch;
                        recvset.clear();
                    }

                    if (n.electionEpoch < electionEpoch) {
                        // 对方的轮次更旧——忽略
                        continue;
                    }

                    // 记录对方投票
                    Vote vote = new Vote(n.leader, n.zxid,
                        n.electionEpoch, n.peerEpoch);
                    recvset.put(n.sid, vote);

                    // 检查是否过半
                    if (termPredicate(recvset, currentVote)) {
                        // 过半同意——选举成功
                        // 等待一段时间确认没有更优投票
                        Thread.sleep(finalizeWait);

                        while (!recvqueue.isEmpty()) {
                            Notification later = recvqueue.poll();
                            // 如果有更优投票，重新选举
                            if (totalOrderPredicate(
                                later.leader, later.zxid, later.peerEpoch,
                                currentVote.getId(),
                                currentVote.getZxid(),
                                currentVote.getPeerEpoch())) {
                                recvqueue.put(later);
                                break;
                            }
                        }

                        // 确认无更优投票——设置Leader
                        self.setPeerState(
                            (currentVote.getId() == self.getId()) ?
                            ServerState.LEADING : ServerState.FOLLOWING);
                        return currentVote;
                    }
                    break;

                case FOLLOWING:
                case LEADING:
                    // 对方已经有Leader了
                    // 验证这个Leader是否有效
                    Vote completedVote = new Vote(n.leader, n.zxid,
                        n.electionEpoch, n.peerEpoch);
                    if (termPredicate(recvset, completedVote)) {
                        self.setPeerState(
                            (n.leader == self.getId()) ?
                            ServerState.LEADING : ServerState.FOLLOWING);
                        return completedVote;
                    }
                    break;
            }
        }
    }

    /**
     * 比较两个候选者
     * 返回true表示候选者1更优
     */
    protected boolean totalOrderPredicate(
            long newId, long newZxid, long newEpoch,
            long curId, long curZxid, long curEpoch) {

        // 1. 比较epoch
        if (newEpoch > curEpoch) return true;
        if (newEpoch < curEpoch) return false;

        // 2. 比较zxid
        if (newZxid > curZxid) return true;
        if (newZxid < curZxid) return false;

        // 3. 比较serverId
        return newId > curId;
    }

    /**
     * 检查投票是否过半
     */
    protected boolean termPredicate(Map<Long, Vote> votes, Vote vote) {
        // 统计同意此投票的数量
        int count = 0;
        for (Vote v : votes.values()) {
            if (v.getId() == vote.getId() &&
                v.getElectionEpoch() == vote.getElectionEpoch()) {
                count++;
            }
        }
        // 是否过半
        return count > self.getQuorumSize() / 2;
    }
}
```

## 4. 数据树与存储

### 4.1 ZNode类型

```
持久节点（PERSISTENT）：创建后一直存在，直到被删除
持久顺序节点（PERSISTENT_SEQUENTIAL）：持久+自动递增序号
临时节点（EPHEMERAL）：客户端Session断开后自动删除
临时顺序节点（EPHEMERAL_SEQUENTIAL）：临时+自动递增序号
容器节点（CONTAINER）：最后一个子节点被删除后，容器节点也会被删除
TTL节点（PERSISTENT_WITH_TTL）：持久节点+TTL，超时无子节点则删除
```

### 4.2 DataTree——内存数据树

```java
public class DataTree {

    // 所有ZNode的内存存储——ConcurrentHashMap
    private final ConcurrentHashMap<String, DataNode> nodes =
        new ConcurrentHashMap<>();

    // Watch管理
    private final WatchManager dataWatches;   // 数据变更Watch
    private final WatchManager childWatches;  // 子节点变更Watch

    /**
     * 创建节点
     */
    public void createNode(String path, byte[] data, long zxid,
            long time, int version) {
        // 1. 创建DataNode
        DataNode node = new DataNode(data, version, time);
        node.setStat(new Stat(...));

        // 2. 加入数据树
        DataNode parent = nodes.get(getParentPath(path));
        parent.addChild(path);
        nodes.put(path, node);

        // 3. 触发Watcher
        dataWatches.triggerWatch(path, Event.EventType.NodeCreated);
        childWatches.triggerWatch(getParentPath(path),
            Event.EventType.NodeChildrenChanged);

        // 4. 记录事务
        addChangeRecord(new ChangeRecord(
            ChangeRecord.CREATE, path, data, zxid));
    }

    /**
     * 读取节点数据
     */
    public byte[] getData(String path, Stat stat, Watcher watcher) {
        DataNode node = nodes.get(path);
        if (node == null) {
            throw new NoNodeException("Node not found: " + path);
        }

        // 注册Watcher
        if (watcher != null) {
            dataWatches.addWatch(path, watcher);
        }

        // 返回数据
        byte[] data = node.getData();
        if (stat != null) {
            copyStat(node.getStat(), stat);
        }
        return data;
    }
}

// DataNode——单个节点
public class DataNode {
    byte[] data;              // 节点数据
    Long acl;                 // ACL ID
    public StatPersisted stat; // 统计信息（版本、时间戳等）
    Set<String> children;     // 子节点列表
}
```

```java
// FileTxnLog——事务日志文件
public class FileTxnLog {

    /**
     * 追加事务日志
     */
    public synchronized boolean append(TxnHeader hdr, Record txn) {
        // 1. 如果日志文件不存在或已满，创建新文件
        if (logStream == null) {
            File logFile = new File(logDir,
                "log." + Long.toHexString(hdr.getZxid()));
            logStream = new BufferedOutputStream(
                new FileOutputStream(logFile));
        }

        // 2. 序列化事务头和事务体
        OutputArchive oa = BinaryOutputArchive.getArchive(logStream);
        oa.writeRecord(hdr, "hdr");  // 事务头（zxid、时间、类型）
        oa.writeRecord(txn, "txn");  // 事务体（具体操作）

        // 3. 计算并写入CRC
        logStream.flush();

        // 4. 按需刷盘
        if (forceSync) {
            ((FileOutputStream) logStream).getFD().sync();
        }

        return true;
    }
}
```

### 4.4 快照

```java
// FileSnap——快照文件
public class FileSnap {

    /**
     * 保存快照
     */
    public synchronized void serialize(DataTree dt, Map<Long, Integer> sessions,
            File snapShot, boolean fsync) {
        // 1. 创建快照文件
        FileOutputStream sessOS = new FileOutputStream(snapShot);
        CheckedOutputStream couts = new CheckedOutputStream(
            sessOS, new Adler32());
        OutputArchive oa = BinaryOutputArchive.getArchive(couts);

        // 2. 写入文件头（magic、version、dbid）
        FileHeader header = new FileHeader(
            SNAP_MAGIC, VERSION, dbid);
        oa.writeRecord(header, "fileheader");

        // 3. 序列化Session列表
        serializeSessions(dt, sessions, oa);

        // 4. 序列化DataTree
        dt.serialize(oa);

        // 5. 写入Checksum
        long checksum = couts.getChecksum().getValue();
        oa.writeLong(checksum, "val");

        // 6. 刷盘
        sessOS.flush();
        if (fsync) {
            sessOS.getFD().sync();
        }
    }

    /**
     * 恢复快照
     */
    public long deserialize(DataTree dt, Map<Long, Integer> sessions) {
        // 1. 找到最新的快照文件
        File snapFile = findNValidSnapshots(1).get(0);

        // 2. 反序列化
        InputArchive ia = BinaryInputArchive.getArchive(
            new FileInputStream(snapFile));
        FileHeader header = new FileHeader();
        ia.readRecord(header, "fileheader");

        // 3. 恢复Session
        deserializeSessions(ia, sessions);

        // 4. 恢复DataTree
        dt.deserialize(ia);

        // 5. 返回快照中的最大zxid
        return header.getZxid();
    }
}

// 快照文件命名：snapshot.{zxid的十六进制}
// /var/lib/zookeeper/version-2/
// ├── snapshot.100000000
// ├── snapshot.200000000
// └── snapshot.300000000
```

## 5. Watcher机制

### 5.1 Watcher类型

```
三种Watcher：
1. 数据Watch（dataWatches）
   └── 监听节点数据变化
   └── getData(path, watcher) 注册
   └── 触发：NodeDataChanged, NodeDeleted

2. 子节点Watch（childWatches）
   └── 监听子节点列表变化
   └── getChildren(path, watcher) 注册
   └── 触发：NodeChildrenChanged, NodeDeleted

3. 存在性Watch（existWatches）
   └── 监听节点是否存在
   └── exists(path, watcher) 注册
   └── 触发：NodeCreated, NodeDeleted, NodeDataChanged

重要特性：一次性触发！
  Watcher触发后自动移除，需要重新注册才能再次监听
```

### 5.2 WatchManager源码

```java
public class WatchManager {

    // path → Watcher集合
    private final Map<String, Set<Watcher>> watchTable =
        new ConcurrentHashMap<>();

    // Watcher → path集合（反向索引，方便Watcher移除）
    private final Map<Watcher, Set<String>> watch2Paths =
        new ConcurrentHashMap<>();

    /**
     * 注册Watcher
     */
    public synchronized void addWatch(String path, Watcher watcher) {
        // 1. path → Watcher
        Set<Watcher> list = watchTable.computeIfAbsent(
            path, k -> new HashSet<>());
        list.add(watcher);

        // 2. Watcher → path
        Set<String> paths = watch2Paths.computeIfAbsent(
            watcher, k -> new HashSet<>());
        paths.add(path);
    }

    /**
     * 触发Watcher
     */
    public Set<Watcher> triggerWatch(String path, EventType type) {
        return triggerWatch(path, type, null);
    }

    public Set<Watcher> triggerWatch(String path, EventType type,
            Set<Watcher> supress) {
        // 1. 构建WatchedEvent
        WatchedEvent e = new WatchedEvent(type,
            KeeperState.SyncConnected, path);

        // 2. 获取并移除Watcher（一次性！）
        Set<Watcher> watchers;
        synchronized (this) {
            watchers = watchTable.remove(path); // 移除
            if (watchers == null || watchers.isEmpty()) {
                return null;
            }

            for (Watcher w : watchers) {
                Set<String> paths = watch2Paths.get(w);
                if (paths != null) {
                    paths.remove(path);
                    if (paths.isEmpty()) {
                        watch2Paths.remove(w);
                    }
                }
            }
        }

        // 3. 通知所有Watcher
        for (Watcher w : watchers) {
            if (supress != null && supress.contains(w)) {
                continue;
            }
            w.process(e); // 回调Watcher
        }

        return watchers;
    }
}
```

```java
        }
    }
}
```

## 6. 事务处理流程

### 6.1 Leader处理写请求

```java
// LeaderZooKeeperServer
public class LeaderZooKeeperServer extends ZooKeeperServer {

    @Override
    public void submitRequest(Request si) {
        if (si.type == OpCode.create || si.type == OpCode.delete ||
            si.type == OpCode.setData) {
            // 写请求——走Proposal流程
            leader.propose(si);
        } else {
            // 读请求——直接处理
            super.submitRequest(si);
        }
    }
}

// Leader.propose
public void propose(Request request) throws IOException {
    // 1. 分配zxid
    long zxid = zk.getZxid();
    request.setZxid(zxid);

    // 2. 写入Leader本地日志
    zk.getZKDatabase().getLog().append(
        request.getHdr(), request.getTxn());

    // 3. 创建Proposal包
    QuorumPacket proposal = new QuorumPacket(
        Leader.PROPOSAL, zxid, request.serialize(), null);

    // 4. 广播给所有Follower
    for (LearnerHandler fh : learners) {
        fh.queuePacket(proposal);
    }

    // 5. 等待过半ACK
    Proposal p = new Proposal(request, zxid);
    outstandingProposals.put(zxid, p);
}
```

### 6.2 Follower处理写请求

```java
// FollowerZooKeeperServer
public class FollowerZooKeeperServer extends ZooKeeperServer {

    /**
     * Follower收到客户端的写请求
     */
    @Override
    public void submitRequest(Request si) {
        if (si.type == OpCode.create || /* ... */) {
            // 转发给Leader
            requestThrottler.addRequest(si);
            pendingSyncs.add(si);
            // 发送请求到Leader
            follower.request(si);
        } else {
            // 读请求——直接本地处理
            super.submitRequest(si);
        }
    }

    /**
     * 收到Leader的PROPOSAL
     */
    void logRequest(Request request) {
        // 1. 写入Follower本地日志
        zk.getZKDatabase().getLog().append(
            request.getHdr(), request.getTxn());

        // 2. 发送ACK给Leader
        QuorumPacket ack = new QuorumPacket(
            Leader.ACK, request.getZxid(), null, null);
        follower.writePacket(ack, false);

        // 3. 等待COMMIT
        pendingTxns.add(request);
    }

    /**
     * 收到Leader的COMMIT
     */
    void commit(long zxid) {
        // 1. 从pending队列取出
        Request request = pendingTxns.remove();

        // 2. 应用到本地DataTree
        zk.getZKDatabase().processTxn(request.getHdr(), request.getTxn());

        // 3. 响应客户端
        request.createResponse();
    }
}
```

## 7. 读写流程

### 7.1 读流程

```
客户端读请求：

1. 客户端发送getData(path, watcher)到任意ZK节点
2. ZK节点从本地DataTree读取数据
3. 返回数据（不需要Leader参与）

特点：
  - 读不需要过半，任何节点都可以响应
  - 读性能高（纯内存操作）
  - 可能读到旧数据（Follower数据可能落后于Leader）

保证线性一致性读：
  - 使用sync()操作——读取前先与Leader同步
  - zk.sync(path, voidCallback, ctx);
  - zk.getData(path, watcher, cb, ctx); // sync后再read
```

### 7.2 写流程

```
客户端写请求（以create为例）：

1. 客户端发送create("/node", data)到Follower
2. Follower转发请求到Leader
3. Leader生成zxid，写入本地日志
4. Leader广播PROPOSAL给所有Follower
5. Follower写入本地日志，回复ACK
6. Leader收到过半ACK，发送COMMIT
7. Follower应用事务到DataTree
8. Leader/Follower响应客户端

特点：
  - 写必须经过Leader
  - 需要过半节点确认（不是全部）
  - 写性能取决于集群规模和网络延迟
```

## 8. Session管理

### 8.1 Session机制

```java
// SessionTracker——Session跟踪器
public class SessionTracker {

    // sessionById: SessionID → SessionImpl
    ConcurrentHashMap<Long, SessionImpl> sessionById;

    // sessionExpiryQueue: 过期时间 → SessionID集合
    // 按过期时间分桶，方便批量检测
    ExpiryQueue<Long> sessionExpiryQueue;

    /**
     * 创建Session
     */
    public long createSession(int sessionTimeout) {
        long sessionId = generateSessionId();
        SessionImpl session = new SessionImpl(sessionId, sessionTimeout);
        sessionById.put(sessionId, session);
        sessionExpiryQueue.update(sessionId, sessionTimeout);
        return sessionId;
    }

    /**
     * 心跳续期
     */
    public void touchSession(long sessionId, int timeout) {
        SessionImpl session = sessionById.get(sessionId);
        if (session != null) {
            // 更新过期时间
            sessionExpiryQueue.update(sessionId, timeout);
        }
    }

    /**
     * 检查过期Session
     */
    public void run() {
        while (running) {
            // 获取已过期的Session
            Map<Long, Set<Long>> expired = sessionExpiryQueue.poll();

            for (Map.Entry<Long, Set<Long>> entry : expired.entrySet()) {
                for (Long sessionId : entry.getValue()) {
                    // 关闭过期Session
                    closeSession(sessionId);
                    // 删除临时节点
                    cleanupEphemerals(sessionId);
                }
            }
        }
    }
}
```

### 8.2 分桶过期检测

```
Session过期检测使用"分桶"机制：

  过期时间按 interval 分桶（默认 tickTime=2000ms）

  桶0 (0-2s):   [session1, session3]
  桶1 (2-4s):   [session2]
  桶2 (4-6s):   [session5, session7, session9]
  桶3 (6-8s):   [session4]

  当前时间到达某个桶的结束时间时
  → 检查该桶内所有Session是否真的过期
  → 过期的Session被关闭，临时节点被删除

  心跳续期：
  → touchSession将Session从当前桶移到新的桶（当前时间+timeout）
  → 相当于"延后过期"

  优点：
  → 不需要遍历所有Session检查过期
  → 只需检查当前桶
  → O(1)的过期检测
```

## 9. ACL权限控制

### 9.1 ACL结构

```
ACL = Scheme:ID:Permissions

Scheme（认证方式）：
  world   —— 任何人
  auth    —— 已认证用户
  digest  —— 用户名密码
  ip      —— IP地址

Permissions（权限）：
  CREATE  (c) —— 创建子节点
  READ    (r) —— 读取节点数据和子节点列表
  WRITE   (w) —— 修改节点数据
  DELETE  (d) —— 删除子节点
  ADMIN   (a) —— 设置ACL
```

## 11. 面试题速查

### Q1: ZAB协议的两个模式是什么？

崩溃恢复模式（Leader选举+数据同步）和消息广播模式（类似2PC的写流程）。消息广播模式下Leader为每个Proposal分配递增zxid，广播给所有Follower，过半ACK后Commit。不需要所有节点ACK，只需Quorum（过半）。

### Q2: Leader选举的规则是什么？

依次比较三个维度：1.epoch（Leader纪元，越大越新）；2.zxid（事务ID，越大数据越完整）；3.myid（服务器配置ID，大的优先）。先选epoch最大的，相同则选zxid最大的，再相同选myid最大的。保证选出数据最完整的节点作为Leader。

### Q3: zxid的结构和作用是什么？

zxid是64位整数：高32位是epoch（Leader纪元），低32位是counter（事务序号）。每次Leader选举epoch+1，counter在每个Leader内递增。zxid保证全局有序——同一epoch内counter越大越新，不同epoch大的epoch更优。Leader选举时用zxid比较数据完整性。

### Q4: Zookeeper的Watcher机制有什么特点？

一次性触发——Watcher触发后自动移除，需要重新注册才能再次监听。三种Watcher：数据Watch（getData注册）、子节点Watch（getChildren注册）、存在性Watch（exists注册）。客户端串行处理Watcher回调。ZK 3.6+引入永久Watcher（addWatch）。

### Q5: Zookeeper如何保证事务的顺序一致性？

1.Leader为每个事务分配全局递增的zxid。2.Follower按zxid顺序应用事务。3.过半ACK后才Commit，保证不会Commit未确认的事务。4.数据同步阶段，Leader确保新Follower的数据与Leader一致后才加入集群。

### Q6: 临时节点有什么用？什么时候被删除？

临时节点在客户端Session断开后自动删除。常用于：1.服务注册与发现（Dubbo Provider注册临时节点，宕机自动摘除）；2.分布式锁（临时节点+Watch实现锁等待）；3.Leader选举（创建临时节点，成功者为Leader）。Session超时后由SessionTracker线程检测并删除。

### Q7: Zookeeper的读写性能为什么不对称？

读不需要Leader参与，任意Follower都能从本地DataTree读取（纯内存操作），性能高。写必须经过Leader，且需要过半Follower ACK（网络往返+日志写入），性能较低。读多写少场景适合ZK。Follower可能读到旧数据（最终一致性），使用sync()可以保证线性一致性读。

### Q8: Zookeeper如何实现分布式锁？

方案1：创建临时节点——成功获取锁，失败则Watch该节点，删除时通知。方案2（更好）：创建临时顺序节点——序号最小者获取锁，否则Watch前一个节点（避免惊群）。临时节点保证客户端宕机时锁自动释放。顺序节点+Watch前驱避免羊群效应。

### Q9: Zookeeper的数据是如何持久化的？

事务日志（FileTxnLog）：每条写操作追加记录到log.{zxid}文件，顺序写。快照（FileSnap）：定期将DataTree全量序列化到snapshot.{zxid}文件。恢复时先加载最新快照，再重放快照之后的事务日志。写请求先写日志再更新内存，保证持久性。

### Q10: ZAB和Raft有什么区别？

ZAB的Leader选举比较epoch+zxid（数据完整性优先），Raft比较term+日志索引。ZAB的日志复制是Leader推送Proposal（push），Raft是Leader等待Follower拉取（pull/append entries）。ZAB有专门的崩溃恢复阶段保证数据一致性，Raft通过日志匹配和选举约束保证。两者都保证强一致性，但ZAB更优化了Leader切换的恢复流程。
