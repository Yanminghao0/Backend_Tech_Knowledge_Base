## 1. 对象头 Mark Word 布局

在 HotSpot JVM 中，每个 Java 对象在内存中由三部分组成：**对象头（Header）、实例数据（Instance Data）、对齐填充（Padding）**。其中对象头是实现 synchronized 的关键。

### 1.1 对象头结构

对象头由两部分组成：
- **Mark Word**：存储对象自身的运行时数据，如哈希码、GC 分代年龄、锁信息等
- **Klass Pointer**：指向对象所属类的元数据的指针

### 1.2 32位 JVM Mark Word 布局

```
|-------------------------------------------------------|--------------------|
|                  Mark Word (32 bits)                  |       State        |
|-------------------------------------------------------|--------------------|
| hashcode:25 | age:4 | biased_lock:0 |    01          |     无锁(Normal)    |
|-------------------------------------------------------|--------------------|
| threadID:23 | epoch:2 | age:4 | biased_lock:1 |  01   |     偏向锁(Biased)  |
|-------------------------------------------------------|--------------------|
|           ptr_to_lock_record:30          |    00      |   轻量级锁(Locked)  |
|-------------------------------------------------------|--------------------|
|           ptr_to_heavyweight_monitor:30  |    10      |   重量级锁(Inflated)|
|-------------------------------------------------------|--------------------|
|                                            |    11    |    GC标记(Marked)   |
|-------------------------------------------------------|--------------------|
```

### 1.3 64位 JVM Mark Word 布局

```
|--------------------------------------------------------------------|--------------------|
|                      Mark Word (64 bits)                           |       State        |
|--------------------------------------------------------------------|--------------------|
| unused:25 | hashcode:31 | unused:1 | age:4 | biased_lock:0 |  01   |     无锁(Normal)    |
|--------------------------------------------------------------------|--------------------|
| threadID:54 | epoch:2  | unused:1 | age:4 | biased_lock:1 |  01   |     偏向锁(Biased)  |
|--------------------------------------------------------------------|--------------------|
|                  ptr_to_lock_record:62                    |   00   |   轻量级锁(Locked)  |
|--------------------------------------------------------------------|--------------------|
|               ptr_to_heavyweight_monitor:62               |   10   |   重量级锁(Inflated)|
|--------------------------------------------------------------------|--------------------|
|                                                           |   11   |    GC标记(Marked)   |
|--------------------------------------------------------------------|--------------------|
```

### 1.4 Mark Word 源码定义

在 HotSpot 源码 `markWord.hpp` 中：

```cpp
// markWord.hpp (HotSpot 源码简化)
class markWord {
private:
    uintptr_t _value;

public:
    // 最低位锁标志位
    static const int locked_value          = 0;  // 00 轻量级锁
    static const int unlocked_value        = 1;  // 01 无锁/偏向锁
    static const int monitor_value         = 2;  // 10 重量级锁
    static const int marked_value          = 3;  // 11 GC标记

    // 偏向锁标志位
    static const int biased_lock_pattern   = 5;  // 101 偏向锁

    // 常量定义
    static const int hash_bits             = 31; // 64位下hash码位数
    static const int age_bits              = 4;  // GC年龄位数
    static const int biased_lock_bits      = 1;  // 偏向锁标志位

    // 获取锁状态
    bool is_locked() const   { return (_value & locked_value) == 0; }
    bool is_unlocked() const { return (_value & locked_value) == 1; }
    bool is_monitor() const  { return (_value & monitor_value) == monitor_value; }
    bool is_neutral() const  { return (_value & 3) == unlocked_value; }

    // 判断是否为偏向锁
    bool has_bias_pattern() const {
        return (_value & biased_lock_mask_in_place) == biased_lock_pattern;
    }

    // 获取哈希码
    uintptr_t hash() const {
        return mask_bits(value() >> hash_shift, hash_mask);
    }

    // 获取偏向线程ID
    JavaThread* biased_locker() const {
        return (JavaThread*) mask_bits(value() >> biased_lock_shift,
                                        biased_lock_mask);
    }
};
```

### 1.5 锁标志位含义

| 锁状态 | biased_lock | lock(2位) | 描述 |
|--------|-------------|-----------|------|
| 无锁 | 0 | 01 | 对象初始状态，可偏向 |
| 偏向锁 | 1 | 01 | 第一个线程访问时设置 |
| 轻量级锁 | 0 | 00 | 多线程交替执行，无竞争 |
| 重量级锁 | 0 | 10 | 多线程同时竞争 |
| GC标记 | - | 11 | 垃圾回收时使用 |

> **关键点**：lock 最低位为 0 表示被锁定（轻量级或重量级），为 1 表示未锁定。biased_lock 位区分普通无锁和偏向锁状态。

## 3. 偏向锁源码解析

### 3.1 偏向锁获取流程

偏向锁的核心思想：**如果一个线程获得了锁，那么锁就进入偏向模式，此时 Mark Word 的结构变为偏向锁结构，存储持有锁的线程 ID。该线程在后续进入和退出同步块时不需要进行 CAS 操作来加锁和解锁。**

源码入口在 `bytecodeInterpreter.cpp` 中的 `monitorenter` 指令处理：

```cpp
// bytecodeInterpreter.cpp (简化)
CASE(_monitorenter): {
    // 获取锁对象
    oop lockee = STACK_OBJECT(-1);
    // 获取对象头
    markWord mark = lockee->mark();
    // 获取当前线程指针
    markWord header = mark.set_locker_thread(THREAD);

    // ====== 步骤1: 检查是否为偏向锁模式 ======
    if (mark.has_bias_pattern()) {
        // 获取偏向的线程ID
        JavaThread* biased_thread = mark.biased_locker();

        if (biased_thread == THREAD) {
            // 步骤1a: 如果是同一个线程重入，直接成功（CAS都不需要）
            // Mark Word 不变，直接返回
            goto done;
        } else if (biased_thread == NULL) {
            // 步骤1b: 匿名偏向状态(可偏向但无线程)
            // 尝试用 CAS 将当前线程ID写入 Mark Word
            markWord new_header = mark.set_biased_locker(THREAD);
            if (lockee->cas_set_mark(new_header, mark) == mark) {
                // CAS 成功，获取偏向锁成功
                goto done;
            }
            // CAS 失败，说明有竞争，走到撤销逻辑
        } else {
            // 步骤1c: 已偏向其他线程，尝试撤销偏向
            // 调用 revoke_bias 撤销偏向锁
            // 如果撤销成功则升级为轻量级锁
            // 如果撤销失败则升级为重量级锁
            markWord prototype = markWord::prototype();
            if (lockee->cas_set_mark(prototype, mark) != mark) {
                // 撤销失败，走慢路径 -> 升级重量级锁
                CALL_VM(InterpreterRuntime::monitorenter(THREAD, lockee),
                        handle_exception);
            }
            goto done;
        }
    }

    // ====== 步骤2: 非偏向锁模式，尝试轻量级锁 ======
    // ... 见轻量级锁章节
}
```

### 3.2 偏向锁撤销（Revoke）

偏向锁撤销是一个复杂的操作，因为需要等待持有偏向锁的线程到达安全点：

```cpp
// biasedLocking.cpp (简化)
void BiasedLocking::revoke_and_rebias(Handle obj, bool attempt_rebias,
                                       TRAPS) {
    markWord mark = obj->mark();

    if (mark.is_biased()) {
        JavaThread* biased_thread = mark.biased_locker();

        if (biased_thread == NULL) {
            // 匿名偏向，直接撤销
            markWord new_mark = markWord::prototype();
            obj->set_mark(new_mark);
            return;
        }

        // 检查偏向线程是否存活
        if (!biased_thread->is_alive()) {
            // 偏向线程已死亡，直接撤销
            markWord new_mark = markWord::prototype();
            obj->set_mark(new_mark);
            return;
        }

        // 偏向线程存活，需要等待安全点进行撤销
        // VMThread 在 safepoint 执行撤销
        if (attempt_rebias) {
            // 尝试重新偏向到当前线程
            markWord new_mark = mark.set_biased_locker(THREAD);
            if (obj->cas_set_mark(new_mark, mark) == mark) {
                return; // 重新偏向成功
            }
        }
        // 走到批量撤销或升级锁
    }
}
```

### 3.3 批量重偏向与批量撤销

JVM 会统计每个类被撤销偏向锁的次数，超过阈值后触发批量操作：

```cpp
// biasedLocking.cpp
class BiasedLockingCounters {
    int _biased_lock_entry_count;   // 偏向锁获取次数
    int _anonymously_biased_lock_entry_count; // 匿名偏向次数
    int _rebiased_lock_entry_count;  // 重偏向次数
    int _revoked_lock_entry_count;   // 撤销次数
    int _slow_path_entry_count;      // 慢路径次数
    int _total_entry_count;
};

// 批量重偏向阈值
static int _biased_lock_rebias_threshold = 20;  // 每类撤销20次后
// 批量撤销阈值
static int _biased_lock_bulk_rebias_threshold = 40;
// 批量撤销最终阈值
static int _biased_lock_bulk_revoke_threshold = 40;
// 延迟(毫秒)后才能批量撤销
static int _biased_lock_bulk_revoke_delay = 25000;

// 当撤销次数达到 rebias_threshold:
// 1. 更新该类的 epoch 值
// 2. 所有该类的对象，如果 epoch 不匹配，说明是旧的偏向，可安全重偏向
// 当撤销次数达到 bulk_revoke_threshold:
// 1. 该类所有对象禁用偏向锁
// 2. 新建对象也不再使用偏向锁
```

## 4. 轻量级锁源码解析

### 4.1 轻量级锁加锁过程

轻量级锁的核心：**在线程的栈帧中创建 Lock Record，通过 CAS 操作将对象头的 Mark Word 拷贝到 Lock Record 中，然后将对象头指向 Lock Record。**

```cpp
// bytecodeInterpreter.cpp (简化)
CASE(_monitorenter): {
    // ... 偏向锁判断后 ...

    // ====== 轻量级锁加锁 ======
    // 在当前线程栈帧中分配 Lock Record
    BasicLock *lock = allocate_lock(this);

    // 将对象当前的 Mark Word 拷贝到 Lock Record 的 displaced_header
    lock->set_displaced_header(mark);

    // CAS 操作: 将对象头替换为指向 Lock Record 的指针
    markWord new_mark = markWord::from_pointer(lock);
    new_mark = new_mark.set_lock_bits(markWord::locked_value);

    if (lockee->cas_set_mark(new_mark, mark) == mark) {
        // CAS 成功，获取轻量级锁成功
        // 对象头现在存储的是 Lock Record 的地址，lock 最低位为 00
        lock->set_obj(lockee);
        goto done;
    } else {
        // CAS 失败，说明有竞争
        // 检查是否是自己重入
        if (mark.has_monitor()) {
            // 已经是重量级锁了
            ObjectMonitor* monitor = mark.monitor();
            if (monitor->try_enter(THREAD)) {
                // 重量级锁重入成功
                goto done;
            } else {
                // 重量级锁竞争失败，走慢路径
                CALL_VM(InterpreterRuntime::monitorenter(THREAD, lockee),
                        handle_exception);
            }
        } else {
            // 两个线程同时竞争轻量级锁
            // 锁膨胀: 轻量级锁 -> 重量级锁
            CALL_VM(InterpreterRuntime::monitorenter(THREAD, lockee),
                        handle_exception);
        }
    }
}
```

### 4.2 Lock Record 结构

```
线程栈帧                          堆内存(对象)
┌─────────────────────┐         ┌──────────────────────┐
│    Lock Record       │         │    Object            │
│  ┌─────────────────┐│         │  ┌────────────────┐  │
│  │ displaced_header││ ←────── │  │   Mark Word     │  │
│  │ (原Mark Word)    ││  拷贝   │  │ (指向Lock Record)│ │
│  ├─────────────────┤│         │  │ lock bits: 00   │  │
│  │ obj (锁对象指针) ││──────→  │  └────────────────┘  │
│  └─────────────────┘│         │                      │
└─────────────────────┘         └──────────────────────┘
```

### 4.3 轻量级锁解锁过程

```cpp
// bytecodeInterpreter.cpp (简化)
CASE(_monitorexit): {
    // 获取 Lock Record
    BasicLock *lock = most_recent_lock;
    oop lockee = lock->obj();
    markWord header = lock->displaced_header();

    // CAS 操作: 将 displaced_header 恢复到对象头
    markWord old_mark = lockee->mark();
    if (old_mark == markWord::from_pointer(lock)) {
        // 如果对象头仍然指向自己的 Lock Record
        // 说明没有竞争，直接恢复
        markWord new_mark = header;
        if (lockee->cas_set_mark(new_mark, old_mark) == old_mark) {
            // CAS 成功，释放锁成功
            goto done;
        }
        // CAS 失败，说明有竞争，需要释放重量级锁
    }

    // 走慢路径: 重量级锁释放
    if (lockee->mark().has_monitor()) {
        ObjectMonitor* monitor = lockee->mark().monitor();
        monitor->exit(true, THREAD);
    }
}
```

## 5. 重量级锁源码解析

### 5.1 锁膨胀（Inflation）

当轻量级锁竞争失败时，会触发锁膨胀，升级为重量级锁：

```cpp
// synchronizer.cpp (简化)
ObjectMonitor* ObjectSynchronizer::inflate(Thread * SELF,
                                            oop object,
                                            const InflateCause cause) {
    for (;;) {
        const markWord mark = object->mark();

        // 已经是重量级锁
        if (mark.has_monitor()) {
            return mark.monitor();
        }

        // 正在进行重量级锁膨胀中
        if (mark == markWord::INFLATING()) {
            // 短暂自旋等待膨胀完成
            read_stablemark(object);
            continue;
        }

        // 无锁状态或轻量级锁状态
        if (mark.is_neutral()) {
            // 无锁状态直接膨胀
            ObjectMonitor * m = omAlloc(SELF);
            m->set_header(mark);  // 保存原 Mark Word
            m->set_object(object);
            m->set_owner(NULL);

            // CAS: 将对象头设为指向 monitor
            markWord monitor_mark = markWord::encode(m);
            if (object->cas_set_mark(monitor_mark, mark) == mark) {
                return m;  // 膨胀成功
            }
            // CAS 失败，重试
        }

        // 轻量级锁状态
        if (mark.has_locker()) {
            // 创建 ObjectMonitor
            ObjectMonitor * m = omAlloc(SELF);
            m->set_header(mark);  // 保存原 Mark Word(含 Lock Record 指针)
            m->set_object(object);
            m->set_owner(mark.locker());
            m->set_anonymous_owner();

            // CAS: 标记正在膨胀
            if (object->cas_set_mark(markWord::INFLATING(), mark) != mark) {
                continue; // 重试
            }

            // 膨胀成功，设置对象头指向 monitor
            markWord monitor_mark = markWord::encode(m);
            object->set_mark(monitor_mark);
            return m;
        }
    }
}
```

### 5.2 ObjectMonitor::enter()

```cpp
// objectMonitor.cpp (简化)
void ObjectMonitor::enter(TRAPS) {
    Thread * const Self = THREAD;
    void * cur = Atomic::cmpxchg(&_owner, (void*)NULL, Self);

    if (cur == NULL) {
        // 无竞争，直接获取锁
        assert(_recursions == 0, "invariant");
        return;
    }

    if (cur == Self) {
        // 重入
        _recursions++;
        return;
    }

    if (Self->is_lock_owned((address)cur)) {
        // 当前线程之前持有轻量级锁，现在膨胀为重量级锁
        assert(_recursions == 0, "invariant");
        _recursions = 1;
        set_owner_from_BasicLock(cur, Self);
        return;
    }

    // ====== 有竞争，进入慢路径 ======
    Self->_Stalled = intptr_t(this);

    // 尝试自旋获取锁（自适应自旋）
    if (Knob_SpinEarly && (TrySpin(Self) > 0)) {
        // 自旋成功，获取锁
        Self->_Stalled = 0;
        return;
    }

    // 自旋失败，进入 EntryList 队列
    Atomic::inc(&_count);

    for (;;) {
        EnterI(THREAD);
        // EnterI 中会 park 线程

        // 被 unpark 唤醒后，检查是否获取到锁
        if (TryLock(Self) > 0) break;
    }

    Self->_Stalled = 0;
    Atomic::dec(&_count);
}

void ObjectMonitor::EnterI(TRAPS) {
    Thread * const Self = THREAD;

    // 再次尝试获取锁
    if (TryLock(Self) > 0) return;

    // 尝试自旋
    if (TrySpin(Self) > 0) return;

    // 创建 ObjectWaiter
    ObjectWaiter node(Self);
    node.TState = ObjectWaiter::TS_CXQ;

    // 加入 cxq 队列（Contention Queue）
    for (;;) {
        ObjectWaiter * nxt = _cxq;
        node._next = nxt;
        if (Atomic::cmpxchg(&_cxq, nxt, &node) == nxt) break;
    }

    // 再次尝试自旋
    if (TryLock(Self) > 0) {
        // 从队列移除自己
        return;
    }

    // park 阻塞当前线程
    for (;;) {
        if (TryLock(Self) > 0) return;

        // 线程 park，等待被 unpark 唤醒
        Self->_ParkEvent->park();

        if (TryLock(Self) > 0) return;
    }
}
```

### 5.3 ObjectMonitor::exit()

```cpp
// objectMonitor.cpp (简化)
void ObjectMonitor::exit(bool not_suspended, TRAPS) {
    Thread * const Self = THREAD;

    // 处理重入
    if (_recursions > 0) {
        _recursions--;
        return;
    }

    // 保存当前 owner 信息
    void * cur = Atomic::load(&_owner);

    for (;;) {
        // 设置 owner 为 NULL（释放锁）
        if (Atomic::cmpxchg(&_owner, cur, (void*)NULL) == cur) {
            break;
        }
        cur = Atomic::load(&_owner);
    }

    // 唤醒等待线程
    // 优先策略：从 _EntryList 或 _cxq 中取出一个线程唤醒
    if (_EntryList != NULL) {
        // 从 EntryList 取第一个
        ObjectWaiter * w = _EntryList;
        if (w != NULL) {
            _EntryList = w->_next;
            w->_next = NULL;
            // unpark 唤醒线程
            w->_thread->_ParkEvent->unpark();
        }
    } else if (_cxq != NULL) {
        // 从 cxq 取出所有线程转移到 EntryList
        ObjectWaiter * w = _cxq;
        _cxq = NULL;
        // 倒序放入 EntryList
        while (w != NULL) {
            ObjectWaiter * nxt = w->_next;
            w->_next = _EntryList;
            _EntryList = w;
            w = nxt;
        }
        // 唤醒第一个
        w = _EntryList;
        if (w != NULL) {
            _EntryList = w->_next;
            w->_next = NULL;
            w->_thread->_ParkEvent->unpark();
        }
    }
}
```

```
┌─────────────┐
│  对象创建    │
│  Mark Word  │

       │

       │ → 撤销偏向锁(等待安全点)
       ↓
┌──────────────┐
│   轻量级锁    │
│   (000)      │

┌─────────────────────────────────────────────────────────┐
│ 无锁 → 偏向锁:   第一个线程访问同步块                     │
```

### 各锁状态对比

| 特性 | 偏向锁 | 轻量级锁 | 重量级锁 |
|------|--------|----------|----------|
| 适用场景 | 单线程访问 | 交替执行，无竞争 | 多线程同时竞争 |
| Mark Word存储 | 线程ID | Lock Record指针 | ObjectMonitor指针 |
| 加锁方式 | CAS(首次) | CAS(每次) | CAS + 自旋 + 阻塞 |
| 解锁方式 | 无操作 | CAS | CAS + 唤醒 |
| 性能 | 最高 | 高 | 低 |
| 开销 | 最低(撤销需安全点) | 中(CAS) | 高(系统调用) |
| 线程阻塞 | 否 | 否(自旋) | 是 |
| 操作系统介入 | 否 | 否 | 是 |

## 7. JIT 锁优化：锁消除与锁粗化

### 7.1 锁消除（Lock Elimination）

JIT 编译器通过**逃逸分析（Escape Analysis）**判断锁对象是否只被单个线程访问。如果锁对象不会逃逸到其他线程，JIT 会直接消除该 synchronized 块。

```java
// 源码
public String concatString(String s1, String s2, String s3) {
    StringBuffer sb = new StringBuffer();
    sb.append(s1);  // append 是 synchronized 方法
    sb.append(s2);
    sb.append(s3);
    return sb.toString();
}

// JIT 编译后(锁消除)
public String concatString(String s1, String s2, String s3) {
    // StringBuffer 没有逃逸，所有 synchronized 被消除
    StringBuffer sb = new StringBuffer();
    sb.append(s1);  // 无锁操作
    sb.append(s2);
    sb.append(s3);
    return sb.toString();
}
```

HotSpot 源码中的实现：

```cpp
// escaped.cpp (简化)
void ConnectionGraph::eliminate_lock(Node *acmp) {
    // 通过逃逸分析判断对象是否逃逸
    if (not_global_escape(acmp)) {
        // 对象未逃逸，可以消除锁
        // 将 monitorenter/monitorexit 节点标记为消除
        acmp->set_eliminated_lock();
    }
}

// 如果 -XX:+DoEscapeAnalysis (默认开启)
// 且 -XX:+EliminateLocks (默认开启)
// 则 JIT 会自动执行锁消除
```

### 7.2 锁粗化（Lock Coarsening）

当 JIT 检测到**连续的 synchronized 块操作同一个对象**时，会将多个锁合并为一个更大的锁，减少加锁解锁的开销。

```java
// 源码
public void method(Object lock) {
    synchronized(lock) {
        // 代码块1
        doSomething1();
    }
    synchronized(lock) {
        // 代码块2
        doSomething2();
    }
    synchronized(lock) {
        // 代码块3
        doSomething3();
    }
}

// JIT 编译后(锁粗化)
public void method(Object lock) {
    synchronized(lock) {
        doSomething1();
        doSomething2();
        doSomething3();
    }
}
```

```cpp
// C2编译器源码(简化)
void PhaseIdealLoop::coarsen_locks() {
    for (each adjacent lock region) {
        if (same_object && no_side_effect_between) {
            // 合并相邻的 monitorenter/monitorexit
            eliminate_inner_unlock_lock_pair();
        }
    }
}
// 需要 -XX:+EliminateLocks (默认开启)
```

### 7.3 适应性自旋（Adaptive Spinning）

```cpp
// objectMonitor.cpp 中的自旋
int ObjectMonitor::TrySpin(Thread * Self) {
    // 适应性自旋: 根据历史成功率调整
    // 如果之前自旋成功率高 → 增加自旋次数
    // 如果之前自旋成功率低 → 减少自旋次数甚至跳过
    int ctr = 0;
    int spins = Knob_FixedSpin;  // 固定自旋次数

    if (Knob_PreSpin > 0) {
        spins = Knob_PreSpin;  // 预自旋
    }

    for (ctr = 0; ctr < spins; ctr++) {
        // 尝试获取锁
        if (TryLock(Self) > 0) return 1;  // 成功

        // PAUSE 指令减少流水线开销
        SpinPause();
    }

    // 动态调整
    if (Knob_AdaptiveSpin) {
        // 根据上次是否成功调整
        if (prev_spin_succeeded) {
            // 增加自旋时间
            _SpinDuration = MIN2(_SpinDuration + 1, max_spin);
        } else {
            // 减少自旋时间
            _SpinDuration = MAX2(_SpinDuration - 1, 0);
        }
    }

    return 0; // 自旋失败
}
```

## 8. wait/notify 底层实现

### 8.1 wait() 源码

```cpp
// objectMonitor.cpp (简化)
void ObjectMonitor::wait(jlong millis, bool interruptible, TRAPS) {
    Thread * const Self = THREAD;

    // 检查当前线程是否持有锁
    if (Self != _owner) {
        if (!Self->is_lock_owned((address)_owner)) {
            throw IllegalMonitorStateException();
        }
    }

    // 保存重入次数
    intptr_t save = _recursions;
    _recursions = 0;

    // 释放锁
    exit(true, Self);

    // 创建 ObjectWaiter
    ObjectWaiter node(Self);
    node.TState = ObjectWaiter::TS_WAIT;

    // 加入 WaitSet 队列
    AddWaiter(&node);

    // park 线程，等待超时或被 notify 唤醒
    if (millis <= 0) {
        Self->_ParkEvent->park();  // 无限等待
    } else {
        Self->_ParkEvent->park(millis);  // 超时等待
    }

    // 被唤醒后，从 WaitSet 移除
    if (node.TState == ObjectWaiter::TS_WAIT) {
        DequeueSpecificWaiter(&node);
    }

    // 重新获取锁
    enter(THREAD);

    // 恢复重入次数
    _recursions = save;
}
```

### 8.2 notify() 源码

```cpp
// objectMonitor.cpp (简化)
void ObjectMonitor::notify(TRAPS) {
    Thread * const Self = THREAD;

    // 检查当前线程是否持有锁
    if (Self != _owner) {
        if (!Self->is_lock_owned((address)_owner)) {
            throw IllegalMonitorStateException();
        }
    }

    // 取 WaitSet 第一个元素
    ObjectWaiter * iterator = DequeueWaiter();
    if (iterator != NULL) {
        // 将状态改为 READY
        iterator->TState = ObjectWaiter::TS_READY;

        // 策略: 放入 EntryList 或 cxq
        // 默认放入 EntryList 头部
        if (Knob_MoveNotifyee == 0) {
            iterator->_next = _EntryList;
            _EntryList = iterator;
        } else if (Knob_MoveNotifyee == 1) {
            // 放入 cxq 头部
            iterator->_next = _cxq;
            _cxq = iterator;
        }

        // 注意: notify 不释放锁，只是将线程从 WaitSet 移到 EntryList
        // 被唤醒的线程需要等当前线程退出同步块后才能竞争锁
    }
}
```

### 8.3 wait/notify 流程图

```
线程调用 wait():
┌───────────┐     release lock      ┌───────────┐     park        ┌──────────┐
│  持有锁    │ ──────────────────→  │  释放锁    │ ────────────→  │ WaitSet  │
│  执行中    │                      │  保存状态  │                │  阻塞    │
└───────────┘                      └───────────┘                └──────────┘

线程调用 notify():
┌──────────┐   move to EntryList    ┌───────────┐   compete lock  ┌───────────┐
│ WaitSet  │ ──────────────────→   │ EntryList │ ────────────→  │  获取锁    │
│  阻塞    │   (状态改READY)        │  就绪     │                │  继续执行  │
└──────────┘                       └───────────┘                └───────────┘

完整时序:
  Thread-A (持有锁)        Thread-B (WaitSet)        Thread-C (EntryList)
       │                         │                         │
   wait()                    被 park                        │
   释放锁                     │                          获取锁
       │                    (在WaitSet)                   执行中
       │                         │                      notify()
       │                    移入EntryList                   │
       │                    被唤醒，竞争锁                  退出同步块
       │                    获取锁(或继续等待)               │
```

## 9. 面试题速查

### Q1: synchronized 和 Lock 的区别？

| 维度 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 实现 | JVM层面(monitorenter/monitorexit) | JDK API层面(AQS) |
| 锁释放 | 自动释放(退出同步块/异常) | 手动释放(finally中unlock) |
| 可中断 | 不可中断 | 可中断(lockInterruptibly) |
| 公平锁 | 非公平 | 支持公平/非公平 |
| 条件变量 | 单个(wait/notify) | 多个Condition |
| 锁绑定 | 对象/类 | Lock对象 |
| 响应中断 | 不支持 | 支持 |
| 尝试获取 | 不支持 | tryLock() |

### Q2: 偏向锁在 JDK 15 后为什么默认关闭？

```
原因：
1. 偏向锁的撤销需要到达安全点(STW)，造成停顿
2. 现代应用多线程使用更频繁，偏向锁带来的收益减小
3. 偏向锁增加了锁升级的复杂性，维护成本高
4. HashCode 计算与偏向锁冲突（一旦计算hashCode就不能偏向）
5. JEP 374: Disable and Deprecate Biased Locking
```

### Q3: synchronized 锁升级过程？

```
无锁 → 偏向锁(首次CAS设线程ID) → 轻量级锁(CAS设LockRecord)
→ 重量级锁(CAS失败+自旋耗尽 → ObjectMonitor)

关键点：
- 升级不可逆
- 偏向锁撤销需要安全点
- 轻量级锁自旋有自适应策略
- 重量级锁依赖操作系统互斥量
```

### Q4: 为什么 wait/notify 必须在 synchronized 块中调用？

```
原因：wait() 会释放锁，notify() 会将线程从 WaitSet 移到 EntryList。
如果没有持有锁，就无法正确释放锁和唤醒等待线程。
源码层面：ObjectMonitor::wait() 第一行就检查 _owner 是否是当前线程，
否则抛出 IllegalMonitorStateException。
```

### Q5: 锁消除和锁粗化的触发条件？

```
锁消除：
- JIT 编译时通过逃逸分析确定对象不逃逸
- -XX:+DoEscapeAnalysis + -XX:+EliminateLocks (默认开启)

锁粗化：
- JIT 检测到连续的 synchronized 块操作同一对象
- 合并为一个更大的同步块
- 减少加锁解锁次数
```

### Q6: Mark Word 中 hashCode 和偏向锁为什么冲突？

```
64位 Mark Word 中：
- hashCode 占31位
- 偏向线程ID 占54位

两者在 Mark Word 中的位置重叠，因此：
1. 对象调用 hashCode() 后，无法进入偏向锁状态
2. 偏向锁状态下调用 hashCode()，会撤销偏向锁升级为轻量级锁
3. 轻量级锁的 Lock Record 的 displaced_header 保存了原 hashCode
```

> 📝 **总结**：synchronized 的核心实现依赖于对象头 Mark Word 的状态转换。从偏向锁的"零成本"到轻量级锁的 CAS 自旋，再到重量级锁的操作系统阻塞，HotSpot 通过多级锁机制在不同竞争强度下自动选择最优策略。理解 Mark Word 布局和 ObjectMonitor 结构是掌握 synchronized 底层原理的关键。
