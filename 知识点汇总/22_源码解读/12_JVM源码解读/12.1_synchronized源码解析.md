# synchronized源码解析

> synchronized是Java中最基本也是最重要的并发原语。从JDK 6开始，JVM对synchronized做了大量优化——偏向锁、轻量级锁、重量级锁的升级机制，使其性能从"重量级锁"时代的大幅落后，到如今与ReentrantLock旗鼓相当。本文从Mark Word到锁升级全流程，深入解析synchronized的底层实现。

## 1. synchronized基础与使用

### 1.1 三种使用方式

```java
// 1. 修饰实例方法——锁当前实例对象
public synchronized void method() {
    // 同一实例串行执行，不同实例并行执行
}

// 2. 修饰静态方法——锁当前Class对象
public static synchronized void method() {
    // 所有实例串行执行
}

// 3. 同步代码块——锁指定对象
public void method() {
    synchronized (obj) {
        // 锁obj对象
    }
}
```

### 1.2 字节码层面的实现

```java
// 源码
public void method() {
    synchronized (obj) {
        System.out.println("hello");
    }
}

// 字节码

    0: aload_0                    // 加载this
    1: getfield      #2           // 获取obj字段
    4: dup                        // 复制引用

    6: monitorenter               // ← 进入monitor
    7: getstatic     #3           // System.out
   10: ldc           #4           // "hello"
   12: invokevirtual #5           // println
   15: aload_1                    // 加载obj
   16: monitorexit                // ← 正常退出monitor
   17: goto          25
   20: astore_2                   // 异常处理
   21: aload_1
   22: monitorexit                // ← 异常退出monitor
   23: aload_2
   24: athrow                     // 重新抛出异常
   25: return

// 异常表
// from  to  target  type
//    7  17    20    any        ← try块异常跳到20

```

### 1.3 同步方法的字节码

```java

    System.out.println("hello");
}

// 而是通过方法标志位 ACC_SYNCHRONIZED

    flags: ACC_PUBLIC, ACC_SYNCHRONIZED  // ← 方法标志

    3: ldc           #4
    5: invokevirtual #5

// 如果是同步方法，自动进入/退出monitor
```

## 2. 对象头与Mark Word详解

### 2.1 Java对象内存布局

```
┌─────────────────────────────────────────────────────┐
│                    对象头 (Object Header)             │
│  ┌───────────────────────┬───────────────────────┐  │
│  │   Mark Word (64 bits)  │  Class Pointer (32/64)│  │
│  └───────────────────────┴───────────────────────┘  │
│  [可选] 数组长度 (32 bits) —— 仅数组对象有              │
├─────────────────────────────────────────────────────┤
│                    实例数据 (Instance Data)           │
├─────────────────────────────────────────────────────┤
│                    对齐填充 (Padding)                 │
└─────────────────────────────────────────────────────┘
```

### 2.2 Mark Word结构（64位JVM）

Mark Word在不同锁状态下存储不同的信息：

```
┌──────────────────────────────────────────────────────────────────┬──────────┐
│                          Mark Word (64 bits)                      │ 锁状态    │
├──────────────────────────────────────────────────────────────────┼──────────┤
│ unused:25 | hash:31 | unused:1 | age:4 | biased:0 | lock:01     │ 无锁      │
├──────────────────────────────────────────────────────────────────┼──────────┤
│ thread:54 | epoch:2  | unused:1 | age:4 | biased:1 | lock:01     │ 偏向锁    │
├──────────────────────────────────────────────────────────────────┼──────────┤
│              ptr_to_lock_record:62              | lock:00         │ 轻量级锁  │
├──────────────────────────────────────────────────────────────────┼──────────┤
│           ptr_to_heavyweight_monitor:62          | lock:10         │ 重量级锁  │
├──────────────────────────────────────────────────────────────────┼──────────┤
│                                                | lock:11         │ GC标记    │
└──────────────────────────────────────────────────────────────────┴──────────┘

字段说明：
  hash        —— 对象的hashCode（31位）
  age         —— GC分代年龄（4位，最大15）
  biased      —— 是否偏向锁（1位）
  lock        —— 锁标志位（2位）
  thread      —— 偏向线程ID（54位）
  epoch       —— 偏向时间戳/版本号（2位）
  ptr_to_lock_record —— 指向栈中Lock Record的指针（62位）
  ptr_to_heavyweight_monitor —— 指向ObjectMonitor的指针（62位）
```

### 2.3 Mark Word的关键特性

```
1. 信息复用：64位空间在不同锁状态下存储不同信息
2. 锁标志位：01=无锁/偏向锁，00=轻量级锁，10=重量级锁，11=GC标记
3. 偏向标志：01且biased=1为偏向锁，01且biased=0为无锁
4. hashCode冲突：调用hashCode()会禁用偏向锁（hashCode需要Mark Word空间）
5. GC年龄：4位最大15，所以MaxTenuringThreshold最大15
```

## 3. Monitor与ObjectMonitor

### 3.1 Monitor概念

Monitor是一种同步机制，每个Java对象天生就是一个Monitor。当线程获取Monitor时，就获得了对象的锁。

### 3.2 ObjectMonitor结构

```cpp
// OpenJDK中的ObjectMonitor（C++）
class ObjectMonitor : public CHeapObj<mtObjectMonitor> {

    // 头部
    volatile markWord _header;    // 对象头（保存原始Mark Word）
    // 持有锁的线程
    void* volatile _owner;        // 指向持有Monitor的线程
    // 等待锁的线程队列（EntryList）
    ObjectWaiter* volatile _EntryList;
    // 等待通知的线程队列（WaitSet）
    ObjectWaiter* volatile _WaitSet;

    // 计数器
    volatile jint _count;         // 等待线程数
    volatile jint _waiters;       // WaitSet中的线程数
    volatile jint _recursions;    // 重入次数

    // 候选线程——用于公平锁优化
    Thread* volatile _succ;

    // 核心方法
    void enter(TRAPS);            // 进入Monitor（获取锁）
    void exit(bool not_suspended, TRAPS); // 退出Monitor（释放锁）
    void wait(jlong millis, bool interruptible, TRAPS); // wait()
    void notify(TRAPS);           // notify()
    void notifyAll(TRAPS);        // notifyAll()
};
```

### 3.3 EntryList与WaitSet

```
ObjectMonitor中的线程队列：

┌────────────────────────────────────────────────────┐
│                  ObjectMonitor                     │
│                                                    │
│  _owner ──→ Thread-A (持有锁)                      │
│                                                    │
│  _EntryList: [Thread-B] → [Thread-C] → [Thread-D] │
│  ↑ 等待获取锁的线程（BLOCKED状态）                     │
│                                                    │
│  _WaitSet: [Thread-E] → [Thread-F]                 │
│  ↑ 调用了wait()的线程（WAITING状态）                   │
│                                                    │
│  _recursions: 1 (Thread-A重入了一次)                 │
└────────────────────────────────────────────────────┘

线程状态转换：
  EntryList中的线程：BLOCKED
  WaitSet中的线程：WAITING
  _owner线程：RUNNABLE

notify(): 从WaitSet移一个线程到EntryList
notifyAll(): 从WaitSet移所有线程到EntryList
wait(): 从_owner移到WaitSet，释放锁
```

### 3.4 enter——获取锁

```cpp
void ObjectMonitor::enter(TRAPS) {
    Thread * const Self = THREAD;

    // 1. CAS尝试获取Monitor
    void * cur = Atomic::cmpxchg(&_owner, (void*)NULL, Self);
    if (cur == NULL) {
        // CAS成功——获取锁
        assert(_recursions == 0, "invariant");
        return;
    }

    // 2. 如果是重入
    if (cur == Self) {
        _recursions++;
        return;
    }

    // 3. CAS失败——进入自旋等待
    if (Self->is_lock_owned((address)cur)) {
        // 当前线程之前已经持有过（轻量级锁膨胀）
        _recursions = 1;
        _owner = Self;
        return;
    }

    // 4. 自旋失败——进入EntryList阻塞
    Self->_Stalled = intptr_t(this);
    // 尝试自旋
    if (Knob_SpinEarly && TrySpin(Self) > 0) {
        // 自旋成功——获取锁
        Self->_Stalled = 0;
        return;
    }

    // 5. 自旋也失败——封装为ObjectWaiter加入EntryList
    ObjectWaiter node(Self);
    node.TState = ObjectWaiter::TS_ENTER;
    // 加入EntryList
    AddWaiter(&node);
    Self->_Stalled = 0;

    // 6. 循环等待
    for (;;) {
        // park（线程阻塞）
        if (node._notified != 0 && node.TState == ObjectWaiter::TS_ENTER) {
            break;
        }
        // 线程挂起
        Self->_ParkEvent->park();
    }
}
```

## 4. 偏向锁

### 4.1 偏向锁的背景

```
现实观察：大多数锁不仅不存在多线程竞争，而且总是由同一个线程多次获得。

偏向锁的核心思想：
  第一次获取锁时，在Mark Word中记录线程ID
  后续同一线程获取锁时，只需检查线程ID匹配——无需CAS
  只有其他线程来竞争时，才撤销偏向锁
```

### 4.2 偏向锁获取流程

```cpp
// OpenJDK中的偏向锁获取（简化）
void ObjectSynchronizer::fast_enter(Handle obj, BasicLock* lock, TRAPS) {
    // 检查是否允许偏向锁
    if (UseBiasedLocking) {
        // 尝试获取偏向锁
        if (BiasedLocking::conditionally_revoke_and_rebias(obj, attempt_rebias, THREAD)) {
            return;
        }
    }
    // 偏向锁不可用——走轻量级锁
    slow_enter(obj, lock, THREAD);
}

// 偏向锁获取逻辑
bool BiasedLocking::conditionally_revoke_and_rebias(Handle obj, bool request_rebias, TRAPS) {
    markWord mark = obj->mark();

    // 1. 如果是可偏向状态（无锁 + biased=1）
    if (mark.has_bias_pattern()) {
        // 检查线程ID是否匹配
        if (mark.biased_locker() == THREAD) {
            // 同一线程——直接返回（无需CAS！）
            return true;
        }
    }

    // 2. 匿名偏向（thread=0）——CAS设置线程ID
    if (mark.is_biased_anonymously()) {
        markWord biased_marker = markWord::encode(
            (JavaThread*)THREAD,     // 线程ID
            mark.age(),
            mark.bias_epoch()
        );
        if (obj->cas_set_mark(biased_marker, mark) == mark) {
            return true; // CAS成功——获得偏向锁
        }
    }

    // 3. 被其他线程偏向——需要撤销
    return false; // 走撤销流程
}
```

### 4.3 偏向锁撤销

```cpp
// 偏向锁撤销——当其他线程尝试获取被偏向的锁时
void BiasedLocking::revoke(Handle obj, TRAPS) {
    markWord mark = obj->mark();
    JavaThread* biased_thread = mark.biased_locker();

    if (biased_thread == NULL) {
        // 匿名偏向——直接撤销
        obj->set_mark(markWord::prototype().set_age(mark.age()));
        return;
    }

    // 检查偏向线程是否存活
    if (!biased_thread->is_alive()) {
        // 线程已终止——直接撤销
        obj->set_mark(markWord::prototype().set_age(mark.age()));
        return;
    }

    // 检查偏向线程是否还在同步块中
    // 如果在——需要升级为轻量级锁
    // 如果不在——直接撤销回无锁
    markWord unbiased = markWord::prototype().set_age(mark.age());
    if (obj->cas_set_mark(unbiased, mark) == mark) {
        // 撤销成功
        return;
    }
    // CAS失败——升级为轻量级锁
}
```

### 4.4 批量重偏向与批量撤销

```
JVM对偏向锁撤销的优化：

批量重偏向（Bulk Rebias）：
  当某个类的偏向锁撤销次数超过阈值（默认20次）
  → JVM认为这些对象不适合偏向当前线程
  → 提升epoch值，使所有旧偏向锁失效
  → 对象可以重新偏向新线程

批量撤销（Bulk Revoke）：
  当某个类的偏向锁撤销次数超过更高阈值（默认40次）
  → JVM认为这个类不适合偏向锁
  → 禁用该类所有对象的偏向锁
  → 后续直接走轻量级锁

相关JVM参数：
  -XX:BiasedLockingBulkRebiasThreshold=20
  -XX:BiasedLockingBulkRevokeThreshold=40
  -XX:BiasedLockingDecayTime=25000
```

### 4.5 JDK 15+的偏向锁废弃

```
JEP 374: Disable and Deprecate Biased Locking (JDK 15)

原因：
1. 维护成本高——HotSpot中大量偏向锁代码增加维护负担
2. 收益降低——现代应用中synchronized竞争场景增多
3. 影响其他优化——偏向锁阻碍了C2编译器的锁优化

JDK 15+: 默认关闭偏向锁 (-XX:-UseBiasedLocking)
JDK 18+: 正式废弃
```

## 5. 轻量级锁

### 5.1 轻量级锁的背景

```
适用场景：多个线程交替执行同步块，没有真正的竞争

核心思想：
  不使用ObjectMonitor（不park线程）
  使用CAS操作在栈帧中创建Lock Record
  适合"短时间持有"的锁
```

### 5.2 Lock Record

```
线程栈帧中的Lock Record结构：

线程栈帧
┌──────────────────────┐
│  Lock Record         │
│  ┌──────────────────┐│
│  │ displaced_header ││ ← 保存对象原始Mark Word
│  │ (mark word copy) ││
│  ├──────────────────┤│
│  │ object pointer   ││ ← 指向锁对象
│  └──────────────────┘│
├──────────────────────┤
│  其他栈帧数据          │
└──────────────────────┘
```

### 5.3 轻量级锁加锁流程

```cpp
void ObjectSynchronizer::slow_enter(Handle obj, BasicLock* lock, TRAPS) {
    markWord mark = obj->mark();

    // 1. 如果是无锁状态
    if (mark.is_unlocked()) {
        // 在栈帧中保存原始Mark Word
        lock->set_displaced_header(mark);

        // CAS：将对象的Mark Word替换为指向Lock Record的指针
        markWord locked = markWord::encode(lock);
        if (obj->cas_set_mark(locked, mark) == mark) {
            // CAS成功——获得轻量级锁
            return;
        }
        // CAS失败——有竞争，走锁膨胀
    }

    // 2. 如果是当前线程重入
    if (mark.has_locker() && THREAD->is_lock_owned((address)mark.locker())) {
        // 重入——不需要CAS
        lock->set_displaced_header(markWord::from_pointers(NULL));
        return;
    }

    // 3. 竞争失败——锁膨胀为重量级锁
    lock->set_displaced_header(markWord::unused_mark());
    inflate(THREAD, obj(), inflate_cause_monitor_enter)->enter(THREAD);
}
```

### 5.4 轻量级锁解锁流程

```cpp
void ObjectSynchronizer::fast_exit(oop object, BasicLock* lock, TRAPS) {
    markWord mark = object->mark();
    markWord dhw = lock->displaced_header();

    // 1. 如果是无锁状态（可能是重入锁）
    if (dhw.is_neutral()) {
        // CAS：恢复对象的Mark Word为原始值
        if (object->cas_set_mark(dhw, mark) == mark) {
            // CAS成功——释放成功
            return;
        }
        // CAS失败——有竞争，锁已膨胀为重量级锁
    }

    // 2. 锁已膨胀——走重量级锁释放
    ObjectSynchronizer::inflate(THREAD, object, inflate_cause_vm_internal)->exit(true, THREAD);
}
```

### 5.5 轻量级锁示意图

```
加锁前：
  对象头: [hash:31 | age:4 | biased:0 | lock:01]  ← 无锁
  线程栈: [empty]

加锁CAS成功后：
  对象头: [ptr_to_lock_record:62 | lock:00]        ← 轻量级锁
  线程栈: [Lock Record: {displaced_header=原始mark, obj=对象}]

解锁CAS成功后：
  对象头: [hash:31 | age:4 | biased:0 | lock:01]  ← 恢复无锁
  线程栈: [Lock Record cleared]
```

## 6. 重量级锁

### 6.1 重量级锁的背景

```
适用场景：多线程同时竞争同一把锁

核心机制：
  使用ObjectMonitor（操作系统互斥量mutex）
  未获取锁的线程进入EntryList，被park阻塞
  释放锁时unpark唤醒等待线程

代价：
  线程阻塞/唤醒需要用户态↔内核态切换
  每次切换开销约1-10微秒
```

### 6.2 锁膨胀

```cpp
// 当轻量级锁CAS失败时，膨胀为重量级锁
ObjectMonitor* ObjectSynchronizer::inflate(Thread * self, oop object,

    for (;;) {
        markWord mark = object->mark();

        // 1. 已经是重量级锁——直接返回
        if (mark.has_monitor()) {
            return mark.monitor();
        }

        // 2. 无锁状态——直接膨胀
        if (mark.is_unlocked()) {
            // 创建ObjectMonitor
            ObjectMonitor * m = ObjectSynchronizer::get_monitor(...);
            // 设置Monitor的header为原始Mark Word
            m->set_header(mark);
            // CAS：将对象头替换为指向Monitor的指针
            if (object->cas_set_mark(markWord::encode(m), mark) == mark) {
                return m;
            }
        }

        // 3. 轻量级锁状态——需要先释放轻量级锁再膨胀
        if (mark.has_locker()) {
            // 唤醒持有轻量级锁的线程释放锁
            // 然后膨胀为重量级锁
            // ...
        }
    }
}
```

### 6.3 重量级锁获取

```cpp
// ObjectMonitor::enter（简化）
void ObjectMonitor::enter(TRAPS) {
    Thread * const Self = THREAD;

    // 1. CAS尝试获取
    if (Atomic::cmpxchg(&_owner, (void*)NULL, Self) == NULL) {
        return; // 成功
    }

    // 2. 重入检查
    if (_owner == Self) {
        _recursions++;
        return;
    }

    // 3. 自旋等待（自适应自旋）
    if (TrySpin(Self) > 0) {
        return; // 自旋成功
    }

    // 4. 进入EntryList阻塞
    ObjectWaiter node(Self);
    node.TState = ObjectWaiter::TS_ENTER;
    AddWaiter(&node);

    // 循环等待——park阻塞
    for (;;) {
        Self->_ParkEvent->park();
        if (TryLock(Self) > 0) break;
    }
}
```

## 7. 锁升级全流程

### 7.1 完整升级链路

```
┌────────────┐     第一次获取     ┌────────────┐    不同线程竞争    ┌────────────┐    CAS失败      ┌────────────┐
│   无锁      │ ──────────────→  │   偏向锁     │ ──────────────→ │  轻量级锁   │ ──────────────→│  重量级锁   │
│ lock:01    │                  │ lock:01    │                  │ lock:00    │                 │ lock:10    │
│ biased:0   │                  │ biased:1   │                  │            │                 │            │
│ hash:31    │                  │ thread:54  │                  │ ptr:62     │                 │ ptr:62     │
└────────────┘                  └────────────┘                  └────────────┘                 └────────────┘
                                     ↑                                  ↑                              │
                                     │    同一线程再次获取                    │  交替执行                       │
                                     └──────────────────────────────────┘                              │  释放锁后
                                                                                                        │  可能降级
                                                                                                        ↓ (GC时)
```

### 7.2 升级触发条件

```
无锁 → 偏向锁：第一次获取锁（JDK 15-默认开启，JDK 15+默认关闭）

偏向锁 → 轻量级锁：其他线程尝试获取被偏向的锁
  - 偏向线程不在同步块中：撤销偏向，升级为轻量级锁
  - 偏向线程在同步块中：其他线程CAS失败，升级为轻量级锁

轻量级锁 → 重量级锁：CAS竞争失败
  - 自旋失败后，膨胀为重量级锁
  - 线程进入EntryList阻塞

注意：锁只能升级，不能降级（GC时可以降级为无锁）
```

### 7.3 完整加锁流程图

```java
// 伪代码——完整的synchronized加锁流程
public void synchronizedBlock(Object obj) {
    // 1. 检查Mark Word的锁标志位
    MarkWord mark = obj.markWord;

    if (mark.isBiased()) {
        // 2. 偏向锁检查
        if (mark.threadId == currentThread.id) {
            // 2a. 同一线程——直接进入（无CAS）
            return;
        } else {
            // 2b. 不同线程——撤销偏向锁
            revokeBias(obj);
            // 走3
        }
    }

    if (mark.isUnlocked()) {
        // 3. 无锁——尝试轻量级锁
        LockRecord lr = new LockRecord(mark);
        if (CAS(obj.markWord, mark, lr.pointer)) {
            // 3a. CAS成功——获得轻量级锁
            return;
        }
        // 3b. CAS失败——有竞争，走4
    }

    if (mark.isLightweight()) {
        // 4. 轻量级锁竞争
        if (mark.lockRecord.owner == currentThread) {
            // 4a. 重入
            return;
        }
        // 4b. 自旋等待
        for (int i = 0; i < spinLimit; i++) {
            if (tryCAS(obj)) return;
        }
        // 4c. 自旋失败——膨胀为重量级锁
    }

    // 5. 重量级锁
    ObjectMonitor monitor = inflate(obj);
    monitor.enter(); // 进入EntryList，可能park
}
```

## 8. 锁消除与锁粗化

### 8.1 锁消除

```java
// JIT编译器的优化——通过逃逸分析消除不必要的锁

public String concat(String s1, String s2) {
    // StringBuffer是线程安全的，内部有synchronized
    StringBuffer sb = new StringBuffer();
    sb.append(s1);
    sb.append(s2);
    return sb.toString();
}

// JIT逃逸分析发现sb不会逃逸出方法（不会被其他线程访问）
// 因此synchronized可以安全消除

// JIT编译后的等效代码
public String concat(String s1, String s2) {
    StringBuffer sb = new StringBuffer();
    sb.appendWithoutSync(s1);  // ← 锁消除
    sb.appendWithoutSync(s2);  // ← 锁消除
    return sb.toString();
}
```

### 8.2 锁粗化

```java
// JIT编译器将多个相邻的synchronized块合并为一个

// 优化前
public void method(Object obj) {
    synchronized (obj) {
        // do something
    }
    // 这里没有其他代码
    synchronized (obj) {
        // do something else
    }
}

// 优化后——锁粗化
public void method(Object obj) {
    synchronized (obj) {
        // do something
        // do something else
    }
}

// StringBuffer连续append
public String method() {
    StringBuffer sb = new StringBuffer();
    for (int i = 0; i < 100; i++) {
        sb.append(i); // 每次append都synchronized
    }
    return sb.toString();
}

// 优化后——100次append的锁合并为1次
public String method() {
    StringBuffer sb = new StringBuffer();
    synchronized (sb) { // 只加锁一次
        for (int i = 0; i < 100; i++) {
            sb.appendWithoutSync(i);
        }
    }
    return sb.toString();
}
```

## 9. synchronized vs ReentrantLock

### 9.1 核心区别

| 特性 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 实现层面 | JVM内置（monitorenter/monitorexit） | JDK API（AQS） |
| 锁释放 | 自动释放（退出同步块/异常） | 手动释放（finally中unlock） |
| 可中断 | 不可中断 | 可中断（lockInterruptibly） |
| 公平锁 | 非公平 | 支持公平/非公平 |
| 条件变量 | 1个（wait/notify） | 多个（Condition） |
| 尝试获取 | 不支持 | tryLock() |
| 锁绑定 | 对象 | Lock对象 |
| 锁状态 | 不可查询 | isLocked(), getHoldCount() |

### 9.2 性能对比

```
JDK 5之前：
  synchronized性能远差于ReentrantLock（重量级锁实现）

JDK 6+：
  偏向锁+轻量级锁优化后，synchronized性能大幅提升
  无竞争场景：synchronized更快（JVM优化好）
  有竞争场景：ReentrantLock略快（AQS优化更细）

结论：
  简单场景用synchronized（简单、不会忘记释放锁）
  高级功能用ReentrantLock（可中断、超时、多Condition）
```

### 9.3 选择建议

```java
// 场景1：简单同步——用synchronized
public synchronized void method() { ... }

// 场景2：需要可中断——用ReentrantLock
ReentrantLock lock = new ReentrantLock();
try {
    lock.lockInterruptibly();
    // ...
} catch (InterruptedException e) {
    // 被中断
} finally {
    lock.unlock();
}

// 场景3：需要超时——用ReentrantLock
if (lock.tryLock(3, TimeUnit.SECONDS)) {
    try {
        // ...
    } finally {
        lock.unlock();
    }
} else {
    // 获取超时
}

// 场景4：多条件变量——用ReentrantLock
ReentrantLock lock = new ReentrantLock();
Condition notEmpty = lock.newCondition();
Condition notFull = lock.newCondition();
```

## 10. 自旋锁与自适应自旋

### 10.1 自旋锁

```
自旋锁：当线程获取锁失败时，不立即阻塞（park），而是循环重试（spin）

优点：避免线程切换开销（用户态↔内核态）
缺点：消耗CPU（如果锁持有时间长）

适用场景：锁持有时间很短，自旋能快速获取
```

### 10.2 自适应自旋

```cpp
// JDK 6引入自适应自旋（Adaptive Spinning）
// 根据历史成功率动态调整自旋次数

int ObjectMonitor::TrySpin(Thread * const Self) {
    // 获取自旋次数限制
    int ctr = 0;
    int spins = 0;
    int delay = 0;

    // 自适应：如果之前自旋成功率高，增加自旋次数
    //         如果之前自旋成功率低，减少甚至跳过自旋
    int prev_spin_count = Self->_SpinStart;
    int spin_limit = adjust_spin_limit(prev_spin_count);

    // 自旋循环
    for (ctr = 0; ctr < spin_limit; ctr++) {
        // 尝试CAS获取锁
        if (TryLock(Self) > 0) {
            // 成功
            Self->_SpinSuccess++;
            return 1;
        }

        // 自旋等待
        if (ctr >= spins) {
            // 增加延迟（指数退避）
            delay = adjust_delay(delay);
            SpinPause();
        }
    }

    // 自旋失败
    Self->_SpinFail++;
    return 0;
}
```

### 10.3 相关JVM参数

```
-XX:+UseSpinning           # 开启自旋（JDK 6+默认开启）
-XX:PreBlockSpin=10        # 自旋次数（JDK 6前，固定值）
                             # JDK 6+使用自适应自旋，此参数忽略

# 偏向锁相关
-XX:+UseBiasedLocking      # 开启偏向锁（JDK 15默认关闭）
-XX:BiasedLockingStartupDelay=4000  # 偏向锁启动延迟（毫秒）

# 锁膨胀相关
-XX:+UseHeavyMonitors      # 禁用偏向锁和轻量级锁（调试用）
```

## 11. 面试题速查

### Q1: synchronized的底层原理是什么？

synchronized基于Monitor实现。同步代码块使用monitorenter/monitorexit字节码指令，同步方法使用ACC_SYNCHRONIZED标志位。每个对象关联一个ObjectMonitor，包含_owner（持有线程）、_EntryList（等待队列）、_WaitSet（等待通知队列）。获取锁就是获取Monitor的_owner。

### Q2: synchronized的锁升级过程是什么？

无锁→偏向锁→轻量级锁→重量级锁。第一次获取锁时进入偏向锁（记录线程ID）。其他线程竞争时撤销偏向，升级为轻量级锁（CAS+Lock Record）。CAS竞争失败自旋也失败后，膨胀为重量级锁（ObjectMonitor+park）。锁只能升级不能降级。

### Q3: 偏向锁、轻量级锁、重量级锁的区别？

偏向锁：记录线程ID，同一线程无需CAS，适合单线程。轻量级锁：CAS竞争，无竞争时无需系统调用，适合交替执行。重量级锁：使用ObjectMonitor，线程park阻塞，适合真正竞争。开销递增。

### Q4: Mark Word中存了什么？

64位Mark Word在不同锁状态存储不同信息：无锁状态存hashCode(31位)+GC年龄(4位)+偏向标志(1位)+锁标志(2位)。偏向锁状态存线程ID(54位)+epoch(2位)。轻量级锁存Lock Record指针(62位)。重量级锁存ObjectMonitor指针(62位)。

### Q5: 为什么调用hashCode()会影响偏向锁？

hashCode()需要将hashCode存储在Mark Word的31位空间中，但偏向锁状态这31位用于存储线程ID。两者冲突。调用hashCode()后，偏向锁被撤销，对象无法再进入偏向锁状态，直接从轻量级锁开始。

### Q6: 锁消除和锁粗化是什么？

锁消除：JIT通过逃逸分析发现锁对象不会被其他线程访问，直接消除synchronized。锁粗化：JIT将多个相邻的synchronized块合并为一个，减少锁获取/释放次数。两者都是JIT编译优化，不需要开发者干预。

### Q7: synchronized和ReentrantLock的区别？

synchronized是JVM内置的，自动释放锁，不可中断，非公平。ReentrantLock是API层面的（基于AQS），手动释放锁，可中断，支持公平锁，支持tryLock超时获取，支持多Condition。简单场景用synchronized，高级功能用ReentrantLock。

### Q8: 什么是自适应自旋？

JIT根据历史自旋成功率动态调整自旋次数。如果之前自旋成功率高（锁持有时间短），增加自旋次数。如果成功率低（锁持有时间长），减少甚至跳过自旋。避免固定自旋次数在某些场景下浪费CPU。

### Q9: 偏向锁为什么在JDK 15被废弃？

维护成本高（HotSpot大量代码）、收益降低（现代应用竞争增多）、阻碍其他JVM优化（如C2编译器的锁优化）、硬件变化（CAS指令性能提升使得轻量级锁开销已足够低）。

### Q10: synchronized异常时会释放锁吗？

会。字节码层面，synchronized同步块编译后有异常表，异常时会执行monitorexit释放锁再athrow。这也是synchronized比ReentrantLock更安全的地方——不会因为忘记unlock导致死锁。ReentrantLock必须在finally中手动unlock。
