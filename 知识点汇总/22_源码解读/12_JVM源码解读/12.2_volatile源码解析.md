# volatile源码解析

> volatile是Java中轻量级的同步机制。它保证可见性、有序性，但不保证原子性。volatile的底层依赖内存屏障和CPU缓存一致性协议（MESI）。理解volatile是理解Java并发编程、DCL单例、AQS等底层机制的基础。

## 1. volatile的基本语义

### 1.1 volatile的两个保证

```
1. 可见性（Visibility）
   一个线程修改了volatile变量，其他线程能立即看到新值
   
2. 有序性（Ordering）
   volatile变量的读写操作不会被编译器和CPU重排序到屏障之外
   
3. 不保证原子性
   volatile变量的复合操作（如i++）不是原子的
```

### 1.2 volatile的基本使用

```java
public class VolatileDemo {

    // volatile标志位——保证可见性
    private volatile boolean running = true;

    public void stop() {
        running = false; // 其他线程能立即看到
    }

    public void run() {
        while (running) {
            // 如果running不是volatile，线程可能永远看不到running=false
            // 因为JIT可能将running缓存到寄存器中
            doWork();
        }
        System.out.println("Stopped");
    }

    private void doWork() {
        // ...
    }
}
```

### 1.3 没有volatile的问题

```java
public class VisibilityProblem {

    private boolean ready = false;  // 没有volatile！

    public void producer() {
        ready = true; // 写线程
    }

    public void consumer() {
        while (!ready) {
            // 可能永远循环下去！
            // 因为读线程可能从CPU缓存读取旧值
            // 看不到ready=true
        }
        System.out.println("Ready!");
    }

    public static void main(String[] args) throws Exception {
        VisibilityProblem demo = new VisibilityProblem();

        new Thread(demo::consumer).start();
        Thread.sleep(100);
        new Thread(demo::producer).start();
    }
}
```

## 2. JMM与happens-before

### 2.1 Java内存模型（JMM）

```
JMM定义了线程和主内存之间的抽象关系：

┌──────────┐         ┌──────────┐         ┌──────────┐
│ Thread A │         │ Thread B │         │ Thread C │
│ ┌──────┐ │         │ ┌──────┐ │         │ ┌──────┐ │
│ │ 工作  │ │         │ │ 工作  │ │         │ │ 工作  │ │
│ │ 内存  │ │         │ │ 内存  │ │         │ │ 内存  │ │
│ └──┬───┘ │         │ └──┬───┘ │         │ └──┬───┘ │
└────┼─────┘         └────┼─────┘         └────┼─────┘
     │                    │                    │
     └────────────────────┼────────────────────┘
                          │
                    ┌─────┴─────┐
                    │  主内存    │
                    │ (Main     │
                    │  Memory)  │
                    └───────────┘

每个线程有自己的工作内存（CPU缓存/寄存器）
线程不能直接读写主内存，必须通过工作内存中转
```

### 2.2 happens-before规则

```
happens-before定义了操作之间的可见性传递关系：
如果A happens-before B，则A的结果对B可见

8条happens-before规则：

1. 程序顺序规则：同一线程中，前面的操作happens-before后面的操作
2. 监视器锁规则：unlock操作happens-before后续对同一锁的lock操作
3. volatile变量规则：volatile写操作happens-before后续对同一变量的读操作
4. 线程启动规则：Thread.start() happens-before该线程的所有操作
5. 线程终止规则：线程的所有操作happens-before Thread.join()返回
6. 线程中断规则：interrupt()调用happens-before被中断线程检测到中断
7. 对象终结规则：构造函数执行结束happens-before finalize()方法
8. 传递性：A happens-before B，B happens-before C → A happens-before C
```

### 2.3 volatile的happens-before

```java
// volatile规则：volatile写 happens-before 后续volatile读

int a = 0;         // 普通变量
volatile boolean flag = false;  // volatile变量

// 线程A
a = 1;             // 1. 普通写
flag = true;       // 2. volatile写

// 线程B
if (flag) {        // 3. volatile读
    int b = a;     // 4. 普通读 → b一定等于1
}

// 为什么b=1？
// 1 happens-before 2（程序顺序规则）
// 2 happens-before 3（volatile规则）
// 3 happens-before 4（程序顺序规则）
// 传递性：1 happens-before 4 → a=1对b=a可见
// 所以b一定等于1

// volatile的"内存语义增强"：
// volatile写之前的所有操作，对volatile读之后的操作可见
```

## 3. 内存屏障

### 3.1 四种内存屏障

```
内存屏障（Memory Barrier）是CPU指令，阻止屏障两侧的指令重排序：

1. LoadLoad屏障
   Load1; LoadLoad; Load2
   保证Load1在Load2之前完成

2. StoreStore屏障
   Store1; StoreStore; Store2
   保证Store1在Store2之前完成（且Store1刷新到主内存）

3. LoadStore屏障
   Load1; LoadStore; Store2
   保证Load1在Store2之前完成

4. StoreLoad屏障
   Store1; StoreLoad; Load2
   保证Store1对其他处理器可见后再执行Load2
   这是最强的屏障，开销最大
```

### 3.2 volatile的屏障插入策略

```
JMM对volatile的内存屏障插入策略：

 volatile写之前：插入StoreStore屏障
   → 保证前面的普通写已刷新到主内存

 volatile写之后：插入StoreLoad屏障
   → 保证volatile写对后续读可见

 volatile读之后：插入LoadLoad屏障
   → 防止后面的读重排序到volatile读之前

 volatile读之后：插入LoadStore屏障
   → 防止后面的写重排序到volatile读之前

屏障布局：
                    StoreStore屏障
                    volatile写
                    StoreLoad屏障
                    
                    volatile读
                    LoadLoad屏障
                    LoadStore屏障
```

### 3.3 屏障的CPU实现

```java
// 不同CPU架构的内存屏障实现不同

// x86架构（强内存模型——天然保证大部分有序性）
// x86只需要StoreLoad屏障（其他三种隐式保证）
// StoreLoad屏障通过 lock addl $0,0(%rsp) 或 mfence 指令实现

// ARM架构（弱内存模型——需要全部四种屏障）
// dmb 指令（Data Memory Barrier）

// Java通过Unsafe类的putOrderedObject/putVolatile等方法
// 在不同CPU上生成对应的屏障指令

// HotSpot中的实现
// orderAccess.hpp
inline void OrderAccess::storeload() {
    // x86: mfence 或 lock addl
    __asm__ volatile ("lock; addl $0,0(%%rsp)" : : : "cc", "memory");
}

inline void OrderAccess::release() {
    // StoreStore + LoadStore
    // x86上不需要额外指令（天然保证）
}

inline void OrderAccess::fence() {
    // 全屏障
    __asm__ volatile ("mfence" : : : "memory");
}
```

### 3.4 volatile写的汇编层面

```java
// Java代码
volatile boolean flag = true;

// JIT编译后的汇编代码（x86）
// 可以看到volatile写后面有lock前缀指令
0x0000000002931350: mov    $0x1,%dl
0x0000000002931352: mov    %dl,0x68(%rsi)      ; 写入flag
0x0000000002931355: lock   addl $0x0,(%rsp)    ; ← StoreLoad屏障！
                                                ; lock前缀确保写操作
                                                ; 对其他CPU可见

// lock指令的作用：
// 1. 锁定缓存行（或总线）
// 2. 将当前CPU的Store Buffer刷入L1 Cache
// 3. 通过MESI协议使其他CPU的缓存行失效
// 4. 阻止后续指令重排序到lock之前
```

## 4. MESI缓存一致性协议

### 4.1 MESI四种状态

```
MESI是CPU缓存一致性协议，每个缓存行有四种状态：

M (Modified)  —— 已修改：缓存行被修改，与主内存不一致
                  只有当前CPU有副本

E (Exclusive) —— 独占：缓存行与主内存一致，只有当前CPU有副本

S (Shared)    —— 共享：缓存行与主内存一致，多个CPU有副本

I (Invalid)   —— 无效：缓存行无效，需要重新从主内存读取

状态转换图：
                 读miss(独占)
    ┌──────────────────────────┐
    ↓                           │
┌─────────┐  其他CPU读   ┌─────────┐
│Exclusive│────────────→│ Shared  │
│    E    │←────────────│    S    │
└────┬────┘  其他CPU写   └────┬────┘
     │ 写                      │ 写miss
     ↓                         ↓
┌─────────┐  其他CPU读   ┌─────────┐
│Modified │────────────→│Invalid  │
│    M    │             │    I    │
└─────────┘             └─────────┘
```

### 4.2 MESI与volatile的关系

```
volatile写的可见性保证流程：

1. CPU-A写volatile变量
   M状态缓存行被修改

2. CPU-A执行lock指令
   Store Buffer刷入L1 Cache
   通过MESI协议发送Invalidate消息给其他CPU

3. CPU-B的缓存行变为I（Invalid）状态

4. CPU-B读volatile变量
   发现缓存行是I状态——Cache Miss
   从CPU-A或主内存读取最新值
   获得最新值——可见性保证！
```

### 4.3 Store Buffer与Invalidate Queue

```
MESI的性能问题：
  CPU-A写数据时，需要等所有CPU确认Invalidate消息后才写入
  这个等待时间可能很长（100+ns）

优化1: Store Buffer
  CPU-A写数据时，先写入Store Buffer，不等确认
  继续执行后续指令
  后续读操作需要先检查Store Buffer（Store Forwarding）

优化2: Invalidate Queue
  CPU-B收到Invalidate消息时，先放入Invalidate Queue
  立即回复ACK
  后续异步处理Invalidate Queue

副作用：
  Store Buffer和Invalidate Queue引入了"重排序"风险
  → 这就是为什么需要内存屏障！

内存屏障的作用：
  StoreStore屏障：刷新Store Buffer
  StoreLoad屏障：刷新Store Buffer + 等待Invalidate Queue处理完
  LoadLoad屏障：处理Invalidate Queue
  LoadStore屏障：处理Invalidate Queue
```

### 4.4 Store Buffer与重排序

```
// Store Buffer导致的重排序

// CPU-A执行：
int data = 1;       // Store1: 写data到Store Buffer
boolean ready = true; // Store2: 写ready到Store Buffer
                      // 由于Store Buffer，Store2可能先于Store1刷入Cache

// CPU-B执行：
if (ready) {        // Load1: 读ready（可能读到true）
    int r = data;   // Load2: 读data（可能读到0！因为data还没刷入Cache）
}

// 这就是为什么需要StoreStore屏障
// volatile boolean ready = true; 会在Store2前插入StoreStore屏障
// 保证data=1先于ready=true刷入Cache
```

## 5. volatile的可见性实现

### 5.1 volatile写的可见性

```java
// volatile写的实现流程

// Java层面
volatile int x = 1;

// JVM层面（JIT编译后）
// 1. 将1写入x的内存位置
// 2. 插入StoreLoad屏障（lock指令）

// CPU层面
// 1. 将1写入Store Buffer
// 2. lock指令触发：
//    a. 将Store Buffer刷入L1 Cache
//    b. 通过MESI协议使其他CPU的x缓存行变为Invalid
//    c. 等待所有Invalidate ACK

// 其他CPU读取x时：
// 1. 发现x的缓存行是Invalid状态
// 2. Cache Miss——从其他CPU或主内存获取最新值
// 3. 读到1——可见性保证
```

### 5.2 volatile读的可见性

```java
// volatile读的实现流程

// Java层面
int y = x; // x是volatile

// JVM层面
// 1. 插入LoadLoad屏障
// 2. 从主内存（或通过MESI从其他CPU Cache）读取x
// 3. 插入LoadStore屏障

// CPU层面
// 1. 处理Invalidate Queue（确保之前的Invalidate被应用）
// 2. 读取x的缓存行
//    a. 如果是Valid状态——从Cache读
//    b. 如果是Invalid状态——Cache Miss，从其他CPU/主内存获取最新值
// 3. 获得最新值

// volatile读的"强制刷新"：
// LoadLoad屏障确保volatile读不会被重排序到前面的读之后
// LoadStore屏障确保volatile读不会被重排序到后面的写之前
```

### 5.3 可见性验证实验

```java
public class VisibilityTest {

    // 不加volatile——可能看不到stop
    // private boolean stop = false;

    // 加volatile——保证可见性
    private volatile boolean stop = false;

    public static void main(String[] args) throws Exception {
        VisibilityTest test = new VisibilityTest();

        Thread reader = new Thread(() -> {
            while (!test.stop) {
                // 如果stop不是volatile
                // JIT可能优化为 if (!test.stop) while(true) {}
                // 一次读取后不再读主内存
            }
            System.out.println("Reader stopped");
        });
        reader.start();

        Thread.sleep(1000);

        test.stop = true;
        System.out.println("Stop set to true");
        reader.join();
    }
}
```

## 6. volatile的有序性实现

### 6.1 指令重排序

```
编译器和CPU为了提高性能，会对指令进行重排序：

// 源代码
int a = 1;   // 语句1
int b = 2;   // 语句2
int c = a + b; // 语句3

// 重排序后可能
int b = 2;   // 语句2先执行
int a = 1;   // 语句1后执行
int c = a + b; // 语句3最后

// 单线程下重排序不影响结果（as-if-serial语义）
// 多线程下可能导致问题
```

### 6.2 重排序导致的问题

```java
public class ReorderingProblem {

    private int x = 0;
    private boolean initialized = false; // 没有volatile

    // 线程A
    public void init() {
        x = 42;           // 1. 写x
        initialized = true; // 2. 写initialized
        // 这两步可能被重排序！
        // 编译器/CPU可能先写initialized=true，再写x=42
    }

    // 线程B
    public void check() {
        if (initialized) {  // 3. 读initialized
            int r = x;      // 4. 读x → r可能为0！
            System.out.println("x = " + r);
        }
    }
}

// 解决方案
public class FixedExample {
    private int x = 0;
    private volatile boolean initialized = true; // volatile！

    public void init() {
        x = 42;              // 1. 普通写
        initialized = true;  // 2. volatile写
        // StoreStore屏障保证1在2之前执行
    }

    public void check() {
        if (initialized) {   // 3. volatile读
            // LoadLoad屏障保证3在4之前执行
            int r = x;       // 4. 普通读 → r一定为42
        }
    }
}
```

### 6.3 volatile禁止重排序的规则

```
volatile读/写的重排序规则：

           第二操作
           普通读/写   volatile读   volatile写
第一操作
普通读/写     ✅           ✅          ✅
volatile读   ❌(LL)       ❌(LL)      ❌(LS)
volatile写   ✅           ❌(SL)      ❌(SS)

✅ = 允许重排序
❌ = 禁止重排序（括号内为需要的屏障）

关键规则：
1. volatile写之前的操作不会被重排序到volatile写之后（StoreStore）
2. volatile读之后的操作不会被重排序到volatile读之前（LoadLoad, LoadStore）
3. volatile写之后的volatile读不会重排序（StoreLoad）
4. volatile读之后的volatile读不会重排序（LoadLoad）
```

## 7. volatile不保证原子性

### 7.1 i++不是原子操作

```java
public class AtomicityTest {

    private volatile int count = 0;

    public void increment() {
        count++; // 不是原子操作！
    }

    // count++ 实际上是三步操作：
    // 1. 读取count的值到工作内存（volatile保证读到最新值）
    // 2. count + 1
    // 3. 写回count（volatile保证写后可见）
    // 但1和2之间可能有其他线程修改了count！

    public static void main(String[] args) throws Exception {
        AtomicityTest test = new AtomicityTest();

        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                for (int j = 0; j < 1000; j++) {
                    test.increment();
                }
            }).start();
        }

        Thread.sleep(2000);
        System.out.println(test.count); // 结果可能小于10000！
    }
}
```

### 7.2 竞争场景分析

```
线程A和线程B同时执行count++（count初始为0）：

时间线：
T1: 线程A读count=0
T2: 线程B读count=0（volatile保证读到最新值0）
T3: 线程A计算0+1=1
T4: 线程B计算0+1=1
T5: 线程A写count=1
T6: 线程B写count=1（覆盖了线程A的写！）

结果：count=1，而不是2
虽然volatile保证了可见性，但读-改-写不是原子的
```

### 7.3 正确的原子方案

```java
// 方案1：AtomicInteger
private AtomicInteger count = new AtomicInteger(0);

public void increment() {
    count.incrementAndGet(); // CAS保证原子性
}

// 方案2：synchronized
private int count = 0;

public synchronized void increment() {
    count++;
}

// 方案3：ReentrantLock
private final ReentrantLock lock = new ReentrantLock();
private int count = 0;

public void increment() {
    lock.lock();
    try {
        count++;
    } finally {
        lock.unlock();
    }
}

// 方案4：LongAdder（高并发推荐）
private LongAdder count = new LongAdder();

public void increment() {
    count.increment();
}
```

### 7.4 volatile适用的场景

```java
// volatile适合"一写多读"的场景

// 场景1：状态标志位
private volatile boolean shutdown = false;
// 一个线程写shutdown=true，多个线程读shutdown

// 场景2：DCL单例
private volatile static Singleton instance;
// 一个线程创建实例，其他线程读取

// 场景3：一次性安全发布
private volatile Configuration config;
// 一个线程初始化config，其他线程读取

// volatile不适合的场景
// 1. i++（复合操作）
// 2. 多线程写同一变量（需要原子性）
// 3. 复杂的状态依赖（需要锁）
```

## 8. 双重检查锁定（DCL）

### 8.1 DCL单例模式

```java
public class Singleton {

    // volatile关键！防止指令重排序
    private volatile static Singleton instance;

    private Singleton() {
        // 防止反射攻击
        if (instance != null) {
            throw new RuntimeException("Use getInstance()");
        }
    }

    public static Singleton getInstance() {
        if (instance == null) {                  // 1. 第一次检查（无锁）
            synchronized (Singleton.class) {     // 2. 加锁
                if (instance == null) {          // 3. 第二次检查
                    instance = new Singleton();  // 4. 创建实例
                }
            }
        }
        return instance;
    }
}
```

### 8.2 为什么DCL需要volatile？

```java
// 问题出在 instance = new Singleton() 这行
// 它不是原子操作，分为三步：

// 1. 分配内存空间
memory = allocate();

// 2. 初始化对象
ctorInstance(memory);

// 3. 将引用指向内存地址
instance = memory;

// 如果不加volatile，步骤2和3可能被重排序：
// 1. 分配内存
// 3. instance指向内存（此时instance != null，但对象未初始化！）
// 2. 初始化对象

// 另一个线程在第一次检查时看到instance != null
// 直接返回未完全初始化的对象——导致NPE或错误行为！

// 加volatile后：
// StoreStore屏障保证步骤2在步骤3之前完成
// 不会出现"指向未初始化对象"的问题
```

### 8.3 DCL的替代方案

```java
// 方案1：静态内部类（推荐，无需volatile）
public class Singleton {

    private Singleton() {}

    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
// 利用类加载机制保证线程安全
// Holder类只在getInstance()首次调用时加载

// 方案2：枚举单例（最安全，防反射）
public enum Singleton {
    INSTANCE;

    public void doSomething() {}
}
// 枚举的天然线程安全 + 防反射 + 防序列化

// 方案3：ThreadLocal
private static final ThreadLocal<Singleton> threadLocal =
    ThreadLocal.withInitial(Singleton::new);
// 每个线程一个实例（不是真正的单例，但线程安全）
```

## 10. volatile vs synchronized

### 10.1 对比

| 特性 | volatile | synchronized |
|------|----------|-------------|
| 原子性 | ❌ 不保证 | ✅ 保证 |
| 可见性 | ✅ 保证 | ✅ 保证 |
| 有序性 | ✅ 保证（部分） | ✅ 保证 |
| 阻塞 | 不阻塞 | 可能阻塞 |
| 粒度 | 变量级 | 对象/代码块级 |
| 编译优化 | 禁止JIT优化volatile变量 | 禁止同步块内重排序 |
| 性能 | 轻量（无锁） | 重量级（锁升级） |

### 10.2 使用选择

```java
// 用volatile：
// 1. 状态标志（一写多读）
// 2. 安全发布对象
// 3. DCL单例
// 4. 配合CAS实现无锁数据结构

// 用synchronized：
// 1. 复合操作（i++）
// 2. 多个变量的原子更新
// 3. 复杂的临界区
// 4. 需要wait/notify的场景

// volatile + CAS = 无锁编程（AQS、ConcurrentHashMap等）
// volatile保证可见性，CAS保证原子性
```

### 10.3 volatile的开销

```
volatile读的开销：
  - 几乎没有额外开销（普通Load + 屏障）
  - 比普通读稍慢（阻止部分优化）

volatile写的开销：
  - Store Buffer刷入Cache
  - MESI协议通知其他CPU（Invalidate）
  - StoreLoad屏障（lock指令）
  - 大约比普通写慢10-100倍

synchronized的开销：
  - 无竞争：偏向锁（几乎无开销）/ 轻量级锁（CAS）
  - 有竞争：重量级锁（线程阻塞/唤醒——内核态切换）
  - 整体比volatile重

结论：
  无竞争时synchronized（偏向锁/轻量级锁）接近volatile
  有竞争时volatile（不阻塞）优于synchronized（阻塞）
  但volatile不保证原子性，适用范围有限
```

## 11. 面试题速查

### Q1: volatile保证什么？不保证什么？

保证可见性（修改后其他线程立即可见）和有序性（禁止指令重排序）。不保证原子性（i++等复合操作不是原子的）。volatile的可见性和有序性通过内存屏障和MESI缓存一致性协议实现。

### Q2: volatile的底层实现原理是什么？

volatile写：在写操作后插入StoreLoad屏障（x86使用lock前缀指令），将Store Buffer刷入Cache，通过MESI协议使其他CPU的缓存行失效。volatile读：在读操作前插入LoadLoad屏障，处理Invalidate Queue，确保读到最新值。内存屏障阻止了指令重排序。

### Q3: 为什么volatile不保证原子性？

i++是读-改-写三步操作，volatile只保证每一步对其他线程可见，但不能保证三步作为一个整体不被打断。线程A读到值0、线程B也读到0，各自+1后都写回1，结果丢失了一次递增。需要用AtomicInteger或synchronized。

### Q4: DCL单例为什么需要volatile？

`instance = new Singleton()`分三步：1.分配内存 2.初始化 3.引用指向内存。不加volatile时步骤2和3可能重排序，导致其他线程在第一次检查时看到非null引用但对象尚未初始化。volatile通过StoreStore屏障保证初始化在引用赋值之前完成。

### Q5: 什么是内存屏障？有哪几种？

内存屏障是CPU指令，阻止屏障两侧的指令重排序。四种：LoadLoad（保证Load1先于Load2）、StoreStore（保证Store1先于Store2且刷新到内存）、LoadStore（保证Load先于Store）、StoreLoad（保证Store对其他CPU可见后再Load，最强也最贵）。

### Q6: MESI协议是什么？

MESI是CPU缓存一致性协议，缓存行有四种状态：Modified（已修改，独占）、Exclusive（独占，与内存一致）、Shared（共享，与内存一致）、Invalid（无效）。写操作通过Invalidate消息使其他CPU的缓存行失效，保证一致性。Store Buffer和Invalidate Queue的引入导致了重排序问题，需要内存屏障解决。

### Q7: volatile和synchronized的区别？

volatile只保证可见性和有序性，不保证原子性，不阻塞线程，作用于变量级。synchronized保证原子性、可见性、有序性，可能阻塞线程，作用于对象/代码块级。volatile轻量（无锁），synchronized较重（锁升级）。volatile适合一写多读，synchronized适合复合操作。

### Q8: volatile的happens-before规则是什么？

volatile写happens-before后续对同一volatile变量的读。这意味着volatile写之前的所有操作，对volatile读之后的操作都可见。通过StoreStore屏障（写前）和LoadLoad屏障（读后）实现。这个规则使得volatile可以作为线程间安全通信的机制。

### Q9: 什么时候用volatile？

1.状态标志位（一写多读，如shutdown标志）。2.安全发布对象（对象完全构造后对其他线程可见）。3.DCL单例模式。4.配合CAS实现无锁数据结构（如AQS的state变量）。不适合：复合操作（i++）、多线程写同一变量。

### Q10: volatile能替代synchronized吗？

不能完全替代。volatile只保证可见性和有序性，不保证原子性。对于i++、多变量原子更新等需要原子性的场景，volatile无法替代synchronized。但volatile + CAS可以替代部分synchronized场景（如AQS、ConcurrentHashMap），实现更高性能的无锁并发。
