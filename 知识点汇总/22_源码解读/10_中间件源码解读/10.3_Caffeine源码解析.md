## 1. Caffeine概述与架构

### 1.1 Caffeine简介

Caffeine是Ben Manes开发的高性能Java本地缓存库，是Spring Boot 2+的默认缓存实现。相比Guava Cache，Caffeine在命中率上提升约30%，吞吐量提升约2倍。

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| W-TinyLFU | 结合LRU和LFU优点的淘汰策略 |
| Count-Min Sketch | 空间高效的频率统计 |
| 异步加载 | 非阻塞的缓存加载和刷新 |
| 时间过期 | 写后过期、读后过期、自定义过期 |
| 大小淘汰 | 基于数量或权重的淘汰 |
| 引用回收 | 基于GC的弱引用/软引用回收 |
| 统计信息 | 命中率、加载时间等指标 |
| 写传播 | 支持监听器同步到外部存储 |

### 1.3 核心类结构

```
com.github.benmanes.caffeine.cache
├── BoundedLocalCache          # 有界本地缓存（核心实现）
│   ├── BoundedBuffer          # 读缓冲（RingBuffer）
│   ├── WriteBuffer            # 写缓冲
│   ├── TimerWheel             # 时间轮（过期管理）
│   └── Node                   # 缓存节点
├── UnboundedLocalCache        # 无界本地缓存
├── LocalCache                 # 缓存接口
├── Cache                      # 对外API接口
├── Caffeine                   # Builder
├── FrequencySketch            # Count-Min Sketch
└── Expires                    # 过期策略接口
```

### 1.4 基本使用

```java
// 基本配置
Cache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)              // 最大缓存数量
    .expireAfterWrite(5, TimeUnit.MINUTES)  // 写后5分钟过期
    .expireAfterAccess(10, TimeUnit.MINUTES) // 访问后10分钟过期
    .recordStats()                    // 开启统计
    .build();

// 写入
cache.put("user:1", user);

// 读取（不存在返回null）
User user = cache.getIfPresent("user:1");

// 读取（不存在则加载）
User user = cache.get("user:1", key -> loadUser(key));

// 删除
cache.invalidate("user:1");

// 统计
CacheStats stats = cache.stats();
System.out.println("命中率: " + stats.hitRate());
```

## 2. W-TinyLFU淘汰策略

### 2.1 传统淘汰策略的问题

```
LRU（最近最少使用）：
  优点：实现简单，对突发访问友好
  缺点：对扫描操作敏感（一次全表扫描会驱逐热数据）
  缺点：无法区分"曾经热"和"现在热"

LFU（最少频率使用）：
  优点：能识别长期热数据
  缺点：旧热点难以驱逐（历史频率高但已不再访问）
  缺点：频率计数器内存开销大

W-TinyLFU：
  结合LRU和LFU的优点
  - 使用LRU处理突发访问（Window区）
  - 使用LFU识别长期热数据（Main区）
  - 使用Count-Min Sketch做频率统计（空间高效）
```

### 2.2 W-TinyLFU结构

```
                    ┌───────────────────────────────────────────────┐
                    │              W-TinyLFU 结构                     │
                    │                                               │
 新增 →  ┌──────────┴──────────┐                                    │
         │   Window LRU (1%)   │ ← 窗口区，处理突发访问                  │
         │   (LRU淘汰)          │                                    │
         └──────────┬──────────┘                                    │
                    │ 满了之后候选                                    │
                    ↓                                               │
         ┌──────────────────────────────────────────────────┐       │
         │                Probation (20%)                     │ ← 试用区
         │                  (LRU)                             │       │
         └────────┬─────────────────────┬─────────────────────┘       │
                  │ 频率足够             │ 频率不够                    │
                  ↓                     ↓                           │
         ┌────────────────┐    ┌─────────────────┐                   │
         │ Protected (80%)│    │   被淘汰          │                   │
         │    (LRU)       │    │                 │                   │
         └────────────────┘    └─────────────────┘                   │
                                                              │
                                                    ┌─────────┘
```

### 2.3 三区域详解

```
Window区（窗口区）：
  - 占总容量约1%
  - 使用LRU淘汰
  - 新条目先进入这里
  - 处理突发访问（scan resistance）
  - 当Window满时，最旧的条目成为候选者

Probation区（试用区）：
  - 占总容量约20%
  - 从Window过来的条目进入这里
  - 使用LRU管理
  - 当需要淘汰时，比较Probation区尾部条目与新进入条目的频率

Protected区（保护区）：
  - 占总容量约80%
  - 从Probation区"晋升"过来的热数据
  - 使用LRU管理
  - Protected区满时，尾部条目降级回Probation区
```

### 2.4 淘汰决策

```java
// BoundedLocalCache中的淘汰逻辑（简化）
public void evict() {
    // 1. 从Window区取出LRU尾部候选者
    Node candidate = windowDeque.peekLast();

    // 2. 从Probation区取出LRU尾部受害者
    Node victim = probationDeque.peekLast();

    // 3. 比较频率——使用Count-Min Sketch
    int candidateFreq = frequencySketch.frequency(candidate.key);
    int victimFreq = frequencySketch.frequency(victim.key);

    // 4. 淘汰决策
    if (candidateFreq > victimFreq) {
        // 候选者频率更高——淘汰受害者
        probationDeque.remove(victim);
        evictEntry(victim);
        // 候选者进入Probation区
        probationDeque.addFirst(candidate);
    } else {
        // 受害者频率更高或相等——淘汰候选者
        evictEntry(candidate);
    }
}
```

## 3. Count-Min Sketch频率统计

### 3.1 为什么不用普通计数器

```
普通频率计数器的问题：
  - 每个Key需要一个int/long计数器
  - 100万Key = 4-8MB内存
  - 无法应对更大的缓存

Count-Min Sketch：
  - 空间固定，与Key数量无关
  - 近似计数（有误差，只会偏大）
  - 4KB空间即可统计百万级Key的频率
```

### 3.2 Count-Min Sketch原理

```
Count-Min Sketch结构：
  - d个哈希函数，d个计数器数组
  - 每个数组宽度w

  Hash 1:  [0, 2, 0, 1, 3, 0, 1, 0]   ← counter array 1
  Hash 2:  [1, 0, 2, 0, 0, 3, 0, 1]   ← counter array 2
  Hash 3:  [0, 1, 0, 3, 0, 1, 2, 0]   ← counter array 3
  Hash 4:  [2, 0, 1, 0, 3, 0, 0, 1]   ← counter array 4

增加Key "A"的频率：
  hash1("A") % 8 = 3 → array1[3]++
  hash2("A") % 8 = 5 → array2[5]++
  hash3("A") % 8 = 3 → array3[3]++
  hash4("A") % 8 = 4 → array4[4]++

查询Key "A"的频率：
  freq = min(array1[3], array2[5], array3[3], array4[4])
  取最小值——减少其他Key哈希冲突带来的误差
```

### 3.3 FrequencySketch源码

```java
public final class FrequencySketch<K> {

    // 4个计数器数组（对应4个哈希函数）
    private long[] table;
    private int tableMask;

    // 增加Key的频率
    public void increment(K key) {
        // 种子值用于哈希函数
        int blockHash = spread(key.hashCode());

        // 4个哈希函数的结果
        int hash0 = blockHash;
        int hash1 = hash0 + ((int)(blockHash >>> 16) | 1);
        int hash2 = hash0 + ((int)(blockHash >>> 16) | 1) * 2;
        int hash3 = hash0 + ((int)(blockHash >>> 16) | 1) * 3;

        // 对4个位置进行+1操作
        incrementAt(hash0);
        incrementAt(hash1);
        incrementAt(hash2);
        incrementAt(hash3);
    }

    // 查询Key的频率
    public int frequency(K key) {
        int blockHash = spread(key.hashCode());
        int hash0 = blockHash;
        int hash1 = hash0 + ((int)(blockHash >>> 16) | 1);
        int hash2 = hash0 + ((int)(blockHash >>> 16) | 1) * 2;
        int hash3 = hash0 + ((int)(blockHash >>> 16) | 1) * 3;

        // 取4个计数器的最小值
        return Math.min(
            Math.min(countAt(hash0), countAt(hash1)),
            Math.min(countAt(hash2), countAt(hash3))
        );
    }

    // 每个long（64位）存储4个4位计数器（每个计数器最大15）
    private void incrementAt(int hash) {
        int index = hash & tableMask;
        // 每个long有16个4位计数器
        // 定位到具体的4位计数器并+1
        int subIndex = (hash >>> 5) & 0xF; // 16个计数器中的一个
        int offset = subIndex * 4;          // 4位偏移
        long mask = 0xFL << offset;

        // 如果计数器还没满（< 15），+1
        if ((table[index] & mask) != mask) {
            table[index] += (1L << offset);
        }
    }
}
```

## 4. 读缓冲与写缓冲

### 4.1 读缓冲——BoundedBuffer

Caffeine不使用传统锁来记录读取操作，而是使用RingBuffer无锁缓冲：

```java
// BoundedLocalCache中的读缓冲
final BoundedBuffer<AccessEvent<K, V>> readBuffer;

// 读操作记录
@Override
public V get(K key, Function<K, V> mappingFunction) {
    Node<K, V> node = data.get(key);
    if (node != null) {
        V value = node.getValue();
        // 记录读取到RingBuffer——无锁
        readBuffer.offer(new AccessEvent<>(key, value));
        // 异步处理（不阻塞读操作）
        afterRead(node);
        return value;
    }
    // 加载...
}

// BoundedBuffer——基于CAS的RingBuffer
class BoundedBuffer<E> {
    final AtomicReferenceArray<RingBuffer<E>> buffers;
    final AtomicInteger tail;
    final AtomicReference<Throwable> drainStatus;

    public boolean offer(E e) {
        // 获取当前线程对应的RingBuffer
        int index = (int)(Thread.currentThread().getId() % buffers.length());
        RingBuffer<E> buffer = buffers.get(index);

        // CAS方式写入
        return buffer.offer(e);
    }
}

// RingBuffer——单线程的无锁环形缓冲区
class RingBuffer<E> {
    final AtomicLong head;
    final AtomicLong tail;
    final Object[] buffer;

    public boolean offer(E e) {
        long currentTail = tail.get();
        long wrapPoint = currentTail - buffer.length;

        if (head.get() <= wrapPoint) {
            return false; // 缓冲区满，丢弃事件
        }

        buffer[(int)(currentTail % buffer.length)] = e;
        tail.lazySet(currentTail + 1);
        return true;
    }
}
```

### 4.2 写缓冲——WriteBuffer

```java
// 写操作通过队列异步处理
final WriteBuffer<K, V> writeBuffer;

// 所有写操作（put/remove/invalidate）先入队
@Override
public void put(K key, V value) {
    // 直接操作ConcurrentHashMap
    data.compute(key, (k, existingNode) -> {
        Node<K, V> node = newNode(key, value, existingNode);
        return node;
    });

    // 写入缓冲队列
    writeBuffer.offer(new WriteEvent<>(key, value, WriteAction.ADD));

    // 触发异步处理
    scheduleAfterWrite();
}

// 写缓冲处理线程
class WriteBufferDrainTask implements Runnable {
    @Override
    public void run() {
        // 批量处理写缓冲
        List<WriteEvent<K, V>> events = writeBuffer.drain();
        for (WriteEvent<K, V> event : events) {
            switch (event.action) {
                case ADD:
                    addToLfu(event.key);
                    break;
                case REMOVE:
                    removeFromLfu(event.key);
                    break;
                case CLEAR:
                    clearLfu();
                    break;
            }
        }

        // 处理完写缓冲后检查是否需要淘汰
        evictIfNecessary();
    }
}
```

### 4.3 缓冲的意义

```
为什么用缓冲而不是直接操作？

读操作：
  - 不加锁 → 高并发读性能
  - 记录到RingBuffer → 不阻塞读线程
  - 后台线程异步drain → 批量更新LRU/LFU
  - 即使RingBuffer满丢弃事件也不影响正确性（只是频率统计不精确）

写操作：
  - 先更新ConcurrentHashMap → 快速返回
  - 写操作入队 → 异步更新LRU队列和频率表
  - 批量处理 → 减少锁竞争
```

## 5. 异步刷新机制

### 5.1 refreshAfterWrite

```java
// 异步刷新——不阻塞读，后台刷新
LoadingCache<String, User> cache = Caffeine.newBuilder()
    .refreshAfterWrite(1, TimeUnit.MINUTES) // 写后1分钟刷新
    .build(key -> loadUser(key));

// 与expireAfterWrite的区别：
// expireAfterWrite: 过期后读取会阻塞等待加载
// refreshAfterWrite: 到期后读取返回旧值，同时异步触发刷新
```

### 5.2 刷新源码

```java
// BoundedLocalCache
@Override
public V get(K key, Function<K, V> loader) {
    Node<K, V> node = data.get(key);
    if (node != null) {
        V value = node.getValue();

        // 检查是否需要刷新
        if (needsRefresh(node)) {
            // 异步刷新——不阻塞当前读取
            scheduleRefresh(node);
        }

        // 立即返回旧值
        return value;
    }

    // 不存在则加载
    return computeIfAbsent(key, loader);
}

// 异步刷新
CompletableFuture<V> scheduleRefresh(Node<K, V> node) {
    // CAS保证只有一个线程触发刷新
    if (node.isRefreshing()) {
        return node.getRefreshFuture();
    }

    CompletableFuture<V> future = CompletableFuture.supplyAsync(() -> {
        try {
            // 调用CacheLoader.reload
            return cacheLoader.reload(node.getKey(), node.getValue());
        } catch (Exception e) {
            // 刷新失败，返回旧值
            return node.getValue();
        }
    }, executor);

    node.setRefreshFuture(future);

    future.thenAccept(newValue -> {
        // 刷新完成后更新缓存
        node.setValue(newValue);
        node.setRefreshFuture(null);
    });

    return future;
}
```

## 6. 过期策略实现

### 6.1 三种过期方式

```java
// 1. 写后过期——从写入时刻开始计时
Caffeine.newBuilder().expireAfterWrite(5, TimeUnit.MINUTES)

// 2. 读后过期——从最后一次访问开始计时
Caffeine.newBuilder().expireAfterAccess(10, TimeUnit.MINUTES)

// 3. 自定义过期——每个条目单独的过期时间
Caffeine.newBuilder().expireVariably(
    new Expiry<String, User>() {
        @Override
        public long expireAfterCreate(String key, User value, long currentTime) {
            // 创建时计算过期时间
            return TimeUnit.MINUTES.toNanos(value.getTtlMinutes());
        }

        @Override
        public long expireAfterUpdate(String key, User value, long currentTime, long currentDuration) {
            // 更新时重新计算
            return TimeUnit.MINUTES.toNanos(value.getTtlMinutes());
        }

        @Override
        public long expireAfterRead(String key, User value, long currentTime, long currentDuration) {
            // 读取时不延长过期时间
            return currentDuration;
        }
    })
```

### 6.2 TimerWheel——时间轮过期

```java
// Caffeine使用分层时间轮管理过期
// 避免了PriorityQueue的O(logN)插入

class TimerWheel<K, V> {

    // 分层时间轮——不同精度的桶
    // [0]: 1.55ms精度，64个桶（覆盖~100ms）
    // [1]: 9.83ms精度，64个桶（覆盖~628ms）
    // [2]: 62.9ms精度，64个桶（覆盖~4s）
    // [3]: 402ms精度，64个桶（覆盖~25.7s）
    // [4]: 2.57s精度，64个桶（覆盖~164s）
    // [5]: 16.4s精度，64个桶（覆盖~1049s）
    // [6]: 105s精度，64个桶（覆盖~6706s≈1.86h）
    // [7]: 671s精度，64个桶（覆盖~42824s≈11.9h）

    final Node<?, ?>[][] wheel;
    final long[] tickAt;

    // 添加到时间轮
    public void schedule(Node<K, V> node) {
        long expirationTime = node.getExpirationTime();
        long currentTime = System.nanoTime();
        long delay = expirationTime - currentTime;

        // 根据delay选择合适的层级和桶
        int level = findLevel(delay);
        int bucketIndex = findBucket(level, delay);

        // 添加到对应桶的链表尾部
        addToBucket(level, bucketIndex, node);
    }

    // 推进时间轮——定期调用
    public void advance(long currentTime) {
        // 找到当前时间应该处理的所有桶
        // 将过期节点移出并清理
        for (int level = 0; level < wheel.length; level++) {
            int tick = (int)(currentTime / tickAt[level]);
            int bucketIndex = tick % wheel[level].length;

            Node<?, ?> bucket = wheel[level][bucketIndex];
            if (bucket != null) {
                // 处理过期的节点
                processExpiredNodes(bucket, currentTime);
            }
        }
    }
}
```

### 6.3 惰性过期 vs 主动过期

```java
// Caffeine采用惰性过期+主动清理的混合策略

// 惰性过期——读操作时检查
@Override
public V getIfPresent(K key) {
    Node<K, V> node = data.get(key);
    if (node == null) return null;

    // 检查是否过期
    if (isExpired(node)) {
        // 异步清理
        scheduleEviction(node);
        return null;
    }

    return node.getValue();
}

// 主动清理——后台线程定期运行
class CleanupTask implements Runnable {
    @Override
    public void run() {
        // 1. 推进时间轮，清理过期节点
        timerWheel.advance(System.nanoTime());

        // 2. 处理读缓冲
        readBuffer.drainTo(this::processReadEvent);

        // 3. 处理写缓冲
        writeBuffer.drain();

        // 4. 执行淘汰
        evictIfNecessary();

        // 5. 调度下次清理
        scheduleNextCleanup();
    }
}
```

### 7.2 computeIfAbsent的并发控制

```java
// BoundedLocalCache
@Override
public V get(K key, Function<K, V> mappingFunction) {
    Node<K, V> node = data.get(key);

    if (node != null) {
        V value = node.getValue();
        if (value != null) {
            // 记录读取
            afterRead(node);
            return value;
        }
    }

    // 缓存不存在——加锁加载
    // 使用ConcurrentHashMap.compute保证原子性
    return computeIfAbsent(key, mappingFunction);
}

V computeIfAbsent(K key, Function<K, V> loader) {
    // ConcurrentHashMap.compute——同一key只有一个线程执行loader
    Node<K, V> node = data.compute(key, (k, existingNode) -> {
        if (existingNode != null && existingNode.getValue() != null) {
            return existingNode; // 已有值，不执行loader
        }

        V value = loader.apply(k);
        if (value == null) {
            return null; // 不缓存null值
        }

        // 创建新节点
        return newNode(k, value, existingNode);
    });

    return node != null ? node.getValue() : null;
}
```

### 10.2 多级缓存

```java
// 本地缓存(Caffeine) + 分布式缓存(Redis)
@Service
public class MultiLevelCacheService {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    // L1: 本地缓存
    private final Cache<String, Object> localCache = Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterWrite(1, TimeUnit.MINUTES)
        .build();

    @SuppressWarnings("unchecked")
    public <T> T get(String key, Supplier<T> dbLoader) {
        // L1: 查本地缓存
        Object l1Value = localCache.getIfPresent(key);
        if (l1Value != null) {
            return (T) l1Value;
        }

        // L2: 查Redis
        Object l2Value = redisTemplate.opsForValue().get(key);
        if (l2Value != null) {
            localCache.put(key, l2Value);
            return (T) l2Value;
        }

        // L3: 查数据库
        T dbValue = dbLoader.get();
        if (dbValue != null) {
            // 回填L2和L1
            redisTemplate.opsForValue().set(key, dbValue, 10, TimeUnit.MINUTES);
            localCache.put(key, dbValue);
        }

        return dbValue;
    }
}
```

## 11. 面试题速查

### Q1: Caffeine的W-TinyLFU是什么？

W-TinyLFU是结合LRU和LFU的淘汰策略。分为Window区（1%，LRU处理突发访问）、Probation区（20%，试用）、Protected区（80%，保护热数据）。新条目先入Window，满后进入Probation，Probation中条目频率足够时晋升到Protected。淘汰时比较候选者和受害者的频率（来自Count-Min Sketch）。

### Q2: Count-Min Sketch是什么？

一种空间高效的近似频率统计数据结构。使用多个哈希函数映射到多个计数器数组，增加时所有对应位置+1，查询时取所有位置的最小值。Caffeine中每个计数器4位（最大15），4KB空间即可统计百万级Key。取最小值减少哈希冲突带来的高估。定期减半重置实现老化。

### Q3: Caffeine的读操作为什么快？

读操作完全无锁：1.ConcurrentHashMap读不需要锁；2.读记录通过RingBuffer的CAS操作写入，不阻塞读线程；3.读缓冲满时丢弃事件也不影响正确性（只是频率统计不精确）；4.统计使用LongAdder分散竞争。读性能接近ConcurrentHashMap。

### Q4: refreshAfterWrite和expireAfterWrite的区别？

expireAfterWrite：到期后读取会阻塞等待新值加载完成。refreshAfterWrite：到期后读取立即返回旧值，同时异步触发刷新。refreshAfterWrite适合"尽量返回最新值但不阻塞"的场景，用户体验更好。两者可以同时使用——refreshAfterWrite先触发刷新，expireAfterWrite作为兜底。

### Q5: Caffeine如何实现自定义过期？

通过Expiry接口实现三个方法：expireAfterCreate（创建时计算TTL）、expireAfterUpdate（更新时重新计算TTL）、expireAfterRead（读取时决定是否延长TTL）。Caffeine使用分层时间轮（TimerWheel）管理过期，避免了PriorityQueue的O(logN)插入复杂度。

### Q6: Caffeine和Guava Cache的主要区别？

淘汰策略：Caffeine用W-TinyLFU（命中率约95%），Guava用LRU（命中率约87%）。性能：Caffeine读快2倍，写快2倍。功能：Caffeine支持异步加载/刷新、自定义过期、异步缓存。API：两者高度兼容，迁移成本低。Spring Boot 2+默认使用Caffeine。

### Q7: Caffeine如何防止缓存穿透？

使用cache.get(key, loader)——当key不存在时只有一个线程执行loader（通过ConcurrentHashMap.compute保证原子性）。可以配合空值缓存：loader返回null时仍然缓存一个标记值。也可以配合布隆过滤器预先过滤不存在的Key。

### Q8: Caffeine的时间轮是怎么工作的？

分层时间轮——8个层级，每层64个桶，精度从1.55ms到671s递增。插入过期条目时根据TTL选择合适的层级和桶。后台线程推进时间轮，处理过期的桶。条目降级机制——高层级的条目随着时间推移降级到低层级，直到最终过期。时间轮插入O(1)，优于PriorityQueue的O(logN)。

### Q9: Caffeine的弱引用和软引用有什么用？

weakKeys()：Key使用弱引用，当Key没有其他强引用时GC可回收缓存条目。weakValues()/softValues()：Value使用弱/软引用，GC在内存不足时回收。注意weakKeys会使用==比较而非equals，因为GC可能随时回收对象。软引用适合做内存敏感的缓存——内存充足时保留，不足时GC回收。

### Q10: Caffeine如何保证同一个Key只加载一次？

使用ConcurrentHashMap.compute(key, biFunction)保证原子性。当多个线程同时请求同一个不存在的Key时，只有第一个线程执行loader函数，其他线程在compute内部等待。compute返回后所有线程都能获取到相同的值。这防止了缓存击穿。
