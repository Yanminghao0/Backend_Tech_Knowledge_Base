### 2.1 日志物理结构

### 2.2 LogSegment——日志段

每个分区由多个日志段（LogSegment）组成，默认每个段1GB（`log.segment.bytes=1073741824`）：

```scala
class LogSegment(
    val log: FileRecords,         // .log文件——存储消息
    val offsetIndex: OffsetIndex,  // .index文件——偏移量索引
    val timeIndex: TimeIndex,      // .timeindex文件——时间戳索引
    val baseOffset: Long,          // 此段的起始偏移量
    val indexIntervalBytes: Int,   // 索引间隔（默认4KB写一条索引）
    val rollJitterMs: Long,
    val time: Time,
    val maxSegmentMs: Long,
    val maxSegmentBytes: Int       // 段最大字节数
) extends Logging {

    // 追加消息到日志段
    def append(
        largestOffset: Long,
        largestTimestamp: Long,
        shallowOffsetOfMaxTimestamp: Long,
        records: MemoryRecords
    ): Unit = {
        if (records.sizeInBytes > 0) {
            // 1. 计算物理位置
            val physicalPosition = log.sizeInBytes()
            if (physicalPosition == 0)
                rollBasedOnTimeOrSize = true

            // 2. 追加消息到.log文件
            val appendedBytes = log.append(records)

            // 3. 每隔indexIntervalBytes字节写一条索引
            if (physicalPosition + appendedBytes > indexIntervalBytes) {
                offsetIndex.append(largestOffset, physicalPosition)
                timeIndex.maybeAppend(largestTimestamp, shallowOffsetOfMaxTimestamp)
            }
        }
    }

    // 读取消息
    def read(
        startOffset: Long,
        maxOffset: Option[Long],
        maxSize: Int,
        fetchPosition: Long
    ): FetchDataInfo = {
        // 1. 通过索引定位物理位置
        val position = offsetIndex.lookup(startOffset).position

        // 2. 从.log文件读取
        val fetchedData = log.read(position, maxSize)
        FetchDataInfo(fetchedData)
    }
}
```

### 2.3 稀疏索引

Kafka使用稀疏索引（Sparse Index）——不是每条消息都有索引，而是每隔4KB消息数据写一条索引：

```
.index文件结构（每条索引8字节：4字节offset + 4字节position）：

  Offset    Position（物理位置）
  0         0
  100       4096        ← 每隔4KB写一条
  250       8192
  380       12288
  ...

查找 offset=200 的消息：
1. 二分查找.index，找到 ≤ 200 的最大索引项 → offset=100, position=4096
2. 从.log的position=4096开始顺序扫描，找到offset=200的消息
```

```scala
class OffsetIndex(file: File, baseOffset: Long, maxIndexSize: Int) {

    // 索引条目：每条8字节（4字节relativeOffset + 4字节physicalPosition）
    private val mmap = // 内存映射文件

    // 二分查找
    def lookup(targetOffset: Long): OffsetPosition = {
        val relativeOffset = targetOffset - baseOffset
        // 二分查找 ≤ relativeOffset 的最大条目
        val slot = largestLowerBoundSlotFor(mmap, relativeOffset)
        if (slot == -1)
            OffsetPosition(baseOffset, 0)
        else
            parseEntry(slot)
    }

    private def parseEntry(slot: Int): OffsetPosition = {
        val relativeOffset = mmap.getInt(slot * 8)
        val position = mmap.getInt(slot * 8 + 4)
        OffsetPosition(baseOffset + relativeOffset, position)
    }
}
```

### 2.4 Log——分区级别管理

```scala
class Log(
    val dir: File,
    val topicPartition: TopicPartition,
    val config: LogConfig,
    val scheduler: Scheduler,
    val time: Time,
    val maxProducerIdExpirationMs: Long,
    val producerIdExpirationCheckIntervalMs: Long,
    val logDirFailureChannel: LogDirFailureChannel
) extends Logging {

    private val segments: ConcurrentNavigableMap[Long, LogSegment] = new ConcurrentSkipListMap

    // 追加消息
    def append(
        records: MemoryRecords,
        origin: AppendOrigin = AppendOrigin.Client,
        interBrokerProtocolVersion: ApiVersion = ApiVersion.latest
    ): LogAppendInfo = {
        // 1. 分配偏移量
        val assignOffsets = origin == AppendOrigin.Client
        var lastOffset = logEndOffset

        if (assignOffsets) {
            lastOffset = logEndOffset + records.records.size()
        }

        // 2. 如果当前段已满，创建新段（roll）
        val segment = maybeRoll(records.sizeInBytes)

        // 3. 追加到日志段
        segment.append(lastOffset, largestTimestamp, shallowOffsetOfMaxTimestamp, records)

        // 4. 更新LEO（Log End Offset）
        updateLogEndOffset(lastOffset + 1)

        LogAppendInfo(lastOffset, ...)
    }

    // 读取消息
    def read(
        startOffset: Long,
        maxLength: Int,
        isolation: FetchIsolation,
        minOneMessage: Boolean
    ): FetchDataInfo = {
        // 1. 找到包含startOffset的日志段
        val segment = segments.floorEntry(startOffset).getValue

        // 2. 从段中读取
        val fetchInfo = segment.read(startOffset, maxLength)

        // 3. 如果跨段，继续读下一个段
        fetchInfo
    }
}
```

## 3. 分区与副本机制

### 3.1 Partition源码

```scala
class Partition(
    val topicPartition: TopicPartition,
    val replicaLagTimeMaxMs: Long,
    val localBrokerId: Int,
    val time: Time,
    val replicaManager: ReplicaManager,
    ...
) extends Logging {

    // 所有副本
    @volatile var allReplicas: Set[Replica] = Set.empty
    // 分配的副本列表（AR）
    @volatile var assignedReplicas: Set[Int] = Set.empty
    // ISR集合
    @volatile var isr: Set[Int] = Set.empty
    // Leader副本ID
    @volatile var leaderReplicaId: Option[Int] = None

    // 追加消息到Leader
    def appendRecordsToLeader(
        records: MemoryRecords,
        origin: AppendOrigin,
        requiredAcks: Int
    ): LogAppendInfo = {
        // 1. 检查是否为Leader
        val leaderLog = leaderLogIfLocal.getOrElse {
            throw new NotLeaderOrFollowerException("Not leader")
        }

        // 2. 检查ISR是否满足min.insync.replicas
        val isrSize = isr.size
        if (requiredAcks == -1 && isrSize < minInSyncReplicas) {
            throw new NotEnoughReplicasException(
                s"ISR size $isrSize < min.insync.replicas $minInSyncReplicas")
        }

        // 3. 追加到本地日志
        val info = leaderLog.append(records, origin)

        // 4. 延迟更新HW（High Watermark）
        maybeIncrementLeaderHW(leaderLog)

        info
    }

    // Follower拉取消息
    def fetchRecordsFromLeader(
        fetchOffset: Long,
        maxLength: Int,
        followerId: Int
    ): LogReadResult = {
        val log = leaderLogIfLocal.getOrElse {
            throw new NotLeaderOrFollowerException("Not leader")
        }

        // 读取消息
        val fetchInfo = log.read(fetchOffset, maxLength, FetchLogEnd)

        // 更新Follower的LEO
        getReplica(followerId).foreach { replica =>
            replica.updateLogReadResult(LogReadResult(
                fetchOffset = fetchOffset,
                ...
            ))
        }

        // 尝试更新ISR
        maybeExpandIsr(followerId)

        fetchInfo
    }
}
```

### 3.2 副本同步

```scala
class ReplicaManager(
    val config: KafkaConfig,
    val time: Time,
    ...
) extends Logging {

    // Follower副本同步线程管理器
    val replicaFetcherManager = new ReplicaFetcherManager(...)

    // 处理来自Follower的Fetch请求
    def fetchMessages(
        timeout: Long,
        replicaId: Int,
        fetchMinBytes: Int,
        fetchMaxBytes: Int,
        hardMaxBytesLimit: Boolean,
        fetchInfos: Seq[(TopicPartition, PartitionData)],
        responseCallback: Seq[(TopicPartition, FetchPartitionData)] => Unit,
        isolationLevel: IsolationLevel
    ): Unit = {

        val logReadResults = readFromLocalLog(
            replicaId, fetchInfos, isolationLevel)

        // 如果是Follower的fetch请求
        if (replicaId != Request.FutureLocalReplicaId) {
            // 更新Follower的LEO和检查ISR
            fetchInfos.foreach { case (tp, _) =>
                getPartition(tp).foreach { partition =>
                    partition.updateReplicaLogReadResult(replicaId,
                        logReadResults(tp))
                    // 检查是否需要扩展ISR
                    partition.maybeExpandIsr(replicaId)
                }
            }
        }

        // 如果有足够数据或超时，立即回调
        if (logReadResults.values.exists(_.info.records.sizeInBytes > 0)) {
            responseCallback(logReadResults)
        } else {
            // 延迟等待数据到达
            val delayedFetch = new DelayedFetch(timeout, ...)
            delayedFetchPurgatory.tryCompleteElseWatch(delayedFetch)
        }
    }
}

// Follower拉取线程
class ReplicaFetcherThread(
    name: String,

    ...
) extends AbstractFetcherThread(name, clientId, sourceBroker, ...) {

        val replica = replicaManager.getReplica(topicPartition).get

        // 追加到本地日志
        replica.log.get.append(records, AppendOrigin.Replication)

        // 更新Follower的HW
        val followerHW = replica.highWatermark
        ...
    }

    // 构建Fetch请求
    override def fetch(topicPartition: TopicPartition): Option[FetchRequest.Builder] = {
        val replica = replicaManager.getReplica(topicPartition)
        val fetchOffset = replica.logEndOffset
        Some(buildFetch(topicPartition, fetchOffset))
    }
}
```

## 4. ISR机制与高可用

### 4.1 ISR概念

```
AR (Assigned Replicas)：分区的所有副本
ISR (In-Sync Replicas)：与Leader保持同步的副本集合
OSR (Out-of-Sync Replicas)：未同步的副本
AR = ISR + OSR

只有ISR中的副本才有资格被选为Leader
```

### 4.2 ISR管理

```scala
class Partition(...) {

    // 检查并扩展ISR
    def maybeExpandIsr(followerId: Int): Unit = {
        val followerReplica = getReplica(followerId).getOrElse(return)

        // 检查Follower是否满足ISR条件：
        // Follower的LEO >= Leader的HW - replica.lag.time.max.ms
        val leaderLogEndOffset = leaderLogIfLocal.get.logEndOffset
        val followerLEO = followerReplica.logEndOffset

        val needsIsrUpdate = !isr.contains(followerId) &&
            inSyncReplica(followerReplica, leaderLogEndOffset)

        if (needsIsrUpdate) {
            val newIsr = isr + followerId
            // 更新ZooKeeper中的ISR
            updateIsr(newIsr)
            isr = newIsr
        }
    }

    // 判断副本是否在同步
    private def inSyncReplica(
        replica: Replica,
        leaderLogEndOffset: Long
    ): Boolean = {
        // 条件：Follower在replica.lag.time.max.ms时间内追上了Leader的LEO
        val followerLEO = replica.logEndOffset
        val lastCaughtUpTimeMs = replica.lastCaughtUpTimeMs

        // Follower的LEO >= Leader上次HW时的LEO
        // 或 Follower在replica.lag.time.max.ms内追上了Leader
        followerLEO >= leaderLogEndOffset ||
            (time.milliseconds - lastCaughtUpTimeMs) <= replicaLagTimeMaxMs
    }

    // 收缩ISR（移除落后的副本）
    def maybeShrinkIsr(): Unit = {
        val leaderLogEndOffset = leaderLogIfLocal.get.logEndOffset
        val outOfSyncReplicas = allReplicas.filter { replicaId =>
            val replica = getReplica(replicaId).get
            !inSyncReplica(replica, leaderLogEndOffset)
        }

        if (outOfSyncReplicas.nonEmpty) {
            val newIsr = isr -- outOfSyncReplicas
            updateIsr(newIsr)
            isr = newIsr
        }
    }
}
```

### 4.3 High Watermark（HW）

```scala
// HW = 所有ISR副本中最小的LEO
// 只有HW以下的消息才对消费者可见

class Partition(...) {

    // 更新Leader的HW
    def maybeIncrementLeaderHW(leaderLog: Log): Unit = {
        // 收集所有ISR副本的LEO
        val allLEOs = allReplicas.map { replica =>
            replica.logEndOffset
        }

        // HW = min(LEO of all ISR replicas)
        val newHW = allLEOs.min

        if (newHW > leaderLog.highWatermark) {
            leaderLog.updateHighWatermark(newHW)
        }
    }
}

// 消息可见性：
// 1. Producer发送消息到Leader
// 2. Leader追加到本地日志，更新LEO
// 3. Follower拉取消息，追加到本地日志，更新LEO
// 4. Leader收到Follower的Fetch请求，更新HW
// 5. 消费者只能消费HW以下的消息

// 数据丢失问题（Kafka 0.11之前）：
// 1. Follower拉取后但未更新HW时崩溃
// 2. 重启后从HW截断日志（可能导致已提交消息丢失）
// 3. Kafka 0.11引入Leader Epoch解决此问题
```

### 4.4 Leader选举

```scala
class KafkaController(
    val config: KafkaConfig,
    val zkClient: KafkaZkClient,
    val time: Time,
    ...
) extends Logging {

    // 分区Leader选举
    def onPartitionFailover(partitions: Set[TopicPartition]): Unit = {
        partitions.foreach { tp =>
            // 从ISR中选择新Leader
            val partitionState = partitionStateMachine.getPartitionState(tp)
            val isr = partitionState.isr

            if (isr.nonEmpty) {
                // ISR不为空——选ISR中第一个存活的副本
                val newLeader = isr.head
                partitionStateMachine.reelectLeader(tp, newLeader)
            } else {
                // ISR为空
                if (uncleanLeaderElectionEnable) {
                    // 允许脏选举——从OSR中选（可能丢数据）
                    val newLeader = partitionState.replicas.head
                    partitionStateMachine.reelectLeader(tp, newLeader)
                } else {
                    // 等待ISR中的副本恢复
                    log.warn(s"No leader for $tp and unclean election disabled")
                }
            }
        }
    }
}
```

## 5. 零拷贝与高吞吐

### 5.1 零拷贝——sendfile

```java
// Kafka使用Java的FileChannel.transferTo()实现零拷贝
// 底层调用Linux的sendfile系统调用

public class FileRecords extends AbstractRecords implements Closeable {
    private final FileChannel channel;

    // 消费者拉取消息时使用零拷贝
    public long writeTo(GatheringByteChannel destChannel, long position, int length)
            throws IOException {
        // 使用transferTo——数据直接从文件到socket，不经过用户空间
        long bytesTransferred = channel.transferTo(position, length, destChannel);
        return bytesTransferred;
    }
}
```

传统IO vs 零拷贝对比：

```
传统IO（4次拷贝 + 4次上下文切换）：
  磁盘 → 内核缓冲区 → 用户缓冲区 → Socket缓冲区 → 网卡
         (DMA)        (CPU)        (CPU)         (DMA)

sendfile零拷贝（2次拷贝 + 2次上下文切换）：
  磁盘 → 内核缓冲区 → 网卡（DMA gather）
         (DMA)        (CPU收集描述符)
```

### 5.2 批量发送

```java
// 生产者批量发送——核心机制
public final class Sender implements Runnable {

    @Override
    public void run() {
        while (running) {
            try {
                // 从RecordAccumulator获取准备好发送的批次
                RecordAccumulator.ReadyCheckResult result = accumulator.ready(cluster, now);

                // 批量发送
                Map<Integer, List<ProducerBatch>> batches = accumulator.drain(cluster, result.readyNodes, maxSize, now);
                sendProduceRequests(batches, now);

                // 等待下一批（batch.size或linger.ms）
                poll(timeout, now);
            } catch (Exception e) {
                log.error("Uncaught error in sender", e);
            }
        }
    }
}
```

### 5.3 顺序写磁盘

```

  - 甚至比内存随机写还快

// Kafka的日志追加
class FileRecords {

        // 只追加，不修改已有数据

    }
}
```

### 5.4 页缓存（Page Cache）

```scala
// Kafka大量依赖操作系统的Page Cache
// 不维护应用层缓存，直接使用OS的文件系统缓存

// 写消息：写入Page Cache → OS异步刷盘
// 读消息：从Page Cache读 → 命中率高（消费者通常读最近的数据）

// Kafka配置
// log.flush.interval.messages=Long.MAX_VALUE  // 几乎不主动刷盘
```

### 6.1 KafkaProducer源码

```java
public class KafkaProducer<K, V> implements Producer<K, V> {

    private final Partitioner partitioner;      // 分区器
    private final int maxRequestSize;           // 最大请求大小
    private final long totalMemorySize;         // 总内存
    private final ProducerMetadata metadata;    // 元数据
    private final RecordAccumulator accumulator; // 记录累加器
    private final Sender sender;                 // 发送线程
    private final Thread ioThread;               // IO线程

    // 发送消息
    @Override
    public Future<RecordMetadata> send(ProducerRecord<K, V> record, Callback callback) {
        // 1. 拦截器链
        ProducerRecord<K, V> interceptedRecord = interceptors.onSend(record);

        // 2. 序列化
        byte[] serializedKey = keySerializer.serialize(topic, record.headers(), record.key());
        byte[] serializedValue = valueSerializer.serialize(topic, record.headers(), record.value());

        // 3. 分区选择
        int partition = partition(record, serializedKey, serializedValue, cluster);

        // 4. 追加到RecordAccumulator
        RecordAccumulator.RecordAppendResult result = accumulator.append(
            tp, timestamp, serializedKey, serializedValue, headers, callback, maxBlockTimeMs);

        // 5. 唤醒Sender线程
        if (result.batchIsFull || result.newBatchCreated) {
            this.sender.wakeup();
        }

        return result.future;
    }

    // 分区选择
    private int partition(ProducerRecord<K, V> record, byte[] key, byte[] value, Cluster cluster) {
        // 1. 如果指定了partition，直接使用
        if (record.partition() != null) return record.partition();

        // 2. 否则使用分区器
        return partitioner.partition(record.topic(), key, value, cluster);
    }
}
```

### 6.2 RecordAccumulator——批次累加

```java
public final class RecordAccumulator {
    // 每个分区的批次队列
    private final ConcurrentMap<TopicPartition, Deque<ProducerBatch>> batches;

    // 追加消息到批次
    public RecordAppendResult append(TopicPartition tp, long timestamp,
            byte[] key, byte[] value, Header[] headers,
            Callback callback, long maxTimeToBlock) {

        // 1. 获取分区的批次队列
        Deque<ProducerBatch> deque = getOrCreateDeque(tp);

        // 2. 尝试追加到现有批次
        synchronized (deque) {
            ProducerBatch last = deque.peekLast();
            if (last != null) {
                FutureRecordMetadata future = last.tryAppend(timestamp, key, value, headers, callback);
                if (future != null) {
                    return new RecordAppendResult(future); // 追加成功
                }
            }
        }

        // 3. 现有批次满了，创建新批次
        ProducerBatch batch = new ProducerBatch(...);
        FutureRecordMetadata future = batch.tryAppend(timestamp, key, value, headers, callback);

        synchronized (deque) {
            deque.addLast(batch);
        }

        return new RecordAppendResult(future);
    }
}
```

### 6.3 分区策略

```java
// 默认分区器
public class DefaultPartitioner implements Partitioner {

    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
            Object value, byte[] valueBytes, Cluster cluster) {
        // 1. 有Key——按Key哈希分区
        if (keyBytes != null) {
            return Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions;
        }
        // 2. 无Key——Sticky Partitioner（粘性分区）
        // 同一批消息发到同一分区，减少请求次数
        int partition = stickyPartitionCache.partition(topic, cluster);
        return partition;
    }
}
```

## 7. 消费者拉取流程

### 7.1 KafkaConsumer源码

```java
public class KafkaConsumer<K, V> implements Consumer<K, V> {

    private final ConsumerNetworkClient client;  // 网络客户端
    private final Fetcher<K, V> fetcher;         // 拉取器
    private final ConsumerCoordinator coordinator; // 消费协调器
    private final SubscriptionState subscriptions; // 订阅状态

    // 拉取消息
    @Override
    public ConsumerRecords<K, V> poll(Duration timeout) {
        // 1. 确保已分配分区
        if (!hasAssignedPartitions()) {
            updateFetchPositions();
        }

        // 2. 拉取消息
        Map<TopicPartition, List<ConsumerRecord<K, V>>> records = pollForFetches(timeout);

        // 3. 触发拦截器
        return interceptors.onConsume(new ConsumerRecords<>(records));
    }

    private Map<TopicPartition, List<ConsumerRecord<K, V>>> pollForFetches(Duration timeout) {
        long now = time.milliseconds();

        // 1. 检查是否已有缓存的数据
        Map<TopicPartition, List<ConsumerRecord<K, V>>> records = fetcher.fetchedRecords();
        if (!records.isEmpty()) {
            return records;
        }

        // 2. 发送Fetch请求
        fetcher.sendFetches();

        // 3. 等待响应
        client.poll(timeout, now);

        // 4. 返回结果
        return fetcher.fetchedRecords();
    }
}
```

### 7.2 Fetcher——拉取器

```java
public class Fetcher<K, V> {

    // 发送Fetch请求
    public void sendFetches() {
        for (Map.Entry<Node, FetchSessionHandler.FetchRequestData> entry :
                fetchRequests.entrySet()) {
            Node node = entry.getKey();
            FetchRequest.Builder request = entry.getValue().toSend();

            // 发送异步请求
            client.send(request, new RequestFutureListener<ClientResponse>() {
                @Override
                public void onSuccess(ClientResponse resp) {
                    FetchResponse<Records> response = (FetchResponse<Records>) resp.responseBody();

                    // 处理响应
                    for (Map.Entry<TopicPartition, FetchResponse.PartitionData<Records>> entry :
                            response.responseData().entrySet()) {
                        TopicPartition tp = entry.getKey();
                        CompletedFetch completedFetch = new CompletedFetch(tp, entry.getValue());
                        completedFetches.add(completedFetch);
                    }
                }
            });
        }
    }

    // 解析已完成的Fetch
    public Map<TopicPartition, List<ConsumerRecord<K, V>>> fetchedRecords() {
        Map<TopicPartition, List<ConsumerRecord<K, V>>> drained = new HashMap<>();

        while (!completedFetches.isEmpty()) {
            CompletedFetch fetch = completedFetches.poll();
            List<ConsumerRecord<K, V>> records = fetchRecords(fetch);
            drained.put(fetch.partition, records);
        }

        return drained;
    }
}
```

## 9. 事务消息机制

### 9.1 事务流程

```
1. Producer初始化事务 → 获取producerId和epoch
2. beginTransaction()
3. 发送消息到多个分区（带事务标记）
4. sendOffsetsToTransaction() （消费+生产模式）
5. commitTransaction() 或 abortTransaction()
6. Coordinator写入COMMIT或ABORT标记
7. 消费者通过read_committed隔离级别只看到已提交的消息
```

## 11. 面试题速查

### Q1: Kafka为什么这么快？

四大原因：1.顺序写磁盘（600MB/s vs 随机写100KB/s）；2.零拷贝（sendfile，2次拷贝vs传统4次）；3.Page Cache（利用OS文件缓存，不维护应用层缓存）；4.批量发送+压缩（减少网络IO）。

### Q2: ISR机制是什么？

ISR（In-Sync Replicas）是 与Leader保持同步的副本集合。Follower的LEO在replica.lag.time.max.ms时间内追上Leader就留在ISR中。只有ISR中的副本才能被选为Leader。当ISR为空时，根据unclean.leader.election.enable决定是否从OSR中选举（可能丢数据）。

### Q3: Kafka的HW是什么？

HW（High Watermark）= 所有ISR副本中最小的LEO。只有HW以下的消息对消费者可见。HW保证了消息在被所有ISR副本确认后才对消费者可见，保证了一致性。Kafka 0.11引入Leader Epoch解决了HW截断导致的数据丢失问题。

### Q4: Kafka如何实现零拷贝？

使用Java NIO的FileChannel.transferTo()，底层调用Linux的sendfile系统调用。数据从磁盘到内核缓冲区后直接送到网卡，不经过用户空间，只需2次DMA拷贝+2次上下文切换（传统IO需要4次CPU拷贝+4次上下文切换）。但零拷贝只能用于消费者拉取，生产者发送仍需经过用户空间。

### Q5: Kafka的分区分配策略有哪些？

RangeAssignor（默认）：按分区范围分配给消费者。RoundRobinAssignor：轮流分配所有分区。StickyAssignor：尽量保持上次分配结果，减少Rebalance时的分区迁移。 CooperativeStickyAssignor：增量式Rebalance，不停顿消费。

### Q6: Kafka如何保证消息不丢失？

生产者端：acks=all（等待所有ISR确认）+ retries + 幂等。Broker端：min.insync.replicas≥2 + replication.factor≥3 + unclean.leader.election.enable=false。消费者端：手动提交offset + 关闭自动提交。综合配置保证消息不丢。

### Q7: Kafka的Rebalance是什么？

当消费者组中成员变化（加入/退出/崩溃）或订阅分区变化时，触发Rebalance——重新分配分区给消费者。Rebalance期间消费者无法消费（Stop The World）。通过CooperativeRebalance协议可以减少停顿。

### Q8: Kafka的稀疏索引是什么？

不是每条消息都建索引，而是每隔4KB消息数据写一条索引项。索引项格式：relativeOffset(4字节) + physicalPosition(4字节)。查找时先二分查找索引定位到大致位置，再顺序扫描.log文件。稀疏索引节省了索引文件空间，同时查找效率仍然很高（O(logN)+少量顺序扫描）。

### Q9: Kafka事务消息的原理？

Producer获取transactionalId和epoch → beginTransaction → 发送消息到多个分区（带事务标记） → commitTransaction → Coordinator向所有相关分区写入COMMIT control batch → 消费者在read_committed模式下只消费已COMMIT的消息。利用两阶段提交和幂等性实现跨分区事务。

### Q10: Kafka和RocketMQ的存储模型有什么区别？

Kafka每个分区一个独立日志文件，优点是查找快（直接定位分区），缺点是分区多时随机写严重。RocketMQ所有Topic共享一个CommitLog（顺序写），通过ConsumeQueue建立索引。RocketMQ的顺序写性能更好但查找多一步索引。Kafka适合分区数适中的大数据场景，RocketMQ适合大量Topic的业务场景。
