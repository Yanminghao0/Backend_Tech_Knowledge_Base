# RocketMQ源码解析

> RocketMQ是阿里巴巴开源的分布式消息中间件，以其高可靠、高吞吐和丰富的消息类型（普通、顺序、延迟、事务消息）著称。与Kafka的分区日志不同，RocketMQ使用所有Topic共享CommitLog的设计，在大量Topic场景下表现更优。本文深入解析RocketMQ的存储引擎、消费队列、事务消息和延迟消息机制。

## 1. RocketMQ整体架构

### 1.1 核心组件

```
┌──────────────────────────────────────────────────────────┐
│                    RocketMQ Cluster                       │
│                                                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐       │
│  │ NameServer1 │  │ NameServer2 │  │ NameServer3 │       │
│  │ (路由信息)    │  │ (独立部署)   │  │             │       │
│  └─────────────┘  └─────────────┘  └─────────────┘       │
│          ↑                ↑                ↑              │
│  ┌───────┴────────────────┴────────────────┴──────┐       │
│  │                                                │       │
│  │  ┌──────────┐     ┌──────────┐                │       │
│  │  │ Broker A │     │ Broker B │   ...          │       │
│  │  │ (Master) │←───→│ (Slave)  │                │       │
│  │  └──────────┘     └──────────┘                │       │
│  └────────────────────────────────────────────────┘       │
│         ↑                              ↑                  │
│    ┌────┴────┐                     ┌────┴────┐            │
│    │Producer │                     │Consumer │            │
│    └─────────┘                     └─────────┘            │
└──────────────────────────────────────────────────────────┘
```

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| NameServer | 轻量级路由中心，Broker注册路由信息，无状态 |
| Broker | 消息存储和转发服务器 |
| Topic | 消息主题，由多个MessageQueue组成 |
| MessageQueue | 消息队列，类似Kafka的Partition |
| CommitLog | 所有消息的统一存储文件 |
| ConsumeQueue | 消息的逻辑索引队列 |
| ConsumerGroup | 消费者组 |
| ProducerGroup | 生产者组 |

### 1.3 与Kafka的架构区别

```
Kafka:
  - 依赖ZooKeeper（或KRaft）管理元数据
  - 每个Partition独立日志文件
  - ISR机制管理副本同步

RocketMQ:
  - 自研NameServer，无状态，不依赖ZK
  - 所有Topic共享CommitLog
  - Master/Slave模式 + Dledger（Raft）选主
  - 支持丰富的消息类型
```

### 2.3 CommitLog源码

```java
public class CommitLog {

    // 默认每个文件1GB
    private final int mappedFileSize = 1024 * 1024 * 1024;

    // MappedFile队列（内存映射文件）
    private final MappedFileQueue mappedFileQueue;

    // 消息序列化器
    private final AppendMessageCallback appendMessageCallback;

    /**
     * 异步写入消息
     */
    public CompletableFuture<PutMessageResult> asyncPutMessage(
            final MessageExtBrokerInner msg) {

        // 1. 设置存储时间
        msg.setStoreTimestamp(System.currentTimeMillis());

        // 2. 获取或创建MappedFile
        MappedFile mappedFile = mappedFileQueue.getLastMappedFile();
        if (mappedFile == null || mappedFile.isFull()) {
            mappedFile = mappedFileQueue.getLastMappedFile(0, true);
        }

        // 3. 写入消息
        PutMessageResult result = mappedFile.appendMessage(msg, appendMessageCallback);

        // 4. 处理写入结果
        switch (result.getPutMessageStatus()) {
            case PUT_OK:
                break;
            case END_OF_FILE:
                // 当前文件已满，创建新文件重试
                mappedFile = mappedFileQueue.getLastMappedFile(0, true);
                result = mappedFile.appendMessage(msg, appendMessageCallback);
                break;
            default:
                return CompletableFuture.completedFuture(result);
        }

        // 5. 异步刷盘
        handleDiskFlush(result, msg);

        // 6. 异步HA同步
        handleHA(result, msg);

        return CompletableFuture.completedFuture(result);
    }
}
```

### 2.4 MappedFile——内存映射文件

```java
public class MappedFile extends ReferenceResource {

    private FileChannel fileChannel;           // 文件通道
    private MappedByteBuffer mappedByteBuffer; // 内存映射缓冲区
    private ByteBuffer writeBuffer;            // 写缓冲区（TransientStorePool模式）
    private long fileFromOffset;               // 文件起始偏移量
    private AtomicInteger wrotePosition;       // 已写入位置
    private AtomicInteger committedPosition;   // 已提交位置
    private AtomicInteger flushedPosition;     // 已刷盘位置

    /**
     * 追加消息
     */
    public PutMessageResult appendMessage(
            final MessageExtBrokerInner msg,
            final AppendMessageCallback cb) {

        // 当前写入位置
        int currentPos = wrotePosition.get();

        if (currentPos < mappedFileSize) {
            // 获取ByteBuffer切片
            ByteBuffer byteBuffer = writeBuffer != null ?
                writeBuffer.slice() : mappedByteBuffer.slice();
            byteBuffer.position(currentPos);

            // 调用回调写入消息
            AppendMessageResult result = cb.doAppend(
                fileFromOffset,    // 文件起始偏移
                byteBuffer,        // 缓冲区
                mappedFileSize - currentPos, // 剩余空间
                msg                // 消息对象
            );

            // 更新写入位置
            wrotePosition.addAndGet(result.getWroteBytes());

            return new PutMessageResult(PutMessageStatus.PUT_OK, result);
        }

        return new PutMessageResult(PutMessageStatus.END_OF_FILE, null);
    }
}
```

### 2.5 消息序列化

```java
public class DefaultAppendMessageCallback implements AppendMessageCallback {

    @Override
    public AppendMessageResult doAppend(
            long fileFromOffset,
            ByteBuffer byteBuffer,
            int maxBlank,
            MessageExtBrokerInner msg) {

        // 序列化消息到ByteBuffer
        byteBuffer.mark();

        // 1. 写入消息总长度
        int msgLen = calMsgLength(msg);
        byteBuffer.putInt(msgLen);

        // 2. 写入魔数
        byteBuffer.putInt(CommitLog.MESSAGE_MAGIC_CODE);

        // 3. 写入bodyCRC
        byteBuffer.putInt(msg.getBodyCRC());

        // 4. 写入queueId
        byteBuffer.putInt(msg.getQueueId());

        // 5. 写入flag
        byteBuffer.putInt(msg.getFlag());

        // 6. 写入queueOffset
        byteBuffer.putLong(msg.getQueueOffset());

        // 7. 写入physicalOffset
        byteBuffer.putLong(fileFromOffset + byteBuffer.position() - 8);

        // 8. 写入sysFlag
        byteBuffer.putInt(msg.getSysFlag());

        // 9. 写入bornTimestamp
        byteBuffer.putLong(msg.getBornTimestamp());

        // ... 继续写入其他字段 ...

        // N. 写入body
        byteBuffer.putInt(msg.getBody().length);
        byteBuffer.put(msg.getBody());

        // N+1. 写入topic
        byte[] topicBytes = msg.getTopic().getBytes(StandardCharsets.UTF_8);
        byteBuffer.putShort((short) topicBytes.length);
        byteBuffer.put(topicBytes);

        // N+2. 写入properties
        byte[] properties = msg.getPropertiesString().getBytes(StandardCharsets.UTF_8);
        byteBuffer.putShort((short) properties.length);
        byteBuffer.put(properties);

        byteBuffer.reset();

        return new AppendMessageResult(
            AppendMessageStatus.PUT_OK,
            msg.getQueueOffset(),
            msgLen,
            System.currentTimeMillis()
        );
    }
}
```

## 3. ConsumeQueue索引机制

### 3.1 ConsumeQueue结构

```
每个Topic的每个MessageQueue对应一个ConsumeQueue：

${ROCKETMQ_HOME}/store/consumequeue/
├── TopicA/
│   ├── 0/                          # queueId=0
│   │   ├── 00000000000000000000    # 每个文件30万条索引
│   │   └── 00000000000006000000
│   ├── 1/                          # queueId=1
│   │   └── 00000000000000000000
│   └── 2/
└── TopicB/
    └── 0/
```

### 3.2 ConsumeQueue条目格式

```
每条索引固定20字节：

┌─────────────────────────────────────────────────┐
│ 字段           │ 长度(字节) │ 说明                  │
├─────────────────────────────────────────────────┤
│ offset         │ 8          │ CommitLog物理偏移量    │
│ msgSize        │ 4          │ 消息大小              │
│ tagsCode       │ 8          │ Tag的哈希值           │
└─────────────────────────────────────────────────┘

每个ConsumeQueue文件存储300,000条索引 = 300,000 × 20 = 6,000,000字节
```

### 3.3 ConsumeQueue源码

```java
public class ConsumeQueue {

    private static final int CQ_STORE_UNIT_SIZE = 20; // 每条20字节

    private final String topic;
    private final int queueId;
    private final MappedFileQueue mappedFileQueue;
    private long maxOffsetInQueue;

    /**
     * 将CommitLog中的消息构建索引
     */
    public void putMessagePositionInfo(
            final long offset,      // CommitLog物理偏移
            final int size,         // 消息大小
            final long tagsCode,    // Tag哈希
            final long cqOffset)   // ConsumeQueue偏移
    {
        // 20字节缓冲区
        ByteBuffer byteBuffer = ByteBuffer.allocate(CQ_STORE_UNIT_SIZE);
        byteBuffer.putLong(offset);     // 8 bytes
        byteBuffer.putInt(size);        // 4 bytes
        byteBuffer.putLong(tagsCode);   // 8 bytes

        // 写入ConsumeQueue文件
        MappedFile mappedFile = mappedFileQueue.getLastMappedFile(cqOffset);
        mappedFile.appendMessage(byteBuffer.array());

        // 更新最大Offset
        this.maxOffsetInQueue = cqOffset + 1;
    }

    /**
     * 根据ConsumeQueue Offset查找CommitLog物理偏移
     */
    public long getIndexBuffer(final long startIndex) {
        int mappedFileSize = mappedFileSize();
        long targetOffset = startIndex * CQ_STORE_UNIT_SIZE;

        MappedFile targetFile = mappedFileQueue.findMappedFileByOffset(targetOffset);
        if (targetFile != null) {
            ByteBuffer byteBuffer = targetFile.sliceByteBuffer();
            byteBuffer.position((int)(targetOffset % mappedFileSize));
            return byteBuffer.getLong(); // 返回CommitLog物理偏移
        }
        return -1;
    }
}
```

```java
// ReputMessageService——后台线程，持续从CommitLog构建ConsumeQueue
public class ReputMessageService extends ServiceThread {

    @Override
    public void run() {
        while (!isStopped()) {
            try {
                Thread.sleep(1);
                doReput();
            } catch (Exception e) {
                log.warn("reput error", e);
            }
        }
    }

    private void doReput() {
        for (int readSize = 0; readSize < result.getSize() && !isStopped(); ) {
            // 1. 从CommitLog读取消息
            DispatchRequest dispatchRequest = commitLog.checkMessageAndReturnSize(...);

            // 2. 构建ConsumeQueue索引
            if (dispatchRequest.isSuccess()) {
                // 写入ConsumeQueue
                DefaultMessageStore.this.putMessagePositionInfo(
                    dispatchRequest.getTopic(),
                    dispatchRequest.getQueueId(),
                    dispatchRequest.getCommitLogOffset(), // CommitLog偏移
                    dispatchRequest.getMsgSize(),
                    dispatchRequest.getTagsCode()
                );

                // 3. 构建IndexFile索引（用于按Key查找）
                DefaultMessageStore.this.indexService.buildIndex(dispatchRequest);
            }
        }
    }
}
```

## 4. 消息存储流程

### 4.1 完整存储流程

```
Producer → Broker → CommitLog → ReputMessageService → ConsumeQueue + IndexFile
                           ↓
                        刷盘（Sync/Async）
                           ↓
                        HA同步到Slave
```

### 4.2 DefaultMessageStore

```java
public class DefaultMessageStore implements MessageStore {

    private final CommitLog commitLog;
    private final ConsumeQueueStore consumeQueueStore;
    private final IndexService indexService;
    private final ReputMessageService reputMessageService;
    private final HAService haService;

    /**
     * 存储消息
     */
    @Override
    public PutMessageResult putMessage(MessageExtBrokerInner msg) {
        // 1. 写入CommitLog
        PutMessageResult result = commitLog.putMessage(msg);

        // 2. 如果写入成功，ConsumeQueue由ReputMessageService异步构建
        // （不需要同步等待ConsumeQueue构建）

        return result;
    }

    /**
     * 获取消息
     */
    @Override
    public GetMessageResult getMessage(
            final String group,
            final String topic,
            final int queueId,
            final long offset,
            final int maxMsgNums,
            final MessageFilter messageFilter) {

        // 1. 根据topic+queueId+offset查找ConsumeQueue
        ConsumeQueue consumeQueue = findConsumeQueue(topic, queueId);
        if (consumeQueue == null) {
            return GetMessageResult.noMessageInQueue();
        }

        // 2. 从ConsumeQueue获取CommitLog物理偏移量
        SelectMappedBufferResult buffer = consumeQueue.getIndexBuffer(offset);

        // 3. 根据物理偏移量从CommitLog读取消息
        GetMessageResult getResult = new GetMessageResult();
        for (int i = 0; i < buffer.getSize() / 20 && i < maxMsgNums; i++) {
            long phyOffset = buffer.getByteBuffer().getLong();
            int size = buffer.getByteBuffer().getInt();
            long tagsCode = buffer.getByteBuffer().getLong();

            // 4. Tag过滤
            if (messageFilter != null && !messageFilter.isMatchedByConsumeQueue(tagsCode)) {
                continue;
            }

            // 5. 从CommitLog读取完整消息
            SelectMappedBufferResult msgResult = commitLog.getMessage(phyOffset, size);

            // 6. Tag精确过滤
            if (messageFilter != null && !messageFilter.isMatchedByCommitLog(msgResult.getByteBuffer())) {
                continue;
            }

            getResult.addMessage(msgResult);
        }

        return getResult;
    }
}
```

### 4.3 刷盘策略

```java
// CommitLog的刷盘处理
public class CommitLog {

    public void handleDiskFlush(PutMessageResult result, MessageExtBrokerInner msg) {
        if (FlushDiskType.SYNC_FLUSH == defaultMessageStore.getMessageStoreConfig().getFlushDiskType()) {
            // 同步刷盘——等待刷盘完成才返回
            GroupCommitService service = (GroupCommitService) flushManager;
            service.putRequest(new GroupCommitRequest(result.getAppendMessageResult().getWroteOffset()));
            service.waitForRunning(5000);
        } else {
            // 异步刷盘——立即返回
            flushManager.wakeUpFlush();
        }
    }
}

// 同步刷盘服务
class GroupCommitService extends FlushManager {

    private void doCommit() {
        for (GroupCommitRequest req : requestsRead) {
            // 等待wrotePosition被刷到flushedPosition
            boolean flushOK = false;
            for (int i = 0; i < 2 && !flushOK; i++) {
                flushOK = mappedFile.getFlushedPosition() >= req.getNextOffset();
                if (!flushOK) {
                    mappedFileQueue.flush(0);
                }
            }

            req.wakeupCustomer(flushOK);
        }
    }
}
```

## 6. 事务消息机制

### 6.1 事务消息流程

```
1. Producer发送半消息（Half Message）→ Broker
2. Broker存储半消息（对消费者不可见）
3. Broker返回发送成功
4. Producer执行本地事务
5. Producer提交事务状态（Commit/Rollback/Unknown）→ Broker
6. 如果Commit → 半消息变为可见
   如果Rollback → 删除半消息
   如果Unknown → Broker回查Producer

回查机制：
7. Broker定期回查Producer事务状态
8. Producer根据本地事务状态返回Commit/Rollback
```

### 6.2 半消息存储

```java
// 事务消息的Topic为 RMQ_SYS_TRANS_HALF_TOPIC
// 消费者不会订阅这个Topic

public class EndTransactionProcessor implements NettyRequestProcessor {

    @Override
    public RemotingCommand processRequest(ChannelHandlerContext ctx, RemotingCommand request) {
        EndTransactionRequestHeader requestHeader = request.decodeCommandCustomHeader();

        // 根据事务状态处理
        switch (requestHeader.getCommitOrRollback()) {
            case MessageSysFlag.TRANSACTION_COMMIT_TYPE:
                // 提交——将半消息从half topic移到真实topic
                OperationResult result = commitMessage(requestHeader);
                // 恢复真实Topic和QueueId
                MessageExtBrokerInner msgInner = endMessageTransaction(result.getPrepareMessage());
                msgInner.setTopic(msgInner.getProperty(MessageConst.PROPERTY_REAL_TOPIC));
                msgInner.setQueueId(Integer.parseInt(
                    msgInner.getProperty(MessageConst.PROPERTY_REAL_QUEUE_ID)));
                // 写入CommitLog（消费者可见）
                messageStore.putMessage(msgInner);
                break;

            case MessageSysFlag.TRANSACTION_ROLLBACK_TYPE:
                // 回滚——标记删除半消息
                break;
        }
    }
}
```

### 6.3 事务回查

```java
// 事务消息回查服务
public class TransactionalMessageService {

    // 定期检查未决事务消息
    public void check(long transactionTimeout, int transactionCheckMax,
            AbstractTransactionalMessageCheckListener listener) {

        // 1. 从RMQ_SYS_TRANS_HALF_TOPIC读取半消息
        String topic = TopicValidator.RMQ_SYS_TRANS_HALF_TOPIC;
        MessageQueue mq = new MessageQueue(topic, brokerName, 0);

        // 2. 遍历半消息
        for (long offset = startOffset; offset < endOffset; offset++) {
            MessageExt msg = getMessage(mq, offset);

            // 3. 判断是否需要回查
            long bornTime = msg.getBornTimestamp();
            if (System.currentTimeMillis() - bornTime > transactionTimeout) {
                // 超时未提交，发起回查
                listener.resolveHalfMsg(msg);
            }
        }
    }
}

// 回查Producer
public abstract class AbstractTransactionalMessageCheckListener {

    public void resolveHalfMsg(final MessageExt msg) {
        executorService.execute(new Runnable() {
            @Override
            public void run() {
                // 向Producer发送回查请求
                requestProcessor.checkTransactionState(
                    msg.getProperty(MessageConst.PROPERTY_PRODUCER_GROUP),
                    msg.getStoreHost(),
                    msg
                );
            }
        });
    }
}
```

## 7. 延迟消息机制

### 7.1 延迟级别

```
RocketMQ支持18个固定延迟级别：
1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h

延迟消息原理：
1. Producer设置delayLevel
2. Broker收到后，不放入真实Topic，而是放入 SCHEDULE_TOPIC_XXXX
3. 后台定时任务扫描到期消息
4. 到期消息恢复真实Topic，消费者可见
```

### 7.2 延迟消息源码

```java
// 存储延迟消息时修改Topic
public class CommitLog {

    public PutMessageResult asyncPutMessage(MessageExtBrokerInner msg) {
        // 处理延迟消息
        final int tranType = MessageSysFlag.getTransactionValue(msg.getSysFlag());
        if (tranType == MessageSysFlag.TRANSACTION_NOT_TYPE) {
            if (msg.getDelayTimeLevel() > 0) {
                // 保存原始Topic和QueueId
                MessageAccessor.putProperty(msg, MessageConst.PROPERTY_REAL_TOPIC, msg.getTopic());
                MessageAccessor.putProperty(msg, MessageConst.PROPERTY_REAL_QUEUE_ID,
                    String.valueOf(msg.getQueueId()));

                // 修改Topic为延迟Topic
                msg.setTopic(TopicValidator.RMQ_SYS_SCHEDULE_TOPIC);
                // 修改QueueId为延迟级别-1
                msg.setQueueId(DelayMessageQueueId(msg.getDelayTimeLevel() - 1));
            }
        }
    }
}

// 延迟消息调度服务
public class ScheduleMessageService {

    private static final String TOPIC = TopicValidator.RMQ_SYS_SCHEDULE_TOPIC;

    // 延迟级别→延迟时间映射
    private ConcurrentMap<Integer, Long> delayLevelTable;

    // 每个延迟级别一个定时任务
    public void start() {
        for (Map.Entry<Integer, Long> entry : delayLevelTable.entrySet()) {
            Integer level = entry.getKey();
            Long timeDelay = entry.getValue();

            Long offset = this.offsetTable.get(level);
            if (null == offset) {
                offset = 0L;
            }

            // 启动定时任务
            timer.schedule(new DeliverDelayedMessageTimerTask(level, offset),
                timeDelay * 1000);
        }
    }

    class DeliverDelayedMessageTimerTask extends TimerTask {

        @Override
        public void run() {
            try {
                deliverDelayedMessage();
            } finally {
                // 重新调度
                timer.schedule(new DeliverDelayedMessageTimerTask(level, offset),
                    delayLevelTable.get(level) * 1000);
            }
        }

        private void deliverDelayedMessage() {
            // 1. 从SCHEDULE_TOPIC_XXXX读取到期的消息
            ConsumeQueue cq = findConsumeQueue(TOPIC, level - 1);

            // 2. 遍历消息
            for (long offset = startOffset; offset < maxOffset; offset++) {
                MessageExt msg = getMessage(TOPIC, level - 1, offset);

                // 3. 检查是否到期
                long bornTime = msg.getBornTimestamp();
                long delayTime = delayLevelTable.get(level);
                if (System.currentTimeMillis() < bornTime + delayTime * 1000) {
                    break; // 未到期，停止扫描
                }

                // 4. 恢复真实Topic和QueueId
                MessageAccessor.putProperty(msg, MessageConst.PROPERTY_REAL_TOPIC,
                    msg.getProperty(MessageConst.PROPERTY_REAL_TOPIC));
                msg.setTopic(msg.getProperty(MessageConst.PROPERTY_REAL_TOPIC));
                msg.setQueueId(Integer.parseInt(
                    msg.getProperty(MessageConst.PROPERTY_REAL_QUEUE_ID)));
                msg.setDelayTimeLevel(0);

                // 5. 重新写入CommitLog（消费者可见）
                MessageExtBrokerInner msgInner = toMessageExtBrokerInner(msg);
                messageStore.putMessage(msgInner);
            }
        }
    }
}
```

## 8. 顺序消息机制

### 8.1 顺序消息原理

```
全局顺序：一个Topic只有一个MessageQueue
  → 吞吐量极低，不推荐

分区顺序：同一业务Key的消息发到同一MessageQueue
  → 同一业务Key的消息在同一个Queue中有序
  → 推荐方式
```

### 8.2 生产者——顺序发送

```java
public class DefaultMQProducer {

        MessageQueue mq = selector.select(messageQueues, msg, arg);

        // 3. 发送到选定的Queue

    }
}
```

### 8.3 消费者——顺序消费

```java
public class ConsumeMessageOrderlyService implements ConsumeMessageService {

    // 顺序消费——对MessageQueue加锁
    @Override
    public void submitConsumeRequest(List<MessageExt> msgs, ProcessQueue pq, MessageQueue mq) {
        ConsumeRequest consumeRequest = new ConsumeRequest(msgs, pq, mq);
        consumeExecutor.submit(consumeRequest);
    }

    class ConsumeRequest implements Runnable {
        @Override
        public void run() {
            // 1. 对MessageQueue加锁（保证同一Queue串行消费）
            final Object objLock = messageQueueLock.fetchLockObject(messageQueue);
            synchronized (objLock) {
                // 2. 加分布式锁（保证集群中只有一个消费者消费这个Queue）
                if (lockManager.isLocked(messageQueue)) {
                    // 3. 逐条消费
                    for (MessageExt msg : msgs) {
                        ConsumeOrderlyContext context = new ConsumeOrderlyContext(messageQueue);
                        ConsumeOrderlyStatus status = messageListener.consumeMessage(msg, context);

                        switch (status) {
                            case SUCCESS:
                                // 消费成功，提交
                                processQueue.removeMessage(Collections.singletonList(msg));
                                break;
                            case SUSPEND_CURRENT_QUEUE_A_MOMENT:
                                // 消费失败，暂停一会重试（不跳过）
                                break;
                        }
                    }
                }
            }
        }
    }
}
```

### 9.1 HAService

```java
public class HAService {

    private final List<HAConnection> connectionList;
    private AcceptSocketService acceptSocketService;

    // Master向Slave同步数据
    class HAConnection {

        // 读取CommitLog数据发送给Slave
        class WriteSocketService extends ServiceThread {
            @Override
            public void run() {
                while (!isStopped()) {
                    long masterOffset = commitLog.getMaxOffset();
                    SelectMappedBufferResult selectResult = commitLog.getData(
                        slaveRequestOffset, slaveRequestOffset == 0 ? 1024 : 0);

                    if (selectResult != null) {
                        // 发送数据到Slave
                        byteBuffer.flip();
                        this.socketChannel.write(byteBuffer);
                        slaveAckOffset = slaveRequestOffset + selectResult.getSize();
                    }
                }
            }
        }

        // 接收Slave的ACK
        class ReadSocketService extends ServiceThread {
            @Override
            public void run() {
                while (!isStopped()) {
                    // 读取Slave的ACK Offset
                    int readSize = socketChannel.read(byteBufferRead);
                    long readOffset = byteBufferRead.getLong();

                    // 更新Slave的ACK Offset
                    slaveAckOffset = readOffset;
                }
            }
        }
    }
```

### 9.2 主从同步模式

```
ASYNC_MASTER（异步主从）：
  Producer → Master写入即返回 → 异步同步到Slave
  优点：低延迟
  缺点：可能丢失数据

SYNC_MASTER（同步主从）：
  Producer → Master写入 → 等待Slave同步 → 返回

  缺点：延迟高
```

## 10. RocketMQ vs Kafka存储对比

### 10.1 存储模型对比

```
Kafka（每个Partition独立日志）：
  TopicA-0/
  ├── 00000000000000000000.log
  └── 00000000000000000000.index
  TopicA-1/
  ├── 00000000000000000000.log
  └── 00000000000000000000.index
  TopicB-0/
  └── ...

优点：查找快（直接定位分区日志）
缺点：Topic/Partition多时随机写严重（多个文件追加写）

RocketMQ（所有Topic共享CommitLog）：
  commitlog/
  ├── 00000000000000000000    ← 所有Topic的消息混在一起
  └── 00000000001073741824
  consumequeue/
  ├── TopicA/0/    ← 索引指向CommitLog
  ├── TopicA/1/
  └── TopicB/0/

优点：顺序写（只有一个CommitLog在写），多Topic场景吞吐稳定
缺点：读取多一步索引查找（ConsumeQueue → CommitLog）
```

### 10.2 零拷贝对比

```
Kafka：sendfile（2次DMA拷贝）
  - 数据不经过用户空间
  - 适合大文件传输

RocketMQ：mmap + write（2次CPU拷贝 + 2次DMA拷贝）
  - 数据映射到用户空间内存
  - 适合小块数据随机读
  - 读取后可以做Tag过滤
```

## 11. 面试题速查

### Q1: RocketMQ的CommitLog和ConsumeQueue是什么关系？

CommitLog是所有Topic消息的统一存储文件（1GB一个文件），所有消息顺序写入CommitLog。ConsumeQueue是消息的逻辑索引队列，每条索引20字节（物理偏移+消息大小+Tag哈希），指向CommitLog中的消息位置。消费者通过ConsumeQueue找到物理偏移，再从CommitLog读取消息。

### Q2: RocketMQ为什么用共享CommitLog而不是像Kafka那样每个分区独立日志？

共享CommitLog保证了所有消息都是顺序写（只有一个文件在追加），在大量Topic场景下吞吐量稳定。Kafka的每个分区独立日志在Topic/分区数多时会产生大量随机写。代价是RocketMQ读取需要多一步ConsumeQueue索引查找。

### Q3: RocketMQ的事务消息原理是什么？

半消息机制：1.Producer发送半消息到Broker（存入特殊Topic，消费者不可见）；2.Producer执行本地事务；3.Producer提交Commit/Rollback给Broker；4.Commit则消息变为可见，Rollback则删除。如果Producer超时未提交，Broker定期回查Producer事务状态。

### Q4: RocketMQ的延迟消息为什么只有18个固定级别？

延迟消息存储在特殊Topic中，每个延迟级别对应一个Queue。后台定时任务扫描到期消息。固定级别简化了实现——同一级别的消息在同一个Queue中按时间有序，扫描效率高。自定义延迟时间需要遍历所有消息，性能差。RocketMQ 5.0引入了基于时间轮的任意延迟消息。

### Q5: RocketMQ如何实现顺序消息？

分区顺序：生产者按业务Key（如orderId）的哈希选择MessageQueue，保证同一Key的消息在同一Queue中有序。消费者使用ConsumeMessageOrderlyService串行消费同一Queue的消息——对Queue加锁，逐条消费，失败重试不跳过。同时有分布式锁保证集群中同一Queue只有一个消费者消费。

### Q6: RocketMQ的NameServer和Kafka的ZooKeeper有什么区别？

NameServer是无状态的路由中心，节点间互不通信，Broker向所有NameServer注册。ZooKeeper是有状态的协调服务，需要多数派达成一致。NameServer更轻量，部署简单，但一致性较弱（各节点数据可能短暂不一致）。ZooKeeper一致性强但运维复杂。

### Q7: RocketMQ如何保证消息不丢失？

生产者：同步发送+重试。Broker：同步刷盘（SYNC_FLUSH）+ 同步主从（SYNC_MASTER）。消费者：手动ACK，消费成功后才提交Offset。极端情况下使用事务消息保证生产端+本地事务的原子性。

### Q8: RocketMQ的刷盘策略有哪些？

同步刷盘（SYNC_FLUSH）：消息写入内存后等待刷盘完成才返回，数据安全但延迟高。异步刷盘（ASYNC_FLUSH）：消息写入内存立即返回，后台线程定期刷盘，延迟低但宕机可能丢数据。生产环境推荐异步刷盘+同步主从，兼顾性能和安全。

### Q9: RocketMQ的Tag过滤是如何实现的？

两阶段过滤：ConsumeQueue中存储了Tag的hashCode（8字节），消费者拉取时先在Broker端按hashCode粗过滤（快速跳过不匹配的消息）。然后从CommitLog读取完整消息后，按Tag字符串精确过滤。两阶段过滤减少了CommitLog的读取量。

### Q10: RocketMQ和Kafka的零拷贝方式有什么区别？

Kafka使用sendfile（FileChannel.transferTo），数据直接从内核缓冲区到网卡，不经过用户空间，2次DMA拷贝。RocketMQ使用mmap（MappedByteBuffer），数据映射到用户空间内存，读取后可以过滤再发送，2次CPU拷贝+2次DMA拷贝。Kafka方式更快但不能过滤，RocketMQ方式稍慢但灵活。
