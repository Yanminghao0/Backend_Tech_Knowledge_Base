## 1. String 类定义与不可变性

### 1.1 类声明

```java
// Java 8 及之前
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {

    /** The value is used for character storage. */
    private final char value[];

    /** Cache the hash code for the string */
    private int hash; // Default to 0
    // ...
}
```

### 1.2 不可变性的三层保证

1. **类被 `final` 修饰**：`String` 类不可被继承，防止子类破坏不可变性。
2. **字段被 `final` 修饰**：`private final char[] value`（Java 9+ 为 `byte[] value`），引用不可变。
3. **数组私有且无修改入口**：`value` 是 `private` 的，且没有任何方法暴露对内部数组的直接修改。

```java
// 所有"修改"操作都返回新 String 对象
String s = "hello";
s.concat(" world"); // 返回新对象 "hello world"
System.out.println(s); // 仍然是 "hello"
```

### 1.3 不可变的好处

| 好处 | 说明 |
|---|---|
| **线程安全** | 不可变对象天生线程安全，无需同步 |
| **哈希缓存** | hashCode 只需计算一次，后续直接返回缓存值 |
| **常量池复用** | 相同字面量共享同一引用，节省内存 |
| **安全性** | 作为参数传递时不会被意外修改（如数据库 URL、文件路径） |
| **用作 HashMap key** | 不可变保证 hashCode 稳定，不会出现 key 丢失 |

### 1.4 不可变性的"破坏"——反射

```java
String s = "Hello";
Field valueField = String.class.getDeclaredField("value");
valueField.setAccessible(true);
char[] value = (char[]) valueField.get(s);
value[0] = 'h';
System.out.println(s); // "hello" — 反射可以修改，但不推荐

// Java 9+ 需要处理 byte[] 和 coder
```

> 反射可以绕过 `final` 限制，但这会破坏 JVM 优化并可能导致不可预知的行为。

## 2. 字符串常量池

### 2.1 常量池的位置演变

| JDK 版本 | 常量池位置 | 说明 |
|---|---|---|
| JDK 1.6 及之前 | 永久代（PermGen） | 常量池在方法区内 |
| JDK 1.7 | 堆（Heap） | 常量池从永久代移到堆中 |
| JDK 1.8+ | 堆（Heap） | 永久代被元空间（Metaspace）取代，常量池仍在堆中 |

> 移到堆中的主要原因：永久代大小固定，容易 OOM（`java.lang.OutOfMemoryError: PermGen space`）。

### 2.2 常量池的工作机制

```java
// 字面量赋值：直接引用常量池中的对象
String s1 = "abc";
String s2 = "abc";
System.out.println(s1 == s2); // true → 常量池复用

// new String：在堆中创建新对象
String s3 = new String("abc");
System.out.println(s1 == s3); // false → 一个在常量池，一个在堆
```

```
内存结构（JDK 8+）:

┌─────────────── 堆 (Heap) ───────────────┐
│                                          │
│  ┌──── 字符串常量池 (String Pool) ────┐ │
│  │  "abc" ──→ String对象              │ │
│  │  "def" ──→ String对象              │ │
│  └────────────────────────────────────┘ │
│                                          │
│  ┌──── 普通 Java 对象区 ──────────────┐ │
│  │  new String("abc") ──→ String对象   │ │  ← s3 指向这里
│  └────────────────────────────────────┘ │
│                                          │
└──────────────────────────────────────────┘
```

### 2.3 常量池的创建时机

```java
// "ab" 在编译期就能确定，放入 Class 文件常量池
// 类加载时，"ab" 进入字符串常量池（懒加载：首次使用时）
String s = "ab";

// 以下不会在编译期确定：
String s2 = "a" + "b";        // 编译期优化为 "ab"，等价于 "ab"
String s3 = "a" + new String("b"); // 运行期拼接，不等价于 "ab"
```

## 3. intern() 方法原理

```java
/**
 * Returns a canonical representation for the string object.
 * <p>
 * A pool of strings, initially empty, is maintained privately by the
 * class {@code String}.
 * <p>
 * When the intern method is invoked, if the pool already contains a
 * string equal to this {@code String} object as determined by
 * the {@link #equals(Object)} method, then the string from the pool is
 * returned. Otherwise, this {@code String} object is added to the
 * pool and a reference to this {@code String} object is returned.
 */
public native String intern();
```

### 3.1 intern() 的行为

```java
String s1 = new String("hello");    // 堆中创建新对象，"hello" 已在常量池
String s2 = "hello";                // 引用常量池中的 "hello"
String s3 = s1.intern();            // 返回常量池中的 "hello"

System.out.println(s1 == s2); // false → s1 是堆对象，s2 是常量池对象
System.out.println(s2 == s3); // true  → 都指向常量池
System.out.println(s1 == s3); // false → s1 是堆对象
```

### 3.2 JDK 6 vs JDK 7+ 的 intern() 差异

**经典面试题：**

```java
String s1 = new String("1");
s1.intern();
String s2 = "1";
System.out.println(s1 == s2); // JDK6: false  JDK7+: false

String s3 = new String("1") + new String("1");
// s3 是动态拼接结果，"11" 此时尚未在常量池中
s3.intern();
String s4 = "11";
System.out.println(s3 == s4); // JDK6: false  JDK7+: true
```

**JDK 6 的 intern()：**
- 常量池在永久代。
- `intern()` 时，如果常量池没有，会**复制**字符串到常量池。
- `s3.intern()` 复制一份到永久代，`s4` 指向永久代中的副本，`s3` 指向堆中原始对象 → `false`。

**JDK 7+ 的 intern()：**
- 常量池在堆中。
- `intern()` 时，如果常量池没有，会**存储堆中对象的引用**（而非复制）。
- `s3.intern()` 将 `s3` 的引用放入常量池，`s4` 通过常量池找到的就是 `s3` → `true`。

```
JDK 7+ intern() 流程:
┌──────── 堆 ────────┐
│                      │
│  常量池: ["11" 的引用] ──→ ┐
│                          │
│  s3 指向的 String("11") ◄┘
│                      │
└──────────────────────┘
```

### 3.3 intern() 的使用场景

```java
// 大量重复字符串的场景，如读取文件中的城市名
// 不使用 intern：每个城市名在堆中可能有多个副本
// 使用 intern：相同城市名共享一个引用

BufferedReader reader = new BufferedReader(new FileReader("data.csv"));
String line;
List<String> cities = new ArrayList<>();
while ((line = reader.readLine()) != null) {
    String city = line.split(",")[2];
    cities.add(city.intern()); // 节省内存
}
```

> **注意**：过度使用 `intern()` 会导致常量池膨胀。JDK 6 中可能导致永久代 OOM。JDK 7+ 中常量池在堆中，可通过 `-XX:StringTableSize` 调整桶数量（默认 60013，JDK 11+ 可动态调整）。

## 4. new String("abc") 创建了几个对象

这是一个经典面试题，答案取决于上下文。

### 4.1 答案：1 个或 2 个

```java
String s = new String("abc");
```

- **如果 "abc" 已在常量池中**：只创建 1 个堆中的 String 对象。
- **如果 "abc" 不在常量池中**：创建 2 个对象——1 个在常量池（字面量 "abc"），1 个在堆中（new 出来的）。

### 4.2 详细分析

```
编译阶段:
    "abc" 字面量被写入 Class 文件的常量池（Constant Pool）

类加载阶段:
    Class 文件常量池中的 "abc" 被加载到运行时常量池

首次执行到 new String("abc") 时:
    1. "abc" 字符串字面量被"驻留"到字符串常量池（StringTable）→ 第1个对象
    2. new String() 在堆中创建新对象，value 指向常量池中 "abc" 的 char[] → 第2个对象
```

### 4.3 更多情况

```java
// 情况1: new String("ab") — "ab" 首次出现
String s1 = new String("ab");  // 2个对象（常量池1 + 堆1）

// 情况2: new String("ab") — "ab" 已在常量池
String s2 = new String("ab");  // 1个对象（堆1）

// 情况3: new String("a") + new String("b")
String s3 = new String("a") + new String("b");
// "a" 常量池1 + "a" 堆1 + "b" 常量池1 + "b" 堆1 + "ab" 堆1（StringBuilder 拼接结果）
// = 5个对象（首次执行时）

// 情况4: new String(new char[]{'a','b'})
String s4 = new String(new char[]{'a', 'b'});
// char[] 1个 + String 1个 = 2个对象，没有常量池对象（没有字面量 "ab"）
```

## 5. String 核心方法源码解析

### 5.1 构造函数

```java
// Java 8
public String(String original) {
    this.value = original.value;
    this.hash = original.hash;
}

public String(char value[]) {
    this.value = Arrays.copyOf(value, value.length);
}

// Java 9+ 引入 coder
public String(byte[] value) {
    this.value = value;
    this.coder = LATIN1; // 默认 Latin-1 编码
}
```

### 5.2 equals()

```java
public boolean equals(Object anObject) {
    if (this == anObject) {
        return true;
    }
    if (anObject instanceof String) {
        String anotherString = (String) anObject;
        int n = value.length;
        if (n == anotherString.value.length) {
            char[] v1 = value;
            char[] v2 = anotherString.value;
            int i = 0;
            while (n-- != 0) {
                if (v1[i] != v2[i]) return false;
                i++;
            }
            return true;
        }
    }
    return false;
}
```

**优化点**：
1. 先用 `==` 快速判断是否同一引用。
2. 用 `instanceof` 做类型检查。
3. 先比较长度，长度不同直接返回 false（快速失败）。
4. 逐字符比较。

### 5.3 hashCode()

```java
public int hashCode() {
    int h = hash;
    if (h == 0 && value.length > 0) {
        char val[] = value;
        for (int i = 0; i < value.length; i++) {
            h = 31 * h + val[i]; // 经典的 31 倍哈希
        }
        hash = h; // 缓存
    }
    return h;
}
```

**关键设计**：
- **延迟计算 + 缓存**：第一次调用时计算，后续直接返回缓存值。这是不可变对象的经典优化。
- **算法**：`h = 31 * h + char[i]`，等价于 `s[0]*31^(n-1) + s[1]*31^(n-2) + ... + s[n-1]`。
- **为什么用 31**：奇质数，分布均匀；`31*i == (i<<5)-i` 可被 JVM 优化为位移+减法。

### 5.4 substring()

```java
// Java 7+ (修复了内存泄漏)
public String substring(int beginIndex, int endIndex) {
    // 参数校验省略...
    int subLen = endIndex - beginIndex;
    // ...
    return new String(value, beginIndex, subLen); // 创建新对象，复制数组
}

// Java 6 (存在内存泄漏)
public String substring(int beginIndex, int endIndex) {
    // ...
    return new String(offset + beginIndex, endIndex - beginIndex, value);
    // 共享原 char[] value，只是改变 offset 和 count
    // 如果原字符串很大，截取一小段后，原 char[] 无法被 GC → 内存泄漏
}
```

> **Java 6 → 7 的变更**：Java 6 的 `substring` 共享 `char[]` 以节省内存，但在大字符串截取小片段时会导致内存泄漏。Java 7+ 改为复制数组，牺牲一点性能换取正确性。

### 5.5 concat()

```java
public String concat(String str) {
    int otherLen = str.length();
    if (otherLen == 0) {
        return this; // 空字符串直接返回原对象
    }
    int len = value.length;
    char buf[] = Arrays.copyOf(value, len + otherLen);
    str.getChars(buf, len); // 将 str 复制到 buf 的后半部分
    return new String(buf, true); // 新建 String 对象
}
```

### 5.6 replace()

```java
public String replace(char oldChar, char newChar) {
    if (oldChar != newChar) {
        int len = value.length;
        int i = -1;
        char[] val = value;

        // 先查找第一个匹配位置
        while (++i < len) {
            if (val[i] == oldChar) break;
        }
        // 找到匹配才创建新数组
        if (i < len) {
            char buf[] = new char[len];
            for (int j = 0; j < i; j++) {
                buf[j] = val[j]; // 复制匹配前的部分
            }
            while (i < len) {
                char c = val[i];
                buf[i] = (c == oldChar) ? newChar : c; // 替换
                i++;
            }
            return new String(buf, true);
        }
    }
    return this; // 无需替换，返回原对象
}
```

**优化点**：先扫描找到第一个匹配，没有匹配直接返回原对象（不创建新数组）。

## 6. StringBuilder vs StringBuffer

### 6.1 继承关系

```
AbstractStringBuilder (abstract)
    ├── StringBuilder   (非线程安全)
    └── StringBuffer    (线程安全，synchronized)
```

```java
// AbstractStringBuilder 核心字段
abstract class AbstractStringBuilder implements Appendable, CharSequence {
    char[] value; // 注意：不是 final！可变
    int count;    // 实际使用的长度

    // 扩容逻辑
    private int newCapacity(int minCapacity) {
        int oldCapacity = value.length >> 1; // 扩容为 (当前容量 + 2) * 2? 不，是：
        // 实际: (value.length << 1) + 2
        int newCapacity = (value.length << 1) + 2 - oldCapacity;
        // 简化理解：newCapacity = oldCapacity * 2 + 2
        if (newCapacity - minCapacity < 0) {
            newCapacity = minCapacity;
        }
        return (newCapacity <= 0 || MAX_ARRAY_SIZE - newCapacity < 0)
            ? hugeCapacity(minCapacity)
            : newCapacity;
    }
}
```

### 6.2 对比

| 维度 | StringBuilder | StringBuffer |
|---|---|---|
| 线程安全 | 否 | 是（方法用 `synchronized` 修饰） |
| 性能 | 高 | 低（锁开销） |
| 引入版本 | JDK 1.5 | JDK 1.0 |
| 典型场景 | 单线程字符串拼接 | 多线程字符串拼接 |

### 6.3 append 方法对比

```java
// StringBuilder (无锁)
@Override
public StringBuilder append(String str) {
    super.append(str);
    return this;
}

// StringBuffer (加锁)
@Override
public synchronized StringBuffer append(String str) {
    toStringCache = null;
    super.append(str);
    return this;
}
```

### 6.4 什么时候用 StringBuffer？

实际上，**绝大多数场景都应该用 `StringBuilder`**。`StringBuffer` 的 `synchronized` 在方法级别，粒度太粗，在现代 JVM 中，如果确认是单线程使用，锁优化（偏向锁→轻量级锁）可以消除大部分开销，但仍然不如 `StringBuilder` 直接。

`StringBuffer` 适用于**多线程共享同一个字符串缓冲区**的场景，但这种场景本身就很罕见——通常多线程下应该用 `ConcurrentHashMap`、`BlockingQueue` 等更合适的数据结构。

### 6.5 StringBuilder 扩容机制

```java
// 初始容量默认 16
StringBuilder sb = new StringBuilder();       // capacity = 16
StringBuilder sb2 = new StringBuilder("abc"); // capacity = 16 + 3 = 19

// 扩容公式: newCapacity = oldCapacity * 2 + 2
// 16 → 34 → 70 → 142 → ...
// 如果需要的容量超过上述值，则直接使用需要的容量

sb.append("very long string..."); // 触发扩容
```

## 7. Java 9 Compact Strings

### 7.1 背景

Java 8 及之前，String 内部使用 `char[]` 存储，每个字符占 2 字节（UTF-16）。但绝大多数应用中的字符串都是纯 Latin-1（ASCII + 欧洲字符），只需要 1 字节。这导致**约 50% 的内存浪费**。

### 7.2 Compact Strings 设计

Java 9 引入了 Compact Strings（JEP 254），将 String 内部存储从 `char[]` 改为 `byte[]` + `coder`：

```java
// Java 9+
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence,
               Constable, ConstantDesc {

    /** The value is used for byte storage. */
    private final byte[] value;

    /**
     * The identifier of the encoding used to encode the bytes in value.
     * Supported values are LATIN1 (0) and UTF16 (1).
     */
    private final byte coder;

    /** Cache the hash code for the string */
    private int hash;

    // ...
    static final boolean COMPACT_STRINGS;
    static {
        COMPACT_STRINGS = true; // 默认开启，可用 -XX:-CompactStrings 关闭
    }
}
```

### 7.3 两种编码模式

```java
static final byte LATIN1 = 0;  // 每个字符1字节，支持 ISO-8859-1
static final byte UTF16  = 1;  // 每个字符2字节，支持完整 Unicode
```

```java
// String 构造时判断编码
private String(byte[] value, byte coder) {
    this.value = value;
    this.coder = coder;
}

// 如果字符串只包含 Latin-1 字符 → coder = LATIN1 → byte[] 每个字节一个字符
// 如果包含非 Latin-1 字符（如中文） → coder = UTF16 → byte[] 每两个字节一个字符
```

### 7.4 效果

```java
// Java 8: "hello" 占用 char[5] = 10 字节（+ 对象头 16 字节 = 26 字节 → 对齐 32 字节）
// Java 9: "hello" 占用 byte[5] = 5 字节（+ 对象头 16 字节 + coder 1 字节 = 22 字节 → 对齐 24 字节）

// "你好" Java 8: char[2] = 4 字节
// "你好" Java 9: byte[4] + coder=1 = 5 字节（UTF16 模式，因为中文不在 Latin-1 中）
```

**内存节省效果**：对纯 Latin-1 字符串，节省约 50% 的内存。JDK 自身大量使用字符串，Compact Strings 还显著减少了 JDK 的内存占用。

### 7.5 charAt 方法的变化

```java
// Java 8
public char charAt(int index) {
    if ((index < 0) || (index >= value.length)) {
        throw new StringIndexOutOfBoundsException(index);
    }
    return value[index]; // 直接返回 char
}

// Java 9+
public char charAt(int index) {
    if (isLatin1()) {
        return StringLatin1.charAt(value, index); // 1字节 → char
    }
    return StringUTF16.charAt(value, index); // 2字节 → char
}
```

> Java 9 的 `charAt` 性能略有下降（多了编码判断），但内存节省的优势远大于此。如果需要极致性能，可用 `-XX:-CompactStrings` 回退到 Java 8 行为。

## 8. String 的 + 拼接原理

### 8.1 编译器优化

```java
// 源码
String s = "a" + "b" + "c";

// 编译后（常量折叠）
String s = "abc"; // 编译期直接优化为字面量 "abc"
```

### 8.2 变量拼接 → StringBuilder

```java
// 源码
String a = "a";
String b = "b";
String c = a + b + "c";

// 编译后等价于（Java 8 及之前使用 StringBuilder.append）
String c = new StringBuilder()
    .append(a)
    .append(b)
    .append("c")
    .toString();
```

> **注意**：Java 9+ 使用 `invokedynamic` + `StringConcatFactory` 来优化字符串拼接，不再生成 `StringBuilder` 代码，但原理类似。

### 8.3 循环中的陷阱

```java
// ❌ 每次循环创建一个 StringBuilder 对象
String result = "";
for (int i = 0; i < 1000; i++) {
    result += i; // 等价于 result = new StringBuilder().append(result).append(i).toString();
}
// 创建了 1000 个 StringBuilder 对象和 1000 个 String 对象

// ✅ 手动使用 StringBuilder
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);
}
String result = sb.toString();
// 只创建 1 个 StringBuilder 对象
```

### 8.4 toString() 源码

```java
// AbstractStringBuilder.toString()
@Override
public abstract String toString();

// StringBuilder.toString()
@Override
public String toString() {
    // 创建新 String，复制有效长度的数据
    return new String(value, 0, count); // Java 8

    // Java 9+
    // return StringFactory.newStringFromBytes(value, coder, 0, count);
}
```

## 9. 面试题速查

### Q1: String 为什么是不可变的？

三层保证：类被 `final` 修饰（不可继承）、内部数组被 `final` 修饰（引用不可变）、数组私有且无修改入口。好处：线程安全、哈希缓存、常量池复用、安全性。

### Q2: new String("abc") 创建了几个对象？

1 个或 2 个。如果 "abc" 不在常量池中，创建 2 个（常量池 1 个 + 堆 1 个）；如果已在常量池中，只创建 1 个堆对象。

### Q3: String s = "a" + "b" 编译后是什么？

编译器会进行常量折叠，直接优化为 `String s = "ab"`。

### Q4: String、StringBuilder、StringBuffer 的区别？

`String` 不可变；`StringBuilder` 可变、非线程安全、性能高；`StringBuffer` 可变、线程安全（synchronized）、性能较低。单线程用 StringBuilder，多线程用 StringBuffer（但实际很少需要）。

### Q5: intern() 方法的作用？

返回字符串常量池中的规范表示。如果常量池中已有等值字符串，返回常量池中的引用；如果没有，JDK 6 复制一份到常量池，JDK 7+ 存储堆中对象的引用。

### Q6: JDK 6 和 JDK 7+ 的 intern() 有什么区别？

JDK 6 常量池在永久代，`intern()` 会复制字符串到永久代。JDK 7+ 常量池在堆中，`intern()` 可以直接存储堆中对象的引用，避免复制。

### Q7: String 的 hashCode 为什么用 31？

31 是奇质数，分布均匀；`31 * i == (i << 5) - i`，可被 JVM 优化为位移和减法，性能高。

### Q8: Java 9 的 Compact Strings 是什么？

将 String 内部存储从 `char[]`（每字符 2 字节）改为 `byte[] + coder`。纯 Latin-1 字符串使用 1 字节/字符，节省约 50% 内存；含非 Latin-1 字符的字符串使用 UTF-16 编码。

### Q9: String substring 在 JDK 6 和 JDK 7 有什么区别？

JDK 6 共享原 `char[]`（通过 offset 和 count），存在内存泄漏问题。JDK 7+ 复制数组到新 String，解决了内存泄漏。

### Q10: 字符串常量池在哪里？

JDK 6 在永久代，JDK 7+ 在堆中。移到堆中的主要原因是永久代大小固定容易 OOM，且 GC 管理不如堆灵活。
