## 1. Object 类概述

`Object` 类位于 `java.lang` 包下，是所有 Java 类的隐式父类。一个类如果没有显式地 `extends` 其他类，那么它默认继承 `Object`。

```java
package java.lang;

/**
 * Class {@code Object} is the root of the class hierarchy.
 * Every class has {@code Object} as a superclass. All objects,
 * including arrays, implement the methods of this class.
 */
public class Object {
    // ...
}
```

Object 类提供的方法一览：

| 方法签名 | 修饰符 | 说明 |
|---|---|---|
| `boolean equals(Object obj)` | public | 判断两个对象是否"相等" |
| `int hashCode()` | public native | 返回对象的哈希码 |
| `String toString()` | public | 返回对象的字符串表示 |
| `Class<?> getClass()` | public final native | 返回运行时类对象 |
| `Object clone()` | protected native | 创建并返回对象的副本 |
| `void wait()` | public final | 使当前线程等待 |
| `void wait(long timeout)` | public final native | 带超时的等待 |
| `void wait(long timeout, int nanos)` | public final | 带超时和纳秒的等待 |
| `void notify()` | public final native | 唤醒一个等待的线程 |
| `void notifyAll()` | public final native | 唤醒所有等待的线程 |
| `void finalize()` | protected deprecated | GC 回收前的回调钩子 |

> 注意：其中 `hashCode`、`getClass`、`clone`、`notify`、`notifyAll`、`wait(long)` 为 `native` 方法，由 JVM 用 C++ 实现。

## 2. JVM 对象内存布局：对象头

在 HotSpot JVM 中，一个 Java 对象在内存中由三部分组成：

```
┌──────────────────────────────────────┐
│            对象头 (Header)             │
│  ┌────────────────────────────────┐  │
│  │  Mark Word (64bit / 32bit)    │  │
│  ├────────────────────────────────┤  │
│  │  Klass Pointer (压缩后4字节)   │  │
│  └────────────────────────────────┘  │
│  (数组对象额外有 4 字节 array length)  │
├──────────────────────────────────────┤
│          实例数据 (Instance Data)     │
├──────────────────────────────────────┤
│          对齐填充 (Padding)           │
└──────────────────────────────────────┘
```

### 2.1 Mark Word

Mark Word 是对象头的核心，64 位 JVM 下占用 8 字节。它存储了对象运行时的关键信息，且内容会随锁状态动态变化：

```
|----------------------------------------------------------------------|----------|
|                          Mark Word (64 bits)                          |   状态   |
|---------------------------------------------------------|------------|----------|
|                无锁状态 (Normal)                          |  hashCode  |  unused  | 1 | 01  | 无锁     |
|  unused:25 | hashCode:31 | cms_free:1 | age:4 | biased:1 |  lock:2   |          |          |
|----------------------------------------------------------------------|----------|
|                偏向锁状态 (Biased)                       |  Thread ID |  epoch   | 1 | 01  | 偏向锁   |
|  ThreadID:54 | epoch:2 | cms_free:1 | age:4 | biased:1   |  lock:2   |          |          |
|----------------------------------------------------------------------|----------|
|                轻量级锁 (Lightweight Locked)                        |  00  | 轻量级锁  |
|  ptr_to_lock_record:62                                 |  lock:2   |       |          |
|----------------------------------------------------------------------|----------|
|                重量级锁 (Heavyweight Locked)                        |  10  | 重量级锁  |
|  ptr_to_heavyweight_monitor:62                         |  lock:2   |       |          |
|----------------------------------------------------------------------|----------|
|                GC 标记 (Marked for GC)                              |  11  | GC标记   |
|  (空)                                                    |  lock:2   |       |          |
|----------------------------------------------------------------------|----------|
```

**关键字段含义：**

- **hashCode**：对象的哈希码，31 位。只有在调用 `hashCode()` 且未被覆盖时才写入。
- **age**：GC 分代年龄，4 位，最大值 15（所以默认晋升老年代的年龄阈值最大为 15）。
- **biased lock**：偏向锁标志位，1 表示可偏向。
- **lock**：锁状态标记，2 位，组合含义：01（无锁/偏向锁）、00（轻量级锁）、10（重量级锁）、11（GC 标记）。
- **Thread ID**：偏向线程 ID，记录哪个线程持有了偏向锁。

### 2.2 Klass Pointer

Klass Pointer 指向方法区（元空间）中该对象的类元信息（`Klass`）。在开启指针压缩（`-XX:+UseCompressedOops`，默认开启）时占用 4 字节，否则占用 8 字节。

### 2.3 实例数据与对齐填充

- **实例数据**：存放类中定义的非静态字段值。字段排列遵循「long/double → int/float → short/char → byte/boolean → reference」的顺序，且父类字段在前。
- **对齐填充**：对象总大小必须是 8 字节的整数倍，不足时填充。

### 2.4 对象大小估算示例

```java
// 开启指针压缩 (-XX:+UseCompressedOops)
public class MyObject {
    int a;    // 4 字节
}

// 对象头: Mark Word(8) + Klass Pointer(4) = 12 字节
// 实例数据: int a = 4 字节
// 对齐填充: 0 字节
// 总计: 16 字节
```

## 3. equals() 方法源码解析

```java
public boolean equals(Object obj) {
    return (this == obj);
}
```

`Object` 的默认 `equals()` 实现就是直接比较引用地址（`==`）。也就是说，**如果不重写 `equals()`，两个对象只有指向同一块内存才被视为相等**。

### 3.1 JDK 中的标准重写示例：String

```java
// java.lang.String.equals()
public boolean equals(Object anObject) {
    if (this == anObject) {          // 1. 先比较引用，快速路径
        return true;
    }
    if (anObject instanceof String) { // 2. 类型检查
        String anotherString = (String) anObject;
        int n = value.length;
        if (n == anotherString.value.length) { // 3. 长度比较，快速失败
            char[] v1 = value;
            char[] v2 = anotherString.value;
            int i = 0;
            while (n-- != 0) {        // 4. 逐字符比较
                if (v1[i] != v2[i])
                    return false;
                i++;
            }
            return true;
        }
    }
    return false;
}
```

> 注意：Java 9+ 的 `String` 内部使用 `byte[] value`，比较逻辑类似但多了 `coder` 判断。

### 3.2 正确重写 equals() 的规范

```java
public class Person {
    private String name;
    private int age;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;                          // 自反性优化
        if (o == null || getClass() != o.getClass()) return false; // 空值+类型检查
        Person person = (Person) o;
        return age == person.age &&
               Objects.equals(name, person.name);            // 字段比较
    }
}
```

**equals() 的五大契约：**

1. **自反性**：`x.equals(x)` 必须为 `true`
2. **对称性**：`x.equals(y)` 为 `true`，则 `y.equals(x)` 也必须为 `true`
3. **传递性**：`x.equals(y)` 且 `y.equals(z)`，则 `x.equals(z)` 必须为 `true`
4. **一致性**：多次调用 `x.equals(y)` 结果不变（对象未被修改）
5. **非空性**：`x.equals(null)` 必须为 `false`

## 4. hashCode() 方法源码解析

```java
@IntrinsicCandidate
public native int hashCode();
```

`hashCode()` 是一个 `native` 方法，由 JVM 实现。在 HotSpot 中，默认的哈希码生成策略有多种，由 `-XX:hashCode=` 参数控制：

| 值 | 策略 | 说明 |
|---|---|---|
| 0 | Park-Miller 伪随机数 | 随机生成 |
| 1 | 内存地址相关 | 将对象地址与某值运算 |
| 2 | 固定值 1 | 所有对象返回 1（测试用） |
| 3 | 自增序列 | 全局计数器递增 |
| 4 | 内存地址 | 直接返回内存地址 |
| **5（默认）** | Marsaglia's xor-shift | 基于线程局部状态的随机算法 |

```cpp
// HotSpot 源码: src/share/vm/runtime/synchronizer.cpp (简化)
static inline unsigned int FastHashCode(Thread *self, oop obj) {
    // 如果 Mark Word 中没有缓存 hashCode，则生成新的
    // 使用 Marsaglia's xor-shift 随机算法
    unsigned int hash = marsaglia_xor_shift(); // xor-shift 伪随机
    hash &= 0x7FFFFFFF; // 确保非负，31位
    // 写入 Mark Word
    obj->set_hash(hash);
    return hash;
}
```

**关键点：**
- `hashCode` 是**懒加载**的，只有在第一次调用时才计算并缓存到 Mark Word。
- 一旦计算，后续调用返回相同的值（除非对象被移动或作为 GC 标记等）。
- `System.identityHashCode(obj)` 获取的就是这个默认哈希码，即使类重写了 `hashCode()`。

## 5. equals() 与 == 的区别

| 维度 | `==` | `equals()` |
|---|---|---|
| 比较类型 | 基本类型比较值，引用类型比较内存地址 | 对象的"逻辑相等"（内容由类定义） |
| 可重写 | 不可（运算符） | 可重写 |
| 默认行为 | 地址比较 | `Object` 默认也是地址比较 |
| 典型场景 | `int a == b` | `str1.equals(str2)` 比较内容 |

```java
String s1 = new String("hello");
String s2 = new String("hello");

System.out.println(s1 == s2);       // false  → 不同对象，地址不同
System.out.println(s1.equals(s2));  // true   → 内容相同（String 重写了 equals）

String s3 = "hello";
String s4 = "hello";
System.out.println(s3 == s4);       // true   → 常量池同一引用
```

> **核心区别**：`==` 比的是"是不是同一个对象"，`equals()` 比的是"内容是否相等"（前提是类正确重写了 `equals()`）。

## 6. hashCode 契约

`Object.hashCode()` 的 Javadoc 明确规定了以下契约：

1. **一致性**：在 Java 应用程序执行期间，对同一对象多次调用 `hashCode` 必须返回相同的整数（前提是对象用于 `equals` 比较的信息没有被修改）。
2. **equals 一致性**：如果两个对象 `equals` 返回 `true`，那么它们的 `hashCode` **必须**相同。
3. **不等不强制**：如果两个对象 `equals` 返回 `false`，它们的 `hashCode` **不要求**不同，但不同时能提升哈希表性能。

### 6.1 为什么重写 equals 必须重写 hashCode？

```java
public class BadPerson {
    String name;

    public BadPerson(String name) { this.name = name; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof BadPerson)) return false;
        return Objects.equals(name, ((BadPerson) o).name);
    }
    // ❌ 没有重写 hashCode！
}

// 放入 HashMap：
Map<BadPerson, String> map = new HashMap<>();
map.put(new BadPerson("Alice"), "Engineer");

// 查不到！因为两个 BadPerson("Alice") 的 hashCode 不同（默认是地址相关）
System.out.println(map.get(new BadPerson("Alice"))); // null
```

### 6.2 标准 hashCode 写法

```java
@Override
public int hashCode() {
    return Objects.hash(name, age); // 使用 Objects.hash 工具方法
}
```

`Objects.hash` 内部使用了高效的散列算法：

```java
// java.util.Objects.hash()
public static int hash(Object... values) {
    return Arrays.hashCode(values);
}

// java.util.Arrays.hashCode()
public static int hashCode(Object a[]) {
    if (a == null) return 0;
    int result = 1;
    for (Object element : a)
        result = 31 * result + (element == null ? 0 : element.hashCode());
    return result;
}
```

**为什么用 31？**
- 31 是奇质数，乘法溢出时分布均匀。
- `31 * i == (i << 5) - i`，JVM 会自动优化为位移和减法，性能高。

## 7. clone() 方法与浅拷贝 vs 深拷贝

```java
@IntrinsicCandidate
protected native Object clone() throws CloneNotSupportedException;
```

`clone()` 是 `protected native` 方法，子类必须实现 `Cloneable` 接口（标记接口，无方法）才能调用，否则抛出 `CloneNotSupportedException`。

### 7.1 浅拷贝（Shallow Clone）

```java
public class Address implements Cloneable {
    String city;

    public Address(String city) { this.city = city; }

    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();
    }
}

public class Person implements Cloneable {
    String name;
    Address address; // 引用类型

    public Person(String name, Address address) {
        this.name = name;
        this.address = address;
    }

    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone(); // 浅拷贝：只复制引用，不复制引用指向的对象
    }
}

// 测试
Person p1 = new Person("Alice", new Address("Beijing"));
Person p2 = (Person) p1.clone();

System.out.println(p1.address == p2.address); // true → 浅拷贝，共享同一个 Address 对象
p2.address.city = "Shanghai";
System.out.println(p1.address.city); // "Shanghai" → 被修改了！
```

**浅拷贝的问题**：基本类型和不可变对象（如 String）没问题，但可变引用类型会被共享，导致数据不一致。

### 7.2 深拷贝（Deep Clone）

```java
@Override
protected Object clone() throws CloneNotSupportedException {
    Person cloned = (Person) super.clone();   // 先浅拷贝
    cloned.address = (Address) address.clone(); // 再深拷贝引用类型
    return cloned;
}

// 测试
Person p1 = new Person("Alice", new Address("Beijing"));
Person p2 = (Person) p1.clone();

System.out.println(p1.address == p2.address); // false → 深拷贝，独立对象
p2.address.city = "Shanghai";
System.out.println(p1.address.city); // "Beijing" → 不受影响
```

### 7.3 浅拷贝 vs 深拷贝对比

| 维度 | 浅拷贝 | 深拷贝 |
|---|---|---|
| 基本类型 | 复制值 | 复制值 |
| 引用类型 | 复制引用（共享） | 递归复制对象（独立） |
| 实现难度 | 简单（`super.clone()`） | 复杂（需逐层 clone 或序列化） |
| 修改影响 | 互相影响 | 互不影响 |
| 性能 | 快 | 慢 |

> **最佳实践**：`clone()` 机制设计上有诸多缺陷（不走构造函数、破坏 final 语义等），Effective Java 建议**优先使用拷贝构造器或拷贝工厂**，或者用序列化实现深拷贝。

## 8. toString() 方法源码解析

```java
public String toString() {
    return getClass().getName() + "@" + Integer.toHexString(hashCode());
}
```

默认实现返回 `类名@十六进制哈希码`，例如 `com.example.Person@1b6d3586`。

### 8.1 为什么要重写 toString()

```java
// 不重写 toString()
Person p = new Person("Alice", 30);
System.out.println(p); // com.example.Person@1b6d3586  ← 无意义

// 重写 toString()
@Override
public String toString() {
    return "Person{name='" + name + "', age=" + age + "}";
}
System.out.println(p); // Person{name='Alice', age=30}  ← 可读
```

> 建议所有类都重写 `toString()`，方便调试和日志输出。IDE 可以一键生成。

## 9. wait() / notify() / notifyAll() 源码解析

这三个方法构成了 Java 线程间通信的基础机制，必须配合 `synchronized` 使用。

```java
public final void wait() throws InterruptedException {
    wait(0L); // 委托给 wait(long)
}

public final void wait(long timeout, int nanos) throws InterruptedException {
    if (timeout < 0)                    throw new IllegalArgumentException("timeout value is negative");
    if (nanos < 0 || nanos > 999999)    throw new IllegalArgumentException("nanosecond timeout value out of range");
    if (nanos > 0 && timeout < Long.MAX_VALUE) timeout++;
    wait(timeout); // 最终都调用 native wait(long)
}

@IntrinsicCandidate
public final native void wait(long timeout) throws InterruptedException;

@IntrinsicCandidate
public final native void notify();

@IntrinsicCandidate
public final native void notifyAll();
```

### 9.1 使用模式

```java
class BoundedBuffer<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;

    public BoundedBuffer(int capacity) {
        this.capacity = capacity;
    }

    public synchronized void put(T item) throws InterruptedException {
        while (queue.size() == capacity) {
            wait(); // 释放锁，进入 WAITING 状态
        }
        queue.offer(item);
        notifyAll(); // 唤醒等待的消费者
    }

    public synchronized T take() throws InterruptedException {
        while (queue.isEmpty()) {
            wait(); // 释放锁，进入 WAITING 状态
        }
        T item = queue.poll();
        notifyAll(); // 唤醒等待的生产者
        return item;
    }
}
```

### 9.2 为什么必须用 while 而不是 if？

```java
// ❌ 错误写法：if
synchronized (obj) {
    if (condition == false) {
        wait(); // 被唤醒后直接继续执行，不再检查条件
    }
    // 此处条件可能已被其他线程改变！
}

// ✅ 正确写法：while
synchronized (obj) {
    while (condition == false) {
        wait(); // 被唤醒后重新检查条件
    }
}
```

**原因**：存在**虚假唤醒（Spurious Wakeup）**——线程可能在没有 `notify` 的情况下被唤醒。`wait` 的 Javadoc 明确要求使用 `while` 循环。

### 9.3 wait/notify 的底层原理

```
线程调用 obj.wait()
    → 释放 obj 的 monitor 锁
    → 线程进入 obj 的 WaitSet（等待队列）
    → 线程状态变为 WAITING / TIMED_WAITING

线程调用 obj.notify()
    → 从 obj 的 WaitSet 中取出一个线程
    → 将其移入 obj 的 EntryList（同步队列）
    → 线程状态变为 BLOCKED，等待重新获取锁

线程调用 obj.notifyAll()
    → 将 obj 的 WaitSet 中所有线程移入 EntryList
```

### 9.4 为什么必须在 synchronized 块中调用？

`wait/notify` 依赖于对象的 **Monitor**（管程）。`synchronized` 会获取对象的 Monitor，而 `wait` 需要释放 Monitor，`notify` 需要操作 Monitor 的等待队列。如果不在 `synchronized` 块中调用，会抛出 `IllegalMonitorStateException`。

## 10. finalize() 方法源码解析

```java
@Deprecated(since="9")
protected void finalize() throws Throwable { }
```

`finalize()` 在 JDK 9 被标记为 `@Deprecated`，在 JDK 18+ 中被标记为 `@Deprecated(forRemoval=true)`，未来将被移除。

### 10.1 finalize 的执行流程

```
对象不可达 → GC 发现该对象重写了 finalize()
    → 将对象放入 Finalizer 队列
    → Finalizer 线程（低优先级守护线程）执行 finalize()
    → finalize() 执行完后，对象再次可达性分析
    → 如果仍不可达 → 回收
    → 如果 finalize() 中让对象重新可达 → "自救"成功
```

### 10.2 finalize 的问题

1. **执行时间不确定**：由 Finalizer 线程决定，可能永远不执行。
2. **只会被调用一次**：如果自救后再次不可达，不会再调用 finalize()。
3. **性能开销大**：有 finalize 的对象需要额外入队处理。
4. **安全风险**：可能被用于对象复活攻击。
5. **替代方案**：使用 `try-with-resources` + `AutoCloseable`，或 `Cleaner`（JDK 9+）。

```java
// ❌ 不推荐
@Override
protected void finalize() throws Throwable {
    try {
        // 清理资源
        close();
    } finally {
        super.finalize();
    }
}

// ✅ 推荐：AutoCloseable
public class MyResource implements AutoCloseable {
    @Override
    public void close() {
        // 清理资源
    }
}

// 使用
try (MyResource res = new MyResource()) {
    // 使用资源
}
```

## 11. getClass() 方法源码解析

```java
@IntrinsicCandidate
public final native Class<?> getClass();
```

`getClass()` 返回对象的运行时 `Class` 对象，是 `final` 方法，不可重写。

```java
class Animal {}
class Dog extends Animal {}

Animal animal = new Dog();
System.out.println(animal.getClass().getName()); // com.example.Dog（运行时类型，不是声明类型）
System.out.println(animal.getClass() == Dog.class); // true
```

> `getClass()` 返回的是**运行时类型**，而不是编译时声明类型。这是它与 `instanceof` 的关键区别之一——`instanceof` 会考虑继承层次，而 `getClass()` 只返回确切的类。

## 12. 面试题速查

### Q1: Object 类有哪些方法？

equals、hashCode、toString、getClass、clone、wait、wait(long)、wait(long, int)、notify、notifyAll、finalize。

### Q2: equals 和 == 的区别？

`==` 对基本类型比较值，对引用类型比较内存地址。`equals` 默认也是比较地址（`Object` 的实现），但通常会被重写为内容比较。`String`、`Integer` 等都重写了 `equals`。

### Q3: 为什么重写 equals 时必须重写 hashCode？

因为 `hashCode` 契约规定：两个 `equals` 为 `true` 的对象，`hashCode` 必须相同。如果不重写 `hashCode`，在 `HashMap`、`HashSet` 等基于哈希的容器中会导致逻辑相等的对象被当作不同的 key。

### Q4: hashCode 的默认实现是什么？

HotSpot JVM 默认使用 Marsaglia's xor-shift 随机算法生成，结果缓存到对象头的 Mark Word 中。与内存地址没有直接关系（虽然 JDK 早期版本曾经是）。

### Q5: clone 是浅拷贝还是深拷贝？

`Object.clone()` 默认是**浅拷贝**——只复制基本类型和引用的值（地址），不递归复制引用指向的对象。深拷贝需要手动在 `clone()` 方法中递归调用引用对象的 `clone()`。

### Q6: 为什么 wait/notify 定义在 Object 而不是 Thread？

因为 Java 的锁是**对象级**的（基于对象的 Monitor）。任意对象都可以作为锁，所以等待和通知机制也定义在 `Object` 上。这样设计使得任何对象都能作为同步监视器。

### Q7: wait() 和 sleep() 的区别？

| 维度 | wait() | sleep() |
|---|---|---|
| 所属类 | Object | Thread |
| 是否释放锁 | 是 | 否 |
| 使用位置 | synchronized 块内 | 任意 |
| 唤醒方式 | notify/notifyAll/超时 | 超时/interrupt |
| 用途 | 线程间通信 | 暂停当前线程 |

### Q8: Object 对象有多大？

64 位 JVM、开启指针压缩时，一个空 `Object` 实例占 16 字节：Mark Word(8) + Klass Pointer(4) + 实例数据(0) + 对齐填充(4)。

### Q9: finalize 方法能用来救活对象吗？

可以，finalize() 中可以让对象重新被引用（"自救"），但只会被调用一次。不推荐使用，JDK 9+ 已废弃。

### Q10: getClass() 和 instanceof 的区别？

`getClass() == X.class` 只匹配确切类型，不考虑子类。`instanceof` 匹配类型及子类。例如 `new Dog() instanceof Animal` 为 true，但 `new Dog().getClass() == Animal.class` 为 false。
