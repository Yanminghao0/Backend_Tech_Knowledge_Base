## 1. Object类概述

### 1.1 Object在类层次中的位置

```
        Object
       /  |  \
      /   |   \
   String Integer Thread ...
      |   
   所有Java类都直接或间接继承Object
```

### 1.2 Object源码全貌

```java
package java.lang;

/**
 * Object是所有类的根类。每个类都以Object作为超类。
 * 所有对象（包括数组）都实现了这个类的方法。
 */
public class Object {

    private static native void registerNatives();
    static {
        registerNatives();
    }

    // ========== 5个native方法 ==========
    public final native Class<?> getClass();
    public native int hashCode();
    protected native Object clone() throws CloneNotSupportedException;
    public final native void notify();
    public final native void notifyAll();
    public final native void wait(long timeout) throws InterruptedException;

    // ========== 6个非native方法 ==========
    public boolean equals(Object obj) { return (this == obj); }
    public String toString() { return getClass().getName() + "@" + Integer.toHexString(hashCode()); }
    public final void wait(long timeout, int nanos) throws InterruptedException { ... }
    public final void wait() throws InterruptedException { wait(0); }
    protected void finalize() throws Throwable { }
    // private 构造方法（由JVM创建）
}
```

方法分类总结：

| 方法 | 类型 | 可重写 | 描述 |
|------|------|--------|------|
| `getClass()` | native, final | ❌ | 返回运行时类对象 |
| `hashCode()` | native | ✅ | 返回哈希码 |
| `equals(Object)` | 普通 | ✅ | 判断相等 |
| `clone()` | native, protected | ✅ | 克隆对象 |
| `toString()` | 普通 | ✅ | 字符串表示 |
| `notify()` | native, final | ❌ | 唤醒单个等待线程 |
| `notifyAll()` | native, final | ❌ | 唤醒所有等待线程 |
| `wait(long)` | native, final | ❌ | 等待指定毫秒 |
| `wait(long, int)` | final | ❌ | 等待指定毫秒+纳秒 |
| `wait()` | final | ❌ | 无限等待 |
| `finalize()` | 普通, 已废弃 | ✅ | GC回收前调用 |

## 2. registerNatives与构造方法

### 2.1 registerNatives

```java
private static native void registerNatives();
static {
    registerNatives();
}
```

`registerNatives`是一个native方法，在类加载时（静态代码块）被调用。它的作用是**将Java中的native方法与JVM中的C/C++实现绑定**。

为什么不用自动绑定而要手动注册？因为手动注册可以让方法名更灵活——JNI自动绑定要求方法名遵循`Java_package_Class_method`格式，而手动注册可以任意命名C函数。

### 2.2 Object没有显式构造方法

Object类没有显式定义构造方法，编译器会提供一个默认的无参构造方法：

```java
// 编译器隐式添加
public Object() {}
```

实际上Object实例由JVM直接创建，构造方法内部是空的——Object的字段（对象头）由JVM在内存分配时设置。

### 2.3 JNI函数注册

在OpenJDK的C源码中（`Object.c`），可以看到注册映射：

```c
// openjdk/src/java.base/share/native/libjava/Object.c

static JNINativeTable methods[] = {
    {"hashCode",     "()I",                    (void *)&JVM_IHashCode},
    {"wait",         "(J)V",                   (void *)&JVM_MonitorWait},
    {"notify",       "()V",                    (void *)&JVM_MonitorNotify},
    {"notifyAll",    "()V",                    (void *)&JVM_MonitorNotifyAll},
    {"clone",        "()Ljava/lang/Object;",   (void *)&JVM_Clone},
    {"getClass",     "()Ljava/lang/Class;",    (void *)&JVM_GetClass},
};

JNIEXPORT void JNICALL
Java_java_lang_Object_registerNatives(JNIEnv *env, jclass cls) {
    (*env)->RegisterNatives(env, cls, methods, sizeof(methods)/sizeof(methods[0]));
}
```

## 3. getClass()方法

### 3.1 源码

```java
/**
 * 返回此对象的运行时类。返回的Class对象是被static synchronized方法
 * 锁定的对象。
 */
public final native Class<?> getClass();
```

`getClass()`是final的，不能重写。它返回对象运行时的实际类型，而不是声明类型。

### 3.2 运行时类型 vs 编译时类型

```java
// 编译时类型 vs 运行时类型
Object obj = new String("hello");

// 编译时类型：Object
// 运行时类型：String

Class<?> clazz = obj.getClass();
System.out.println(clazz.getName()); // java.lang.String，不是java.lang.Object
```

### 3.3 getClass()的应用

```java
// 1. 类型判断（比instanceof更精确）
public void process(Object obj) {
    // instanceof会匹配子类
    if (obj instanceof Number) { /* 匹配Integer, Double等 */ }

    // getClass()精确匹配
    if (obj.getClass() == Integer.class) { /* 只匹配Integer */ }
}

// 2. 反射创建实例
public <T> T createInstance(T template) throws Exception {
    Class<?> clazz = template.getClass();
    return (T) clazz.getDeclaredConstructor().newInstance();
}

// 3. 获取类信息
public void inspect(Object obj) {
    Class<?> clazz = obj.getClass();
    System.out.println("类名: " + clazz.getSimpleName());
    System.out.println("包名: " + clazz.getPackage().getName());
    System.out.println("是否数组: " + clazz.isArray());
    System.out.println("是否接口: " + clazz.isInterface());
}
```

### 3.4 getClass()与synchronized

```java
//getClass()返回的Class对象常用于static synchronized的锁
public class Counter {
    private static int count = 0;

    // 等价于 synchronized (Counter.class)
    public static synchronized void increment() {
        count++;
    }

    // 手动使用Class对象作为锁
    public static void increment2() {
        Class<?> clazz = new Counter().getClass(); // 返回Counter.class
        synchronized (clazz) {
            count++;
        }
    }
}
```

## 4. hashCode()方法

### 4.1 源码

```java
/**
 * 返回此对象的哈希码。此方法支持HashMap等哈希表。
 *
 * hashCode的一般约定：
 * 1. 在Java应用执行期间，对同一对象多次调用hashCode必须返回相同的整数
 *    （前提是equals比较的信息没有被修改）
 * 2. 如果两个对象equals返回true，则它们的hashCode必须相同
 * 3. 如果两个对象equals返回false，不要求hashCode不同，但不同的话能提高哈希表性能
 */
public native int hashCode();
```

### 4.2 JVM中的hashCode实现

在HotSpot JVM中，hashCode的生成有6种策略，通过`-XX:hashCode=N`参数控制：

| 值 | 策略 | 描述 |
|----|------|------|
| 0 | Park-Miller RNG | 随机数生成器 |
| 1 | Object address | 对象内存地址（经过处理） |
| 2 | Always 1 | 所有对象都返回1（测试用） |
| 3 | Incrementing | 自增序列号 |
| 4 | Object address | 原始内存地址 |
| 5（默认） | Marsaglia XOR-Shift | 基于线程状态的XOR-Shift随机算法 |

默认策略（5）的实现（OpenJDK C++伪代码）：

```cpp
// openjdk/src/hotspot/share/runtime/synchronizer.cpp

static inline uintptr_t get_next_hash(Thread* current, oop obj) {
    uintptr_t value = 0;
    if (hashCode == 0) {
        // Park-Miller RNG
        value = os::random();
    } else if (hashCode == 1) {
        // 对象地址 + 一些运算
        value = cast_from_oop<uintptr_t>(obj) >> 3;
    } else if (hashCode == 5) {
        // Marsaglia's xor-shift scheme with thread-specific state
        unsigned t = current->_hashStateX;
        t ^= (t << 11);
        current->_hashStateX = current->_hashStateY;
        current->_hashStateY = current->_hashStateZ;
        current->_hashStateZ = current->_hashStateW;
        unsigned v = current->_hashStateW;
        v = (v ^ (v >> 19)) ^ (t ^ (t >> 8));
        current->_hashStateW = v;
        value = v;
    }
    // ...
    value &= markWord::hash_mask;  // 截取到位
    return (value == 0) ? 0xBAD : value;  // 避免返回0
}
```

### 4.3 hashCode的缓存

在HotSpot中，hashCode只在第一次调用时计算，之后缓存在对象头的Mark Word中：

```
对象头（64位）的Mark Word结构：
|----------------------------------------------|------------------|
| 未使用 (25bit) | 未使用 (31bit) | hash (31bit) | 状态 (1bit) | 锁标志 (2bit) |
|----------------------------------------------|------------------|

一旦计算了hashCode，它会存储在Mark Word中
后续调用直接返回缓存的值，不会重新计算
```

### 4.4 自定义hashCode的推荐写法

```java
public class Person {
    private String name;
    private int age;
    private String email;

    // Objects.hashCode方式（推荐）
    @Override
    public int hashCode() {
        return Objects.hash(name, age, email);
    }

    // 手动写法（性能更好）
    @Override
    public int hashCode() {
        int result = 17;
        result = 31 * result + (name != null ? name.hashCode() : 0);
        result = 31 * result + age;
        result = 31 * result + (email != null ? email.hashCode() : 0);
        return result;
    }
}
```

为什么用31作为乘数？
- 31是奇素数，减少冲突
- `31 * i == (i << 5) - i`，JVM可以优化为位运算
- 不是太大也不是太小，避免溢出和分布不均

## 5. equals()方法与hashCode契约

### 5.1 Object的默认实现

```java
public boolean equals(Object obj) {
    return (this == obj);  // 默认比较引用地址
}
```

Object的默认实现就是`==`——比较两个引用是否指向同一个对象。也就是说，如果不重写equals，两个不同对象永远不相等，即使内容完全一样。

### 5.2 equals的通用约定

```
1. 自反性（Reflexive）：x.equals(x) == true
2. 对称性（Symmetric）：x.equals(y) == y.equals(x)
3. 传递性（Transitive）：x.equals(y) && y.equals(z) → x.equals(z)
4. 一致性（Consistent）：多次调用结果不变（前提是对象未被修改）
5. 非空性：x.equals(null) == false
```

### 5.3 正确重写equals的模板

```java
public class Person {
    private String name;
    private int age;

    @Override
    public boolean equals(Object o) {
        // 1. 检查是否同一引用
        if (this == o) return true;

        // 2. 检查null和类型
        if (o == null || getClass() != o.getClass()) return false;

        // 3. 类型转换
        Person person = (Person) o;

        // 4. 逐个比较关键字段
        if (age != person.age) return false;
        return name != null ? name.equals(person.name) : person.name == null;
    }

    @Override
    public int hashCode() {
        int result = name != null ? name.hashCode() : 0;
        result = 31 * result + age;
        return result;
    }
}
```

### 5.4 常见错误

```java
// 错误1：参数类型不是Object——这不是重写，是重载
public boolean equals(Person other) {  // ← 不是override！
    return this.name.equals(other.name);
}

// 错误2：破坏对称性
class Parent {
    String name;
    public boolean equals(Object o) {
        if (o instanceof Parent) {
            return this.name.equals(((Parent)o).name);
        }
        return false;
    }
}
class Child extends Parent {
    int age;
    public boolean equals(Object o) {
        if (o instanceof Child) {
            return super.equals(o) && this.age == ((Child)o).age;
        }
        return false; // Parent.equals(Child)可能为true，Child.equals(Parent)为false → 破坏对称性
    }
}

// 错误3：重写了equals但没有重写hashCode
class BadPerson {
    String name;
    public boolean equals(Object o) {
        return o instanceof BadPerson && ((BadPerson)o).name.equals(this.name);
    }
    // 没有重写hashCode！
    // BadPerson p1 = new BadPerson(); p1.name = "Alice";
    // BadPerson p2 = new BadPerson(); p2.name = "Alice";
    // p1.equals(p2) == true，但 p1.hashCode() != p2.hashCode()
    // 在HashMap中会被分到不同的桶！
}
```

### 5.5 hashCode契约详解

```
equals返回true的两个对象，hashCode必须相同
equals返回false的两个对象，hashCode可以相同（但不同更好）

HashMap的工作原理依赖于这个契约：
1. 先用hashCode定位桶（bucket）
2. 桶内用equals判断是否相等

如果equals相同但hashCode不同 → 两个"相等"的对象被分到不同的桶 → 查找时找不到！
```

```java
// 验证hashCode契约
Map<BadPerson, String> map = new HashMap<>();
BadPerson p1 = new BadPerson();
p1.name = "Alice";
BadPerson p2 = new BadPerson();
p2.name = "Alice";

map.put(p1, "value1");
System.out.println(map.get(p2)); // null！因为p2的hashCode与p1不同，找不到对应的桶
```

## 6. clone()方法与深浅拷贝

### 6.1 源码

```java
/**
 * 创建并返回此对象的副本。"副本"的确切含义可能取决于对象的类。
 * 通常约定：x.clone() != x 为true，x.clone().getClass() == x.getClass() 为true
 *
 * 按照惯例，返回的对象应该通过super.clone()获取。
 * 如果一个类及其所有超类（除Object外）都遵守此约定，
 * 则 x.clone().getClass() == x.getClass()
 */
protected native Object clone() throws CloneNotSupportedException;
```

### 6.2 Cloneable接口

`clone()`是protected的，只有实现了`Cloneable`接口的类才能调用。`Cloneable`是一个标记接口（没有方法），它只是告诉JVM"这个类允许被clone"：

```java
public interface Cloneable {
    // 空接口——标记接口
}
```

如果调用`clone()`时没有实现`Cloneable`，会抛出`CloneNotSupportedException`。

### 6.3 浅拷贝（Shallow Clone）

```java
public class Address {
    String city;
    public Address(String city) { this.city = city; }
}

public class Person implements Cloneable {
    String name;
    Address address;  // 引用类型

    public Person(String name, Address address) {
        this.name = name;
        this.address = address;
    }

    @Override
    protected Person clone() throws CloneNotSupportedException {
        return (Person) super.clone(); // 浅拷贝
    }
}

// 浅拷贝的问题
Person p1 = new Person("Alice", new Address("Beijing"));
Person p2 = p1.clone();

p2.name = "Bob";
p2.address.city = "Shanghai";

System.out.println(p1.name);         // Alice（基本类型不受影响）
System.out.println(p1.address.city); // Shanghai！被修改了！
// p1.address == p2.address → true（共享同一个Address对象）
```

### 6.4 深拷贝（Deep Clone）

```java
// 方式1：手动深拷贝
@Override
protected Person clone() throws CloneNotSupportedException {
    Person cloned = (Person) super.clone();
    cloned.address = new Address(this.address.city); // 手动拷贝引用类型
    return cloned;
}

// 方式2：Address也实现Cloneable
public class Address implements Cloneable {
    String city;

    @Override
    protected Address clone() throws CloneNotSupportedException {
        return (Address) super.clone();
    }
}

@Override
protected Person clone() throws CloneNotSupportedException {
    Person cloned = (Person) super.clone();
    cloned.address = this.address.clone();
    return cloned;
}

// 方式3：通过序列化实现深拷贝（最通用）
@SuppressWarnings("unchecked")
public Person deepCopy() {
    try {
        ByteArrayOutputStream bos = new ByteArrayOutputStream();
        ObjectOutputStream oos = new ObjectOutputStream(bos);
        oos.writeObject(this);
        oos.flush();

        ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
        ObjectInputStream ois = new ObjectInputStream(bis);
        return (Person) ois.readObject();
    } catch (Exception e) {
        throw new RuntimeException("深拷贝失败", e);
    }
}
```

### 6.5 clone()的问题与替代方案

```
clone()的问题：
1. 浅拷贝容易导致共享引用的bug
2. 不调用构造方法，可能破坏不变性
3. Cloneable是标记接口，没有类型约束
4. 数组的clone()是浅拷贝，容易踩坑

推荐替代方案：
1.拷贝构造方法：Person(Person other) { ... }
2. 拷贝工厂：static Person copyOf(Person other) { ... }
3. Builder模式 + toBuilder()
4. 序列化方式：Jackson/Gson序列化再反序列化
```

// Integer.toString

// AbstractCollection（List、Set的父类）
public String toString() {
    Iterator<E> it = iterator();
    if (! it.hasNext())
        return "[]";

    StringBuilder sb = new StringBuilder();
    sb.append('[');
    for (;;) {

## 8. 线程相关方法：wait/notify/notifyAll

### 8.1 源码

```java
/**
 * 导致当前线程等待，直到另一个线程调用此对象的notify()或notifyAll()。
 * 当前线程必须拥有此对象的监视器锁。
 */
public final void wait() throws InterruptedException {
    wait(0);  // 0表示无限等待
}

public final native void wait(long timeoutMillis) throws InterruptedException;

public final void wait(long timeout, int nanos) throws InterruptedException {
    if (timeout < 0) {
        throw new IllegalArgumentException("timeout value is negative");
    }
    if (nanos < 0 || nanos > 999999) {
        throw new IllegalArgumentException("nanosecond timeout value out of range");
    }
    if (nanos > 0) {
        timeout++;  // 纳秒精度实际上被忽略了，只是+1ms
    }
    wait(timeout);
}

/**
 * 唤醒正在此对象监视器上等待的单个线程。
 * 如果有多个线程在等待，选择任意一个。
 */
public final native void notify();

/**
 * 唤醒正在此对象监视器上等待的所有线程。
 */
public final native void notifyAll();
```

### 8.2 wait/notify的使用模式

```java
// 生产者-消费者模式
public class BoundedBuffer<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;

    public BoundedBuffer(int capacity) {
        this.capacity = capacity;
    }

    // 生产者
    public synchronized void put(T item) throws InterruptedException {
        // 必须用while，不能用if（防止虚假唤醒）
        while (queue.size() == capacity) {
            wait(); // 释放锁，进入等待
        }
        queue.offer(item);
        notifyAll(); // 唤醒消费者
    }

    // 消费者
    public synchronized T take() throws InterruptedException {
        while (queue.isEmpty()) {
            wait(); // 释放锁，进入等待
        }
        T item = queue.poll();
        notifyAll(); // 唤醒生产者
        return item;
    }
}
```

### 8.3 为什么wait必须在synchronized块中

```java
// wait/notify的正确使用前提：当前线程必须持有对象的监视器锁
// 否则抛出 IllegalMonitorStateException

synchronized (obj) {
    while (!condition) {
        obj.wait();    // 合法：当前线程持有obj的锁
    }
    // ...
    obj.notify();     // 合法
}

// obj.wait(); // 非法！没有持有obj的锁 → IllegalMonitorStateException
```

底层原理：
1. `wait()`会先**释放对象的监视器锁**，然后将线程加入该对象的WaitSet
2. `notify()`会从WaitSet中移出一个线程，放入EntryList
3. 被唤醒的线程需要重新竞争锁

### 8.4 wait(long timeout)的超时机制

```java
synchronized (lock) {
    long start = System.currentTimeMillis();
    long timeout = 5000; // 5秒超时

    while (!condition) {
        long elapsed = System.currentTimeMillis() - start;
        long remaining = timeout - elapsed;
        if (remaining <= 0) {
            break; // 超时退出
        }
        lock.wait(remaining);
    }
    // condition满足或超时
}
```

### 8.5 虚假唤醒（Spurious Wakeup）

```java
// 错误写法——用if
synchronized (obj) {
    if (!condition) {
        obj.wait();
    }
    // condition不一定为true！可能是虚假唤醒
}

// 正确写法——用while
synchronized (obj) {
    while (!condition) {  // 每次醒来都重新检查
        obj.wait();
    }
    // condition一定为true
}
```

虚假唤醒是指线程在没有被notify/notifyAll/interrupt的情况下从wait中返回。虽然概率很低，但规范要求使用while循环来防止这种情况。

## 9. finalize()方法（已废弃）

### 9.1 源码

```java
/**
 * 当GC确定不存在对该对象的更多引用时，由GC调用此方法。
 * 子类可以重写此方法来释放系统资源或执行其他清理。
 *
 * 已废弃（Java 9+），推荐使用Cleaner或try-with-resources替代。
 */
@Deprecated(since="9")
protected void finalize() throws Throwable { }
```

### 9.2 finalize的问题

```
1. 不保证执行：JVM退出时可能不执行finalize
2. 执行时间不确定：GC决定何时调用，可能永远不调用
3. 性能开销：finalize会严重影响GC性能
4. 可能"复活"对象：在finalize中重新建立引用
5. 异常被忽略：finalize中抛出的异常会被忽略
6. 安全风险：可能被攻击者利用
```

### 9.3 替代方案

```java
// 方案1：try-with-resources（推荐）
public class Resource implements AutoCloseable {
    @Override
    public void close() {
        // 释放资源
    }
}

try (Resource res = new Resource()) {
    // 使用资源
} // 自动调用res.close()

// 方案2：Cleaner（Java 9+）
public class NativeResource implements AutoCloseable {
    private static final Cleaner cleaner = Cleaner.create();

    private static class State implements Runnable {
        private long nativeHandle;

        State(long handle) {
            this.nativeHandle = handle;
        }

        @Override
        public void run() {
            // 清理逻辑——由Cleaner在GC时调用
            if (nativeHandle != 0) {
                freeNative(nativeHandle);
                nativeHandle = 0;
            }
        }
    }

    private final State state;
    private final Cleaner.Cleanable cleanable;

    public NativeResource(long handle) {
        this.state = new State(handle);
        this.cleanable = cleaner.register(this, state);
    }

    @Override
    public void close() {
        cleanable.clean(); // 显式清理
    }
}
```

## 10. 对象头与Mark Word

### 10.1 Java对象内存布局

```
一个Java对象在内存中由三部分组成：

┌───────────────────────────────────┐
│         对象头 (Object Header)     │  ← Mark Word + Class Pointer
├───────────────────────────────────┤
│         实例数据 (Instance Data)   │  ← 字段值
├───────────────────────────────────┤
│         对齐填充 (Padding)         │  ← 8字节对齐
└───────────────────────────────────┘
```

### 10.2 Mark Word结构（64位JVM）

Mark Word是对象头的核心，存储了hashCode、GC年龄、锁状态等信息。64位JVM中Mark Word为64位：

```
|----------------------------------------------------------------------|-------------|
|                          Mark Word (64 bits)                           |   状态位    |
|----------------------------------------------------------------------|-------------|
| unused:25 | hashCode:31 | cms_free:1 | age:4 | biased_lock:1 | lock:2 |   无锁       |
|----------------------------------------------------------------------|-------------|
| thread:54 | epoch:2     | cms_free:1 | age:4 | biased_lock:1 | lock:2 |   偏向锁     |
|----------------------------------------------------------------------|-------------|
|                       ptr_to_lock_record:62                | lock:2 |   轻量级锁   |
|----------------------------------------------------------------------|-------------|
|                   ptr_to_heavyweight_monitor:62             | lock:2 |   重量级锁   |
|----------------------------------------------------------------------|-------------|
|                                                          | lock:2 |   GC标记     |
|----------------------------------------------------------------------|-------------|
```

### 10.3 锁状态与Mark Word变化

```java
// 无锁状态：存储hashCode
Object obj = new Object();
obj.hashCode(); // Mark Word记录hashCode

// 偏向锁状态：存储线程ID
synchronized (obj) {
    // 第一次进入：Mark Word记录当前线程ID
    // 后续同一线程进入：只需比较线程ID，无需CAS
}

// 轻量级锁状态：指向栈中Lock Record
// 当有第二个线程竞争时，偏向锁撤销，升级为轻量级锁
// Mark Word指向线程栈帧中的Lock Record

// 重量级锁状态：指向Monitor对象
// 当竞争激烈（自旋失败）时，升级为重量级锁
// Mark Word指向ObjectMonitor
```

### 10.4 hashCode与锁的互斥

```java
// 重要：调用hashCode()会禁用偏向锁！

Object obj = new Object();
obj.hashCode(); // Mark Word记录了hashCode
// 此时偏向锁位被覆盖，无法进入偏向锁状态
// 直接从无锁→轻量级锁→重量级锁

// 这也是为什么不建议在锁对象上调用hashCode()
```

### 10.5 验证Mark Word

```java
import org.openjdk.jol.info.ClassLayout;

public class MarkWordDemo {
    public static void main(String[] args) {
        Object obj = new Object();

        // 打印对象内存布局
        System.out.println("无锁状态：");
        System.out.println(ClassLayout.parseInstance(obj).toPrintable());

        synchronized (obj) {
            System.out.println("加锁状态：");
            System.out.println(ClassLayout.parseInstance(obj).toPrintable());
        }

        System.out.println("释放锁后：");
        System.out.println(ClassLayout.parseInstance(obj).toPrintable());
    }
}

// 输出可以看到Mark Word中锁标志位的变化
// 000 = 无锁/偏向锁
// 001 = 偏向锁
// 010 = 轻量级锁
// 011 = 重量级锁
```

## 11. 面试题速查

### Q1: Object类有哪些方法？哪些是native的？

Object有11个方法。5个native方法：getClass、hashCode、clone、notify、notifyAll、wait(long)。其余6个是非native：equals、toString、wait()、wait(long,int)、finalize、registerNatives（private static native）。

### Q2: equals和hashCode的契约是什么？

1. equals返回true的两个对象，hashCode必须相同。2. equals返回false的两个对象，hashCode最好不同（但不强制）。3. 同一对象多次调用hashCode返回值必须一致。如果违反第一条，HashMap/HashSet会出问题——两个"相等"的对象被分到不同的桶。

### Q3: 为什么重写equals必须重写hashCode？

因为HashMap先用hashCode定位桶，再用equals判断相等。如果两个对象equals为true但hashCode不同，它们会被放入不同的桶，查找时找不到。这违反了hashCode契约，导致HashMap工作异常。

### Q4: hashCode()的默认实现是什么？

HotSpot JVM默认使用Marsaglia XOR-Shift算法（策略5），基于线程状态生成随机数。不是对象内存地址！可以通过`-XX:hashCode=`参数切换策略。hashCode只在第一次调用时计算，之后缓存在对象头的Mark Word中。

### Q5: wait/notify为什么必须在synchronized块中调用？

因为wait/notify操作的是对象的监视器（Monitor）。wait会释放监视器锁，notify需要操作监视器的等待队列。如果没有持有锁就调用，会抛出IllegalMonitorStateException。这是JVM层面的强制检查。

### Q6: 为什么wait要用while循环而不是if？

防止虚假唤醒（Spurious Wakeup）。线程可能在没有被notify的情况下从wait中返回（操作系统层面的行为）。用while循环可以每次醒来后重新检查条件，确保条件真正满足才继续执行。

### Q7: clone()是浅拷贝还是深拷贝？

Object.clone()是浅拷贝——基本类型直接复制值，引用类型只复制引用（共享同一个对象）。要实现深拷贝需要手动在clone方法中递归clone引用类型，或通过序列化方式实现。

### Q8: 为什么不建议使用finalize？

finalize不保证执行、执行时间不确定、性能开销大、可能导致对象"复活"、异常被忽略。Java 9已标记为@Deprecated，推荐使用try-with-resources或Cleaner替代。

### Q9: Mark Word是什么？它存储了什么信息？

Mark Word是Java对象头的一部分（64位），存储了hashCode、GC分代年龄、锁状态标志、线程ID（偏向锁）、Lock Record指针（轻量级锁）、Monitor指针（重量级锁）等。不同锁状态下Mark Word的布局不同，信息复用同一块内存。

### Q10: 调用hashCode()对锁有什么影响？

调用hashCode()会将hashCode写入Mark Word，覆盖了偏向锁的线程ID信息，导致该对象无法进入偏向锁状态。后续加锁直接从轻量级锁开始。因此不建议在锁对象上调用hashCode()。
