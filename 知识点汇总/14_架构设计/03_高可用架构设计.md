## 1. 高可用概述

### SLA定义

| SLA等级 | 可用性 | 年宕机时间 | 适用场景 |
|---------|--------|----------|---------|
| 2个9 | 99% | 3.65天 | 内部工具 |
| 3个9 | 99.9% | 8.76小时 | 一般业务 |
| 4个9 | 99.99% | 52.6分钟 | 核心业务 |
| 5个9 | 99.999% | 5.26分钟 | 金融/电信 |

### 衡量指标

```
MTBF (Mean Time Between Failures) 平均故障间隔时间
MTTR (Mean Time To Recovery) 平均恢复时间

可用性 = MTBF / (MTBF + MTTR)

提升可用性 = 增大MTBF（减少故障） + 减小MTTR（快速恢复）
```

## 2. 高可用核心原则

```
1. 冗余：多实例部署，单点故障不影响整体
2. 故障转移：自动检测故障，切换到健康节点
3. 降级：非核心功能降级，保住核心链路
4. 限流：控制流量，防止雪崩
5. 隔离：故障隔离，避免级联失败
6. 可观测：监控告警，快速发现和定位问题
```

### 冗余层次

```
┌─────────────────────────────────────┐
│           DNS / CDN                  │ ← 接入层冗余
├─────────────────────────────────────┤
│        LVS / Nginx (多台)            │ ← 负载层冗余
├─────────────────────────────────────┤
│     应用服务 (多实例 + K8s)           │ ← 应用层冗余
├─────────────────────────────────────┤
│  MySQL主从 / Redis集群 / MQ集群      │ ← 数据层冗余
├─────────────────────────────────────┤
│        多机房 / 多可用区              │ ← 机房层冗余
└─────────────────────────────────────┘
```

## 3. 多机房部署

### 同城双活

```
机房A ←10ms→ 机房B（同城，距离<50km）

特点：
  - 网络延迟低（<10ms），可做同步复制
  - 一个机房故障，另一个机房接管
  - 成本适中

实现：
  - MySQL半同步复制（同城延迟可接受）
  - Redis集群跨机房部署
  - 应用层无状态，DNS轮询
```

### 异地多活

```
北京机房 ←─30-100ms─→ 上海机房 ←─30-100ms─→ 广州机房

特点：
  - 网络延迟高（30-100ms），只能异步复制
  - 单地域故障不影响其他地域
  - 成本高，架构复杂

实现：
  - 按用户地域路由（GSLB）
  - 数据按租户/地域分片
  - 异步消息同步跨地域数据
```

### 两地三中心

```
北京主中心 ←→ 北京备中心  （同城双活）
     ↓
上海灾备中心              （异步复制）

特点：
  - 同城双活 + 异地灾备
  - 同城机房故障：切换到同城备中心
  - 城市级故障：切换到异地灾备中心
  - 金融行业标配
```

## 4. 负载均衡

### 四层 vs 七层

| 层次 | 技术 | 特点 | 适用 |
|------|------|------|------|
| 四层 | LVS | 基于IP+Port转发，性能高 | TCP/UDP流量 |
| 七层 | Nginx/Gateway | 基于HTTP头/URL路由，灵活 | HTTP/HTTPS流量 |

### 健康检查

```yaml
# Nginx健康检查
upstream backend {
    server 10.0.0.1:8080 max_fails=3 fail_timeout=30s;
    server 10.0.0.2:8080 max_fails=3 fail_timeout=30s;
    # max_fails=3: 30秒内失败3次标记为不可用
    # fail_timeout=30s: 30秒后重试
}

# K8s健康检查
readinessProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
  failureThreshold: 3
```

## 5. 故障转移

### MySQL主从切换

```java
// 使用ShardingSphere或动态数据源实现自动切换
@Configuration
public class DataSourceConfig {
    
    @Bean
    @Primary
    public DynamicDataSource dynamicDataSource(
            @Qualifier("masterDataSource") DataSource master,
            @Qualifier("slaveDataSource") DataSource slave) {
        DynamicDataSource ds = new DynamicDataSource();
        ds.setDefaultTargetDataSource(master);
        Map<Object, Object> target = new HashMap<>();
        target.put("master", master);
        target.put("slave", slave);
        ds.setTargetDataSources(target);
        return ds;
    }
}

// 故障检测 + 自动切换
@Scheduled(fixedDelay = 5000)
public void healthCheck() {
    try {
        masterDataSource.getConnection().close();
    } catch (Exception e) {
        // 主库故障，切换到从库
        dynamicDataSource.setDefaultTargetDataSource(slaveDataSource);
        alertService.sendAlert("主库故障，已切换到从库");
    }
}
```

### K8s自动故障转移

```yaml
# Pod故障 → K8s自动重启
spec:
  restartPolicy: Always
  containers:
  - name: app
    livenessProbe:
      httpGet:
        path: /actuator/health/liveness
        port: 8080
      failureThreshold: 3       # 连续3次失败重启
      periodSeconds: 10

# 节点故障 → Pod自动迁移
# K8s Controller检测到节点不可达，在健康节点重新调度Pod
```

## 6. 限流降级

### Sentinel限流

```java
@SentinelResource(value = "createOrder", 
    blockHandler = "createOrderBlockHandler",
    fallback = "createOrderFallback")
public Order createOrder(OrderRequest request) {
    // 正常下单逻辑
    return orderService.create(request);
}

// 限流后的处理
public Order createOrderBlockHandler(OrderRequest request, BlockException ex) {
    if (ex instanceof FlowException) {
        throw new BusinessException("系统繁忙，请稍后重试");
    }
    throw new BusinessException("服务不可用");
}

// 降级处理
public Order createOrderFallback(OrderRequest request, Throwable e) {
    // 降级：记录请求，异步处理
    log.warn("下单降级，异步处理: {}", request, e);
    asyncOrderService.processAsync(request);
    return Order.pending(request);
}
```

### 限流策略对比

| 策略 | 说明 | 适用 |
|------|------|------|
| 固定窗口 | 时间窗口内计数 | 简单场景 |
| 滑动窗口 | 滑动时间窗口计数 | 精度要求高 |
| 令牌桶 | 匀速生成令牌 | 允许突发流量 |
| 漏桶 | 匀速消费请求 | 严格限流 |

### 降级策略

```
读降级：
  - 缓存降级：Redis不可用时返回本地缓存或默认值
  - 搜索降级：ES不可用时返回数据库查询结果

写降级：
  - 同步转异步：将同步写转为MQ异步写
  - 合并写：多个写请求合并批量执行

功能降级：
  - 关闭推荐、评论等非核心功能
  - 简化展示（只展示核心信息）
```

## 7. 容灾设计

### 容灾等级

| 等级 | RPO（数据丢失） | RTO（恢复时间） | 方案 |
|------|----------------|----------------|------|
| 数据级 | 少量 | 数小时 | 数据备份恢复 |
| 应用级 | 零或少量 | 分钟级 | 应用级主备切换 |
| 业务级 | 零 | 秒级 | 多活自动切换 |

### 混沌工程

```
目标：主动注入故障，验证系统容错能力

Netflix Chaos Monkey理念：
  - 随机杀实例 → 验证实例级容错
  - 模拟网络延迟 → 验证超时处理
  - 模拟网络分区 → 验证分区容错
  - 模拟CPU/内存满载 → 验证资源隔离

工具：
  - Chaos Mesh (K8s原生)
  - Chaos Monkey (Netflix)
  - 阿里云AHAS
```

### 容灾演练流程

```
1. 制定演练计划（注入什么故障，预期什么结果）
2. 准备监控和回滚方案
3. 小范围注入故障（先在预发环境）
4. 观察系统行为（是否自动恢复）
5. 记录问题并改进
6. 逐步扩大范围（生产环境灰度）
```

## 8. 面试要点

### Q1: 如何实现99.99%的可用性？

```
99.99% = 年宕机 < 52.6分钟

1. 消除单点：多实例 + 多机房
2. 自动故障转移：健康检查 + 自动切换
3. 限流降级：防止雪崩
4. 快速恢复：灰度发布 + 快速回滚
5. 监控告警：MTTR最小化
6. 容灾演练：验证容错能力
```

### Q2: 同城双活和异地多活的区别？

```
同城双活：
  - 距离<50km，延迟<10ms
  - 可做同步复制
  - 应对机房级故障
  - 成本适中

异地多活：
  - 跨城市，延迟30-100ms
  - 只能异步复制
  - 应对城市级故障
  - 成本高，架构复杂
  - 需要按地域分片数据
```

### Q3: 如何防止缓存雪崩？

```
1. 缓存高可用：Redis集群 + 哨兵
2. 多级缓存：Caffeine + Redis
3. 熔断降级：Sentinel限流，缓存不可用时降级到DB
4. 随机TTL：避免大量Key同时过期
5. 预热：大促前提前加载缓存
```

### Q4: 限流算法怎么选？

```
令牌桶：允许突发流量，适合大多数场景（Sentinel默认）
漏桶：匀速处理，适合保护下游系统
滑动窗口：精度高，适合需要精确控制的场景

实际选择：
  - API网关：令牌桶（允许突发）
  - 消息消费：漏桶（匀速消费）
  - 秒杀：滑动窗口（精确控制）
```

### Q5: 什么是优雅停机？

```
1. 应用收到SIGTERM信号
2. 从注册中心注销（停止接收新请求）
3. 等待正在处理的请求完成（graceful period）
4. 关闭数据库连接、线程池等资源
5. 进程退出

Spring Boot配置：
  server.shutdown=graceful
  spring.lifecycle.timeout-per-shutdown-phase=30s
```
