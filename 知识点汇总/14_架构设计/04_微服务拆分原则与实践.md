## 1. 为什么要拆分微服务

### 单体架构的问题

```
痛点：
  - 代码膨胀：百万行代码，编译5分钟
  - 耦合严重：改一处影响多处
  - 部署效率低：一个小改动需要重新部署整个应用
  - 技术栈固化：无法对新模块使用新技术
  - 扩展性差：只能整体扩展，资源浪费
```

### 微服务的优势

```
优势：
  - 独立部署：小团队独立迭代
  - 技术自由：每个服务可选合适技术栈
  - 按需扩展：只扩展压力大的服务
  - 故障隔离：一个服务故障不影响其他

代价：
  - 分布式复杂性：网络、一致性、事务
  - 运维成本：监控、日志、链路追踪
  - 开发成本：接口定义、服务间通信
```

## 2. 拆分原则

### 高内聚低耦合

```
高内聚：一个服务内的功能高度相关
  ✅ 订单服务：创建订单、查询订单、取消订单
  ❌ 订单服务：创建订单 + 发送邮件 + 生成报表

低耦合：服务间依赖最小化
  ✅ 通过API/事件通信
  ❌ 共享数据库表、共享代码库
```

### 单一职责原则

```
每个服务只负责一个业务领域：
  - 用户服务：注册、登录、用户信息
  - 商品服务：商品CRUD、库存
  - 订单服务：下单、取消、查询
  - 支付服务：支付、退款
```

### 闭包原则（CCP）

```
同时变化的类放在同一个服务中：
  - 订单状态和订单详情一起变 → 放在订单服务
  - 商品价格和促销规则一起变 → 放在商品服务
```

### 数据所有权

```
每个服务拥有自己的数据，不共享数据库：
  ✅ 订单服务 → order_db
  ✅ 用户服务 → user_db
  ❌ 订单服务直接查user_db的用户表

需要用户信息 → 调用用户服务的API
```

## 3. DDD驱动拆分

### 限界上下文（Bounded Context）

```
DDD的核心概念：一个限界上下文对应一个微服务

电商系统限界上下文划分：

┌──────────┐ ┌──────────┐ ┌──────────┐
│ 用户上下文 │ │ 商品上下文 │ │ 订单上下文 │
│  User BC  │ │Product BC │ │ Order BC │
├──────────┤ ├──────────┤ ├──────────┤
│ 注册/登录  │ │ 商品管理   │ │ 下单/取消  │
│ 用户信息   │ │ 库存管理   │ │ 订单查询   │
│ 权限管理   │ │ 价格管理   │ │ 订单状态   │
└──────────┘ └──────────┘ └──────────┘
      ↕            ↕            ↕
┌──────────┐ ┌──────────┐ ┌──────────┐
│ 支付上下文 │ │ 物流上下文 │ │ 营销上下文 │
│Payment BC │ │Shipping BC│ │Market BC │
├──────────┤ ├──────────┤ ├──────────┤
│ 支付/退款  │ │ 发货/签收  │ │ 优惠券/活动│
│ 支付渠道   │ │ 物流跟踪   │ │ 促销规则   │
└──────────┘ └──────────┘ └──────────┘
```

### 上下文映射（Context Map）

```
上下文之间的关系：

1. 合作关系（Partnership）：两个上下文紧密协作
2. 共享内核（Shared Kernel）：共享部分领域模型
3. 客户-供应商（Customer-Supplier）：上游提供API，下游消费
4. 遵奉者（Conformist）：下游无条件遵从上游模型
5. 防腐层（ACL）：下游通过适配器隔离上游模型
6. 开放主机服务（OHS）：上游提供标准化API
7. 发布语言（PL）：用事件/消息通信

推荐：OHS + PL（API + 事件驱动），最小化耦合
```

### 防腐层实现

```java
// 防腐层：隔离外部服务的模型
@Service
public class UserAntiCorruptionLayer {
    
    @DubboReference
    private UserRemoteService userRemoteService;  // 外部服务
    
    // 将外部模型转换为本地模型
    public LocalUser getUser(Long userId) {
        UserDTO remoteUser = userRemoteService.getById(userId);
        // 转换：外部UserDTO → 本地LocalUser
        return LocalUser.builder()
            .id(remoteUser.getUserId())
            .name(remoteUser.getNickname())
            .level(remoteUser.getMemberLevel())
            .build();
    }
}
```

## 4. 拆分粒度

### 粒度判断标准

| 维度 | 粒度粗 | 粒度细 |
|------|--------|--------|
| 团队规模 | 大团队 | 小团队 |
| 服务数 | <20个 | >50个 |
| 部署频率 | 低 | 高 |
| 数据一致性 | 强一致 | 最终一致 |
| 团队人数 | 8-15人/服务 | 3-5人/服务 |

### 粒度过细的问题

```
❌ 过度拆分：
  - 订单创建服务和订单查询服务分开 → 不必要的网络开销
  - 用户登录服务和用户信息服务分开 → 频繁跨服务调用
  - 50+微服务，但团队只有10人 → 运维灾难

✅ 合理粒度：
  - 一个业务领域一个服务
  - 读写分离可用CQRS，不必拆成两个服务
  - 服务的部署频率和团队结构匹配
```

### 康威定律

```
"设计系统的架构受制于产生这些设计的组织的沟通结构"

含义：
  - 微服务拆分应该匹配团队组织结构
  - 10人团队 → 3-5个服务
  - 100人团队 → 15-30个服务
  - 服务边界 = 团队边界
```

## 5. 拆分策略

### 绞杀者模式（Strangler Fig）

```
逐步替换单体中的模块，而非一次性重写：

1. 在单体前加API网关
2. 新功能用微服务实现，通过网关路由
3. 逐步将单体中的旧功能迁移到微服务
4. 旧功能迁移完成后，删除单体中的对应代码
5. 最终单体被完全替代

        API Gateway
        /        \
   微服务A      单体应用
   (新功能)    (逐步减少)
```

### 分支策略

```
1. 按领域拆分（优先）：用户、商品、订单各自独立
2. 按变更频率拆分：高频变更的模块独立
3. 按性能要求拆分：高性能模块独立扩展
4. 按团队拆分：匹配团队组织结构

推荐顺序：领域 → 变更频率 → 性能 → 团队
```

### 数据拆分

```sql
-- 阶段1：共享数据库，应用层隔离
-- 各服务通过DAO访问同一数据库的不同表

-- 阶段2：数据库分库
-- 用户服务 → user_db
-- 订单服务 → order_db
-- 跨库JOIN改为API调用

-- 阶段3：分库分表
-- 订单库按用户ID分片
-- 使用ShardingSphere管理
```

## 6. 实战案例

### 电商系统拆分

```
阶段1：单体 → 按领域拆分
  用户服务、商品服务、订单服务、支付服务

阶段2：核心服务再拆分
  订单服务 → 订单交易 + 订单查询（CQRS）
  商品服务 → 商品管理 + 库存管理

阶段3：基础设施服务独立
  认证服务、消息服务、文件服务

最终架构：
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │ 用户  │  │ 商品  │  │ 订单  │  │ 支付  │  │ 物流  │
  │ 服务  │  │ 服务  │  │ 服务  │  │ 服务  │  │ 服务  │
  └──┬───┘  └──┬───┘  └──┬───┘  └──┬───┘  └──┬───┘
     │         │         │         │         │
  ┌──┴───┐ ┌──┴───┐ ┌──┴───┐ ┌──┴───┐ ┌──┴───┐
  │user_db│ │prod_db│ │order_db│ │pay_db│ │ship_db│
  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘
```

### 服务间通信

```java
// 同步调用（Dubbo）
@DubboReference
private ProductService productService;

public Order createOrder(OrderRequest request) {
    // 同步调用商品服务
    Product product = productService.getById(request.getProductId());
    // 创建订单
    return orderMapper.insert(buildOrder(product, request));
}

// 异步事件（MQ）
@Service
public class OrderService {
    
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    public void payOrder(Long orderId) {
        orderMapper.updateStatus(orderId, "PAID");
        // 发布订单支付事件
        mqTemplate.syncSend("order-paid", new OrderPaidEvent(orderId));
        // 物流服务消费事件，自动创建发货单
    }
}
```

## 7. 面试要点

### Q1: 微服务拆分的原则是什么？

```
1. 高内聚低耦合：功能相关的放一起，依赖最小化
2. 单一职责：一个服务一个业务领域
3. DDD驱动：限界上下文定义服务边界
4. 数据自治：每个服务拥有自己的数据库
5. 康威定律：服务边界匹配团队边界
```

### Q2: 微服务拆分过细有什么问题？

```
1. 网络开销：服务间调用增加延迟
2. 运维复杂：服务数量多，监控/部署/排障成本高
3. 数据一致性：跨服务事务难以保证
4. 团队负担：小团队维护大量服务力不从心
5. 建议：从粗粒度开始，按需拆分
```

### Q3: 如何从单体演进到微服务？

```
绞杀者模式：
1. 加API网关，流量先到网关
2. 新功能用微服务开发
3. 逐步迁移单体功能到微服务
4. 每次迁移后验证，可随时回退
5. 最终单体完全被替代

关键：增量迁移，不搞大爆炸式重写
```

### Q4: 微服务之间如何通信？

```
同步：Dubbo/gRPC/HTTP（适合实时性要求高）
异步：Kafka/RocketMQ/RabbitMQ（适合解耦和削峰）

选型：
  - 核心链路（下单→支付）：同步调用，实时反馈
  - 非核心链路（支付→发货）：异步事件，解耦
  - 查询类：同步调用或CQRS读模型
```

### Q5: 微服务的数据一致性怎么保证？

```
1. 最终一致性 + 事件驱动
2. Saga模式：长事务拆分为多个本地事务+补偿
3. TCC模式：Try-Confirm-Cancel
4. 本地消息表：本地事务+消息表，定时投递
5. Seata：AT模式（自动补偿）

推荐：非金融场景用最终一致性+MQ，金融场景用TCC
```
