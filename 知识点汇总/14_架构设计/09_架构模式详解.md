## 1. 分层架构

```
分层架构（Layered Architecture）= 最常见的单体架构

┌─────────────────────┐
│  Presentation Layer  │  ← Controller / API
├─────────────────────┤
│  Business Layer      │  ← Service / Domain
├─────────────────────┤
│  Persistence Layer   │  ← Repository / DAO
├─────────────────────┤
│  Database Layer      │  ← MySQL / Redis
└─────────────────────┘

特点：
  - 每层只依赖下层
  - 关注点分离
  - 简单易懂

缺点：
  - 容易出现"穿透"（Controller直接调DAO）
  - 领域逻辑散落在Service中
  - 难以替换数据库
```

## 2. 六边形架构

```
六边形架构（Hexagonal / Ports and Adapters）

         ┌──────────────────┐
Web API ←→│                  │←→ REST Adapter
         │   Domain Core     │
   DB   ←→│  (端口+适配器)    │←→ Message Queue
         │                  │←→ External API
  Test  ←→│                  │←→ Test Mock
         └──────────────────┘

核心思想：
  - 领域逻辑在中心，不依赖任何外部
  - 端口（Port）= 接口定义
  - 适配器（Adapter）= 接口实现
  - 外部技术可替换（换DB/换MQ不影响领域）

端口 = 定义领域需要什么能力
  interface OrderRepository {  // 端口
      Order findById(Long id);
      void save(Order order);
  }

适配器 = 具体技术实现
  class JpaOrderRepository implements OrderRepository {  // 适配器
      // JPA实现
  }
  class MongoOrderRepository implements OrderRepository {  // 另一个适配器
      // MongoDB实现
  }
```

## 3. 洋葱架构

```
洋葱架构（Onion Architecture）= 六边形的演进

       ┌───────────────────────┐
       │    Infrastructure      │ ← 外层：DB/MQ/Email
       │  ┌─────────────────┐  │
       │  │  Application     │  │ ← 应用服务：用例编排
       │  │  ┌───────────┐  │  │
       │  │  │  Domain    │  │  │ ← 核心：实体/值对象/领域服务
       │  │  │  (纯业务)  │  │  │     不依赖任何外部
       │  │  └───────────┘  │  │
       │  └─────────────────┘  │
       └───────────────────────┘

依赖方向：外层依赖内层，内层不依赖外层

层次：
  Domain：实体、值对象、领域事件、领域服务（纯业务，无框架依赖）
  Application：用例编排、事务管理、DTO转换
  Infrastructure：数据库、消息队列、外部API
  Presentation：Controller、API
```

## 4. CQRS

```
CQRS = Command Query Responsibility Segregation（命令查询职责分离）

传统：一个Model同时负责读和写
CQRS：读模型和写模型分离

  ┌──────────┐         ┌──────────┐
  │  Write    │         │   Read    │
  │  Model    │         │   Model   │
  │ (Command) │         │  (Query)  │
  └─────┬────┘         └─────┬────┘
        │                    │
  ┌─────▼────┐         ┌─────▼────┐
  │  写入DB   │ ─同步→  │  读DB     │
  │ (规范化)  │ ─异步→  │ (反范式)  │
  │           │  MQ     │ (ES/Redis)│
  └──────────┘         └──────────┘

适用场景：
  - 读写比例悬殊（读多写少）
  - 读需要复杂查询（报表/搜索）
  - 写需要严格事务一致性

示例：
  写：INSERT INTO orders (...)  → 规范化表，强一致
  读：SELECT * FROM order_view  → 反范式视图/ES，高性能
```

## 5. 事件驱动架构

```
事件驱动架构（EDA = Event-Driven Architecture）

  Producer → [Event Bus / MQ] → Consumer

  ┌─────────┐                    ┌──────────┐
  │ Order    │ ──OrderCreated──→ │ Inventory │ ← 扣减库存
  │ Service  │ ──OrderCreated──→ │ Points    │ ← 加积分
  │          │ ──OrderCreated──→ │ Notify    │ ← 发通知
  └─────────┘                    └──────────┘

特点：
  - 松耦合：生产者不关心消费者
  - 异步：事件发布后立即返回
  - 可扩展：新增消费者不影响生产者
  - 最终一致：通过事件最终达到一致

事件类型：
  - 事件通知（Event Notification）：只通知，不含完整数据
  - 事件携带状态转移（Event-Carried State Transfer）：含完整数据
  - 事件溯源（Event Sourcing）：事件即真相
```

## 6. 事件溯源

```
事件溯源（Event Sourcing）= 所有状态变更记录为事件序列

传统：只存当前状态
  orders表：id=1, status=shipped, amount=100

事件溯源：存所有变更事件
  events表：
  1. OrderCreated(orderId=1, amount=100)
  2. OrderPaid(orderId=1, paidAmount=100)
  3. OrderShipped(orderId=1, trackingNo=SF123)

当前状态 = 重放所有事件

  events → replay → Order(id=1, status=shipped, amount=100)

优势：
  - 完整审计日志
  - 可回溯任意时间点状态
  - 支持时间旅行调试
  - 天然适配CQRS

劣势：
  - 重放开销
  - 事件Schema演化困难
  - 快照优化复杂
```

## 7. SAGA模式

```
SAGA = 分布式事务的最终一致方案

  事务T = T1 → T2 → T3（每个T是本地事务）
  补偿 = C1 ← C2 ← C3（每个C是对应T的补偿操作）

  正常流程：
  T1(扣款) → T2(扣库存) → T3(创建订单) → 成功

  T2失败：
  T1(扣款) → T2(扣库存)❌ → C1(退款) → 失败

  两种实现：
  1. 编排（Orchestration）：中央协调器控制流程
     SagaCoordinator → 调用各服务 → 失败则调用补偿

  2. 协同（Choreography）：事件驱动，无中央协调器
     OrderService → OrderCreated → InventoryService → InventoryDeducted → ...
```

## 8. Sidecar模式

```
Sidecar = 与主容器同Pod的辅助容器

  ┌─────────────────── Pod ───────────────────┐
  │                                            │
  │  ┌──────────┐        ┌──────────────┐    │
  │  │  App      │ ←───→ │  Sidecar      │    │
  │  │  Container│        │  (Proxy/Agent)│    │
  │  │  (业务)   │        │  (治理逻辑)    │    │
  │  └──────────┘        └──────────────┘    │
  │                                            │
  └────────────────────────────────────────────┘

Istio Envoy就是Sidecar：
  - 拦截所有出入流量
  - 负载均衡、熔断、限流
  - mTLS加密
  - 链路追踪
  - 业务代码无感知

优势：业务与治理解耦
劣势：每Pod一个Sidecar，资源开销
```

## 9. Strangler Fig模式

```
绞杀者模式（Strangler Fig）= 渐进式单体→微服务迁移

  阶段1：单体应用 + 少量新服务
  ┌────────────────┐
  │    Monolith     │ ← 大部分请求
  │    ┌────────┐   │
  │    │ Feature │ ← 新功能独立服务
  │    └────────┘   │
  └────────────────┘
       ↑ API Gateway分流

  阶段2：逐步"绞杀"单体
  ┌──────┐ ┌──────┐ ┌──────────┐
  │ Svc A │ │ Svc B │ │ Monolith  │ ← 越来越小
  └──────┘ └──────┘ └──────────┘

  阶段3：单体消失
  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
  │ Svc A │ │ Svc B │ │ Svc C │ │ Svc D │
  └──────┘ └──────┘ └──────┘ └──────┘

优势：
  - 渐进式迁移，风险可控
  - 不影响现有功能
  - 可以随时暂停
```

## 10. 架构对比与选型

| 架构模式 | 核心思想 | 适用场景 | 复杂度 |
|----------|----------|----------|--------|
| 分层架构 | 按层分离关注点 | 小型应用、CRUD系统 | ⭐ |
| 六边形架构 | 端口适配器，领域隔离 | 需要替换外部技术的系统 | ⭐⭐⭐ |
| 洋葱架构 | 依赖向内，领域为核心 | DDD领域驱动设计 | ⭐⭐⭐ |
| CQRS | 读写分离 | 读多写少、复杂查询 | ⭐⭐⭐ |
| 事件驱动 | 松耦合异步 | 微服务间通信 | ⭐⭐⭐ |
| 事件溯源 | 事件即真相 | 需要完整审计、时间旅行 | ⭐⭐⭐⭐ |
| SAGA | 分布式事务补偿 | 微服务跨服务事务 | ⭐⭐⭐⭐ |
| Sidecar | 治理逻辑旁路 | 服务网格(Istio) | ⭐⭐⭐ |
| Strangler Fig | 渐进式迁移 | 单体→微服务改造 | ⭐⭐ |

```
选型建议：
  小项目 → 分层架构
  DDD项目 → 洋葱架构/六边形架构
  高并发读 → CQRS + 事件驱动
  微服务事务 → SAGA
  微服务治理 → Sidecar (Istio)
  系统重构 → Strangler Fig
```

## 面试题速查

**Q1: 六边形架构的核心思想？**
```
领域逻辑在中心，不依赖外部技术
通过端口（接口）定义能力，适配器（实现）连接外部
外部技术可替换（换DB/MQ不影响领域逻辑）
```

**Q2: CQRS解决什么问题？**
```
读写分离：
- 写模型：规范化设计，强一致性
- 读模型：反范式/ES/Redis，高性能查询
适用：读写比例悬殊、读需要复杂查询
```

**Q3: 事件溯源和审计日志的区别？**
```
审计日志：额外记录操作日志，状态存在DB
事件溯源：事件本身就是状态来源，不存最终状态
事件溯源可以重放事件重建任意时间点状态
```

**Q4: SAGA的编排和协同区别？**
```
编排（Orchestration）：中央协调器控制流程，逻辑集中
协同（Choreography）：事件驱动，各服务自治，无中心
编排适合复杂流程，协同适合简单场景
```
