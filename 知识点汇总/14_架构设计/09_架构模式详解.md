## 1. 分层架构模式

### 1.1 模式概述

分层架构是最常见的软件架构模式，将系统按职责水平划分为若干层，每层只与相邻层通信：

```
┌─────────────────────────┐
│   表现层 (Presentation)   │  ← Controller / API
├─────────────────────────┤
│   业务层 (Business)      │  ← Service / Domain Logic
├─────────────────────────┤
│   持久层 (Persistence)   │  ← DAO / Repository
├─────────────────────────┤
│   数据库层 (Database)    │  ← MySQL / Redis
└─────────────────────────┘
```

### 1.2 分层规则

- **单向依赖**：上层依赖下层，下层不感知上层
- **封闭层**：请求必须逐层传递，不能跨层调用（Controller不能直接访问DAO）
- **开放层**（可选）：允许跨层调用，如Service可以跳过DAO直接访问数据库

### 1.3 代码示例

```java
// ===== 表现层 =====
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @Autowired
    private OrderService orderService;

    @PostMapping
    public ResponseEntity<OrderVO> createOrder(@RequestBody OrderCreateDTO dto) {
        OrderVO vo = orderService.createOrder(dto);
        return ResponseEntity.ok(vo);
    }

    @GetMapping("/{id}")
    public ResponseEntity<OrderVO> getOrder(@PathVariable Long id) {
        return ResponseEntity.ok(orderService.getOrder(id));
    }
}

// ===== 业务层 =====
@Service
@Transactional
public class OrderService {

    @Autowired
    private OrderRepository orderRepository;
    @Autowired
    private ProductService productService;
    @Autowired
    private PaymentService paymentService;

    public OrderVO createOrder(OrderCreateDTO dto) {
        // 1. 校验商品
        Product product = productService.findById(dto.getProductId());
        if (product.getStock() < dto.getQuantity()) {
            throw new BusinessException("库存不足");
        }

        // 2. 创建订单
        Order order = new Order();
        order.setProductId(dto.getProductId());
        order.setQuantity(dto.getQuantity());
        order.setTotalPrice(product.getPrice().multiply(
            BigDecimal.valueOf(dto.getQuantity())));
        order.setStatus(OrderStatus.PENDING);

        order = orderRepository.save(order);

        // 3. 扣减库存
        productService.deductStock(dto.getProductId(), dto.getQuantity());

        return OrderVO.from(order);
    }
}

// ===== 持久层 =====
@Repository
public interface OrderRepository extends JpaRepository<Order, Long> {

    List<Order> findByUserIdAndStatus(Long userId, OrderStatus status);

    @Query("SELECT o FROM Order o WHERE o.createdAt BETWEEN :start AND :end")
    List<Order> findByDateRange(@Param("start") LocalDateTime start,
                                 @Param("end") LocalDateTime end);
}
```

### 1.4 优缺点

| 优点 | 缺点 |
|------|------|
| 简单易懂，团队上手快 | 容易沦为"贫血模型" |
| 层职责清晰，便于分工 | 单体架构扩展性差 |
| 可独立替换某层实现 | 容易出现跨层调用 |
| 大部分IDE和框架原生支持 | 数据库驱动设计，业务逻辑散落 |

## 2. 六边形架构（端口与适配器）

### 2.1 模式概述

六边形架构由Alistair Cockburn提出，核心思想是**将业务逻辑放在中心，通过端口（Port）与外部交互，适配器（Adapter）实现端口**：

```
                    ┌───────────────┐
    HTTP Adapter →  │               │  → JPA Adapter
                    │   Application │
    gRPC Adapter →  │    Domain     │  → Redis Adapter
                    │    (Core)     │
    CLI Adapter  →  │               │  → Kafka Adapter
                    └───────────────┘
                    ↑ 端口(Port)    ↑ 端口(Port)
                    驱动端           被驱动端
```

### 2.2 核心概念

- **端口（Port）**：接口，定义应用与外界的交互契约
  - **驱动端口（Driving Port）**：外部调用应用的接口（如OrderService）
  - **被驱动端口（Driven Port）**：应用调用外部的接口（如OrderRepository）
- **适配器（Adapter）**：端口的实现
  - **主适配器**：实现驱动端口（如REST Controller）
  - **次适配器**：实现被驱动端口（如JPA Repository）

### 2.3 代码示例

```java
// ===== 领域模型（核心，不依赖任何框架）=====
public class Order {
    private OrderId id;
    private CustomerId customerId;
    private List<OrderLine> lines;
    private OrderStatus status;
    private Money totalAmount;

    // 领域逻辑
    public void confirm() {
        if (status != OrderStatus.PENDING) {
            throw new DomainException("只有待处理订单才能确认");
        }
        this.status = OrderStatus.CONFIRMED;
    }

    public void cancel(String reason) {
        if (status == OrderStatus.DELIVERED) {
            throw new DomainException("已送达订单不能取消");
        }
        this.status = OrderStatus.CANCELLED;
    }
}

// ===== 驱动端口（应用接口）=====
public interface OrderUseCase {
    OrderResult createOrder(CreateOrderCommand cmd);
    OrderResult confirmOrder(Long orderId);
    OrderResult cancelOrder(Long orderId, String reason);
}

// ===== 被驱动端口（基础设施接口）=====
public interface OrderRepository {
    void save(Order order);
    Optional<Order> findById(OrderId id);
    List<Order> findByCustomer(CustomerId customerId);
}

public interface PaymentGateway {
    PaymentResult charge(OrderId orderId, Money amount);
}

// ===== 应用服务（实现驱动端口，使用被驱动端口）=====
@Service
public class OrderService implements OrderUseCase {

    private final OrderRepository orderRepository;
    private final PaymentGateway paymentGateway;

    public OrderService(OrderRepository orderRepository,
                        PaymentGateway paymentGateway) {
        this.orderRepository = orderRepository;
        this.paymentGateway = paymentGateway;
    }

    @Override
    public OrderResult createOrder(CreateOrderCommand cmd) {
        Order order = Order.create(cmd.getCustomerId(), cmd.getLines());
        orderRepository.save(order);
        return OrderResult.from(order);
    }

    @Override
    public OrderResult confirmOrder(Long orderId) {
        Order order = orderRepository.findById(new OrderId(orderId))
            .orElseThrow(() -> new NotFoundException("订单不存在"));
        order.confirm();
        paymentGateway.charge(order.getId(), order.getTotalAmount());
        orderRepository.save(order);
        return OrderResult.from(order);
    }
}

// ===== 次适配器（实现被驱动端口）=====
@Repository
public class JpaOrderRepository implements OrderRepository {

    @Autowired
    private OrderJpaRepository jpaRepository;

    @Override
    public void save(Order order) {
        OrderEntity entity = OrderEntity.fromDomain(order);
        jpaRepository.save(entity);
    }

    @Override
    public Optional<Order> findById(OrderId id) {
        return jpaRepository.findById(id.getValue())
            .map(OrderEntity::toDomain);
    }

    @Override
    public List<Order> findByCustomer(CustomerId customerId) {
        return jpaRepository.findByCustomerId(customerId.getValue())
            .stream()
            .map(OrderEntity::toDomain)
            .collect(Collectors.toList());
    }
}

// ===== 主适配器（REST Controller）=====
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final OrderUseCase orderUseCase;

    public OrderController(OrderUseCase orderUseCase) {
        this.orderUseCase = orderUseCase;
    }

    @PostMapping
    public ResponseEntity<OrderResult> create(@RequestBody CreateOrderCommand cmd) {
        return ResponseEntity.ok(orderUseCase.createOrder(cmd));
    }

    @PostMapping("/{id}/confirm")
    public ResponseEntity<OrderResult> confirm(@PathVariable Long id) {
        return ResponseEntity.ok(orderUseCase.confirmOrder(id));
    }
}
```

### 2.4 优缺点

| 优点 | 缺点 |
|------|------|
| 业务逻辑完全隔离，可测试性极强 | 接口和转换层多，代码量大 |
| 可轻松替换外部组件（换DB、换MQ） | 学习曲线较陡 |
| 领域模型是富模型，不是贫血模型 | 对简单CRUD应用过度设计 |

## 3. 洋葱架构

### 3.1 模式概述

洋葱架构由Jeffrey Palermo提出，是六边形架构的演进。核心思想类似，但强调**层次依赖方向始终向内**：

```
            ┌─────────────────────────────┐
            │      Infrastructure         │  ← 外层：DB, MQ, Email
            │  ┌───────────────────────┐  │
            │  │    Application        │  │  ← 应用服务
            │  │  ┌─────────────────┐  │  │
            │  │  │   Domain         │  │  │  ← 核心：实体、值对象、领域服务
            │  │  │  ┌───────────┐   │  │  │
            │  │  │  │  Models    │   │  │  │
            │  │  │  └───────────┘   │  │  │
            │  │  └─────────────────┘  │  │
            │  └───────────────────────┘  │
            └─────────────────────────────┘

    依赖方向：外层 → 内层（内层不知道外层的存在）
```

### 3.2 层次定义

1. **Domain Services（领域服务层）**：领域接口定义（如Repository接口）
2. **Domain Models（领域模型层）**：实体、值对象、领域事件
3. **Application Services（应用服务层）**：用例编排，事务管理
4. **Infrastructure（基础设施层）**：数据库实现、消息队列、外部API

### 3.3 与六边形架构的区别

```
六边形架构：强调端口与适配器，内外通过端口交互
洋葱架构：  强调依赖方向，外层依赖内层，层次更严格

实际项目中两者经常混用：
- 洋葱架构的分层思想 + 六边形架构的端口适配器机制
```

## 4. CQRS模式

### 4.1 模式概述

CQRS（Command Query Responsibility Segregation）将系统分为**命令端（写）和查询端（读）**，各自有独立的模型和数据存储：

```
         ┌─────────┐                    ┌─────────┐
  写请求  │ Command │                    │  Query  │  读请求
 ────────→│  Side   │                    │  Side   │←───────
          │         │                    │         │
          │ Write DB│                    │ Read DB │
          │ (MySQL) │                    │(Elastic)│
          └────┬────┘                    └────▲────┘
               │                              │
               │     事件/消息同步              │
               └──────────────────────────────┘
```

### 4.2 命令端实现

```java
// 命令对象（不返回数据，只返回执行结果）
public record CreateOrderCommand(
    Long customerId,
    List<OrderLineDTO> lines,
    String shippingAddress
) {}

public record CancelOrderCommand(
    Long orderId,
    String reason
) {}

// 命令处理器
@Service
public class OrderCommandHandler {

    @Autowired
    private OrderRepository orderRepository; // 写模型Repository
    @Autowired
    private ApplicationEventPublisher eventPublisher;

    @Transactional
    public Long handle(CreateOrderCommand cmd) {
        // 1. 验证和创建领域对象
        Order order = Order.create(cmd.customerId(), cmd.lines());

        // 2. 持久化到写库
        order = orderRepository.save(order);

        // 3. 发布领域事件（用于同步读库）
        eventPublisher.publishEvent(new OrderCreatedEvent(
            order.getId(),
            order.getCustomerId(),
            order.getTotalAmount(),
            order.getLines(),
            Instant.now()
        ));

        return order.getId();
    }

    @Transactional
    public void handle(CancelOrderCommand cmd) {
        Order order = orderRepository.findById(cmd.orderId())
            .orElseThrow(() -> new NotFoundException("订单不存在"));

        order.cancel(cmd.reason());
        orderRepository.save(order);

        eventPublisher.publishEvent(new OrderCancelledEvent(
            order.getId(), cmd.reason(), Instant.now()
        ));
    }
}
```

```java
            rm.setStatus("CANCELLED");
            readRepository.save(rm);
        });
    }
}
```

## 5. 事件驱动架构

### 5.1 模式概述

事件驱动架构（EDA）中，组件通过**发布/订阅事件**进行通信，而非直接调用：

```
  ┌──────────┐   事件    ┌────────────┐   事件    ┌──────────────┐
  │ Order    │──────────→│ Event Bus  │──────────→│ Inventory    │
  │ Service  │           │ (Kafka)    │           │ Service      │
  └──────────┘           │            │           └──────────────┘
                         │            │   事件    ┌──────────────┐
                         │            │──────────→│ Notification │
                         │            │           │ Service      │
                         └────────────┘           └──────────────┘
```

### 5.2 事件定义


### 5.3 事件发布与处理

```java
// 事件发布者
@Service
public class OrderService {

    @Autowired
    private OrderRepository orderRepository;
    @Autowired
    private KafkaTemplate<String, DomainEvent> kafkaTemplate;

    private static final String ORDER_EVENTS_TOPIC = "order-events";

    @Transactional
    public Long placeOrder(OrderRequest request) {
        Order order = createOrder(request);
        orderRepository.save(order);

        // 发布事件到Kafka
        OrderPlacedEvent event = new OrderPlacedEvent(
            order.getId(),
            order.getCustomerId(),
            order.getItems(),
            order.getTotalAmount()
        );

        kafkaTemplate.send(ORDER_EVENTS_TOPIC,
            String.valueOf(order.getId()), event);

        return order.getId();
    }
}

// 事件消费者——库存服务
@Service
public class InventoryEventConsumer {

    @KafkaListener(topics = "order-events", groupId = "inventory-service")
    public void handle(OrderPlacedEvent event) {
        // 扣减库存
        for (OrderItem item : event.getItems()) {
            inventoryService.deductStock(item.getProductId(),
                item.getQuantity());
        }
    }
}

// 事件消费者——通知服务
@Service
public class NotificationEventConsumer {

    @KafkaListener(topics = "order-events", groupId = "notification-service")
    public void handle(OrderPlacedEvent event) {
        // 发送订单确认邮件
        emailService.sendOrderConfirmation(
            event.getCustomerId(),
            event.getOrderId(),
            event.getTotalAmount()
        );
    }
}
```

### 5.4 事件编排 vs 事件链式

```java
// 方案1：事件链式（每个服务消费事件后发布新事件）
// Order → OrderPlacedEvent → Inventory → StockDeductedEvent → Payment → ...

// 方案2：事件编排器（Orchestrator集中管理流程）
@Service
public class OrderOrchestrator {

    @KafkaListener(topics = "order-events")
    public void onOrderPlaced(OrderPlacedEvent event) {
        // 1. 调用库存服务
        boolean stockOk = inventoryClient.deductStock(event);
        if (!stockOk) {
            publishEvent(new OrderFailedEvent(event.getOrderId(), "库存不足"));
            return;
        }

        // 2. 调用支付服务
        boolean paymentOk = paymentClient.charge(event);
        if (!paymentOk) {
            // 补偿：恢复库存
            inventoryClient.restoreStock(event);
            publishEvent(new OrderFailedEvent(event.getOrderId(), "支付失败"));
            return;
        }

        // 3. 确认订单
        publishEvent(new OrderConfirmedEvent(event.getOrderId()));
    }
}
```

## 6. 事件溯源

### 6.1 模式概述

事件溯源（Event Sourcing）不存储对象的当前状态，而是**存储所有导致状态变化的事件**。当前状态通过重放事件得到：

```
传统方式：存储最终状态
  UPDATE account SET balance = 850 WHERE id = 1;

事件溯源：存储事件序列
  INSERT INTO events (type, data) VALUES ('AccountCreated', {balance: 1000});
  INSERT INTO events (type, data) VALUES ('Deposited', {amount: 200});
  INSERT INTO events (type, data) VALUES ('Withdrew', {amount: 350});
  -- 当前余额 = 1000 + 200 - 350 = 850（通过重放事件计算）
```

### 6.2 事件存储

```java
// 事件基类

// 事件存储
public interface EventStore {

// 事件存储实现
@Repository
public class JpaEventStore implements EventStore {

    @Autowired
    private EventEntityRepository repository;

    @Override
    @Transactional
    public void append(UUID aggregateId, List<Event> events, long expectedVersion) {
        // 乐观并发控制
        long currentVersion = repository
            .findMaxVersionByAggregateId(aggregateId).orElse(0L);
        if (currentVersion != expectedVersion) {
            throw new OptimisticConcurrencyException(
                "并发冲突：期望版本 " + expectedVersion + "，实际版本 " + currentVersion);
        }

        long version = expectedVersion;
        for (Event event : events) {
            version++;
            EventEntity entity = EventEntity.builder()
                .aggregateId(aggregateId)
                .eventType(event.getClass().getName())
                .eventData(serialize(event))
                .version(version)
                .timestamp(event.getTimestamp())
                .build();
            repository.save(entity);
        }
    }

    @Override
    public List<Event> load(UUID aggregateId) {
        return repository.findByAggregateIdOrderByVersionAsc(aggregateId)
            .stream()
            .map(this::deserialize)
            .collect(Collectors.toList());
    }
}
```

### 6.3 聚合根重放

```java
// 聚合根——通过重放事件恢复状态
public class Account {
    private UUID id;
    private String owner;
    private BigDecimal balance;
    private long version = 0;
    private List<Event> uncommittedEvents = new ArrayList<>();

    // 从历史事件重建
    public static Account fromEvents(List<Event> events) {
        Account account = new Account();
        for (Event event : events) {
            account.apply(event);
            account.version++;
        }
        return account;
    }

    // 命令方法——产生事件
    public void deposit(BigDecimal amount, String description) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new DomainException("存款金额必须大于0");
        }
        raise(new MoneyDeposited(id, amount, description));
    }

    public void withdraw(BigDecimal amount) {
        if (amount.compareTo(balance) > 0) {
            throw new DomainException("余额不足");
        }
        raise(new MoneyWithdrawn(id, amount));
    }

    // 事件应用——修改状态
    private void apply(AccountCreated event) {
        this.id = event.getAccountId();
        this.owner = event.getOwner();
        this.balance = event.getInitialBalance();
    }

    private void apply(MoneyDeposited event) {
        this.balance = this.balance.add(event.getAmount());
    }

    private void apply(MoneyWithdrawn event) {
        this.balance = this.balance.subtract(event.getAmount());
    }

    // 统一事件应用入口
    private void raise(Event event) {
        apply(event);
        uncommittedEvents.add(event);
    }

    @SuppressWarnings("unchecked")
    private void apply(Event event) {
        try {
            String methodName = "apply";
            Class<?> eventClass = event.getClass();
            Method method = this.getClass().getDeclaredMethod(methodName, eventClass);
            method.setAccessible(true);
            method.invoke(this, event);
        } catch (Exception e) {
            throw new RuntimeException("无法应用事件: " + event.getClass(), e);
        }
    }
}
```

### 6.4 快照优化

当事件数量过多时，重放性能下降。使用快照定期保存状态：

```java
@Service
public class AccountRepository {

    @Autowired
    private EventStore eventStore;
    @Autowired
    private SnapshotStore snapshotStore;

    private static final int SNAPSHOT_INTERVAL = 100; // 每100个事件做一次快照

    public Account load(UUID accountId) {
        // 1. 先查最新快照
        Optional<Snapshot> snapshot = snapshotStore
            .findLatest(accountId);

        Account account;
        long fromVersion;

        if (snapshot.isPresent()) {
            // 2. 从快照恢复
            account = (Account) snapshot.get().getState();
            fromVersion = snapshot.get().getVersion();
        } else {
            // 3. 无快照，从头重放
            account = new Account();
            fromVersion = 0;
        }

        // 4. 重放快照之后的事件
        List<Event> events = eventStore.loadFrom(accountId, fromVersion);
        for (Event event : events) {
            account.apply(event);
        }

        return account;
    }

    public void save(Account account) {
        // 1. 追加事件
        eventStore.append(account.getId(),
            account.getUncommittedEvents(), account.getVersion());

        // 2. 判断是否需要快照
        if (account.getVersion() % SNAPSHOT_INTERVAL == 0) {
            snapshotStore.save(new Snapshot(
                account.getId(),
                account.getVersion(),
                account // 深拷贝
            ));
        }
    }
}
```

## 7. SAGA分布式事务模式

### 7.1 模式概述

SAGA将一个分布式事务拆分为一系列本地事务，每个事务有对应的**补偿操作**。失败时通过执行补偿操作回滚：

```
正常流程：
  T1(创建订单) → T2(扣减库存) → T3(扣减余额) → T4(确认订单)

失败回滚（T3失败）：
  T1(创建订单) → T2(扣减库存) → T3(扣减余额)✗
                                         ↓
  C1(取消订单) ← C2(恢复库存) ← ────────┘
```

### 7.2 编排式SAGA（Choreography）

```java
// 各服务监听事件并自主决策，无中央协调器

// 订单服务
@Service
public class OrderSagaParticipant {

    @Transactional
    public Long createOrder(OrderRequest req) {
        Order order = Order.create(req);
        orderRepository.save(order);
        // 发布事件触发下一个步骤
        eventPublisher.publish(new OrderCreatedEvent(order));
        return order.getId();
    }

    @EventListener
    public void onPaymentFailed(PaymentFailedEvent event) {
        // 补偿：取消订单
        orderRepository.findById(event.getOrderId()).ifPresent(order -> {
            order.cancel("支付失败");
            orderRepository.save(order);
        });
    }
}

// 库存服务
@Service
public class InventorySagaParticipant {

    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        try {
            inventoryService.deductStock(event.getItems());
            eventPublisher.publish(new StockDeductedEvent(event.getOrderId()));
        } catch (Exception e) {
            eventPublisher.publish(new StockDeductFailedEvent(
                event.getOrderId(), e.getMessage()));
        }
    }

    @EventListener
    public void onPaymentFailed(PaymentFailedEvent event) {
        // 补偿：恢复库存
        inventoryService.restoreStock(event.getOrderId(), event.getItems());
    }
}

// 支付服务
@Service
public class PaymentSagaParticipant {

    @EventListener
    public void onStockDeducted(StockDeductedEvent event) {
        try {
            paymentService.charge(event.getOrderId(), event.getAmount());
            eventPublisher.publish(new PaymentSucceededEvent(event.getOrderId()));
        } catch (Exception e) {
            eventPublisher.publish(new PaymentFailedEvent(
                event.getOrderId(), event.getAmount(), event.getItems()));
        }
    }
}
```

### 7.3 协调式SAGA（Orchestration）

```java
// 中央协调器管理整个流程
@Service
public class OrderSagaOrchestrator {

    @Autowired
    private OrderClient orderClient;
    @Autowired
    private InventoryClient inventoryClient;
    @Autowired
    private PaymentClient paymentClient;
    @Autowired
    private SagaStateRepository sagaStateRepository;

    @Transactional
    public void execute(OrderSagaContext context) {
        // 保存Saga状态
        SagaState state = new SagaState(context.getOrderId(), "STARTED");
        sagaStateRepository.save(state);

        try {
            // Step 1: 创建订单
            state.setCurrentStep("CREATE_ORDER");
            sagaStateRepository.save(state);
            orderClient.createOrder(context);

            // Step 2: 扣减库存
            state.setCurrentStep("DEDUCT_INVENTORY");
            sagaStateRepository.save(state);
            inventoryClient.deduct(context.getOrderId(), context.getItems());

            // Step 3: 支付
            state.setCurrentStep("PAYMENT");
            sagaStateRepository.save(state);
            paymentClient.charge(context.getOrderId(), context.getAmount());

            // 完成
            state.setStatus("COMPLETED");
            sagaStateRepository.save(state);

        } catch (Exception e) {
            log.error("Saga失败，开始补偿: {}", context.getOrderId(), e);
            compensate(state, context);
        }
    }

    @Transactional
    public void compensate(SagaState state, OrderSagaContext context) {
        state.setStatus("COMPENSATING");
        sagaStateRepository.save(state);

        // 按逆序执行补偿
        try {
            if (state.getCompletedSteps().contains("PAYMENT")) {
                paymentClient.refund(context.getOrderId());
            }
            if (state.getCompletedSteps().contains("DEDUCT_INVENTORY")) {
                inventoryClient.restore(context.getOrderId(), context.getItems());
            }
            if (state.getCompletedSteps().contains("CREATE_ORDER")) {
                orderClient.cancelOrder(context.getOrderId());
            }
            state.setStatus("COMPENSATED");
        } catch (Exception e) {
            state.setStatus("COMPENSATION_FAILED");
            // 重试或人工介入
        }
        sagaStateRepository.save(state);
    }
}
```

### 7.4 两种方式对比

| 特性 | 编排式（Choreography） | 协调式（Orchestration） |
|------|----------------------|----------------------|
| 协调方式 | 事件驱动，去中心化 | 中央协调器管理 |
| 耦合度 | 低 | 中 |
| 流程可见性 | 差（散落各服务） | 好（集中管理） |
| 复杂度 | 简单流程适合 | 复杂流程适合 |
| 循环依赖风险 | 高 | 低 |
| 调试难度 | 难 | 相对容易 |

## 9. Strangler Fig（绞杀者）模式

### 9.1 模式概述

Strangler Fig模式是一种**渐进式迁移策略**：新系统逐步替代旧系统，像绞杀植物一样慢慢包裹最终取代宿主：

```
阶段1: 初始状态
┌─────────────────────────────┐
│     旧系统（Monolith）       │
│  ┌─────┐ ┌─────┐ ┌─────┐   │
│  │模块A │ │模块B │ │模块C │   │
│  └─────┘ └─────┘ └─────┘   │
└─────────────────────────────┘

阶段2: 新服务接管部分功能
┌─────────────────────────────┐
│     旧系统（Monolith）       │
│  ┌─────┐ ┌─────┐           │      ┌─────────┐
│  │模块A │ │模块C │           │      │ 新服务B  │
│  └─────┘ └─────┘           │      └─────────┘
└─────────────────────────────┘
         ↑                         ↑
    ┌─────────────────────────────┐
    │      API Gateway / Facade   │  ← 统一入口路由
    └─────────────────────────────┘
              ↑
           客户端请求

阶段3: 完全替代
                                    ┌─────────┐ ┌─────────┐
                                    │ 新服务A  │ │ 新服务C  │
                                    └─────────┘ └─────────┘
    ┌─────────────────────────────┐
    │      API Gateway            │
    └─────────────────────────────┘
```

### 9.2 实现示例

```java
// API Gateway / Facade——路由请求到新系统或旧系统
@RestController
@RequestMapping("/api")
public class MigrationFacade {

    @Autowired
    private LegacySystemClient legacyClient;
    @Autowired
    private NewOrderService newOrderService;
    @Autowired
    private FeatureToggleManager featureToggle;

    @PostMapping("/orders")
    public ResponseEntity<?> createOrder(@RequestBody OrderRequest request) {
        // 特性开关决定走新系统还是旧系统
        if (featureToggle.isEnabled("use-new-order-service")) {
            try {
                OrderResult result = newOrderService.createOrder(request);
                return ResponseEntity.ok(result);
            } catch (Exception e) {
                log.error("新系统异常，降级到旧系统", e);
                // 可选：降级到旧系统
                return legacyClient.createOrder(request);
            }
        } else {
            return legacyClient.createOrder(request);
        }
    }

    @GetMapping("/orders/{id}")
    public ResponseEntity<?> getOrder(@PathVariable Long id) {
        // 可以同时查新旘认系统做对比验证（暗部署）
        if (featureToggle.isEnabled("shadow-new-order-service")) {
            // 影子调用——不影响主流程
            CompletableFuture.runAsync(() -> {
                try {
                    newOrderService.getOrder(id);
                } catch (Exception e) {
                    log.warn("影子调用新系统失败: orderId={}", id, e);
                }
            });
        }
        // 主请求走旧系统
        return legacyClient.getOrder(id);
    }
}
```

### 9.3 迁移步骤

```
1. 识别迁移边界
   └── 选择一个独立的、耦合度低的模块开始

2. 部署Facade（API Gateway）
   └── 所有请求先经过Facade，Facade路由到旧系统

3. 构建新服务
   └── 用新架构重写目标模块，部署为独立服务

4. 特性开关切换
   └── 通过特性开关逐步将流量从旧系统切到新系统
   └── 0% → 1% → 10% → 50% → 100%

5. 影子验证
   └── 同时调用新旧系统，对比结果但不影响主流程

6. 完全切换
   └── 确认新系统稳定后，移除旧系统代码

7. 重复1-6
   └── 对下一个模块执行同样的迁移
```

## 10. 架构模式对比与选型

### 10.1 全面对比

| 模式 | 复杂度 | 适用场景 | 典型应用 |
|------|--------|---------|---------|
| 分层架构 | 低 | 中小项目，CRUD为主 | 传统MVC应用 |

| 事件溯源 | 很高 | 需要完整审计、时间旅行 | 金融系统、订单系统 |

```
项目规模？
```

## 11. 面试题速查

### Q1: 分层架构和六边形架构的核心区别？

分层架构按技术职责水平分层（表现层-业务层-持久层），依赖方向从上到下。六边形架构将领域逻辑放在中心，通过端口与外部交互，外部适配器实现端口。六边形架构的可测试性和可替换性更强，但更复杂。

### Q2: CQRS的本质是什么？

CQRS将写操作（Command）和读操作（Query）分离为不同的模型和数据存储。写端保证业务一致性，读端为查询优化。两端通过事件同步。适合读写比例悬殊、读写模型差异大的场景。

### Q3: 事件溯源和传统CRUD有什么区别？

传统CRUD存储当前状态（覆盖更新）。事件溯源存储所有状态变更事件，当前状态通过重放事件得到。事件溯源提供完整审计日志、时间旅行能力，但复杂度高、查询性能需要快照优化。

### Q4: SAGA的编排式和协调式如何选择？

简单流程（2-3步）用编排式（事件驱动去中心化）。复杂流程（多步骤、有条件分支）用协调式（中央协调器集中管理）。编排式耦合低但流程不可见，协调式流程清晰但有单点风险。

### Q5: Sidecar模式解决了什么问题？

将服务治理功能（熔断、限流、追踪、安全等）从业务代码中剥离到Sidecar进程，业务代码只关注业务逻辑。多语言微服务中无需每种语言实现一套治理库。Istio/Linkerd的Service Mesh就是Sidecar模式的典型实现。

### Q6: Strangler Fig模式为什么比"大爆炸"重写更好？

大爆炸重写风险极高——周期长、不能中断服务、新旧需求并行。Strangler Fig渐进式迁移，每次迁移一个模块，通过Facade保持接口不变，可随时回滚。风险可控、价值持续交付。

### Q7: 事件驱动架构的At-Least-Once语义如何处理重复消费？

消费者需要实现**幂等性**——同一事件被处理多次结果一致。常见方案：用事件ID做去重表、数据库唯一约束、Redis SetNX标记已处理事件。

### Q8: 洋葱架构的依赖规则是什么？

依赖方向始终向内：外层依赖内层，内层不依赖外层。Domain层（最内）不依赖任何框架和外部组件。Application层依赖Domain层。Infrastructure层依赖Application层和Domain层的接口定义。

### Q9: 事件溯源中快照的作用是什么？

当聚合根的事件数量过多时，每次重放所有事件性能很差。快照定期保存聚合根的当前状态，加载时先恢复快照再重放快照之后的事件，大幅减少重放事件数量。

### Q10: 架构模式选型的核心原则是什么？

没有银弹——根据业务规模、团队能力、时间约束选型。简单项目不要过度设计（分层架构足够）。复杂项目要权衡一致性、可用性、性能。模式可以组合使用，但每增加一种模式都会增加系统复杂度。
