# 架构模式详解

> 架构模式是软件系统结构设计的经验总结，它定义了系统组件的组织方式、交互规则和约束条件。从经典的分层架构到现代的微服务架构，每种模式都针对特定的问题场景而生。理解各种架构模式的核心思想、适用场景和权衡取舍，是架构师必备的核心能力。本文系统梳理分层架构、六边形架构、洋葱架构、CQRS、事件驱动、事件溯源、SAGA、Sidecar、Service Mesh等主流架构模式，帮助你在架构选型时做出合理决策。

## 1. 分层架构（Layered）

### 1.1 核心思想

分层架构是最经典的架构模式，将系统按职责划分为若干水平层，每层只与相邻层通信。最常见的四层架构：

```
┌──────────────────────────────┐
│       表现层 (Presentation)    │  ← Controller / View
├──────────────────────────────┤
│       业务层 (Business)        │  ← Service / Domain Logic
├──────────────────────────────┤
│       持久层 (Persistence)     │  ← DAO / Repository
├──────────────────────────────┤
│       数据库层 (Database)      │  ← MySQL / Redis
└──────────────────────────────┘
```

### 1.2 层间规则

- **向下依赖**：上层可以调用下层，下层不能调用上层
- **跨层禁止**：表现层不能直接访问数据库层（跳过业务层）
- **同层隔离**：同层组件之间通过接口通信，避免直接依赖实现

### 1.3 代码示例

```java
// ==================== 表现层 ====================
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @Autowired private OrderService orderService;

    @PostMapping
    public Result<OrderVO> createOrder(@RequestBody @Validated OrderCreateDTO dto) {
        OrderVO vo = orderService.createOrder(dto);
        return Result.success(vo);
    }

    @GetMapping("/{id}")
    public Result<OrderVO> getOrder(@PathVariable Long id) {
        return Result.success(orderService.getOrderById(id));
    }
}

// ==================== 业务层 ====================
@Service
@Transactional
public class OrderServiceImpl implements OrderService {

    @Autowired private OrderRepository orderRepository;
    @Autowired private ProductService productService;
    @Autowired private InventoryService inventoryService;

    @Override
    public OrderVO createOrder(OrderCreateDTO dto) {
        // 业务校验
        Product product = productService.getById(dto.getProductId());
        if (product.getStock() < dto.getQuantity()) {
            throw new BusinessException("库存不足");
        }

        // 构建领域对象
        Order order = new Order();
        order.setProductId(dto.getProductId());
        order.setQuantity(dto.getQuantity());
        order.setTotalPrice(product.getPrice().multiply(BigDecimal.valueOf(dto.getQuantity())));
        order.setStatus(OrderStatus.PENDING);

        // 持久化
        orderRepository.save(order);

        // 扣减库存
        inventoryService.deduct(dto.getProductId(), dto.getQuantity());

        return OrderVO.from(order);
    }
}

// ==================== 持久层 ====================
public interface OrderRepository {
    Order save(Order order);
    Order findById(Long id);
    List<Order> findByUserId(Long userId);
}

@Repository
public class OrderRepositoryImpl implements OrderRepository {

    @Autowired private OrderMapper orderMapper;

    @Override
    public Order save(Order order) {
        if (order.getId() == null) {
            orderMapper.insert(order);
        } else {
            orderMapper.update(order);
        }
        return order;
    }

    @Override
    public Order findById(Long id) {
        return orderMapper.selectById(id);
    }

    @Override
    public List<Order> findByUserId(Long userId) {
        return orderMapper.selectByUserId(userId);
    }
}
```

### 1.4 优缺点

| 优点 | 缺点 |
|------|------|
| 结构清晰，易于理解 | 容易产生"流水线"式代码（简单CRUD也走四层） |
| 层间解耦，可独立替换 | 性能开销（每层都可能做数据转换） |
| 团队分工明确 | 业务逻辑容易分散在多层中 |
| 适合大多数企业应用 | 单体架构扩展性有限 |

## 2. 六边形架构（Hexagonal）

### 2.1 核心思想

六边形架构由Alistair Cockburn提出，又称**端口与适配器架构**（Ports and Adapters）。核心思想是将业务逻辑放在中心，通过端口（接口）定义与外界的交互，适配器实现这些端口。业务逻辑不关心外部是HTTP、gRPC还是消息队列。

```
                    ┌─────────┐
     HTTP Adapter   │         │   JPA Adapter
     ──────────────>│  Domain │<──────────────
                    │  Core   │
     gRPC Adapter   │ (Ports) │   Redis Adapter
     ──────────────>│         │<──────────────
                    │         │
     Kafka Adapter  │         │   Elasticsearch
     ──────────────>│         │<──────────────
                    └─────────┘
```

### 2.2 端口与适配器

- **驱动端口（Driving Port / 入站端口）**：外部调用应用核心的接口（如REST Controller调用）
- **被驱动端口（Driven Port / 出站端口）**：应用核心调用外部的接口（如Repository接口）

```java
// ==================== 领域模型 ====================
// 领域对象，不依赖任何框架
public class Order {
    private OrderId id;
    private CustomerId customerId;
    private List<OrderLine> lines;
    private OrderStatus status;
    private Money totalAmount;

    // 领域逻辑封装在实体中
    public void confirm() {
        if (status != OrderStatus.PENDING) {
            throw new DomainException("只有待处理订单才能确认");
        }
        this.status = OrderStatus.CONFIRMED;
    }

    public void addLine(ProductId productId, int quantity, Money price) {
        OrderLine line = new OrderLine(productId, quantity, price);
        this.lines.add(line);
        recalculateTotal();
    }

    private void recalculateTotal() {
        this.totalAmount = lines.stream()
            .map(OrderLine::getSubtotal)
            .reduce(Money.ZERO, Money::add);
    }
}

// ==================== 驱动端口（入站） ====================
// 定义应用核心对外提供的服务接口
public interface OrderUseCase {
    OrderResult createOrder(CreateOrderCommand command);
    void confirmOrder(OrderId orderId);
    OrderResult queryOrder(OrderId orderId);
}

// ==================== 被驱动端口（出站） ====================
// 定义应用核心需要的外部依赖接口
public interface OrderRepository {
    void save(Order order);
    Optional<Order> findById(OrderId id);
}

public interface PaymentGateway {
    PaymentResult charge(OrderId orderId, Money amount, CreditCard card);
}

public interface EventPublisher {
    void publish(OrderEvent event);
}

// ==================== 应用服务（实现驱动端口） ====================
@Service
public class OrderApplicationService implements OrderUseCase {

    private final OrderRepository orderRepository;  // 被驱动端口
    private final PaymentGateway paymentGateway;     // 被驱动端口
    private final EventPublisher eventPublisher;     // 被驱动端口

    public OrderApplicationService(OrderRepository orderRepository,
                                    PaymentGateway paymentGateway,
                                    EventPublisher eventPublisher) {
        this.orderRepository = orderRepository;
        this.paymentGateway = paymentGateway;
        this.eventPublisher = eventPublisher;
    }

    @Override
    public OrderResult createOrder(CreateOrderCommand cmd) {
        Order order = Order.create(cmd.getCustomerId());
        for (OrderItem item : cmd.getItems()) {
            order.addLine(item.getProductId(), item.getQuantity(), item.getPrice());
        }
        orderRepository.save(order);
        eventPublisher.publish(new OrderCreatedEvent(order.getId()));
        return OrderResult.from(order);
    }

    @Override
    public void confirmOrder(OrderId orderId) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new NotFoundException("订单不存在"));
        order.confirm();
        orderRepository.save(order);
        eventPublisher.publish(new OrderConfirmedEvent(order.getId()));
    }
}

// ==================== 适配器（实现端口） ====================
// HTTP适配器（驱动适配器）
@RestController
@RequestMapping("/api/orders")
public class OrderRestAdapter {

    private final OrderUseCase orderUseCase;

    public OrderRestAdapter(OrderUseCase orderUseCase) {
        this.orderUseCase = orderUseCase;
    }

    @PostMapping
    public ResponseEntity<OrderResult> create(@RequestBody CreateOrderRequest req) {
        CreateOrderCommand cmd = CreateOrderCommand.from(req);
        return ResponseEntity.ok(orderUseCase.createOrder(cmd));
    }

    @PostMapping("/{id}/confirm")
    public ResponseEntity<Void> confirm(@PathVariable String id) {
        orderUseCase.confirmOrder(new OrderId(id));
        return ResponseEntity.noContent().build();
    }
}

// JPA适配器（被驱动适配器）
@Repository
public class JpaOrderRepository implements OrderRepository {

    private final OrderJpaRepository jpaRepository;
    private final OrderMapper mapper;

    public JpaOrderRepository(OrderJpaRepository jpaRepository, OrderMapper mapper) {
        this.jpaRepository = jpaRepository;
        this.mapper = mapper;
    }

    @Override
    public void save(Order order) {
        OrderJpaEntity entity = mapper.toJpaEntity(order);
        jpaRepository.save(entity);
    }

    @Override
    public Optional<Order> findById(OrderId id) {
        return jpaRepository.findById(id.getValue())
            .map(mapper::toDomain);
    }
}
```

### 2.3 核心优势

- **业务逻辑与基础设施完全隔离**：替换数据库、消息队列等不影响核心逻辑
- **可测试性极强**：用Mock适配器即可测试所有业务逻辑
- **技术替换灵活**：从REST切换到gRPC只需新增适配器

## 3. 洋葱架构（Onion）

### 3.1 核心思想

洋葱架构由Jeffrey Palermo提出，与六边形架构思想相似，但更强调层次的同心圆结构。依赖方向始终从外向内——外层依赖内层，内层不依赖外层。

```
              ┌─────────────────────────────────┐
              │  Infrastructure (基础设施层)      │
              │  ┌───────────────────────────┐  │
              │  │  Application Services      │  │
              │  │  (应用服务层)               │  │
              │  │  ┌───────────────────────┐│  │
              │  │  │  Domain Services       ││  │
              │  │  │  (领域服务层)           ││  │
              │  │  │  ┌───────────────────┐││  │
              │  │  │  │  Domain Model      │││  │
              │  │  │  │  (领域模型/核心)    │││  │
              │  │  │  └───────────────────┘││  │
              │  │  └───────────────────────┘│  │
              │  └───────────────────────────┘  │
              └─────────────────────────────────┘
   依赖方向：外 ──────> 内（绝不反向）
```

### 3.2 各层职责

| 层 | 职责 | 依赖 |
|----|------|------|
| **领域模型** | 实体、值对象、领域事件，纯粹的业务逻辑 | 无（零依赖） |
| **领域服务** | 跨实体的业务操作 | 领域模型 |
| **应用服务** | 用例编排、事务管理、调用领域服务 | 领域服务+领域模型 |
| **基础设施** | 数据库、消息队列、外部API、UI | 应用服务（实现其接口） |

### 3.3 与六边形架构的区别

| 维度 | 六边形架构 | 洋葱架构 |
|------|-----------|---------|
| 结构隐喻 | 六边形（端口+适配器） | 同心圆（洋葱层） |
| 层次划分 | 核心vs外围，无严格层次 | 明确的同心圆层次 |
| 依赖方向 | 外→内 | 外→内 |
| 接口位置 | 端口在核心边界 | 接口在内层，实现在外层 |
| 关注重点 | 可替换性 | 依赖方向 |

## 4. CQRS命令查询职责分离

### 4.1 核心思想

CQRS（Command Query Responsibility Segregation）将系统分为命令端（写操作）和查询端（读操作），各自有独立的模型和数据存储。传统CRUD中读写共用同一模型，CQRS认为读和写的需求往往不同，分离后可以各自优化。

```
        ┌─────────┐         ┌──────────┐         ┌──────────┐
用户 ──>│ Command │────────>│  Write   │────────>│  Event   │
        │  Side   │         │  Model   │         │  Store   │
        └─────────┘         └──────────┘         └────┬─────┘
                                                       │ 同步事件
                         ┌──────────┐         ┌───────▼─────┐
用户 <───│  Query   │<────────│  Read    │<────────│  Projector │
        │  Side   │         │  Model   │         │            │
        └─────────┘         └──────────┘         └────────────┘
```

### 4.2 代码实现

```java
// ==================== 命令端（写模型） ====================

// Command对象
@Data
public class CreateProductCommand {
    private String name;
    private BigDecimal price;
    private String category;
    private int stock;
}

// Command Handler
@Service
public class ProductCommandHandler {

    @Autowired private ProductRepository productRepository;
    @Autowired private ApplicationEventPublisher eventPublisher;

    @Transactional
    public Long handle(CreateProductCommand cmd) {
        // 写模型使用领域对象进行业务校验
        Product product = Product.create(cmd.getName(), cmd.getPrice(), cmd.getCategory());
        product.setStock(cmd.getStock());

        productRepository.save(product);

        // 发布事件，通知查询端更新
        eventPublisher.publishEvent(new ProductCreatedEvent(
            product.getId(), product.getName(), product.getPrice(),
            product.getCategory(), product.getStock()
        ));

        return product.getId();
    }
}

// 写模型Repository
public interface ProductRepository {
    Product save(Product product);
    Optional<Product> findById(Long id);
}

// ==================== 查询端（读模型） ====================

// 查询DTO（与写模型不同，为查询优化）
@Data
public class ProductQueryDTO {
    private Long id;
    private String name;
    private String price;       // 格式化后的价格字符串
    private String category;
    private String categoryName; // 关联的分类名称
    private int stock;
    private String status;      // 格式化的状态文本
}

// 查询服务（直接查读库，可能是Elasticsearch或专门的查询视图）
@Service
public class ProductQueryService {

    @Autowired private ProductQueryMapper queryMapper;

    public Page<ProductQueryDTO> search(ProductSearchCondition condition) {
        // 查询端可以做复杂的JOIN、聚合等
        List<ProductQueryDTO> list = queryMapper.search(condition);
        int total = queryMapper.count(condition);
        return new Page<>(list, total, condition.getPageNum(), condition.getPageSize());
    }

    public ProductQueryDTO detail(Long id) {
        return queryMapper.findDetailById(id);
    }

    // 统计查询，与写模型完全独立
    public List<CategoryStatDTO> statByCategory() {
        return queryMapper.statByCategory();
    }
}

// ==================== 事件投影器（同步读写模型） ====================

@Component
public class ProductProjector {

    @Autowired private ProductReadRepository readRepository;

    @EventListener
    @Async
    public void on(ProductCreatedEvent event) {
        // 将写模型的事件投影到读模型
        ProductReadModel readModel = new ProductReadModel();
        readModel.setId(event.getProductId());
        readModel.setName(event.getName());
        readModel.setPrice(event.getPrice().toPlainString());
        readModel.setCategory(event.getCategory());
        readModel.setStock(event.getStock());
        readModel.setStatus("在售");

        readRepository.save(readModel);
    }

    @EventListener
    @Async
    public void on(ProductPriceChangedEvent event) {
        readRepository.updatePrice(event.getProductId(), event.getNewPrice().toPlainString());
    }
}
```

### 4.3 适用场景

- 读写比例严重不均衡（读多写少或写多读少）
- 读端需要复杂查询（搜索、聚合、多表JOIN）
- 读写性能要求不同（写需要强一致，读可以最终一致）

## 5. 事件驱动架构（EDA）

### 5.1 核心思想

事件驱动架构中，组件通过生产/消费事件进行通信，而非直接调用。生产者发布事件后不等待消费者处理，实现松耦合。

```
  Producer ──发布事件──> Event Bus / Broker ──投递事件──> Consumer
     │                                                   │
     │  不关心谁消费                                      │  不关心谁生产
     │  不等待响应                                        │  异步处理
```

### 5.2 事件类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **事件通知** | 仅告知发生了什么，不含详细数据 | `OrderCreatedEvent(orderId)` |
| **事件携带状态转移** | 事件中包含完整的变更数据 | `OrderCreatedEvent(orderId, items, total)` |
| **领域事件** | 领域专家关心的业务事件 | `PaymentCompletedEvent(orderId, amount)` |

### 5.3 代码实现（基于Spring事件）

```java
// ==================== 领域事件定义 ====================
@Data
@AllArgsConstructor
public class OrderPlacedEvent {
    private String orderId;
    private String customerId;
    private List<OrderItem> items;
    private BigDecimal totalAmount;
    private LocalDateTime placedAt;
}

// ==================== 事件发布者 ====================
@Service
public class OrderService {

    @Autowired private ApplicationEventPublisher eventPublisher;
    @Autowired private OrderRepository orderRepository;

    @Transactional
    public String placeOrder(OrderRequest request) {
        Order order = createOrder(request);
        orderRepository.save(order);

        // 发布事件，不关心谁消费
        eventPublisher.publishEvent(new OrderPlacedEvent(
            order.getId(), order.getCustomerId(), order.getItems(),
            order.getTotalAmount(), LocalDateTime.now()
        ));

        return order.getId();
    }
}

// ==================== 事件消费者 ====================

// 库存服务消费事件，扣减库存
@Component
public class InventoryEventHandler {

    @Autowired private InventoryService inventoryService;

    @EventListener
    @Async
    public void on(OrderPlacedEvent event) {
        for (OrderItem item : event.getItems()) {
            inventoryService.deduct(item.getProductId(), item.getQuantity());
        }
    }
}

// 通知服务消费事件，发送通知
@Component
public class NotificationEventHandler {

    @Autowired private EmailService emailService;

    @EventListener
    @Async
    public void on(OrderPlacedEvent event) {
        emailService.sendOrderConfirmation(event.getCustomerId(), event.getOrderId());
    }
}

// 积分服务消费事件，增加积分
@Component
public class PointsEventHandler {

    @Autowired private PointsService pointsService;

    @EventListener
    @Async
    public void on(OrderPlacedEvent event) {
        int points = event.getTotalAmount().intValue();
        pointsService.addPoints(event.getCustomerId(), points);
    }
}
```

## 6. 事件溯源（Event Sourcing）

### 6.1 核心思想

传统架构中，系统存储的是对象的当前状态。事件溯源则存储所有改变状态的事件，当前状态通过回放事件得到。事件是不可变的、可追溯的。

```
  传统存储：                    事件溯源存储：

  ┌──────────┐                 ┌──────────────────────────────────┐
  │ Account  │                 │ Event Store                      │
  │ id: 001  │                 │ ┌──────────────────────────────┐ │
  │ balance: │                 │ │ 1. AccountCreated (balance=0)│ │
  │   800    │                 │ │ 2. Deposited (amount=1000)   │ │
  └──────────┘                 │ │ 3. Withdrawn (amount=200)    │ │
                               │ │ 4. Deposited (amount=100)    │ │
                               │ │ 5. Withdrawn (amount=100)    │ │
                               │ └──────────────────────────────┘ │
                               │ 当前状态 = 0+1000-200+100-100 = 800 │
                               └──────────────────────────────────┘
```

### 6.2 代码实现

```java
// ==================== 事件定义 ====================
public abstract class AccountEvent {
    private final String accountId;
    private final LocalDateTime timestamp;

    protected AccountEvent(String accountId) {
        this.accountId = accountId;
        this.timestamp = LocalDateTime.now();
    }
    public String getAccountId() { return accountId; }
    public LocalDateTime getTimestamp() { return timestamp; }
}

@Data
public class AccountCreatedEvent extends AccountEvent {
    private final String owner;

    public AccountCreatedEvent(String accountId, String owner) {
        super(accountId);
        this.owner = owner;
    }
}

@Data
public class MoneyDepositedEvent extends AccountEvent {
    private final BigDecimal amount;

    public MoneyDepositedEvent(String accountId, BigDecimal amount) {
        super(accountId);
        this.amount = amount;
    }
}

@Data
public class MoneyWithdrawnEvent extends AccountEvent {
    private final BigDecimal amount;

    public MoneyWithdrawnEvent(String accountId, BigDecimal amount) {
        super(accountId);
        this.amount = amount;
    }
}

// ==================== 聚合根（通过事件重建状态） ====================
public class Account {
    private String id;
    private String owner;
    private BigDecimal balance;
    private long version;

    // 从事件历史重建状态
    public static Account fromEvents(List<AccountEvent> events) {
        if (events.isEmpty()) {
            throw new IllegalStateException("无事件历史");
        }
        Account account = new Account();
        for (AccountEvent event : events) {
            account.apply(event);
        }
        account.version = events.size();
        return account;
    }

    // 业务命令：存款
    public MoneyDepositedEvent deposit(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new DomainException("存款金额必须大于0");
        }
        return new MoneyDepositedEvent(id, amount);
    }

    // 业务命令：取款
    public MoneyWithdrawnEvent withdraw(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new DomainException("取款金额必须大于0");
        }
        if (balance.compareTo(amount) < 0) {
            throw new DomainException("余额不足");
        }
        return new MoneyWithdrawnEvent(id, amount);
    }

    // 事件应用器（修改内部状态）
    void apply(AccountEvent event) {
        if (event instanceof AccountCreatedEvent e) {
            this.id = e.getAccountId();
            this.owner = e.getOwner();
            this.balance = BigDecimal.ZERO;
        } else if (event instanceof MoneyDepositedEvent e) {
            this.balance = this.balance.add(e.getAmount());
        } else if (event instanceof MoneyWithdrawnEvent e) {
            this.balance = this.balance.subtract(e.getAmount());
        }
    }
}

// ==================== 事件存储 ====================
@Repository
public class EventStoreRepository {

    @Autowired private JdbcTemplate jdbc;

    public void append(String aggregateId, List<AccountEvent> events, long expectedVersion) {
        // 乐观锁检查
        Long currentVersion = jdbc.queryForObject(
            "SELECT COUNT(*) FROM event_store WHERE aggregate_id = ?",
            Long.class, aggregateId);
        if (currentVersion != null && currentVersion != expectedVersion) {
            throw new ConcurrentModificationException("并发冲突");
        }

        for (AccountEvent event : events) {
            jdbc.update(
                "INSERT INTO event_store (aggregate_id, event_type, event_data, version) VALUES (?, ?, ?, ?)",
                aggregateId,
                event.getClass().getSimpleName(),
                toJson(event),
                ++expectedVersion
            );
        }
    }

    public List<AccountEvent> load(String aggregateId) {
        List<String> jsonEvents = jdbc.queryForList(
            "SELECT event_data FROM event_store WHERE aggregate_id = ? ORDER BY version",
            String.class, aggregateId);

        return jsonEvents.stream()
            .map(this::fromJson)
            .collect(Collectors.toList());
    }
}

// ==================== 应用服务 ====================
@Service
public class AccountApplicationService {

    @Autowired private EventStoreRepository eventStore;

    public void deposit(String accountId, BigDecimal amount) {
        // 1. 加载事件历史，重建当前状态
        List<AccountEvent> history = eventStore.load(accountId);
        Account account = Account.fromEvents(history);

        // 2. 执行业务命令，生成新事件
        MoneyDepositedEvent newEvent = account.deposit(amount);

        // 3. 应用新事件
        account.apply(newEvent);

        // 4. 追加事件到Event Store
        eventStore.append(accountId, List.of(newEvent), account.getVersion());
    }
}
```

### 6.3 事件溯源+快照优化

当事件数量巨大时，每次回放所有事件效率低下。可以定期做快照：

```java
// 每100个事件做一次快照
public class AccountSnapshot {
    private String aggregateId;
    private long version;
    private BigDecimal balance;
    private String owner;
    private LocalDateTime snapshotTime;
}

// 加载时优先使用快照
public Account load(String aggregateId) {
    // 先查快照
    Optional<AccountSnapshot> snapshot = snapshotStore.findLatest(accountId);
    Account account;
    long startVersion;

    if (snapshot.isPresent()) {
        // 从快照恢复
        account = Account.fromSnapshot(snapshot.get());
        startVersion = snapshot.get().getVersion();
    } else {
        // 无快照，从第一个事件开始
        account = new Account();
        startVersion = 0;
    }

    // 只回放快照之后的事件
    List<AccountEvent> events = eventStore.loadAfterVersion(accountId, startVersion);
    for (AccountEvent event : events) {
        account.apply(event);
    }
    return account;
}
```

## 7. SAGA模式

### 7.1 核心思想

SAGA是管理分布式事务的模式。它将一个长事务拆分为一系列本地事务，每个本地事务都有对应的补偿操作。如果某个步骤失败，通过执行已成功步骤的补偿操作来回滚。

```
  正向流程：
  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
  │ 创建订单 │───>│ 扣减库存 │───>│ 扣减余额 │───>│ 发货    │
  │ T1     │    │ T2     │    │ T3     │    │ T4     │
  └────────┘    └────────┘    └────────┘    └────────┘
                     │
                     │ T3失败
                     ▼
  补偿流程（逆序）：
  ┌────────┐    ┌────────┐
  │ 恢复库存 │<───│ 取消订单 │
  │ C2     │    │ C1     │
  └────────┘    └────────┘
```

### 7.2 编排式SAGA（Orchestration）

中央协调器负责编排所有步骤：

```java
// SAGA编排器
@Service
public class OrderSagaOrchestrator {

    @Autowired private OrderService orderService;
    @Autowired private InventoryService inventoryService;
    @Autowired private PaymentService paymentService;
    @Autowired private ShippingService shippingService;

    public SagaResult execute(CreateOrderRequest request) {
        SagaContext context = new SagaContext();
        List< CompensationAction> compensations = new ArrayList<>();

        try {
            // Step 1: 创建订单
            Order order = orderService.createOrder(request);
            context.setOrder(order);
            compensations.add(() -> orderService.cancelOrder(order.getId()));

            // Step 2: 扣减库存
            inventoryService.deduct(request.getProductId(), request.getQuantity());
            compensations.add(() -> inventoryService.restore(request.getProductId(), request.getQuantity()));

            // Step 3: 扣减余额
            paymentService.charge(request.getCustomerId(), order.getTotalAmount());
            compensations.add(() -> paymentService.refund(request.getCustomerId(), order.getTotalAmount()));

            // Step 4: 发货
            shippingService.ship(order.getId());
            // 发货通常不可逆，不做补偿

            return SagaResult.success(order.getId());

        } catch (Exception e) {
            // 补偿：逆序执行所有补偿操作
            Collections.reverse(compensations);
            for (CompensationAction action : compensations) {
                try {
                    action.compensate();
                } catch (Exception ce) {
                    log.error("补偿操作失败", ce);
                    // 记录失败，后续由定时任务重试
                }
            }
            return SagaResult.failure(e.getMessage());
        }
    }

    @FunctionalInterface
    interface CompensationAction {
        void compensate() throws Exception;
    }
}
```

### 7.3 协同式SAGA（Choreography）

各服务通过事件协同，无中央协调器：

```java
// 订单服务
@Service
public class OrderService {

    @Transactional
    public Order createOrder(CreateOrderRequest request) {
        Order order = buildOrder(request);
        order.setStatus(OrderStatus.PENDING);
        orderRepository.save(order);
        // 发布事件，触发库存服务
        eventPublisher.publish(new OrderCreatedEvent(order.getId(), order.getProductId(), order.getQuantity()));
        return order;
    }

    // 监听支付失败事件，执行补偿
    @EventListener
    @Async
    @Transactional
    public void onPaymentFailed(PaymentFailedEvent event) {
        Order order = orderRepository.findById(event.getOrderId()).orElseThrow();
        if (order.getStatus() == OrderStatus.PENDING) {
            order.setStatus(OrderStatus.CANCELLED);
            order.setCancelReason("支付失败: " + event.getReason());
            orderRepository.save(order);
            // 发布取消事件，触发库存恢复
            eventPublisher.publish(new OrderCancelledEvent(order.getId(), order.getProductId(), order.getQuantity()));
        }
    }
}

// 库存服务
@Service
public class InventoryService {

    @EventListener
    @Async
    @Transactional
    public void onOrderCreated(OrderCreatedEvent event) {
        inventoryService.deduct(event.getProductId(), event.getQuantity());
        // 发布库存扣减成功事件
        eventPublisher.publish(new InventoryDeductedEvent(event.getOrderId()));
    }

    // 补偿：监听订单取消事件，恢复库存
    @EventListener
    @Async
    @Transactional
    public void onOrderCancelled(OrderCancelledEvent event) {
        inventoryService.restore(event.getProductId(), event.getQuantity());
    }
}

// 支付服务
@Service
public class PaymentService {

    @EventListener
    @Async
    @Transactional
    public void onInventoryDeducted(InventoryDeductedEvent event) {
        try {
            paymentService.charge(event.getOrderId());
            eventPublisher.publish(new PaymentSucceededEvent(event.getOrderId()));
        } catch (PaymentException e) {
            eventPublisher.publish(new PaymentFailedEvent(event.getOrderId(), e.getMessage()));
        }
    }
}
```

### 7.4 两种方式对比

| 维度 | 编排式 | 协同式 |
|------|--------|--------|
| 协调方式 | 中央协调器 | 事件驱动 |
| 耦合度 | 中（协调器知道所有服务） | 低（服务间无直接依赖） |
| 可观测性 | 强（集中可见） | 弱（分散在各服务） |
| 复杂度 | 集中在协调器 | 分散在各服务 |
| 适用场景 | 步骤多、逻辑复杂 | 步骤少、逻辑简单 |

## 9. Sidecar模式

### 9.1 核心思想

Sidecar模式将辅助功能（日志收集、监控、熔断、认证等）从主应用中剥离，部署在独立的辅助进程中，与主应用共生死。主应用只关注业务逻辑，基础设施能力由Sidecar提供。

```
  ┌─────────────────────────────┐
  │          Pod / VM           │
  │  ┌──────────┐ ┌──────────┐ │
  │  │  Main    │ │ Sidecar  │ │
  │  │  App     │<-->│ Proxy  │ │
  │  │(业务逻辑) │ │(基础设施) │ │
  │  └──────────┘ └──────────┘ │
  │       │            │       │
  │       │     共享网络命名空间  │
  │       │            │       │
  └───────┼────────────┼───────┘
          │            │
          ▼            ▼
     ┌─────────────────────┐
     │    External Services  │
     └─────────────────────┘
```

### 9.2 Sidecar典型应用

| 应用场景 | 说明 | 代表实现 |
|----------|------|----------|
| 服务代理 | 拦截入站/出站流量，实现路由、负载均衡 | Envoy、Nginx |
| 安全认证 | TLS终止、JWT验证、mTLS | Istio Citadel |
| 可观测性 | 指标采集、链路追踪、日志收集 | Jaeger、Prometheus |
| 熔断限流 | 出站请求的熔断、限流、重试 | Envoy、Resilience4j |
| 配置管理 | 动态配置下发 | Consul、Spring Cloud Config |
