### 1.2 各阶段对比

| 阶段 | 部署方式 | 通信方式 | 数据一致性 | 运维成本 | 适用规模 |
|------|----------|----------|------------|----------|----------|
| 单体 | 单包部署 | 方法调用 | 本地事务 | 低 | < 50人团队 |
| 垂直拆分 | 多包部署 | HTTP/RPC | 跨库查询难 | 中 | 50-200人 |

| 平台化 | 云原生 | 多协议 | 最终一致 | 极高 | 超大规模 |

### 2.2 单体架构的核心问题

```
1. 代码膨胀
   - 一个工程百万行代码，编译时间 > 10分钟
   - 新人上手困难，修改一处可能影响全局

2. 部署效率低
   - 一个小改动需要全量部署
   - 部署窗口受限，影响所有业务

3. 技术栈锁定
   - 整个系统统一技术栈，无法按场景选择
   - 想引入新框架/语言需要整体重构

4. 扩展性差
   - 无法按需扩展某个模块
   - 热点模块拖垮整个系统

5. 故障隔离弱
   - 一个模块的OOM导致整个应用宕机
   - 一个模块的线程耗尽影响所有请求

6. 团队协作困难
   - 多团队在同一代码库开发
   - 合并冲突频繁，发布互相阻塞
```

### 2.3 何时应该拆分微服务

```
拆分信号检查清单：

✅ 代码量 > 50万行，编译超过5分钟
✅ 团队 > 5个，频繁合并冲突
✅ 不同模块的技术需求差异大
✅ 需要独立伸缩某些模块
✅ 故障需要隔离到特定业务域
✅ 需要独立部署和发布节奏

不拆的理由：

❌ 团队 < 10人，微服务治理成本 > 收益
❌ 业务边界不清晰，强耦合严重
❌ 没有DevOps基础设施支撑
❌ 没有完善的监控和链路追踪
```

## 三、微服务拆分策略

### 3.1 按业务域拆分（DDD方法）

```java
// DDD 领域驱动设计：通过领域模型识别限界上下文（Bounded Context）
// 每个限界上下文对应一个微服务

// 示例：电商系统的领域划分
// 1. 用户域（User Context）
//    - 用户注册、登录、个人信息、权限管理
// 2. 商品域（Product Context）
//    - 商品发布、分类、库存、搜索
// 3. 订单域（Order Context）
//    - 下单、支付、取消、退款
// 4. 营销域（Marketing Context）
//    - 优惠券、促销活动、积分
// 5. 物流域（Logistics Context）
//    - 发货、配送、签收

// 领域服务接口定义示例
public interface OrderService {

    /**
     * 创建订单
     * 跨域协作：调用商品域扣减库存，调用营销域使用优惠券
     */
    OrderDTO createOrder(CreateOrderCommand command);

    /**
     * 支付订单
     * 跨域协作：调用支付网关，通知物流域
     */
    PaymentResult payOrder(String orderId, PaymentRequest request);

    /**
     * 取消订单
     * 跨域协作：通知商品域回滚库存，通知营销域退回优惠券
     */
    void cancelOrder(String orderId, String reason);
}
```

### 3.2 拆分原则

```
微服务拆分六大原则：

1. 单一职责（SRP）
   - 每个服务专注于一个业务域
   - 一个服务不应承担多个业务能力

2. 高内聚低耦合
   - 服务内功能高度相关
   - 服务间通过API通信，避免共享数据库

3. 限界上下文
   - 每个服务有清晰的领域边界
   - 跨边界通过API或事件通信

4. 数据自主
   - 每个服务拥有自己的数据库
   - 不允许跨服务直接访问数据库

5. 独立部署
   - 服务可独立构建、部署、升级
   - 不依赖其他服务的发布周期

6. 渐进拆分
   - 优先拆分边界清晰的模块
   - 边拆分边验证，避免大爆炸式重构
```

### 3.3 拆分粒度把控

```
拆分粒度参考维度：

┌──────────────┬──────────────────┬───────────────────┐
│ 维度          │ 偏粗（合并）       │ 偏细（拆分）        │
├──────────────┼──────────────────┼───────────────────┤
│ 服务团队      │ > 8人共享一个服务   │ 1人维护多个服务     │
│ 代码量        │ > 10万行           │ < 5000行           │
│ 接口数        │ > 50个             │ < 5个              │
│ 数据库表      │ > 30张             │ 1张               │
│ 部署频率      │ 各模块发布节奏一致   │ 每个服务每天部署    │
└──────────────┴──────────────────┴───────────────────┘

经验法则：
- 一个服务由一个小团队（3-5人）负责
- 一个服务对应2-5张核心表
- 服务间调用链路不超过3层
- 能用包（module）解决的就不要拆服务
```

### 3.4 数据库拆分策略

```java
// 数据库拆分的三种模式

// 模式1：每服务独立数据库（推荐）
// Order Service → order_db
// Product Service → product_db
// User Service → user_db

// 模式2：共享数据库实例，独立Schema
// 同一个MySQL实例，不同Database

// 模式3：共享表（过渡阶段）
// 逐步从共享表迁移到独立库

// 跨服务数据查询的处理策略：

// 策略1：API组合查询（适合简单场景）
@RestController
public class OrderDetailController {

    @Autowired
    private OrderService orderService;

    @Autowired
    private ProductService productApiService; // Feign调用

    @Autowired
    private UserService userApiService;       // Feign调用

    @GetMapping("/order/{id}/detail")
    public OrderDetailVO getOrderDetail(@PathVariable String id) {
        OrderDTO order = orderService.getById(id);

        // 并行调用其他服务获取关联数据
        CompletableFuture<ProductDTO> productFuture =
            CompletableFuture.supplyAsync(() -> productApiService.getById(order.getProductId()));
        CompletableFuture<UserDTO> userFuture =
            CompletableFuture.supplyAsync(() -> userApiService.getById(order.getUserId()));

        CompletableFuture.allOf(productFuture, userFuture).join();

        return OrderDetailVO.builder()
            .order(order)
            .product(productFuture.join())
            .user(userFuture.join())
            .build();
    }
}

// 策略2：CQRS + 数据冗余（适合复杂查询场景）
// 将其他服务的数据同步到本服务的查询库（如ES）
// 通过事件驱动保持最终一致性

// 策略3：数据联邦（适合BI报表场景）
// 通过Presto/Trino等联邦查询引擎跨库查询
```

## 五、服务治理核心机制

### 5.1 负载均衡

```java
// Spring Cloud LoadBalancer 自定义负载策略

public class WeightedLoadBalancer implements ReactorServiceInstanceLoadBalancer {

    private final String serviceId;
    private final ObjectProvider<ServiceInstanceListSupplier> supplierProvider;
    private final AtomicInteger position;

    public WeightedLoadBalancer(String serviceId,
                                 ObjectProvider<ServiceInstanceListSupplier> supplierProvider) {
        this.serviceId = serviceId;
        this.supplierProvider = supplierProvider;
        this.position = new AtomicInteger(new Random().nextInt(1000));
    }

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        return supplierProvider.getIfAvailable().get(request)
            .next()
            .map(instances -> {
                if (instances.isEmpty()) {
                    return new EmptyResponse();
                }
                // 按权重选择
                ServiceInstance selected = selectByWeight(instances);
                return new DefaultResponse(selected);
            });
    }

    private ServiceInstance selectByWeight(List<ServiceInstance> instances) {
        // 从元数据中读取权重
        int totalWeight = instances.stream()
            .mapToInt(i -> {
                String weight = i.getMetadata().get("weight");
                return weight != null ? Integer.parseInt(weight) : 1;
            })
            .sum();

        int randomWeight = new Random().nextInt(totalWeight);
        int currentWeight = 0;

        for (ServiceInstance instance : instances) {
            String weightStr = instance.getMetadata().get("weight");
            int weight = weightStr != null ? Integer.parseInt(weightStr) : 1;
            currentWeight += weight;
            if (randomWeight < currentWeight) {
                return instance;
            }
        }
        return instances.get(0);
    }
}
```

### 5.2 熔断与限流（Sentinel）

```java
// Sentinel 熔断降级配置

// 1. 定义熔断规则
@PostConstruct
public void initDegradeRules() {
    List<DegradeRule> rules = new ArrayList<>();

    // RT熔断：慢调用比例熔断
    DegradeRule rtRule = new DegradeRule("createOrder")
        .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())
        .setCount(500)           // 慢调用阈值：500ms
        .setSlowRatioThreshold(0.5)  // 慢调用比例阈值：50%
        .setMinRequestAmount(5)  // 最小请求数
        .setStatIntervalMs(10000) // 统计窗口：10s
        .setTimeWindow(10);       // 熔断时长：10s
    rules.add(rtRule);

    // 异常比例熔断
    DegradeRule exceptionRule = new DegradeRule("createOrder")
        .setGrade(CircuitBreakerStrategy.EXCEPTION_RATIO.getType())
        .setCount(0.5)           // 异常比例阈值：50%
        .setMinRequestAmount(5)
        .setStatIntervalMs(10000)
        .setTimeWindow(10);
    rules.add(exceptionRule);

    DegradeRuleManager.loadRules(rules);
}

// 2. 限流规则
@PostConstruct
public void initFlowRules() {
    List<FlowRule> rules = new ArrayList<>();

    FlowRule rule = new FlowRule("createOrder");
    rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
    rule.setCount(1000);  // QPS限流：1000
    rule.setLimitApp("default");
    rule.setStrategy(RuleConstant.STRATEGY_DIRECT);
    // 匀速排队模式
    rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_RATE_LIMITER);
    rule.setMaxQueueingTimeMs(2000);  // 排队超时：2s
    rules.add(rule);

    FlowRuleManager.loadRules(rules);
}

// 3. 使用注解保护方法
@SentinelResource(
    value = "createOrder",
    blockHandler = "createOrderBlockHandler",
    fallback = "createOrderFallback"
)
public OrderDTO createOrder(CreateOrderRequest request) {
    // 业务逻辑
    return orderRepository.save(buildOrder(request));
}

// 限流后的处理
public OrderDTO createOrderBlockHandler(CreateOrderRequest request, BlockException ex) {
    throw new ServiceException("系统繁忙，请稍后重试");
}

// 降级后的处理
public OrderDTO createOrderFallback(CreateOrderRequest request, Throwable e) {
    log.error("创建订单降级", e);
    // 返回降级数据或走备用流程

        .status("PENDING")
        .message("系统处理中，请稍后查询")
        .build();
}
```

## 六、API网关设计

### 6.1 网关核心职责

```
┌─────────────────────────────────────────────────────┐
│                  API Gateway 职责                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 路由转发   ──── 根据路径/Header转发到目标服务       │
│  2. 负载均衡   ──── 多实例间负载均衡                    │
│  3. 认证鉴权   ──── JWT/OAuth2统一鉴权                 │
│  4. 限流熔断   ──── 全局限流，保护后端服务              │
```

```java
public class RequestLogFilter implements GlobalFilter, Ordered {

    private static final Logger log = LoggerFactory.getLogger(RequestLogFilter.class);

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        long startTime = System.currentTimeMillis();
        ServerHttpRequest request = exchange.getRequest();

        log.info("[Gateway] {} {} traceId={} from={}",
            request.getMethod(),
            request.getPath(),
            traceId,
            request.getRemoteAddress()
        );

        return chain.filter(exchange).doFinally(signal -> {
            long duration = System.currentTimeMillis() - startTime;
            HttpStatus status = exchange.getResponse().getStatusCode();

            log.info("[Gateway] {} {} status={} duration={}ms traceId={}",
                request.getMethod(),
                request.getPath(),
                status,
                duration,
                traceId
            );
        });
    }

    @Override
    public int getOrder() {
        return Ordered.HIGHEST_PRECEDENCE;
    }
}

// 鉴权过滤器
@Component
public class AuthFilter implements GlobalFilter, Ordered {

    private static final Set<String> WHITE_LIST = Set.of(
        "/api/auth/login",
        "/api/auth/register",
        "/api/health"
    );

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().value();

        // 白名单放行
        if (WHITE_LIST.stream().anyMatch(path::startsWith)) {
            return chain.filter(exchange);
        }

        // 获取Token
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (token == null || !token.startsWith("Bearer ")) {
            return unauthorized(exchange, "缺少认证Token");
        }

        try {
            // 验证JWT
            Claims claims = JwtUtil.parseToken(token.substring(7));
            // 将用户信息传递给下游服务
            ServerHttpRequest mutatedRequest = exchange.getRequest().mutate()
                .header("X-User-Id", claims.getSubject())
                .header("X-User-Role", claims.get("role", String.class))
                .build();
            return chain.filter(exchange.mutate().request(mutatedRequest).build());
        } catch (Exception e) {
            return unauthorized(exchange, "Token无效或已过期");
        }
    }

    private Mono<Void> unauthorized(ServerWebExchange exchange, String message) {
        exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
        exchange.getResponse().getHeaders().setContentType(MediaType.APPLICATION_JSON);
        String body = "{\"code\":401,\"message\":\"" + message + "\"}";
        DataBuffer buffer = exchange.getResponse().bufferFactory()
            .wrap(body.getBytes(StandardCharsets.UTF_8));
        return exchange.getResponse().writeWith(Mono.just(buffer));
    }

    @Override
    public int getOrder() {
        return Ordered.HIGHEST_PRECEDENCE + 1;
    }
}
```

### 10.2 数据迁移策略

```java
// 双写模式：新老数据源同时写入，保证数据一致

@Service
public class OrderMigrationService {

    @Autowired
    private OrderRepository legacyRepository;  // 单体数据库

    @Autowired
    private OrderMongoRepository newRepository; // 新微服务数据库

    /**
     * 双写：同时写入新老数据源
     */
    @Transactional
    public void createOrderWithDualWrite(Order order) {
        // 1. 写入旧库
        legacyRepository.save(order);

        // 2. 写入新库
        try {
            newRepository.save(order);
        } catch (Exception e) {
            // 记录补偿日志，异步重试
            compensationLogService.record("DUAL_WRITE_FAIL",
                "createOrder", order.getId(), e.getMessage());
        }
    }

    /**
     * 数据校验：定期比对新老库数据一致性
     */
    @Scheduled(cron = "0 0 2 * * ?")
    public void verifyDataConsistency() {
        int pageSize = 1000;
        int pageNum = 0;

        while (true) {
            List<Order> legacyOrders = legacyRepository.findAll(pageNum, pageSize);
            if (legacyOrders.isEmpty()) break;

            List<String> ids = legacyOrders.stream()
                .map(Order::getId)
                .collect(Collectors.toList());

            List<Order> newOrders = newRepository.findAllById(ids);
            Map<String, Order> newOrderMap = newOrders.stream()
                .collect(Collectors.toMap(Order::getId, o -> o));

            for (Order legacyOrder : legacyOrders) {
                Order newOrder = newOrderMap.get(legacyOrder.getId());
                if (newOrder == null || !dataEquals(legacyOrder, newOrder)) {
                    // 数据不一致，记录并修复
                    log.warn("数据不一致: legacy={}, new={}",
                        legacyOrder, newOrder);
                    fixInconsistentData(legacyOrder);
                }
            }
            pageNum++;
        }
    }
}
```

## 十一、面试题速查

### Q1：单体架构什么时候该拆分为微服务？

> 当出现以下信号时考虑拆分：代码量超过50万行、团队超过5个且合并冲突频繁、不同模块需要独立伸缩、故障需要隔离到特定域。但要注意：团队少于10人、没有DevOps基础设施、业务边界不清晰时不要拆，治理成本会大于收益。

### Q2：微服务拆分有哪些原则？

> 六大原则：单一职责（每服务专注一个业务域）、高内聚低耦合（服务间通过API通信）、限界上下文（清晰领域边界）、数据自主（每服务独立数据库）、独立部署（不依赖其他服务发布周期）、渐进拆分（优先拆边界清晰的模块，边拆边验证）。

### Q3：服务注册中心的AP和CP如何选择？

> 注册中心通常选AP（如Nacos AP模式、Eureka）。因为注册中心的核心目标是可用性——即使注册中心节点故障，服务间仍能通过本地缓存继续调用。CP模式（如Consul）在Leader选举期间不可用，影响服务发现。Nacos支持AP/CP切换，默认AP。

### Q4：微服务如何处理跨服务的数据一致性？

> 三种方案：1）API组合查询——简单场景并行调用多个服务获取数据；2）CQRS+数据冗余——通过事件驱动将数据同步到查询库，适合复杂查询；3）分布式事务——Seata AT/TCC模式保证写操作一致性，或用Saga模式实现最终一致。核心原则是优先最终一致性，避免强一致带来的性能损耗。

### Q5：服务网格和传统微服务SDK有什么区别？

> 传统SDK模式（如Spring Cloud）将治理能力（负载均衡、熔断、链路追踪）以SDK形式嵌入业务代码，与语言绑定，升级需要重新部署。服务网格（如Istio）将治理能力下沉到Sidecar代理（Envoy），业务代码零侵入，支持多语言，治理策略通过配置动态下发。代价是增加了一层网络代理的性能开销。

### Q6：从单体迁移到微服务的策略是什么？

> 采用**绞杀者模式**（Strangler Fig Pattern）。在单体前部署API网关，逐步将功能从单体提取为独立微服务，网关按路由将流量分发到新服务或单体。每提取一个服务，就完成对应的数据迁移（双写+校验）。整个过程渐进式进行，避免大爆炸式重构。单体功能逐渐被"绞杀"直至完全替换。

### Q7：微服务架构下如何做灰度发布？

> 三种方式：1）网关层路由——在API网关按Header/Cookie/权重路由到灰度实例；2）注册中心元数据——服务实例携带版本元数据，消费者按版本路由；3）服务网格——Istio VirtualService按权重/Header精确控制流量比例。推荐网关层做粗粒度灰度，服务网格做细粒度灰度。

### Q8：微服务监控需要关注哪些核心指标？

> 四大维度（USE/RED方法）：1）资源层——CPU、内存、磁盘IO、网络IO；2）应用层——QPS、RT（P50/P99）、错误率、并发数；3）中间件层——数据库连接池、Redis响应时间、MQ积压量；4）业务层——订单成功率、支付成功率等业务指标。配合链路追踪（SkyWalking/Jaeger）定位跨服务调用瓶颈。
