## 1. 中台概述

### 什么是中台

```
中台 = 企业级能力复用平台

核心思想：将共性能力抽取出来，统一建设，供多个前台业务复用。

前台：面向用户的业务系统（商城、APP、小程序）
中台：共性能力中心（用户中心、商品中心、订单中心）
后台：基础设施（数据库、中间件、云资源）

┌─────────────────────────────────────┐
│            前台业务                   │
│  ┌──────┐ ┌──────┐ ┌──────┐        │
│  │ 商城  │ │APP   │ │小程序 │        │
│  └──┬───┘ └──┬───┘ └──┬───┘        │
├─────┼────────┼────────┼────────────┤
│     │   中台能力复用    │            │
│  ┌──┴────────┴────────┴───┐        │
│  │ 用户 商品 订单 支付 营销 │        │
│  └────────────────────────┘        │
├─────────────────────────────────────┤
│            后台基础设施               │
└─────────────────────────────────────┘
```

### 中台类型

| 类型 | 定位 | 核心能力 | 典型代表 |
|------|------|---------|---------|
| 业务中台 | 业务能力复用 | 用户/商品/订单/支付 | 阿里淘宝 |
| 数据中台 | 数据能力复用 | 数仓/BI/用户画像 | 字节跳动 |
| 技术中台 | 技术能力复用 | 消息/缓存/搜索/权限 | 美团 |
| AI中台 | AI能力复用 | 模型训练/推理/特征 | 蚂蚁金服 |

### 为什么需要中台

```
无中台的问题：
  - 各业务线重复造轮子（A部门和B部门各写一套用户系统）
  - 数据孤岛（用户数据分散在不同系统）
  - 研发效率低（新业务从零开始）
  - 标准不统一（各系统接口/数据格式不同）

中台的价值：
  - 能力复用：一次建设，多次使用
  - 数据统一：打通数据孤岛
  - 快速试错：新业务基于中台快速搭建
  - 标准化：统一接口规范和数据标准
```

### 能力中心划分

```
┌──────────────────────────────────────────┐
│                业务中台                    │
│                                          │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │
│  │用户中心│ │商品中心│ │订单中心│ │支付中心│   │
│  │      │ │      │ │      │ │      │   │
│  │注册登录│ │商品管理│ │下单   │ │支付   │   │
│  │用户信息│ │库存管理│ │取消   │ │退款   │   │
│  │权限管理│ │价格管理│ │查询   │ │对账   │   │
│  │会员体系│ │分类管理│ │状态   │ │渠道   │   │
│  └──────┘ └──────┘ └──────┘ └──────┘   │
│                                          │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │
│  │营销中心│ │物流中心│ │评价中心│ │搜索中心│   │
│  │优惠券 │ │发货   │ │商品评价│ │商品搜索│   │
│  │活动   │ │物流跟踪│ │商家评分│ │推荐   │   │
│  └──────┘ └──────┘ └──────┘ └──────┘   │
└──────────────────────────────────────────┘
```

### 能力中心设计原则

```
1. 高内聚：一个中心负责一个业务域
2. 松耦合：中心间通过API/事件通信
3. 数据自治：每个中心拥有独立数据库
4. 接口标准化：统一的API规范（REST/Dubbo）
5. 独立部署：各中心可独立发布
```

### 用户中心示例

```java
// 用户中心：统一用户管理
@Service
public class UserCenterService {
    
    @DubboService  // 对外提供Dubbo接口
    public UserDTO getUser(Long userId) {
        return userMapper.selectById(userId).toDTO();
    }
    
    @DubboService
    public UserDTO getUserByPhone(String phone) {
        return userMapper.selectByPhone(phone).toDTO();
    }
    
    @DubboService
    public boolean checkPermission(Long userId, String resource) {
        return permissionService.check(userId, resource);
    }
    
    // 注册成功后发布事件
    public User register(RegisterRequest request) {
        User user = doRegister(request);
        eventPublisher.publish(new UserRegisteredEvent(user.getId()));
        // 营销中心消费事件 → 发新手券
        // 评价中心消费事件 → 初始化用户评分
        return user;
    }
}
```

## 3. 数据中台

### 数据中台架构

```
┌──────────────────────────────────────────┐
│              数据应用层                    │
│  BI报表 用户画像 推荐系统 风控系统         │
├──────────────────────────────────────────┤
│              数据服务层                    │
│  数据API 指标引擎 标签服务                 │
├──────────────────────────────────────────┤
│              数据资产层                    │
│  数仓(ODS→DWD→DWS→ADS) 元数据管理         │
├──────────────────────────────────────────┤
│              数据采集层                    │
│  Canal( binlog) Flume(日志) API(业务)     │
├──────────────────────────────────────────┤
│              数据源                        │
│  MySQL Redis MongoDB 业务日志              │
└──────────────────────────────────────────┘
```

### 数仓分层

```sql
-- ODS层：原始数据（贴源层）
-- 直接从业务库同步，保持原样
CREATE TABLE ods_orders (
    id BIGINT, user_id BIGINT, amount DECIMAL,
    create_time DATETIME, ...
);

-- DWD层：明细数据（清洗+标准化）
-- 清洗脏数据、统一格式
CREATE TABLE dwd_orders (
    order_id BIGINT, user_id BIGINT, 
    amount DECIMAL, pay_type STRING,
    dt DATE  -- 分区字段
);

-- DWS层：汇总数据（轻度聚合）
-- 按天/周/月聚合
CREATE TABLE dws_user_order_daily (
    user_id BIGINT, order_count INT, 
    total_amount DECIMAL, dt DATE
);

-- ADS层：应用数据（面向业务）
-- 直接供BI/报表使用
CREATE TABLE ads_user_value (
    user_id BIGINT, user_level STRING,
    total_consumption DECIMAL, last_order_date DATE
);
```

### 用户画像

```java
// 标签服务：基于数据中台构建用户画像
@Service
public class UserProfileService {
    
    // 查询用户标签
    public UserProfile getProfile(Long userId) {
        return UserProfile.builder()
            .basicInfo(getBasicInfo(userId))       // 基本信息
            .consumptionLevel(getConsumeLevel(userId))  // 消费等级
            .interestTags(getInterests(userId))     // 兴趣标签
            .activityLevel(getActivityLevel(userId)) // 活跃度
            .riskScore(getRiskScore(userId))        // 风险评分
            .build();
    }
    
    // 标签存储在HBase/Redis中，支持实时查询
}
```

### 技术中台的价值

```
1. 统一技术栈：全公司使用统一的中间件和框架
2. 运维标准化：统一监控、日志、部署
3. 开发效率：业务团队专注业务逻辑
4. 故障隔离：技术中台故障不影响业务中台（降级）
5. 资源利用率：统一资源池，按需分配
```

## 5. 中台架构实战

### 电商平台中台架构

```
┌──────────┐ ┌──────────┐ ┌──────────┐
│  商城前台  │ │  移动APP  │ │  小程序   │ ← 前台
└────┬─────┘ └────┬─────┘ └────┬─────┘
     │             │            │
┌────┴─────────────┴────────────┴─────┐
│            API网关 + BFF             │ ← 聚合层
└────────────────┬────────────────────┘
                 │
┌────────────────┴────────────────────┐
│             业务中台                  │
│  用户中心 商品中心 订单中心 支付中心  │
│  营销中心 物流中心 评价中心 搜索中心  │
└────────────────┬────────────────────┘
                 │
┌────────────────┴────────────────────┐
│             技术中台                  │
│  消息 缓存 搜索 配置 网关 日志 监控   │
└────────────────┬────────────────────┘
                 │
┌────────────────┴────────────────────┐
│             数据中台                  │
│  数据采集 数仓 画像 BI 推荐 风控     │
└─────────────────────────────────────┘
```

### BFF层（Backend for Frontend）

```java
// BFF：为不同前端聚合数据
@RestController
@RequestMapping("/bff/mall")
public class MallBFFController {
    
    @DubboReference private UserCenterService userCenter;
    @DubboReference private ProductCenterService productCenter;
    @DubboReference private OrderCenterService orderCenter;
    
    // 商城首页：聚合用户+推荐商品+优惠券
    @GetMapping("/home")
    public CompletableFuture<HomeVO> home(@RequestHeader Long userId) {
        // 并行调用各中台服务
        CompletableFuture<UserDTO> userFuture = CompletableFuture.supplyAsync(
            () -> userCenter.getUser(userId));
        CompletableFuture<List<ProductDTO>> productFuture = CompletableFuture.supplyAsync(
            () -> productCenter.getRecommendProducts(userId, 10));
        CompletableFuture<List<CouponDTO>> couponFuture = CompletableFuture.supplyAsync(
            () -> marketingCenter.getUserCoupons(userId));
        
        // 聚合结果
        return CompletableFuture.allOf(userFuture, productFuture, couponFuture)
            .thenApply(v -> HomeVO.builder()
                .user(userFuture.join())
                .products(productFuture.join())
                .coupons(couponFuture.join())
                .build());
    }
}
```

## 6. 中台 vs 微服务

| 维度 | 中台 | 微服务 |
|------|------|--------|
| 定位 | 业务能力复用 | 技术架构风格 |
| 关注点 | 业务域划分和能力沉淀 | 服务拆分和技术治理 |
| 边界 | 按业务能力中心划分 | 按领域边界(DDD)划分 |
| 规模 | 通常10-20个能力中心 | 通常20-100个微服务 |
| 关系 | 中台由多个微服务组成 | 微服务是中台的实现方式 |

```
关系：中台是业务概念，微服务是技术实现

一个业务中台（如用户中心）可能包含：
  - 用户认证微服务
  - 用户信息微服务
  - 会员等级微服务
  - 权限管理微服务
```

### 中台的适用条件

```
适合建设中台的企业：
  1. 多业务线（3+产品线有共性能力）
  2. 团队规模大（100+研发）
  3. 业务复杂度高
  4. 有重复建设问题

不适合：
  1. 单一业务线
  2. 小团队（<50人）
  3. 业务快速变化（中台建设周期长）
  4. 初创公司（先跑通业务再建中台）
```

## 7. 面试要点

### Q1: 中台和微服务的关系？

```
中台是业务概念：强调能力复用
微服务是技术实现：强调服务拆分

中台由多个微服务组成：
  用户中台 = 认证微服务 + 用户信息微服务 + 权限微服务

中台关注：业务能力划分、数据统一、接口标准化
微服务关注：服务拆分、独立部署、技术治理
```

### Q2: 业务中台怎么划分能力中心？

```
1. 按DDD限界上下文划分
2. 高内聚低耦合
3. 一个能力中心 = 一个业务域
4. 常见中心：用户/商品/订单/支付/营销/物流
5. 数据自治：每个中心独立数据库
6. 接口标准化：统一API规范
```

### Q3: 数据中台的价值是什么？

```
1. 打通数据孤岛：各系统数据统一采集
2. 统一数据标准：一致的指标定义和口径
3. 数据资产化：数据可复用、可交易
4. 赋能业务：用户画像、推荐、风控
5. 决策支持：BI报表、实时大屏
```

### Q4: 中台建设的坑有哪些？

```
1. 过度中台化：简单业务也搞中台 → 过度设计
2. 中台太重：能力中心职责不清 → 耦合严重
3. 建设周期长：1-2年才见效 → 业务等不起
4. 缺乏治理：接口不规范、文档缺失 → 难以复用
5. 组织不匹配：中台需要专门团队维护

建议：先跑通业务，再抽取共性能力建中台
```

### Q5: BFF层的作用是什么？

```
BFF (Backend for Frontend) 面向前端的后端：

1. 数据聚合：一次调用聚合多个中台服务
2. 接口适配：为不同前端(APP/小程序/Web)提供定制化接口
3. 协议转换：HTTP → Dubbo/gRPC
4. 缓存：BFF层缓存聚合结果
5. 限流降级：在BFF层做限流和降级

价值：前端只需一次请求获取所有数据，减少多次请求
```
