## 1. 架构评审概述

```
架构评审目的：
  - 识别风险（技术/安全/性能）
  - 保证方案合理性
  - 知识共享
  - 控制技术债

评审类型：
  - 方案评审（设计阶段）：评审技术方案文档
  - 代码评审（开发阶段）：Review关键代码
  - 上线评审（发布前）：评估上线风险

参与角色：
  - 架构师（主导）
  - 技术负责人（执行）
  - 运维代表（部署/监控）
  - 安全代表（安全合规）
  - 相关业务方（需求确认）
```

## 2. 评审流程

```
1. 准备阶段（1-2天）
   ├── 方案负责人编写《技术方案设计书》
   ├── 提前发给参与者阅读
   └── 收集预审问题

2. 评审会议（1-2小时）
   ├── 方案讲解（15分钟）
   ├── Q&A讨论（30-60分钟）
   ├── 风险识别（15分钟）
   └── 结论：通过/修改后通过/重新设计

3. 跟进阶段
   ├── 修改方案文档
   ├── 落实评审意见
   └── 关键节点复查

《技术方案设计书》结构：
  1. 背景与目标
  2. 需求分析
  3. 方案设计（架构图+核心流程）
  4. 技术选型（对比+理由）
  5. 数据模型设计
  6. 接口设计
  7. 非功能性（性能/可用性/安全）
  8. 风险与对策
  9. 里程碑计划
```

## 3. Checklist检查清单

### 3.1 功能完整性

```
✅ 是否覆盖所有需求场景
✅ 异常流程是否处理
✅ 边界条件是否考虑
✅ 幂等性设计
✅ 兼容性（新旧版本共存）
```

### 3.2 性能

```
✅ 预估QPS/TPS是否满足需求
✅ 响应时间目标（P99 < 200ms?）
✅ 数据库查询是否走索引
✅ 是否需要缓存
✅ 批量操作是否分批
✅ 异步化（MQ/线程池）
✅ 数据量增长后的扩展方案
```

### 3.3 可用性

```
✅ 单点故障识别（是否有备份/故障转移）
✅ 服务降级方案
✅ 限流方案
✅ 熔断方案
✅ 重试策略（次数/退避）
✅ 超时设置
✅ 滚动发布/灰度发布方案
✅ 回滚方案
```

### 3.4 数据一致性

```
✅ 分布式事务方案（Saga/TCC/本地消息表）
✅ 缓存一致性方案
✅ 消息幂等消费
✅ 并发修改控制（乐观锁/分布式锁）
✅ 数据校验（对账机制）
```

### 3.5 安全

```
✅ 认证鉴权方案
✅ 敏感数据加密存储
✅ SQL注入防护
✅ XSS/CSRF防护
✅ 接口限流防刷
✅ 日志脱敏
✅ API权限校验
```

### 3.6 可观测性

```
✅ 日志（关键链路日志）
✅ 指标（QPS/RT/错误率）
✅ 链路追踪（TraceId）
✅ 告警规则（阈值/通知方式）
✅ 关键业务大盘
```

## 4. 架构反模式

```
常见架构反模式：

1. 分布式单体
   症状：微服务拆分但共享数据库、同步调用链太长
   解决：独立数据库、异步解耦、事件驱动

2. 过度设计
   症状：日活100的系统用了K8s+Service Mesh+事件溯源
   解决：匹配业务规模，可演进而非一步到位

3. 共享数据库
   症状：多个微服务共享同一个数据库
   解决：每个服务独立数据库，通过API/事件通信

4. 链式调用
   症状：A→B→C→D→E，一环挂全链挂
   解决：异步化、聚合调用、缓存

5. 无限重试
   症状：失败无限重试，放大流量雪崩
   解决：有限重试+退避+熔断

6. 烟囱系统
   症状：每个系统独立，无法复用
   解决：识别共性，平台化/中台化

7. 大泥球
   症状：所有逻辑在一个系统里，边界模糊
   解决：DDD划分限界上下文，逐步拆分
```

## 5. 面试题速查

**Q1: 架构评审的核心目标？**
```
识别风险、保证方案合理、知识共享、控制技术债
不是挑刺，而是提前发现问题降低风险
```

**Q2: 架构评审Checklist的关键维度？**
```
功能完整性、性能、可用性、数据一致性、安全、可观测性
每个维度有具体检查项
```

**Q3: 什么是分布式单体？**
```
看起来是微服务（拆了服务），实际还是单体（共享DB、同步链路长）
问题：有微服务的复杂度，没有微服务的优势
解决：独立数据库+异步解耦
```

**Q4: 如何避免过度设计？**
```
1. 匹配业务规模（日活/QPS/数据量）
2. 可演进而非一步到位
3. YAGNI原则（不需要的不做）
4. 简单方案优先，需要时再升级
```
