## 1. 架构评审概述

```
架构评审目的：
  - 识别风险（技术/安全/性能）
  - 保证方案合理性
  - 知识共享
  - 控制技术债

评审类型：
  - 方案评审（设计阶段）：评审技术方案文档
  - 代码评审（开发阶段）：Review关键代码
  - 上线评审（发布前）：评估上线风险
  - 架构复盘（上线后）：事后分析问题

参与角色：
  - 架构师（主导）
  - 技术负责人（执行）
  - 运维代表（部署/监控）
  - 安全代表（安全合规）
  - 相关业务方（需求确认）
  - QA代表（测试策略）
  - DBA（数据库设计审核）

评审原则：
  ┌──────────────────────────────────────────────────┐
  │  1. 对事不对人 — 评审方案质量，非评审个人能力     │
  │  2. 数据驱动 — 用压测数据/监控数据说话             │
  │  3. 风险优先 — 先识别高风险项，细节后续跟进        │
  │  4. 可执行 — 评审意见必须具体可落地                │
  │  5. 时间盒 — 控制评审时间，不无限延伸              │
  │  6. 记录留档 — 评审记录归档，便于追溯              │
  └──────────────────────────────────────────────────┘
```

## 2. 评审流程

```
1. 准备阶段（1-2天）
   ├── 方案负责人编写《技术方案设计书》
   ├── 提前2天发给参与者阅读
   ├── 收集预审问题
   └── 确定评审会议时间和参与人

2. 评审会议（1-2小时）
   ├── 方案讲解（15分钟）
   ├── Q&A讨论（30-60分钟）
   ├── 风险识别（15分钟）
   └── 结论：通过/修改后通过/重新设计

3. 跟进阶段
   ├── 修改方案文档
   ├── 落实评审意见
   └── 关键节点复查

4. 上线后复盘（可选）
   ├── 对比实际效果与设计预期
   ├── 记录偏差和原因
   └── 更新架构知识库
```

### 2.1 技术方案设计书模板

```
《技术方案设计书》结构：

  1. 背景与目标
     ├── 业务背景（为什么要做）
     ├── 目标（做成什么样）
     └── 非目标（不做什么，明确边界）

  2. 需求分析
     ├── 功能需求（用户故事/用例）
     ├── 非功能需求（性能/可用性/安全）
     └── 约束条件（预算/时间/团队能力）

  3. 方案设计
     ├── 整体架构图
     ├── 核心流程图（时序图/流程图）
     ├── 关键模块设计
     └── 数据流图

  4. 技术选型
     ├── 候选方案对比（至少2个）
     ├── 选型理由（评分矩阵）
     └── 风险评估

  5. 数据模型设计
     ├── ER图/表结构
     ├── 数据量预估
     ├── 读写QPS预估
     └── 分库分表方案

  6. 接口设计
     ├── API文档（Swagger）
     ├── 内部接口
     └── 消息格式

  7. 非功能性设计
     ├── 性能（QPS/RT/数据量）
     ├── 可用性（SLA/降级/熔断）
     ├── 安全（认证/加密/脱敏）
     └── 可观测性（日志/指标/链路）

  8. 风险与对策
     ├── 技术风险
     ├── 依赖风险
     ├── 人员风险
     └── 回滚方案

  9. 里程碑计划
     ├── 开发计划
     ├── 测试计划
     ├── 上线计划
     └── 监控告警计划
```

## 3. Checklist检查清单

### 3.1 功能完整性

```
✅ 是否覆盖所有需求场景
✅ 异常流程是否处理
✅ 边界条件是否考虑
✅ 幂等性设计
✅ 兼容性（新旧版本共存）
✅ 是否考虑数据迁移
✅ 是否考虑灰度/AB测试
✅ 回滚方案是否可行
```

### 3.2 性能

```
✅ 预估QPS/TPS是否满足需求
✅ 响应时间目标（P99 < 200ms?）
✅ 数据库查询是否走索引
✅ 是否需要缓存
✅ 批量操作是否分批
✅ 异步化（MQ/线程池）
✅ 数据量增长后的扩展方案
✅ 是否做了压测验证
✅ 慢查询预防（explain分析）
✅ 连接池配置是否合理
✅ 序列化方式是否高效
✅ 是否有N+1查询问题

性能Checklist分级:
  ┌──────────────────────────────────────────────┐
  │  P0 (必须达标)                                │
  │  ├── 核心接口 P99 < 200ms                     │
  │  ├── 数据库查询走索引                         │
  │  ├── 无N+1查询                                │
  │  └── 关键路径无外部阻塞调用                    │
  │                                              │
  │  P1 (应当达标)                                │
  │  ├── 缓存策略明确                             │
  │  ├── 批量操作分页                             │
  │  └── 异步化非核心链路                         │
  │                                              │
  │  P2 (建议达标)                                │
  │  ├── 连接池参数调优                           │
  │  └── 序列化优化                              │
  └──────────────────────────────────────────────┘
```

### 3.3 可用性

```
✅ 单点故障识别（是否有备份/故障转移）
✅ 服务降级方案
✅ 限流方案（Sentinel/网关层）
✅ 熔断方案
✅ 重试策略（次数/退避）
✅ 超时设置（连接超时/读取超时）
✅ 滚动发布/灰度发布方案
✅ 回滚方案
✅ 多机房/多可用区部署
✅ 数据备份与恢复
✅ 容灾切换流程
✅ 优雅停机
✅ 健康检查（就绪探针/存活探针）

可用性目标分级:
  ┌──────────────────────────────────────────────┐
  │  99.9% (三个九) — 年停机 ≤ 8.76小时          │
  │  99.95% → 年停机 ≤ 4.38小时                  │
  │  99.99% (四个九) — 年停机 ≤ 52.6分钟         │
  │  99.999% (五个九) — 年停机 ≤ 5.26分钟         │
  └──────────────────────────────────────────────┘
```

### 3.4 数据一致性

```
✅ 分布式事务方案（Saga/TCC/本地消息表）
✅ 缓存一致性方案（Cache Aside/Write Through）
✅ 消息幂等消费
✅ 并发修改控制（乐观锁/分布式锁）
✅ 数据校验（对账机制）
✅ 数据迁移一致性（双写/灰度切换）
✅ 主从延迟处理
✅ 分布式ID方案（雪花算法/号段模式）

分布式事务选型矩阵:
  ┌──────────────────────────────────────────────────┐
  │  场景              │  推荐方案                    │
  │  ──────          │  ──────                     │
  │  强一致性+高TPS   │  TCC                         │
  │  最终一致性       │  本地消息表/Saga              │
  │  跨库聚合查询     │  不建议(改设计)               │
  │  资金类交易       │  TCC/Seata AT                │
  │  异步通知         │  本地消息表+MQ               │
  └──────────────────────────────────────────────────┘
```

### 3.5 安全

```
✅ 认证鉴权方案（OAuth2/JWT/SSO）
✅ 敏感数据加密存储（AES/RSA）
✅ SQL注入防护（参数化查询）
✅ XSS/CSRF防护
✅ 接口限流防刷
✅ 日志脱敏
✅ API权限校验
✅ 数据传输加密（HTTPS/TLS）
✅ 密码安全（BCrypt）
✅ 审计日志
✅ 越权检查（水平/垂直越权）
✅ 文件上传安全（类型/大小/路径）
✅ 依赖漏洞扫描
```

### 3.6 可观测性

```
✅ 日志（关键链路日志 + TraceId）
✅ 指标（QPS/RT/错误率/资源使用率）
✅ 链路追踪（TraceId/SpanId）
✅ 告警规则（阈值/通知方式/升级策略）
✅ 关键业务大盘
✅ 容量规划数据
✅ SLI/SLO定义

可观测性三支柱:
  ┌──────────────────────────────────────────────────┐
  │  Metrics (指标)     │  Prometheus + Grafana       │
  │  Logging (日志)     │  ELK / Loki                  │
  │  Tracing (链路)     │  Jaeger / SkyWalking         │
  │                                                    │
  │  三者通过TraceId关联                               │
  │  故障排查: 大盘看异常 → 链路定位 → 日志看详情      │
  └──────────────────────────────────────────────────┘
```

## 4. 架构反模式

```
常见架构反模式：

1. 分布式单体
   症状：微服务拆分但共享数据库、同步调用链太长
   解决：独立数据库、异步解耦、事件驱动

2. 过度设计
   症状：日活100的系统用了K8s+Service Mesh+事件溯源
   解决：匹配业务规模，可演进而非一步到位

3. 共享数据库
   症状：多个微服务共享同一个数据库
   解决：每个服务独立数据库，通过API/事件通信

4. 链式调用
   症状：A→B→C→D→E，一环挂全链挂
   解决：异步化、聚合调用、缓存

5. 无限重试
   症状：失败无限重试，放大流量雪崩
   解决：有限重试+退避+熔断

6. 烟囱系统
   症状：每个系统独立，无法复用
   解决：识别共性，平台化/中台化

7. 大泥球
   症状：所有逻辑在一个系统里，边界模糊
   解决：DDD划分限界上下文，逐步拆分

8. 热点Key
   症状：缓存/数据库某Key访问量极大，打挂单节点
   解决：多级缓存、本地缓存、Key分散

9. 连接泄漏
   症状：连接池/文件句柄不释放，逐渐耗尽
   解决：try-with-resources、连接池监控

10. 分布式事务滥用
    症状：2PC/XA到处用，性能极差
    解决：最终一致性(Saga/消息表)替代强一致性
```

### 5.2 选型原则

```
技术选型六大原则:

  1. 适用性原则 — 匹配业务场景，不盲目追新
     日活100 → 单体Spring Boot
     日活10万 → 微服务+Redis+MQ
     日活千万 → 微服务+分库分表+ES+MQ+缓存

  2. 成熟度原则 — 选择生产验证过的技术
     评估: GitHub Star/Issue活跃度、生产案例、社区规模
     避坑: 0.x版本慎用，无生产案例的新技术慎用

  3. 团队匹配原则 — 团队能hold住
     考虑: 团队技术栈、学习曲线、招聘难度
     Java团队 → 不选Go/Rust(除非有特殊理由)

  4. 生态优先原则 — 优先选择生态好的
     Spring Cloud生态 > 自研微服务框架
     有现成集成 > 手动对接

  5. 可退出原则 — 能换掉
     抽象层(接口隔离) + 数据可迁移
     避免厂商锁定(除非成本可接受)

  6. 成本原则 — TCO(总拥有成本)
     开发成本 + 运维成本 + 服务器成本 + 培训成本
     开源 ≠ 免费(运维成本可能更高)
```

## 6. 评审实战案例

### 6.1 案例: 电商订单系统架构评审

```
背景: 新电商系统，预计日订单50万，大促日500万

方案A: 单体Spring Boot + MySQL + Redis
  优点: 开发快，运维简单
  缺点: 单库压力大，扩展性差

方案B: 微服务 + 分库分表 + Kafka + ES
  优点: 高扩展，高可用
  缺点: 复杂度高，开发周期长

方案C: 模块化单体 + 读写分离 + Redis + 异步MQ
  优点: 平衡复杂度和扩展性
  缺点: 需要设计好模块边界

评审结论: 方案C，理由:
  - 日50万订单非超大规模，微服务过度设计
  - 读写分离解决读压力
  - 异步MQ削峰填谷
  - 模块化为未来微服务化预留
  - 可以先上线再拆分
```

## 7. 持续改进机制

```
架构评审不是一次性活动，需要持续改进:

  ┌──────────────────────────────────────────────────┐
  │              架构评审持续改进闭环                  │
  │                                                  │
  │  设计 → 评审 → 实施 → 上线 → 复盘 → 更新Checklist │
  │   ↑                                      │       │
  │   └──────────────────────────────────────┘       │
  └──────────────────────────────────────────────────┘

  1. 架构决策记录(ADR)
     每个重要技术决策记录: 上下文+决策+理由+后果
     形成: ADR(Architecture Decision Records)知识库

  2. 架构度量和体检
     定期度量: 代码复杂度、耦合度、技术债
     工具: SonarQube + 架构守护(ArchUnit)

  3. 技术雷达
     维护团队技术雷达: 采纳/试验/评估/淘汰
     定期Review更新

  4. 事后复盘
     上线后对比预期 → 记录偏差 → 更新设计指南
     事故分析 → 根因 → 补充Checklist

  5. 技术分享
     月度技术分享: 架构经验/踩坑/新技术
     建立团队架构知识库
```

### 7.1 ADR模板

```
# ADR-001: 订单系统使用RocketMQ替代Kafka

  日期: 2026-07-13
  状态: 已采纳
  决策者: 架构组

  ## 上下文
  订单系统需要消息队列支撑:
  - 订单创建异步通知
  - 订单状态变更广播
  - 日订单50万，大促500万
  - 需要事务消息保证一致性

  ## 决策
  选择RocketMQ而非Kafka

  ## 理由
  1. 事务消息: RocketMQ原生支持，Kafka需自行实现
  2. 消息可靠性: RocketMQ同步刷盘+同步双写
  3. 延迟消息: RocketMQ支持延迟消息(Kafka需自己实现)
  4. 运维: 团队已有RocketMQ运维经验
  5. 吞吐: 日500万订单RocketMQ可承受(Kafka更高但不需要)

  ## 后果
  - 正面: 事务消息简化分布式事务，延迟消息支持订单超时
  - 负面: 生态不如Kafka丰富，大数据集成需适配
  - 风险: 如果日订单超千万需重新评估
```

## 8. 面试题速查

**Q1: 架构评审的核心目标？**

```
识别风险、保证方案合理、知识共享、控制技术债
不是挑刺，而是提前发现问题降低风险
评审结论: 通过/修改后通过/重新设计
```

**Q2: 架构评审Checklist的关键维度？**

```
功能完整性、性能、可用性、数据一致性、安全、可观测性
每个维度有具体检查项，分P0/P1/P2优先级
```

**Q3: 什么是分布式单体？**

```
看起来是微服务（拆了服务），实际还是单体（共享DB、同步链路长）
问题：有微服务的复杂度，没有微服务的优势
解决：独立数据库+异步解耦
```

**Q4: 如何避免过度设计？**

```
1. 匹配业务规模（日活/QPS/数据量）
2. 可演进而非一步到位
3. YAGNI原则（不需要的不做）
4. 简单方案优先，需要时再升级
5. 模块化单体为微服务化预留
```

**Q5: 技术选型的核心原则？**

```
适用性(匹配场景)、成熟度(生产验证)、团队匹配(能hold住)、
生态优先(集成好)、可退出(能换掉)、成本(TCO)
用评分矩阵量化决策，记录ADR留档
```

**Q6: 什么是ADR？**

```
Architecture Decision Record — 架构决策记录
记录: 上下文+决策+理由+后果
价值: 知识传承、追溯决策原因、避免重复讨论
每个重要技术决策一条ADR，形成知识库
```

**Q7: 如何保证架构持续演进？**

```
1. ADR记录每个决策
2. 定期架构体检(复杂度/耦合/技术债)
3. 技术雷达跟踪技术趋势
4. 上线后复盘对比预期
5. 事故分析补充Checklist
6. 月度技术分享传播知识
```
