## 1. RESTful API规范

### 1.1 设计原则

```
RESTful API六大约束:
  1. 客户端-服务器分离
  2. 无状态 — 每个请求包含所有信息, 服务器不保存会话状态
  3. 可缓存 — 响应应标识是否可缓存
  4. 统一接口 — 资源URI + HTTP方法 + 表征
  5. 分层系统 — 客户端不感知中间层
  6. 按需代码(可选) — 服务端可下发可执行代码

HTTP方法语义:
  ┌──────────────────────────────────────────────────┐
  │  方法    │  语义        │  幂等  │  安全  │  示例    │
  │  ────   │  ─────      │  ──── │  ──── │  ────   │
  │  GET    │  查询资源     │  是    │  是    │  安全    │
  │  POST   │  创建资源     │  否    │  否    │  不安全  │
  │  PUT    │  全量更新     │  是    │  否    │  不安全  │
  │  PATCH  │  部分更新     │  否    │  否    │  不安全  │
  │  DELETE │  删除资源     │  是    │  否    │  不安全  │
  └──────────────────────────────────────────────────┘
  幂等: 多次执行结果相同
  安全: 不修改服务器数据
```

### 1.2 URI设计

```
好的URI设计 — 用名词, 不用动词:

  ✅ GET    /api/users          — 获取用户列表
  ✅ GET    /api/users/123      — 获取单个用户
  ✅ POST   /api/users          — 创建用户
  ✅ PUT    /api/users/123      — 更新用户(全量)
  ✅ PATCH  /api/users/123      — 更新用户(部分)
  ✅ DELETE /api/users/123      — 删除用户

  ✅ GET    /api/users/123/orders — 获取用户123的订单

  ❌ GET    /api/getUser        — 不要用动词
  ❌ GET    /api/createUser     — 用POST
  ❌ GET    /api/deleteUser/123 — 用DELETE
  ❌ POST   /api/users/123/delete — 用DELETE /api/users/123

  特殊操作(非CRUD):
  ✅ POST   /api/users/123/actions/activate   — 激活用户
  ✅ POST   /api/orders/123/actions/cancel     — 取消订单

  查询参数:
  ✅ GET /api/users?page=1&size=20&sort=name,desc&status=active
  ✅ GET /api/users?keyword=alice&fields=id,name,email
```

### 1.3 统一响应格式

```java
// 统一响应体
@Data
@AllArgsConstructor
public class ApiResponse<T> {
    private int code;        // 业务状态码: 0=成功, 非0=失败
    private String message;  // 提示信息
    private T data;          // 数据
    private long timestamp;  // 时间戳

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(0, "success", data, System.currentTimeMillis());
    }

    public static <T> ApiResponse<T> error(int code, String message) {
        return new ApiResponse<>(code, message, null, System.currentTimeMillis());
    }
}

// 分页响应
@Data
@AllArgsConstructor
public class PageResponse<T> {
    private List<T> list;
    private long total;
    private int page;
    private int size;
    private int totalPages;

    public static <T> PageResponse<T> of(List<T> list, long total, int page, int size) {
        int totalPages = (int) Math.ceil((double) total / size);
        return new PageResponse<>(list, total, page, size, totalPages);
    }
}

// Controller
@RestController
@RequestMapping("/api/users")
public class UserController {

    @GetMapping
    public ApiResponse<PageResponse<UserDTO>> list(
            @RequestParam(defaultValue = "1") int page,
            @RequestParam(defaultValue = "20") int size,
            @RequestParam(required = false) String keyword) {
        PageResponse<UserDTO> result = userService.list(page, size, keyword);
        return ApiResponse.success(result);
    }

    @GetMapping("/{id}")
    public ApiResponse<UserDTO> getById(@PathVariable Long id) {
        return ApiResponse.success(userService.getById(id));
    }

    @PostMapping
    public ApiResponse<UserDTO> create(@Valid @RequestBody CreateUserRequest req) {
        return ApiResponse.success(userService.create(req));
    }

    @PutMapping("/{id}")
    public ApiResponse<UserDTO> update(@PathVariable Long id,
                                        @Valid @RequestBody UpdateUserRequest req) {
        return ApiResponse.success(userService.update(id, req));
    }

    @DeleteMapping("/{id}")
    public ApiResponse<Void> delete(@PathVariable Long id) {
        userService.delete(id);
        return ApiResponse.success(null);
    }
}
```

### 1.4 HTTP状态码

```
常用HTTP状态码:

  2xx 成功:
    200 OK — 请求成功
    201 Created — 资源创建成功(POST)
    204 No Content — 成功但无返回(DELETE)

  3xx 重定向:
    301 Moved Permanently — 永久重定向
    304 Not Modified — 资源未修改(缓存)

  4xx 客户端错误:
    400 Bad Request — 参数错误
    401 Unauthorized — 未认证(没登录)
    403 Forbidden — 无权限(登录了但没权限)
    404 Not Found — 资源不存在
    409 Conflict — 资源冲突(重复创建)
    422 Unprocessable Entity — 参数格式正确但语义错误
    429 Too Many Requests — 限流

  5xx 服务器错误:
    500 Internal Server Error — 服务器内部错误
    502 Bad Gateway — 网关错误
    503 Service Unavailable — 服务不可用
    504 Gateway Timeout — 网关超时

  争议: 业务错误用HTTP状态码还是业务码?
    方案A: HTTP 200 + 业务码(0成功/非0失败) — 国内主流
    方案B: HTTP状态码反映结果(4xx/5xx) — RESTful标准
    推荐: 方案A(前端统一处理, 不需要判断HTTP状态码)
```

```
REST vs GraphQL:

  ┌──────────────────────────────────────────────────┐
  │  REST                │  GraphQL                  │
  │  ────               │  ───────                  │
  │  多个端点             │  单端点(/graphql)          │
  │  固定返回结构          │  按需查询字段              │
  │  过度获取(返回不需要的) │  精确获取                  │
  │  不足获取(需多次请求)   │  一次查询关联数据           │
  │  HTTP缓存(ETag)      │  无HTTP缓存               │
  │  简单直接              │  灵活但复杂               │
  │  适合简单CRUD          │  适合复杂数据关系           │
  └──────────────────────────────────────────────────┘

  GraphQL适合: 移动端(省流量)、复杂关联查询、BFF层
  REST适合: 简单API、需要HTTP缓存、微服务间通信
```

## 3. WebSocket实时通信

### 3.1 前端WebSocket

```javascript
// 原生WebSocket
const ws = new WebSocket('ws://localhost:8080/ws/chat');

ws.onopen = () => {
    console.log('Connected');
    ws.send(JSON.stringify({ type: 'join', room: 'general' }));
};

ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    console.log('Received:', data);
    // 处理消息
    switch (data.type) {
        case 'message':
            displayMessage(data);
            break;
        case 'notification':
            showNotification(data);
            break;
    }
};

ws.onerror = (error) => {
    console.error('WebSocket error:', error);
};

ws.onclose = () => {
    console.log('Disconnected, reconnecting...');
    // 自动重连
    setTimeout(connect, 3000);
};

// 发送消息
function sendMessage(text) {
    ws.send(JSON.stringify({ type: 'message', text, room: 'general' }));
}
```

### 3.2 心跳与重连

```javascript
class ReconnectingWebSocket {
    constructor(url, options = {}) {
        this.url = url;
        this.options = {
            heartbeatInterval: 30000,  // 30秒心跳
            reconnectInterval: 3000,   // 3秒重连
            maxReconnectAttempts: 10,
            ...options
        };
        this.reconnectAttempts = 0;
        this.ws = null;
        this.heartbeatTimer = null;
        this.connect();
    }

    connect() {
        this.ws = new WebSocket(this.url);

        this.ws.onopen = () => {
            console.log('WebSocket connected');
            this.reconnectAttempts = 0;
            this.startHeartbeat();
        };

        this.ws.onmessage = (event) => {
            const data = JSON.parse(event.data);
            if (data.type === 'pong') return;  // 心跳响应
            this.options.onMessage?.(data);
        };

        this.ws.onclose = () => {
            this.stopHeartbeat();
            this.reconnect();
        };

        this.ws.onerror = (error) => {
            console.error('WebSocket error:', error);
        };
    }

    startHeartbeat() {
        this.heartbeatTimer = setInterval(() => {
            if (this.ws.readyState === WebSocket.OPEN) {
                this.ws.send(JSON.stringify({ type: 'ping' }));
            }
        }, this.options.heartbeatInterval);
    }

    stopHeartbeat() {
        if (this.heartbeatTimer) {
            clearInterval(this.heartbeatTimer);
            this.heartbeatTimer = null;
        }
    }

    reconnect() {
        if (this.reconnectAttempts >= this.options.maxReconnectAttempts) {
            console.error('Max reconnection attempts reached');
            return;
        }
        this.reconnectAttempts++;
        const delay = this.options.reconnectInterval * this.reconnectAttempts;
        setTimeout(() => this.connect(), delay);
    }

    send(data) {
        if (this.ws.readyState === WebSocket.OPEN) {
            this.ws.send(JSON.stringify(data));
        }
    }

    close() {
        this.stopHeartbeat();
        this.ws.close();
    }
}

// 使用
const ws = new ReconnectingWebSocket('ws://localhost:8080/ws', {
    onMessage: (data) => console.log('Message:', data)
});
```

### 3.3 SSE(Server-Sent Events)

```javascript
// SSE — 服务器单向推送(比WebSocket简单)
const eventSource = new EventSource('/api/events');

eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data);
    console.log('Event:', data);
};

eventSource.addEventListener('notification', (event) => {
    const data = JSON.parse(event.data);
    showNotification(data);
});

eventSource.onerror = () => {
    console.log('SSE error, auto-reconnecting...');
    // SSE自动重连
};

// 关闭
// eventSource.close();

// SSE vs WebSocket:
// SSE: 服务器→客户端单向, HTTP协议, 自动重连, 简单
// WebSocket: 双向, 独立协议, 需手动重连, 复杂
// 场景: 通知/消息推送用SSE, 聊天/实时交互用WebSocket
```

### 5.2 前后端联调流程

```
标准联调流程:

  1. 需求评审
     ├── 确认接口需求
     └── 确认数据格式

  2. API设计(后端)
     ├── 后端编写Swagger/OpenAPI文档
     ├── 定义请求/响应格式
     └── 发布到Apifox/YApi

  3. 并行开发
     ├── 前端: 用Mock数据开发页面
     └── 后端: 实现API

  4. 联调
     ├── 前端切换到真实API
     ├── 验证接口数据格式
     └── 修复不一致问题

  5. 集成测试
     ├── 端到端测试
     └── 性能测试

  联调常见问题:
  ┌──────────────────────────────────────────────────┐
  │  问题                  │  解决方案               │
  │  ────                │  ──────                │
  │  字段名不一致           │  以API文档为准, 统一命名 │
  │  数据类型不一致         │  API文档标注类型         │
  │  空值处理不一致         │  约定null/空字符串/0    │
  │  时间格式不一致         │  统一ISO 8601格式       │
  │  分页参数不一致         │  统一page/size          │
  │  错误码不一致           │  统一错误码表           │
  │  跨域问题              │  后端配置CORS或前端代理  │
  └──────────────────────────────────────────────────┘
```

@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://www.example.com", "https://admin.example.com")

## 6. 面试题速查

**Q1: RESTful API的设计原则？**

```
资源URI(名词) + HTTP方法(动词) + 统一响应格式
GET查询, POST创建, PUT全量更新, PATCH部分更新, DELETE删除
无状态, 幂等性(GET/PUT/DELETE幂等, POST/PATCH不幂等)
```

**Q2: REST和GraphQL的区别？**

```
REST: 多端点, 固定返回, 可能过度/不足获取
GraphQL: 单端点, 按需查询, 精确获取, 关联查询一次完成
REST适合简单CRUD, GraphQL适合复杂关系/移动端
```

**Q3: WebSocket和SSE的区别？**

```
WebSocket: 双向通信, 独立协议, 需手动重连
SSE: 服务器→客户端单向, HTTP协议, 自动重连
聊天/互动用WebSocket, 通知/推送用SSE
```

**Q4: 前后端联调流程？**

```
需求评审 → API文档(Swagger) → 并行开发(前端Mock) → 联调 → 集成测试
关键: API文档先行, 前端用Mock不等后端, 统一数据格式
```

**Q5: 跨域问题怎么解决？**

```
开发: Vite/Webpack代理(前端)
生产: 后端CORS配置 / Nginx反向代理
注意: allowCredentials=true时不能allowedOrigins=*
```
