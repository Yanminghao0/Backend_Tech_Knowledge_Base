## 1. 同源策略与跨域

```
同源策略 — 浏览器安全基石

  同源 = 协议 + 域名 + 端口 三者完全一致

  ┌──────────────────────────────────────────────────┐
  │  URL A: http://www.example.com/page              │
  │  URL B: https://www.example.com/page  ❌ 协议不同  │
  │  URL C: http://api.example.com/page   ❌ 域名不同  │
  │  URL D: http://www.example.com:8080    ❌ 端口不同 │
  │  URL E: http://www.example.com/other   ✅ 同源     │
  └──────────────────────────────────────────────────┘

  同源策略限制:
  1. 无法读取跨域源的Cookie/LocalStorage/IndexedDB
  2. 无法操作跨域源的DOM(iframe)
  3. 无法发送跨域AJAX请求(XMLHttpRequest/Fetch)

  不受同源策略限制:
  - <img> <script> <link> <video> <iframe> 标签的src
  - <form> 提交(但无法读取响应)
  - WebSocket

  跨域解决方案:
  1. CORS — 标准方案(推荐)
  2. 代理 — 开发环境/Nginx反向代理
  3. JSONP — 利用<script>标签(已过时, 只支持GET)
  4. postMessage — 跨窗口通信(iframe)
  5. document.domain — 同主域不同子域(已废弃)
```

## 2. CORS详解

### 2.1 简单请求

```
CORS(Cross-Origin Resource Sharing) — 跨域资源共享

  简单请求(不触发预检):
    条件:
    - 方法: GET / HEAD / POST
    - Content-Type: text/plain / multipart/form-data / application/x-www-form-urlencoded
    - 不含自定义Header

  流程:
    浏览器 → 服务器
    请求头: Origin: https://www.example.com

    服务器 → 浏览器
    响应头:
      Access-Control-Allow-Origin: https://www.example.com
      Access-Control-Allow-Credentials: true

    浏览器检查Allow-Origin → 匹配则放行
```

### 2.2 预检请求(Preflight)

```
非简单请求(触发预检):
    条件:
    - 方法: PUT / DELETE / PATCH
    - Content-Type: application/json
    - 含自定义Header(如Authorization)

  流程:
    1. 预检请求(OPTIONS方法)
       浏览器 → 服务器
       OPTIONS /api/users HTTP/1.1
       Origin: https://www.example.com
       Access-Control-Request-Method: PUT
       Access-Control-Request-Headers: Content-Type, Authorization

    2. 预检响应
       服务器 → 浏览器
       HTTP/1.1 200 OK
       Access-Control-Allow-Origin: https://www.example.com
       Access-Control-Allow-Methods: GET, POST, PUT, DELETE
       Access-Control-Allow-Headers: Content-Type, Authorization
       Access-Control-Allow-Credentials: true
       Access-Control-Max-Age: 3600  // 预检缓存1小时

    3. 实际请求(预检通过后)
       浏览器 → 服务器
       PUT /api/users/1 HTTP/1.1
       Origin: https://www.example.com
       Content-Type: application/json
       Authorization: Bearer eyJhbG...
       {"name": "Alice"}

    4. 实际响应
       服务器 → 浏览器
       Access-Control-Allow-Origin: https://www.example.com
       {"code": 0, "data": {...}}
```

### 2.3 后端CORS配置

```java
// Spring Boot CORS配置

// 方式1: 全局配置
@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins(
                "https://www.example.com",
                "https://admin.example.com"
            )
            .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
            .allowedHeaders("*")
            .exposedHeaders("Authorization", "X-Total-Count")
            .allowCredentials(true)
            .maxAge(3600);
    }
}

// 方式2: Filter(更灵活)
@Component
public class CorsFilter implements Filter {

    @Override
    public void doFilter(ServletRequest req, ServletResponse res,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletResponse response = (HttpServletResponse) res;
        HttpServletRequest request = (HttpServletRequest) req;

        String origin = request.getHeader("Origin");
        // 白名单校验
        List<String> allowedOrigins = List.of(
            "https://www.example.com",
            "https://admin.example.com"
        );

        if (allowedOrigins.contains(origin)) {
            response.setHeader("Access-Control-Allow-Origin", origin);
            response.setHeader("Access-Control-Allow-Credentials", "true");
            response.setHeader("Access-Control-Allow-Methods",
                "GET, POST, PUT, DELETE, OPTIONS");
            response.setHeader("Access-Control-Allow-Headers",
                "Content-Type, Authorization, X-Requested-With");
            response.setHeader("Access-Control-Max-Age", "3600");
        }

        // 预检请求直接返回200
        if ("OPTIONS".equalsIgnoreCase(request.getMethod())) {
            response.setStatus(HttpServletResponse.SC_OK);
            return;
        }

        chain.doFilter(req, res);
    }
}

// 方式3: 注解(单个Controller)
@CrossOrigin(origins = "https://www.example.com", allowCredentials = "true")
@RestController
public class UserController { }
```

```
CORS注意事项:

  1. Allow-Credentials与Allow-Origin
     allowCredentials=true时, Allow-Origin不能为*
     必须指定具体域名(或动态回填Origin)

  2. Cookie跨域
     前端: fetch(url, { credentials: 'include' })
            axios.defaults.withCredentials = true
     后端: allowCredentials(true)

  3. 安全考虑
     不要无脑设allowedOrigins=*(不安全)
     白名单校验Origin
     生产环境只允许已知域名
```

## 3. 代理方案

### 3.1 开发环境代理

```javascript
// Vite代理
// vite.config.js
export default {
    server: {
        proxy: {
            '/api': {
                target: 'http://localhost:8080',
                changeOrigin: true,
                // 路径重写
                rewrite: (path) => path.replace(/^\/api/, ''),
                // cookie处理
                cookieDomainRewrite: 'localhost',
            },
            '/ws': {
                target: 'ws://localhost:8080',
                ws: true,  // WebSocket代理
            }
        }
    }
};

// 前端请求(同源, 代理转发)
// fetch('/api/users') → 代理 → http://localhost:8080/users
// 浏览器认为同源, 无跨域问题
```

### 3.2 Nginx反向代理

```nginx
# 生产环境Nginx配置
server {
    listen 443 ssl;
    server_name www.example.com;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    # 前端静态资源
    location / {
        root /var/www/html;
        try_files $uri $uri/ /index.html;  # SPA路由
    }

    # API代理
    location /api/ {
        proxy_pass http://backend:8080/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # WebSocket代理
    location /ws/ {
        proxy_pass http://backend:8080/ws/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 86400;
    }

    # 文件上传
    location /upload {
        proxy_pass http://backend:8080/upload;
        client_max_body_size 50m;
        proxy_request_buffering off;
    }
}
```

## 4. JWT认证

### 4.1 JWT结构

```
JWT(JSON Web Token) — 无状态认证令牌

  结构: Header.Payload.Signature

  Header(头部):
  {
    "alg": "HS256",  // 签名算法
    "typ": "JWT"
  }
  → Base64URL编码

  Payload(载荷):
  {
    "sub": "1001",           // 用户ID
    "name": "Alice",
    "role": "admin",
    "iat": 1719500000,       // 签发时间
    "exp": 1719586400,       // 过期时间
    "jti": "unique-token-id" // 唯一标识
  }
  → Base64URL编码
  ⚠️ Payload不加密, 不要放敏感信息

  Signature(签名):
  HMACSHA256(
    base64url(Header) + "." + base64url(Payload),
    secret
  )

  完整JWT:
  eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMDAxIiwibmFtZSI6IkFsaWNlIn0.signature
```

### 4.2 前端JWT流程

```javascript
import axios from 'axios';

// axios实例
const api = axios.create({
    baseURL: '/api',
    timeout: 10000,
});

// 请求拦截器 — 自动添加Token
api.interceptors.request.use(config => {
    const token = localStorage.getItem('token');
    if (token) {
        config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
});

// 响应拦截器 — Token刷新
let isRefreshing = false;
let failedQueue = [];

api.interceptors.response.use(
    response => response,
    async error => {
        const originalRequest = error.config;

        // Token过期(401)
        if (error.response?.status === 401 && !originalRequest._retry) {
            if (isRefreshing) {
                // 等待刷新完成
                return new Promise((resolve, reject) => {
                    failedQueue.push({ resolve, reject, config: originalRequest });
                });
            }

            originalRequest._retry = true;
            isRefreshing = true;

            try {
                const refreshToken = localStorage.getItem('refreshToken');
                const response = await axios.post('/api/auth/refresh', { refreshToken });
                const { token, refreshToken: newRefreshToken } = response.data.data;

                localStorage.setItem('token', token);
                localStorage.setItem('refreshToken', newRefreshToken);

                // 重试队列中的请求
                failedQueue.forEach(({ resolve, config }) => {
                    config.headers.Authorization = `Bearer ${token}`;
                    resolve(api(config));
                });
                failedQueue = [];

                originalRequest.headers.Authorization = `Bearer ${token}`;
                return api(originalRequest);
            } catch (refreshError) {
                // 刷新失败 → 跳转登录
                localStorage.removeItem('token');
                localStorage.removeItem('refreshToken');
                window.location.href = '/login';
                return Promise.reject(refreshError);
            } finally {
                isRefreshing = false;
            }
        }

        return Promise.reject(error);
    }
);

export default api;
```

### 4.3 Token存储方案

```
JWT存储位置对比:

  ┌──────────────────────────────────────────────────┐
  │  存储位置      │  XSS  │  CSRF  │  说明           │
  │  ────────    │  ──── │  ────  │  ──────        │
  │  LocalStorage │  ❌   │  ✅    │  JS可读, XSS可偷 │
  │  SessionStorage│ ❌   │  ✅    │  同上, 关页消失  │
  │  Cookie(httpOnly)│ ✅ │  ❌    │  JS不可读, 防XSS │
  │  Cookie+SameSite│ ✅ │  ✅    │  防CSRF          │
  └──────────────────────────────────────────────────┘

  推荐方案:
    Access Token → 内存(JavaScript变量), 刷新时用Refresh Token
    Refresh Token → HttpOnly + Secure + SameSite Cookie

    纯前端方案(LocalStorage): 简单但XSS风险
    安全方案(HttpOnly Cookie): 安全但跨域复杂

  Access Token有效期: 短(15-30分钟)
  Refresh Token有效期: 长(7-30天)
```

## 5. SSO单点登录

```
SSO(Single Sign-On) — 一次登录, 多系统访问

  ┌──────────────────────────────────────────────────────┐
  │                    SSO认证中心                        │
  │              (sso.example.com)                        │
  │              ┌──────────┐                             │
  │              │ 登录页面  │                             │
  │              │ Token管理 │                             │
  │              └─────┬────┘                             │
  └────────────────────┼─────────────────────────────────┘
                       │
          ┌────────────┼────────────┐
          ↓            ↓            ↓
     ┌─────────┐ ┌─────────┐ ┌─────────┐
     │ 系统A    │ │ 系统B   │ │ 系统C   │
     │ app1.com│ │app2.com │ │app3.com │
     └─────────┘ └─────────┘ └─────────┘

  SSO流程(未登录访问系统A):
    1. 用户 → 系统A → 检查Token → 无Token
    2. 系统A → 重定向到SSO认证中心
       https://sso.example.com/login?redirect=https://app1.com
    3. SSO认证中心 → 展示登录页 → 用户输入账号密码
    4. SSO认证中心 → 验证成功 → 生成全局Ticket
    5. SSO → 重定向回系统A, 携带Ticket
       https://app1.com?ticket=abc123
    6. 系统A → 用Ticket向SSO验证 → 获取Token → 存入Session
    7. 用户访问系统B → 系统B重定向到SSO → SSO发现已登录 → 直接返回Ticket
    8. 系统B → 用Ticket获取Token → 登录成功

  关键: SSO认证中心维护全局登录状态, 各系统通过Ticket换取Token
```

```
CAS(Central Authentication Service) — SSO经典实现

  核心概念:
    TGT(Ticket Granting Ticket) — SSO中心的全局会话凭证(Cookie)
    ST(Service Ticket) — 一次性的服务票据(用于换取Token)
    TGT存在SSO中心的Cookie中, 用户不可见

  流程:
    1. 访问app1 → 无Session → 重定向到SSO
    2. SSO检查TGT → 无TGT → 显示登录页
    3. 登录成功 → 生成TGT(Cookie存SSO) + 生成ST(URL参数回app1)
    4. app1用ST向SSO验证 → 获取用户信息 → 建立app1的Session
    5. 访问app2 → 重定向到SSO → SSO检查TGT → 有TGT → 生成ST → 回app2
    6. app2用ST验证 → 建立Session → 登录成功
```

## 6. OAuth2前端流程

### 6.1 授权码模式(Authorization Code)

```
OAuth2.0授权码模式 — 第三方登录(如微信/GitHub登录)

  ┌──────────┐                    ┌──────────┐
  │  用户     │                    │ GitHub   │
  │  浏览器   │                    │ OAuth    │
  └─────┬────┘                    └────┬─────┘
        │                              │
        │ 1. 点击"GitHub登录"           │
        │ → 重定向到GitHub授权页         │
        │   https://github.com/login/oauth/authorize
        │   ?client_id=xxx
        │   &redirect_uri=https://app.com/callback
        │   &scope=user:email
        │   &state=random_string  ← 防CSRF
        │                              │
        │ 2. 用户授权                   │
        │─────────────────────────────→│
        │                              │
        │ 3. 重定向回app, 携带code      │
        │←─────────────────────────────│
        │ https://app.com/callback?code=xxx&state=random_string
        │                              │
        │ 4. 前端把code发给后端          │
        │ POST /api/auth/github        │
        │ { code: "xxx" }              │
        │                              │
        │                   5. 后端用code换access_token
        │                   POST https://github.com/login/oauth/access_token
        │                   { client_id, client_secret, code }
        │                              │
        │                   6. 后端用access_token获取用户信息
        │                   GET https://api.github.com/user
        │                   Authorization: Bearer xxx
        │                              │
        │ 7. 后端生成JWT返回前端         │
        │←─────────────────────────────│
        │ { token: "jwt..." }          │
```

### 6.2 前端实现

```javascript
// GitHub OAuth2登录

// 1. 跳转到GitHub授权页
function loginWithGitHub() {
    const clientId = 'your_client_id';
    const redirectUri = encodeURIComponent('https://app.com/callback');
    const scope = encodeURIComponent('user:email');
    const state = generateRandomString();  // 防CSRF
    sessionStorage.setItem('oauth_state', state);

    window.location.href =
        `https://github.com/login/oauth/authorize` +
        `?client_id=${clientId}` +
        `&redirect_uri=${redirectUri}` +
        `&scope=${scope}` +
        `&state=${state}`;
}

// 2. 回调页面处理
async function handleCallback() {
    const params = new URLSearchParams(window.location.search);
    const code = params.get('code');
    const state = params.get('state');
    const savedState = sessionStorage.getItem('oauth_state');

    // 验证state防CSRF
    if (state !== savedState) {
        alert('Invalid state, possible CSRF attack');
        return;
    }
    sessionStorage.removeItem('oauth_state');

    // 把code发给后端
    try {
        const response = await fetch('/api/auth/github', {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ code })
        });
        const result = await response.json();

        if (result.code === 0) {
            localStorage.setItem('token', result.data.token);
            window.location.href = '/';
        } else {
            alert('Login failed: ' + result.message);
        }
    } catch (error) {
        console.error('OAuth callback error:', error);
    }
}

// 回调页调用
if (window.location.pathname === '/callback') {
    handleCallback();
}
```

### 6.3 PKCE扩展

```
PKCE(Proof Key for Code Exchange) — 移动端/SPA安全增强

  问题: 前端(client_id公开)无法安全存储client_secret
  解决: 用code_verifier + code_challenge代替client_secret

  流程:
    1. 前端生成随机code_verifier(43-128字符)
    2. 计算code_challenge = SHA256(code_verifier) + Base64URL
    3. 授权请求带code_challenge
    4. 换Token时带code_verifier
    5. 服务端验证: SHA256(code_verifier) == code_challenge

  适用: SPA/移动App(无后端或无法安全存储secret)
```

```javascript
// PKCE实现
async function loginWithPKCE() {
    // 1. 生成code_verifier
    const codeVerifier = generateRandomString(64);

    // 2. 计算code_challenge
    const encoder = new TextEncoder();
    const data = encoder.encode(codeVerifier);
    const hashBuffer = await crypto.subtle.digest('SHA-256', data);
    const codeChallenge = base64UrlEncode(new Uint8Array(hashBuffer));

    // 3. 存储code_verifier
    sessionStorage.setItem('code_verifier', codeVerifier);

    // 4. 跳转授权
    const params = new URLSearchParams({
        client_id: 'your_client_id',
        redirect_uri: 'https://app.com/callback',
        response_type: 'code',
        scope: 'openid profile email',
        state: generateRandomString(),
        code_challenge: codeChallenge,
        code_challenge_method: 'S256',
    });
    window.location.href = `https://auth.example.com/authorize?${params}`;
}
```

## 7. 面试题速查

**Q1: 什么是同源策略？**

```
同源 = 协议+域名+端口一致
同源策略限制: 跨域读取Cookie/DOM、跨域AJAX
目的: 防止恶意网站读取其他网站的数据
跨域方案: CORS(标准)、代理、JSONP(过时)
```

**Q2: CORS的预检请求？**

```
非简单请求(PUT/DELETE/JSON/自定义Header)触发OPTIONS预检
预检请求询问服务器是否允许该跨域
服务器返回Allow-Origin/Methods/Headers
预检通过后发送实际请求
Access-Control-Max-Age可缓存预检结果
```

**Q3: JWT存在哪里最安全？**

```
Access Token → 内存(JS变量), 短有效期(15-30分钟)
Refresh Token → HttpOnly + Secure + SameSite Cookie
不推荐LocalStorage(XSS风险)和普通Cookie(CSRF风险)
```

**Q4: SSO单点登录的原理？**

```
SSO认证中心维护全局登录状态(TGT)
各系统通过Ticket(ST)向SSO换取Token
首次访问系统A → 跳SSO登录 → 获取TGT+ST → 回A换Token
访问系统B → 跳SSO → 有TGT直接给ST → 回B换Token
用户只登录一次
```

**Q5: OAuth2授权码模式的流程？**

```
1. 前端跳转到授权服务器(带client_id+redirect_uri+state)
2. 用户授权
3. 授权服务器重定向回前端(带code)
4. 前端把code发给后端
5. 后端用code+client_secret换access_token
6. 后端用access_token获取用户信息
7. 后端生成JWT给前端
state防CSRF, PKCE增强SPA安全
```
