## 1. 零信任概述

### 传统安全模型的问题

```
传统模型（城堡护城河）：
  - 内网可信，外网不可信
  - 一旦突破边界，内网畅通无阻
  - VPN接入后即"可信"

问题：
  - 内网横向移动攻击
  - VPN凭据泄露 → 全网沦陷
  - 云原生无明确边界
```

### 零信任核心思想

```
"永不信任，始终验证"（Never Trust, Always Verify）

1. 不区分内外网 → 所有访问都需要验证
2. 不基于IP信任 → 基于身份+上下文
3. 最小权限 → 只授予完成任务所需权限
4. 持续验证 → 不是一次性认证
```

---

## 2. 核心原则

```
1. 身份为基石：用户身份+设备身份+服务身份
2. 最小权限：按需授权，定期回收
3. 微隔离：服务间默认拒绝，显式允许
4. 持续评估：实时风险评估，异常即降权
5. 加密一切：传输加密+存储加密
```

## 5. 面试要点

### Q1: 零信任和传统安全模型的区别？

```
传统：内网可信，边界防护（防火墙/VPN）
零信任：永不信任，每次访问都验证

区别：
1. 信任边界：内网vs外网 → 无信任边界
2. 认证方式：一次性登录 → 持续验证
3. 授权：基于IP/网段 → 基于身份+上下文
4. 隔离：大区隔离 → 微隔离（服务级）
```

### Q2: 微服务中如何实现零信任？

```
1. mTLS：服务间双向TLS认证（Istio自动实现）
2. NetworkPolicy：K8s网络策略，默认拒绝
3. JWT + RBAC：每次API调用验证身份和权限
4. 设备健康：检查设备是否合规（补丁/安全软件）
5. 审计日志：记录所有访问行为
```

## 6. 零信任实现方案：SPIFFE/SPIRE 详解

### 6.1 什么是 SPIFFE

```
SPIFFE（Secure Production Identity Framework for Everyone）
  - CNCF 项目，为工作负载提供统一身份标准
  - 解决问题：云原生环境中服务没有固定 IP，无法基于 IP 建立信任
  - 核心概念：每个工作负载拥有一个加密可验证的身份标识

两大规范：
  1. SPIFFE ID —— 身份标识格式
     spiffe://<trust domain>/<workload identifier>
     例：spiffe://prod.acme.com/ns/prod/sa/payment-service

  2. SVID（SPIFFE Verifiable Identity Document）—— 身份凭证
     - X.509 SVID：短有效期证书（通常 1 小时），用于 mTLS
     - JWT SVID：JWT 格式，用于跨域 API 调用
     - 自动轮转，无需人工干预
```

### 6.2 SPIRE 架构

```
SPIRE（SPIFFE Runtime Environment）—— SPIFFE 的参考实现

┌─────────────────────────────────────────────┐
│              SPIRE Server                    │
│  - 签发 SVID（CA 角色）                       │
│  - 管理信任域（Trust Domain）                 │
│  - 注册节点和工作负载身份映射                  │
│  - 默认监听 8081（HTTPS）/ 8080（GRPC）        │
└──────────────┬──────────────────────────────┘
               │ Node attestation + Workload attestation
┌──────────────┴──────────────────────────────┐
│            SPIRE Agent（每个节点一个）          │
│  - 从 Server 拉取 SVID 并缓存                  │
│  - 通过 Workload API 向工作负载分发 SVID        │
│  - 监听 Unix Domain Socket：/tmp/spire/...     │
│  - 自动轮转即将过期的证书                       │
└─────────────────────────────────────────────┘
```

### 6.3 工作负载注册与认证流程

```
1. 节点认证（Node Attestation）
   Agent 启动时向 Server 证明节点身份
   支持的认证方式：K8s Node Attestor / AWS IID / GCP / TPM

2. 工作负载认证（Workload Attestation）
   Agent 通过选择器（Selector）识别工作负载
   K8s 选择器示例：
     - spiffe_id: spiffe://acme.com/ns/prod/sa/payment
     - selectors:
       - k8s:ns:prod
       - k8s:sa:payment-service
       - k8s:pod-label:app:payment

3. SVID 分发
   工作负载通过 Workload API（gRPC over Unix Socket）获取 SVID
   - 无需挂载证书文件
   - 无需重启即可获取新证书
   - 证书过期前自动刷新
```

### 6.4 SPIRE 部署示例（K8s）

```yaml

kind: Deployment
metadata:

        country = ["CN"]

    }
```

## 7. 微服务 mTLS 实战（Istio 配置）

### 7.1 Istio mTLS 工作原理

```
Istio 通过 Sidecar（Envoy）代理实现服务间 mTLS：

  Service A Pod              Service B Pod
  ┌──────────────┐           ┌──────────────┐
  │  App A       │           │  App B       │
  │  ↓ HTTP      │           │  ↑ HTTP      │
  │  Envoy Sidecar│ ←mTLS→  │  Envoy Sidecar│
  └──────────────┘           └──────────────┘

特点：
  - 应用无感知：业务代码仍是明文 HTTP，Envoy 自动升级为 mTLS
  - 证书自动管理：集成 SPIRE 或内置 CA，自动签发轮转
  - 策略灵活：可全局 PERMISSIVE（渐进迁移）或 STRICT（强制）
```

### 7.2 启用 mTLS：PeerAuthentication

```yaml
# 方式一：命名空间级别——STRICT 模式（全量强制 mTLS）
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production     # 对 production 命名空间生效
spec:
  mtls:
    mode: STRICT            # 拒绝所有非 mTLS 流量

---
# 方式二：PERMISSIVE 模式（迁移期使用，同时接受明文和 mTLS）
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: staging
spec:
  mtls:
    mode: PERMISSIVE        # 平滑过渡：旧客户端不受影响

---
# 方式三：端口级别覆盖（特定端口关闭 mTLS）
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: payment-mtls
  namespace: production
spec:
  selector:
    matchLabels:
      app: payment-service
  mtls:
    mode: STRICT
  portLevelMtls:
    8080:                   # 健康检查端口允许明文
      mode: DISABLE         # K8s liveness probe 不走 mTLS
```

### 7.3 授权策略：AuthorizationPolicy

```yaml
# 只允许 order-service 访问 payment-service 的 /api/v1/pay
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: payment-authz
  namespace: production
spec:
  selector:
    matchLabels:
      app: payment-service
  action: ALLOW
  rules:
    - from:
        - source:
            principals: ["cluster.local/ns/production/sa/order-service"]
      to:
        - operation:
            methods: ["GET", "POST"]
            paths: ["/api/v1/pay*"]
    # 允许监控抓取
    - from:
        - source:
            namespaces: ["monitoring"]
      to:
        - operation:
            paths: ["/metrics"]
```

### 7.4 渐进式迁移实战步骤

```
步骤 1：部署 Istio 并开启 PERMISSIVE 模式
  → 现有流量不受影响，新流量自动 mTLS

步骤 2：监控指标，确认所有客户端已完成 Sidecar 注入
  → istioctl experimental authz check <pod>
  → 查看 Grafana Istio 面板的 mTLS 流量占比

步骤 3：逐步切换为 STRICT 模式
  → 先对非核心服务切换，观察 1-2 天
  → 再对核心服务切换

步骤 4：添加 AuthorizationPolicy（默认拒绝 → 显式允许）
  → 先 PERMISSIVE（只记日志不拦截），确认无误
  → 再改为 ALLOW 强制执行

常见坑：
  - Sidecar 未注入的 Pod 无法与 STRICT 模式通信
  - 健康检查端口需要 portLevelMtls DISABLE
  - 外部入口流量需通过 Ingress Gateway 终止 TLS
```

## 8. K8s NetworkPolicy 微隔离

### 8.1 NetworkPolicy 基本原理

```
默认行为：
  - K8s 集群中 Pod 默认网络是全通的（任何 Pod 可访问任何 Pod）
  - 一旦对某 Pod 定义了 NetworkPolicy（Ingress/Egress），则默认拒绝
  - 需要依赖支持 NetworkPolicy 的 CNI 插件：
    Calico / Cilium / Weave Net / Antrea

三大维度：
  - podSelector：选择目标 Pod（受策略保护的 Pod）
  - namespaceSelector：按命名空间选择来源/目标
  - ipBlock：按 CIDR 选择外部 IP 段
```

### 8.2 典型微隔离策略

```yaml
# 策略 1：默认拒绝所有入口流量（基线策略）
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}           # 空选择器 = 命名空间所有 Pod
  policyTypes:
    - Ingress
  ingress: []               # 空 = 拒绝所有入口

---
# 策略 2：只允许 order → payment 的 8080 端口
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-order-to-payment
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: payment
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: order
      ports:
        - protocol: TCP
          port: 8080

---
# 策略 3：允许监控系统命名空间抓取 metrics
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-monitoring-scrape
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: payment
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              team: monitoring
      ports:
        - protocol: TCP
          port: 9090
```

### 8.3 出口流量控制（Egress）

```yaml
# 限制 payment 服务只能访问 DB 和外部支付网关
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: payment-egress
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: payment
  policyTypes:
    - Egress
  egress:
    # 允许访问集群内 DNS
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
    # 允许访问数据库 Pod
    - to:
        - podSelector:
            matchLabels:
              app: mysql
      ports:
        - protocol: TCP
          port: 3306
    # 允许访问外部支付网关 IP（白名单）
    - to:
        - ipBlock:
            cidr: 203.0.113.50/32
      ports:
        - protocol: TCP
          port: 443
```

### 8.4 NetworkPolicy vs Service Mesh 授权

```
                NetworkPolicy          Istio AuthorizationPolicy
工作层级         L3/L4（IP/端口）       L7（HTTP 方法/路径/Header）
策略粒度         Pod 间网络可达          API 级别细粒度控制
加密支持         无（仅访问控制）         有（mTLS）
默认拒绝         是（定义后默认拒绝）     否（需显式 ALLOW）
性能开销         低（内核 eBPF/iptables） 较高（Sidecar 代理）
最佳实践         两者配合使用：NetworkPolicy 做网络层隔离，
                Istio 做应用层授权 + 加密
```

## 9. 持续信任评估机制

### 9.1 信任不是一次性的

```
传统认证：登录一次 → 获得 Token → Token 有效期内完全信任
零信任：持续评估 → 信任分数实时变化 → 异常即降权/断开

信任评分（Trust Score）输入因子：
  ┌──────────────────────────────────────┐
  │ 身份因子    │ 用户身份、角色、MFA 状态  │
  │ 设备因子    │ 设备指纹、合规性、补丁状态  │
  │ 行为因子    │ 访问时间、地点、频率       │
  │ 上下文因子  │ 网络、会话、请求模式       │
  │ 威胁情报    │ IP 信誉、已知攻击特征      │
  └──────────────────────────────────────┘
```

### 9.2 风险评分与动态授权

```java
// 信任评估服务示例
@Service
public class TrustEvaluationService {

    @Autowired
    private RiskEngine riskEngine;
    @Autowired
    private DeviceHealthService deviceHealthService;
    @Autowired
    private UserBehaviorAnalyzer behaviorAnalyzer;

    /**
     * 实时计算信任分数（0-100）
     * > 80：正常访问
     * 50-80：需要二次验证（MFA）
     * < 50：拒绝访问 / 降级为只读
     */
    public TrustResult evaluate(AuthContext ctx) {
        int score = 100;

        // 1. 设备健康检查（-20 分）
        if (!deviceHealthService.isCompliant(ctx.getDeviceId())) {
            score -= 20;
        }

        // 2. 异常行为检测（-30 分）
        if (behaviorAnalyzer.isAnomalous(ctx.getUserId(), ctx.getIp())) {
            score -= 30;  // 异地登录/异常时间/异常频率
        }

        // 3. IP 信誉检查（-25 分）
        if (riskEngine.isIpBlacklisted(ctx.getIp())) {
            score -= 25;
        }

        // 4. MFA 状态（+10 分补偿）
        if (ctx.hasMfaVerified()) {
            score += 10;
        }

        TrustLevel level = determineLevel(score);
        return new TrustResult(score, level, getSuggestedAction(level));
    }

    private TrustLevel determineLevel(int score) {
        if (score >= 80) return TrustLevel.HIGH;
        if (score >= 50) return TrustLevel.MEDIUM;
        return TrustLevel.LOW;
    }
}

// 网关层拦截，每次请求都评估
@Component
public class TrustFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        AuthContext ctx = extractContext(exchange);
        TrustResult result = trustService.evaluate(ctx);

        switch (result.getLevel()) {
            case HIGH:
                return chain.filter(exchange);              // 正常放行
            case MEDIUM:
                // 要求二次验证
                return requireStepUpAuth(exchange);
            case LOW:
                // 拒绝访问或降级
                return denyOrDegrade(exchange, result);
        }
    }
}
```

### 9.3 持续评估的关键信号

```
实时降权触发条件：
  1. 会话期间 IP 地址突变（可能的会话劫持）
  2. 请求频率激增（可能的自动化攻击/爬虫）
  3. 访问敏感资源超出角色权限范围
  4. 设备安全状态恶化（新漏洞/补丁过期）
  5. UEBA 检测到行为模式偏离基线

响应动作（梯度响应）：
  低风险 → 记录日志 + 静默观察
  中风险 → 要求 Step-up 认证（短信/TOTP）
  高风险 → 降级权限（只读模式）
  极高风险 → 立即终止会话 + 通知安全团队
```

## 10. 零信任成熟度模型

### 10.1 CISA 零信任成熟度模型（美国网络安全与基础设施安全局）

```
五大支柱（Pillar）：
  1. Identity（身份）       —— 统一身份管理、MFA、条件访问
  2. Devices（设备）        —— 设备清单、合规检测、健康证明
  3. Networks（网络）       —— 微隔离、加密、软件定义边界
  4. Applications & Workloads（应用与工作负载）—— API 安全、mTLS
  5. Data（数据）           —— 数据分类、DLP、加密

三个横切能力（Cross-Cutting）：
  - Visibility & Analytics（可见性与分析）
  - Automation & Orchestration（自动化与编排）
  - Governance（治理）
```

### 10.2 四级成熟度评估

```
┌──────────┬───────────────────────────────────────────────┐
│ 级别      │ 特征描述                                      │
├──────────┼───────────────────────────────────────────────┤
│ Traditional │ 传统模式：依赖边界防护，VPN+防火墙           │
│          │ 无统一身份，无微隔离，无持续评估                 │
├──────────┼───────────────────────────────────────────────┤
│ Initial  │ 初始零信任：开始实施 MFA，部分系统接入 SSO      │
│          │ 有基本的访问日志，但无实时风险评估               │
├──────────┼───────────────────────────────────────────────┤
│ Advanced │ 进阶零信任：统一身份，设备健康检查               │
│          │ 微隔离部分覆盖，有 SIEM 集成                   │
│          │ 策略半自动化，有条件访问                        │
├──────────┼───────────────────────────────────────────────┤
│ Optimal  │ 最优零信任：全量 mTLS + 微隔离                  │
│          │ 持续信任评估，自动化响应                        │
│          │ 策略即代码，全链路可观测                        │
└──────────┴───────────────────────────────────────────────┘
```

### 10.3 成熟度自评检查清单

```
Identity（身份）：
  [ ] 所有用户使用统一 SSO（OIDC/SAML）
  [ ] 关键系统强制 MFA
  [ ] 服务身份使用 SPIFFE/SVID（非静态密钥）
  [ ] 实施条件访问策略

Devices（设备）：
  [ ] 有完整设备清单（MDM 管理）
  [ ] 设备健康检查（补丁/加密/安全软件）
  [ ] 不合规设备无法访问资源

Networks（网络）：
  [ ] K8s NetworkPolicy 默认拒绝
  [ ] 服务间 mTLS 全覆盖
  [ ] 无 VPN 全网可达，采用 SDP/应用级网关

Applications（应用）：
  [ ] API 有认证和授权（非匿名）
  [ ] 敏感操作有审计日志
  [ ] 密钥/证书自动轮转

Data（数据）：
  [ ] 数据分级分类
  [ ] 传输和存储全加密
  [ ] DLP 防泄漏策略
```

## 11. 面试要点补充

### Q3: SPIFFE/SPIRE 解决了什么问题？和传统 PKI 有什么区别？

```
解决问题：云原生环境中服务身份认证
  - 容器 IP 不固定，无法基于 IP 建立信任
  - 传统 PKI 证书管理复杂，需人工签发/轮转
  - 多云/混合云中无统一身份标准

与传统 PKI 的区别：
  1. 身份标识：CN/SAN（域名） → SPIFFE ID（服务身份语义化）
  2. 证书有效期：1 年 → 1 小时（短有效期降低泄露风险）
  3. 轮转方式：人工/脚本 → Workload API 自动轮转
  4. 认证粒度：机器/域名 → 工作负载级别（每个 Pod 独立身份）
  5. 适用场景：传统数据中心 → 云原生/服务网格
```

### Q4: 如何在生产环境中渐进式实施零信任？

```
分阶段推进（不要 Big Bang）：

第一阶段：身份统一
  → 全员 SSO + MFA，消灭共享账号
  → 服务身份引入 SPIFFE/SVID

第二阶段：网络隔离
  → K8s NetworkPolicy 默认拒绝（先观察模式）
  → Istio mTLS PERMISSIVE 模式（不阻断现有流量）

第三阶段：策略收紧
  → mTLS 切换 STRICT（逐命名空间）
  → AuthorizationPolicy 从 PERMISSIVE → ALLOW
  → NetworkPolicy 逐个白名单

第四阶段：持续评估
  → 接入 SIEM/UEBA，实时风险评分
  → 条件访问策略（异常即降权）
  → 全链路审计日志 + 自动化告警

关键原则：先可观测、后阻断；先非核心、后核心服务
```

### Q5: 零信任和 Service Mesh 是什么关系？

```
关系：Service Mesh 是零信任在微服务领域的关键落地工具

Service Mesh 提供的零信任能力：
  1. 身份：SPIFFE 集成，每个 Pod 有 SVID 身份
  2. 加密：Sidecar 自动 mTLS，应用无感知
  3. 授权：AuthorizationPolicy 细粒度 L7 策略
  4. 可观测：全链路遥测，支撑持续评估

但 Service Mesh ≠ 零信任：
  - Service Mesh 只覆盖东西向（服务间）流量
  - 南北向（用户→服务）流量还需 API 网关/SDP
  - 用户身份/设备健康/行为分析需额外系统
  - 数据层加密/DLP 不在 Service Mesh 范畴

总结：Service Mesh 是零信任的重要拼图，但不是全部。
      零信任是安全理念，Service Mesh 是实现手段之一。
```
