## 1. 零信任概述

### 1.1 传统安全模型的问题

```
传统模型（城堡护城河）：
  - 内网可信，外网不可信
  - 一旦突破边界，内网畅通无阻
  - VPN接入后即"可信"

问题：
  - 内网横向移动攻击
  - VPN凭据泄露 → 全网沦陷
  - 云原生无明确边界
  - 远程办公模糊了内外网边界
  - 第三方供应链攻击
  - 内部威胁（Insider Threat）
```

### 1.2 零信任核心思想

```
"永不信任，始终验证"（Never Trust, Always Verify）

1. 不区分内外网 → 所有访问都需要验证
2. 不基于IP信任 → 基于身份+上下文
3. 最小权限 → 只授予完成任务所需权限
4. 持续验证 → 不是一次性认证
```

### 1.3 零信任演进历程

```
2010年: John Kindervag (Forrester) 首次提出零信任概念
  ↓
2014年: Google发布BeyondCorp论文，实践零信任
  ↓
2017年: Gartner提出SDP（软件定义边界）和持续自适应风险与信任评估
  ↓
2020年: NIST发布SP 800-207零信任架构标准
  ↓
2021年: 美国政府行政令要求联邦机构实施零信任
  ↓
2023年: 零信任成为企业安全建设的核心战略
```

## 2. 核心原则

```
1. 身份为基石：用户身份+设备身份+服务身份
2. 最小权限：按需授权，定期回收
3. 微隔离：服务间默认拒绝，显式允许
4. 持续评估：实时风险评估，异常即降权
5. 加密一切：传输加密+存储加密
```

### 2.1 深入理解核心原则

```
┌──────────────────────────────────────────────────────┐
│                  零信任五大核心原则                     │
├──────────────────────────────────────────────────────┤
│                                                        │
│  1. 显式验证（Explicit Verification）                  │
│     - 每次访问都验证：身份、设备、上下文                 │
│     - 不依赖网络位置（IP/网段）                         │
│     - 多因素认证（MFA）+ 风险评分                       │
│                                                        │
│  2. 最小权限（Least Privilege）                        │
│     - Just-In-Time（JIT）：按需临时授权                 │
│     - Just-Enough-Access（JEA）：仅授予必要的权限       │
│     - 定期审查和回收权限                               │
│                                                        │
│  3. 假设被入侵（Assume Breach）                        │
│     - 假设攻击者已在内部                                │
│     - 分段隔离，限制爆炸半径                            │
│     - 持续监控和检测异常                                │
│                                                        │
│  4. 持续评估（Continuous Assessment）                  │
│     - 用户行为分析（UBA）                              │
│     - 设备健康状态实时检查                              │
│     - 风险变化时动态调整权限                            │
│                                                        │
│  5. 加密所有数据（Encrypt Everything）                 │
│     - 传输加密：mTLS / TLS 1.3                         │
│     - 存储加密：磁盘加密 + 应用层加密                    │
│     - 零知识证明：敏感操作不暴露明文                     │
└──────────────────────────────────────────────────────┘
```

## 3. 架构组件（PE/PA/PEP）

### 3.1 零信任架构组件模型

```
┌──────────────────────────────────────────────────────┐
│                  NIST 800-207 零信任架构                │
├──────────────────────────────────────────────────────┤
│                                                        │
│  ┌─────────────────┐  主体（Subject）                   │
│  │  User / Device  │──┐                                │
│  │  / Service      │  │                                │
│  └─────────────────┘  │                                │
│                       ▼                                │
│  ┌───────────────────────────────────────────────┐    │
│  │          Policy Engine (PE) 策略引擎            │    │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────────┐  │    │
│  │  │ 身份信息  │ │ 设备状态  │ │ 行为分析      │  │    │
│  │  │ (IdP)    │ │ (MDM)    │ │ (SIEM/UEBA)  │  │    │
│  │  └──────────┘ └──────────┘ └──────────────┘  │    │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────────┐  │    │
│  │  │ 威胁情报  │ │ 合规策略  │ │ 风险评分      │  │    │
│  │  │ (TI)     │ │ (Policy) │ │ (Risk Score) │  │    │
│  │  └──────────┘ └──────────┘ └──────────────┘  │    │
│  │                                                │    │
│  │  → 综合评估 → 输出信任决策                      │    │
│  └───────────────────┬───────────────────────────┘    │
│                      │ 授权决策                         │
│  ┌───────────────────▼───────────────────────────┐    │
│  │       Policy Administrator (PA) 策略管理员      │    │
│  │  → 接收PE的决策                                │    │
│  │  → 生成令牌/会话凭证                            │    │
│  │  → 下发策略到PEP                               │    │
│  └───────────────────┬───────────────────────────┘    │
│                      │ 策略下发                         │
│  ┌───────────────────▼───────────────────────────┐    │
│  │    Policy Enforcement Point (PEP) 策略执行点    │    │
│  │  → 网关/代理/Sidecar                           │    │
│  │  → 拦截请求 → 验证凭证 → 执行访问控制            │    │
│  │  → 允许/拒绝/降级                              │    │
│  └───────────────────┬───────────────────────────┘    │
│                      │                                 │
│                      ▼                                 │
│  ┌───────────────────────────────────────────────┐    │
│  │             企业资源（Resource）                │    │
│  │  Application / API / Database / Service       │    │
│  └───────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────┘
```

### 3.2 三大组件详解

```
┌──────────────────────────────────────────────────────┐
│  Policy Engine (PE) - 策略引擎                        │
├──────────────────────────────────────────────────────┤
│  职责：做出信任决策                                    │
│  输入：                                                │
│    - 用户身份（Who）：IdP提供的认证信息                │
│    - 设备状态（What）：MDM/EDR提供的设备健康度          │
│    - 上下文（Where/When）：IP、时间、地理位置           │
│    - 行为模式（How）：UEBA的行为分析结果               │
│    - 威胁情报：SIEM的威胁检测信号                      │
│  输出：                                                │
│    - 允许（Allow）                                     │
│    - 拒绝（Deny）                                      │
│    - 允许但需附加认证（Allow + Step-up Auth）          │
│    - 允许但降级（Allow + Reduced Access）              │
│  实现：OPA（Open Policy Agent）、自定义决策引擎         │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│  Policy Administrator (PA) - 策略管理员               │
├──────────────────────────────────────────────────────┤
│  职责：根据PE的决策生成和分发凭证                      │
│  功能：                                                │
│    - 生成短期令牌（Short-lived Token）                 │
│    - 建立会话（Session）                              │
│    - 向PEP下发访问策略                                │
│    - 管理会话生命周期（超时、撤销）                    │
│  实现：Auth0/Keycloak/Okcita + 自定义PA               │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│  Policy Enforcement Point (PEP) - 策略执行点          │
├──────────────────────────────────────────────────────┤
│  职责：在每个资源访问点执行访问控制                     │
│  形态：                                                │
│    - 反向代理（Nginx/Envoy + Auth模块）               │
│    - API网关（Kong/APISIX + 插件）                    │
│    - Sidecar代理（Istio Envoy）                       │
│    - 零信任网络访问网关（ZTNA Gateway）                │
│  功能：                                                │
│    - 拦截所有请求                                     │
│    - 验证令牌/会话                                     │
│    - 执行PE/PA的决策                                   │
│    - 记录审计日志                                      │
└──────────────────────────────────────────────────────┘
```

### 3.3 工作流程示例

```
用户访问内部API的完整流程：

  1. 用户发起请求 → PEP拦截
  2. PEP检查请求中是否有有效令牌
     - 无令牌 → 重定向到IdP认证
     - 有令牌 → 验证令牌签名和有效期
  3. PEP向PE发送评估请求
     - 用户身份: user_id=alice, role=developer
     - 请求资源: api://order-service/api/orders
     - 设备状态: device_id=xxx, compliance=trusted
     - 上下文: ip=10.0.1.5, time=14:30, location=office
  4. PE综合评估
     - 身份验证通过 ✓
     - 设备合规 ✓
     - 时间在工作时间 ✓
     - 角色有权限访问 ✓
     - 威胁情报无异常 ✓
     - 风险评分: 低 → 决策=Allow
  5. PA生成短期令牌（5分钟有效）
  6. PEP允许请求通过，附加令牌到请求头
  7. 请求到达后端服务

异常场景：
  - 设备不合规 → PE决策=Deny → PEP拒绝
  - 非工作时间访问 → PE决策=Step-up → 要求MFA二次认证
  - 异常IP登录 → PE决策=Allow+Reduced → 只允许只读操作
```

## 4. 设备信任评估

### 4.1 设备信任模型

```
┌──────────────────────────────────────────────┐
│              设备信任评估维度                   │
├──────────────────────────────────────────────┤
│                                                │
│  1. 设备身份（Device Identity）                 │
│     - 设备证书（X.509 / SCEP）                  │
│     - 设备指纹（硬件特征）                       │
│     - 设备注册状态（已注册/BYOD）               │
│                                                │
│  2. 设备合规性（Device Compliance）             │
│     - 操作系统版本和补丁状态                    │
│     - 杀毒软件/EDR是否运行                      │
│     - 磁盘加密状态（BitLocker/FileVault）      │
│     - 屏幕锁是否启用                            │
│     - 越狱/Root状态                             │
│                                                │
│  3. 设备健康度（Device Health）                 │
│     - EDR实时健康分                             │
│     - 最近漏洞扫描结果                          │
│     - 安全基线配置评分                          │
│                                                │
│  4. 设备行为（Device Behavior）                 │
│     - 异常网络连接                              │
│     - 可疑进程活动                              │
│     - 数据外泄迹象                              │
└──────────────────────────────────────────────┘
```

### 4.2 设备信任评估实现

```yaml
# MDM + EDR集成的设备信任策略（OPA Rego）
package zero_trust.device

# 设备信任评分计算
default allow = false

allow {
    device.registered
    device.compliant
    device.health_score > 70
    not device.jailbroken
    not device.threat_detected
}

# 设备合规检查
device.compliant {
    device.os_version >= "14.0"
    device.disk_encrypted
    device.screen_lock_enabled
    device.antivirus_running
}

# 信任级别
trust_level := "high" {
    device.health_score > 90
    device.compliant
} else := "medium" {
    device.health_score > 70
    device.compliant
} else := "low" {
    device.health_score > 50
} else := "untrusted"

# 根据信任级别决定访问策略
access_decision := "allow" {
    trust_level == "high"
} else := "allow_readonly" {
    trust_level == "medium"
} else := "require_mfa" {
    trust_level == "low"
} else := "deny"
```

## 5. 微隔离

### 5.1 微隔离概念

```
传统网络隔离（VLAN/子网）：
  ┌─────────────────────────────────────────┐
  │  DMZ区: Web服务器                         │
  │  ┌───────────────────────────────────┐  │
  │  │ Web1 ←→ Web2 ←→ Web3              │  │
  │  └───────────────────────────────────┘  │
  │  内网区: 应用服务器                       │
  │  ┌───────────────────────────────────┐  │
  │  │ App1 ←→ App2 ←→ App3 ←→ DB       │  │
  │  └───────────────────────────────────┘  │
  │  → 同区互通，跨区防火墙控制               │
  │  → 问题：同区内任何服务可互相访问          │
  └─────────────────────────────────────────┘

微隔离（Micro-segmentation）：
  ┌─────────────────────────────────────────┐
  │  每个服务/Pod独立隔离                      │
  │                                          │
  │  Web1 ←✗→ Web2  (默认拒绝)               │
  │  Web1 ←✓→ App1 (显式允许)               │
  │  App1 ←✗→ App2  (默认拒绝)               │
  │  App1 ←✓→ DB   (显式允许)               │
  │  App2 ←✗→ DB   (默认拒绝)               │
  │                                          │
  │  → 爆炸半径最小化                         │
  │  → 纵向+横向全控制                        │
  └─────────────────────────────────────────┘
```

### 5.2 微隔离实现方式

```yaml
# 方式1: K8s NetworkPolicy
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: payment-default-deny
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: payment
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: order    # 只允许order访问payment
      ports:
        - port: 8080
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: mysql     # payment只能访问mysql
      ports:
        - port: 3306
    - to:
        - namespaceSelector:
            matchLabels:
              name: kube-system  # 允许DNS
      ports:
        - port: 53
          protocol: UDP
---
# 方式2: Istio AuthorizationPolicy（L7层隔离）
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: payment-authz
  namespace: production
spec:
  selector:
    matchLabels:
      app: payment
  rules:
    - from:
        - source:
            principals: ["cluster.local/ns/production/sa/order-sa"]
      to:
        - operation:
            methods: ["GET", "POST"]
            paths: ["/api/payment/*"]
```

### 5.3 微隔离策略设计

```
微隔离策略设计原则：
  1. 默认拒绝（Default Deny）：所有流量默认拒绝，显式允许
  2. 最小连接：只开放必要的端口和路径
  3. 身份基础：基于服务身份而非IP
  4. 分层隔离：
     - L3/L4: NetworkPolicy（IP+端口）
     - L7: Istio AuthorizationPolicy（HTTP方法+路径）
  5. 持续观察：先观察模式，再切换到强制模式

实施步骤：
  Step 1: 流量观察（Monitor Mode）
    → 收集服务间调用关系
    → 生成依赖图
  Step 2: 策略生成
    → 基于观察数据生成白名单策略
    → 人工审核和调整
  Step 3: 试运行（Permissive Mode）
    → 策略生效但不拒绝，只记录告警
    → 验证策略不阻断正常业务
  Step 4: 强制执行（Enforce Mode）
    → 切换为拒绝模式
    → 持续监控和调优
```

## 6. 软件定义边界（SDP）

### 6.1 SDP架构

```
SDP (Software Defined Perimeter) 软件定义边界

传统VPN的问题：
  - 接入后获得整个网络访问权
  - 网络层信任，缺乏应用层控制
  - VPN网关是单点故障和攻击目标

SDP架构：
  ┌──────────────────────────────────────────────────────┐
│                   SDP 架构                              │
├──────────────────────────────────────────────────────┤
│                                                        │
│  ┌──────────────┐                                      │
│  │  SDP Client   │  (Initiating Host, IH)              │
│  │  - 设备认证   │                                      │
│  │  - 用户认证   │                                      │
│  │  - SPA握手    │                                      │
│  └──────┬───────┘                                      │
│         │ 1. 认证请求                                    │
│         ▼                                              │
│  ┌──────────────┐                                      │
│  │  SDP Controller │  (Policy Decision Point)         │
│  │  - 身份验证     │                                    │
│  │  - 设备验证     │                                    │
│  │  - 策略评估     │                                    │
│  │  - 下发授权     │                                    │
│  └──┬────────┬───┘                                     │
│     │ 2. 授权 │ 3. 下发策略                              │
│     ▼        ▼                                         │
│  ┌──────────────┐                                      │
│  │  SDP Gateway  │  (Accepting Host, AH)               │
│  │  - 默认隐身   │  (所有端口关闭，不响应未认证请求)       │
│  │  - 验证令牌   │                                      │
│  │  - 代理转发   │                                      │
│  └──────┬───────┘                                      │
│         │ 4. 建立连接                                    │
│         ▼                                              │
│  ┌──────────────┐                                      │
│  │  受保护的应用  │                                      │
│  └──────────────┘                                      │
└──────────────────────────────────────────────────────┘
```

### 6.2 SPA（Single Packet Authorization）

```
SPA单包授权机制：

  1. SDP Gateway默认隐身
     - 所有端口关闭
     - 不响应任何探测（ping/port scan）
     - 外部看起来像"不存在"的服务器

  2. SDP Client发送SPA包
     - 单个UDP包（256字节）
     - 包含加密的认证信息（设备ID+用户ID+时间戳+HMAC）
     - 使用预共享密钥加密

  3. SDP Gateway验证SPA包
     - 解密验证
     - 检查时间戳防重放
     - 验证通过 → 为该Client临时打开端口

  4. 建立mTLS连接
     - 端口打开后，Client建立mTLS连接
     - 双向证书验证
     - 加密通道传输数据

优势：
  - 服务隐身：攻击者无法发现攻击面
  - 抗DDoS：未认证流量不消耗资源
  - 抗端口扫描：端口默认关闭
```

### 6.3 SDP vs VPN

```
| 维度          | 传统VPN              | SDP                  |
|---------------|---------------------|----------------------|
| 信任模型      | 网络层信任           | 身份+设备+上下文      |
| 访问粒度      | 网络级（子网/路由）  | 应用级（HTTP/API）   |
| 连接方式      | 隧道（全流量）       | 按需连接（单应用）    |
| 服务可见性    | 接入后可扫描内网     | 服务隐身，不可发现    |
| 认证频率      | 一次认证持续访问     | 持续验证             |
| 扩展性        | 集中式VPN网关瓶颈   | 分布式Gateway        |
| 远程办公      | 全隧道流量过VPN     | 按应用分流           |
```

## 7. BeyondCorp案例

### 7.1 BeyondCorp概述

```
Google BeyondCorp = 零信任的工业级实践

核心思想：
  "不存在内网，所有访问都经过认证和授权"

关键理念：
  1. 取消VPN，所有访问通过反向代理
  2. 设备和用户均需注册和认证
  3. 访问决策基于设备+用户+上下文
  4. 每次请求都验证，而非一次性登录

BeyondCorp组件：
  ┌──────────────────────────────────────────────┐
  │  BeyondCorp 架构组件                          │
  ├──────────────────────────────────────────────┤
  │  设备清单服务（Device Inventory）              │
  │  - 跟踪所有设备的清单和状态                    │
  │  - 设备指纹和证书管理                         │
  │  - 设备信任评分                               │
  │                                                │
  │  设备证书颁发（Certificate Authority）         │
  │  - 每台设备有唯一证书                         │
  │  - 证书定期轮转                               │
  │  - 证书用于mTLS认证                           │
  │                                                │
  │  信任推断服务（Trust Inferer）                 │
  │  - 综合设备+用户+上下文计算信任分              │
  │  - 实时更新信任级别                           │
  │                                                │
  │  访问代理（Access Proxy）                     │
  │  - 所有应用的前置反向代理                      │
  │  - 验证用户和设备                             │
  │  - 执行访问控制策略                            │
  │  - 终止TLS，代理到后端                         │
  │                                                │
  │  身份提供者（Identity Provider）               │
  │  - 用户认证（MFA/SAML/OAuth）                 │
  │  - 用户身份和组信息                            │
  └──────────────────────────────────────────────┘
```

### 7.2 BeyondCorp访问流程

```
用户访问内部应用流程：

  1. 用户通过浏览器访问 app.corp.google.com
  2. Access Proxy拦截请求
  3. 检查请求中的Cookie和设备证书
     - 无有效凭证 → 重定向到身份认证
     - 有凭证 → 验证凭证
  4. Trust Inferer评估信任
     - 用户信任分: 85/100
     - 设备信任分: 92/100
     - 上下文: 办公网络，工作时间
     - 综合信任级别: Sufficient
  5. 策略引擎匹配
     - 应用: internal-app
     - 要求信任级别: Sufficient
     - 用户权限: has access
     - 决策: Allow
  6. Access Proxy代理请求到后端应用
  7. 用户获得应用访问权限

持续验证：
  - 每个HTTP请求都经过Access Proxy验证
  - 信任分实时变化（如设备被标记不合规 → 降权）
  - 会话超时后需重新认证
```

## 8. NIST 800-207框架

### 8.1 NIST SP 800-207 核心要求

```
NIST SP 800-207 零信任架构标准

七大基本原则：
  1. 所有数据源和计算服务都视为资源
  2. 无论网络位置，所有通信都必须安全
  3. 对单个企业资源的访问授权基于每个会话
  4. 访问决策是动态的，在访问请求之前确定
  5. 企业监控和测量所有资产的完整性和安全状态
  6. 所有资源认证和授权是动态的，在允许访问前严格 enforced
  7. 企业收集尽可能多的资产、网络、通信状态信息以改进安全态势

零信任架构部署模型：
  ┌──────────────────────────────────────────────┐
  │  模型1: 基于设备代理                         │
  │  - 每台设备部署Agent                         │
  │  - Agent负责认证和策略执行                    │
  │  - 适合：企业自有设备                         │
  ├──────────────────────────────────────────────┤
  │  模型2: 基于沙箱                             │
  │  - 资源在沙箱中运行                          │
  │  - 沙箱控制所有进出流量                       │
  │  - 适合：BYOD场景                            │
  ├──────────────────────────────────────────────┤
  │  模型3: 基于资源门户                         │
  │  - 所有访问通过统一门户                       │
  │  - 门户负责认证和代理                         │
  │  - 适合：Web应用访问                         │
  ├──────────────────────────────────────────────┤
  │  模型4: 混合模式                             │
  │  - 多种模式组合使用                           │
  │  - 根据场景灵活选择                           │
  │  - 适合：复杂企业环境                         │
  └──────────────────────────────────────────────┘
```

### 8.2 实施路线图

```
零信任实施四阶段：

  阶段一：身份与访问管理（Identity）
    - 统一身份管理（SSO + MFA）
    - 设备注册和证书管理
    - 最小权限策略定义
    → 产出：身份基线、设备清单、策略草案

  阶段二：网络与微隔离（Network）
    - NetworkPolicy默认拒绝
    - 服务间mTLS
    - 分段隔离
    → 产出：网络隔离策略、流量可视化

  阶段三：应用与数据保护（Application）
    - 所有应用通过PEP访问
    - 动态访问决策
    - 数据加密和分类
    → 产出：零信任网关、数据保护策略

  阶段四：持续监控与优化（Operations）
    - UEBA行为分析
    - 实时风险评分
    - 策略持续调优
    → 产出：自动化响应、风险仪表盘
```

## 9. 面试要点

### Q1: 零信任和传统安全模型的区别？

```
传统：内网可信，边界防护（防火墙/VPN）
零信任：永不信任，每次访问都验证

区别：
1. 信任边界：内网vs外网 → 无信任边界
2. 认证方式：一次性登录 → 持续验证
3. 授权：基于IP/网段 → 基于身份+上下文
4. 隔离：大区隔离 → 微隔离（服务级）
```

### Q2: PE、PA、PEP分别是什么？

```
PE (Policy Engine) 策略引擎：
  - 做出信任决策（Allow/Deny/Step-up）
  - 综合评估身份、设备、上下文、行为

PA (Policy Administrator) 策略管理员：
  - 根据PE的决策生成凭证
  - 管理会话生命周期

PEP (Policy Enforcement Point) 策略执行点：
  - 拦截请求，验证凭证
  - 执行PE/PA的决策
  - 形态：反向代理、API网关、Sidecar
```

### Q3: 微隔离如何实现？

```
实现层次：
  L3/L4: K8s NetworkPolicy（IP+端口级隔离）
  L7: Istio AuthorizationPolicy（HTTP方法+路径级隔离）

实施步骤：
  1. 观察模式 → 收集服务调用关系
  2. 策略生成 → 基于观察数据生成白名单
  3. 试运行 → 记录但不拒绝
  4. 强制执行 → 切换为拒绝模式

关键原则：默认拒绝，显式允许
```

### Q4: SDP和VPN的区别？

```
VPN：
  - 网络层信任，接入后获得网络访问权
  - 全隧道流量
  - 服务可被发现

SDP：
  - 身份+设备+上下文验证
  - 按应用授权
  - 服务隐身（SPA单包授权）
  - 默认所有端口关闭

核心区别：VPN是"网络接入"，SDP是"应用访问"
```

### Q5: BeyondCorp的核心做法是什么？

```
Google BeyondCorp的核心理念：
  "不存在内网，所有访问都经过认证和授权"

关键做法：
  1. 取消VPN，通过Access Proxy代理所有应用
  2. 每台设备有唯一证书（设备身份）
  3. Trust Inferer实时计算信任分
  4. 每个HTTP请求都验证
  5. 访问决策基于设备+用户+上下文
```

### Q6: NIST 800-207的核心要求？

```
七大原则：
  1. 所有资源都需保护（数据+计算）
  2. 所有通信加密
  3. 基于会话的授权
  4. 动态访问决策
  5. 持续监控资产完整性
  6. 严格enforced认证和授权
  7. 收集信息改进安全态势

三大组件：PE + PA + PEP
```
