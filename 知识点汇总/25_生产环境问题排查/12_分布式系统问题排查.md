## 分布式锁问题

### 排查方法

**1. 查看锁持有情况**

```bash
# Redis分布式锁查看
redis-cli KEYS "lock:*"
redis-cli GET "lock:order:123"

# ZooKeeper分布式锁查看
zkCli.sh get /locks/order/lock-0000000001
```

**2. 检查锁超时设置**

```java
// 检查Redis锁超时设置
String lockKey = "lock:order:" + orderId;
boolean locked = redisTemplate.opsForValue().setIfAbsent(lockKey, "1", 30, TimeUnit.SECONDS);
```

### 常见原因

**1. 死锁**

```java
// 错误示例：未设置超时时间
boolean locked = redisTemplate.opsForValue().setIfAbsent(lockKey, "1");
// 正确示例：设置超时时间
boolean locked = redisTemplate.opsForValue().setIfAbsent(lockKey, "1", 30, TimeUnit.SECONDS);
```

**2. 锁释放问题**

```java
// 错误示例：未在finally中释放锁
if (locked) {
    try {
        // 业务逻辑
    } catch (Exception e) {
        // 异常处理
    }
    redisTemplate.delete(lockKey);
}

// 正确示例：在finally中释放锁
if (locked) {
    try {
        // 业务逻辑
    } catch (Exception e) {
        // 异常处理
    } finally {
        redisTemplate.delete(lockKey);
    }
}
```

### 解决方案

**1. 分布式锁最佳实践**

```java
// Redis分布式锁最佳实践
String lockKey = "lock:order:" + orderId;
String requestId = UUID.randomUUID().toString();
try {
    boolean locked = redisTemplate.opsForValue().setIfAbsent(lockKey, requestId, 30, TimeUnit.SECONDS);
    if (locked) {
        // 业务逻辑
    } else {
        // 获取锁失败
        throw new RuntimeException("获取分布式锁失败");
    }
} finally {
    // 只能释放自己的锁
    String value = redisTemplate.opsForValue().get(lockKey);
    if (requestId.equals(value)) {
        redisTemplate.delete(lockKey);
    }
}
```

**2. 使用成熟的分布式锁框架**

```xml
<!-- Redisson依赖 -->
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson</artifactId>
    <version>3.16.1</version>
</dependency>
```

```java
// Redisson分布式锁使用
RLock lock = redissonClient.getLock(lockKey);
try {
    boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);
    if (locked) {
        // 业务逻辑
    }
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

### 案例一：分布式锁死锁导致订单处理阻塞

**问题描述**：电商平台订单支付后状态更新缓慢，出现大量订单处理超时，数据库中出现大量未完成订单。

**排查过程**：
1. 查看应用日志发现获取分布式锁失败的错误
```bash
 grep "获取分布式锁失败" application.log
```
2. 检查Redis中的锁状态
```bash
 redis-cli KEYS "lock:order:*"
 # 发现大量长时间未释放的锁
 redis-cli TTL lock:order:12345
 # 返回值为-1，表示锁未设置过期时间
```
3. 分析代码发现未正确设置锁超时时间
```java
// 问题代码
String lockKey = "lock:order:" + orderId;
boolean locked = redisTemplate.opsForValue().setIfAbsent(lockKey, "1");
// 未设置超时时间，发生异常时锁无法自动释放
```

**解决方案**：
1. 修复分布式锁实现，添加超时时间
```java
String lockKey = "lock:order:" + orderId;
String requestId = UUID.randomUUID().toString();
// 设置30秒超时时间
boolean locked = redisTemplate.opsForValue().setIfAbsent(lockKey, requestId, 30, TimeUnit.SECONDS);
if (locked) {
    try {
        // 订单处理逻辑
        processOrder(orderId);
    } finally {
        // 释放锁
        String value = redisTemplate.opsForValue().get(lockKey);
        if (requestId.equals(value)) {
            redisTemplate.delete(lockKey);
        }
    }
}
```
2. 部署Redisson分布式锁框架
```xml
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson-spring-boot-starter</artifactId>
    <version>3.16.1</version>
</dependency>
```
3. 清理残留的死锁
```bash
# 编写脚本清理过期锁
redis-cli KEYS "lock:order:*" | while read key; do
    if [ $(redis-cli TTL $key) -eq -1 ]; then
        redis-cli DEL $key
        echo "Deleted expired lock: $key"
    fi
 done
```

**优化效果**：订单处理成功率从75%提升至99.9%，平均处理时间从15秒降至2秒。

### 案例二：Raft集群脑裂导致数据不一致

**问题描述**：基于Etcd的配置中心出现配置数据不一致，部分服务加载了旧配置，导致服务行为异常。

**排查过程**：
1. 检查Etcd集群状态
```bash
etcdctl endpoint status --write-out=table
```
2. 发现集群出现两个Leader节点
```
+----------------+------------------+---------+---------+-----------+------------+
|    ENDPOINT    |        ID        | VERSION | DB SIZE | IS LEADER | RAFT TERM |
+----------------+------------------+---------+---------+-----------+------------+
| 10.0.1.10:2379 | 6e3bd2f48db52441 |   3.5.0 |   45 MB |      true |         15 |
| 10.0.1.11:2379 | 924e2e83e93f256a |   3.5.0 |   45 MB |      true |         16 |
| 10.0.1.12:2379 | c8a7c74f2e6a5f1d |   3.5.0 |   45 MB |     false |         15 |
+----------------+------------------+---------+---------+-----------+------------+
```
3. 检查网络发现节点间网络分区

**解决方案**：
1. 临时处理：移除异常节点
```bash
etcdctl member remove 924e2e83e93f256a
```
2. 调整Raft参数
```yaml
# etcd配置
etcd:
  name: etcd-1
  data-dir: /var/lib/etcd
  initial-cluster: etcd-1=https://10.0.1.10:2380,etcd-2=https://10.0.1.11:2380,etcd-3=https://10.0.1.12:2380
  initial-cluster-state: existing
  heartbeat-interval: 100
  election-timeout: 1000
  auto-compaction-mode: periodic
  auto-compaction-retention: "1h"
```
3. 配置监控告警
```yaml
# Prometheus告警规则
groups:
- name: etcd_alert
  rules:
  - alert: EtcdClusterUnhealthy
    expr: sum(etcd_server_is_leader) > 1
    for: 30s
    labels:
      severity: critical
    annotations:
      summary: "Etcd集群脑裂"
      description: "Etcd集群存在{{ $value }}个leader节点，正常应为1个"
```

**优化效果**：集群恢复稳定，配置同步延迟从30秒降至2秒，未再发生数据不一致问题。

```java
// 错误示例：未使用分布式追踪上下文
```

```bash
etcdctl endpoint status --write-out=table

# Consul集群状态
consul operator raft list-peers
```

**2. 查看日志复制情况**
```bash
# Etcd日志复制状态
etcdctl -w table endpoint status
```

### 常见原因

**1. 集群脑裂**
```bash
# 检查集群成员
etcdctl member list

# 检查leader状态
etcdctl endpoint status --write-out=json | jq '.leaderInfo'
```

**2. 日志复制延迟**
```bash
# 查看 follower 与 leader 的差距
etcdctl -w json endpoint status | jq '.raftAppliedIndex - .raftIndex'
```

### 解决方案

**1. 调整Raft参数**
```yaml
# Etcd配置调整
etcd:
  raft:
    election-timeout: 5000ms
    heartbeat-interval: 500ms
```

**2. 修复集群脑裂**
```bash
# 强制重新选举leader
etcdctl elect -w table /leader/lock my-node
```
