## 慢SQL排查

### 排查方法

**1. 查看慢查询日志**

```sql
-- 查看慢查询配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 1秒

-- 查看慢查询
SELECT * FROM mysql.slow_log 
ORDER BY start_time DESC 
LIMIT 10;
```

**2. 使用EXPLAIN分析**

```sql
-- 分析SQL执行计划
EXPLAIN SELECT * FROM orders 
WHERE user_id = 123 AND status = 1;

-- 重点关注：
-- type: 访问类型（ALL最差，const最好）
-- key: 使用的索引
-- rows: 扫描行数
-- Extra: 额外信息
```

**3. 查看当前执行SQL**

```sql
-- 查看当前连接和SQL
SHOW PROCESSLIST;

-- 查看详细执行信息
SELECT * FROM information_schema.processlist 
WHERE command != 'Sleep' 
ORDER BY time DESC;
```

### 常见原因

**1. 缺少索引**

```sql
-- ❌ 全表扫描
SELECT * FROM orders WHERE user_id = 123;

-- ✅ 添加索引
CREATE INDEX idx_user_id ON orders(user_id);
```

**2. SQL语句问题**

```sql
-- ❌ 使用函数破坏索引
SELECT * FROM orders WHERE DATE(create_time) = '2025-10-29';

-- ✅ 改写SQL
SELECT * FROM orders 
WHERE create_time >= '2025-10-29 00:00:00' 
AND create_time < '2025-10-30 00:00:00';
```

**3. 索引选择不当**

```sql
-- 分析索引使用情况
EXPLAIN SELECT * FROM orders 
WHERE user_id = 123 AND status = 1;

-- 如果未使用索引，检查：
-- 1. 索引是否存在
-- 2. 索引是否有效
-- 3. 是否需要联合索引
```

### 优化方案

**1. 添加索引**

```sql
-- 单列索引
CREATE INDEX idx_user_id ON orders(user_id);

-- 联合索引（注意顺序）
CREATE INDEX idx_user_status ON orders(user_id, status);

-- 覆盖索引
CREATE INDEX idx_user_status_time ON orders(user_id, status, create_time);
```

**2. 优化SQL**

```sql
-- 避免SELECT *
SELECT user_id, order_id, status FROM orders WHERE user_id = 123;

-- 避免函数计算
-- ❌ WHERE DATE(create_time) = '2025-10-29'
-- ✅ WHERE create_time >= '2025-10-29 00:00:00' AND create_time < '2025-10-30 00:00:00'

-- 使用LIMIT
SELECT * FROM orders WHERE user_id = 123 LIMIT 10;
```

**3. 分页优化**

```sql
-- ❌ 深分页（慢）
SELECT * FROM orders LIMIT 100000, 10;

-- ✅ 使用索引优化
SELECT * FROM orders 
WHERE id > 100000 
ORDER BY id 
LIMIT 10;
```

## 连接池问题

### 排查方法

**1. 查看连接池状态**

```java
// HikariCP连接池监控
HikariPool pool = (HikariPool) dataSource.getHikariPoolMXBean();
System.out.println("Active: " + pool.getActiveConnections());
System.out.println("Idle: " + pool.getIdleConnections());
System.out.println("Total: " + pool.getTotalConnections());
```

**2. 查看数据库连接数**

```sql
-- 查看当前连接数
SHOW PROCESSLIST;
SHOW STATUS LIKE 'Threads_connected';

-- 查看最大连接数
SHOW VARIABLES LIKE 'max_connections';

-- 查看连接数使用率
SELECT 
    VARIABLE_VALUE AS max_connections,
    (SELECT VARIABLE_VALUE 
     FROM information_schema.GLOBAL_STATUS 
     WHERE VARIABLE_NAME = 'Threads_connected') AS current_connections,
    ROUND((SELECT VARIABLE_VALUE 
           FROM information_schema.GLOBAL_STATUS 
           WHERE VARIABLE_NAME = 'Threads_connected') / 
          VARIABLE_VALUE * 100, 2) AS usage_percent
FROM information_schema.GLOBAL_VARIABLES
WHERE VARIABLE_NAME = 'max_connections';
```

### 常见原因

**1. 连接池配置过小**

```yaml
# application.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10  # 太小
      minimum-idle: 5
```

**2. 连接泄漏**

```java
// ❌ 连接未关闭
Connection conn = dataSource.getConnection();
PreparedStatement ps = conn.prepareStatement(sql);
// 忘记关闭连接

// ✅ 使用try-with-resources
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    // 执行查询
}
```

**3. 连接持有时间过长**

```java
// ❌ 长时间持有连接
Connection conn = dataSource.getConnection();
// ... 大量业务逻辑 ...
conn.close();

// ✅ 及时释放连接
try (Connection conn = dataSource.getConnection()) {
    // 快速执行SQL
}
```

### 解决方案

**1. 调整连接池配置**

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 10
      connection-timeout: 30000
      max-lifetime: 1800000
      idle-timeout: 600000
```

**2. 检查连接泄漏**

```java
// 启用连接泄漏检测
spring:
  datasource:
    hikari:
      leak-detection-threshold: 60000  # 60秒
```

**3. 增加数据库连接数**

```sql
-- 增加最大连接数
SET GLOBAL max_connections = 200;

-- 永久生效（修改配置文件）
[mysqld]
max_connections = 200
```

## 死锁问题

### 排查方法

**1. 查看死锁日志**

```sql
-- 查看最近死锁
SHOW ENGINE INNODB STATUS;

-- 查看死锁信息
SELECT * FROM information_schema.innodb_locks;
SELECT * FROM information_schema.innodb_lock_waits;
```

**2. 查看锁等待**

```sql
-- 查看锁等待情况
SELECT 
    r.trx_id waiting_trx_id,
    r.trx_mysql_thread_id waiting_thread,
    r.trx_query waiting_query,
    b.trx_id blocking_trx_id,
    b.trx_mysql_thread_id blocking_thread,
    b.trx_query blocking_query
FROM information_schema.innodb_lock_waits w
INNER JOIN information_schema.innodb_trx b ON b.trx_id = w.blocking_trx_id
INNER JOIN information_schema.innodb_trx r ON r.trx_id = w.requesting_trx_id;
```

### 常见原因

**1. 事务顺序不一致**

```java
// ❌ 事务顺序不一致，容易死锁
// 事务1：先锁A，再锁B
// 事务2：先锁B，再锁A

// ✅ 统一事务顺序
// 所有事务：先锁A，再锁B
```

**2. 长事务**

```java
// ❌ 长事务持有锁时间过长
@Transactional
public void longTransaction() {
    // 大量业务逻辑
    // 持有锁时间过长
}

// ✅ 减少事务范围
public void shortTransaction() {
    // 只包含必要的数据库操作
}
```

**3. 索引缺失**

```sql
-- 缺少索引导致锁范围扩大
-- 全表扫描会锁更多行
CREATE INDEX idx_user_id ON orders(user_id);
```

### 解决方案

**1. 统一事务顺序**

```java
// 所有事务按相同顺序加锁
// 例如：先锁用户表，再锁订单表
```

**2. 减少事务范围**

```java
// 只包含必要的数据库操作
@Transactional
public void quickTransaction() {
    // 快速执行SQL
}
```

**3. 使用乐观锁**

```java
// 使用版本号避免锁等待
UPDATE orders 
SET status = 1, version = version + 1 
WHERE id = 123 AND version = 1;
```

**4. 设置锁超时**

```sql
-- 设置锁等待超时
SET innodb_lock_wait_timeout = 10;  -- 10秒
```

**2. 查看binlog位置**

```sql
-- 主库执行
SHOW MASTER STATUS;

SHOW SLAVE STATUS;
-- Master_Log_File: 主库binlog文件

```

### 常见原因

```bash
# 检查从库资源
```

```bash

traceroute master_host
```

**3. 主库写入量大**

```sql
-- 查看主库写入量
SHOW STATUS LIKE 'Com_insert';
SHOW STATUS LIKE 'Com_update';
SHOW STATUS LIKE 'Com_delete';
```

**4. 大事务**

```sql
-- 大事务导致延迟
-- 解决方案：拆分大事务
```

### 解决方案

**1. 提高从库性能**

```
- 增加从库资源（CPU、内存）
- 优化从库配置
```

**2. 优化网络**

**3. 读写分离策略**

**4. 监控告警**

### 分库分表问题排查

### 常见问题类型

**1. 数据分布不均**

**症状**：
```
✅ 部分分表数据量过大
✅ 部分分表查询缓慢
✅ 分表负载不均衡
```

**排查方法**：
```sql
-- 查看各分表数据量分布
SELECT table_name, table_rows FROM information_schema.tables WHERE table_schema = 'db_name' AND table_name LIKE 'order_%';
```

**解决方案**：
```java
// Sharding-JDBC 自动分片策略配置
@Configuration
public class ShardingConfig {
    @Bean
    public ShardingRuleConfiguration shardingRuleConfig() {
        // 使用一致性哈希分片算法
        TableRuleConfiguration orderTableRule = new TableRuleConfiguration("t_order", "ds_${0..1}.t_order_${0..31}");
        orderTableRule.setDatabaseShardingStrategyConfig(
            new StandardShardingStrategyConfiguration("user_id", new ModuloDatabaseShardingAlgorithm()));
        orderTableRule.setTableShardingStrategyConfig(
            new StandardShardingStrategyConfiguration("order_id", new ConsistentHashTableShardingAlgorithm()));
        
        ShardingRuleConfiguration shardingRuleConfig = new ShardingRuleConfiguration();
        shardingRuleConfig.getTableRuleConfigs().add(orderTableRule);
        return shardingRuleConfig;
    }
}
```

**2. 分布式事务问题**

**症状**：
```
✅ 跨库事务不一致
✅ 数据提交失败
✅ 分布式锁冲突
```

**解决方案**：

**Seata AT模式配置**：
```yaml
# application.yml
seata:
  enabled: true
  application-id: order-service
  tx-service-group: my_test_tx_group
  service:
    vgroup-mapping:
      my_test_tx_group: default
    grouplist:
      default: 127.0.0.1:8091
  registry:
    type: nacos
    nacos:
      server-addr: 127.0.0.1:8848
```

**分布式事务代码示例**：
```java
@GlobalTransactional
public void createOrder(OrderDTO orderDTO) {
    // 订单服务本地事务
    orderMapper.insert(orderDTO);
    
    // 库存服务远程调用
    inventoryFeignClient.deduct(orderDTO.getProductId(), orderDTO.getQuantity());
    
    // 支付服务远程调用
    paymentFeignClient.createPayment(orderDTO);
}
```

**3. 跨库联合查询问题**

**症状**：
```
✅ 跨库JOIN查询失败
✅ 分页查询异常
✅ 全局表数据不一致
```

**解决方案**：
```java
// 使用Sharding-JDBC的绑定表和广播表
@Bean
public ShardingRuleConfiguration shardingRuleConfig() {
    // 绑定表配置
    TableRuleConfiguration orderTableRule = new TableRuleConfiguration("t_order", "ds_${0..1}.t_order_${0..31}");
    TableRuleConfiguration orderItemTableRule = new TableRuleConfiguration("t_order_item", "ds_${0..1}.t_order_item_${0..31}");
    
    ShardingRuleConfiguration shardingRuleConfig = new ShardingRuleConfiguration();
    shardingRuleConfig.getTableRuleConfigs().add(orderTableRule);
    shardingRuleConfig.getTableRuleConfigs().add(orderItemTableRule);
    shardingRuleConfig.getBindingTableGroups().add("t_order,t_order_item");
    
    // 广播表配置
    TableRuleConfiguration dictTableRule = new TableRuleConfiguration("t_dict", "ds_${0..1}.t_dict");
    dictTableRule.setDatabaseShardingStrategyConfig(new NoneShardingStrategyConfiguration());
    dictTableRule.setTableShardingStrategyConfig(new NoneShardingStrategyConfiguration());
    shardingRuleConfig.getTableRuleConfigs().add(dictTableRule);
    
    return shardingRuleConfig;
}
```
