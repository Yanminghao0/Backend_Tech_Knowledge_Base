## 1. 数据仓库概述

```
数据仓库(Data Warehouse) vs 数据库(Database):

  ┌──────────────────────────────────────────────────┐
  │  数据库(DB)        │  数据仓库(DW)                │
  │  ──────────       │  ──────────                  │
  │  面向业务操作      │  面向分析决策                 │
  │  OLTP             │  OLAP                        │
  │  当前数据          │  历史数据                     │
  │  频繁增删改        │  批量加载，少更新             │
  │  规范化(3NF)       │  维度建模(星型/雪花)          │
  │  GB~TB            │  TB~PB                       │
  └──────────────────────────────────────────────────┘

数据仓库四大特征(Inmon定义):
  1. 面向主题 — 按业务主题组织(如客户、销售、库存)
  2. 集成性 — 多数据源统一口径，消除不一致
  3. 非易失 — 数据一旦进入不修改，只追加
  4. 时变性 — 保留历史变化，支持时间维度分析
```

## 2. 维度建模

### 2.1 星型模型 vs 雪花模型

```
星型模型(Star Schema):
  一个事实表 + 多个维度表(不展开)

                    ┌──────────┐
                    │ dim_date │
                    └────┬─────┘
                         │
  ┌──────────┐    ┌─────┴──────┐    ┌──────────┐
  │dim_store │────│ fact_sales │────│dim_product│
  └──────────┘    └─────┬──────┘    └──────────┘
                         │
                    ┌────┴─────┐
                    │dim_customer│
                    └──────────┘

  特点: 维度表只一层，结构简单，JOIN少，查询快
  推荐: 大多数数仓场景

雪花模型(Snowflake Schema):
  维度表展开多级

                    ┌──────────┐
                    │ dim_date │
                    └────┬─────┘
                         │
  ┌──────────┐    ┌─────┴──────┐    ┌──────────┐
  │dim_store │────│ fact_sales │────│dim_product│
  └────┬─────┘    └─────┬──────┘    └────┬─────┘
       │                 │                │
  ┌────┴─────┐     ┌─────┴──────┐   ┌────┴─────┐
  │dim_region│     │dim_customer│   │dim_category│
  └──────────┘     └─────┬──────┘   └──────────┘
                         │
                   ┌─────┴──────┐
                   │dim_customer│
                   │  _segment  │
                   └────────────┘

  特点: 维度表多级展开，规范化高，JOIN多
  适用: 存储空间敏感，维度层级深
```

### 2.2 事实表

```sql
-- 事实表: 存储度量值(可聚合的数值)和外键(关联维度表)

-- 事务事实表 — 每条业务记录一行
CREATE TABLE fact_order (
    order_id BIGINT,
    dt DATE,
    customer_key BIGINT,
    store_key BIGINT,
    product_key BIGINT,
    quantity INT,
    unit_price DECIMAL(10,2),
    total_amount DECIMAL(10,2),
    discount_amount DECIMAL(10,2)
);

-- 周期快照事实表 — 每天固定周期记录状态
CREATE TABLE fact_daily_inventory (
    dt DATE,
    store_key BIGINT,
    product_key BIGINT,
    stock_quantity INT,
    stock_value DECIMAL(10,2)
);

-- 累积快照事实表 — 跟踪业务流程各阶段时间
CREATE TABLE fact_order_lifecycle (
    order_id BIGINT,
    create_time TIMESTAMP,
    pay_time TIMESTAMP,
    ship_time TIMESTAMP,
    deliver_time TIMESTAMP,
    complete_time TIMESTAMP,
    customer_key BIGINT,
    total_amount DECIMAL(10,2)
);
-- 一个订单一行，各阶段时间逐步填充
```

### 2.3 维度表

```sql
-- 维度表: 存储描述性属性

-- 客户维度
CREATE TABLE dim_customer (
    customer_key BIGINT,      -- 代理键(自增)
    customer_id BIGINT,       -- 业务键(来自源系统)
    customer_name STRING,
    phone STRING,
    email STRING,
    gender STRING,
    age_group STRING,         -- 分组: 18-25/26-35/36-45/46+
    customer_level STRING,    -- 等级: 普通/银卡/金卡/钻石
    register_date DATE,
    start_date DATE,          -- SCD Type2 有效起始
    end_date DATE,            -- SCD Type2 有效结束
    is_current BOOLEAN        -- 是否当前版本
);

-- 缓慢变化维度(SCD) — 维度属性变化如何处理
-- Type 1: 直接覆盖(丢失历史)
-- Type 2: 新增行(保留历史，用start/end_date标记)★推荐
-- Type 3: 新增列(只保留前一个版本)

-- 日期维度(固定生成)
CREATE TABLE dim_date (
    dt DATE PRIMARY KEY,
    year INT,
    month INT,
    day INT,
    quarter INT,
    week_of_year INT,
    day_of_week INT,
    day_name STRING,        -- Monday/Tuesday...
    month_name STRING,      -- January/February...
    is_weekend BOOLEAN,
    is_holiday BOOLEAN,
    fiscal_year INT,        -- 财年
    fiscal_quarter INT
);
```

## 3. 数仓分层架构

```
数仓标准分层:

  ┌──────────────────────────────────────────────────────┐
  │                    ADS (应用层)                       │
  │  面向应用的结果数据，直接给BI报表/API使用              │
  │  如: 每日销售报表、用户画像标签、推荐特征             │
  ├──────────────────────────────────────────────────────┤
  │                    DWS (汇总层)                       │
  │  按主题+时间粒度汇总，宽表                            │
  │  如: 用户日汇总表、商品日汇总表、门店日汇总表         │
  ├──────────────────────────────────────────────────────┤
  │                    DWD (明细层)                       │
  │  清洗后的明细数据，统一口径                           │
  │  如: 订单明细表(清洗+标准化+维度退化)                 │
  ├──────────────────────────────────────────────────────┤
  │                    ODS (贴源层)                       │
  │  原始数据，与源系统结构一致                           │
  │  如: MySQL binlog、日志文件、API数据                 │
  └──────────────────────────────────────────────────────┘

  数据流: ODS → DWD → DWS → ADS
  每层职责清晰，避免重复计算
```

```sql
-- ODS层 — 原始数据(贴源)
CREATE TABLE ods_order (
    order_id BIGINT,
    customer_id BIGINT,
    product_id BIGINT,
    amount DECIMAL(10,2),
    status STRING,
    create_time TIMESTAMP,
    update_time TIMESTAMP,
    dt STRING  -- 分区字段
) STORED AS ORC;

-- DWD层 — 清洗+标准化
CREATE TABLE dwd_order_detail (
    order_id BIGINT,
    customer_key BIGINT,     -- 维度代理键
    product_key BIGINT,
    store_key BIGINT,
    order_status STRING,     -- 标准化: 1→待支付 2→已支付...
    original_amount DECIMAL(10,2),
    discount_amount DECIMAL(10,2),
    final_amount DECIMAL(10,2),
    create_time TIMESTAMP,
    dt STRING
) STORED AS ORC;

-- DWD清洗逻辑:
-- 1. 去重(order_id唯一)
-- 2. 空值处理(默认值或丢弃)
-- 3. 格式统一(日期格式、金额精度)
-- 4. 维度关联(业务键→代理键)
-- 5. 状态码标准化

-- DWS层 — 汇总
CREATE TABLE dws_user_daily (
    dt STRING,
    customer_key BIGINT,
    order_count INT,         -- 下单次数
    order_amount DECIMAL(10,2), -- 下单金额
    pay_count INT,           -- 支付次数
    pay_amount DECIMAL(10,2),
    refund_count INT,
    refund_amount DECIMAL(10,2),
    last_order_time TIMESTAMP
) STORED AS ORC;

-- ADS层 — 应用结果
CREATE TABLE ads_daily_sales_report (
    dt STRING,
    total_orders INT,
    total_amount DECIMAL(10,2),
    avg_order_amount DECIMAL(10,2),
    new_customers INT,
    returning_customers INT,
    top_product STRING,
    top_store STRING
) STORED AS ORC;
```

## 4. ETL与ELT

```
ETL vs ELT:

  ETL (Extract-Transform-Load):
    源系统 → 提取 → 转换(在外部引擎) → 加载 → 目标库
    转换在专门的ETL引擎(DataX/Kettle)中完成
    适合: 传统数仓，数据量中等

  ELT (Extract-Load-Transform):
    源系统 → 提取 → 加载(原始) → 转换(在数仓内SQL) → 目标表
    转换在数仓引擎(Spark SQL/Hive SQL/ClickHouse)中完成
    适合: 现代数仓/数据湖，大数据量

  趋势: ELT逐渐成为主流
    - 数仓引擎算力强(SQL on Spark/Flink)
    - 避免外部ETL引擎的瓶颈
    - SQL更易维护
```

```
数据同步工具:

  ┌──────────────────────────────────────────────────┐
  │  工具          │  适用场景                        │
  │  ─────        │  ──────                         │
  │  DataX        │  阿里开源，异构数据源同步         │
  │  Canal        │  MySQL binlog → Kafka/MQ         │
```

## 5. 数据湖技术

### 5.1 Hudi

```
Hudi (Hadoop Upserts Deletes and Incrementals):

  核心能力:
    - Upsert: 支持更新(传统HDFS不支持)
    - Delete: 支持删除
    - Incremental: 增量查询(只查变化部分)
    - Time Travel: 时间旅行(查询历史版本)

  表类型:
    COW (Copy On Write):
      写入时合并 → 读快写慢
      每次upsert重写整个Parquet文件

    MOR (Merge On Read):
      写入到Log文件 → 读时合并 → 写快读慢
      后台Compaction合并Log到Base文件

  ┌──────────────────────────────────────────────────┐
  │  COW适用: 读多写少，查询性能要求高                │
  │  MOR适用: 写多读少，实时写入场景                  │
  └──────────────────────────────────────────────────┘
```

```sql
-- Hudi建表(Spark SQL)
CREATE TABLE IF NOT EXISTS hudi_orders (
    id BIGINT,
    user_id BIGINT,
    amount DECIMAL(10,2),
    status STRING,
    ts TIMESTAMP
) USING HUDI
PARTITIONED BY (dt STRING)
OPTIONS (
    primaryKey = 'id',
    preCombineField = 'ts',
    type = 'mor',           -- MERGE ON READ
    hoodie.cleaner.commits.retained = '10',
    hoodie.keep.max.commits = '20'
);

-- Upsert(自动更新)
INSERT INTO hudi_orders VALUES
    (1, 1001, 99.9, 'paid', '2026-07-13 10:00:00');

-- 增量查询(只查上次同步后的变化)
SELECT * FROM hudi_orders
/*+ OPTIONS('hoodie.datasource.query.type'='incremental',
            'hoodie.datasource.read.begin.instanttime'='20260713100000') */;
```

### 5.2 Iceberg

```
Iceberg — 开放表格式(Netflix开源):

  核心特性:
    - 快照隔离: 读写不冲突(MVCC)
    - Schema演进: 加/删/改名列，不影响历史数据
    - 分区演进: 可改变分区策略，不需要重写数据
    - 时间旅行: 查询任意历史快照
    - 精确分区剪裁: 文件级统计信息，比Hive更高效

  Iceberg vs Hudi vs Delta Lake:
    ┌──────────────────────────────────────────────┐
    │  维度      │ Iceberg  │ Hudi    │ Delta     │
    │  ─────    │ ──────── │ ──────  │ ─────     │
    │  引擎绑定  │ 开放     │ Spark   │ Spark     │
    │  Schema演进│ 强       │ 中      │ 中        │
    │  Upsert   │ 支持     │ 强项    │ 支持      │
    │  生态     │ 多引擎   │ Flink好 │ Databricks│
    │  适用     │ 通用     │ CDC实时 │ Databricks│
    └──────────────────────────────────────────────┘
```

### 6.2 数据质量

```sql
-- 数据质量六大维度:
-- 1. 完整性 — 空值率、缺失率
-- 2. 准确性 — 数据是否正确
-- 3. 一致性 — 多源数据口径一致
-- 4. 及时性 — 数据是否按时到达
-- 5. 唯一性 — 主键是否重复
-- 6. 有效性 — 格式/范围是否合规

-- 质量检查SQL示例
-- 完整性检查: 空值率
SELECT
    'user_id' AS field,
    COUNT(*) AS total,
    SUM(CASE WHEN user_id IS NULL THEN 1 ELSE 0 END) AS null_count,
    ROUND(SUM(CASE WHEN user_id IS NULL THEN 1.0 ELSE 0 END) / COUNT(*) * 100, 2) AS null_rate
FROM dwd_order_detail
WHERE dt = '2026-07-13';

-- 唯一性检查: 主键重复
SELECT order_id, COUNT(*) AS cnt
FROM dwd_order_detail
WHERE dt = '2026-07-13'
GROUP BY order_id
HAVING COUNT(*) > 1;

-- 有效性检查: 金额范围
SELECT COUNT(*) AS invalid_count
FROM dwd_order_detail
WHERE dt = '2026-07-13'
  AND (amount < 0 OR amount > 1000000);  -- 金额应在0-100万之间

-- 一致性检查: 上下游数据量对比
SELECT
    'ODS' AS source,
    COUNT(*) AS cnt
FROM ods_order WHERE dt = '2026-07-13'
UNION ALL
SELECT
    'DWD' AS source,
    COUNT(*) AS cnt
FROM dwd_order_detail WHERE dt = '2026-07-13';
```

## 7. 面试题速查

**Q1: 星型模型和雪花模型的区别？**

```
星型: 维度表一层，JOIN少，查询快，推荐
雪花: 维度表多级展开，存储省，JOIN多，性能差
大多数OLAP场景选星型模型
```

**Q2: 数仓分层的意义？**

```
ODS→DWD→DWS→ADS
1. 复用性: DWD/DWS被多个ADS复用
2. 清晰性: 每层职责明确
3. 性能: 预聚合减少重复计算
4. 解耦: 源系统变化不影响ADS
```

**Q3: 缓慢变化维度(SCD)怎么处理？**

```
Type1: 直接覆盖(丢历史)
Type2: 新增行+有效日期(保留历史)★推荐
Type3: 新增列(保留前一版本)
根据业务需求: 需要历史分析用Type2，不需要用Type1
```

**Q4: Hudi的COW和MOR区别？**

```
COW(Copy On Write): 写入时合并，读快写慢，适合读多写少
MOR(Merge On Read): 写入Log，读时合并，写快读慢，适合实时写入
MOR有后台Compaction合并Log→Base文件
```

**Q5: 数据质量有哪些维度？**

```
完整性(空值)、准确性(正确)、一致性(口径)、
及时性(按时)、唯一性(不重复)、有效性(格式/范围)
生产中通常自动化检查 + 告警
```
