## 2. 架构对比

### Kafka架构

```
Producer → [Broker] → Consumer
              ↓
         Zookeeper/KRaft
         
核心概念：
- Topic → Partition → Replica
- Consumer Group
- Offset 管理消费进度
```

### RocketMQ架构

```
Producer → NameServer → Broker → Consumer
                           ↓
                      CommitLog + ConsumeQueue
                      
核心概念：
- Topic → MessageQueue
- 消息存储：CommitLog统一存储
- 消费模式：集群/广播
```

### RabbitMQ架构

```
Producer → Exchange → Queue → Consumer
              ↓
          Binding(RoutingKey)
          
核心概念：
- Exchange: Direct/Fanout/Topic/Headers
- Queue: 消息存储
- Binding: 绑定关系
```

### Pulsar架构

```
Producer → Broker → BookKeeper → Consumer
              ↓
        Bookie(存储节点)
        
核心概念：
- 计算与存储分离
- Topic → Partition → Segment → Ledger
- 多租户原生支持
```

### 架构差异总结

| 特性 | Kafka | RocketMQ | RabbitMQ | Pulsar |
|------|-------|----------|----------|--------|
| 存储模型 | 分区日志 | CommitLog+ConsumeQueue | 内存+磁盘 | 分层存储(BookKeeper) |
| 计算存储分离 | 否 | 否 | 否 | 是 |
| 多租户 | 弱 | 一般 | 一般 | 原生支持 |
| 元数据管理 | KRaft(3.3+) | NameServer | Mnesia | Zookeeper |

## 3. 性能对比

### 吞吐量

| 消息队列 | 单机吞吐 | 延迟 | 适用TPS |
|---------|---------|------|---------|
| Kafka | 100万/s | ~5ms | 10万-100万 |
| RocketMQ | 10万/s | ~3ms | 1万-10万 |
| RabbitMQ | 5万/s | ~1ms | 1千-5万 |
| Pulsar | 100万/s | ~5ms | 10万-100万 |

### 延迟对比

```
消息大小 1KB, 持久化场景:

RabbitMQ:  ~1ms  (最低延迟, Erlang优化)
RocketMQ:  ~3ms  (顺序写磁盘+异步刷盘)
Kafka:     ~5ms  (批量发送+顺序写)
Pulsar:    ~5ms  (BookKeeper分布式写)
```

### 资源消耗

| 消息队列 | CPU | 内存 | 磁盘 | JVM |
|---------|-----|------|------|-----|
| Kafka | 中 | 高 | 高 | 是 |
| RocketMQ | 中 | 中 | 中 | 是 |
| RabbitMQ | 低 | 低 | 低 | 否(Erlang) |
| Pulsar | 高 | 高 | 中 | 是 |

## 4. 功能特性对比

### 消息可靠性

| 特性 | Kafka | RocketMQ | RabbitMQ | Pulsar |
|------|-------|----------|----------|--------|
| 同步刷盘 | ✅ | ✅ | ✅ | ✅ |
| 异步刷盘 | ✅ | ✅ | ✅ | ✅ |
| 消息确认 | Offset ACK | Consumer ACK | Publisher Confirm | Cursor ACK |
| 死信队列 | 需自实现 | ✅ 内置 | ✅ 内置 | ✅ 内置 |
| 消息回溯 | ✅ 按Offset | ✅ 按时间 | ❌ | ✅ 按时间 |

### 消息顺序性

| 方案 | Kafka | RocketMQ | RabbitMQ | Pulsar |
|------|-------|----------|----------|--------|
| 分区有序 | ✅ | ✅ | ✅ | ✅ |
| 全局有序 | 单分区 | 单Queue | 单Queue | 单分区 |
| 延迟消息 | 需自实现 | ✅ 内置(18级) | 插件支持 | ✅ 内置 |
| 事务消息 | ✅ | ✅ | ❌ | ✅ |

## 5. 选型决策矩阵

### 按需求场景选型

| 需求场景 | 推荐 | 备选 | 选择理由 |
|---------|------|------|---------|
| 日志收集（高吞吐） | Kafka | Pulsar | 吞吐量最高，生态成熟 |
| 订单交易（高可靠） | RocketMQ | Kafka | 事务消息，延迟消息内置 |
| 业务解耦（灵活路由） | RabbitMQ | RocketMQ | Exchange路由灵活，低延迟 |
| IoT/流处理 | Pulsar | Kafka | 计算存储分离，多租户 |
| 微服务事件驱动 | RabbitMQ | RocketMQ | 消息确认机制完善 |
| 大数据流处理 | Kafka | Pulsar | Kafka Connect/Streams生态 |
| 金融支付 | RocketMQ | Kafka | 同步刷盘+事务消息 |

### 按团队技术栈选型

| 团队背景 | 推荐 | 理由 |
|---------|------|------|
| Java团队 | RocketMQ/Kafka | Java生态，源码可读 |
| 运维能力强 | Kafka | 需要较多运维投入 |
| 追求简单 | RabbitMQ | 开箱即用，管理界面友好 |
| 云原生团队 | Pulsar | 计算存储分离，弹性扩缩 |

## 6. 适用场景分析

### Kafka适用场景

```
✅ 日志收集系统（ELK/EFK数据管道）
✅ 用户行为追踪（埋点数据采集）
✅ 流处理（Flink/Spark Streaming数据源）
✅ 事件溯源（Event Sourcing）
✅ CDC数据同步（Debezium → Kafka → 下游）

❌ 不适合：需要延迟消息、消息回溯按时间、复杂路由
```

### RocketMQ适用场景

```
✅ 电商交易（订单、支付、库存）
✅ 分布式事务（事务消息）
✅ 延迟任务（订单超时取消）
✅ 金融场景（高可靠+审计）
✅ 消息推送（广播模式）

❌ 不适合：超大规模日志收集（Kafka更合适）
```

### RabbitMQ适用场景

```
✅ 微服务异步通信
✅ 任务队列（Celery/Sidekiq后端）
✅ 发布订阅（WebSocket推送）
✅ 复杂路由（Topic Exchange + 通配符）
✅ 小规模系统（快速接入）

❌ 不适合：超高吞吐（>5万TPS）、消息堆积
```

### Pulsar适用场景

```
✅ 多租户SaaS平台
✅ IoT数据接入
✅ 跨地域复制
✅ 计算存储分离架构
✅ 流批一体处理

❌ 不适合：团队经验不足（社区相对小）
```

## 7. 面试要点

### Q1: Kafka为什么吞吐量高？

```
1. 顺序写磁盘：磁盘顺序写速度接近内存随机写
2. 零拷贝（Zero Copy）：sendfile系统调用，减少用户态/内核态切换
3. 批量发送：Producer批量聚合消息，减少网络IO
4. 页缓存（Page Cache）：利用OS缓存，减少磁盘读取
5. 分区并行：多Partition并行读写
```

### Q2: RocketMQ和Kafka的区别？

```
1. 存储模型：Kafka每个Partition独立文件，RocketMQ所有Topic共享CommitLog
2. 消息可靠：RocketMQ支持同步刷盘+同步复制，Kafka默认异步
3. 事务消息：RocketMQ内置事务消息，Kafka需幂等+事务API
4. 延迟消息：RocketMQ内置18级延迟，Kafka需自行实现
5. 消息过滤：RocketMQ支持Tag/SQL过滤，Kafka需消费端过滤
```

### Q3: RabbitMQ的Exchange类型？

```
Direct: 精确匹配RoutingKey
Fanout: 广播到所有绑定的Queue
Topic: 通配符匹配（*匹配一个词，#匹配多个词）
Headers: 根据消息头属性匹配
```

### Q4: 如何保证消息不丢失？

```
生产端：
- Kafka: acks=all + retries
- RocketMQ: 同步发送 + 重试
- RabbitMQ: Publisher Confirm模式

Broker：
- Kafka: 副本因子>=2 + min.insync.replicas=2
- RocketMQ: 同步刷盘 + 同步复制
- RabbitMQ: 持久化队列 + 持久化消息

消费端：
- 手动ACK，处理完成后再确认
- 消费失败重试 + 死信队列
```

### Q5: 消息积压怎么处理？

```
1. 扩容Consumer实例（注意不能超过Partition数）
2. 临时方案：先消费再处理（转存到另一个Topic/Queue）
3. 优化消费逻辑：批量处理、异步处理
4. 如果消息可丢弃：设置TTL让消息过期
5. 根因分析：是消费慢还是生产过快
```
