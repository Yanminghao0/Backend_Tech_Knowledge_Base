## 1. 向量检索基础

### 1.1 为什么需要向量检索

```
传统检索(BM25/全文搜索):
  基于"关键词匹配"
  "Java并发编程" → 搜索"Java"和"并发"关键词
  问题: "Java多线程" 搜不到(虽然语义相同但关键词不同)

向量检索:
  基于"语义相似度"
  "Java并发编程" → 向量 [0.3, -0.5, 0.8, ...]
  "Java多线程"   → 向量 [0.3, -0.4, 0.7, ...] (相似!)
  → 余弦相似度高 → 能搜到

  应用场景:
    RAG知识库 — 语义检索文档
    推荐系统 — 相似物品推荐
    图片搜索 — 相似图片
    去重 — 语义去重
    Agent记忆 — 检索历史交互
```

### 1.2 相似度计算

```
┌──────────────────────────────────────────────────────┐
  │  方法          │  公式                    │  特点     │
  │  ──────     │  ──────               │  ────   │
  │  余弦相似度    │  cos(A,B) = A·B/|A||B|  │  最常用   │
  │  点积          │  A·B                     │  快      │
  │  欧氏距离      │  √Σ(aᵢ-bᵢ)²            │  几何距离│
  │  曼哈顿距离    │  Σ|aᵢ-bᵢ|              │  网格距离│
  └──────────────────────────────────────────────────────┘

  余弦相似度:
    值域: [-1, 1]
    1 = 完全相同
    0 = 正交(无关)
    -1 = 完全相反

  向量归一化后: 点积 = 余弦相似度
    大多数嵌入模型输出已归一化 → 用点积更快
```

### 1.3 暴力检索 vs 索引检索

```
暴力检索(Flat):
  遍历所有向量，计算相似度，取Top-K
  优点: 100%精确
  缺点: O(N)复杂度，数据量大时慢
  适用: <10万条数据

  示例: 100万条向量 × 1024维
    一次查询需计算100万个点积
    约500ms-1s(单线程)
    → 不可接受

索引检索:
  预建索引结构，近似检索(ANN)
  优点: O(logN)或O(1)，极快
  缺点: 近似结果(可能遗漏)
  适用: >10万条数据

  常用索引:
    HNSW — 图索引(主流)
    IVF — 倒排索引
    PQ — 乘积量化(压缩)
```

## 2. 索引算法

### 2.1 HNSW

```
HNSW(Hierarchical Navigable Small World) — 分层小世界图

  核心思想: 构建多层图，上层稀疏(快速定位区域)，下层密集(精确搜索)

  ┌──────────────────────────────────────────────┐
  │  Layer 2 (最稀疏):  A ──── B                  │
  │                  ↓        ↓                   │
  │  Layer 1:        A ── C ── B ── D             │
  │                  ↓    ↓    ↓    ↓             │
  │  Layer 0 (最密): A─E─C─F─B─G─D─H (全部节点)   │
  └──────────────────────────────────────────────┘

  搜索过程:
    1. 从Layer 2开始(最稀疏) → 快速定位到大致区域
    2. 逐层下降 → 在每层局部搜索
    3. Layer 0(最密) → 精确搜索最近邻

  类比: 坐飞机找城市
    Layer 2: 北京→上海(跨省)
    Layer 1: 上海→浦东(跨区)
    Layer 0: 浦东→张江(街道)

  HNSW关键参数:
    M: 每个节点的连接数(如16)
       M越大 → 精度高但内存大
    ef_construction: 构建索引时的搜索宽度(如200)
       越大 → 构建越慢但质量越好
    ef_search: 查询时的搜索宽度(如50)
       越大 → 查询越精确但越慢

  HNSW优势:
    1. 查询速度快(亚毫秒级)
    2. 召回率高(95%+)
    3. 支持动态增删
    4. 大多数向量数据库默认选择
```

### 2.2 IVF

```
IVF(Inverted File) — 倒排索引

  核心思想: K-means聚类，只搜索最近的几个聚类

  ┌──────────────────────────────────────────┐
  │  1. 训练: K-means聚类                     │
  │     将N个向量聚成K个簇                    │
  │     每个簇有一个中心点                     │
  │                                          │
  │  2. 查询:                                 │
  │     查询向量 → 找最近的nprobe个簇中心      │
  │     → 只在这些簇内搜索                    │
  │                                          │
  │  ┌───┐  ┌───┐  ┌───┐                     │
  │  │簇1│  │簇2│  │簇3│  ... K个簇          │
  │  └─┬─┘  └─┬─┘  └─┬─┘                     │
  │    │      │      │                        │
  │  v1,v2  v3,v4  v5,v6 (属于各簇的向量)    │
  └──────────────────────────────────────────┘

  查询示例:
    100万向量聚成1000个簇
    nprobe=10 → 只搜索10个最近的簇
    → 只需比较100万/1000×10 = 1万个向量
    → 加速100倍

  IVF参数:
    nlist: 簇的数量(如1000)
      nlist越大 → 每簇越小 → 搜索快但可能遗漏
    nprobe: 查询时搜索的簇数(如10)
      nprobe越大 → 精度高但慢

  IVF vs HNSW:
    IVF: 聚类+倒排，内存占用小，查询速度中等
    HNSW: 图索引，内存占用大，查询速度快
    推荐: 默认用HNSW，内存敏感用IVF
```

### 2.3 PQ乘积量化

```
PQ(Product Quantization) — 向量压缩

  问题: 100万 × 1024维 × 4字节 = 4GB内存
  PQ压缩: 100万 × 8字节(8个子码) = 8MB (压缩500倍)

  原理:
    1. 将d维向量切分成m个子向量
       1024维 → 8个子向量 × 128维
    2. 每个子向量独立K-means聚类(256个中心)
       128维 → 1字节(256个中心点的索引)
    3. 原始向量 → 8字节编码

  ┌──────────────────────────────────────────┐
  │  原始: [0.3, -0.5, 0.8, ...] (1024维)    │
  │  切分: [0.3,...128维] [-0.5,...] ... ×8  │
  │  量化: [码1, 码2, ..., 码8] (8字节)       │
  │                                          │
  │  1024维 × 4字节 = 4096字节               │
  │  → 8字节 (压缩512倍)                     │
  └──────────────────────────────────────────┘

  精度损失: 约2-5%的召回率下降
  适用: 内存有限的大规模数据(亿级)

  IVF + PQ组合:
    IVF粗筛(选簇) + PQ精排(簇内比较)
    最常用的内存优化组合
```

## 3. Milvus实战

### 3.1 Milvus概述

```
Milvus — 开源分布式向量数据库(国产/Zilliz)

  特点:
    1. 分布式 — 支持亿级向量
    2. 多索引 — HNSW/IVF/Flat/DiskANN
    3. 混合检索 — 稠密+稀疏向量
    4. 混合存储 — 内存+SSD
    5. 云原生 — K8s部署

  版本:
    Milvus Lite — 嵌入式(开发用)
    Milvus Standalone — 单机(Docker)
    Milvus Cluster — 分布式(生产)
```

### 3.2 Java SDK实战

```java
// Milvus Java SDK
import io.milvus.client.MilvusClient;
import io.milvus.grpc.*;
import io.milvus.param.*;

// 1. 连接Milvus
MilvusClient client = MilvusClient.newBuilder()
    .withHost("localhost")
    .withPort(19530)
    .build();

// 2. 创建Collection(表)
CreateCollectionParam createParam = CreateCollectionParam.newBuilder()
    .withCollectionName("documents")
    .withDescription("文档向量")
    .addFieldType(FieldType.newBuilder()
        .withName("id")
        .withDataType(DataType.Int64)
        .withPrimaryKey(true)
        .withAutoID(true)
        .build())
    .addFieldType(FieldType.newBuilder()
        .withName("embedding")
        .withDataType(DataType.FloatVector)
        .withDimension(1024)  // BGE-M3输出1024维
        .build())
    .addFieldType(FieldType.newBuilder()
        .withName("content")
        .withDataType(DataType.VarChar)
        .withMaxLength(65535)
        .build())
    .addFieldType(FieldType.newBuilder()
        .withName("metadata")
        .withDataType(DataType.JSON)
        .build())
    .build();
client.createCollection(createParam);

// 3. 创建索引
CreateIndexParam indexParam = CreateIndexParam.newBuilder()
    .withCollectionName("documents")
    .withFieldName("embedding")
    .withIndexType(IndexType.HNSW)
    .withMetricType(MetricType.COSINE)
    .withExtraParam("{\"M\":16,\"efConstruction\":200}")
    .build();
client.createIndex(indexParam);

// 4. 插入数据
List<JSONObject> rows = new ArrayList<>();
for (Document doc : documents) {
    JSONObject row = new JSONObject();
    row.put("embedding", doc.getEmbedding());  // float[]
    row.put("content", doc.getText());
    row.put("metadata", doc.getMetadata());
    rows.add(row);
}
InsertParam insertParam = InsertParam.newBuilder()
    .withCollectionName("documents")
    .withRows(rows)
    .build();
client.insert(insertParam);

// 5. 向量检索
SearchParam searchParam = SearchParam.newBuilder()
    .withCollectionName("documents")
    .withVectorFieldName("embedding")
    .withFloatVectors(queryVector)  // 查询向量
    .withTopK(10)
    .withMetricType(MetricType.COSINE)
    .withVectorParams("{\"ef\":50}")  // HNSW查询参数
    .withExpr("metadata[\"source\"] == \"wiki\"")  // 元数据过滤
    .withOutFields(Arrays.asList("content", "metadata"))
    .build();
SearchResults results = client.search(searchParam);

// 6. 处理结果
for (QueryResult result : results.getResults()) {
    System.out.println("Score: " + result.getScore());
    System.out.println("Content: " + result.get("content"));
}
```

## 4. Qdrant实战

### 4.1 Qdrant概述

```
Qdrant — Rust编写的高性能向量数据库

  特点:
    1. Rust — 高性能+内存安全
    2. 过滤强 — 向量+Payload(JSON)混合过滤
    3. 轻量 — 单二进制部署
    4. 简单API — REST/gRPC

  适用: 中小规模(100万-1亿)、需要复杂过滤
```

```java
// Qdrant Java Client
import io.qdrant.client.QdrantClient;
import io.qdrant.client.QdrantGrpcClient;

QdrantClient qdrant = new QdrantClient(
    QdrantGrpcClient.newBuilder("localhost", 6334, false).build()
);

// 创建Collection
qdrant.createCollectionAsync("documents",
    VectorParams.newBuilder()
        .setSize(1024)
        .setDistance(Distance.Cosine)
        .build()
).get();

// 插入 + 过滤
qdrant.upsertAsync("documents",
    List.of(
        PointStruct.newBuilder()
            .setId(id1)
            .setVectors(Vectors.newBuilder()
                .setVector(Vector.newBuilder().addAllVector(embedding1).build())
                .build())
            .putAllPayload(Map.of(
                "content", Value.newBuilder().setStringValue("文本内容").build(),
                "source", Value.newBuilder().setStringValue("wiki").build(),
                "timestamp", Value.newBuilder().setIntegerValue(20260713).build()
            ))
            .build()
    )
).get();

// 检索 + 过滤
List<ScoredPoint> results = qdrant.searchAsync(
    SearchPoints.newBuilder()
        .setCollectionName("documents")
        .addAllVector(queryVector)
        .setLimit(10)
        .setFilter(Filter.newBuilder()
            .addMust(FieldCondition.newBuilder()
                .setKey("source")
                .setMatch(Match.newBuilder().setKeyword("wiki").build())
                .build())
            .addMust(FieldCondition.newBuilder()
                .setKey("timestamp")
                .setRange(Range.newBuilder().setGte(20260101).build())
                .build())
            .build())
        .setWith(WithPayloadSelector.newBuilder().setEnable(true).build())
        .build()
).get();
```

## 5. pgvector实战

### 5.1 pgvector概述

```
pgvector — PostgreSQL向量扩展

  特点:
    1. 最简单 — 已有PostgreSQL直接加扩展
    2. 事务支持 — ACID(其他向量库通常不支持)
    3. 混合查询 — SQL + 向量混合
    4. 中小规模 — 100万以下推荐
    5. 运维简单 — 和PG一样运维
```

```sql
-- 1. 安装扩展
CREATE EXTENSION vector;

-- 2. 建表
CREATE TABLE documents (
    id BIGSERIAL PRIMARY KEY,
    content TEXT,
    embedding VECTOR(1024),  -- BGE-M3的1024维
    metadata JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 3. 创建索引
-- HNSW索引(推荐)
CREATE INDEX idx_documents_embedding
ON documents USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 200);

-- 或IVFFlat索引
CREATE INDEX idx_documents_embedding_ivf
ON documents USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- 4. 插入数据
INSERT INTO documents (content, embedding, metadata)
VALUES (
    'Java是一门面向对象的编程语言',
    '[0.3, -0.5, 0.8, ...]'::vector,
    '{"source": "wiki", "author": "张三"}'::jsonb
);

-- 5. 向量检索 + SQL混合
SELECT
    id,
    content,
    metadata,
    1 - (embedding <=> '[0.3, -0.5, ...]'::vector) AS similarity
FROM documents
WHERE metadata->>'source' = 'wiki'  -- SQL过滤
ORDER BY embedding <=> '[0.3, -0.5, ...]'::vector  -- 向量排序
LIMIT 10;

-- <=> = 余弦距离
-- <-> = 欧氏距离
-- <#> = 负内积

-- 6. 混合检索(BM25 + 向量)
WITH vector_search AS (
    SELECT id, content,
           1 - (embedding <=> query_vec) AS vector_score
    FROM documents
    ORDER BY embedding <=> query_vec
    LIMIT 50
),
keyword_search AS (
    SELECT id, content,
           ts_rank_cd(content_tsv, query) AS keyword_score
    FROM documents
    WHERE content_tsv @@ to_tsquery('chinese', 'Java & 并发')
    LIMIT 50
)
SELECT
    COALESCE(v.id, k.id) AS id,
    COALESCE(v.content, k.content) AS content,
    COALESCE(v.vector_score, 0) * 0.7 +
    COALESCE(k.keyword_score, 0) * 0.3 AS final_score
FROM vector_search v
FULL OUTER JOIN keyword_search k ON v.id = k.id
ORDER BY final_score DESC
LIMIT 10;
```

## 6. 混合检索

### 6.1 为什么需要混合检索

```
纯向量检索的问题:
  "Java 8的Stream API" → 向量检索可能返回"Java流式处理"(语义近但关键词不同)
  但用户就是要找"Stream API"这个特定关键词

纯关键词检索(BM25)的问题:
  "Java多线程" → BM25搜不到"Java并发编程"(关键词不同)

混合检索 = 向量(语义) + BM25(关键词) → 取长补短

  ┌──────────────────────────────────────────────┐
  │  查询: "Java并发编程"                         │
  │                                              │
  │  向量检索: Java多线程, Java线程安全(语义近)    │
  │  BM25检索: Java并发, 并发编程(关键词匹配)     │
  │                                              │
  │  融合: 两种结果合并 → 最佳排名                 │
  └──────────────────────────────────────────────┘
```

### 6.2 融合策略

```
Reciprocal Rank Fusion(RRF) — 最常用:

  score(d) = Σ 1/(k + rank_i(d))

  k: 平滑参数(通常60)
  rank_i(d): 文档d在第i个检索系统中的排名

  示例:
    向量检索: 文档A排第1, 文档B排第3
    BM25检索: 文档A排第5, 文档B排第1

    RRF(A) = 1/(60+1) + 1/(60+5) = 0.0164 + 0.0154 = 0.0318
    RRF(B) = 1/(60+3) + 1/(60+1) = 0.0159 + 0.0164 = 0.0323

    → B排名更高(BM25第1 + 向量第3 优于 A的向量第1+BM25第5)

  加权融合:
    final_score = w1 × vector_score + w2 × keyword_score
    典型: w1=0.7, w2=0.3 (向量为主,关键词补充)

  重排序(Rerank):
    1. 向量检索取Top-50
    2. BM25检索取Top-50
    3. 融合后取Top-20
    4. Cross-encoder Rerank精排 → Top-10最终结果
    → 三级检索: 向量召回→融合→Rerank精排
```

## 7. 性能调优

### 7.1 索引选择

```
┌──────────────────────────────────────────────────────┐
  │  数据规模      │  推荐索引        │  典型延迟        │
  │  ──────     │  ──────      │  ──────      │
  │  <10万        │  Flat(暴力)      │  <1ms           │
  │  10万-100万   │  HNSW            │  1-5ms          │
  │  100万-1亿    │  HNSW/IVF_PQ    │  5-20ms         │
  │  >1亿         │  DiskANN         │  10-50ms        │
  └──────────────────────────────────────────────────────┘

  HNSW调优:
    M=16 (默认，精度与内存平衡)
    M=32 (高精度，内存翻倍)
    ef_construction=200 (构建质量)
    ef_search=50 (查询精度，越大越精确)
      ef_search=10 → 快但召回低
      ef_search=100 → 慢但召回高
      建议: ef_search = 2 × top_k(至少)

  IVF调优:
    nlist = sqrt(N) × 10 (N=数据量)
    nprobe = nlist / 100 (查询时的簇数)
    例: 100万数据 → nlist=10000, nprobe=100
```

### 7.2 数据库选型

```
┌──────────────────────────────────────────────────────────┐
  │  向量库      │  规模       │  优势          │  适用      │
  │  ──────   │  ────     │  ────       │  ────   │
  │  Chroma     │  <100万    │  最简单        │  原型/开发 │
  │  pgvector   │  <100万    │  事务+SQL      │  已有PG   │
  │  Qdrant     │  <1亿      │  过滤+性能     │  生产     │
  │  Milvus     │  <100亿    │  分布式        │  大规模   │
  │  Pinecone   │  无限      │  全托管        │  免运维   │
  │  ES         │  <1亿      │  全文+向量     │  已有ES   │
  └──────────────────────────────────────────────────────────┘

  选型建议:
    开发阶段 → Chroma(最简单) 或 pgvector(如有PG)
    生产中小规模 → Qdrant(过滤强) 或 pgvector(事务)
    生产大规模 → Milvus(分布式)
    不想运维 → Pinecone(云服务)
    已有ES → Elasticsearch 8.x(向量+全文)
```

## 8. 面试题速查

**Q1: HNSW的原理？**

```
分层小世界图: 多层图结构
上层稀疏(快速定位区域)→下层密集(精确搜索)
查询: 从顶层开始→逐层下降→Layer 0精确搜索
参数: M(连接数)、ef_construction(构建)、ef_search(查询)
主流向量数据库默认选择
```

**Q2: 向量检索和BM25的区别？**

```
向量检索: 语义相似度("Java并发"≈"Java多线程")
BM25: 关键词匹配(精确但无语义)
混合检索: 向量+BM25融合，取长补短
Rerank: Cross-encoder精排，融合后Top-20→Top-10
```

**Q3: 如何选择向量数据库？**

```
开发: Chroma(简单) / pgvector(有PG)
中小规模生产: Qdrant(过滤强) / pgvector(事务)
大规模生产: Milvus(分布式)
免运维: Pinecone(云服务)
已有ES: ES 8.x向量检索
关键维度: 规模/过滤/事务/运维/成本
```

**Q4: PQ乘积量化是什么？**

```
将向量切分成m个子向量→每个子向量独立K-means→编码为m字节
1024维×4字节=4096字节 → 8字节(压缩512倍)
精度损失约2-5%召回率
适用: 内存有限的亿级数据(IVF+PQ组合)
```

**Q5: 混合检索的融合策略？**

```
RRF(Reciprocal Rank Fusion): 1/(k+rank)求和，最常用
加权融合: w1×向量分 + w2×关键词分
Rerank: 融合后用Cross-encoder精排
推荐: 向量Top50 + BM25 Top50 → RRF融合 → Rerank Top20 → 最终Top10
```
