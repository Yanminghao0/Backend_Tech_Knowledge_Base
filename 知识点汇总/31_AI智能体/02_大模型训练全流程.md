## 1. 训练流程概览

```
大模型训练三阶段(类比人类学习):

  ┌──────────────────────────────────────────────────────┐
  │  阶段1: 预训练         │  类比: 读万卷书(学知识)        │
  │  Pre-training          │  海量文本 → 自回归预测下一token │
  │                        │  学会语言规则和世界知识          │
  │                        │  产出: 基座模型(Base Model)     │
  ├────────────────────────┼─────────────────────────────┤
  │  阶段2: SFT            │  类比: 做练习题(学格式)        │
  │  监督微调               │  指令-响应数据对               │
  │                        │  学会听懂指令并回答             │
  │                        │  产出: 对话模型(Chat Model)    │
  ├────────────────────────┼─────────────────────────────┤
  │  阶段3: RLHF/DPO      │  类比: 老师反馈(学好坏)        │
  │  对齐训练               │  人类偏好数据                   │
  │                        │  学会安全/有用/诚实             │
  │                        │  产出: 对齐模型(Aligned Model)  │
  └────────────────────────┴─────────────────────────────┘

  能力层次:
    预训练 → 知识(知道"法国首都是巴黎")
    SFT → 指令跟随(知道怎么回答"法国首都是哪？")
    RLHF → 对齐(知道不该回答"如何制造炸弹")

  数据量:
    预训练: 万亿token (T级别，如2T tokens)
    SFT: 百万级样本 (如100万条指令)
    RLHF: 十万级偏好对 (如20万对)
```

## 2. 预训练(Pre-training)

### 2.1 训练目标

```
因果语言建模(Causal Language Modeling, CLM):
  给定前面的token，预测下一个token

  输入: "Java是一门"
  目标: "编"  (下一个token)

  输入: "Java是一门编"
  目标: "程"

  输入: "Java是一门编程"
  目标: "语"

  损失函数: 交叉熵
    L = -Σ log P(token_i | token_{<i})

  特点:
    1. 自监督 — 不需要人工标注，原始文本即可
    2. 通用 — 同一目标学习所有语言能力
    3. 可扩展 — 数据越多模型越大效果越好(Scaling Laws)
```

### 2.2 预训练数据

```
┌──────────────────────────────────────────────────────┐
  │  数据来源          │  占比    │  说明                │
  │  ──────         │  ────  │  ────             │
  │  Common Crawl    │  60-70% │  网页爬虫(最大来源)   │
  │  Wikipedia       │  5-10%  │  百科(高质量)         │
  │  书籍            │  5-10%  │  Books3/Gutenberg   │
  │  代码            │  5-10%  │  GitHub/StackOverflow│
  │  学术论文        │  3-5%   │  arXiv/PubMed       │
  │  新闻            │  2-5%   │  新闻文章             │
  │  对话数据        │  2-5%   │  Reddit/论坛          │
  │  数学数据        │  2-5%   │  数学证明/竞赛         │
  └──────────────────────────────────────────────────────┘

  数据清洗流程:
    1. 去重 — 精确去重(MD5) + 模糊去重(MinHash/SimHash)
    2. 质量过滤 — 启发式规则/分类器评分/语言检测
    3. 毒性过滤 — 有害内容检测器
    4. PII脱敏 — 个人信息(电话/邮箱/身份证)删除或替换
    5. 格式清洗 — 去除HTML标签/特殊字符/乱码
    6. 长度过滤 — 去除过短(无信息)或过长(质量差)文档

  数据配比(Different Languages/Domains):
    英文:中文:代码 = 50:30:20 (Qwen典型配比)
    过多英文 → 中文能力弱
    过多代码 → 日常对话能力弱
    需要实验找到最佳配比
```

### 2.3 训练超参数

```
预训练关键超参数:

  ┌──────────────────────────────────────────────────────┐
  │  参数              │  典型值(7B模型)  │  说明           │
  │  ──────         │  ──────────   │  ────        │
  │  学习率            │  3e-4           │  峰值，预热后衰减  │
  │  学习率调度        │  Cosine         │  预热+余弦衰减    │
  │  预热步数          │  2000           │  线性预热         │
  │  Batch Size       │  4M tokens      │  有效批次(梯度累积)│
  │  序列长度          │  4096           │  训练时的上下文    │
  │  训练步数          │  ~500K steps    │  约2T tokens     │
  │  优化器            │  AdamW          │  β1=0.9,β2=0.95  │
  │  权重衰减          │  0.1            │  L2正则化         │
  │  梯度裁剪          │  1.0            │  防止梯度爆炸      │
  │  精度              │  BF16           │  混合精度训练      │
  │  ZeRO阶段          │  ZeRO-2/3       │  分布式分片        │
  └──────────────────────────────────────────────────────┘

  Chinchilla最优配比:
    训练数据量 ≈ 20 × 模型参数量
    例: 7B模型 → 最优训练140B tokens
    但实践中往往overtrain(训练更多数据)以降低推理成本
    LLaMA-3 8B训练了15T tokens(远超Chinchilla最优)
```

### 2.4 Scaling Laws

```
Scaling Laws(缩放定律) — 模型越大效果越好

  ┌──────────────────────────────────────────────────┐
  │  Kaplan定律(OpenAI 2020):                         │
  │  Loss ∝ N^-0.076 (参数量N)                        │
  │  Loss ∝ D^-0.095 (数据量D)                        │
  │  Loss ∝ C^-0.05 (算力C)                           │
  │  → 参数量对Loss影响最大                             │
  │                                                  │
  │  Chinchilla定律(DeepMind 2022):                   │
  │  最优: 数据和参数等比例增长                         │
  │  给定算力C, 最优: N ∝ C^0.5, D ∝ C^0.5            │
  │  → 70B模型应训练1.4T tokens                       │
  └──────────────────────────────────────────────────┘

  涌现能力(Emergent Abilities):
    模型规模到达某个阈值后，突然出现新能力
    例: 小模型无法做数学推理，大模型突然可以
    典型涌现能力: CoT推理、少样本学习、代码生成

  注意: 新研究表明小模型也能通过训练数据质量提升获得"涌现"能力
    → 可能不是规模导致，而是训练策略导致
```

## 3. SFT监督微调

### 3.1 SFT目标

```
SFT(Supervised Fine-Tuning) = 用指令-响应对微调基座模型

  预训练模型的问题:
    输入: "法国的首都是"
    输出: "巴黎" (续写，但可能续写"巴黎，它是..."而非直接回答)

    输入: "请告诉我法国的首都"
    输出: "请告诉我法国的首都和它的人口" (续写，而非回答)

  SFT后:
    输入: "请告诉我法国的首都"
    输出: "法国的首都是巴黎。" (正确回答)

  SFT让模型学会:
    1. 听懂指令 — 理解"请告诉我"是要求回答
    2. 格式对齐 — 以对话格式回答，而非续写
    3. 角色定位 — 作为AI助手回答问题
```

### 3.2 数据格式

```
Alpaca格式(最经典):
    {
      "instruction": "将以下句子翻译成英文",
      "input": "今天天气很好",
      "output": "The weather is nice today."
    }

  ShareGPT格式(多轮对话):
    {
      "conversations": [
        {"from": "human", "value": "你好"},
        {"from": "gpt", "value": "你好！有什么可以帮你的？"},
        {"from": "human", "value": "解释一下什么是递归"},
        {"from": "gpt", "value": "递归是一种..."}
      ]
    }

  ChatML格式(Qwen/通义千问用):
    <|im_start|>system
    你是一个AI助手<|im_end|>
    <|im_start|>user
    你好<|im_end|>
    <|im_start|>assistant
    你好！有什么可以帮你的？<|im_end|>

  Mistral格式:
    [INST]你好[/INST]你好！有什么可以帮你的？</s>[INST]解释递归[/INST]

  注意: 不同模型有不同的对话模板(Special Tokens)
    SFT时必须用正确的模板
    推理时也必须用相同的模板
    否则模型性能会严重下降
```

### 3.3 数据构建

```
SFT数据来源:

  1. 人工标注(质量最高)
     标注员编写指令和高质量回答
     成本高(每条$0.5-5)、速度慢
     代表: OpenAI的SFT数据

  2. Self-Instruct(用大模型生成)
     用GPT-4/Claude生成指令-响应对
     成本低、量大
     质量取决于生成模型
     流程: 种子指令 → LLM扩展 → 质量过滤

  3. 公开数据集
     Alpaca(52K) / ShareGPT / FLAN / Dolly
     开源免费但质量参差

  4. Evol-Instruct(进化指令)
     逐步增加指令复杂度
     简单指令 → 加深/加宽 → 复杂指令
     WizardLM用的方法

  数据质量 > 数据数量:
    1000条高质量 > 100000条低质量
    LIMA论文: 1000条高质量SFT数据就能显著提升
```

## 4. RLHF人类反馈强化学习

### 4.1 为什么需要RLHF

```
SFT的问题:
  SFT学的是"怎么回答"的格式
  但不区分"好回答"和"坏回答"
  可能学到: 有害内容/幻觉/不安全回答

  RLHF的目标:
    让模型学会"什么是好的回答"
    安全(Safe) — 不生成有害内容
    有用(Helpful) — 给出有用的信息
    诚实(Honest) — 不编造虚假信息

  类比:
    预训练 = 读万卷书(学知识)
    SFT = 做练习题(学格式)
    RLHF = 老师反馈(学好坏标准)
```

### 4.2 RLHF三阶段

```
RLHF = SFT + 奖励模型 + PPO

  ┌──────────────────────────────────────────────────────┐
  │  阶段1: SFT (已完成)                                  │
  │  用指令-响应数据微调基座模型                            │
  │  产出: SFT模型 π_SFT                                  │
  ├──────────────────────────────────────────────────────┤
  │  阶段2: 奖励模型训练(Reward Model)                    │
  │  人类标注: 对同一问题，给两个回答排序(A>B)              │
  │  训练: 奖励模型学习预测人类偏好                         │
  │  损失: L = -log(σ(r(x,y_winning) - r(x,y_losing)))  │
  │  产出: 奖励模型 r(x,y) → score                       │
  ├──────────────────────────────────────────────────────┤
  │  阶段3: PPO强化学习                                    │
  │  SFT模型生成回答 → 奖励模型打分 → PPO优化              │
  │  目标: max E[r(x,y)] - β·KL(π || π_SFT)              │
  │  KL散度项防止偏离SFT模型太远(防止"奖励黑客")           │
  │  产出: RLHF模型 π_RLHF                                │
  └──────────────────────────────────────────────────────┘

  奖励模型训练详解:
    数据: (prompt, response_A, response_B, preference)
      preference = "A>B" 或 "B>A"
    标注员: 比较两个回答哪个更好(安全性/有用性/诚实性)
    模型: 从SFT模型初始化，去掉LM Head，加一个标量输出头
    损失: 让好回答的分数比坏回答高

  PPO训练详解:
    1. SFT模型对prompt生成多个回答
    2. 奖励模型对每个回答打分
    3. 用分数作为奖励信号，PPO更新策略(SFT模型)
    4. KL散度约束防止模型退化(只追求高分而乱说)

  RLHF的问题:
    1. 复杂 — 需要训练奖励模型+PPO，工程复杂
    2. 不稳定 — PPO训练容易不稳定
    3. 奖励黑客 — 模型找到奖励模型的漏洞而非真正变好
    4. 成本高 — 需要大量人类标注
    → DPO是简化方案
```

## 5. DPO直接偏好优化

### 5.1 DPO原理

```
DPO(Direct Preference Optimization) — 不需要奖励模型和PPO

  核心思想:
    RLHF: 偏好数据 → 训练奖励模型 → PPO优化策略
    DPO:  偏好数据 → 直接优化策略(跳过奖励模型)

  数学等价:
    DPO证明了最优策略可以通过偏好数据直接求解
    不需要显式的奖励模型和强化学习

  DPO损失函数:
    L_DPO = -log σ(β·log(π(y_w)/π_ref(y_w)) - β·log(π(y_l)/π_ref(y_l)))

    y_w = 偏好的回答(chosen)
    y_l = 不偏好的回答(rejected)
    π = 当前策略(训练中)
    π_ref = 参考策略(SFT模型，固定不变)
    β = 温度参数(控制偏离程度)

  直觉:
    让模型对chosen回答的概率高于rejected回答
    同时不偏离参考模型太远(KL约束内置在公式中)
```

### 5.2 DPO vs RLHF

```
┌──────────────────────────────────────────────────────┐
  │  维度        │  RLHF            │  DPO              │
  │  ──────    │  ──────        │  ──────         │
  │  奖励模型    │  需要            │  不需要            │
  │  强化学习    │  需要(PPO)       │  不需要            │
  │  训练流程    │  3阶段(SFT+RM+PPO)│ 2阶段(SFT+DPO)  │
  │  稳定性      │  较差(PPO不稳)   │  好(分类任务)      │
  │  工程复杂度  │  高              │  低               │
  │  效果        │  略好(理论上)    │  接近RLHF         │
  │  计算成本    │  高(4个模型)     │  低(2个模型)       │
  │  显存        │  大              │  小               │
  └──────────────────────────────────────────────────────┘

  RLHF需要同时加载4个模型:
    1. 策略模型(训练) 2. 参考模型(固定) 3. 奖励模型(固定) 4. 价值模型(训练)

  DPO只需2个模型:
    1. 策略模型(训练) 2. 参考模型(固定)

  实践建议:
    先用DPO(简单、稳定、效果好)
    如果效果不够再考虑RLHF
    大多数场景DPO已经够用
```

### 5.3 其他对齐方法

```
ORPO(Odds Ratio Preference Optimization):
    不需要参考模型(比DPO更简化)
    将SFT和偏好优化合并为一步
    损失 = SFT Loss + 偏好Loss

  KTO(Kahneman-Tversky Optimization):
    不需要成对的偏好数据
    只需要二元反馈(好/坏)
    基于前景理论(人类对损失更敏感)

  SimPO(Simple Preference Optimization):
    不需要参考模型
    用序列长度归一化

  GRPO(Group Relative Policy Optimization):
    DeepSeek提出的RL算法
    不需要价值模型(用组内相对优势代替)
    DeepSeek-R1推理模型用的就是GRPO

  Constitutional AI(Anthropic):
    AI自我反馈对齐
    给定原则(Constitution) → AI评估自己的回答 → 自我修正
    减少人类标注需求
```

## 6. 分布式训练

### 6.1 并行策略

```
┌──────────────────────────────────────────────────────┐
  │  策略          │  原理           │  适用场景         │
  │  ──────     │  ────        │  ──────      │
  │  数据并行(DDP) │  每GPU完整模型  │  模型装得下单GPU   │
  │  张量并行(TP)  │  模型层内切分   │  单层装不下        │
  │  流水线并行(PP)│  模型层间切分   │  层数多            │
  │  ZeRO         │  优化器状态分片  │  通用加速          │
  │  FSDP         │  ZeRO-3的PyTorch原生实现│  通用加速   │
  └──────────────────────────────────────────────────────┘

  数据并行(DDP):
    GPU1: 模型完整 + 数据batch1 → 梯度1
    GPU2: 模型完整 + 数据batch2 → 梯度2
    → 梯度AllReduce平均 → 各GPU更新
    问题: 每GPU装完整模型(显存浪费)

  张量并行(TP):
    将每一层的权重矩阵按行/列切分到多GPU
    GPU1: W的左半 → GPU2: W的右半
    每层需要AllReduce通信
    适合: 单层参数大(如70B模型的FFN)

  流水线并行(PP):
    GPU1: Layer 1-20
    GPU2: Layer 21-40
    数据像流水线一样通过
    问题: 气泡(某GPU等待前一GPU)
    解决: Micro-batching(将batch切小，交错执行)

  3D并行 = DP + TP + PP (训练超大模型)
    如GPT-3 175B: 约10000个V100 GPU(3D并行示意: TP×PP×DP)
```

### 6.2 ZeRO优化

```
ZeRO(Zero Redundancy Optimizer) — 分片减少冗余

  ┌──────────────────────────────────────────────────────┐
  │  阶段      │  分片内容              │  显存节省        │
  │  ────   │  ──────            │  ──────      │
  │  ZeRO-1  │  优化器状态(Adam)      │  4x             │
  │  ZeRO-2  │  优化器状态 + 梯度     │  8x             │
  │  ZeRO-3  │  优化器+梯度+模型参数  │  N×(N=GPU数)    │
  └──────────────────────────────────────────────────────┘

  举例(7B模型 FP16 + AdamW):
    模型参数: 14GB (FP16)
    梯度: 14GB (FP16)
    优化器状态: 84GB (FP32主参数28GB + FP32的m和v各28GB)
    总计: 112GB → 单卡A100(80GB)装不下

  ZeRO-3分片到8卡:
    每卡: 112/8 = 14GB → 轻松装下
    代价: 前向/反向需要All-Gather通信

  FSDP(Fully Sharded Data Parallel):
    PyTorch原生的ZeRO-3实现
    比DeepSpeed更易用(PyTorch原生支持)
    生产中FSDP和DeepSpeed ZeRO-3都常用
```

### 6.3 混合精度训练

```
混合精度 = FP32主参数 + FP16/BF16计算

  ┌──────────────────────────────────────────────────┐
  │  精度    │  字节  │  范围           │  适用        │
  │  ────  │  ────│  ──────      │  ────     │
  │  FP32   │  4     │  ±3.4×10³⁸     │  主参数/损失  │
  │  FP16   │  2     │  ±65504         │  前向/反向    │
  │  BF16   │  2     │  ±3.4×10³⁸     │  前向/反向(推荐)│
  │  FP8    │  1     │  有限           │  实验中       │
  └──────────────────────────────────────────────────┘

  FP16 vs BF16:
    FP16: 精度高但范围小 → 容易溢出(梯度爆炸)
    BF16: 精度低但范围大 → 不溢出(与FP32同范围)
    → BF16是大模型训练首选(A100/H100支持)

  混合精度训练流程:
    1. 主参数保持FP32(精度)
    2. 前向传播: FP32→BF16转换 → BF16计算
    3. 反向传播: BF16梯度
    4. 梯度缩放(FP16需要) → FP32梯度
    5. 优化器更新FP32主参数

  梯度检查点(Gradient Checkpointing):
    不保存中间激活值 → 反向传播时重新计算
    以约30%的时间换约60%的显存
    大模型训练几乎必开
```

## 7. 面试题速查

**Q1: 预训练、SFT、RLHF分别做什么？**

```
预训练: 海量无标注文本→自回归预测下一token→学语言和知识
SFT: 指令-响应对→微调→学指令跟随和对话格式
RLHF: 人类偏好→奖励模型+PPO→学安全有用诚实
类比: 读书→做题→老师反馈
```

**Q2: 为什么需要DPO替代RLHF？**

```
RLHF: 需要训练奖励模型+PPO，4个模型同时在线，复杂且不稳定
DPO: 直接从偏好数据优化策略，2个模型，简单且稳定
DPO数学上等价于RLHF的最优解，但更易实现
大多数场景DPO效果已足够
```

**Q3: 数据并行和模型并行的区别？**

```
数据并行(DDP): 每GPU完整模型+不同数据 → 梯度平均
模型并行(TP/PP): 模型切分到多GPU → 通信开销大但能装大模型
ZeRO: 数据并行基础上分片优化器/梯度/参数
生产中常用3D并行(DP+TP+PP)或ZeRO-3/FSDP
```

**Q4: BF16为什么比FP16好？**

```
FP16: 精度高但范围小(±65504)→大梯度溢出→训练崩溃
BF16: 精度低但范围与FP32相同(±3.4×10³⁸)→不溢出
大模型训练首选BF16(A100/H100原生支持)
```

**Q5: Chinchilla定律说明了什么？**

```
给定算力，数据和参数应等比例增长
最优: N ∝ C^0.5, D ∝ C^0.5
70B模型最优训练1.4T tokens
但实践中overtrain(如LLaMA-3 8B训15T)以降低推理成本
```

**Q6: 梯度检查点是什么？**

```
反向传播需要前向的中间激活值
不保存→反向时重新计算(以时间换显存)
省约60%显存，慢约30%
大模型训练几乎必开(否则显存不够)
```
