## 2. Cursor

### 核心特性

```
Cursor = VS Code Fork + 深度AI集成

优势：
  1. 全项目理解：索引整个代码库，跨文件理解
  2. Agent模式：自主完成多步骤编码任务
  3. 多模型支持：GPT-5/Claude 4/Claude 4 Opus
  4. 代码库聊天：@Codebase 引用整个项目
  5. Composer：多文件同时编辑
  6. 终端AI：命令行辅助
```

### 核心功能

```
Tab补全：基于上下文的智能补全（比Copilot更准）
Cmd+K：选中代码→AI修改/重构/解释
Cmd+L：侧边栏聊天，@文件引用
Composer：多文件编辑模式（Ctrl+I）
Agent：自主完成"添加一个用户注册功能"等复杂任务
@Codebase：全项目搜索+理解
@Web：联网搜索最新文档
```

### 适用场景

```
✅ 从零搭建项目（Agent模式）
✅ 跨文件重构
✅ 理解大型代码库
✅ 快速原型开发
```

## 7. 面试要点

### Q1: AI编程助手的原理？

```
代码补全：
  1. 收集上下文：当前文件+光标位置+导入语句+相关文件
  2. 构建Prompt：上下文 + "请补全以下代码"
  3. LLM生成：流式输出补全建议
  4. Ghost Text：灰色文本展示，Tab确认

全项目理解：
  1. 代码库索引：向量化所有文件
  2. 检索相关文件：根据当前任务检索相关代码
  3. 构建上下文：相关代码 + 当前文件
  4. LLM生成：基于完整上下文的回答
```

### Q2: Cursor和Copilot的区别？

```
Cursor：
  - 独立IDE（VS Code Fork）
  - Agent模式（自主完成多步任务）
  - Composer（多文件同时编辑）
  - 全项目理解更深
  - 多模型可选

Copilot：
  - 插件形式（多IDE支持）
  - GitHub生态集成
  - 行内补全体验好
  - 企业管理功能
```

### Q3: 如何安全使用AI编程助手？

```
1. 敏感代码：不发送到云端（用Continue+本地模型）
2. 代码审查：AI生成的代码必须人工审查
3. 安全扫描：AI代码可能引入漏洞
4. 版权合规：AI可能生成与开源项目相似的代码
5. 企业策略：制定AI编程工具使用规范
6. 敏感信息：不在Prompt中包含密码/密钥
```

## 8. Cursor Agent模式详解

### Agent模式工作原理

```
Agent模式 = LLM自主决策 + 工具调用 + 多步执行

核心循环（ReAct模式）：
  1. 理解任务：解析用户自然语言指令
  2. 制定计划：拆解为多个子步骤
  3. 执行工具：读写文件 / 运行命令 / 搜索代码
  4. 观察结果：读取执行输出
  5. 调整策略：根据结果修正计划
  6. 循环直到任务完成

与传统补全的区别：
  补全：你写代码，AI猜下一行（被动）
  聊天：你提问，AI回答（问答）
  Agent：你下指令，AI自主完成（主动）
```

### Agent可用工具集

```
文件操作：
  - read_file：读取指定文件内容
  - edit_file：精确修改文件（find-replace）
  - create_file：创建新文件
  - delete_file：删除文件

代码检索：
  - grep_search：正则搜索代码库
  - file_search：按文件名查找
  - codebase_search：语义搜索（向量化检索）

终端：
  - run_command：执行shell命令
  - run_tests：运行测试套件

网络：
  - web_search：联网搜索文档
  - web_fetch：抓取网页内容

外部能力（MCP）：
  - 通过MCP协议接入数据库 / API / Jira / Slack等
```

### Agent模式典型工作流

```
场景：用户输入"添加一个用户注册接口，包含邮箱验证"

Agent执行过程：
  Step 1: 搜索现有代码结构
    → grep "Controller" 了解项目分层
    → 读取 UserController.java 了解现有风格

  Step 2: 分析数据模型
    → 读取 User.java 实体类
    → 读取 UserRepository.java

  Step 3: 生成代码
    → 创建 UserRegisterDTO.java（请求体）
    → 修改 UserController.java 添加注册接口
    → 创建 EmailService.java 发送验证邮件
    → 修改 application.yml 添加邮件配置

  Step 4: 自验证
    → 运行 ./mvnw compile 检查编译
    → 若报错，自动读取错误并修复

  Step 5: 生成测试
    → 创建 UserControllerTest.java
    → 运行测试，根据失败修正代码

  Step 6: 汇报结果
    → 列出所有改动文件
    → 说明实现逻辑和注意事项
```

### Agent模式使用技巧

```
1. 提供清晰上下文：
   ❌ "加个登录功能"
   ✅ "基于现有的UserController风格，添加JWT登录接口，
      密码用BCrypt加密，参考 @UserController 的代码风格"

2. 分阶段下指令：
   复杂任务拆分为多个Agent会话，每个聚焦一个子目标
   避免单次任务过大导致Agent迷失

3. 及时Review：
   Agent每步执行后查看diff，用Cmd+K微调
   不要等全部完成才检查

4. 善用@引用：
   @文件名 引用特定文件
   @Codebase 让Agent搜索全项目
   @Web 引用最新文档

5. 设置边界：
   在.cursorrules中定义编码规范
   限制Agent的修改范围（如不碰config文件）

6. 模型选择：
   复杂Agent任务 → Claude 4 Opus / GPT-5（推理强）
   简单补全 → 默认模型即可（省额度）
```

### .cursorrules 配置示例

```
# 项目编码规范（Cursor会遵循）

## 技术栈
- Java 21 + Spring Boot 3.x
- 数据库：MySQL 8.0 + MyBatis-Plus
- 缓存：Redis + Spring Cache

## 代码规范
- Controller只做参数校验和转发，不含业务逻辑
- Service层使用接口+实现类模式
- 所有DTO使用Record定义
- 统一返回Result<T>包装类
- 异常使用全局异常处理器，禁止在Controller捕获

## 命名规范
- 类名：大驼峰
- 方法/变量：小驼峰
- 常量：全大写下划线
- 数据库字段：下划线

## Agent约束
- 不要修改application.yml中的数据库配置
- 新增接口必须同步编写单元测试
- 不要引入新的第三方依赖，除非我确认
```

## 11. 安全使用规范

### 代码安全

```
1. 分级管理：
   🔴 绝密级（核心算法/密钥）：禁止使用任何云端AI
   🟡 内部级（业务代码）：可用企业版（关闭训练）
   🟢 公开级（文档/测试）：无限制

2. 敏感信息过滤：
   - 禁止在Prompt中包含：数据库密码、API Key、
     内部IP、客户数据、商业逻辑
   - 使用占位符替代：如 DB_PASSWORD → <REDACTED>

3. 本地优先：
   涉密项目强制使用Continue + Ollama本地模型
   配置 .cursorignore / .copilotignore 排除敏感目录
```

### 代码审查规范

```
AI生成代码必须人工审查，重点检查：

1. 安全性：
   - SQL注入风险（AI可能拼接SQL）
   - XSS漏洞（未转义输出）
   - 硬编码密钥/密码
   - 不安全的反序列化

2. 正确性：
   - 边界条件处理
   - 空指针/异常处理
   - 并发安全
   - 事务一致性

3. 可维护性：
   - 是否符合项目编码规范
   - 命名是否清晰
   - 是否引入不必要的依赖
   - 是否过度设计

4. 版权合规：
   - AI可能生成与开源项目高度相似的代码
   - 关注License兼容性（如GPL传染性）
   - 企业可配置Copilot重复检测
```

### 企业使用规范模板

```
《AI编程助手使用规范》

第一章 适用范围
  - 适用于全员使用的AI编程工具
  - 涉密项目遵循单独规定

第二章 工具准入
  - 允许：Cursor（开Privacy Mode）/ Copilot Business
  - 禁止：未经审批的其他AI工具
  - 涉密：仅限Continue + 本地模型

第四章 审查要求
  - AI生成代码需标记来源
```

## 12. 面试要点补充（5问）

### Q4: Cursor的Agent模式和普通补全有什么本质区别？

```
本质区别在于"主动性"和"自主性"：

普通补全：
  - 被动响应：用户写代码，AI猜下一行
  - 单点预测：只关注光标位置附近
  - 无工具调用：不能读写文件、运行命令
  - 即时返回：毫秒级延迟

Agent模式：
  - 主动执行：用户下指令，AI自主完成多步任务
  - 全局理解：索引整个项目，跨文件推理
  - 工具调用：可读写文件、运行测试、搜索代码
  - 多轮迭代：观察→调整→再执行，直到完成
  - 自验证：运行编译/测试，根据结果自我修正

技术实现：
  补全 = 单次LLM推理（上下文→补全文本）
  Agent = ReAct循环（思考→行动→观察→再思考）
        + 工具调用（Function Calling）
        + 代码库语义检索（RAG）
```

### Q5: AI编程助手如何实现全项目理解？

```
核心技术：RAG（检索增强生成）+ 代码库索引

实现步骤：
  1. 索引构建（首次打开项目时）：
     - 遍历所有源文件
     - 按语义切分（函数/类/块）
     - 用Embedding模型向量化
     - 存入向量数据库（如本地SQLite+向量索引）

  2. 查询时检索：
     - 用户问题 → 向量化
     - 向量相似度搜索 → Top-K相关代码块
     - 可能结合：AST解析、符号表、调用关系图

  3. 上下文构建：
     相关代码片段 + 当前文件 + 用户问题
     → 组装为Prompt送入LLM

  4. 生成回答：
     LLM基于检索到的完整上下文生成回答

Cursor增强：
  - 增量索引：文件修改时只更新变化部分
  - 混合检索：向量检索 + 关键词检索
  - 代码结构理解：解析import/调用链/继承关系
```

### Q6: 企业引入AI编程助手面临哪些风险？如何应对？

```
风险一：代码泄露
  风险：代码发送到云端可能泄露商业机密
  应对：①启用Privacy Mode ②敏感项目用本地模型
        ③配置.ignore排除敏感目录

风险二：代码质量
  风险：AI生成代码可能有隐藏Bug/安全漏洞
  应对：①强制人工Code Review ②集成SAST扫描
        ③关键路径必须有测试覆盖

风险三：版权合规
  风险：AI可能生成与开源项目相似代码，引发版权纠纷
  应对：①启用重复检测 ②关注License兼容性
        ③核心算法人工实现

风险四：过度依赖
  风险：开发者能力退化，不理解AI生成代码
  应对：①要求理解后才能合并 ②新人限制使用
        ③定期考察手写代码能力

风险五：成本失控
  风险：API调用费用不可控
  应对：①设置用量上限 ②监控使用数据
        ③选择合适套餐
```

### Q7: 本地部署AI编程助手（Continue+Ollama）的方案？

```
架构：
  开发者IDE → Continue插件 → Ollama（本地）→ 开源大模型
  代码全程不出本机，满足数据合规

部署步骤：
  1. 安装Ollama：curl -fsSL https://ollama.com/install.sh | sh
  2. 拉取模型：ollama pull qwen3:72b（代码模型）
  3. 拉取Embedding：ollama pull nomic-embed-text
  4. 安装Continue插件（VS Code/JetBrains）
  5. 配置 ~/.continue/config.json 指向本地Ollama

硬件要求：
  qwen3:7b    → 8GB显存（入门）
  qwen3:32b   → 24GB显存（推荐，RTX 4090）
  qwen3:72b   → 48GB+显存（高端，多卡）

权衡：
  ✅ 代码不外传，数据绝对安全
  ✅ 无API费用，长期成本低
  ⚠️ 本地模型能力弱于GPT-5/Claude 4
  ⚠️ 需要GPU硬件投入
  ⚠️ 无云端Agent的深度推理能力
```

### Q8: 如何评估AI编程助手对团队效率的提升？

```
量化指标：

1. 开发效率：
   - 代码提交频率（PR数/人/周）
   - 首次提交到合并时长
   - 故事点完成速度

2. 代码质量：
   - Bug率（千行缺陷数）
   - Code Review返工次数
   - 测试覆盖率变化

3. 使用数据：
   - 补全接受率（建议被采纳比例）
   - Agent任务完成率
   - 聊天使用频次

4. 主观反馈：
   - 开发者满意度问卷
   - 减少重复劳动的感受
   - 学习新技术的效率

评估方法：
  A/B测试：对照组（不用AI）vs 实验组（用AI）
  周期：至少1-2个月，避免 novelty effect（新鲜感效应）

典型结果：
  代码编写速度提升 30-50%
  测试编写效率提升 50%+
  文档编写效率提升 60%+
  Bug率：略有下降（审查到位时）
```
