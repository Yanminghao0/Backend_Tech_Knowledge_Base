## 1. 本地部署概述

### 为什么需要本地部署

```
1. 数据安全：敏感数据不出内网
2. 成本控制：无API费用，一次性GPU投入
3. 低延迟：无网络往返
4. 合规要求：金融/医疗/政务
5. 离线场景：无网络环境
```

### 部署方案对比

| 方案 | 特点 | 适用 | GPU需求 |
|------|------|------|---------|
| Ollama | 一键部署，简单易用 | 开发/09_测试/小规模 | 消费级GPU |
| vLLM | 高性能推理引擎 | 生产环境 | 数据中心GPU(A100) |
| TGI | HuggingFace推理服务 | 生产环境 | 数据中心GPU |
| LM Studio | 桌面GUI应用 | 个人使用 | 消费级GPU |
| llama.cpp | CPU推理，无GPU | 边缘设备 | 无GPU需求 |

## 2. Ollama

### 安装与使用

```bash
# 安装
curl -fsSL https://ollama.com/install.sh | sh

# 拉取模型
ollama pull qwen3:72b        # 通义千问72B
ollama pull llama4:70b       # Llama 4 70B
ollama pull deepseek-v4      # DeepSeek
ollama pull nomic-embed-text # Embedding模型

# 运行
ollama run qwen3:72b
>>> 你好，介绍一下你自己

# API服务（默认端口11434）
curl http://localhost:11434/api/chat -d '{
  "model": "qwen3:72b",
  "messages": [{"role": "user", "content": "你好"}]
}'
```

### Ollama优势

```
1. 一键安装：无需配置CUDA/Python环境
2. 模型管理：ollama pull/run/list
3. OpenAI兼容API：/v1/chat/completions
4. Modelfile：自定义模型（类似Dockerfile）
5. 多平台：Mac/Linux/Windows
6. 自动量化：自动选择适合硬件的量化级别
```

### Modelfile自定义

```dockerfile
# Modelfile（类似Dockerfile）
FROM qwen3:72b

# 系统Prompt
SYSTEM """
你是一个Java技术专家，专注于Spring Boot开发。
回答要简洁专业，包含代码示例。
"""

# 参数
PARAMETER temperature 0.7
PARAMETER top_p 0.9
PARAMETER num_ctx 8192

# 构建自定义模型
# ollama create my-java-expert -f Modelfile
```

## 3. vLLM

### 高性能推理

```bash
# 安装
pip install vllm

# 启动OpenAI兼容API服务
python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen3-72B \
  --tensor-parallel-size 2 \    # 2卡并行
  --max-model-len 32768 \       # 最大上下文
  --port 8000
```

### vLLM核心优化

```
1. PagedAttention：分页式KV Cache管理
   - 减少内存碎片
   - 提高GPU利用率
   - 支持更高并发

2. Continuous Batching：连续批处理
   - 动态组装batch
   - 请求完成即释放槽位
   - 吞吐量提升2-4倍

3. Tensor Parallelism：张量并行
   - 多GPU并行计算
   - 线性扩展吞吐量

4. 量化推理：支持AWQ/GPTQ/INT8
   - 减少显存占用
   - 提升推理速度
```

### 性能对比

```
模型：Llama 4 70B，A100 80G × 2

vLLM:       吞吐 4000 token/s，并发 256
TGI:        吞吐 2800 token/s，并发 128
Ollama:     吞吐 800 token/s，  并发 1

vLLM适合生产高并发，Ollama适合开发测试
```

## 4. LiteLLM

### 统一API代理

```python
# LiteLLM：统一100+模型的API接口

# 启动Proxy Server
# litellm --model ollama/qwen3:72b --port 4000

# 统一调用不同模型
from litellm import completion

# OpenAI
response = completion(model="gpt-5", messages=[...])

# Claude
response = completion(model="claude-4-sonnet", messages=[...])

# 本地Ollama
response = completion(model="ollama/qwen3:72b", messages=[...])

# 本地vLLM
response = completion(
    model="openai/qwen3-72b",
    api_base="http://localhost:8000/v1",
    messages=[...]
)
```

### Proxy模式（推荐生产）

```yaml
# litellm_config.yaml
model_list:
  - model_name: gpt-5
    litellm_params:
      model: openai/gpt-5
      api_key: sk-xxx
  - model_name: claude
    litellm_params:
      model: anthropic/claude-4-sonnet
      api_key: sk-xxx
  - model_name: local
    litellm_params:
      model: ollama/qwen3:72b
      api_base: http://localhost:11434

# 启动
# litellm --config litellm_config.yaml --port 4000

# 所有模型统一通过 http://localhost:4000/v1 访问
# 自动负载均衡、故障转移、成本追踪
```

## 5. 模型量化

### 量化方案对比

| 方案 | 精度损失 | 压缩比 | 速度 | 适用 |
|------|---------|--------|------|------|
| FP16 | 基准 | 1x | 基准 | A100/H100 |
| INT8 | <1% | 2x | +30% | 通用 |
| GPTQ | 1-2% | 3-4x | +50% | 消费级GPU |
| AWQ | <1% | 3-4x | +40% | 消费级GPU（推荐） |
| GGUF | 1-3% | 4-8x | CPU可用 | 无GPU/边缘 |
| INT4 | 2-5% | 4x | +80% | 极致压缩 |

### GGUF格式（Ollama使用）

```bash
# GGUF：llama.cpp的量化格式，支持CPU/GPU混合

# 量化级别
q8_0:  8位量化，精度最高，体积约FP16的50%
q5_K_M: 5位量化，精度好，体积约FP16的35%
q4_K_M: 4位量化，推荐，体积约FP16的25%
q3_K_S: 3位量化，极限压缩

# 70B模型不同量化大小
FP16:    140GB（需2×A100 80G）
q8_0:    70GB（需1×A100 80G）
q4_K_M:  40GB（需2×RTX 4090或CPU offload，单卡24G不够）
q3_K_S:  30GB（Mac M2 32G可跑）
```

## 6. Java调用本地模型

### 通过Ollama API

```java
@Service
public class LocalLlmService {
    
    private final WebClient webClient = WebClient.create("http://localhost:11434");
    
    public String chat(String model, String prompt) {
        return webClient.post()
            .uri("/api/chat")
            .bodyValue(Map.of(
                "model", model,
                "messages", List.of(Map.of("role", "user", "content", prompt)),
                "stream", false
            ))
            .retrieve()
            .bodyToMono(String.class)
            .block();
    }
    
    // 流式响应
    public Flux<String> streamChat(String model, String prompt) {
        return webClient.post()
            .uri("/api/chat")
            .bodyValue(Map.of(
                "model", model,
                "messages", List.of(Map.of("role", "user", "content", prompt)),
                "stream", true
            ))
            .retrieve()
            .bodyToFlux(String.class)
            .map(this::extractContent);
    }
}
```

### Spring AI + Ollama

```yaml
spring:
  ai:
    ollama:
      base-url: http://localhost:11434
      chat:
        model: qwen3:72b
        options:
          temperature: 0.7
          num-ctx: 8192
      embedding:
        model: nomic-embed-text
```

```java

private ChatClient chatClient;

// 直接使用，和调用OpenAI完全一样
String answer = chatClient.prompt()

    .call()
    .content();
```

## 7. 面试要点

### Q1: 什么时候用本地部署？

```
必须本地：
  1. 数据安全合规（金融/医疗/政务）
  2. 无网络环境（边缘/IoT）
  3. 超高并发（API成本过高）

可以云API：
  1. 一般业务场景
  2. 团队无GPU运维能力
  3. 需要最强模型（GPT-5/Claude 4）
```

### Q2: Ollama和vLLM怎么选？

```
Ollama：开发/09_测试/小规模，一键安装，简单易用
vLLM：生产环境，高并发，PagedAttention优化

Ollama：单并发，吞吐800 token/s
vLLM：256并发，吞吐4000 token/s
```

### Q3: 模型量化有什么影响？

```
量化 = 降低精度 → 减少内存 → 提升速度 → 损失精度

FP16 → INT4：
  - 内存减少4倍
  - 速度提升80%
  - 精度损失2-5%

选择：
  - A100/H100：FP16（不差钱）
  - RTX 4090：AWQ INT4（消费级最优）
  - Mac M2：GGUF q4_K_M
  - CPU only：GGUF q3_K_S
```

### Q4: LiteLLM的作用是什么？

```
统一API代理：100+模型统一接口
1. 统一调用方式：OpenAI格式调用所有模型
2. 自动负载均衡：多实例轮询
3. 故障转移：主模型故障自动切换备模型
4. 成本追踪：按模型/用户统计Token消耗
5. 缓存：语义缓存减少重复调用
```
