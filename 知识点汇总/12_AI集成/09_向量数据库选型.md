## 1. 向量数据库概述

### 为什么需要向量数据库

```
传统数据库：精确匹配（WHERE id = 1, WHERE name LIKE '%java%'）
向量数据库：语义相似度搜索（找到"最接近"的向量）

应用场景：
  - RAG知识库问答：检索相关文档片段
  - 语义搜索：理解用户意图而非关键词匹配
  - 推荐系统：找到相似商品/内容
  - 图片搜索：以图搜图
  - 去重检测：语义级别去重
```

### 向量相似度计算

```
余弦相似度（Cosine）：向量夹角，不考虑长度
  cos(A, B) = (A·B) / (|A|×|B|)
  值域：[-1, 1]，越大越相似

欧氏距离（L2）：空间直线距离
  d(A, B) = √(Σ(Ai - Bi)²)
  值域：[0, ∞)，越小越相似

内积（Inner Product）：点积
  IP(A, B) = Σ(Ai × Bi)
  适用于已归一化的向量
```

## 2. 主流方案对比

| 方案 | 类型 | 部署方式 | 性能 | 生态 | 适用规模 |
|------|------|---------|------|------|---------|
| Milvus | 专用向量DB | 自部署/云 | 极高 | 丰富 | 亿级 |
| Pinecone | 云托管SaaS | 仅云 | 高 | 闭源 | 千万级 |
| Weaviate | 专用向量DB | 自部署/云 | 高 | 丰富 | 千万级 |
| pgvector | PG插件 | 自部署 | 中 | PG生态 | 百万级 |
| Redis VSearch | Redis内置 | 自部署/云 | 高 | Redis生态 | 百万级 |
| Elasticsearch | 搜索引擎 | 自部署/云 | 中 | ES生态 | 千万级 |

### 选型速查

```
亿级向量 + 高性能 → Milvus
百万级 + 不想引入新组件 → pgvector (PostgreSQL)
百万级 + 已有Redis → Redis VSearch
千万级 + 全托管 → Pinecone
已有ES + 向量搜索 → Elasticsearch kNN
对象存储+图+向量 → Weaviate
```

## 3. 索引算法

### 暴力搜索 vs 近似搜索

```
暴力搜索（Flat）：
  - 计算查询向量与所有向量的距离
  - 精确但慢：O(N)
  - 适用于小数据集（<10万）

近似搜索（ANN）：
  - 牺牲少量精度换取大幅性能提升
  - 主流算法：HNSW、IVF、PQ
```

### HNSW（分层导航小世界图）

```
原理：
  - 构建多层图结构，上层稀疏（快速跳转），下层密集（精确搜索）
  - 从最上层开始搜索，逐层下沉
  - 类似跳表的思想

优势：
  - 查询速度快（O(logN)）
  - 召回率高（>95%）
  - 支持动态插入

劣势：
  - 内存占用大（图结构）
  - 构建时间长

适用：对召回率和延迟都要求高的场景（最常用）
```

### IVF（倒排文件）

```
原理：
  - K-Means聚类，将向量空间划分为N个簇
  - 查询时只搜索最近的nprobe个簇
  - 牺牲少量未搜索簇中的结果

优势：
  - 内存占用小
  - 构建速度快
  - 可配合PQ压缩

劣势：
  - 召回率依赖nprobe参数
  - 簇数选择影响效果

适用：大规模数据 + 内存敏感场景
```

### PQ（乘积量化）

```
原理：
  - 将高维向量切分为多个子向量
  - 每个子向量用聚类中心编码（1字节代替4字节float）
  - 压缩比可达8-32倍

优势：
  - 极高压缩比
  - 减少内存占用

劣势：
  - 精度损失（量化误差）
  - 通常配合IVF使用（IVF_PQ）

适用：超大规模数据 + 内存受限
```

### 算法对比

| 算法 | 召回率 | 速度 | 内存 | 构建时间 | 适用 |
|------|--------|------|------|---------|------|
| Flat | 100% | 慢 | 高 | 无 | <10万 |
| HNSW | >95% | 快 | 高 | 中 | <1千万（最常用） |
| IVF_FLAT | 90-95% | 中 | 中 | 快 | <1千万 |
| IVF_PQ | 85-92% | 快 | 低 | 中 | >1千万 |

## 4. Milvus详解

### 架构

```
Milvus 2.x架构：
  ┌─────────────────────────────────────┐
  │           SDK / REST API             │
  ├─────────────────────────────────────┤
  │          Proxy (接入层)              │
  ├──────────┬──────────┬───────────────┤
  │ Query    │ Data     │ Index         │
  │ Node     │ Node     │ Node          │
  │ (查询)   │ (写入)   │ (索引构建)    │
  ├──────────┴──────────┴───────────────┤
  │     MinIO (对象存储)                  │
  │     Pulsar/Kafka (消息)              │
  │     etcd (元数据)                    │
  └─────────────────────────────────────┘
```

### Java SDK使用

```java
// Milvus Java SDK
MilvusServiceClient client = MilvusServiceClient.newBuilder()
    .withHost("localhost").withPort(19530).build();

// 1. 创建Collection
CreateCollectionParam createParam = CreateCollectionParam.newBuilder()
    .withCollectionName("articles")
    .withDescription("文章向量库")
    .addFieldType(FieldType.newBuilder()
        .withName("id")
        .withDataType(DataType.Int64)
        .withPrimaryKey(true)
        .withAutoID(true)
        .build())
    .addFieldType(FieldType.newBuilder()
        .withName("embedding")
        .withDataType(DataType.FloatVector)
        .withDimension(1536)
        .build())
    .addFieldType(FieldType.newBuilder()
        .withName("title")
        .withDataType(DataType.VarChar)
        .withMaxLength(256)
        .build())
    .build();
client.createCollection(createParam);

// 2. 创建索引（HNSW）
CreateIndexParam indexParam = CreateIndexParam.newBuilder()
    .withCollectionName("articles")
    .withFieldName("embedding")
    .withIndexType(IndexType.HNSW)
    .withMetricType(MetricType.COSINE)
    .withExtraParam("{\"M\":16,\"efConstruction\":256}")
    .build();
client.createIndex(indexParam);

// 3. 插入向量
List<InsertParam.Field> fields = new ArrayList<>();
fields.add(new InsertParam.Field("embedding", floatVectors));
fields.add(new InsertParam.Field("title", titles));
client.insert(InsertParam.newBuilder()
    .withCollectionName("articles")
    .withFields(fields)
    .build());

// 4. 向量检索
SearchParam searchParam = SearchParam.newBuilder()
    .withCollectionName("articles")
    .withVectorFieldName("embedding")
    .withFloatVectors(queryVector)
    .withTopK(10)
    .withMetricType(MetricType.COSINE)
    .withVectorFieldType(DataType.FloatVector)
    .withParams("{\"ef\":64}")
    .build();
SearchResults results = client.search(searchParam);
```

## 5. pgvector实践

### 安装与使用

```sql
-- 安装pgvector扩展
CREATE EXTENSION IF NOT EXISTS vector;

-- 创建表
CREATE TABLE articles (
    id BIGSERIAL PRIMARY KEY,
    title VARCHAR(256),
    content TEXT,
    embedding VECTOR(1536)
);

-- 创建HNSW索引
CREATE INDEX ON articles USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- 插入数据
INSERT INTO articles (title, content, embedding) 
VALUES ('Java 25新特性', '...', '[0.1, 0.2, ...]');

-- 向量检索
SELECT id, title, content, 
       1 - (embedding <=> '[0.1, 0.2, ...]') AS similarity
FROM articles
ORDER BY embedding <=> '[0.1, 0.2, ...]'
LIMIT 10;

-- <=> = 余弦距离
-- <-> = 欧氏距离
-- <#> = 负内积
```

### Spring AI + pgvector

```java
// Spring AI 自动管理pgvector
@Configuration
public class VectorConfig {
    
    @Bean
    public VectorStore vectorStore(JdbcTemplate jdbcTemplate, 
                                    EmbeddingModel embeddingModel) {
        return PgVectorStore.builder(jdbcTemplate, embeddingModel)
            .dimensions(1536)
            .distanceType(PgDistanceType.COSINE_DISTANCE)
            .indexType(PgIndexType.HNSW)
            .build();
        // Spring AI自动创建表、索引，无需手动DDL
    }
}

// 使用
@Autowired
private VectorStore vectorStore;

// 存储
vectorStore.add(List.of(
    new Document("Java 25正式发布，引入值类型...", 
        Map.of("source", "技术综述", "date", "2026-05"))
));

// 检索
List<Document> results = vectorStore.similaritySearch(
    SearchRequest.builder()
        .query("Java新特性")
        .topK(5)
        .similarityThreshold(0.7)
        .build()
);
```

## 6. 选型决策

### 决策矩阵

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| RAG知识库（百万级） | pgvector | 复用PG，运维简单 |
| RAG知识库（千万级） | Milvus | 专用DB，性能好 |
| 全托管SaaS | Pinecone | 免运维 |
| 已有ES | ES kNN | 复用ES基础设施 |
| 已有Redis | Redis VSearch | 复用Redis |
| 图片搜索（亿级） | Milvus | 高性能+大规模 |
| 小项目快速验证 | Chroma | 轻量级，嵌入式 |

### 成本对比

```
自部署Milvus：
  - 3节点集群，16核64G
  - 支持约1亿条768维向量
  - 月成本：约3000元（云服务器）

Pinecone SaaS：
  - 同等规模月费：约$2000+
  - 优势：免运维

pgvector：
  - 复用现有PostgreSQL
  - 额外成本：几乎为零
  - 限制：百万级以内性能好
```

## 7. 面试要点

### Q1: 向量数据库和传统数据库的区别？

```
传统数据库：精确匹配，B+Tree索引，WHERE/JOIN
向量数据库：相似度搜索，ANN索引，语义理解

核心差异：
  - 索引结构：B+Tree → HNSW/IVF
  - 查询方式：精确匹配 → 近似最近邻（ANN）
  - 结果：精确结果 → TopK相似结果
```

### Q2: HNSW和IVF的区别？

```
HNSW（图索引）：
  - 多层图结构，逐层搜索
  - 召回率高（>95%），查询快
  - 内存占用大
  - 适合百万级、对精度要求高

IVF（聚类索引）：
  - K-Means划分簇，搜索最近簇
  - 召回率中（90%），速度中
  - 内存占用小
  - 适合千万级、内存敏感
```

### Q3: 如何选择向量维度？

```
维度越高 → 表达能力越强，但内存和计算成本越大

常见选择：
  - OpenAI text-embedding-3-small: 1536维
  - BGE-large-zh: 1024维
  - M3E-base: 768维
  - 降维：可用PCA将高维降到低维

权衡：
  - 通用场景：768-1536维
  - 精度优先：1536-3072维
  - 性能优先：384-768维
```

### Q4: 向量检索的召回率怎么提升？

```
1. 索引参数调优：HNSW的ef越大召回越高（但更慢）
2. 混合检索：向量+BM25，RRF融合
3. 重排序：Cross-Encoder重排初步结果
4. Query改写：同义词扩展、HyDE
5. 多路召回：不同Embedding模型 + 不同索引
```

### Q5: pgvector能支撑多大规模？

```
HNSW索引 + 百万级向量：查询延迟<50ms，良好
IVF_FLAT索引 + 千万级向量：查询延迟<100ms，可接受
超过千万级：建议迁移到Milvus等专用向量DB

pgvector优势：复用PG，支持事务、JOIN、全文检索
pgvector劣势：大规模性能不如专用DB
```
