## 1. AI工作流平台概述

### 什么是AI工作流平台

```
低代码/无代码平台，通过可视化拖拽构建AI应用：

  用户输入 → [LLM节点] → [条件判断] → [工具调用] → [输出]
                ↑                         ↓
            [知识库检索] ← ← ← ← ← ←

价值：
  - 降低AI应用开发门槛
  - 快速原型验证
  - 可视化调试
  - 非技术人员可参与构建
```

### 应用场景

```
1. 智能客服：多轮对话+知识库+人工转接
2. 文档处理：解析→提取→分类→存储
3. 内容生成：选题→大纲→写作→审核
4. 数据分析：查询→分析→可视化→报告
5. 流程自动化：触发→处理→通知
```

## 2. Dify

### 核心特性

```
Dify = 开源LLM应用开发平台

功能：
  - 可视化Prompt编排
  - RAG知识库管理
  - Agent工具链
  - 工作流编排
  - 模型管理（多模型切换）
  - API发布
  - 日志和标注

部署：Docker自部署 / 云服务
```

### 工作流示例

```
Dify工作流：智能客服

  [开始] → [意图识别(LLM)] → [条件判断]
                                    ├─ 知识库问答 → [RAG检索] → [LLM生成] → [输出]
                                    ├─ 查询订单 → [API调用] → [LLM总结] → [输出]
                                    └─ 人工转接 → [通知客服] → [输出]
```

### Dify优势

```
1. 一站式：Prompt编排+RAG+Agent+发布
2. 可视化：拖拽式工作流
3. 多模型：OpenAI/Claude/Qwen/Ollama统一管理
4. 企业级：多租户、权限、审计
5. 开源：Apache 2.0，可私有部署
```

## 3. Flowise

### 核心特性

```
Flowise = LangChain的可视化版本

特点：
  - 拖拽式构建LangChain流程
  - 节点即LangChain组件（LLM/Tool/Memory/Retriever）
  - 实时预览和调试
  - API导出
  - 开源
```

### vs Dify

```
Flowise：
  - 更灵活（任意LangChain组件组合）
  - 适合开发者快速原型
  - 无知识库管理（需自己搭建）
  - 无模型管理

Dify：
  - 更完整（含RAG/知识库/模型管理/发布）
  - 适合企业级应用
  - 工作流更强大
  - 更重
```

## 7. 面试要点

### Q1: 什么时候用工作流平台，什么时候写代码？

```
用工作流平台：
  - 快速原型验证
  - 非技术人员参与
  - 流程频繁变更
  - 简单AI应用

写代码：
  - 复杂业务逻辑
  - 性能要求高
  - 需要细粒度控制
  - 企业级生产环境
```

### Q2: Dify的核心能力？

```
1. Prompt编排：可视化构建多轮对话
2. RAG知识库：文档管理+向量化+检索
3. Agent：工具调用+多步推理
4. 工作流：条件分支+循环+并行
5. 模型管理：多模型统一管理+路由
6. API发布：一键暴露为REST API
7. 运维：日志/监控/标注/评估
```

### Q3: n8n与Dify在AI场景下如何选型？

```
n8n：
  - 强项是系统集成（400+节点）+ 自动化触发
  - 适合"业务流程中嵌入AI"（邮件分类、工单分发）
  - AI是流程的一环，不是核心

Dify：
  - 强项是AI应用编排（Prompt/RAG/Agent）
  - 适合"以AI为核心的应用"（客服、知识问答）
  - 流程围绕LLM展开

判断标准：
  - 流程为主、AI为辅 → n8n
  - AI为主、流程为辅 → Dify
  - 两者都需要 → n8n做外部编排 + Dify做AI能力（通过API互调）
```

### Q4: Dify工作流与Chatflow的区别？

```
Dify两种应用类型：
  - Chatflow：对话型，多轮交互，带对话记忆
    适合：客服、助手类应用
  - Workflow：任务型，单次输入输出，无对话记忆
    适合：文档处理、数据分析、内容生成

选择：
  - 需要多轮对话 → Chatflow
  - 单次处理任务 → Workflow
```

### Q5: 这些平台如何与Java后端集成？

```
三种方式：
  1. API调用：平台暴露REST API，Java通过HTTP调用
     - Dify：/v1/chat-messages、/v1/workflows/run
     - n8n：Webhook节点暴露为HTTP接口
     - 最常用，解耦清晰

  2. Webhook回调：平台处理完回调Java服务
     - 适合长耗时任务（异步）
     - Java提供回调接口

  3. 嵌入式：通过iframe嵌入前端
     - Dify支持iframe嵌入
     - 快速上线，但定制性低

Java侧技术选型：
  - HTTP客户端：OkHttp / WebClient / RestTemplate
  - 流式响应：WebClient（SSE）处理Dify流式输出
  - Spring AI：直接集成，无需中间平台（代码级控制）
```

### Q6: Dify工作流的变量系统是怎么工作的？

```
核心机制：节点间通过变量引用传递数据

  - 每个节点定义输入变量和输出变量
  - 下游节点通过 {{node_id.output_var}} 引用上游输出
  - Start节点变量为全局入口，所有节点可访问

示例：
  [Start: question] → [LLM: answer] → [End]
  LLM节点引用：{{#start.question#}}
  End节点引用：{{#llm.answer#}}

迭代节点（Iteration）：
  - 输入一个数组，对每个元素执行子流程
  - 输出处理后的数组
  - 适合批量处理（逐段翻译/分类/摘要）
```

### Q7: 低代码AI平台在生产环境有哪些坑？

```
1. 性能：多一跳网络调用 + 平台内部编排开销
   → 高频场景考虑Spring AI直连，不走平台
   → 平台适合中低频、流程复杂的场景

2. 版本管理：工作流JSON导出，但diff不友好
   → 纳入Git管理（导出DSL）
   → 环境（开发/测试/生产）间迁移需脚本化

3. 监控盲区：平台内部执行对后端不可见
   → 开启平台日志 + 关键节点打点
   → 业务侧记录调用的traceId关联

4. 供应商锁定：工作流高度依赖平台特性
   → 核心逻辑保留代码实现能力
   → 平台做编排，业务核心用Spring AI/LangChain4j

5. 成本：LLM调用 + 平台资源 + 向量库
   → 缓存常见查询结果
   → 模型路由（简单任务用小模型）
```

## 8. Dify工作流编排详解

### 节点类型

```
Dify Workflow节点体系：

基础节点：
  - Start：定义输入变量（文本/数字/文件/下拉选项）
  - End：定义输出变量
  - Answer：直接输出文本（Chatflow专用）

逻辑节点：
  - LLM：调用大模型，支持系统/用户Prompt模板
  - Knowledge Retrieval：RAG知识库检索（多知识库）
  - Question Classifier：意图分类（多分类路由）
  - IF/ELSE：条件分支（支持多条件组合）
  - Iteration：循环迭代（处理数组元素）
  - Template Transform：Jinja2模板渲染转换
  - Code：Python代码节点（数据处理/计算）
  - Variable Aggregator：变量聚合（合并多分支输出）

工具节点：
  - HTTP Request：调用外部API/微服务
  - Tool：调用内置工具或自定义工具
```

### 编排示例：文档智能处理

```
输入：上传文档（PDF/Word）

[Start: file]
  → [LLM: 文档解析] 提取结构化内容（JSON）
  → [Code: 拆分段落] 输出段落数组
  → [Iteration: 逐段处理]
       ├─ [LLM: 分类] 判断段落类型
       ├─ [IF/ELSE: 是否需要翻译]
       │    ├─ 是 → [LLM: 翻译]
       │    └─ 否 → [直接通过]
       └─ [LLM: 摘要] 生成段落摘要
  → [Template Transform: Jinja2] 汇总为报告
  → [End: 输出报告]
```

### 关键机制

```
变量系统：
  - 每个节点有输入变量和输出变量
  - 节点间通过变量引用传递数据
  - 格式：{{#node_id.output_var#}}（Dify DSL语法）

迭代节点（Iteration）：
  - 输入一个数组，对每个元素并行/串行执行子流程
  - 输出处理后的数组
  - 适合批量处理（逐段翻译、分类、摘要）

并行执行：
  - 多个无依赖节点可并行（Dify自动调度）
  - 降低端到端延迟

错误处理：
  - 节点失败可配置重试次数
  - IF/ELSE可做异常分支兜底
  - 目前不支持全局try-catch（需用条件分支模拟）

调试：
  - 单节点运行：独立测试某个节点
  - 完整运行：端到端测试
  - 查看每节点输入输出：便于定位问题
```

## 9. n8n AI节点实战

### AI节点体系

```
n8n AI节点（基于LangChain封装）：

核心节点：
  - AI Agent：创建Agent（自动决定调用哪些工具）
  - AI Memory：对话记忆（Buffer Window/Summary/Postgres）
  - AI Language Model：LLM模型（OpenAI/Anthropic/Ollama等）

工具节点（供Agent调用）：
  - Call n8n Workflow Tool：调用其他n8n工作流（嵌套）
  - Custom Tool：自定义工具
  - HTTP Request Tool：HTTP调用外部接口
  - Calculator Tool：数学计算
  - Code Tool：执行JavaScript/Python代码

RAG相关节点：
  - Document Loader：加载文档（PDF/网页/文本）
  - Text Splitter：文本分割（递归/字符/Token）
  - Embeddings：向量化（OpenAI/Cohere/本地）
  - Vector Store：向量存储（Pinecone/Qdrant/Supabase）

输出节点：
  - Output Parser：结构化输出解析（JSON/列表）
  - Structured Output Parser：按Schema解析
```

### 实战示例：RAG问答机器人

```
n8n工作流：基于知识库的问答

【文档预处理（一次性/定时）】
  [Schedule Trigger] → [Document Loader: PDF]
    → [Text Splitter: 递归分割]
    → [Embeddings: OpenAI]
    → [Vector Store: Qdrant 存入]

【实时问答（Chat）】
  [Chat Trigger] → [AI Agent]
                    ├─ [Memory: Buffer Window]
                    ├─ [Language Model: GPT-4]
                    └─ [Tool: Vector Store 检索]
                         ├─ [Embeddings: OpenAI]
                         └─ [Vector Store: Qdrant 查询]

流程说明：
  1. 文档预处理：加载→分割→向量化→存入向量库
  2. 实时问答：用户提问→Agent判断→检索相关文档→LLM生成答案
```

### 实战示例：多步骤内容审核

```
[Webhook: 接收UGC内容]
  → [AI Agent]
       ├─ [LLM: 判断是否违规]
       ├─ [IF: 违规]
       │    ├─ 是 → [LLM: 生成违规原因] → [Slack通知] → [DB标记]
       │    └─ 否 → [LLM: 质量评分] → [DB存储] → [返回成功]
```

### n8n vs Dify AI能力对比

```
n8n AI：
  - 基于LangChain节点，灵活组合
  - 强在"AI + 系统集成"（调用各种SaaS/API/数据库）
  - 无内置知识库管理（需自建Vector Store + 文档处理流程）
  - 无统一模型管理界面（每个节点单独配模型）
  - 优势：触发方式丰富（定时/Webhook/事件/手动）

Dify AI：
  - 内置完整RAG（知识库上传→自动向量化→检索）
  - 强在"AI应用编排"（Prompt/Agent/一键发布API）
  - 模型统一管理（一个界面切换所有模型）
  - 工作流可视化更直观，非技术人员友好
  - 优势：开箱即用的知识库 + 应用发布
```

### 生产部署通用建议

```
1. 反向代理：Nginx + HTTPS + WebSocket支持（流式输出必需）
2. 数据库：独立PostgreSQL实例 + 定期备份（pg_dump）
3. 向量库：独立部署Qdrant/Milvus（生产不要用嵌入式）
4. 监控：Prometheus + Grafana（n8n/Dify均暴露指标）
5. 日志：ELK / Loki集中收集
6. 模型代理：统一通过LiteLLM/OneAPI管理多模型API Key

Dify生产特别注意：
  - Sandbox必须部署（隔离用户Code节点的Python执行）
  - 向量库单独部署（Weaviate/Qdrant/Milvus）
  - Worker可水平扩展（处理并发文档解析）

n8n生产特别注意：
  - 启用队列模式（EXECUTIONS_MODE=queue，Redis + 多Worker）
  - WEBHOOK_URL必须配置正确（否则外部回调失败）
  - 执行历史数据定期清理（避免DB膨胀）
  - 敏感凭证用Credentials管理，不硬编码
```

## 11. Java集成方案

### 方案一：API调用Dify

```java
// Dify Workflow / Chatflow API调用（WebClient流式）
@Configuration
public class DifyConfig {
    @Bean
    public WebClient difyWebClient(@Value("${dify.api-url}") String url,
                                   @Value("${dify.api-key}") String apiKey) {
        return WebClient.builder()
            .baseUrl(url)
            .defaultHeader("Authorization", "Bearer " + apiKey)
            .defaultHeader("Content-Type", "application/json")
            .codecs(c -> c.defaultCodecs().maxInMemorySize(16 * 1024 * 1024))
            .build();
    }
}

@Service
public class DifyService {

    @Autowired
    private WebClient difyWebClient;

    // 阻塞式调用Workflow（response_mode=blocking）
    public DifyResponse runWorkflow(Map<String, Object> inputs) {
        return difyWebClient.post()
            .uri("/v1/workflows/run")
            .bodyValue(Map.of(
                "inputs", inputs,
                "response_mode", "blocking",
                "user", "java-backend"
            ))
            .retrieve()
            .bodyToMono(DifyResponse.class)
            .block();
    }

    // 流式调用（SSE）—— 实时获取生成内容
    public Flux<String> runWorkflowStream(Map<String, Object> inputs) {
        return difyWebClient.post()
            .uri("/v1/workflows/run")
            .accept(MediaType.TEXT_EVENT_STREAM)
            .bodyValue(Map.of(
                "inputs", inputs,
                "response_mode", "streaming",
                "user", "java-backend"
            ))
            .retrieve()
            .bodyToFlux(String.class);
    }

    // 对话型应用（Chatflow），带会话记忆
    public Flux<String> chat(String message, String conversationId) {
        return difyWebClient.post()
            .uri("/v1/chat-messages")
            .accept(MediaType.TEXT_EVENT_STREAM)
            .bodyValue(Map.of(
                "query", message,
                "conversation_id", conversationId != null ? conversationId : "",
                "response_mode", "streaming",
                "user", "java-backend"
            ))
            .retrieve()
            .bodyToFlux(String.class);
    }
}
```

```java
// n8n Webhook调用（n8n工作流暴露为HTTP接口）
@Service
public class N8nService {

    @Autowired
    private WebClient.Builder webClientBuilder;

    @Value("${n8n.webhook-base-url}")
    private String webhookBaseUrl;

    // 同步调用n8n工作流（Webhook节点配置为Respond Immediately）
    public Mono<JsonNode> callWorkflow(String webhookPath, Map<String, Object> payload) {
        return webClientBuilder.build().post()
            .uri(webhookBaseUrl + webhookPath)
            .contentType(MediaType.APPLICATION_JSON)
            .bodyValue(payload)
            .retrieve()
            .bodyToMono(JsonNode.class);
    }

    // 异步：n8n处理后通过Webhook回调Java服务
    public Mono<JsonNode> callAsyncWorkflow(String webhookPath,
                                            Map<String, Object> payload,
                                            String callbackUrl) {
        payload.put("callback_url", callbackUrl);
        return webClientBuilder.build().post()
            .uri(webhookBaseUri + webhookPath)
            .bodyValue(payload)
            .retrieve()
            .bodyToMono(JsonNode.class);
    }
}

// n8n回调接收接口
@RestController
@RequestMapping("/api/n8n/callback")
```

### 方案三：Spring AI直连（无需中间平台）

```java
// 直接用Spring AI / LangChain4j，不依赖Dify/n8n
@RestController
@RequestMapping("/api/ai")
public class SpringAIController {

    @Autowired
    private ChatClient.Builder chatClientBuilder;

    // 简单对话
    @PostMapping("/chat")
    public String chat(@RequestBody String message) {
        return chatClientBuilder.build()
            .prompt()
            .user(message)
            .call()
            .content();
    }

    // 流式输出
    @PostMapping(value = "/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<String> chatStream(@RequestBody String message) {
        return chatClientBuilder.build()
            .prompt()
            .user(message)
            .stream()
            .content();
    }

    // 结构化输出（直接映射为Java对象）
    @PostMapping("/analyze")
    public SentimentResult analyze(@RequestBody String text) {
        return chatClientBuilder.build()
            .prompt()
            .user("分析情感倾向: " + text)
            .call()
            .entity(SentimentResult.class);
    }
}
```

### 集成方案选型

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 调用Dify API | 可视化编排、非开发可改流程 | 多一跳延迟、平台运维成本 | 流程多变、需运营参与 |
| 调用n8n Webhook | 集成丰富、自动化强 | AI能力弱于Dify | 业务流程自动化 |
| Spring AI直连 | 性能好、控制精细、无外部依赖 | 纯代码、改流程需发版 | 稳定AI逻辑、高性能场景 |

```
推荐组合策略：
  - 对外AI能力（客服/问答）→ Dify编排 + Java调用API
  - 内部流程自动化（邮件/工单）→ n8n + Java Webhook
  - 核心AI逻辑（推荐/风控/搜索）→ Spring AI直连
  - 三者通过消息队列/事件总线解耦协作
```
