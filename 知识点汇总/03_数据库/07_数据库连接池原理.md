## 1. 连接池概述

### 为什么需要连接池

```
无连接池：
  每次请求 → 创建TCP连接 → 认证 → 执行SQL → 关闭连接
  开销：TCP三次握手 + MySQL认证 + TCP四次挥手 ≈ 50-100ms

有连接池：
  启动时创建N个连接 → 复用连接 → 释放回池
  开销：仅SQL执行时间

连接池核心参数：
  - minimumIdle：最小空闲连接数
  - maximumPoolSize：最大连接数
  - connectionTimeout：获取连接超时
  - idleTimeout：空闲连接超时
  - maxLifetime：连接最大存活时间
```

## 2. HikariCP原理

### 为什么HikariCP最快

```
1. 字节码优化：使用Javassist生成代理类，比反射快
2. ConcurrentBag：自定义无锁并发集合，减少锁竞争
3. FastList替代ArrayList：无范围检查，速度更快
4. 代理类精简：Connection/Statement/ResultSet轻量代理
5. 微优化：用copyOnWrite、state等减少volatile读写
```

### ConcurrentBag

```java
// HikariCP核心数据结构：ConcurrentBag
// 无锁获取连接，ThreadLocal优先

public class ConcurrentBag<T> {
    // 共享队列（所有线程可见）
    private final CopyOnWriteArrayList<T> sharedList;
    // ThreadLocal缓存（线程本地优先）
    private final ThreadLocal<List<T>> threadList;
    
    public T borrow(long timeout) {
        // 1. 先从ThreadLocal找（无竞争，最快）
        List<T> list = threadList.get();
        for (int i = list.size() - 1; i >= 0; i--) {
            T entry = list.remove(i);
            if (entry.compareAndSet(STATE_NOT_IN_USE, STATE_IN_USE)) {
                return entry;  // 命中ThreadLocal
            }
        }
        
        // 2. 再从共享队列找
        for (T entry : sharedList) {
            if (entry.compareAndSet(STATE_NOT_IN_USE, STATE_IN_USE)) {
                return entry;  // CAS无锁获取
            }
        }
        
        // 3. 都没有，等待
        return handoffQueue.poll(timeout, TimeUnit.MILLISECONDS);
    }
}
```


## 4. 连接池调优

### 参数调优公式

```
最大连接数 = (核心数 * 2) + 有效磁盘数
  但实际更依赖业务：IO密集型需更多连接

经验值：
  - 小型应用：10-20
  - 中型应用：30-50
  - 大型应用：50-100
  - 超过100需分库分表

公式（MySQL官方建议）：
  连接数 = (CPU核心数 * 2) + 有效磁盘数
```

### 常见问题

```
问题1: 连接泄漏
  现象：连接数持续增长直到耗尽
  原因：忘记close连接
  解决：HikariCP的leakDetectionThreshold参数

问题2: 获取连接超时
  现象：Connection is not available
  原因：连接数不足或慢SQL占满连接
  解决：增大maxPoolSize + 优化慢SQL

问题3: 连接断开
  现象：Communications link failure
  原因：MySQL wait_timeout关闭了空闲连接
  解决：maxLifetime < MySQL wait_timeout（通常设为28800秒的60%）
```

## 5. 面试要点

### Q1: HikariCP为什么快？

```
1. ConcurrentBag无锁获取连接（ThreadLocal优先 + CAS）
2. FastList无范围检查
3. Javassist字节码生成代理（比反射快）
4. 精简的代理类实现
5. 避免不必要的volatile读写
```

### Q2: 连接池最大连接数怎么设？

```
不是越多越好！
  - 连接过多 → MySQL端锁竞争加剧
  - 连接过多 → 上下文切换开销

经验值：
  - 公式：(CPU核心 * 2) + 磁盘数
  - 实际：根据QPS和平均RT计算
  - QPS=1000, RT=50ms → 需要50个连接
  - 建议从20开始，逐步调优
```

### Q3: HikariCP和Druid怎么选？

```
HikariCP：性能最优，Spring Boot默认
Druid：监控功能强大，SQL防火墙，阿里生态

选HikariCP：追求性能，监控用APM替代
选Druid：需要详细SQL监控和SQL防火墙
```
