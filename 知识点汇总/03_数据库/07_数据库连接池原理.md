## 1. 连接池概述

### 为什么需要连接池

```
无连接池：
  每次请求 → 创建TCP连接 → 认证 → 执行SQL → 关闭连接
  开销：TCP三次握手 + MySQL认证 + TCP四次挥手 ≈ 50-100ms

有连接池：
  启动时创建N个连接 → 复用连接 → 释放回池
  开销：仅SQL执行时间

连接池核心参数：
  - minimumIdle：最小空闲连接数
  - maximumPoolSize：最大连接数
  - connectionTimeout：获取连接超时
  - idleTimeout：空闲连接超时
  - maxLifetime：连接最大存活时间
```

## 2. HikariCP原理

### 为什么HikariCP最快

```
1. 字节码优化：使用Javassist生成代理类，比反射快
2. ConcurrentBag：自定义无锁并发集合，减少锁竞争
3. FastList替代ArrayList：无范围检查，速度更快
4. 代理类精简：Connection/Statement/ResultSet轻量代理
5. 微优化：用copyOnWrite、state等减少volatile读写
```

### ConcurrentBag

```java
// HikariCP核心数据结构：ConcurrentBag
// 无锁获取连接，ThreadLocal优先

public class ConcurrentBag<T> {
    // 共享队列（所有线程可见）
    private final CopyOnWriteArrayList<T> sharedList;
    // ThreadLocal缓存（线程本地优先）
    private final ThreadLocal<List<T>> threadList;
    
    public T borrow(long timeout) {
        // 1. 先从ThreadLocal找（无竞争，最快）
        List<T> list = threadList.get();
        for (int i = list.size() - 1; i >= 0; i--) {
            T entry = list.remove(i);
            if (entry.compareAndSet(STATE_NOT_IN_USE, STATE_IN_USE)) {
                return entry;  // 命中ThreadLocal
            }
        }
        
        // 2. 再从共享队列找
        for (T entry : sharedList) {
            if (entry.compareAndSet(STATE_NOT_IN_USE, STATE_IN_USE)) {
                return entry;  // CAS无锁获取
            }
        }
        
        // 3. 都没有，等待
        return handoffQueue.poll(timeout, TimeUnit.MILLISECONDS);
    }
}
```

## 4. 连接池调优

### 参数调优公式

```
最大连接数 = (核心数 * 2) + 有效磁盘数
  但实际更依赖业务：IO密集型需更多连接

经验值：
  - 小型应用：10-20
  - 中型应用：30-50
  - 大型应用：50-100
  - 超过100需分库分表

公式（MySQL官方建议）：
  连接数 = (CPU核心数 * 2) + 有效磁盘数
```

### 常见问题

```
问题1: 连接泄漏
  现象：连接数持续增长直到耗尽
  原因：忘记close连接
  解决：HikariCP的leakDetectionThreshold参数

问题2: 获取连接超时
  现象：Connection is not available
  原因：连接数不足或慢SQL占满连接
  解决：增大maxPoolSize + 优化慢SQL

问题3: 连接断开
  现象：Communications link failure
  原因：MySQL wait_timeout关闭了空闲连接
  解决：maxLifetime < MySQL wait_timeout（通常设为28800秒的60%）
```

## 6. HikariCP ConcurrentBag 源码分析

### 设计背景

```
传统连接池使用BlockingQueue获取连接：
  - 多线程竞争同一把锁 → 上下文切换开销大
  - 吞吐量受限于队列锁

ConcurrentBag目标：
  - 无锁优先（ThreadLocal + CAS）
  - 只有在连接紧张时才退化到阻塞等待
  - 借出/归还的读写分离设计
```

### 核心数据结构

```java
public class ConcurrentBag<T extends IConcurrentBagEntry> {

    // 共享列表：所有线程可见，CopyOnWrite保证遍历安全
    private final CopyOnWriteArrayList<T> sharedList;

    // ThreadLocal缓存：每个线程"收藏"自己用过的连接
    // 下次borrow优先从这里取，无任何竞争
    private final ThreadLocal<List<Object>> threadList;

    // 手递手队列：连接紧张时的阻塞等待
    private final SynchronousQueue<T> handoffQueue;

    // 等待者数量（ AtomicInteger ）
    private final AtomicInteger waiters;

    // 关闭标记
    private volatile boolean closed;
}
```

### borrow：借出连接的三级策略

```java
public T borrow(long timeout, final TimeUnit timeUnit) throws InterruptedException {
    // ① 获取当前线程的ThreadLocal列表
    final List<Object> list = threadList.get();
    // ② 从ThreadLocal逆序遍历（最近用过的优先）
    for (int i = list.size() - 1; i >= 0; i--) {
        final T entry = (T) list.remove(i);
        // CAS：STATE_NOT_IN_USE → STATE_IN_USE
        // 如果CAS失败说明被其他线程抢走了（handoff场景）
        if (entry.compareAndSet(STATE_NOT_IN_USE, STATE_IN_USE)) {
            return entry;
        }
    }

    // ③ ThreadLocal没有，遍历共享列表
    final int count = sharedList.size();
    for (int i = 0; i < count; i++) {
        final T entry = sharedList.get(i);
        if (entry.compareAndSet(STATE_NOT_IN_USE, STATE_IN_USE)) {
            return entry;
        }
    }

    // ④ 共享列表也没有，进入等待
    timeout = timeUnit.toNanos(timeout);
    do {
        final long start = currentTime();
        final T entry = handoffQueue.poll(timeout, NANOSECONDS);
        if (entry == null || entry.compareAndSet(STATE_NOT_IN_USE, STATE_IN_USE)) {
            return entry;
        }
        // 被唤醒但CAS失败（被抢），重新计算剩余超时继续等待
        timeout -= (currentTime() - start);
    } while (timeout > 10_000L);

    return null;  // 超时返回null
}
```

### requite：归还连接

```java
public void requite(final T bagEntry) {
    // CAS：STATE_IN_USE → STATE_NOT_IN_USE
    bagEntry.setState(STATE_NOT_IN_USE);

    // 如果有等待线程，尝试直接递交给等待者（handoff）
    for (int i = 0; waiters.get() > 0; i++) {
        if (bagEntry.getState() != STATE_NOT_IN_USE 
            || handoffQueue.offer(bagEntry)) {
            return;
        }
        // 自旋一段时间后yield让出CPU
        if (i > 256) {
            Thread.yield();
        }
    }

    // 没有等待者，放入ThreadLocal缓存
    threadList.get().add(bagEntry);
}
```

### 状态流转

```
STATE_NOT_IN_USE (0)  ←——CAS——  STATE_IN_USE (1)
       ↑                               ↑
       |                            STATE_REMOVED (-1)
       |                               ↑
       └———remove()—————————————————┘

关键：所有状态转换都通过CAS保证原子性
  - borrow：NOT_IN_USE → IN_USE
  - requite：IN_USE → NOT_IN_USE
  - remove：任意 → REMOVED（从池中移除）
```

### 为什么快？核心总结

```
1. ThreadLocal优先：同一线程连续使用同一连接，零竞争
2. CAS无锁：共享列表遍历不持有任何锁，CAS失败即跳过
3. 读写分离：
   - requite写ThreadLocal（不阻塞borrow的读）
   - borrow读ThreadLocal（不阻塞requite的写）
4. handoff优化：有等待者时直接递交，避免归还后再借出的开销
5. CopyOnWriteArrayList：遍历时不用加锁，写时复制保证一致性
```

## 7. 连接泄漏检测

### 泄漏场景

```java
// ❌ 泄漏：忘记close，连接永远不归还
Connection conn = dataSource.getConnection();
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery("SELECT ...");
// 没有 rs.close() / stmt.close() / conn.close()
// 连接被GC前不会归还，池中可用连接逐渐减少

// ✅ 正确：try-with-resources
try (Connection conn = dataSource.getConnection();
     Statement stmt = conn.createStatement();
     ResultSet rs = stmt.executeQuery("SELECT ...")) {
    // 自动close，连接归还池
}
```

### HikariCP泄漏检测机制

```yaml
spring:
  datasource:
    hikari:
      # 泄漏检测阈值，单位毫秒，0=禁用，最小2000ms
      leak-detection-threshold: 60000
```

```java
// ProxyConnection中启动泄漏检测
public Connection getConnection() {
    final long now = System.currentTimeMillis();
    // 记录借出时间
    borrowTime = now;

    if (leakDetectionThreshold > 0) {
        // 提交定时任务，在阈值后检查
        houseKeepingExecutorService.schedule(
            new LeakTask(this, leakDetectionThreshold),
            leakDetectionThreshold, TimeUnit.MILLISECONDS);
    }
    return proxyConnection;
}

// LeakTask：泄漏检测任务
class LeakTask implements Runnable {
    public void run() {
        // 如果连接仍未归还（borrowTime没变），判定为泄漏
        if (borrowTime == lastUsed) {
            logger.warn("Apparent connection leak detected. " +
                "Connection {} has been checked out for {} ms", 
                proxyConnection, elapsed);
            // 打印获取连接时的堆栈（调试关键）
            // 可以看到是哪行代码借出后没归还
        }
    }
}
```

### 排查泄漏的步骤

```
步骤1：开启leakDetectionThreshold
  leak-detection-threshold: 30000   # 30秒

步骤2：观察日志
  WARN  c.z.h.p.ProxyLeakTask - Apparent connection leak detected.
  连接被借出超过30秒未归还，日志会打印获取连接时的堆栈

步骤3：根据堆栈定位代码
  - 通常在Service层的try-catch中忘记finally close
  - 或者在循环中获取连接但只close了最后一个

步骤4：代码修复
  - 使用try-with-resources
  - 使用Spring事务管理（@Transactional自动归还连接）
  - 使用Template模式（JdbcTemplate/MyBatis SqlSession）
```

### 生产环境建议

```
1. 泄漏检测阈值建议设为最大SQL执行时间的2-3倍
   - 正常SQL 50ms → 设为5000ms
   - 报表SQL 5s → 设为30000ms
   - 过小会误报，过大会延迟发现

2. 上线初期开启，稳定后可以关闭或调大
   - 泄漏检测有额外开销（定时任务 + 堆栈采集）

3. 配合监控告警
   - 活跃连接数 > maxPoolSize * 80% 时告警
   - 等待获取连接的线程数 > 0 时告警
```

## 8. Spring Boot 连接池配置最佳实践

### 生产级配置模板

```yaml
spring:
  datasource:
    url: jdbc:mysql://mysql-master:3306/mydb?useUnicode=true&characterEncoding=utf8&useSSL=false&serverTimezone=Asia/Shanghai&rewriteBatchedStatements=true
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
    driver-class-name: com.mysql.cj.jdbc.Driver
    hikari:
      # ===== 连接池大小 =====
      minimum-idle: 10           # 最小空闲，官方建议=min(maximumPoolSize)
      maximum-pool-size: 30      # 最大连接，根据QPS和RT计算

      # ===== 超时配置 =====
      connection-timeout: 30000      # 获取连接超时30s
      idle-timeout: 600000           # 空闲超时10min
      max-lifetime: 1800000          # 最大存活30min（必须<MySQL wait_timeout）
      validation-timeout: 5000       # 连接校验超时5s

      # ===== 连接校验 =====
      connection-test-query: SELECT 1  # 仅在不支持JDBC4时需要
      # HikariCP默认用JDBC4的isValid()，比SELECT 1更高效

      # ===== 泄漏检测 =====
      leak-detection-threshold: 30000  # 30s泄漏检测

      # ===== 其他 =====
      pool-name: MyHikariPool
      auto-commit: true
      connection-init-sql: SET NAMES utf8mb4  # 连接初始化SQL
```

### 关键参数详解

```
maximumPoolSize（最大连接数）
  - 公式：QPS * 平均RT(秒)  +  冗余
  - 示例：QPS=500, RT=50ms → 500*0.05=25，设30
  - 上限：单库不超过100，超过考虑读写分离/分库

minimumIdle（最小空闲）
  - 官方建议：等于maximumPoolSize（固定大小池）
  - 原因：动态扩缩容有创建/销毁开销
  - 低峰期固定占用连接，但避免频繁创建

maxLifetime（连接最大存活时间）
  - 必须小于MySQL的wait_timeout（默认28800s=8h）
  - 建议设为28800s的60%≈4h，即14400000ms
  - 原因：避免连接被MySQL静默关闭后报错
  - HikariCP会平滑替换（错开时间关闭，避免批量重建）

idleTimeout（空闲超时）
  - 仅当minimumIdle < maximumPoolSize时生效
  - 超过此时间的空闲连接会被关闭
  - 建议设为maxLifetime的一半

connectionTimeout（获取超时）
  - 超过此时间拿不到连接抛异常
  - 不能设太大（用户等不了太久）
  - 不能设太小（高峰期正常排队都超时）
  - 建议10-30秒
```

### 常见配置陷阱

```
陷阱1: maximumPoolSize过大
  错误：max=200，以为连接越多越好
  实际：MySQL端锁竞争加剧，RT反而上升
  正确：根据公式计算，从20开始压测调优

陷阱2: minimumIdle过小
  错误：min=1，省资源
  实际：突发流量时需要创建连接，响应变慢
  正确：min=max，固定池大小（官方推荐）

陷阱3: 忘记设置maxLifetime
  错误：使用默认值（30min）但MySQL wait_timeout被改过
  实际：连接被MySQL关闭，报Communications link failure
  正确：maxLifetime < wait_timeout * 0.6

陷阱4: 使用connection-test-query代替isValid
  错误：配置connection-test-query: SELECT 1
  实际：每次借出都执行SELECT 1，多一次RTT
  正确：依赖JDBC4的Connection.isValid()，HikariCP默认使用

陷阱5: 连接初始化SQL位置错误
  错误：在URL中拼SET NAMES，每次execute都发
  实际：浪费性能
  正确：用connection-init-sql，只在创建连接时执行一次
```

```
  ✓ 已有APM系统（SkyWalking/Pinpoint）做监控
  ✓ 团队偏好"做好一件事"的简洁设计
  ✓ 微服务架构，连接池轻量化

选 Druid：
  ✓ 需要详细SQL监控（DBA驱动开发/排查频繁）
  ✓ 需要SQL防火墙（安全要求高/外部输入多）
  ✓ 传统企业应用，需要可视化运维
  ✓ 阿里技术栈（Dubbo/Nacos等生态统一）
  ✓ 需要密码加密等安全特性

混合方案（推荐）：
  - 生产用HikariCP + SkyWalking做APM监控
  - 测试/Staging环境用Druid做SQL分析
  - 或者Druid + 关闭部分Filter（保留StatFilter）
```

## 10. 面试要点补充

### Q4: ConcurrentBag的borrow流程是怎样的？为什么要这样设计？

```
borrow流程（三级获取）：
  1. 先查ThreadLocal：无竞争，最快，利用"线程局部性"
  2. 再查共享列表：CAS无锁遍历，失败即跳过
  3. 最后handoffQueue：阻塞等待，连接紧张时才走

设计原因：
  - 80%场景命中ThreadLocal（同一线程重复获取）
  - 共享列表CAS避免BlockingQueue的全局锁
  - handoffQueue只在连接不足时阻塞，正常不进入
  - requite归还时优先handoff给等待者，减少排队
```

### Q5: 连接泄漏如何检测和排查？

```
检测：
  - HikariCP: leakDetectionThreshold参数，超时未归还打WARN日志+堆栈
  - Druid: removeAbandoned + removeAbandonedTimeout
  - 监控: active连接数持续增长不下降

排查：
  1. 开启leakDetectionThreshold=30000
  2. 看日志中的堆栈，定位到获取连接的代码行
  3. 检查该处是否try-with-resources / finally close
  4. 常见原因：异常路径忘记close、循环中只close最后一个

预防：
  - 强制使用try-with-resources
  - 使用@Transactional / JdbcTemplate管理连接生命周期
  - Code Review检查手动getConnection的场景
```

### Q6: maxLifetime为什么要小于MySQL的wait_timeout？

```
原因：
  MySQL有个wait_timeout参数（默认8小时）
  连接空闲超过这个时间，MySQL会主动关闭连接（不发通知）

  如果连接池的连接在池中空闲超过wait_timeout：
  - MySQL端已关闭
  - 连接池还以为连接有效
  - 借出后执行SQL → Communications link failure

解决方案：
  maxLifetime < wait_timeout

  建议设为wait_timeout的60%：
  - wait_timeout=28800s（8h）→ maxLifetime=17280s（4.8h）
  - 留出余量，避免边界问题

  HikariCP的maxLifetime默认30min，通常已满足
  但如果MySQL wait_timeout被改小（如60s），需要同步调整
```

### Q7: HikariCP的HouseKeeper做什么？

```
HouseKeeper是HikariCP的内部定时任务，每30秒执行一次：

职责：
  1. 清理空闲连接
     - 如果minimumIdle < 当前空闲数
     - 关闭超过idleTimeout的空闲连接

  2. 补充连接
     - 如果当前连接数 < minimumIdle
     - 创建新连接补齐到minimumIdle

  3. 连接老化处理
     - 检查连接是否超过maxLifetime
     - 平滑关闭（错开时间，避免批量重建）
     - 标记为待移除，下次borrow时跳过

  4. 维护池大小在minimumIdle ~ maximumPoolSize之间

执行频率：
  housekeeping-period-ms = 30s（可配置）
  不是实时，是定期巡检
```

### Q8: 如何排查"连接池耗尽"问题？

```
现象：
  Connection is not available, request timed out after 30000ms

排查步骤：
  1. 看监控指标
     - hikaricp.connections.active → 是否打满maxPoolSize
     - hikaricp.connections.pending → 多少线程在等连接

  2. 如果active打满：
     a. 慢SQL占连接 → 查Druid/slow query log
        解决：优化SQL / 加索引 / 拆分大事务
     b. 连接泄漏 → 开leakDetectionThreshold
        解决：修复泄漏代码
     c. 并发量突增 → QPS超预期
        解决：扩容maxPoolSize / 加缓存 / 限流

  3. 如果active未打满但拿不到连接：
     a. 连接被MySQL关闭但池不知道
        → 检查maxLifetime vs wait_timeout
     b. 网络问题导致创建连接慢
        → 检查网络延迟和MySQL负载

  4. 临时措施
     - 适当增大maxPoolSize（治标）
     - 设置获取连接超时connectionTimeout，快速失败
     - 降级/限流保护核心业务

  5. 长期方案
     - 读写分离（分散读连接）
     - 分库分表（分散写连接）
     - 加缓存减少DB访问
     - 异步化（消息队列削峰）
```
