## 1. 重构概述

```
重构 = 在不改变外部行为的前提下，改善代码内部结构

  ┌────────────────────────────────────┐
  │           重构的黄金法则             │
  │                                    │
  │  1. 先有测试，再重构                  │
  │  2. 小步修改，频繁验证               │
  │  3. 每次只做一种重构                 │
  │  4. 新功能不重构，重构不加功能        │
  └────────────────────────────────────┘
```

## 2. 重构时机判断

```
什么时候该重构？

  ✅ 代码坏味道（Code Smell）出现时：
     - 方法超过50行
     - 类超过500行
     - 圈复杂度>15
     - 重复代码>3处
     - 深层嵌套>3层
     - 参数列表>5个

  ✅ 添加新功能前：
     - 现有代码结构不支持新功能
     - 修改一处影响多处

  ✅ 修复Bug后：
     - Bug根源是设计问题

  ✅ Code Review时：
     - 审查者难以理解逻辑

  ❌ 什么时候不该重构：
     - 临近发布截止日期
     - 没有测试覆盖
     - 需要重写而非重构（架构级问题）
     - 团队不熟悉代码
```

## 3. 重构手法实战

### 3.1 提取方法

```java
// 重构前：一个方法做太多事情
public void printOwing() {
    Enumeration e = orders.elements();
    double outstanding = 0.0;
    
    System.out.println("**** 客户欠款 ****");
    System.out.println("姓名：" + name);
    
    while (e.hasMoreElements()) {
        Order each = (Order) e.nextElement();
        outstanding += each.getAmount();
    }
    
    System.out.println("应付金额：" + outstanding);
}

// 重构后：提取方法
public void printOwing() {
    printBanner();
    double outstanding = calculateOutstanding();
    printDetails(outstanding);
}

private void printBanner() {
    System.out.println("**** 客户欠款 ****");
    System.out.println("姓名：" + name);
}

private double calculateOutstanding() {
    return orders.stream()
        .mapToDouble(Order::getAmount)
        .sum();
}

private void printDetails(double outstanding) {
    System.out.println("应付金额：" + outstanding);
}
```

### 3.3 用多态替换条件

```java
// 重构前：大量if-else
public double calculatePay(Employee e) {
    switch (e.getType()) {
        case "SALARIED": return e.getMonthlySalary();
        case "HOURLY": return e.getHourlyRate() * e.getHoursWorked();
        case "COMMISSIONED": return e.getBaseSalary() + e.getCommission();
        default: throw new IllegalArgumentException("未知类型");
    }
}

// 重构后：策略模式
public interface PayCalculator {
    double calculate(Employee e);
}

public class SalariedCalculator implements PayCalculator {
    public double calculate(Employee e) { return e.getMonthlySalary(); }
}

public class HourlyCalculator implements PayCalculator {
    public double calculate(Employee e) { return e.getHourlyRate() * e.getHoursWorked(); }
}

public class CommissionedCalculator implements PayCalculator {
    public double calculate(Employee e) { return e.getBaseSalary() + e.getCommission(); }
}

// Employee持有Calculator
public class Employee {
    private PayCalculator payCalculator;
    public double calculatePay() { return payCalculator.calculate(this); }
}
```

## 4. 风险控制

```
重构风险控制清单：

  1. 测试保障
     ├── 重构前确保测试覆盖率≥70%
     ├── 每次小步修改后运行测试
     └── 关键路径必须有集成测试

  2. 分支策略
     ├── 重构在独立分支进行
     ├── 小批量频繁合并（每1-2天）
     └── Code Review必须通过

  3. 灰度发布
     ├── 重构后先灰度10%流量
     ├── 观察1-2天无异常
     └── 逐步全量

  4. 回滚方案
     ├── 保留旧代码（标记@Deprecated）
     ├── 特征开关控制新旧逻辑
     └── 出问题可秒级回滚

  5. 禁止事项
     ├── 禁止重构+加功能同时进行
     ├── 禁止无测试重构
     ├── 禁止大批量重构一次性提交
     └── 禁止在发版前重构
```

## 6. 面试题速查

**Q1: 什么时候该重构？**
```
代码坏味道出现时：方法过长、重复代码、圈复杂度高、参数过多
添加新功能前：现有结构不支持
修复Bug后：根源是设计问题
原则：新功能不重构，重构不加功能
```

**Q2: 重构的风险怎么控制？**
```
1. 先有测试（覆盖率≥70%）
2. 小步修改，频繁验证
3. 独立分支，小批量合并
4. 灰度发布，观察后全量
5. 保留回滚方案（特征开关）
```

**Q3: 提取方法的判断标准？**
```
1. 方法名能清晰表达意图（注释能表达的用方法名替代）
2. 方法长度：一屏内（约30-50行）
3. 单一职责：一个方法只做一件事
4. 避免过短的参数列表和过深的嵌套
```

**Q4: 重构和重写的区别？**
```
重构：改善内部结构，不改变外部行为，增量进行
重写：推翻重来，风险高，周期长
重构是持续过程，重写是最后手段
当重构成本 > 重写成本时，才考虑重写
```
