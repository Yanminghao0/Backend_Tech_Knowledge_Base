## 1. 日志平台概述

### 1.1 为什么需要日志平台

```
传统日志问题:
├── 日志分散在数百台服务器上，排查问题需要逐台登录
├── 日志格式不统一，难以自动化分析
├── 无法实时监控和告警
├── 历史日志难以检索
└── 无法关联多个服务的调用链

日志平台价值:
├── 统一采集: 所有日志汇聚到一个平台
├── 实时检索: 秒级搜索全量日志
├── 可视化: 仪表盘直观展示系统状态
├── 告警: 异常日志实时告警
└── 关联分析: TraceID串联全链路日志
```

### 1.2 日志分类

| 日志类型 | 来源 | 格式 | 日均量 |
|----------|------|------|--------|
| 应用日志 | Java/Go/Python应用 | JSON/文本 | 500GB |
| 访问日志 | Nginx/Apache | Combined/JSON | 200GB |
| 系统日志 | syslog/journal | 文本 | 50GB |
| 慢查询日志 | MySQL/Redis | 文本 | 10GB |
| 审计日志 | 安全/合规 | JSON | 5GB |

## 2. 整体架构设计

### 2.1 架构全景

```
┌────────────────────────────────────────────────────────────────┐
│                        日志数据源                                │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐        │
│  │App-1 │ │App-2 │ │Nginx │ │MySQL │ │Redis │ │Syslog│        │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘        │
└─────┼────────┼────────┼────────┼────────┼────────┼────────────┘
      │        │        │        │        │        │
┌─────▼────────▼────────▼────────▼────────▼────────▼────────────┐
│                    Filebeat (采集层)                             │
│              部署在每台服务器上，轻量级采集                       │
└─────────────────────┬──────────────────────────────────────────┘
                      │
┌─────────────────────▼──────────────────────────────────────────┐
│                  Kafka (缓冲层)                                  │
│              日志削峰填谷，解耦采集和处理                         │
│              Topic: app-log / access-log / slow-log             │
└─────────────────────┬──────────────────────────────────────────┘
                      │
┌─────────────────────▼──────────────────────────────────────────┐
│                Logstash (处理层)                                 │
│              解析、过滤、富化、转换                               │
└─────────────────────┬──────────────────────────────────────────┘
                      │
┌─────────────────────▼──────────────────────────────────────────┐
│              Elasticsearch (存储+检索)                           │
│              按天建索引，冷热分离                                 │
└─────────────────────┬──────────────────────────────────────────┘
                      │
┌─────────────────────▼──────────────────────────────────────────┐
│         Kibana (可视化) + ElastAlert (告警)                     │
│              仪表盘 / 搜索 / 告警规则                             │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 数据流

```
日志文件 → Filebeat采集 → Kafka缓存 → Logstash解析 → ES索引 → Kibana展示
                    ↑                                    ↓
                    ←────── 告警通知 ←── ElastAlert ←────┘
```

### 2.3 容量规划

```yaml
日均日志量: 765GB
峰值QPS: 50,000 条/秒
保留天数: 30天 (热数据7天 + 温数据23天)
压缩比: 约3:1

ES集群规划:
  数据节点: 15台 (32C 128GB 4TB SSD)
  主节点: 3台 (8C 16GB)
  协调节点: 5台 (16C 32GB)

总存储需求: 765GB * 30天 / 3 (压缩) ≈ 7.6TB (含副本约15TB)
```

```yaml
  events: 4096
```

## 4. Kafka缓冲层

### 4.1 Kafka在日志架构中的价值

```
没有Kafka:
  Filebeat → Logstash → ES
  问题: 日志洪峰时Logstash处理不过来，Filebeat会被反压阻塞
        ES写入慢时整个管道阻塞
        Logstash宕机时日志丢失

有Kafka:
  Filebeat → Kafka → Logstash → ES
  优势: 解耦采集和处理，缓冲峰值流量
        消费者可以按自己速率处理
        Logstash宕机不丢数据(Kafka持久化)
```

### 4.2 Kafka Topic设计

```yaml
# Topic设计
topics:
  - name: application    # 应用日志
    partitions: 12       # 按服务量分区
    replication: 3
    retention: 168h      # 7天
  - name: access         # 访问日志
    partitions: 6
    replication: 3
    retention: 72h       # 3天
  - name: slow_query     # 慢查询
    partitions: 3
    replication: 3
    retention: 168h
  - name: audit          # 审计日志
    partitions: 3
    replication: 3
    retention: 720h      # 30天(合规要求)
  - name: error          # 错误日志(高优先级)
    partitions: 6
    replication: 3
    retention: 168h
```

### 4.3 Kafka生产者配置(Filebeat)

```yaml
# Filebeat Kafka输出优化
output.kafka:
  hosts: ["kafka-1:9092", "kafka-2:9092", "kafka-3:9092"]
  topic: "%{[log_type]}"
  # 分区策略: 按服务名分区，同一服务日志有序
  key: "%{[service]}"
  partition.round_robin:
    reachable_only: false
  # ACK级别: all 确保不丢数据
  required_acks: 1
  # 压缩: gzip减少网络传输
  compression: gzip
  compression_level: 3
  # 批量发送
  bulk_max_size: 2048
  bulk_flush_frequency: 1s
  # 工作线程
  worker: 4
  channel_buffer_size: 256
  max_message_bytes: 1048576  # 1MB
```

## 5. Logstash过滤处理

### 5.1 Logstash配置

```ruby
# logstash.conf — 完整管道配置
input {
  kafka {
    bootstrap_servers => "kafka-1:9092,kafka-2:9092,kafka-3:9092"
    topics => ["application", "access", "slow_query", "error", "audit"]
    group_id => "logstash-consumer"
    consumer_threads => 8
    auto_offset_reset => "latest"
    decorate_events => true
  }
}

filter {
  # 根据日志类型路由到不同处理逻辑
  if [log_type] == "application" {
    # JSON解析
    json {
      source => "message"
      target => "log"
      remove_field => ["message"]
    }

    # 时间戳处理
    date {
      match => ["[@timestamp]", "ISO8601"]
      target => "@timestamp"
    }

    # 日志级别标准化
    mutate {
      lowercase => ["[log][level]"]
    }

    # 添加应用元数据
    mutate {
      add_field => {
        "platform" => "backend"
        "datacenter" => "cn-east-1"
      }
    }

    # 异常堆栈提取
    if [log][stack_trace] {
      grok {
        match => {
          "[log][stack_trace]" => [
            "(?<exception_class>[\w.]+Exception):\s*(?<exception_message>.*)"
          ]
        }
      }
    }
  }

  # Nginx访问日志解析
  if [log_type] == "access" {
    json {
      source => "message"
      remove_field => ["message"]
    }

    # 提取地理信息
    geoip {
      source => "client_ip"
      target => "geoip"
      fields => ["country_name", "region_name", "city_name"]
    }

    # User-Agent解析
    useragent {
      source => "user_agent"
      target => "ua"
    }

    # 响应时间分类
    mutate {
      convert => { "request_time" => "float" }
    }
    if [request_time] {
      ruby {
        code => '
          t = event.get("request_time")
          if t
            event.set("response_speed", t > 2 ? "slow" : (t > 1 ? "normal" : "fast"))
          end
        '
      }
    }
  }

  # MySQL慢查询解析
  if [log_type] == "slow_query" {
    grok {
      match => {
        "message" => [
          "# Time: %{TIMESTAMP_ISO8601:query_time}\n# User@Host: %{DATA:user}\[%{DATA:user}\] @ %{DATA:host} \[%{IP:client_ip}\]\n# Query_time: %{NUMBER:query_duration:float}  Lock_time: %{NUMBER:lock_duration:float}  Rows_sent: %{NUMBER:rows_sent:int}  Rows_examined: %{NUMBER:rows_examined:int}\n%{GREEDYDATA:sql_query}"
        ]
      }
    }

    # SQL指纹提取
    mutate {
      gsub => [
        "sql_query", "'[^']*'", "?",
        "sql_query", "\d+", "?"
      ]
    }
  }

  # 通用字段清理
  mutate {
    remove_field => ["@version", "input_type", "offset", "prospector"]
  }

  # 环境标记
  mutate {
    add_field => {
      "environment" => "%{[fields][env]}"
    }
  }
}

output {
  # 按日志类型输出到不同ES索引
  elasticsearch {
    hosts => ["es-data-1:9200", "es-data-2:9200", "es-data-3:9200"]
    index => "logs-%{[log_type]}-%{+YYYY.MM.dd}"
    document_type => "_doc"

    # 索引模板
    template => "/etc/logstash/templates/logstash-template.json"
    template_name => "logs-template"
    template_overwrite => true

    # 批量写入
    flush_size => 500
    idle_flush_time => 5

    # 重试策略
    retry_max_interval => 30
    retry_on_conflict => 5
  }
}
```

### 5.2 Grok解析优化

```ruby
# 自定义Grok模式
# /etc/logstash/patterns/custom_patterns

# Java异常类名
JAVA_EXCEPTION [\w.]+(?:Exception|Error|Throwable)
# 应用日志时间戳
APP_TIMESTAMP \d{4}-\d{2}-\d{2}\s\d{2}:\d{2}:\d{2}\.\d{3}

# 使用预编译Grok模式提高性能
filter {
  if [log_type] == "application" and ![log][parsed] {
    grok {
      patterns_dir => ["/etc/logstash/patterns"]
      match => {
        "message" => "%{APP_TIMESTAMP:timestamp}\s+%{LOGLEVEL:level}\s+%{JAVACLASS:logger}\s+-\s+%{GREEDYDATA:msg}"
      }
      timeout_millis => 500
      tag_on_timeout => ["_groktimeout"]
    }
    mutate { add_field => { "[log][parsed]" => "true" } }
  }
}
```

### 6.4 ES查询优化

```java
/**
 * 日志查询服务
 */
@Service
public class LogQueryService {

    @Autowired
    private RestHighLevelClient esClient;

    /**
     * 全文搜索日志
     */
    public SearchResponse searchLogs(LogSearchRequest request) throws IOException {
        SearchRequest searchRequest = new SearchRequest(getIndexPattern(request));

        SearchSourceBuilder sourceBuilder = new SearchSourceBuilder();
        BoolQueryBuilder boolQuery = QueryBuilders.boolQuery();

        // 关键词搜索
        if (StringUtils.isNotBlank(request.getKeyword())) {
            boolQuery.must(QueryBuilders.matchQuery("message", request.getKeyword()));
        }

        // 时间范围
        boolQuery.filter(QueryBuilders.rangeQuery("@timestamp")
            .gte(request.getStartTime())
            .lte(request.getEndTime()));

        // 日志级别
        if (CollectionUtils.isNotEmpty(request.getLevels())) {
            boolQuery.filter(QueryBuilders.termsQuery("level", request.getLevels()));
        }

        // 服务过滤
        if (CollectionUtils.isNotEmpty(request.getServices())) {
            boolQuery.filter(QueryBuilders.termsQuery("service", request.getServices()));
        }

        // TraceID精确查询
        if (StringUtils.isNotBlank(request.getTraceId())) {
            boolQuery.filter(QueryBuilders.termQuery("traceId", request.getTraceId()));
        }

        sourceBuilder.query(boolQuery);
        sourceBuilder.from((request.getPage() - 1) * request.getSize());
        sourceBuilder.size(request.getSize());
        sourceBuilder.sort("@timestamp", SortOrder.DESC);
        sourceBuilder.timeout("10s");

        // 高亮
        sourceBuilder.highlighter(new HighlightBuilder()
            .field("message")
            .preTags("<em>")
            .postTags("</em>"));

        // 聚合统计
        sourceBuilder.aggregation(AggregationBuilders.terms("level_agg").field("level").size(10));
        sourceBuilder.aggregation(AggregationBuilders.terms("service_agg").field("service").size(20));
        sourceBuilder.aggregation(
            AggregationBuilders.dateHistogram("timeline_agg")
                .field("@timestamp")
                .calendarInterval(DateHistogramInterval.MINUTE)
                .minDocCount(0)
        );

        searchRequest.source(sourceBuilder);
        return esClient.search(searchRequest, RequestOptions.DEFAULT);
    }

    /**
     * 根据时间范围生成索引模式
     */
    private String[] getIndexPattern(LogSearchRequest request) {
        List<String> indices = new ArrayList<>();
        LocalDate start = request.getStartTime().toLocalDate();
        LocalDate end = request.getEndTime().toLocalDate();
        while (!start.isAfter(end)) {
            indices.add("logs-application-" + start.format(DateTimeFormatter.ofPattern("yyyy.MM.dd")));
            start = start.plusDays(1);
        }
        return indices.toArray(new String[0]);
    }
}
```

## 10. 面试题速查

**Q1: ELK日志平台的完整数据流是什么？**

应用产生日志文件 → Filebeat采集(轻量级Agent部署在每台机器) → Kafka缓冲(削峰填谷) → Logstash解析过滤(Grok/JSON解析、字段提取、富化) → Elasticsearch存储索引 → Kibana可视化展示。告警通过ElastAlert/ES Watcher定时查询ES，触发条件后发送钉钉/邮件通知。

**Q2: 为什么中间要加Kafka？**

① 解耦：Filebeat不直接依赖Logstash，Logstash宕机不影响采集；② 削峰：日志洪峰时Kafka暂存，Logstash按自己速率消费；③ 持久化：Kafka持久化存储，Logstash重启不丢数据；④ 扇出：同一份日志可以被多个消费者消费(如同时做实时告警和离线分析)。

**Q3: ES日志索引怎么设计？**

按天建索引(logs-app-2026.07.13)，便于按天删除和管理。分片数根据日数据量确定(一般3-5个)，副本1个。使用索引生命周期管理(ILM)：热阶段(0-3天SSD)→温阶段(3-15天HDD)→冷阶段(15-30天)→删除(30天)。刷新间隔设30秒(日志不需要实时)。

**Q4: Filebeat如何处理Java异常堆栈的多行日志？**

配置multiline规则：pattern匹配行首时间戳，negate=true表示不匹配的行属于前一条，match=after表示追加到前一行后面。这样异常堆栈的后续行(不以时间戳开头)会被合并到第一条日志中。设置multiline.timeout=5s防止最后一条多行日志无法触发。

**Q5: 日志平台数据量太大怎么优化？**

① 采样：DEBUG日志按比例采样，不全部入库；② 分级：ERROR日志实时入库，INFO日志延迟批量入库；③ 压缩：Kafka用gzip压缩，ES开启_source压缩；④ 冷热分离：热数据SSD，冷数据HDD；⑤ 归档：超期数据归档到对象存储(S3/OSS)，ES只保留近期数据；⑥ 字段裁剪：不索引不需要搜索的字段。

**Q6: Logstash和Filebeat的区别？为什么不能只用Logstash？**

Filebeat是轻量级采集器(Go编写)，资源占用极低，只做采集和转发。Logstash是重量级处理器(JVM)，功能强大但资源占用高。如果用Logstash部署到每台服务器采集，资源开销太大。正确做法是Filebeat部署到每台机器采集，Logstash集中部署做解析处理。

**Q7: 如何保证日志不丢失？**

Filebeat：registry文件记录读取位置，重启后从断点继续。Kafka：required_acks=1，持久化存储。Logstash：auto_offset_reset=latest，手动提交offset，处理成功后才提交。ES：translog异步刷盘(30秒)，副本数1。极端情况可能丢失最后几秒日志，可接受。

**Q8: 如何做日志链路追踪？**

在请求入口生成全局traceId，通过MDC放入日志上下文，所有日志自动携带traceId。跨服务调用时通过HTTP Header传递traceId。在ES中通过traceId可以串联一次请求经过的所有服务的全部日志，快速定位问题。
