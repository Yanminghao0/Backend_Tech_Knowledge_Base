## 1. 消息队列选型

```
场景：电商平台，日均订单100万，峰值TPS 5000
需求：
  - 订单创建后异步扣库存
  - 支付结果通知
  - 日志收集
  - 要求消息不丢失、支持顺序消费

候选：Kafka / RocketMQ / RabbitMQ / Pulsar

对比：
  | 维度       | Kafka    | RocketMQ  | RabbitMQ | Pulsar   |
  |-----------|----------|-----------|----------|----------|
  | 吞吐量     | 百万级    | 十万级     | 万级      | 百万级    |
  | 消息可靠性  | 高        | 非常高     | 高        | 非常高    |
  | 顺序消息    | Partition | 支持      | 队列级    | 支持      |
  | 事务消息    | 弱        | 支持      | 不支持    | 支持      |
  | 延迟消息    | 不支持    | 支持      | 插件      | 支持      |
  | 运维复杂度  | 中        | 中        | 低        | 高        |
  | 生态       | 大数据    | 电商      | 企业集成  | 云原生    |

决策：RocketMQ
  理由：
  1. 电商场景天然匹配（阿里出品，双11验证）
  2. 事务消息解决订单+库存的分布式事务
  3. 延迟消息用于超时取消订单
  4. 顺序消息保证同一订单的操作顺序

复盘：
  优点：事务消息大幅简化了分布式事务代码
  缺点：运维比RabbitMQ复杂，Dashboard不如Kafka生态丰富
  改进：日志收集用Kafka（大数据团队已有Kafka集群）
```

## 2. 数据库选型

```
场景：SaaS平台，多租户，核心业务+数据分析
需求：
  - 交易型：订单/支付/账户（强一致）
  - 分析型：报表/统计/大屏（高并发读）
  - 日均数据增长100万行

候选：MySQL / PostgreSQL / TiDB / MongoDB

对比：
  | 维度       | MySQL    | PostgreSQL | TiDB     | MongoDB  |
  |-----------|----------|-----------|----------|----------|
  | 事务ACID   | 强       | 强         | 强       | 弱(4.0+) |
  | 分库分表   | 需中间件  | 需中间件   | 原生分布式| 内置分片  |
  | HTAP      | 不支持   | 不支持     | 支持      | 不支持    |
  | JSON支持   | 基础     | 强         | 强        | 原生     |
  | 运维成熟度  | 非常成熟  | 成熟       | 较新      | 成熟     |
  | 生态       | 最广     | 广         | 发展中    | 广       |

决策：MySQL（交易型）+ ClickHouse（分析型）
  理由：
  1. 交易型用MySQL：团队最熟悉、生态最成熟、运维成本低
  2. 分库分表用ShardingSphere（透明化分库分表）
  3. 分析型用ClickHouse：列存、查询快几十倍、MySQL同步简单
  4. 没选TiDB：团队没有分布式DB运维经验，风险高
  5. 没选PostgreSQL：虽然功能强但招人难

复盘：
  优点：MySQL+ClickHouse组合稳定，分库分表扛住了3年增长
  缺点：ShardingSphere跨库JOIN痛苦，后期改用CQRS
  教训：如果重来，业务初期就用TiDB（避免分库分表的痛苦）
```

## 3. 缓存选型

```
场景：高并发读的商品详情页，QPS 10000+
需求：
  - 商品信息缓存（读多写少）
  - 库存缓存（高频更新）
  - 热点数据本地缓存

候选：Redis / Caffeine / Memcached / 多级缓存

对比：
  | 维度       | Redis    | Caffeine  | Memcached |
  |-----------|----------|-----------|-----------|
  | 数据结构   | 丰富     | Map       | 简单KV    |
  | 持久化     | 支持     | 不支持     | 不支持    |
  | 分布式     | 原生     | 本地       | 原生      |
  | 性能       | 10万QPS  | 百万QPS    | 10万QPS   |
  | 过期策略   | 惰性+定期 | W-TinyLFU | LRU       |
  | 内存效率   | 中       | 高         | 高        |

决策：Caffeine（本地）+ Redis（分布式）多级缓存
  理由：
  1. Caffeine本地缓存抗热点：百万QPS，无网络开销
  2. Redis分布式缓存一致性：Caffeine miss后查Redis
  3. 多级缓存：Caffeine → Redis → DB
  4. 没选Memcached：数据结构太简单，不如Redis

  架构：
  请求 → Caffeine(miss) → Redis(miss) → DB → 回填
  
复盘：
  优点：Caffeine扛住了热点商品80%的流量
  缺点：Caffeine缓存一致性需要消息通知（Redis Pub/Sub）
  改进：热点商品预热，非热点只走Redis
```

## 4. 微服务框架选型

```
场景：新项目，20+微服务，Java技术栈
需求：
  - 服务注册发现
  - 配置中心
  - 流量控制/熔断降级
  - 分布式事务
  - API网关

候选：Spring Cloud Alibaba / Spring Cloud Netflix / Dubbo

对比：
  | 维度       | SC Alibaba | SC Netflix | Dubbo    |
  |-----------|-----------|-----------|----------|
  | 注册中心   | Nacos     | Eureka    | Nacos/ZK |
  | 配置中心   | Nacos     | Config    | Apollo   |
  | 网关      | Gateway   | Zuul      | Dubbo3   |
  | 熔断      | Sentinel  | Hystrix   | Sentinel |
  | 事务      | Seata     | 无        | Seata    |
  | RPC协议   | HTTP      | HTTP      | Dubbo/Tri|
  | 维护状态   | 活跃      | 停止维护   | 活跃     |
  | 性能      | 中        | 中        | 高       |

决策：Spring Cloud Alibaba + Dubbo3
  理由：
  1. Nacos统一注册+配置（一套搞定）
  2. Sentinel流量控制（规则丰富、Dashboard好用）
  3. 内部服务间调用用Dubbo3（Triple协议，性能高）
  4. 外部请求走Gateway（HTTP/REST）
  5. Seata解决分布式事务（AT模式免侵入）
  6. 没选Netflix：已停止维护，Eureka无配置中心

复盘：
  优点：Nacos+Sentinel+Seata全家桶减少了技术碎片
  缺点：Dubbo3 Triple协议文档少，踩坑多
  改进：内部RPC全部用Triple，对外用Gateway+REST
```

## 5. 部署方案选型

```
场景：20+微服务，需要高可用、弹性伸缩、灰度发布
需求：
  - 自动伸缩
  - 滚动更新/蓝绿/金丝雀
  - 配置管理
  - 日志收集

候选：物理机+脚本 / Docker Compose / Kubernetes / Serverless

对比：
  | 维度       | 物理机    | Docker Compose | K8s      | Serverless |
  |-----------|----------|---------------|----------|-----------|
  | 弹性伸缩   | 手动      | 手动           | 自动     | 自动       |
  | 滚动更新   | 脚本      | 简单           | 原生     | 原生       |
  | 灰度发布   | 困难      | 困难           | 原生     | 原生       |
  | 运维成本   | 高        | 低             | 高(初期) | 低         |
  | 资源利用   | 低        | 中             | 高       | 最高       |
  | 学习曲线   | 低        | 低             | 陡峭     | 陡峭       |
  | 冷启动     | 无        | 无             | 秒级     | 秒~分钟    |

决策：Kubernetes
  理由：
  1. 20+服务需要统一编排和管理
  2. HPA自动伸缩应对流量波动
  3. 原生滚动更新+Argo Rollouts金丝雀
  4. Helm管理配置，ConfigMap/Secret管理敏感信息
  5. 生态成熟：监控(Prometheus)、日志(ELK)、网格(Istio)
  6. 没选Serverless：Java冷启动太慢，不适合
  7. 没选Docker Compose：不支持多节点编排

复盘：
  优点：K8s自动化运维，弹性伸缩节省30%服务器
  缺点：初期学习成本高，排查问题需要熟悉K8s
  教训：先上Docker Compose跑通业务，再迁移K8s
```

## 6. 面试题速查

**Q1: 技术选型的核心原则？**
```
1. 适合团队：技术栈匹配、招人容易、学习成本可控
2. 适合业务：满足性能/可用性需求，不过度设计
3. 成熟稳定：生产验证过、社区活跃、文档完善
4. 可运维：监控告警完善、排障工具齐全
5. 可演进：不会锁死技术栈，有迁移路径
```

**Q2: 为什么选RocketMQ而不选Kafka？**
```
电商场景需要：事务消息（订单+库存）、延迟消息（超时取消）、顺序消息
Kafka不支持事务消息和延迟消息
RocketMQ天生为电商设计（阿里双11验证）
但日志收集用Kafka（大数据生态更好）
```

**Q3: 什么时候用TiDB替代MySQL？**
```
当MySQL分库分表维护成本 > TiDB运维成本时：
  - 数据量TB级，分库分表>16库
  - 跨库JOIN频繁，ShardingSphere搞不定
  - 需要HTAP（同时支持OLTP+OLAP）
前提：团队有分布式DB运维能力
```
