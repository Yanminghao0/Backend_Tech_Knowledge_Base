## 1. 多级缓存架构概述

### 为什么需要多级缓存

| 缓存层级 | 技术 | 延迟 | 容量 | 特点 |
|---------|------|------|------|------|
| L1 本地缓存 | Caffeine | ~1μs | GB级 | 最快，无网络开销，单机隔离 |
| L2 分布式缓存 | Redis | ~1ms | TB级 | 跨实例共享，一致性保证 |
| L3 数据库 | MySQL | ~10ms | PB级 | 持久化存储，最终数据源 |

### 单一缓存的问题

```
仅用Redis：
  - 每次请求都有网络IO开销
  - Redis故障时全部穿透到数据库
  - 热点Key造成Redis压力集中

仅用Caffeine：
  - 数据不跨实例共享
  - 应用重启后缓存丢失
  - 多实例间数据不一致

多级缓存：
  - L1拦截大部分读请求，减少Redis压力
  - L2保证多实例数据共享和一致性
  - 故障降级：L2不可用时L1兜底
```

## 2. 架构设计

### 整体架构

```
┌─────────────────────────────────────────────┐
│                  应用层                      │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐    │
│  │ App-1   │  │ App-2   │  │ App-3   │    │
│  │ L1缓存  │  │ L1缓存  │  │ L1缓存  │    │
│  │Caffeine │  │Caffeine │  │Caffeine │    │
│  └────┬────┘  └────┬────┘  └────┬────┘    │
│       │            │            │          │
└───────┼────────────┼────────────┼──────────┘
        │            │            │
        └────────────┼────────────┘
                     │
              ┌──────┴──────┐
              │  L2 分布式   │
              │   缓存Redis  │
              └──────┬──────┘
                     │
              ┌──────┴──────┐
              │  L3 数据库   │
              │    MySQL    │
              └─────────────┘
```

### 读流程

```
1. 查L1(Caffeine) → 命中 → 返回
2. L1未命中 → 查L2(Redis) → 命中 → 回填L1 → 返回
3. L2未命中 → 查L3(MySQL) → 命中 → 回填L2+L1 → 返回
4. L3未命中 → 返回空或默认值
```

### 写流程

```
方案A: Cache-Aside（推荐）
1. 更新数据库
2. 删除Redis缓存
3. 广播清除本地缓存

方案B: Write-Through
1. 更新数据库
2. 更新Redis
3. 广播更新本地缓存
```

## 3. 缓存一致性方案

### 多实例本地缓存同步

```
问题：App-1更新了数据，清除了自己的L1缓存，但App-2/App-3的L1缓存还是旧数据

方案1: Redis Pub/Sub广播
  - App-1更新后通过Redis发布缓存失效消息
  - App-2/App-3订阅消息，清除对应L1缓存

方案2: 消息队列
  - 通过MQ广播缓存失效事件
  - 各实例消费消息清除L1缓存

方案3: 短TTL兜底
  - L1缓存设置较短TTL（如30秒）
  - 即使广播丢失，30秒后L1自动过期
```

### 一致性保证

```java
// Cache-Aside + 消息广播 完整流程
public void updateProduct(Long id, Product product) {
    // 1. 更新数据库
    productMapper.updateById(product);
    
    // 2. 删除Redis
    redisTemplate.delete("product:" + id);
    
    // 3. 广播清除本地缓存
    cacheEventPublisher.publish(new CacheInvalidEvent("product:" + id));
}
```

## 4. 缓存异常处理

### 多级缓存下的穿透/击穿/雪崩

| 问题 | 场景 | L1对策 | L2对策 |
|------|------|--------|--------|
| 缓存穿透 | 查询不存在的数据 | 缓存空值（短TTL） | 布隆过滤器 |
| 缓存击穿 | 热点Key过期 | L1未过期则L1兜底 | 互斥锁+双重检查 |
| 缓存雪崩 | 大量Key同时过期 | TTL加随机值 | 多级TTL策略 |

### 降级策略

```java
public Product getProduct(Long id) {
    try {
        // 1. L1: Caffeine
        Product product = caffeineCache.getIfPresent(id);
        if (product != null) return product;
        
        // 2. L2: Redis (降级：Redis不可用时跳过)
        try {
            product = redisTemplate.opsForValue().get("product:" + id);
            if (product != null) {
                caffeineCache.put(id, product);  // 回填L1
                return product;
            }
        } catch (Exception e) {
            log.warn("Redis不可用，降级到数据库: {}", e.getMessage());
        }
        
        // 3. L3: MySQL
        product = productMapper.selectById(id);
        if (product != null) {
            // 回填L2+L1
            redisTemplate.opsForValue().set("product:" + id, product, 30, TimeUnit.MINUTES);
            caffeineCache.put(id, product);
        } else {
            // 缓存空值防穿透
            caffeineCache.put(id, Product.EMPTY, 5, TimeUnit.SECONDS);
        }
        return product;
    } catch (Exception e) {
        log.error("多级缓存查询异常", e);
        return null;  // 最终降级
    }
}
```

## 5. Spring Boot实现

### 配置类

```java
@Configuration
public class MultiLevelCacheConfig {

    // L1: Caffeine
    @Bean
    public Cache<Long, Product> productCaffeineCache() {
        return Caffeine.newBuilder()
                .maximumSize(10_000)
                .expireAfterWrite(30, TimeUnit.SECONDS)  // 短TTL
                .recordStats()
                .build();
    }

    // L2: Redis
    @Bean
    public RedisTemplate<String, Object> redisTemplate(
            RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        template.setKeySerializer(new StringRedisSerializer());
        template.setValueSerializer(new GenericJackson2JsonRedisSerializer());
        return template;
    }
}
```

### 多级缓存服务

```java
@Service
public class MultiLevelCacheService {

    @Autowired
    private Cache<Long, Product> caffeineCache;
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    @Autowired
    private ProductMapper productMapper;
    
    @Autowired
    private CacheEventPublisher cacheEventPublisher;
    
    private static final String CACHE_PREFIX = "product:";
    private static final long REDIS_TTL = 30;  // 分钟
    private static final long CAFFEINE_TTL = 30; // 秒
    
    public Product getProduct(Long id) {
        // L1
        Product product = caffeineCache.getIfPresent(id);
        if (product != null) {
            return product;
        }
        
        // L2
        String key = CACHE_PREFIX + id;
        product = (Product) redisTemplate.opsForValue().get(key);
        if (product != null) {
            caffeineCache.put(id, product);
            return product;
        }
        
        // L3
        product = productMapper.selectById(id);
        if (product != null) {
            redisTemplate.opsForValue().set(key, product, REDIS_TTL, TimeUnit.MINUTES);
            caffeineCache.put(id, product);
        }
        return product;
    }
    
    public void updateProduct(Product product) {
        // 1. 更新DB
        productMapper.updateById(product);
        // 2. 删Redis
        redisTemplate.delete(CACHE_PREFIX + product.getId());
        // 3. 广播清L1
        cacheEventPublisher.publish(
            new CacheInvalidEvent(CACHE_PREFIX + product.getId()));
    }
}
```

### 缓存事件监听

```java
@Component
public class CacheEventListener {
    
    @Autowired
    private Cache<Long, Product> caffeineCache;
    
    @RabbitListener(queues = "cache.invalidate")
    public void handleCacheInvalid(CacheInvalidEvent event) {
        // 清除本地缓存
        if (event.getKey().startsWith("product:")) {
            Long id = Long.parseLong(event.getKey().substring(8));
            caffeineCache.invalidate(id);
        }
    }
}
```

## 6. 性能对比

### 基准测试数据

| 方案 | QPS | 平均延迟 | P99延迟 | DB压力 |
|------|-----|---------|--------|--------|
| 仅MySQL | 500 | 12ms | 50ms | 100% |
| Redis + MySQL | 5,000 | 2ms | 10ms | 20% |
| Caffeine + Redis + MySQL | 50,000 | 0.1ms | 1ms | 2% |

### 缓存命中率

```
L1 Caffeine命中率: ~85%（热点数据）
L2 Redis命中率:    ~95%（L1未命中的95%）
L3 DB命中率:       ~5%（L2未命中的5%）

整体命中率: 85% + 15%*95% = 99.25%
```

## 7. 面试要点

### Q1: 多级缓存如何保证一致性？

```
1. 写操作：先更新DB，再删Redis，最后广播清L1
2. L1设置短TTL（30秒）作为兜底，即使广播失败也能最终一致
3. 强一致场景可使用分布式锁，但性能开销大，一般不用
4. 最终一致性是大多数场景的合理选择
```

### Q2: L1和L2的TTL如何设计？

```
L1(Caffeine): 30秒 — 短TTL保证最终一致性
L2(Redis):    30分钟 — 长TTL减少DB访问
L3(DB):       持久 — 数据最终来源

设计原则：L1 TTL << L2 TTL << DB TTL
```

### Q3: Redis宕机时多级缓存如何降级？

```
1. L1(Caffeine)继续工作，命中L1的请求不受影响
2. L1未命中的请求直接查DB
3. DB压力增大，需配合限流降级
4. Redis恢复后自动回填
```

### Q4: 多级缓存的适用场景？

```
适合：
- 读多写少（读:写 > 10:1）
- 数据一致性要求不极高（可接受秒级延迟）
- 高并发场景（QPS > 5000）

不适合：
- 写频繁的场景
- 强一致性要求（金融交易）
- 数据量极大且访问分散
```
