# 知识库强制规范

> 所有对知识库的增删改操作必须严格遵守本规范，无例外

---

## 一、目录规范

### 1.1 一级目录

- 格式: `NN_目录名称`，NN为两位数字编号
- 编号从00开始，连续递增，不得跳号
- 00_综合 为综合索引区，99_计划 为规划管理区
- 技术领域目录从01开始，到25（当前），新增目录从26开始
- 目录名使用中文，简洁明了，不带"详解"/"实战"等后缀
- 不得随意创建新目录，新增目录必须先在99_计划中提交规划

### 1.2 子目录

- 格式: `NN_子目录名称`，NN为两位数字编号
- 子目录名应带父目录的领域前缀，如06_微服务下用 `01_微服务核心组件`
- 编号从01开始，连续递增
- 不得有无编号的子目录（00_综合/归档 例外）

### 1.3 禁止事项

- 禁止创建不带编号的目录
- 禁止用"计划"/"temp"/"tmp"等非规范名称作为目录名
- 禁止在知识点汇总根目录直接放md文件（README.md除外）
- 所有规划文档统一放到 `99_计划/` 目录

---

## 二、文件命名规范

### 2.1 技术文档

- 格式: `NN_文档标题.md`，NN为两位数字编号
- 编号从01开始，连续递增，不得跳号
- 文档标题使用中文，简洁明确
- 同一目录内文件编号不得重复
- 文件名不得包含特殊字符（空格用下划线替代）

### 2.2 子目录内文件

- 格式: `N.N_文档标题.md`，如 `2.1_HashMap源码解析.md`
- 编号与父目录编号对应，如02_集合框架下的文件用 2.1, 2.2...

### 2.3 README

- 每个目录必须有 `README.md`（不含编号）
- README作为目录索引，列出所有文档及简要说明

### 2.4 规划/脚本文件

- 统一放在 `99_计划/` 目录
- 规划文档: `NN_规划标题.md`
- 脚本文件: `xxx.py` / `xxx.sh`
- 日志文件: `变更日志.md`

---

## 三、文档内容规范

### 3.1 文档结构（强制）

```markdown
# 文档标题

> 一句话描述文档核心内容

---

## 📋 目录

1. [章节一](#1-章节一)
2. [章节二](#2-章节二)
...

---

## 1. 章节一

### 1.1 子章节

内容...

---

## N. 面试题速查

**Q1: 问题？**
```
答案...
```

---

*最后更新：YYYY-MM-DD*
```

### 3.2 必须包含的部分

1. 标题（# 开头）
2. 引言（> 开头，一句话）
3. --- 分隔线
4. 📋 目录（带锚点链接）
5. 编号章节（## 1. / ### 1.1）
6. 代码块标注语言（```java / ```bash / ```yaml / ```sql / ```mermaid）
7. 面试题速查（技术文档必须）
8. 最后更新时间

### 3.3 内容质量要求

- 每篇技术文档不少于 8KB
- README不少于 2KB
- 月度综述不少于 3KB
- 全中文（代码/命令/配置除外）
- 代码示例必须可运行（语法正确）
- 面试题答案必须准确

## 四、Git提交规范

### 4.1 提交格式

```
<type>: <简要描述>

<详细说明（可选）>
```

### 4.2 type 取值

| type | 说明 |
|------|------|
| feat | 新增文档/功能 |
| fix | 修复问题（编号/命名/引用） |
| docs | 更新文档内容 |
| refactor | 重构目录结构 |
| chore | 维护性操作（脚本/配置） |

### 4.3 提交规则

- 每次操作完成后必须检查（scan_health.py）
- 检查通过后必须提交（git add -A && git commit）
- 一次提交只做一件事（新增文档/修复命名/更新引用 分开提交）
- 不得push，由用户自行push

## 五、操作流程规范

### 5.1 新增文档流程

```
1. 确认目标目录是否存在 → 不存在则先在99_计划提交规划
2. 确认文件编号 → 读取目录现有最大编号+1
3. 创建文档 → 严格遵守文档结构规范
4. 更新README → 在目录README中添加新文档引用
5. 运行 scan_health.py → 确认无问题
6. git commit → 按提交规范
```

### 5.2 新增目录流程

```
1. 在99_计划中提交新目录规划 → 说明目录名/编号/内容范围
2. 创建目录 → NN_目录名称
3. 创建README.md → 目录索引
4. 创建首批文档 → 至少2-3篇
5. 更新知识点汇总根README → 添加新目录引用
6. 运行 scan_health.py
7. git commit
```

### 5.3 重命名流程

```
1. 确认新命名符合命名规范
2. git mv 或 os.rename
3. 全库搜索旧名称引用 → 更新所有交叉引用
4. 更新父目录README
5. 运行 scan_health.py
6. git commit
```

### 5.4 扩充文档流程

```
1. read_file 读取现有内容
2. 保留原有结构，在现有基础上扩充
3. write_file 写入完整内容（不追加，覆盖写入）
4. 确认扩充后大小 ≥ 8KB
5. git commit
```

## 六、交叉引用规范

### 6.1 引用格式

```markdown
[文档名](./相对路径/文件名.md)
[文档名](../其他目录/文件名.md)
```

### 6.2 规则

- 使用相对路径，不得使用绝对路径
- 目录重命名后必须全库搜索更新引用
- 文档间有内容交叉时，在各自文档中添加"详见"链接
- README中必须列出本目录所有文档的链接

---

## 七、自更新规范

### 7.1 定时任务

| 任务 | 频率 | 类型 | 说明 |
|------|------|------|------|
| 健康扫描 | 每周一09:00 | 纯脚本 | scan_health.py，静默运行 |
| 月度技术综述 | 每月1日10:00 | LLM驱动 | 搜索+生成+提交 |
| 内容质量提升 | 每月15日10:00 | LLM驱动 | 扩充偏薄+版本检查 |

### 7.2 变更日志

每次自更新后在 `99_计划/变更日志.md` 追加记录:
```
## YYYY-MM-DD 操作名称
- 新建/扩充/更新: 文件列表
- 提交: commit hash
```

## 八、禁止事项

1. 禁止随意创建目录（必须先规划）
2. 禁止创建不带编号的目录/文件
3. 禁止在根目录放非README的md文件
4. 禁止覆盖文档时不保留原有内容
5. 禁止不检查就提交
6. 禁止push到远程（用户自行操作）
7. 禁止删除文件不检查交叉引用
8. 禁止文档不带面试题速查（技术文档）
9. 禁止规划文档放到99_计划以外的地方
