## 1. 评审流程

### 1.1 评审生命周期

```
开发者提交PR → 自动化检查(CI) → 评审者分配 → 代码评审 → 
   ↓                                              ↓
  请求修改 ←─────────────────── 提出评审意见
   ↓                                              ↓
  修改后重新提交 → 再评审 → 通过 → 合并 → 关闭PR
```

### 1.2 提交前自检

开发者提交PR前应完成的自检清单：

```markdown
## 提交前自检清单

### 功能检查
- [ ] 代码是否实现了需求描述的所有功能点？
- [ ] 边界条件是否处理（空值、极值、并发）？
- [ ] 是否添加了对应的单元测试？
- [ ] 测试是否覆盖了正常路径和异常路径？

### 代码质量
- [ ] 是否有魔法值（硬编码的数字/字符串）？
- [ ] 命名是否清晰、符合规范？
- [ ] 是否有重复代码（可提取公共方法）？
- [ ] 函数长度是否合理（建议不超过50行）？
- [ ] 类的职责是否单一？

### 安全检查
- [ ] SQL是否使用参数化查询（防注入）？
- [ ] 用户输入是否做了校验和转义？
- [ ] 敏感信息是否避免出现在日志中？
- [ ] 权限检查是否到位？

### 性能检查
- [ ] 是否有N+1查询问题？
- [ ] 大数据量是否有分页？
- [ ] 是否在循环中执行数据库/网络操作？
- [ ] 缓存使用是否合理？
```

```java
// 评审时间指南：
// - 小型PR（<100行变更）：15-30分钟
// - 中型PR（100-500行变更）：30-60分钟
// - 大型PR（500-1000行变更）：1-2小时，建议分多次评审
```

## 2. 评审检查清单

### 2.1 通用检查项

```java
// ✅ 命名检查
public class OrderService {
    // 常量命名：全大写下划线
    private static final int MAX_RETRY_COUNT = 3;
    
    // 变量命名：见名知意
    private OrderRepository orderRepository;  // ✅ 清晰

    public List<Order> orders(Long userId) { }              // ❌ 缺少动词

// ✅ 结构检查
// - 单一职责：一个类/方法只做一件事
// - 依赖方向：避免循环依赖
// - 抽象层级：不混合高层和低层逻辑
// - 重复代码：DRY原则

// ✅ 可读性检查
// - 复杂条件表达式是否提取为有意义的方法
// - 魔法值是否替换为常量
// - 注释是否解释"为什么"而非"是什么"
```

```java
public class OrderService {

// ✅ 使用分布式锁
public void processOrder(Long orderId) {
    String lockKey = "order:lock:" + orderId;
    try {

            throw new BusinessException("订单正在处理中，请稍后重试");
```

```java
// ❌ 非线程安全的单例状态
@Service
public class OrderService {
    private List<Order> cache = new ArrayList<>();  // 非线程安全

    private final List<Order> cache = Collections.synchronizedList(new ArrayList<>());
    // 或
    private final CopyOnWriteArrayList<Order> cache = new CopyOnWriteArrayList<>();
    private final DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd");
    
    // 不可变对象天然线程安全
    private final ConcurrentHashMap<String, Order> orderCache = new ConcurrentHashMap<>();
}

// ❌ 双重检查锁缺少volatile
public class Singleton {
    private static Singleton instance;  // 缺少volatile
    
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                    // 指令重排序可能导致其他线程获取到未初始化完成的对象
                }
            }
        }
        return instance;
    }
}

// ✅ volatile防止指令重排序
private static volatile Singleton instance;
```

### 3.3 死锁检查

```java
// ❌ 嵌套锁可能导致死锁
public class TransferService {
    public void transfer(Account from, Account to, BigDecimal amount) {
        synchronized (from) {
            synchronized (to) {  // 另一线程可能反向锁：to -> from
                from.debit(amount);
                to.credit(amount);
            }
        }
    }
}

// ✅ 按固定顺序加锁
public void transfer(Account from, Account to, BigDecimal amount) {
    Account first = from.getId() < to.getId() ? from : to;
    Account second = from.getId() < to.getId() ? to : from;
    
    synchronized (first) {
        synchronized (second) {
            from.debit(amount);
            to.credit(amount);
        }
    }
}

// ✅ 使用tryLock避免死锁
public void transfer(Account from, Account to, BigDecimal amount) throws InterruptedException {
    while (true) {
        if (from.getLock().tryLock(1, TimeUnit.SECONDS)) {
            try {
                if (to.getLock().tryLock(1, TimeUnit.SECONDS)) {
                    try {
                        from.debit(amount);
                        to.credit(amount);
                        return;
                    } finally {
                        to.getLock().unlock();
                    }
                }
            } finally {
                from.getLock().unlock();
            }
        }
    }
}
```

## 4. 异常与安全检查

### 4.1 异常处理评审

```java
// ❌ 吞异常
try {
    riskyOperation();
} catch (Exception e) {
    // 什么都没做，问题被隐藏
}

// ❌ 异常信息不够
catch (Exception e) {
    log.error("Error");  // 没有异常堆栈和上下文
}

// ❌ 捕获范围过大
catch (Throwable e) {   // 不应捕获Throwable
    log.error("Error", e);
}

// ❌ finally块中的异常掩盖try块中的异常
try {
    riskyOperation();
} finally {
    closeResource();  // 如果closeResource()抛异常，riskyOperation()的异常会被掩盖
}

// ✅ 正确的异常处理
try {
    riskyOperation();
} catch (SpecificException e) {
    log.error("操作失败: param={}", param, e);  // 包含上下文和堆栈
    throw new BusinessException(ErrorCode.OPERATION_FAILED, e);  // 包装后重新抛出
} catch (Exception e) {
    log.error("未知异常: param={}", param, e);
    throw new BusinessException(ErrorCode.INTERNAL_ERROR, e);
}

// ✅ try-with-resources自动处理资源关闭
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    // ...
} catch (SQLException e) {
    throw new DataAccessException("查询失败", e);
}
```

```java

List<User> findByName(String name);

String sql = "SELECT * FROM orders WHERE status = " + status;  // 直接拼接
jdbcTemplate.query(sql, ...);

// ✅ 参数化查询

jdbcTemplate.query("SELECT * FROM orders WHERE status = ?", 
                    new Object[]{status}, ...);

// ❌ XSS漏洞
@GetMapping("/search")
public String search(@RequestParam String keyword) {
    return "<div>搜索结果: " + keyword + "</div>";  // 未转义，XSS风险
}

// ✅ 输出转义

    return "<div>搜索结果: " + HtmlUtils.htmlEscape(keyword) + "</div>";
```

```java
// ❌ N+1查询问题
public List<OrderDTO> getOrders() {
    List<Order> orders = orderRepository.findAll();
    return orders.stream()
            .map(order -> {
                User user = userRepository.findById(order.getUserId());  // N+1！

            .collect(Collectors.toList());

// ✅ 使用JOIN FETCH或批量查询
@Query("SELECT o FROM Order o JOIN FETCH o.user")
List<Order> findAllWithUser();

// 或
public List<OrderDTO> getOrders() {
    List<Order> orders = orderRepository.findAll();
    Set<Long> userIds = orders.stream().map(Order::getUserId).collect(Collectors.toSet());
    Map<Long, User> userMap = userRepository.findAllById(userIds).stream()
            .collect(Collectors.toMap(User::getId, Function.identity()));
    
    return orders.stream()
            .map(order -> toDTO(order, userMap.get(order.getUserId())))
            .collect(Collectors.toList());
}

// ❌ 循环中执行数据库操作
for (OrderItem item : items) {
    Product product = productRepository.findById(item.getProductId());  // 循环中查询
    product.setStock(product.getStock() - item.getQuantity());
    productRepository.save(product);  // 循环中保存
}

// ✅ 批量操作
List<Long> productIds = items.stream().map(OrderItem::getProductId).collect(Collectors.toList());

        .collect(Collectors.toMap(Product::getId, Function.identity()));

for (OrderItem item : items) {

}
productRepository.saveAll(productMap.values());  // 批量保存
```

### 6.2 设计模式评审

```java
// ❌ 过度设计——简单逻辑使用复杂模式
public class SimpleCalculator {
    private CalculationStrategy strategy;  // 只有加法，不需要策略模式

// ❌ 设计不足——需要灵活扩展的地方硬编码
public class PaymentService {
    public void pay(Order order, String paymentType) {
        if ("ALIPAY".equals(paymentType)) {
            // 支付宝支付逻辑
        } else if ("WECHAT".equals(paymentType)) {
            // 微信支付逻辑
        } else if ("UNIONPAY".equals(paymentType)) {
            // 银联支付逻辑
        }
        // 每次添加新支付方式都要修改此类
    }

// ✅ 适度设计——策略模式
public interface PaymentStrategy {
    PaymentResult pay(Order order);
}

@Component("ALIPAY")
public class AlipayStrategy implements PaymentStrategy { }

@Component("WECHAT")
public class WechatPayStrategy implements PaymentStrategy { }

@Service
public class PaymentService {
    private final Map<String, PaymentStrategy> strategies;
    
    public PaymentService(Map<String, PaymentStrategy> strategies) {
        this.strategies = strategies;
    }
    
    public PaymentResult pay(Order order, String paymentType) {
        PaymentStrategy strategy = strategies.get(paymentType);
        if (strategy == null) {
            throw new IllegalArgumentException("不支持的支付方式: " + paymentType);
        }
        return strategy.pay(order);
    }
}
```

## 7. 沟通技巧

### 7.1 评审意见分类

```markdown
## 评审意见标注规范

### 🔴 必须修改 (Blocking)
- Bug：逻辑错误、边界遗漏、安全漏洞
- 必须修复后才能合并

示例：
> 🔴 这里的用户ID没有判空，如果传null会NPE。请添加空值检查。

### 🟡 建议修改 (Suggestion)
- 改进建议：性能优化、可读性提升、设计改进
- 讨论后决定是否修改

示例：
> 🟡 建议这里使用StringBuilder替代字符串拼接，在大量数据时性能更好。不过如果数据量很小也可以不改。

### 💬 提问 (Question)
- 理解确认：确认代码意图、了解背景
- 不影响合并

示例：
> 💬 这里为什么使用LinkedHashMap而不是HashMap？是否有顺序要求？

### 📝 备注 (Note)
- 信息分享：最佳实践、相关知识
- 不需要修改

示例：
> 📝 Java 16+可以使用record简化这个DTO：
> `public record OrderDTO(Long id, String orderNo, BigDecimal amount) {}`

### ✅ 赞赏 (Praise)
- 肯定好的实践
- 鼓励团队学习

示例：
> ✅ 这个抽象很优雅，把策略选择逻辑封装得很好！
```

### 7.2 沟通原则

```markdown
## 代码评审沟通原则

### 1. 对事不对人
❌ "你这段代码写得很烂"
✅ "这段代码在并发场景下可能有问题，建议使用ConcurrentHashMap"

### 2. 具体而非笼统
❌ "这个方法需要重构"
✅ "这个方法有200行，建议把验证逻辑提取到validateOrder()方法中"

### 3. 解释原因
❌ "不要用static"
✅ "建议避免使用static变量，因为它在测试间共享状态，可能导致测试相互影响"

### 4. 提供方案
❌ "这里有性能问题"
✅ "这里每次循环都查询数据库，可以改为先批量查询ID，再一次性获取所有数据"

### 5. 尊重作者
❌ "这不对，应该这样做"
✅ "我理解你的思路，但这里可能有个边界问题。你觉得用X方案怎么样？"

### 6. 知道何时线下讨论
- 超过3轮来回讨论的，应该线下/视频沟通
- 涉及架构决策的，应该邀请更多人参与
- 有分歧时，以数据和测试结果为准
```

## 9. 面试题速查

**Q1: 代码评审的核心价值是什么？**
> 1）发现缺陷：人工评审能发现自动化工具难以覆盖的逻辑问题和设计缺陷。2）知识共享：团队成员通过评审了解其他模块的代码。3）标准对齐：确保团队编码规范和设计原则的统一执行。4）持续改进：通过评审反馈促进开发者提升技能。5）质量保障：是代码进入主分支前的最后一道防线。

**Q2: 如何处理评审中的分歧？**
> 1）基于数据和事实讨论，而非个人偏好。2）如果讨论超过3轮仍未达成一致，应转为线下沟通。3）涉及架构决策时邀请技术负责人参与。4）可以采用"异议记录"方式——记录分歧点，允许暂时保留意见，后续通过实验或数据验证。5）始终以代码质量和团队标准为评判依据，而非"谁说了算"。

**Q3: 如何避免代码评审成为瓶颈？**
> 1）控制PR大小——鼓励小而频繁的PR。2）设定SLA——24小时内首次响应，72小时内完成。3）轮值评审制度——避免评审负担集中在个别人。4）自动化检查前置——让CI处理风格/格式问题，人工评审聚焦逻辑和设计。5）限制每日评审时间——避免疲劳导致质量下降。

**Q4: 评审中如何平衡"完美主义"和"效率"？**
> 区分"必须改"和"可以改"：Bug和安全问题必须改，风格偏好和微优化可记录为技术债后续处理。评估投入产出比——如果修改成本远高于收益，可以接受不完美。团队应就"最低质量标准"达成共识，避免反复打磨。

**Q5: 如何评审自己不熟悉模块的代码？**
> 1）先阅读PR描述和关联Issue理解背景。2）关注接口和契约的正确性，而非实现细节。3）从用户视角验证功能完整性。4）检查通用问题（命名、异常处理、日志等）。5）提问而非质疑——"这里这样设计的原因是什么？"6）邀请模块维护者协助评审。

**Q6: N+1查询问题如何在评审中发现？**
> 关注循环中的数据库调用：如果在for/forEach/stream.map中调用了Repository或Mapper方法，很可能存在N+1。检查ORM的关联查询是否使用了懒加载（LAZY fetch在循环中触发N次查询）。建议使用JOIN FETCH、@EntityGraph或批量查询+内存组装。

**Q7: 什么是"特征依恋"（Feature Envy）？如何识别？**
> 一个方法大量调用另一个对象的getter和方法，而非操作自身数据。识别标准：方法中超过一半的调用是针对其他对象。这通常意味着方法应该搬到它"依恋"的类中。例如OrderService的方法大量使用Customer的属性，该逻辑可能应该移到Customer类中。
