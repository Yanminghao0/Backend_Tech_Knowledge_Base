## 1. 可观测性概述

### 1.1 什么是可观测性？

**定义**：可观测性（Observability）是指通过系统外部输出（日志、指标、追踪）来理解系统内部状态的能力。

**监控 vs 可观测性**：
```
监控（Monitoring）：
- 告诉你系统出了什么问题
- 预定义的指标和仪表盘
- 被动式

可观测性（Observability）：
- 帮助你理解为什么出了问题
- 允许提出任意问题
- 主动式探索
```

### 1.2 三支柱模型

```
可观测性
├── 日志（Logging）
│   └── 离散的事件记录
│       └── 谁在什么时候做了什么
│
├── 指标（Metrics）  
│   └── 聚合的数值数据
│       └── 系统在某个时刻的状态
│
└── 追踪（Tracing）
    └── 请求的完整路径
        └── 一个请求经过了哪些服务
```

### 1.3 三支柱关系

| 维度 | 数据类型 | 基数 | 成本 | 典型用途 |
|------|---------|------|------|---------|
| 日志 | 事件 | 高 | 高 | 排查具体问题 |
| 指标 | 数值 | 低 | 低 | 监控趋势和告警 |
| 追踪 | 调用链 | 中 | 中 | 定位性能瓶颈 |

## 2. 日志（Logging）

### 2.1 日志规范

**结构化日志格式**：
```json
{
  "timestamp": "2025-12-22T10:30:00.123Z",
  "level": "INFO",
  "service": "order-service",
  "traceId": "abc123def456",
  "spanId": "span789",
  "userId": "USR-001",
  "action": "createOrder",
  "message": "Order created successfully",
  "duration": 150,
  "request": {
    "path": "/api/orders",
    "method": "POST",
    "ip": "192.168.1.100"
  }
}
```

**Java日志配置（Logback + JSON）**：
```xml
<!-- logback-spring.xml -->
<configuration>
    <appender name="JSON" class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <includeMdcKeyName>traceId</includeMdcKeyName>
            <includeMdcKeyName>spanId</includeMdcKeyName>
            <includeMdcKeyName>userId</includeMdcKeyName>
        </encoder>
    </appender>
    
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/application.json</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>logs/application.%d{yyyy-MM-dd}.json.gz</fileNamePattern>
            <maxHistory>30</maxHistory>
        </rollingPolicy>
        <encoder class="net.logstash.logback.encoder.LogstashEncoder"/>
    </appender>
</configuration>
```

### 2.3 日志级别使用规范

```
ERROR：影响业务运行的错误，需要立即处理
  → 数据库连接失败、第三方服务不可用、数据一致性问题

WARN：潜在问题，不影响当前业务但需要关注
  → 响应时间超过阈值、重试请求、资源接近上限

INFO：关键业务操作记录
  → 用户登录/登出、订单创建/支付、配置变更

DEBUG：调试信息，生产环境通常关闭
  → 方法入参出参、中间结果、详细执行路径

TRACE：非常详细的调试信息
  → SQL语句、网络请求详情、对象序列化
```

## 3. 指标（Metrics）

### 3.1 指标类型

```
Counter（计数器）：只增不减
  → 请求总数、错误总数、订单总数

Gauge（仪表盘）：可增可减
  → 当前在线用户数、队列长度、内存使用量

Histogram（直方图）：分布统计
  → 请求延迟分布、响应大小分布

Summary（摘要）：分位数统计
  → P50/P90/P95/P99延迟
```

### 3.2 Prometheus + Micrometer集成

**Spring Boot配置**：

```yaml

  endpoints:

    export:
```

**自定义指标**：
```java
@Service
public class OrderService {
    
    private final Counter orderCounter;
    private final Timer orderTimer;
    private final Gauge activeOrdersGauge;
    
    public OrderService(MeterRegistry registry) {
        this.orderCounter = Counter.builder("orders.created.total")
            .description("Total number of orders created")
            .tag("service", "order-service")
            .register(registry);
            
        this.orderTimer = Timer.builder("orders.processing.duration")
            .description("Order processing duration")
            .publishPercentiles(0.5, 0.9, 0.95, 0.99)
            .register(registry);
    }
    
    public Order createOrder(OrderRequest request) {
        return orderTimer.record(() -> {
            Order order = doCreateOrder(request);
            orderCounter.increment();
            return order;
        });
    }
}
```

### 3.3 四个黄金指标

```
1. 延迟（Latency）
   - 请求响应时间
   - 关注P50、P90、P95、P99
   - 区分成功和失败的请求延迟

2. 流量（Traffic）
   - 请求吞吐量（QPS/TPS）
   - 带宽使用量
   - 活跃连接数

3. 错误（Errors）
   - 错误率（5xx比例）
   - 异常类型分布
   - 超时率

4. 饱和度（Saturation）
   - CPU使用率
   - 内存使用率
   - 磁盘IO
   - 线程池使用率
   - 连接池使用率
```

## 4. 链路追踪（Tracing）

### 4.1 分布式追踪原理

```
客户端 → API网关 → 用户服务 → 订单服务 → 库存服务
  |          |          |          |          |
  |──Trace───|──────────|──────────|──────────|
  |──Span────|          |          |          |
  |          ├──Span────|          |          |
  |          |          ├──Span────|          |
  |          |          |          ├──Span────|

TraceID：全局唯一，贯穿整个请求链路
SpanID：每个服务调用的唯一标识
ParentSpanID：父调用的SpanID
```

### 4.2 Spring Cloud Sleuth + Zipkin

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-sleuth</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-sleuth-zipkin</artifactId>
</dependency>
```

```yaml
# application.yml

  sleuth:

    web:
      skip-pattern: /actuator.*,/health

      type: web
```

**Java Agent方式**：

**自定义追踪**：
```java
@Service
public class OrderService {
    
    @TraceCrossThread  // 跨线程追踪
    @Trace             // 自定义Span
    @Tags({
        @Tag(key = "order.id", value = "returnedObj.orderId"),
        @Tag(key = "order.amount", value = "returnedObj.amount")
    })
    public Order createOrder(OrderRequest request) {
        // 业务逻辑
        return order;
    }
    
    // 手动创建Span
    public void processPayment(Order order) {
        AbstractSpan span = ContextManager.createLocalSpan("processPayment");
        try {
            span.tag("order.id", order.getOrderId());
            // 业务处理
        } catch (Exception e) {
            span.log(e);
            throw e;
        } finally {
            ContextManager.stopSpan();
        }
    }
}
```

## 5. 告警体系

### 5.1 告警分级

```
P0（紧急）：影响核心业务，需要5分钟内响应
  → 服务完全不可用
  → 数据丢失
  → 安全漏洞被利用

P1（重要）：影响部分业务，需要30分钟内响应
  → 接口错误率 > 5%
  → 响应时间 > 3秒
  → 消息队列积压 > 10万

P2（一般）：影响较小，需要2小时内处理
  → 磁盘使用率 > 80%
  → 慢SQL数量增加
  → 告警频繁

P3（提示）：优化建议，可计划处理
  → 缓存命中率 < 70%
  → 线程池使用率 > 60%
  → 配置变更
```

### 5.2 Prometheus告警规则

```yaml
# 告警规则
groups:
- name: java-app-alerts
  rules:
  - alert: HighErrorRate
    expr: rate(http_server_requests_seconds_count{status=~"5.."}[5m]) / rate(http_server_requests_seconds_count[5m]) > 0.05
    for: 2m
    labels:
      severity: P1
    annotations:
      summary: "服务错误率过高"
      description: "{{ $labels.application }} 错误率超过5%，当前值: {{ $value | humanizePercentage }}"
      
  - alert: HighLatency
    expr: histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m])) > 3
    for: 5m
    labels:
      severity: P2
    annotations:
      summary: "P99延迟过高"
      description: "{{ $labels.application }} P99延迟超过3秒，当前值: {{ $value }}s"
      
  - alert: ServiceDown
    expr: up{job="spring-boot-apps"} == 0
    for: 1m
    labels:
      severity: P0
    annotations:
      summary: "服务不可用"
      description: "{{ $labels.instance }} 已宕机超过1分钟"
```

## 6. 最佳实践

### 6.1 日志最佳实践

```
1. 使用结构化日志（JSON格式）
2. 每条日志必须包含TraceID
3. 敏感信息脱敏（密码、手机号、身份证）
4. 合理设置日志级别
5. 避免在循环中打印日志
6. 异步日志写入
7. 日志轮转和清理策略
```

### 6.2 指标最佳实践

```
1. 使用RED方法（Rate、Error、Duration）监控服务
2. 使用USE方法（Utilization、Saturation、Errors）监控资源
3. 指标命名规范：namespace_subsystem_name_unit
4. 合理设置采集间隔（15s-60s）
5. 控制指标基数（避免高基数标签）
```

### 6.3 追踪最佳实践

```
1. 控制采样率（生产环境10%-30%）
2. 关键路径必须追踪
3. 添加业务标签（orderId、userId）
4. 跨线程、跨异步传递TraceContext
5. 追踪数据保留7-30天
```

## 7. 常见问题

### Q1: 如何选择追踪工具？

```
Zipkin：
  ✅ 轻量级，与Spring Cloud集成好
  ❌ 功能相对简单

SkyWalking：
  ✅ 功能全面，无侵入Agent
  ✅ 支持告警、拓扑图
  ❌ 资源消耗较大

Jaeger：
  ✅ 云原生，支持OpenTracing
  ✅ 性能优秀
  ❌ 功能不如SkyWalking全面
```

### Q2: 如何控制可观测性成本？

```
1. 日志：设置合理的保留期限，冷热分离
2. 指标：降采样（长期数据1分钟→5分钟→1小时）
3. 追踪：控制采样率，只保留异常追踪
4. 使用开源方案替代商业APM
5. 按重要性分级存储
```
