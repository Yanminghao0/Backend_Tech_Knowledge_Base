## 1. TCP协议概述

```
TCP = Transmission Control Protocol（传输控制协议）

特点：
  - 面向连接：先建连接再传数据
  - 可靠传输：保证数据不丢失、不重复、按序到达
  - 全双工：双向同时传输
  - 字节流：以字节为单位传输

TCP头部（20字节）：
  ┌────────┬────────┬──────────────┐
  │ 源端口   │ 目的端口 │ 序列号(32bit)  │
  ├────────┴────────┼──────────────┤
  │    序列号 seq    │  确认号 ack    │
  ├─────────────────┼──────────────┤
  │ 数据偏移│ 标志位   │   窗口大小     │
  ├─────────────────┼──────────────┤
  │   校验和         │   紧急指针     │
  └─────────────────┴──────────────┘

标志位：
  SYN: 建立连接
  ACK: 确认
  FIN: 关闭连接
  RST: 重置连接
  PSH: 推送（立即交给应用层）
  URG: 紧急
```

## 2. 三次握手

```
客户端                    服务端
  │                        │
  │  SYN, seq=x            │  LISTEN
  │ ──────────────────────→ │
  │                        │  SYN_RCVD
  │  SYN+ACK, seq=y, ack=x+1│
  │ ←────────────────────── │
  │  ESTABLISHED            │
  │  ACK, ack=y+1           │
  │ ──────────────────────→ │  ESTABLISHED
  │                        │

为什么三次而不是两次？
  两次握手：服务端收到SYN就建连接，可能建立无效连接（旧SYN延迟到达）
  三次握手：客户端必须确认服务端的SYN+ACK，确保双方都准备好了

seq和ack的关系：
  客户端发送seq=x → 服务端回复ack=x+1（表示收到了x，期望下一个是x+1）
  服务端发送seq=y → 客户端回复ack=y+1
```

## 3. 四次挥手

```
客户端                    服务端
  │  FIN, seq=u             │  ESTABLISHED
  │ ──────────────────────→ │  FIN_WAIT_1
  │  FIN_WAIT_2             │
  │  ACK, ack=u+1           │  CLOSE_WAIT
  │ ←────────────────────── │
  │                        │  （服务端处理剩余数据）
  │  FIN, seq=v             │  LAST_ACK
  │ ←────────────────────── │
  │  TIME_WAIT              │
  │  ACK, ack=v+1           │  CLOSED
  │ ──────────────────────→ │
  │  (等待2MSL)              │
  │  CLOSED                 │

为什么四次而不是三次？
  TCP全双工，每个方向需要单独关闭
  服务端收到FIN后可能还有数据要发，先ACK，发完再FIN

为什么TIME_WAIT等待2MSL？
  1. 确保最后的ACK到达服务端（如果丢了对端会重发FIN）
  2. 确保旧连接的报文消失（防止新连接收到旧数据）
  MSL = Maximum Segment Lifetime（报文最大生存时间，通常2分钟）
```

## 4. 可靠传输机制

```
1. 序列号 + 确认号
   每个字节有序列号，接收方确认收到的序列号
   
2. 超时重传
   发送后启动定时器，超时未确认则重传
   RTO(Retransmission Timeout)动态计算

3. 滑动窗口
   ┌──────────────────────────────┐
   │  已确认    │  已发送未确认  │ 可发送  │ 不可发送 │
   │           │  (窗口内)     │        │         │
   └──────────────────────────────┘
   窗口大小 = 接收方告知的可用缓冲区
   窗口滑动：收到确认后窗口右移

4. 流量控制
   接收方通过ACK中的窗口大小告知发送方
   窗口=0时发送方暂停发送（定期发探测包）

5. 快速重传
   收到3个重复ACK → 立即重传（不等超时）
```

## 5. 拥塞控制

```
拥塞控制四个算法：

1. 慢启动
   cwnd从1开始，每收到ACK翻倍（指数增长）
   到达ssthresh后转为拥塞避免

2. 拥塞避免
   cwnd线性增长（每RTT加1）
   超时 → ssthresh = cwnd/2，cwnd = 1，重新慢启动

3. 快速重传 + 快速恢复
   3个重复ACK → ssthresh = cwnd/2，cwnd = ssthresh + 3
   不回到慢启动，而是从ssthresh开始线性增长

   ┌─────────────────────────────────────┐
   │ cwnd                                 │
   │     /\    /\    /\                   │
   │    /  \  /  \  /  \   线性增长        │
   │   /    \/    \/    \─── ───          │
   │  / 慢启动(指数)                       │
   │ /                                    │
   └─────────────────────────────────────→ 时间

4. BBR (Google)
   不基于丢包，基于带宽和RTT估算
   Linux 4.9+ 内核支持
   net.ipv4.tcp_congestion_control=bbr
```

## 6. 面试题速查

**Q1: 三次握手为什么不是两次？**
```
两次握手的问题：
1. 服务端会建立无效连接（旧SYN延迟到达）
2. 无法确认客户端的接收能力
三次确保双向通信都就绪
```

**Q2: 四次挥手为什么TIME_WAIT要等2MSL？**
```
1. 确保最后的ACK到达对端（丢了则对端重发FIN）
2. 让旧连接的报文在网络中消失
TIME_WAIT过多怎么办？调整tcp_tw_reuse/tcp_tw_recycle
```

**Q3: 滑动窗口的作用？**
```
流量控制：接收方通过窗口大小告知发送方能发多少
提高吞吐量：不必等一个ACK才发下一个（流水线传输）
窗口大小=0时发送方暂停
```

**Q4: TCP和UDP的区别？**
```
TCP：面向连接、可靠、有序、字节流、有流量控制/拥塞控制
UDP：无连接、不可靠、无序、数据报、轻量
TCP适合文件传输/HTTP，UDP适合视频/直播/DNS
```
