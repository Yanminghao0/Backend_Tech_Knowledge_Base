## 1. 缓存一致性问题

```
问题：数据在DB和Cache两处存储，更新时可能不一致

更新策略选择：
  1. 先更新DB，再更新Cache
  2. 先更新Cache，再更新DB
  3. 先删除Cache，再更新DB
  4. 先更新DB，再删除Cache ← 推荐

并发问题：
  线程A更新DB → 线程B更新DB → 线程B删除Cache → 线程A删除Cache
  结果：Cache被删，下次读会从DB加载，最终一致 ✅

  线程A读Cache(miss) → 线程B更新DB → 线程B删除Cache → 线程A写旧值到Cache
  结果：Cache是旧值，DB是新值，不一致 ❌
  解决：延迟双删 / 消息队列重删
```

## 2. Cache Aside模式

```java
// Cache Aside（旁路缓存）：最常用的模式
// 读：先读缓存，miss则读DB并回填缓存
// 写：先更新DB，再删除缓存（不是更新缓存）

@Service
public class UserService {
    
    @Autowired
    private RedisTemplate<String, User> redisTemplate;
    @Autowired
    private UserMapper userMapper;
    
    private static final String KEY_PREFIX = "user:";
    
    // 读操作
    public User getUser(Long id) {
        String key = KEY_PREFIX + id;
        // 1. 先读缓存
        User user = redisTemplate.opsForValue().get(key);
        if (user != null) {
            return user;  // 缓存命中
        }
        // 2. 缓存miss，读DB
        user = userMapper.selectById(id);
        if (user != null) {
            // 3. 回填缓存（带随机过期防雪崩）
            int ttl = 1800 + ThreadLocalRandom.current().nextInt(300);
            redisTemplate.opsForValue().set(key, user, ttl, TimeUnit.SECONDS);
        } else {
            // 空值缓存防穿透
            redisTemplate.opsForValue().set(key, new User(), 60, TimeUnit.SECONDS);
        }
        return user;
    }
    
    // 写操作：先更新DB，再删除缓存
    @Transactional
    public void updateUser(Long id, UserDTO dto) {
        // 1. 更新DB
        userMapper.updateById(id, dto);
        // 2. 删除缓存（不是更新！）
        redisTemplate.delete(KEY_PREFIX + id);
    }
}
```

### 2.1 延迟双删

```java
// 解决"先删缓存再更新DB"的并发问题
@Transactional
public void updateUserWithDoubleDelete(Long id, UserDTO dto) {
    // 1. 先删除缓存
    redisTemplate.delete(KEY_PREFIX + id);
    // 2. 更新DB
    userMapper.updateById(id, dto);
    // 3. 延迟再删一次（防止并发读回填旧值）
    CompletableFuture.runAsync(() -> {
        try {
            Thread.sleep(500);  // 延迟500ms
            redisTemplate.delete(KEY_PREFIX + id);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    });
}
```

## 3. Write Through模式

```java
// Write Through：写时同时更新DB和Cache
// 优点：强一致
// 缺点：写性能差（同步双写）

public void writeThrough(Long id, User user) {
    // 同时写DB和Cache
    userMapper.updateById(id, user);
    redisTemplate.opsForValue().set(KEY_PREFIX + id, user);
    // 两步要么都成功，要么都失败（需要事务或补偿）
}
```

## 4. Write Behind模式

```java
// Write Behind（Write Back）：只写缓存，异步刷DB
// 优点：写性能极高
// 缺点：可能丢数据（缓存宕机）

public void writeBehind(Long id, User user) {
    // 1. 只写缓存
    redisTemplate.opsForValue().set(KEY_PREFIX + id, user);
    // 2. 异步刷DB（消息队列/定时任务）
    mqTemplate.asyncSend("db-flush", new FlushMessage(id, user));
}

// MQ消费者异步刷DB
@RocketMQMessageListener(topic = "db-flush", consumerGroup = "flush-group")
public class FlushConsumer implements RocketMQListener<FlushMessage> {
    @Override
    public void onMessage(FlushMessage msg) {
        userMapper.updateById(msg.getId(), msg.getUser());
    }
}
```

## 5. 一致性保障策略

```
最终一致性保障方案：

1. 延迟双删
   删缓存 → 更新DB → 延迟500ms → 再删缓存

2. 消息队列重试删除
   更新DB → 发MQ → 消费者删缓存 → 失败重试
   
3. Binlog订阅（Canal）
   DB更新 → Canal监听Binlog → 解析变更 → 删除缓存
   优点：不影响业务代码，与DB操作解耦

4. 设置合理的TTL
   所有缓存设置过期时间 → 兜底方案
   即使不一致，过期后也会从DB重新加载

5. 读写分离+缓存版本号
   写操作递增版本号 → 读操作检查版本号 → 不匹配则重读DB
```

```java
// Canal Binlog订阅方案
@Component
public class CanalCacheInvalidator {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    @CanalEventListener
    public void onTableChange(CanalEntry.EventType eventType, 
                               CanalEntry.RowData rowData) {
        if (eventType == CanalEntry.EventType.UPDATE || 
            eventType == CanalEntry.EventType.DELETE) {
            // 解析变更的行数据
            Map<String, String> afterMap = rowData.getAfterColumnsList().stream()
                .collect(Collectors.toMap(
                    CanalEntry.Column::getName,
                    CanalEntry.Column::getValue
                ));
            
            String id = afterMap.get("id");
            // 删除对应缓存
            redisTemplate.delete("user:" + id);
        }
    }
}
```

## 6. 面试题速查

**Q1: 为什么是删除缓存而不是更新缓存？**
```
1. 并发安全问题：两个线程同时更新缓存可能产生脏数据
2. 性能问题：有些缓存值计算复杂，每次更新都重算浪费
3. 懒加载：删除后下次读时按需加载，更高效
4. 一致性：删除是幂等操作，多次执行结果一致
```

**Q2: 先更新DB还是先删缓存？**
```
推荐：先更新DB，再删缓存（Cache Aside）
原因：先删缓存有并发问题（删了→读回填旧值→更新DB→不一致）
先更新DB再删缓存的问题概率更低（读比写快，并发窗口小）
```

**Q3: 如何保证缓存最终一致？**
```
1. 设置TTL兜底
2. 延迟双删
3. 消息队列重试删除
4. Canal监听Binlog自动删缓存
5. 强一致性场景用Write Through
```

**Q4: Cache Aside、Write Through、Write Behind的区别？**
```
Cache Aside：应用层管理缓存，读miss回填，写删缓存（最常用）
Write Through：缓存和DB同步写（强一致，写性能差）
Write Behind：只写缓存异步刷DB（高性能，可能丢数据）
```
