## 1. 可观测性概述

### 1.1 定义与三大支柱

微服务可观测性(Observability)是指通过外部输出了解系统内部状态的能力，使运维和开发人员能够快速识别、诊断和解决系统问题。它建立在三大支柱之上：

- **日志(Logs)**：系统事件的离散记录
- **指标(Metrics)**：系统状态的量化数据
- **追踪(Traces)**：请求在分布式系统中的路径

![可观测性三大支柱](https://i.imgur.com/xyz123.png)

### 1.2 微服务可观测性挑战

- **分布式系统复杂性**：请求跨多个服务和网络
- **数据量爆炸**：大规模集群产生海量数据
- **异构环境**：多种技术栈并存
- **动态性**：服务实例频繁扩缩容
- **故障定位困难**：单一故障可能引发连锁反应

### 1.3 核心价值

- **快速故障定位**：缩短MTTD(平均检测时间)和MTTR(平均恢复时间)
- **性能优化**：识别系统瓶颈
- **容量规划**：基于实际数据预测资源需求
- **用户体验提升**：发现并解决影响用户的问题
- **系统可靠性**：提高整体系统稳定性

## 2. 日志收集与分析

### 2.1 日志设计原则

#### 2.1.1 结构化日志

采用JSON格式记录日志，便于检索和分析：

```java
@Slf4j
@Service
public class OrderService {
    public void createOrder(OrderRequest request) {
        // 记录结构化日志
        log.info("{}", JsonUtil.toJson(
            new LogEvent()
                .setEventType("ORDER_CREATE")
                .setTraceId(MDC.get("traceId"))
                .setOrderId(request.getOrderId())
                .setUserId(request.getUserId())
                .setItems(request.getItems().size())
                .setTimestamp(LocalDateTime.now())
        ));

        // 业务逻辑...
    }
}
```

#### 2.1.2 关键日志字段

每条日志应包含以下关键字段：
- `timestamp`: 时间戳
- `level`: 日志级别
- `service`: 服务名称
- `traceId`: 分布式追踪ID
- `spanId`: 当前追踪跨度ID
- `userId`: 用户ID
- `eventType`: 事件类型
- `message`: 日志消息
- `stackTrace`: 异常堆栈(错误日志)

## 3. 指标监控体系（Prometheus）

### 3.1 Prometheus 架构

Prometheus 采用拉取(Pull)模式采集指标，核心组件包括：

- **Prometheus Server**: 指标采集与存储（时序数据库 TSDB）
- **Exporters**: 暴露指标端点（node_exporter、mysql_exporter 等）
- **Alertmanager**: 告警规则评估与通知
- **Grafana**: 指标可视化看板

### 3.2 指标类型

| 类型 | 说明 | 示例 |
|------|------|------|
| Counter | 单调递增计数器 | http_requests_total |
| Gauge | 可增减的瞬时值 | memory_usage_bytes |
| Histogram | 分布统计（分桶） | request_duration_seconds |
| Summary | 分位数统计 | request_latency_p99 |

### 3.4 自定义业务指标

```java
@Component
public class OrderMetrics {

    private final Counter orderCreatedCounter;
    private final Counter orderFailedCounter;
    private final Timer orderProcessTimer;
    private final Gauge orderPendingGauge;

    public OrderMetrics(MeterRegistry registry) {
        this.orderCreatedCounter = Counter.builder("order.created.total")
            .description("Total created orders")
            .tag("type", "order")
            .register(registry);

        this.orderFailedCounter = Counter.builder("order.failed.total")
            .description("Total failed orders")
            .register(registry);

        this.orderProcessTimer = Timer.builder("order.process.duration")
            .description("Order processing duration")
            .register(registry);

        this.orderPendingGauge = Gauge.builder("order.pending.count",
            this, OrderMetrics::getPendingCount)
            .description("Current pending order count")
            .register(registry);
    }

    public void recordOrderCreated() {
        orderCreatedCounter.increment();
    }

    public Timer.Sample startTimer() {
        return Timer.start();
    }

    public void recordProcessDuration(Timer.Sample sample) {
        sample.stop(orderProcessTimer);
    }

    private int getPendingCount() {
        return orderRepository.countByStatus("PENDING");
    }
}
```

### 3.6 PromQL 常用查询示例

```promql
# QPS - 每秒请求数

# P99 延迟

# 错误率
sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m]))
  / sum(rate(http_server_requests_seconds_count[5m]))

# CPU 使用率
100 - (avg by(instance)(irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# 内存使用率
(node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes)
  / node_memory_MemTotal_bytes * 100
```

### 4.4 手动埋点（TraceContext API）

```java
import org.apache.skywalking.apitoolkit.trace.TraceContext;
import org.apache.skywalking.apitoolkit.trace.Tag;
import org.apache.skywalking.apitoolkit.trace.Tags;
import org.apache.skywalking.apitoolkit.trace.ActiveSpan;

@Service
public class PaymentService {

    @Tag(key = "paymentMethod", value = "arg[0]")
    @Tag(key = "orderId", value = "arg[1]")
    public void processPayment(String paymentMethod, String orderId, BigDecimal amount) {
        // 获取当前 traceId，用于关联日志
        String traceId = TraceContext.traceId();
        log.info("traceId={}, Processing payment orderId={}", traceId, orderId);

        ActiveSpan.tag("payment.amount", amount.toString());

        try {
            callPaymentGateway(paymentMethod, amount);
            ActiveSpan.event("payment-success");
        } catch (Exception e) {
            Tags.STATUS_CODE.set(ActiveSpan.get(), "ERROR");
            ActiveSpan.errorLog(e);
            throw e;
        }
    }
}
```

### 5.3 统一 TraceId 传递过滤器

```java
@Component
public class TraceIdFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        String traceId = request.getHeader("X-Trace-Id");
        if (StringUtils.isBlank(traceId)) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }
        MDC.put("traceId", traceId);
        response.setHeader("X-Trace-Id", traceId);

        try {
            chain.doFilter(request, response);
        } finally {
            MDC.clear();
        }
    }
}
```

## 7. 告警与通知机制

### 7.1 Prometheus 告警规则

```yaml
# alert_rules.yml
groups:
  - name: application-alerts
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m])) by(service)
            / sum(rate(http_server_requests_seconds_count[5m])) by(service) > 0.05
        for: 2m
        labels: {severity: critical}
        annotations:
          summary: "{{ $labels.service }} 错误率过高"
          description: "服务 5xx 错误率 {{ $value | humanizePercentage }}"

      - alert: HighLatency
        expr: |
          histogram_quantile(0.99, sum(rate(http_server_requests_seconds_bucket[5m])) by(le,service)) > 1
        for: 5m
        labels: {severity: warning}
        annotations:
          summary: "{{ $labels.service }} P99 延迟超过 1s"

      - alert: PodCrashLooping
        expr: rate(kube_pod_container_status_restarts_total[15m]) > 0
        for: 5m
        labels: {severity: critical}
        annotations:
          summary: "Pod {{ $labels.pod }} 频繁重启"
```

## 8. 面试要点

### Q1: 可观测性的三大支柱是什么？它们之间如何关联？

可观测性三大支柱为 Metrics（指标）、Logging（日志）、Tracing（追踪）。

- Metrics 关注系统整体状态的聚合数据，适合监控与告警，开销低；
- Logging 记录离散事件详情，适合根因分析，但数据量大；
- Tracing 记录单个请求在分布式系统中的完整调用路径，适合链路分析与性能定位。

三者通过 traceId 关联：Metrics 提供告警触发 → Tracing 定位慢/错链路 → Logging 查看具体错误堆栈，形成闭环。实践中应优先建设 Metrics（成本最低），再补齐 Tracing，最后完善结构化 Logging。

### Q2: Prometheus 的 Pull 模式与 Push 模式相比有何优劣？

Pull 模式优势：服务端主动控制采集频率，避免客户端突发推送压垮服务端；天然支持服务发现（K8s、Consul）；目标健康状态可由 Prometheus 探测感知；调试方便（直接 curl /metrics 端点）。

劣势：不适合短生命周期任务（如 Cron Job），实例可能在拉取前已退出；防火墙/NAT 环境需配合 Pushgateway。

实践：长生命周期服务用 Pull；短任务通过 Pushgateway 中转推送，Prometheus 再从 Pushgateway 拉取。

### Q3: SkyWalking 与 Zipkin/Jaeger 相比有何优势？

SkyWalking 核心优势：
1. 无侵入字节码增强（Java Agent），业务零代码改动即可自动埋点，接入成本极低；
2. 自动生成服务拓扑图与依赖关系，无需手动配置；
3. 同时支持 Metrics + Tracing + Logging 关联（统一 OAP 后端），而非纯 Tracing 系统；
4. 对国内生态（Dubbo、Spring Cloud Alibaba）支持更完善；
5. 支持告警规则配置。

Zipkin/Jaeger 更轻量、社区标准化（OpenTracing/OpenTelemetry），适合多语言、多云环境。新项目推荐 OpenTelemetry SDK + SkyWalking/Tempo 后端。

### Q4: 分布式追踪中 TraceId 如何在服务间传递？

TraceId 通过请求上下文在服务间透传：

- HTTP 同步调用：通过请求头（如 sw8、traceparent、X-B3-TraceId）传递，下游解析后写入 MDC；
- 消息队列：TraceId 注入消息 Header（Kafka headers / RabbitMQ properties），消费者启动时从 Header 恢复上下文；
- 线程内传递：通过 ThreadLocal（SkyWalking 用 ContextManager，Sleuth 用 MDC + TraceContext）；
- 异步/线程池：需使用包装后的 Runnable（如 TraceRunnable.wrap()）或 Spring 的 TaskDecorator 将上下文复制到子线程。

核心风险：忽略异步场景会导致 Trace 断链，这是生产中最常见的追踪问题。

### Q5: 生产环境日志采样策略如何设计？

日志采样与级别策略：
1. 错误日志（ERROR/WARN）全量保留，不采样；
2. INFO 日志按 traceId 采样（如仅保留 10% 成功请求的 INFO 日志），但同一 traceId 的所有日志要么全保留要么全丢弃，保证链路完整；
3. DEBUG 日志默认关闭，通过动态配置（如 Apollo/Nacos）按需开启特定服务的 DEBUG；
4. 日志体积控制：单条日志限制大小（如 stackTrace 截断），JSON 结构化输出避免冗余字段；
5. 存储分层：热数据（7天）存 Elasticsearch，冷数据归档到对象存储（S3/OSS）。
